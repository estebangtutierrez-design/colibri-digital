Bienvenido al capítulo 26 de Java Master Book. Hasta ahora hemos visto cómo la
herencia nos permite reutilizar código y el polimorfismo nos da flexibilidad. Pero
hay un concepto fundamental que necesitamos para diseñar jerarquías robustas:
la abstracción.

La abstracción es el proceso de ocultar los detalles de implementación y mostrar
solo las funcionalidades esenciales. En Java, se logra mediante clases abstractas
y métodos abstractos.

Una clase abstracta es como un plano incompleto: define qué se debe hacer (el
contrato), pero deja algunos detalles para las subclases. Es una herramienta
poderosa para diseñar frameworks y APIs extensibles.

1. ¿QUÉ ES LA ABSTRACCIÓN?

La abstracción identifica las características esenciales de una entidad e ignora
detalles irrelevantes. En programación, define qué hace un objeto sin especificar
cómo lo hace.

Abstracción vs. Encapsulación:
- Encapsulación: oculta los datos (private, getters/setters).
- Abstracción: oculta la implementación (clases abstractas, interfaces).

En Java la abstracción se logra con:
1. Clases abstractas (parcialmente implementadas).
2. Interfaces (contratos puros, capítulo 27).

═══════════════════════════════════════════════════════════════════════════════════

2. CLASES ABSTRACTAS

Una clase abstracta:
- Se declara con abstract.
- NO puede instanciarse (no se usa new).
- Puede tener métodos abstractos y concretos.
- Puede tener constructores, atributos, métodos estáticos.
- Sirve como base para subclases.

```java
abstract class Figura {
    protected String color;

    public Figura(String color) {
        this.color = color;
    }

    public String getColor() {
        return color;
    }

    public abstract double calcularArea();

    public void mostrar() {
        System.out.println("Figura " + color + " | Área: " +
                         String.format("%.2f", calcularArea()));
    }
}
```

Si una clase tiene al menos un método abstracto, debe declararse abstract. Una
clase abstracta puede tener todos sus métodos concretos.

═══════════════════════════════════════════════════════════════════════════════════

3. MÉTODOS ABSTRACTOS

Un método abstracto:
- Se declara con abstract.
- No tiene cuerpo (termina en ;).
- Debe implementarse en la primera subclase concreta.
- No puede ser private, static ni final.

```java
abstract class Animal {
    public abstract void hacerSonido();
    public abstract void moverse();

    public void dormir() {
        System.out.println("El animal está durmiendo.");
    }
}
```

4. SUBCLASES CONCRETAS

Deben implementar TODOS los métodos abstractos. Si no, también deben declararse
abstractas.

```java
class Perro extends Animal {
    private String nombre;

    public Perro(String nombre) {
        this.nombre = nombre;
    }

    @Override
    public void hacerSonido() {
        System.out.println(nombre + " dice: ¡Guau guau!");
    }

    @Override
    public void moverse() {
        System.out.println(nombre + " corre en cuatro patas.");
    }
}

class Pajaro extends Animal {
    private String nombre;

    public Pajaro(String nombre) {
        this.nombre = nombre;
    }

    @Override
    public void hacerSonido() {
        System.out.println(nombre + " dice: ¡Pío pío!");
    }

    @Override
    public void moverse() {
        System.out.println(nombre + " vuela por el cielo.");
    }
}
```

5. EJEMPLO COMPLETO: FIGURAS GEOMÉTRICAS

```java
abstract class FormaGeometrica {
    protected String nombre;
    protected String color;

    public FormaGeometrica(String nombre, String color) {
        this.nombre = nombre;
        this.color = color;
    }

    public abstract double calcularArea();
    public abstract double calcularPerimetro();
    public abstract int contarLados();

    public void mostrarInfo() {
        System.out.println("=== " + nombre.toUpperCase() + " ===");
        System.out.println("Color: " + color);
        System.out.println("Lados: " + contarLados());
        System.out.println("Área: " + String.format("%.2f", calcularArea()));
        System.out.println("Perímetro: " + String.format("%.2f", calcularPerimetro()));
    }
}

class Circulo extends FormaGeometrica {
    private double radio;

    public Circulo(double radio, String color) {
        super("Círculo", color);
        this.radio = radio;
    }

    @Override
    public double calcularArea() {
        return Math.PI * radio * radio;
    }

    @Override
    public double calcularPerimetro() {
        return 2 * Math.PI * radio;
    }

    @Override
    public int contarLados() {
        return 0;
    }
}

class Rectangulo extends FormaGeometrica {
    private double base, altura;

    public Rectangulo(double base, double altura, String color) {
        super("Rectángulo", color);
        this.base = base;
        this.altura = altura;
    }

    @Override
    public double calcularArea() { return base * altura; }

    @Override
    public double calcularPerimetro() { return 2 * (base + altura); }

    @Override
    public int contarLados() { return 4; }
}

class Triangulo extends FormaGeometrica {
    private double l1, l2, l3;

    public Triangulo(double l1, double l2, double l3, String color) {
        super("Triángulo", color);
        if (l1 + l2 <= l3 || l1 + l3 <= l2 || l2 + l3 <= l1)
            throw new IllegalArgumentException("Triángulo inválido");
        this.l1 = l1; this.l2 = l2; this.l3 = l3;
    }

    @Override
    public double calcularArea() {
        double s = (l1 + l2 + l3) / 2;
        return Math.sqrt(s * (s - l1) * (s - l2) * (s - l3));
    }

    @Override
    public double calcularPerimetro() { return l1 + l2 + l3; }

    @Override
    public int contarLados() { return 3; }
}
```

```java
public class TestAbstraccion {
    public static void main(String[] args) {
        FormaGeometrica[] formas = {
            new Circulo(5.0, "Rojo"),
            new Rectangulo(4.0, 6.0, "Azul"),
            new Triangulo(3.0, 4.0, 5.0, "Verde")
        };

        for (FormaGeometrica f : formas) {
            f.mostrarInfo();
            System.out.println();
        }
    }
}
```

6. CONSTRUCTORES EN CLASES ABSTRACTAS

Aunque no se puedan instanciar, tienen constructores llamados via super().

```java
abstract class Empleado {
    private String nombre;
    private String id;
    private double salarioBase;

    public Empleado(String nombre, String id, double salarioBase) {
        this.nombre = nombre;
        this.id = id;
        this.salarioBase = salarioBase;
    }

    public abstract double calcularSalario();

    public void mostrarInfo() {
        System.out.println(nombre + " | $" +
                         String.format("%.2f", calcularSalario()));
    }

    public String getNombre() { return nombre; }
    public String getId() { return id; }
    public double getSalarioBase() { return salarioBase; }
}
```

7. PATRÓN TEMPLATE METHOD

Las clases abstractas son ideales para el patrón Template Method: defines el
algoritmo y dejas pasos específicos a las subclases.

```java
abstract class BebidaCaliente {
    public final void prepararReceta() {
        hervirAgua();
        prepararIngrediente();
        servirEnTaza();
        agregarComplementos();
    }

    private void hervirAgua() {
        System.out.println("Hirviendo agua...");
    }

    protected abstract void prepararIngrediente();

    private void servirEnTaza() {
        System.out.println("Sirviendo en la taza...");
    }

    protected abstract void agregarComplementos();
}

class Cafe extends BebidaCaliente {
    @Override
    protected void prepararIngrediente() {
        System.out.println("Agregando café molido...");
    }

    @Override
    protected void agregarComplementos() {
        System.out.println("Agregando azúcar y leche...");
    }
}

class Te extends BebidaCaliente {
    @Override
    protected void prepararIngrediente() {
        System.out.println("Colocando bolsa de té...");
    }

    @Override
    protected void agregarComplementos() {
        System.out.println("Agregando limón...");
    }
}
```

8. CUÁNDO USAR CLASES ABSTRACTAS

Úsalas cuando:
- Las subclases comparten estado y comportamiento común.
- Necesitas proporcionar implementación parcial.
- La relación es claramente "es-un".
- Necesitas constructores con parámetros.

NO uses cuando:
- Solo necesitas un contrato (usa interface).
- No hay estado compartido.
- Necesitas herencia múltiple (usa interfaces).

```java
// BUEN USO: estado y comportamiento compartidos
abstract class Vehicle {
    protected String marca;
    protected int velocidad;
    public abstract void acelerar();
    public void mostrarVelocidad() {
        System.out.println(velocidad + " km/h");
    }
}

// MAL USO: mejor como interface
abstract class Volador { // → interface Volador
    public abstract void volar();
}
```

9. EJEMPLO: SISTEMA DE PAGOS

```java
abstract class Pago {
    protected String id;
    protected double monto;
    protected String estado;

    public Pago(String id, double monto) {
        this.id = id;
        this.monto = monto;
        this.estado = "PENDIENTE";
    }

    public abstract boolean autorizar();
    public abstract void capturar();
    public abstract void reembolsar();

    public void mostrarResumen() {
        System.out.println("ID: " + id + " | $" + monto +
                         " | Estado: " + estado);
    }
}

class PagoTarjeta extends Pago {
    private String numeroTarjeta;

    public PagoTarjeta(String id, double monto, String numeroTarjeta) {
        super(id, monto);
        this.numeroTarjeta = numeroTarjeta;
    }

    @Override
    public boolean autorizar() {
        if (monto > 0 && numeroTarjeta.length() == 16) {
            estado = "AUTORIZADO";
            System.out.println("Tarjeta autorizada: $" + monto);
            return true;
        }
        estado = "RECHAZADO";
        return false;
    }

    @Override
    public void capturar() {
        if (estado.equals("AUTORIZADO")) {
            estado = "CAPTURADO";
            System.out.println("Pago capturado.");
        }
    }

    @Override
    public void reembolsar() {
        if (estado.equals("CAPTURADO")) {
            estado = "REEMBOLSADO";
            System.out.println("Reembolso procesado.");
        }
    }
}
```

10. DIFERENCIAS: ABSTRACTA vs CONCRETA

| Característica       | Abstracta          | Concreta            |
|----------------------|--------------------|---------------------|
| Instanciable         | No                 | Sí                  |
| Métodos abstractos   | Sí (opcional)      | No                  |
| Métodos concretos    | Sí                 | Sí                  |
| Constructores        | Sí                 | Sí                  |
| Uso típico           | Clase base        | Clase completa      |

═══════════════════════════════════════════════════════════════════════════════════

11. BUENAS PRÁCTICAS

1. Minimizar métodos abstractos: solo lo esencial.
2. Usar Template Method para algoritmos reutilizables.
3. Proteger invariantes con métodos final.
4. Preferir interfaces para contratos puros.
5. Documentar el contrato de cada método abstracto.
6. Nombrar con prefijo Abstract cuando ayude.

14. PREGUNTAS FRECUENTES (FAQ)

P: ¿Puedo instanciar una clase abstracta?
R: No. new ClaseAbstracta() produce error de compilación.

P: ¿Una clase abstracta puede tener solo métodos concretos?
R: Sí. Puede declararse abstract incluso sin métodos abstractos, para evitar
instanciación.

P: ¿Puedo tener métodos abstractos private?
R: No. Los métodos abstractos deben ser accesibles desde subclases (public o
protected).

P: ¿Cuándo usar clase abstracta vs interface?
R: Abstracta cuando hay estado/comportamiento compartido. Interface para contrato
puro.

P: ¿Puede una clase abstracta tener final?
R: Sí, puede tener métodos final (que no se sobrescriben) y atributos final.

P: ¿Qué pasa si una subclase no implementa todos los métodos abstractos?
R: Debe declararse también como abstracta. No se puede instanciar.

P: ¿Las clases abstractas soportan polimorfismo?
R: Sí, totalmente. Una referencia de tipo abstracto puede apuntar a cualquier
subclase concreta.

15. DESAFÍO DEL ARQUITECTO

Construye un sistema de procesamiento de datos abstracto.

Clase abstracta ProcesadorDatos:
- Atributos: nombreProceso, datos (String[]), resultados (String[]).
- Constructor con nombre y datos.
- Métodos abstractos: void validarDatos(), String procesarElemento(String).
- Método concreto (Template): ejecutarProceso() que:
  1. Llama a validarDatos()
  2. Para cada elemento, llama a procesarElemento()
  3. Almacena resultados
  4. Muestra resumen

Subclases:
- ProcesadorTexto: validar que no esté vacío, procesar = convertir a mayúsculas.
- ProcesadorNumerico: validar que sean números, procesar = calcular cuadrado.
- ProcesadorFechas: validar formato dd/mm/aaaa, procesar = mostrar día de semana.

Clase de prueba que procese arrays de diferentes tipos.

Este desafío integra: abstracción, template method, polimorfismo, arrays.

16. CONCLUSIÓN

La abstracción nos permite diseñar jerarquías flexibles donde:
- Las clases abstractas definen el contrato y comportamiento parcial.
- Las subclases concretas completan la implementación.
- El patrón Template Method estructura algoritmos reutilizables.
- El código se vuelve extensible y mantenible.

siguiente nivel con contratos completamente abstractos.
