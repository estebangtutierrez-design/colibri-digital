1. ¿QUÉ ES UN PAQUETE?

Un paquete organiza tipos en grupos lógicos. Proporciona:
- Organización: agrupa clases relacionadas.
- Espacio de nombres: evita conflictos de nombres.
- Control de acceso: define niveles de visibilidad.
- Mantenibilidad: facilita navegación y mantenimiento.

La estructura de paquetes refleja la estructura de directorios:
com.empresa.proyecto.modelo → com/empresa/proyecto/modelo/

═══════════════════════════════════════════════════════════════════════════════════

2. DECLARACIÓN DE PAQUETES

package debe ser la primera línea del archivo (antes de import):

```java
package com.empresa.miaplicacion.modelo;

public class Usuario {
    private String nombre;
    private String email;
}
```

Convenciones de nombres:
- Se usan minúsculas.
- Se usa el dominio inverso de la empresa: com.google, org.apache.
- Se siguen con el nombre del proyecto y el módulo.
- Los niveles se separan con puntos.

Ejemplos:
- com.google.android.gms
- org.apache.commons.lang3
- com.empresa.sistema.modelo
- com.empresa.sistema.controlador
- com.empresa.sistema.vista

3. LA SENTENCIA import

Para usar clases de otro paquete se usa import:

```java
import java.util.Scanner;
import java.util.ArrayList;
import java.util.List;

public class MiPrograma {
    public static void main(String[] args) {
        Scanner sc = new Scanner(System.in);
        List<String> lista = new ArrayList<>();
    }
}
```

Import con wildcard (*):
```java
import java.util.*;  // Importa todas las clases de java.util (no subpaquetes)
import java.io.*;    // Importa todas las clases de java.io
```

Import estático (Java 5+):
```java
import static java.lang.Math.PI;
import static java.lang.Math.pow;
import static java.lang.System.out;

public class TestStaticImport {
    public static void main(String[] args) {
        out.println("Área: " + (PI * pow(5, 2)));
    }
}
```

Fully Qualified Name (sin import):
```java
public class SinImport {
    public static void main(String[] args) {
        java.util.Scanner sc = new java.util.Scanner(System.in);
        java.util.ArrayList<String> lista = new java.util.ArrayList<>();
    }
}
```

4. ESTRUCTURA DE DIRECTORIOS

La estructura de directorios debe coincidir con los paquetes:

```
Proyecto/
  src/
    com/
      empresa/
        sistema/
          Main.java                // package com.empresa.sistema;
          modelo/
            Usuario.java           // package com.empresa.sistema.modelo;
            Producto.java
          controlador/
            UsuarioCtrl.java
          vista/
            MenuPrincipal.java
  bin/     (clases compiladas)
  lib/     (dependencias)
```

Compilación y ejecución:
```bash
# Compilar desde src/
javac -d ../bin com/empresa/sistema/Main.java

# Ejecutar
java -cp bin com.empresa.sistema.Main
```

6. EJEMPLO COMPLETO: PROYECTO BANCARIO EN PAQUETES

Estructura del proyecto:
```
com/masterbook/banco/
  Main.java
  modelo/Cliente.java
  modelo/Cuenta.java
  servicio/CuentaServicio.java
  util/Validador.java
  util/GeneradorId.java
```

```java
// com/masterbook/banco/modelo/Cliente.java
package com.masterbook.banco.modelo;

public class Cliente {
    private String id;
    private String nombre;
    private String email;

    public Cliente(String id, String nombre, String email) {
        this.id = id;
        this.nombre = nombre;
        this.email = email;
    }

    public String getId() { return id; }
    public String getNombre() { return nombre; }
    public String getEmail() { return email; }

    @Override
    public String toString() {
        return nombre + " (" + email + ")";
    }
}
```

```java
// com/masterbook/banco/modelo/Cuenta.java
package com.masterbook.banco.modelo;

public class Cuenta {
    private String numero;
    private Cliente titular;
    private double saldo;

    public Cuenta(String numero, Cliente titular, double saldoInicial) {
        this.numero = numero;
        this.titular = titular;
        this.saldo = saldoInicial;
    }

    public String getNumero() { return numero; }
    public Cliente getTitular() { return titular; }
    public double getSaldo() { return saldo; }

    public void depositar(double monto) {
        if (monto > 0) this.saldo += monto;
    }

    public boolean retirar(double monto) {
        if (monto > 0 && saldo >= monto) {
            saldo -= monto;
            return true;
        }
        return false;
    }

    @Override
    public String toString() {
        return "Cuenta " + numero + " - " + titular.getNombre() +
               " | $" + String.format("%.2f", saldo);
    }
}
```

```java
// com/masterbook/banco/util/Validador.java
package com.masterbook.banco.util;

public class Validador {
    public static boolean validarMonto(double monto) {
        return monto > 0 && monto < 1000000;
    }

    public static boolean validarNumeroCuenta(String numero) {
        return numero != null && numero.matches("\\d{10,20}");
    }
}
```

```java
// com/masterbook/banco/util/GeneradorId.java
package com.masterbook.banco.util;

public class GeneradorId {
    public static String generarNumeroCuenta() {
        StringBuilder sb = new StringBuilder();
        for (int i = 0; i < 10; i++) {
            sb.append((int) (Math.random() * 10));
        }
        return sb.toString();
    }
}
```

```java
// com/masterbook/banco/servicio/CuentaServicio.java
package com.masterbook.banco.servicio;

import com.masterbook.banco.modelo.Cuenta;
import com.masterbook.banco.modelo.Cliente;
import com.masterbook.banco.util.Validador;
import com.masterbook.banco.util.GeneradorId;

public class CuentaServicio {
    private Cuenta[] cuentas;
    private int contador;

    public CuentaServicio(int capacidad) {
        this.cuentas = new Cuenta[capacidad];
        this.contador = 0;
    }

    public Cuenta crearCuenta(Cliente cliente, double depositoInicial) {
        if (!Validador.validarMonto(depositoInicial)) return null;
        String num = GeneradorId.generarNumeroCuenta();
        Cuenta c = new Cuenta(num, cliente, depositoInicial);
        if (contador < cuentas.length) cuentas[contador++] = c;
        return c;
    }

    public void listarTodas() {
        for (int i = 0; i < contador; i++) {
            System.out.println(cuentas[i]);
        }
    }
}
```

```java
// com/masterbook/banco/Main.java
package com.masterbook.banco;

import com.masterbook.banco.modelo.Cliente;
import com.masterbook.banco.servicio.CuentaServicio;

public class Main {
    public static void main(String[] args) {
        CuentaServicio servicio = new CuentaServicio(10);
        Cliente ana = new Cliente("C-001", "Ana López", "ana@email.com");
        Cliente carlos = new Cliente("C-002", "Carlos Ruiz", "carlos@email.com");
        servicio.crearCuenta(ana, 1000);
        servicio.crearCuenta(carlos, 2500);
        System.out.println("=== CUENTAS ===");
        servicio.listarTodas();
    }
}
```

7. IMPORTANCIA DEL PAQUETE POR DEFECTO

Si no se declara package, la clase pertenece al paquete por defecto (default
package). Esto solo es aceptable para ejemplos pequeños y prototipos rápidos.

En proyectos reales:
- SIEMPRE usa paquetes explícitos.
- NO uses el paquete por defecto en código profesional.
- Las clases sin paquete no pueden ser importadas por clases con paquete.

```java
// MAL: Sin paquete (default package)
public class HolaMundo {
    public static void main(String[] args) {
        System.out.println("Hola");
    }
}

// BIEN: Con paquete
package com.miempresa.app;
public class HolaMundo {
    public static void main(String[] args) {
        System.out.println("Hola");
    }
}
```

8. VISIBILIDAD Y PAQUETES

El modificador default (package-private) permite acceso solo desde clases del
mismo paquete:

```java
package com.empresa.internal;

class ClaseInterna {
    // Solo visible en com.empresa.internal
    void metodoAyuda() { }
}
```

Esto es útil para:
- Clases de utilidad internas.
- Implementaciones que no deben exponerse.
- Testing (clases de test en el mismo paquete).

9. CONFLICTOS DE NOMBRES

Si dos paquetes tienen clases con el mismo nombre, usa FQN para resolver:

```java
import java.util.Date;
// import java.sql.Date; // Error de compilación

public class Test {
    public static void main(String[] args) {
        Date utilDate = new Date();              // java.util.Date
        java.sql.Date sqlDate = new java.sql.Date(0); // Fully Qualified
    }
}
```

También puedes importar una y usar FQN para la otra:
```java
import java.util.Date;

// No importar java.sql.Date, usar FQN:
java.sql.Date sqlDate = new java.sql.Date(System.currentTimeMillis());
```

10. EL CLASSPATH Y LOS ARCHIVOS JAR

El classpath es la ruta donde Java busca las clases compiladas y los .jar:

```bash
# Compilar con classpath externo
javac -cp lib/dependencia.jar -d bin src/com/empresa/Main.java

# Ejecutar con classpath
java -cp bin:lib/dependencia.jar com.empresa.Main

# En Windows usar ; como separador
java -cp bin;lib\dependencia.jar com.empresa.Main

# Usando el flag -classpath (equivalente a -cp)
java -classpath bin:lib/* com.empresa.Main
```

Archivos JAR (Java ARchive): empaquetan clases compiladas:

```bash
# Crear un JAR
jar cf miApp.jar -C bin .

# Ver contenido de un JAR
jar tf miApp.jar

# Ejecutar un JAR
java -jar miApp.jar

# Crear JAR ejecutable (con MANIFEST.MF)
jar cfm miApp.jar MANIFEST.MF -C bin .
```

Contenido típico de MANIFEST.MF:
```
Manifest-Version: 1.0
Main-Class: com.empresa.Main
Class-Path: lib/otro.jar lib/util.jar
```

11. EL SISTEMA DE MÓDULOS (JPMS) EN JAVA 9+

Java 9 introdujo el sistema de módulos (Java Platform Module System). Un módulo
agrupa uno o más paquetes y declara dependencias y exportaciones explícitas.

```java
// module-info.java (archivo especial en la raíz de src/)
module com.empresa.miaplicacion {
    requires java.sql;
    requires java.logging;
    exports com.empresa.miaplicacion.api;
    exports com.empresa.miaplicacion.modelo;
    opens com.empresa.miaplicacion.internal;
}
```

Directivas principales:
- requires: declara dependencia de otro módulo.
- exports: hace visible un paquete a otros módulos.
- opens: permite acceso por reflexión.
- provides/uses: para servicios (ServiceLoader).
- transitive: dependencia transitiva.

Los paquetes NO desaparecen con los módulos. Siguen existiendo dentro de los
módulos. Los módulos son una capa adicional de organización y encapsulación.

package com.miempresa.app.servicio;

import com.miempresa.app.modelo.Usuario;

public class UsuarioServicio {

}

14. BUENAS PRÁCTICAS

1. Usar dominio inverso: com.miempresa.miproyecto para evitar conflictos.
2. Un paquete por funcionalidad: modelo, servicio, controlador, vista, util.
3. Nombres en minúscula, sin guiones, espacios ni caracteres especiales.
4. No usar default package en proyectos reales (solo ejemplos pequeños).
5. Agrupar interfaces, clases abstractas y concretas relacionadas en un mismo
   paquete si están estrechamente acopladas.
6. Un archivo .java por clase pública, con nombre idéntico a la clase.
7. Usar import explícito en lugar de comodín (*) en código profesional.
8. Mantener las importaciones ordenadas (IDE puede ayudar con esto).
9. No crear dependencias circulares entre paquetes.
10. Documentar el propósito de cada paquete con package-info.java:

```java
// Archivo: package-info.java en com/empresa/servicio/
/**
 * Contiene las clases de servicio del sistema.
 * Implementa la lógica de negocio de la aplicación.
 */
package com.empresa.servicio;
```

15. PREGUNTAS FRECUENTES (FAQ)

P: ¿Cuál es la diferencia entre package y import?
R: package declara a qué paquete pertenece la clase. import trae clases de otros
paquetes al ámbito actual.

P: ¿Puedo tener múltiples clases públicas en un archivo?
R: No. Solo una clase pública por archivo, y debe llamarse igual que el archivo.

P: ¿Qué pasa si no uso package?
R: La clase va al paquete por defecto (default package). No recomendado para
proyectos reales.

P: ¿El import con * afecta el rendimiento?
R: No en tiempo de ejecución. En compilación puede ser ligeramente más lento.

P: ¿Cómo se compila un proyecto con paquetes?
R: javac -d directorio_salida archivo.java. Luego java paquete.Clase.

P: ¿Qué es el classpath?
R: Es la ruta donde Java busca las clases compiladas y JARs. Se especifica con
-cp o -classpath.

P: ¿Java 9+ módulos reemplazan paquetes?
R: No. Los módulos (module-info.java) agrupan paquetes. Los paquetes siguen
existiendo dentro de los módulos.

P: ¿Puedo tener una clase sin paquete que importe otra con paquete?
R: No. Si una clase está en un paquete, puede importar de otros paquetes. Pero
clases en el paquete por defecto no pueden importar clases con paquete.

P: ¿Qué es el classpath por defecto?
R: El directorio actual (.) si no se especifica -cp.

P: ¿Cómo se organizan los tests en paquetes?
R: Convencionalmente en src/test/java con la misma estructura de paquetes que
src/main/java.

  nomina, produccion, calidad, logistica.

Cada módulo debe tener subpaquetes:
- modelo: entidades del dominio.
- servicio: lógica de negocio.
- controlador: manejo de solicitudes.
- repositorio: acceso a datos.
- util: utilidades específicas del módulo.
- excepcion: excepciones del módulo.
- dto: objetos de transferencia de datos.

Además:
- Un paquete compartido: com.corporacion.erp.comun con clases como Direccion,
  Contacto, DocumentoIdentidad, Moneda, FechaUtil.
- Un paquete de configuración: com.corporacion.erp.config con ConfiguracionSistema,
  BaseDatosConfig, SeguridadConfig.
- Un paquete de seguridad: com.corporacion.erp.seguridad con Usuario, Rol,
  Permiso, AutenticacionServicio, AutorizacionServicio.
- Un paquete de reportes: com.corporacion.erp.reporte con GeneradorReportes,
  FormateadorPDF, FormateadorExcel, Graficador.

Escribe la estructura completa de directorios y al menos una clase de ejemplo
para cada módulo usando package e import correctamente.

Incluye una clase Main en com.corporacion.erp que demuestre interacción entre
módulos usando imports.

Este desafío integra: organización de paquetes, jerarquía de directorios,
importación entre paquetes, diseño de arquitectura de software, y buenas
prácticas de estructuración de proyectos empresariales.

═══════════════════════════════════════════════════════════════════════════════════

17. CONCLUSIÓN

Los paquetes son una herramienta fundamental para:
- Organizar proyectos de cualquier tamaño.
- Evitar conflictos de nombres entre clases.
- Controlar la visibilidad a nivel de paquete.
- Facilitar el mantenimiento y la evolución.
- Documentar la estructura del proyecto.
- Permitir la distribución modular (JARs, módulos).

Una buena estructura de paquetes es la base de una arquitectura de software
sólida y mantenible. Invertir tiempo en diseñarla correctamente al inicio del
proyecto ahorra incontables horas de refactorización.

En el próximo capítulo veremos los modificadores de acceso, que trabajan junto
con los paquetes para controlar la visibilidad de clases, métodos y atributos.
