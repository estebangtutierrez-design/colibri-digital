-------------------------------------------------------------------------------
  71.1. ¿QUÉ ES MVC?
-------------------------------------------------------------------------------

MVC (Model-View-Controller) es un patrón arquitectónico que separa una
aplicación en tres componentes principales:

  ┌─────────────────────────────────────────────────────────────────┐
  │                        MVC ARCHITECTURE                         │
  │                                                                 │
  │   ┌──────────┐     ┌──────────────┐     ┌──────────┐           │
  │   │  VIEW    │◄────►│ CONTROLLER  │◄────►│  MODEL   │           │
  │   │ (Vista)  │      │(Controlador)│      │ (Modelo) │           │
  │   │          │      │             │      │          │           │
  │   │Interfaz  │      │Lógica de    │      │Datos y   │           │
  │   │de usuario│      │aplicación   │      │negocio   │           │
  │   └──────────┘      └──────────────┘      └──────────┘           │
  │                                                                 │
  └─────────────────────────────────────────────────────────────────┘

  • Modelo (Model): Gestiona los datos, la lógica de negocio y las reglas
    de la aplicación. Es independiente de la interfaz de usuario.

  • Vista (View): Presenta los datos al usuario. Se actualiza cuando el
    modelo cambia. No contiene lógica de negocio.

  • Controlador (Controller): Maneja las entradas del usuario, interpreta
    las acciones y actualiza el modelo o la vista según corresponda.

-------------------------------------------------------------------------------
  71.2. HISTORIA Y ORIGEN DEL MVC
-------------------------------------------------------------------------------

El patrón MVC fue descrito por primera vez en 1979 por Trygve Reenskaug
mientras trabajaba en el Smalltalk-80 en el Xerox PARC. La idea original era
resolver el problema de las interfaces de usuario complejas.

  • 1979: Trygve Reenskaug propone MVC para Smalltalk-80
  • 1980: Se implementa en el entorno Smalltalk-80
  • 1990: Se populariza en aplicaciones web con frameworks como Struts
  • 2000: Spring MVC lo adopta como núcleo de su framework web
  • 2010: JavaScript frameworks (Backbone, Angular, React) reinterpretan MVC
  • 2020: MVC sigue siendo el patrón dominante en aplicaciones empresariales

Java ha incorporado MVC de diversas formas:
  - AWT/Swing: Modelo delegado (variante MVC)
  - JavaServer Faces (JSF): Componente-based MVC
  - Spring MVC: El framework MVC más usado en el mundo empresarial
  - Jakarta MVC: Especificación oficial de MVC para Java EE

-------------------------------------------------------------------------------
  71.3. SEPARACIÓN DE CONCERNAS (Separation of Concerns)
-------------------------------------------------------------------------------

La separación de concernas es el principio fundamental detrás de MVC.
Cada componente tiene una responsabilidad única y bien definida:

  RESPONSABILIDAD DEL MODELO:
  - Almacenar el estado de la aplicación
  - Implementar reglas de negocio
  - Realizar validaciones de datos
  - Gestionar la persistencia (base de datos, archivos, API)
  - Notificar cambios a la vista (mediante Observer)

  RESPONSABILIDAD DE LA VISTA:
  - Renderizar la interfaz de usuario
  - Capturar eventos del usuario (clics, teclas, táctil)
  - Delegar acciones al controlador
  - Actualizar la presentación cuando el modelo cambia
  - Mantenerse libre de lógica de negocio

  RESPONSABILIDAD DEL CONTROLADOR:
  - Recibir peticiones del usuario
  - Interpretar la acción solicitada
  - Invocar métodos del modelo
  - Seleccionar la vista adecuada para la respuesta
  - Actuar como puente entre vista y modelo

-------------------------------------------------------------------------------
  71.4. ACOPLAMIENTO Y COHESIÓN
-------------------------------------------------------------------------------

Dos conceptos críticos en el diseño arquitectónico:

  ACOPLAMIENTO (Coupling):
  Mide el grado de dependencia entre módulos. Buscamos un BAJO acoplamiento.

  • Alto acoplamiento: Cambiar un módulo fuerza cambios en muchos otros
  • Bajo acoplamiento: Los módulos son independientes, fáciles de mantener

  COHESIÓN (Cohesion):
  Mide cuán relacionadas están las responsabilidades dentro de un módulo.
  Buscamos una ALTA cohesión.

  • Baja cohesión: Un módulo hace muchas cosas no relacionadas
  • Alta cohesión: Cada módulo tiene un propósito claro y único

  MVC promueve:
  - Bajo acoplamiento entre capas (se comunican mediante interfaces)
  - Alta cohesión dentro de cada capa (cada una tiene su responsabilidad)

```java
// Entidad del dominio: representa una tarea en el sistema

package com.lucasega.mvc.model;

import java.time.LocalDateTime;
import java.util.Objects;
import java.util.UUID;

public class Tarea {

    private String id;
    private String titulo;
    private String descripcion;
    private boolean completada;
    private LocalDateTime fechaCreacion;
    private LocalDateTime fechaVencimiento;

    // Constructor por defecto
    public Tarea() {
        this.id = UUID.randomUUID().toString();
        this.fechaCreacion = LocalDateTime.now();
        this.completada = false;
    }

    // Constructor parametrizado
    public Tarea(String titulo, String descripcion, LocalDateTime fechaVencimiento) {
        this();
        this.titulo = titulo;
        this.descripcion = descripcion;
        this.fechaVencimiento = fechaVencimiento;
    }

    // Getters y Setters
    public String getId() { return id; }
    public void setId(String id) { this.id = id; }

    public String getTitulo() { return titulo; }
    public void setTitulo(String titulo) {
        if (titulo == null || titulo.trim().isEmpty()) {
            throw new IllegalArgumentException("El título no puede estar vacío");
        }
        this.titulo = titulo;
    }

    public String getDescripcion() { return descripcion; }
    public void setDescripcion(String descripcion) { this.descripcion = descripcion; }

    public boolean isCompletada() { return completada; }
    public void setCompletada(boolean completada) { this.completada = completada; }

    public LocalDateTime getFechaCreacion() { return fechaCreacion; }
    public void setFechaCreacion(LocalDateTime fechaCreacion) { this.fechaCreacion = fechaCreacion; }

    public LocalDateTime getFechaVencimiento() { return fechaVencimiento; }
    public void setFechaVencimiento(LocalDateTime fechaVencimiento) { this.fechaVencimiento = fechaVencimiento; }

    // Métodos de negocio
    public void marcarCompletada() {
        this.completada = true;
    }

    public void marcarPendiente() {
        this.completada = false;
    }

    public boolean estaVencida() {
        return fechaVencimiento != null && 
               LocalDateTime.now().isAfter(fechaVencimiento) && 
               !completada;
    }

    @Override
    public String toString() {
        String estado = completada ? "[✓]" : "[ ]";
        String vencida = estaVencida() ? " (VENCIDA)" : "";
        return String.format("%s %s - %s%s", estado, titulo, descripcion, vencida);
    }

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof Tarea)) return false;
        Tarea tarea = (Tarea) o;
        return Objects.equals(id, tarea.id);
    }

    @Override
    public int hashCode() {
        return Objects.hash(id);
    }
}
```

```java
// com/lucasega/mvc/model/TareaRepository.java
// Repositorio: capa de acceso a datos (simulada en memoria)

package com.lucasega.mvc.model;

import java.util.ArrayList;
import java.util.Collections;
import java.util.List;
import java.util.Optional;
import java.util.stream.Collectors;

public class TareaRepository {

    private final List<Tarea> tareas;
    private static TareaRepository instancia;

    // Singleton: única instancia del repositorio
    private TareaRepository() {
        this.tareas = new ArrayList<>();
    }

    public static synchronized TareaRepository getInstancia() {
        if (instancia == null) {
            instancia = new TareaRepository();
        }
        return instancia;
    }

    // CRUD básico
    public Tarea guardar(Tarea tarea) {
        tareas.add(tarea);
        return tarea;
    }

    public Optional<Tarea> buscarPorId(String id) {
        return tareas.stream()
                .filter(t -> t.getId().equals(id))
                .findFirst();
    }

    public List<Tarea> buscarTodas() {
        return Collections.unmodifiableList(new ArrayList<>(tareas));
    }

    public List<Tarea> buscarPorEstado(boolean completada) {
        return tareas.stream()
                .filter(t -> t.isCompletada() == completada)
                .collect(Collectors.toList());
    }

    public List<Tarea> buscarVencidas() {
        return tareas.stream()
                .filter(Tarea::estaVencida)
                .collect(Collectors.toList());
    }

    public Tarea actualizar(Tarea tareaActualizada) {
        Optional<Tarea> existente = buscarPorId(tareaActualizada.getId());
        if (existente.isPresent()) {
            Tarea t = existente.get();
            t.setTitulo(tareaActualizada.getTitulo());
            t.setDescripcion(tareaActualizada.getDescripcion());
            t.setCompletada(tareaActualizada.isCompletada());
            t.setFechaVencimiento(tareaActualizada.getFechaVencimiento());
            return t;
        }
        throw new IllegalArgumentException("Tarea no encontrada: " + tareaActualizada.getId());
    }

    public boolean eliminar(String id) {
        return tareas.removeIf(t -> t.getId().equals(id));
    }

    public int contar() {
        return tareas.size();
    }

    public void limpiar() {
        tareas.clear();
    }
}
```

```java
// com/lucasega/mvc/model/TareaService.java
// Servicio: lógica de negocio de la aplicación

package com.lucasega.mvc.model;

import java.time.LocalDateTime;
import java.util.List;
import java.util.Optional;

public class TareaService {

    private final TareaRepository repositorio;

    public TareaService() {
        this.repositorio = TareaRepository.getInstancia();
    }

    // Constructor para inyección de dependencias (útil para testing)
    public TareaService(TareaRepository repositorio) {
        this.repositorio = repositorio;
    }

    // Crear una nueva tarea con validaciones
    public Tarea crearTarea(String titulo, String descripcion, LocalDateTime vencimiento) {
        if (titulo == null || titulo.trim().length() < 3) {
            throw new IllegalArgumentException(
                "El título debe tener al menos 3 caracteres");
        }
        if (vencimiento != null && vencimiento.isBefore(LocalDateTime.now())) {
            throw new IllegalArgumentException(
                "La fecha de vencimiento no puede estar en el pasado");
        }
        Tarea tarea = new Tarea(titulo, descripcion, vencimiento);
        return repositorio.guardar(tarea);
    }

    // Obtener todas las tareas
    public List<Tarea> obtenerTodasLasTareas() {
        return repositorio.buscarTodas();
    }

    // Obtener tareas pendientes
    public List<Tarea> obtenerTareasPendientes() {
        return repositorio.buscarPorEstado(false);
    }

    // Obtener tareas completadas
    public List<Tarea> obtenerTareasCompletadas() {
        return repositorio.buscarPorEstado(true);
    }

    // Marcar tarea como completada
    public Tarea completarTarea(String id) {
        Optional<Tarea> optional = repositorio.buscarPorId(id);
        if (optional.isPresent()) {
            Tarea tarea = optional.get();
            tarea.marcarCompletada();
            return repositorio.actualizar(tarea);
        }
        throw new IllegalArgumentException("Tarea no encontrada: " + id);
    }

    // Marcar tarea como pendiente
    public Tarea desmarcarTarea(String id) {
        Optional<Tarea> optional = repositorio.buscarPorId(id);
        if (optional.isPresent()) {
            Tarea tarea = optional.get();
            tarea.marcarPendiente();
            return repositorio.actualizar(tarea);
        }
        throw new IllegalArgumentException("Tarea no encontrada: " + id);
    }

    // Eliminar una tarea
    public boolean eliminarTarea(String id) {
        return repositorio.eliminar(id);
    }

    // Obtener estadísticas
    public int obtenerTotalTareas() {
        return repositorio.contar();
    }

    public int obtenerTareasCompletadasCount() {
        return repositorio.buscarPorEstado(true).size();
    }

    public int obtenerTareasPendientesCount() {
        return repositorio.buscarPorEstado(false).size();
    }

    // Buscar tarea por ID
    public Optional<Tarea> buscarTarea(String id) {
        return repositorio.buscarPorId(id);
    }
}
```

```java
// com/lucasega/mvc/controller/TareaController.java
// Controlador: maneja las solicitudes del usuario y coordina el flujo

package com.lucasega.mvc.controller;

import com.lucasega.mvc.model.Tarea;
import com.lucasega.mvc.model.TareaService;
import com.lucasega.mvc.view.TareaView;
import com.lucasega.mvc.view.TareaConsoleView;

import java.time.LocalDateTime;
import java.time.format.DateTimeFormatter;
import java.time.format.DateTimeParseException;
import java.util.List;
import java.util.Optional;

public class TareaController {

    private final TareaService servicio;
    private final TareaView vista;

    public TareaController() {
        this.servicio = new TareaService();
        this.vista = new TareaConsoleView();
    }

    // Constructor para inyección de dependencias
    public TareaController(TareaService servicio, TareaView vista) {
        this.servicio = servicio;
        this.vista = vista;
    }

    // Método principal que inicia la aplicación
    public void iniciar() {
        vista.limpiarPantalla();
        vista.mostrarBienvenida();
        boolean ejecutando = true;

        while (ejecutando) {
            try {
                vista.mostrarMenu();
                String opcion = vista.leerEntrada();

                switch (opcion) {
                    case "1" -> manejarCrearTarea();
                    case "2" -> manejarListarTodas();
                    case "3" -> manejarListarPendientes();
                    case "4" -> manejarListarCompletadas();
                    case "5" -> manejarCompletarTarea();
                    case "6" -> manejarEliminarTarea();
                    case "7" -> manejarEstadisticas();
                    case "8" -> manejarBuscarTarea();
                    case "9" -> manejarDesmarcarTarea();
                    case "0" -> ejecutando = false;
                    default -> vista.mostrarError("Opción no válida");
                }
            } catch (Exception e) {
                vista.mostrarError("Error inesperado: " + e.getMessage());
            }
        }
        vista.mostrarMensaje("¡Gracias por usar el sistema de tareas!");
    }

    private void manejarCrearTarea() {
        vista.mostrarMensaje("--- CREAR NUEVA TAREA ---");

        vista.mostrarMensaje("Ingrese el título:");
        String titulo = vista.leerEntrada();

        vista.mostrarMensaje("Ingrese la descripción:");
        String descripcion = vista.leerEntrada();

        vista.mostrarMensaje("Ingrese fecha de vencimiento (opcional, formato: dd/MM/yyyy HH:mm):");
        String fechaStr = vista.leerEntrada();

        LocalDateTime vencimiento = null;
        if (!fechaStr.isEmpty()) {
            try {
                DateTimeFormatter formatter = DateTimeFormatter.ofPattern("dd/MM/yyyy HH:mm");
                vencimiento = LocalDateTime.parse(fechaStr, formatter);
            } catch (DateTimeParseException e) {
                vista.mostrarError("Formato de fecha inválido. Se creará sin vencimiento.");
            }
        }

        try {
            Tarea tarea = servicio.crearTarea(titulo, descripcion, vencimiento);
            vista.mostrarMensaje("Tarea creada exitosamente:");
            vista.mostrarTarea(tarea);
        } catch (IllegalArgumentException e) {
            vista.mostrarError(e.getMessage());
        }
    }

    private void manejarListarTodas() {
        List<Tarea> tareas = servicio.obtenerTodasLasTareas();
        vista.mostrarTareas(tareas);
    }

    private void manejarListarPendientes() {
        List<Tarea> tareas = servicio.obtenerTareasPendientes();
        vista.mostrarMensaje("--- TAREAS PENDIENTES ---");
        vista.mostrarTareas(tareas);
    }

    private void manejarListarCompletadas() {
        List<Tarea> tareas = servicio.obtenerTareasCompletadas();
        vista.mostrarMensaje("--- TAREAS COMPLETADAS ---");
        vista.mostrarTareas(tareas);
    }

    private void manejarCompletarTarea() {
        vista.mostrarMensaje("--- COMPLETAR TAREA ---");
        vista.mostrarMensaje("Ingrese el ID de la tarea a completar:");
        String id = vista.leerEntrada();

        try {
            Tarea tarea = servicio.completarTarea(id);
            vista.mostrarMensaje("Tarea completada:");
            vista.mostrarTarea(tarea);
        } catch (IllegalArgumentException e) {
            vista.mostrarError(e.getMessage());
        }
    }

    private void manejarEliminarTarea() {
        vista.mostrarMensaje("--- ELIMINAR TAREA ---");
        vista.mostrarMensaje("Ingrese el ID de la tarea a eliminar:");
        String id = vista.leerEntrada();

        if (servicio.eliminarTarea(id)) {
            vista.mostrarMensaje("Tarea eliminada exitosamente.");
        } else {
            vista.mostrarError("No se pudo eliminar la tarea.");
        }
    }

    private void manejarEstadisticas() {
        int total = servicio.obtenerTotalTareas();
        int completadas = servicio.obtenerTareasCompletadasCount();
        int pendientes = servicio.obtenerTareasPendientesCount();
        vista.mostrarEstadisticas(total, completadas, pendientes);
    }

    private void manejarBuscarTarea() {
        vista.mostrarMensaje("--- BUSCAR TAREA ---");
        vista.mostrarMensaje("Ingrese el ID de la tarea:");
        String id = vista.leerEntrada();

        Optional<Tarea> tarea = servicio.buscarTarea(id);
        if (tarea.isPresent()) {
            vista.mostrarTarea(tarea.get());
        } else {
            vista.mostrarError("Tarea no encontrada.");
        }
    }

    private void manejarDesmarcarTarea() {
        vista.mostrarMensaje("--- DESMARCAR TAREA ---");
        vista.mostrarMensaje("Ingrese el ID de la tarea a desmarcar:");
        String id = vista.leerEntrada();

        try {
            Tarea tarea = servicio.desmarcarTarea(id);
            vista.mostrarMensaje("Tarea marcada como pendiente:");
            vista.mostrarTarea(tarea);
        } catch (IllegalArgumentException e) {
            vista.mostrarError(e.getMessage());
        }
    }
}
```

-------------------------------------------------------------------------------
  71.6. FLUJO DE INTERACCIÓN EN MVC
-------------------------------------------------------------------------------

El flujo típico de una petición en una aplicación MVC es el siguiente:

  1. El usuario interactúa con la Vista (hace clic, escribe, etc.)
  2. La Vista notifica al Controlador sobre la acción del usuario
  3. El Controlador interpreta la petición y llama al Modelo adecuado
  4. El Modelo ejecuta la lógica de negocio y accede a datos si es necesario
  5. El Modelo devuelve el resultado al Controlador
  6. El Controlador selecciona la Vista apropiada y le pasa los datos
  7. La Vista renderiza la respuesta y la presenta al usuario

  DIAGRAMA DE SECUENCIA:

  Usuario       Vista         Controlador       Modelo
    │             │               │               │
    │──Acción────►│               │               │
    │             │──Petición────►│               │
    │             │               │──Operación───►│
    │             │               │◄──Resultado───│
    │             │◄──Respuesta───│               │
    │◄──Render────│               │               │
    │             │               │               │

-------------------------------------------------------------------------------
  71.7. MVC Y EL PATRÓN OBSERVER
-------------------------------------------------------------------------------

El patrón MVC clásico utiliza el patrón Observer para mantener la vista
sincronizada con el modelo. El modelo es el "sujeto" y las vistas son los
"observadores".

```java
// com/lucasega/mvc/model/TareaModeloObservable.java
// Modelo que implementa el patrón Observer para notificar cambios

package com.lucasega.mvc.model;

import java.util.ArrayList;
import java.util.List;

// Interfaz Observer
interface TareaObserver {
    void onTareaCreada(Tarea tarea);
    void onTareaActualizada(Tarea tarea);
    void onTareaEliminada(String id);
}

// Sujeto observable
public class TareaModeloObservable {

    private final List<TareaObserver> observadores;
    private final TareaService servicio;

    public TareaModeloObservable(TareaService servicio) {
        this.observadores = new ArrayList<>();
        this.servicio = servicio;
    }

    public void agregarObservador(TareaObserver obs) {
        observadores.add(obs);
    }

    public void eliminarObservador(TareaObserver obs) {
        observadores.remove(obs);
    }

    private void notificarCreacion(Tarea tarea) {
        observadores.forEach(obs -> obs.onTareaCreada(tarea));
    }

    private void notificarActualizacion(Tarea tarea) {
        observadores.forEach(obs -> obs.onTareaActualizada(tarea));
    }

    private void notificarEliminacion(String id) {
        observadores.forEach(obs -> obs.onTareaEliminada(id));
    }

    // Métodos que notifican cambios
    public Tarea crearTarea(String titulo, String descripcion,
                           LocalDateTime vencimiento) {
        Tarea tarea = servicio.crearTarea(titulo, descripcion, vencimiento);
        notificarCreacion(tarea);
        return tarea;
    }

    public Tarea completarTarea(String id) {
        Tarea tarea = servicio.completarTarea(id);
        notificarActualizacion(tarea);
        return tarea;
    }

    public boolean eliminarTarea(String id) {
        boolean resultado = servicio.eliminarTarea(id);
        if (resultado) {
            notificarEliminacion(id);
        }
        return resultado;
    }

    // Métodos de consulta (no notifican cambios)
    public List<Tarea> obtenerTodas() {
        return servicio.obtenerTodasLasTareas();
    }
}
```

-------------------------------------------------------------------------------
  71.8. VARIANTES DE MVC
-------------------------------------------------------------------------------

Existen varias variantes del patrón MVC, cada una adaptada a diferentes
contextos:

  MVP (Model-View-Presenter):
  - El Presenter reemplaza al Controller
  - La Vista es pasiva: solo delega eventos al Presenter
  - El Presenter maneja toda la lógica de presentación
  - Usado en: GWT, Android (tradicional)

  MVVM (Model-View-ViewModel):
  - El ViewModel expone datos y comandos para la Vista
  - La Vista se suscribe a cambios del ViewModel (data binding)
  - Usado en: JavaFX, WPF, Angular

  HMVC (Hierarchical MVC):
  - Cada componente es un MVC completo (sub-aplicaciones)
  - Los controladores se organizan jerárquicamente
  - Usado en: aplicaciones modulares complejas

  PAC (Presentation-Abstraction-Control):
  - Similar a HMVC pero con agentes independientes
  - Usado en: sistemas multiagente, interfaces multimodales

-------------------------------------------------------------------------------
  71.9. BUENAS PRÁCTICAS EN MVC
-------------------------------------------------------------------------------

  1. Mantén las capas separadas:
     - No mezcles código de presentación con lógica de negocio
     - No pongas SQL o llamadas a API en las vistas

  2. Usa interfaces para las dependencias:
     - Las capas deben depender de abstracciones, no de implementaciones
     - Facilita el testing y el cambio de implementaciones

  3. Controladores delgados (Thin Controllers):
     - Los controladores solo deben orquestar, no contener lógica pesada
     - La lógica de negocio pertenece al modelo

  4. Vistas tontas (Dumb Views):
     - Las vistas solo deben mostrar datos, no procesarlos
     - Sin condicionales complejos ni transformaciones de datos

  5. Modelos ricos (Rich Models):
     - El modelo debe contener toda la lógica de negocio
     - Incluye validaciones, cálculos, reglas de negocio

  6. Inyección de dependencias:
     - Pasa las dependencias por constructor, no las crees internamente
     - Facilita las pruebas unitarias con mocks

  7. Pruebas unitarias:
     - Testea el modelo sin la vista
     - Testea el controlador con mocks de vista y modelo
     - Usa la vista real solo en tests de integración

  RESUMEN DEL CAPÍTULO
-------------------------------------------------------------------------------

  • MVC es un patrón arquitectónico que separa la aplicación en tres capas:
    Modelo (datos y negocio), Vista (presentación) y Controlador
    (orquestación).

  • La separación de concernas (SoC) es el principio fundamental:
    cada capa tiene una responsabilidad única y bien definida.

  • Buscamos bajo acoplamiento entre capas y alta cohesión dentro de
    cada capa.

  • El modelo contiene la lógica de negocio y los datos. No debe tener
    conocimiento de la vista ni del controlador.

  • La vista solo se encarga de la presentación. No debe contener
    lógica de negocio.

  • El controlador actúa como intermediario: recibe peticiones del
    usuario, invoca al modelo y selecciona la vista.

  • El modelo clásico usa el patrón Observer para notificar cambios
    a las vistas.

  • Existen variantes: MVP, MVVM, HMVC, cada una adaptada a
    contextos específicos.

  • Las buenas prácticas incluyen: controladores delgados, vistas
    tontas, modelos ricos, e inyección de dependencias.

  • Spring MVC es la implementación más usada en el mundo empresarial
    Java.

  • MVC no es solo un patrón: es una forma de pensar el software.
    Domínalo y podrás construir sistemas de cualquier escala.

================================================================================
  PREGUNTAS FRECUENTES (FAQ)
================================================================================

  ❓ ¿MVC es un patrón de diseño o una arquitectura?
  R: Es un patrón arquitectónico. Opera a un nivel más alto que los
     patrones de diseño (Singleton, Factory, etc.). Define la estructura
     general de la aplicación, no cómo resolver un problema específico.

  ❓ ¿Puedo tener múltiples vistas para un mismo modelo?
  R: Sí, y es una de las grandes ventajas de MVC. Puedes tener una vista
     de consola, una vista web y una vista móvil compartiendo el mismo
     modelo y controlador (o controladores específicos).

  ❓ ¿Los controladores deben ser clases separadas o pueden ser métodos?
  R: En aplicaciones pequeñas, un solo controlador con métodos puede ser
     suficiente. En aplicaciones grandes, se organizan por recurso o
     dominio (UsuarioController, PedidoController, etc.).

  ❓ ¿Qué tan estricto debo ser con la separación de capas?
  R: En proyectos profesionales, muy estricto. En prototipos o pruebas
     rápidas, puedes relajarlo. Pero recuerda: la deuda técnica se paga
     con intereses.

  ❓ ¿MVC es adecuado para aplicaciones móviles?
  R: Sí, pero en Android se usa más MVP o MVVM. iOS usa MVC de forma
     nativa (UIKit). El concepto es el mismo, los nombres cambian.

  ❓ ¿Cómo manejo la navegación entre pantallas en MVC?
  R: El controlador es responsable de la navegación. Decide qué vista
     mostrar después de cada acción del usuario.

  ❓ ¿Puedo usar MVC sin un framework?
  R: Sí, como hemos demostrado en este capítulo. Los frameworks facilitan
     la implementación pero no son obligatorios.
