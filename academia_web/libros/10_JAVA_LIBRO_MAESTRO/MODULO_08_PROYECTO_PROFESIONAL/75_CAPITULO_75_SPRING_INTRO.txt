  75.1. ¿QUÉ ES SPRING FRAMEWORK?
-------------------------------------------------------------------------------

Spring Framework es un framework de código abierto para el desarrollo de
aplicaciones empresariales en Java. Creado por Rod Johnson en 2002 y
publicado inicialmente en 2003, Spring revolucionó el desarrollo Java al
ofrecer una alternativa más simple y menos intrusiva a los EJB (Enterprise
JavaBeans).

  MÓDULOS PRINCIPALES DE SPRING:

-------------------------------------------------------------------------------
  75.2. EL PROBLEMA QUE SPRING RESUELVE
-------------------------------------------------------------------------------

Antes de entender Spring, debemos entender el problema que resuelve.

  EL PROBLEMA: ACOPLAMIENTO DIRECTO

```java
// Sin Spring: acoplamiento fuerte
public class ServicioUsuario {
    private RepositorioUsuario repositorio;

    public ServicioUsuario() {
        // MAL: la clase crea sus propias dependencias
        this.repositorio = new RepositorioUsuarioMySQL();
    }

    public Usuario buscarPorId(Long id) {
        return repositorio.findById(id);
    }
}
```

  Problemas de este enfoque:
  1. ServicioUsuario está acoplado a RepositorioUsuarioMySQL
  2. Si cambiamos de base de datos, modificamos el código
  3. Difícil de testear (no podemos usar un mock)
  4. Difícil de extender

  LA SOLUCIÓN: INVERSIÓN DE CONTROL (IoC)

```java
// Con Spring: el control se invierte
@Service
public class ServicioUsuario {

    private final RepositorioUsuario repositorio;

    // Spring inyecta la dependencia automáticamente
    public ServicioUsuario(RepositorioUsuario repositorio) {
        this.repositorio = repositorio;
    }

    public Usuario buscarPorId(Long id) {
        return repositorio.findById(id);
    }
}

@Repository
public class RepositorioUsuarioMySQL implements RepositorioUsuario {
    // implementación con MySQL
}

@Repository
public class RepositorioUsuarioMock implements RepositorioUsuario {
    // implementación mock para pruebas
}
```

  Ventajas de IoC/DI:
  1. Bajo acoplamiento: la clase no crea sus dependencias
  2. Fácil testing: podemos inyectar mocks
  3. Flexibilidad: cambiamos implementaciones sin modificar código
  4. Código más limpio y mantenible

-------------------------------------------------------------------------------
  75.3. INVERSIÓN DE CONTROL (IoC)
-------------------------------------------------------------------------------

La Inversión de Control es un principio de diseño donde el flujo de
control de un programa se invierte: en lugar de que tu código controle
el flujo, un contenedor externo (Spring IoC Container) lo controla.

  "Hollywood Principle": "Don't call us, we'll call you."
  (No nos llames, nosotros te llamaremos.)

  SIN IoC (tú controlas):

```java
public class Main {
    public static void main(String[] args) {
        ServicioUsuario servicio = new ServicioUsuario();
        servicio.buscarPorId(1L);
    }
}
```

  CON IoC (Spring controla):

```java
@SpringBootApplication
public class Main {
    public static void main(String[] args) {
        // Spring inicia, crea el contexto, instancia beans,
        // inyecta dependencias, y ejecuta la aplicación
        SpringApplication.run(Main.class, args);
    }
}
```

-------------------------------------------------------------------------------
  75.4. DEPENDENCY INJECTION (DI)
-------------------------------------------------------------------------------

La Inyección de Dependencias es el patrón mediante el cual el contenedor
IoC suministra las dependencias a un objeto en lugar de que el objeto
las cree por sí mismo.

  TIPOS DE INYECCIÓN:

  75.4.1. INYECCIÓN POR CONSTRUCTOR (RECOMENDADA)

```java
@Component
public class ServicioCliente {
    private final RepositorioCliente repositorio;
    private final ServicioNotificacion notificacion;

    // Spring inyecta automáticamente las dependencias
    public ServicioCliente(
            RepositorioCliente repositorio,
            ServicioNotificacion notificacion) {
        this.repositorio = repositorio;
        this.notificacion = notificacion;
    }
}
```

  Ventajas:
  - Dependencias inmutables (final)
  - Fácil testing (constructor con parámetros)
  - Spring puede detectar dependencias circulares
  - Claro qué necesita la clase para funcionar

  75.4.2. INYECCIÓN POR SETTER

```java
@Component
public class ServicioCliente {
    private RepositorioCliente repositorio;
    private ServicioNotificacion notificacion;

    @Autowired
    public void setRepositorio(RepositorioCliente repositorio) {
        this.repositorio = repositorio;
    }

    @Autowired
    public void setNotificacion(ServicioNotificacion notificacion) {
        this.notificacion = notificacion;
    }
}
```

  75.4.3. INYECCIÓN POR CAMPO (NO RECOMENDADA)

```java
@Component
public class ServicioCliente {
    @Autowired
    private RepositorioCliente repositorio;

    @Autowired
    private ServicioNotificacion notificacion;
}
```

  Desventajas:
  - Dependencias no finales
  - Difícil testing (requiere reflexión)
  - Oculta las dependencias de la clase
  - Viola el principio de inmutabilidad

-------------------------------------------------------------------------------
  75.5. EL CONTENEDOR IOC: APPLICATION CONTEXT
-------------------------------------------------------------------------------

El ApplicationContext es el contenedor central de Spring. Gestiona el ciclo
de vida de los Beans, resuelve dependencias, y proporciona servicios
adicionales.

```java
package com.lucasega.spring.ejemplos;

import org.springframework.context.ApplicationContext;
import org.springframework.context.annotation.AnnotationConfigApplicationContext;
import org.springframework.context.support.ClassPathXmlApplicationContext;

public class EjemploApplicationContext {

    public static void main(String[] args) {
        // === 1. Contexto con configuración Java (recomendado) ===
        ApplicationContext context = new AnnotationConfigApplicationContext(
            AppConfig.class
        );

        // Obtener un bean por tipo
        ServicioSaludo servicio = context.getBean(ServicioSaludo.class);
        servicio.saludar("Lucas");

        // Obtener un bean por nombre
        Object bean = context.getBean("miServicio");

        // Verificar si existe un bean
        boolean existe = context.containsBean("miServicio");

        // === 2. Contexto con XML (legado) ===
        ApplicationContext xmlContext = new ClassPathXmlApplicationContext(
            "applicationContext.xml"
        );

        // === 3. Cerrar el contexto correctamente ===
        ((AnnotationConfigApplicationContext) context).close();

        // === 4. Registrar shutdown hook para cierre automático ===
        ((AnnotationConfigApplicationContext) context).registerShutdownHook();
    }
}
```

  75.6. CONFIGURACIÓN DE SPRING
-------------------------------------------------------------------------------

  75.6.1. CONFIGURACIÓN CON JAVA (JAVACONFIG) - RECOMENDADA

```java
package com.lucasega.config;

import com.lucasega.repository.RepositorioUsuario;
import com.lucasega.repository.RepositorioUsuarioJPA;
import com.lucasega.service.ServicioUsuario;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.ComponentScan;
import org.springframework.context.annotation.Configuration;
import org.springframework.context.annotation.Primary;

@Configuration // Indica que esta clase es una fuente de configuración de beans
@ComponentScan(basePackages = "com.lucasega") // Escanea paquetes en busca de componentes
public class AppConfig {

    // Definición explícita de un bean (cuando no puedes usar anotaciones)
    @Bean
    public RepositorioUsuario repositorioUsuario() {
        return new RepositorioUsuarioJPA();
    }

    @Bean
    @Primary // Bean primario si hay múltiples del mismo tipo
    public ServicioUsuario servicioUsuario() {
        return new ServicioUsuario(repositorioUsuario());
    }

    // Bean con nombre personalizado
    @Bean(name = "servicioAlternativo")
    public ServicioUsuario servicioUsuarioAlternativo() {
        return new ServicioUsuario(repositorioAlternativo());
    }

    @Bean
    public RepositorioUsuario repositorioAlternativo() {
        // Implementación alternativa (mock para pruebas, por ejemplo)
        return datos -> System.out.println("Mock: " + datos);
    }
}
```

  75.6.2. CONFIGURACIÓN CON ANOTACIONES (COMPONENT SCANNING)

```java
// Las anotaciones de estereotipo activan el escaneo automático
@Component  // Genérico (cualquier bean)
@Service    // Capa de servicio (lógica de negocio)
@Repository // Capa de persistencia (acceso a datos)
@Controller // Capa web (controladores MVC)
@Configuration // Clase de configuración
```

```java
package com.lucasega.service;

import com.lucasega.repository.RepositorioUsuario;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.stereotype.Service;

@Service // Spring detecta esta anotación y crea un bean
public class ServicioUsuario {

    private final RepositorioUsuario repositorio;

    @Autowired // Spring inyecta la dependencia (opcional en constructor desde Spring 4.3+)
    public ServicioUsuario(RepositorioUsuario repositorio) {
        this.repositorio = repositorio;
    }

    public void registrarUsuario(String nombre, String email) {
        // Lógica de negocio
        repositorio.guardar(new Usuario(nombre, email));
    }
}
```

```java
package com.lucasega.repository;

import com.lucasega.model.Usuario;
import org.springframework.stereotype.Repository;

@Repository // Bean de repositorio (también traduce excepciones JDBC)
public class RepositorioUsuarioJPA implements RepositorioUsuario {

    @Override
    public Usuario guardar(Usuario usuario) {
        System.out.println("Guardando usuario: " + usuario);
        return usuario;
    }

    @Override
    public Usuario buscarPorId(Long id) {
        System.out.println("Buscando usuario: " + id);
        return new Usuario("Juan", "juan@email.com");
    }
}
```

-------------------------------------------------------------------------------
  75.7. CICLO DE VIDA DE UN BEAN
-------------------------------------------------------------------------------

Los Beans en Spring pasan por un ciclo de vida bien definido:

  ┌──────────────────────────────────────────────────────────────────┐
  │                    CICLO DE VIDA DEL BEAN                        │
  │                                                                  │
  │  1. Instanciación: Spring crea la instancia del bean             │
  │  2. Poblado de propiedades: Asigna valores y referencias         │
  │  3. Aware interfaces: setBeanName, setBeanFactory, etc.          │
  │  4. PostProcessBeforeInitialization: BeanPostProcessor pre       │
  │  5. @PostConstruct / afterPropertiesSet: Inicialización          │
  │  6. PostProcessAfterInitialization: BeanPostProcessor post       │
  │  7. Bean listo para usar                                         │
  │  8. @PreDestroy / destroy: Limpieza antes de cerrar              │
  │                                                                  │
  └──────────────────────────────────────────────────────────────────┘

```java
package com.lucasega.beans;

import jakarta.annotation.PostConstruct;
import jakarta.annotation.PreDestroy;
import org.springframework.beans.factory.DisposableBean;
import org.springframework.beans.factory.InitializingBean;
import org.springframework.stereotype.Component;

@Component
public class BeanEjemplo implements InitializingBean, DisposableBean {

    private String nombre;

    public BeanEjemplo() {
        System.out.println("1. Instanciación: constructor");
    }

    // Método 1: Usando anotación @PostConstruct (recomendado)
    @PostConstruct
    public void init() {
        System.out.println("3. @PostConstruct: inicialización personalizada");
        this.nombre = "Bean de ejemplo";
    }

    // Método 2: Implementando InitializingBean
    @Override
    public void afterPropertiesSet() throws Exception {
        System.out.println("4. afterPropertiesSet: propiedades asignadas");
    }

    // Método 3: initMethod en @Bean

    // Método de negocio
    public void trabajar() {
        System.out.println("5. Bean trabajando: " + nombre);
    }

    // @PreDestroy (recomendado)
    @PreDestroy
    public void cleanup() {
        System.out.println("6. @PreDestroy: limpiando recursos");
    }

    // Implementando DisposableBean
    @Override
    public void destroy() throws Exception {
        System.out.println("7. destroy: método de destrucción");
    }
}
```

-------------------------------------------------------------------------------
  75.8. BEAN SCOPES
-------------------------------------------------------------------------------

El scope de un bean define su ciclo de vida y visibilidad.

  ┌──────────────┬──────────────────────────────────────────────────┐
  │ Scope        │ Descripción                                      │
  ├──────────────┼──────────────────────────────────────────────────┤
  │ singleton    │ [Por defecto] Una única instancia por contenedor │
  │              │ IoC. Todos los clientes comparten la misma       │
  │              │ instancia.                                       │
  ├──────────────┼──────────────────────────────────────────────────┤
  │ prototype    │ Una nueva instancia cada vez que se solicita     │
  │              │ el bean. No se gestiona el ciclo de vida         │
  │              │ completo (destroy no se llama).                  │
  ├──────────────┼──────────────────────────────────────────────────┤
  │ request     │ Una instancia por petición HTTP (solo en web)     │
  ├──────────────┼──────────────────────────────────────────────────┤
  │ session     │ Una instancia por sesión HTTP (solo en web)       │
  ├──────────────┼──────────────────────────────────────────────────┤
  │ application │ Una instancia por ServletContext (solo en web)    │
  ├──────────────┼──────────────────────────────────────────────────┤
  │ websocket   │ Una instancia por sesión WebSocket (solo en web)  │
  └──────────────┴──────────────────────────────────────────────────┘

```java
import org.springframework.context.annotation.Scope;
import org.springframework.stereotype.Component;

@Component
@Scope("singleton") // Scope explícito (por defecto es singleton)
public class ServicioCompartido {
    // Una sola instancia para toda la aplicación
    private int contador = 0;

    public int incrementar() {
        return ++contador;
    }
}

@Component
@Scope("prototype")
public class ServicioPorPeticion {
    // Nueva instancia cada vez que se inyecta
    private final String id = java.util.UUID.randomUUID().toString();

    public String getId() {
        return id;
    }
}
```

-------------------------------------------------------------------------------
  75.10. CALIFICADORES (@Qualifier Y @Primary)
-------------------------------------------------------------------------------

Cuando hay múltiples beans del mismo tipo, necesitamos calificarlos.

```java
// Definiciones
@Repository
public class RepositorioUsuarioMySQL implements RepositorioUsuario {
    // ...
}

@Repository
@Primary // Este es el principal (se inyecta por defecto)
public class RepositorioUsuarioPostgreSQL implements RepositorioUsuario {
    // ...
}

@Repository("mongoRepo") // Bean con nombre personalizado
public class RepositorioUsuarioMongoDB implements RepositorioUsuario {
    // ...
}

// --- Uso con @Qualifier ---

@Service
public class ServicioUsuario {

    private final RepositorioUsuario repositorio;

    public ServicioUsuario(
            @Qualifier("repositorioUsuarioMySQL")
            RepositorioUsuario repositorio) {
        this.repositorio = repositorio;
    }
}

@Service
public class ServicioAuditoria {

    private final RepositorioUsuario repositorio;

    public ServicioAuditoria(
            @Qualifier("mongoRepo")
            RepositorioUsuario repositorio) {
        this.repositorio = repositorio;
    }
}
```

  75.11. SPRING EXPRESSION LANGUAGE (SpEL)

@Component
public class BeanConSpEL {

    @Value("#{systemProperties['user.name']}")
    private String nombreUsuario;

    @Value("#{T(java.lang.Math).random() * 100}")
    private double numeroAleatorio;

    @Value("#{servicioUsuario.obtenerNombreDefault()}")
    private String nombreDefault;

    @Value("#{systemProperties['java.home']}")
    private String javaHome;

    @Value("#{42 > 10 ? 'mayor' : 'menor'}")
    private String comparacion;

    @Value("${app.nombre:ValorDefault}") // Property placeholder con default
    private String appNombre;
}

-------------------------------------------------------------------------------
  75.12. APPLICATION EVENTS
-------------------------------------------------------------------------------

Spring permite la comunicación desacoplada mediante eventos.

```java
// 1. Definir el evento
public class UsuarioRegistradoEvent extends ApplicationEvent {
    private final String email;

    public UsuarioRegistradoEvent(Object source, String email) {
        super(source);
        this.email = email;
    }

    public String getEmail() { return email; }
}

// 2. Publicar el evento
@Component
public class ServicioRegistro {
    @Autowired
    private ApplicationEventPublisher publisher;

    public void registrarUsuario(String email) {
        // ... lógica de registro
        publisher.publishEvent(new UsuarioRegistradoEvent(this, email));
    }
}

// 3. Escuchar el evento
@Component
public class ServicioNotificacion {

    @EventListener
    public void handleUsuarioRegistrado(UsuarioRegistradoEvent event) {
        System.out.println("Enviar email de bienvenida a: " + event.getEmail());
    }

    @EventListener(condition = "#event.email.contains('admin')")
    public void handleAdminRegistration(UsuarioRegistradoEvent event) {
        System.out.println("Notificar a seguridad: admin registrado");
    }
}
```

  75.13. ENTORNOS Y PERFILES
-------------------------------------------------------------------------------

Spring Profiles permiten activar/desactivar beans según el entorno.

```java
@Configuration
@Profile("desarrollo")
public class ConfigDesarrollo {
    @Bean
    public DataSource dataSource() {
        return new EmbeddedDatabaseBuilder()
            .setType(EmbeddedDatabaseType.H2)
            .build();
    }
}

@Configuration
@Profile("produccion")
public class ConfigProduccion {
    @Bean
    public DataSource dataSource() {
        HikariConfig config = new HikariConfig();
        config.setJdbcUrl("jdbc:mysql://prod-db:3306/miapp");
        config.setUsername(System.getenv("DB_USER"));
        config.setPassword(System.getenv("DB_PASS"));
        return new HikariDataSource(config);
    }
}

// En application.properties:
// spring.profiles.active=desarrollo

// O al ejecutar:
// java -jar app.jar --spring.profiles.active=produccion
```

  - El singleton mantiene el estado entre inyecciones
  - El prototype crea una nueva instancia cada vez

  - Dos listeners manejen el evento:
    - Enviar email de confirmación
    - Actualizar inventario
  - Verifica que ambos listeners se ejecuten

  RESUMEN DEL CAPÍTULO
-------------------------------------------------------------------------------

  • Spring es el framework empresarial más importante del ecosistema
    Java, basado en los principios de IoC y DI.

  • La Inversión de Control (IoC) invierte el flujo de control:
    el contenedor Spring gestiona el ciclo de vida de los objetos.

  • La Inyección de Dependencias (DI) suministra las dependencias
    desde el exterior, reduciendo el acoplamiento.

  • Spring ofrece tres tipos de inyección: por constructor
    (recomendada), por setter y por campo (no recomendada).

  • El ApplicationContext es el contenedor IoC central que gestiona
    los beans y sus dependencias.

  • Las anotaciones @Component, @Service, @Repository y @Controller
    permiten el descubrimiento automático de beans.

  • La configuración con Java (JavaConfig) es la forma moderna y
    recomendada de configurar Spring.

  • Los beans tienen un ciclo de vida con callbacks: @PostConstruct,
    @PreDestroy, InitializingBean, DisposableBean.

  • Los scopes (singleton, prototype, request, session) definen el
    ciclo de vida y visibilidad de los beans.

  • @Qualifier y @Primary resuelven ambigüedades cuando hay
    múltiples beans del mismo tipo.

  • Los eventos de aplicación permiten comunicación desacoplada
    entre componentes.

  • Los perfiles (Profiles) permiten configuraciones específicas
    por entorno.

================================================================================
  PREGUNTAS FRECUENTES (FAQ)
================================================================================

  ❓ ¿Spring es lo mismo que Spring Boot?
  R: No. Spring Framework es la base. Spring Boot es una extensión que
     simplifica la configuración y el despliegue. Lo veremos en el
     Capítulo 76.

  ❓ ¿Debo usar XML o JavaConfig?
  R: JavaConfig (anotaciones) es el estándar moderno. XML solo se usa
     en proyectos legacy.

  ❓ ¿Qué pasa si hay un error en el constructor de un bean?
  R: Spring propaga la excepción y la aplicación no inicia. Esto es
     bueno: detectas errores de configuración temprano.

  ❓ ¿Cómo inyecto una implementación específica si hay varias?
  R: Usa @Qualifier("nombreDelBean") o @Primary en una de las
     implementaciones.

  ❓ ¿Los beans singleton son thread-safe?
  R: No automáticamente. Si un bean singleton tiene estado mutable,
     debes sincronizar el acceso o usar objetos inmutables.

  ❓ ¿Cómo cierro el ApplicationContext correctamente?
  R: registerShutdownHook() asegura que el contexto se cierre al
     finalizar la JVM, llamando a los métodos @PreDestroy.

  ❓ ¿Puedo tener beans con el mismo nombre en diferentes paquetes?
  R: Técnicamente sí, pero es una mala práctica. El nombre del bean
     se deriva del nombre de la clase (en camelCase).

  🏆 MINI-FRAMEWORK IOC

  Implementa tu propio contenedor IoC ligero en Java que emule el
  comportamiento básico de Spring:

  1. Escáner de componentes:
     - Encuentra clases anotadas con @MiComponente en el classpath
     - Usa reflexión para analizar las clases

  2. Contenedor de dependencias:
     - Instancia los beans registrados
     - Resuelve dependencias por constructor
     - Detecta y maneja dependencias circulares

  3. Inyección automática:
     - Implementa @MiAutowired en constructores
     - Soporta @MiQualifier para beans con nombre
     - Maneja scopes singleton y prototype

  4. Ciclo de vida:
     - @MiPostConstruct y @MiPreDestroy
     - BeanPostProcessor personalizado

  5. Mensajes de error claros:
     - Bean no encontrado
     - Dependencia circular
     - Ambigüedad

  💡 PISTA: Usa reflexión (java.lang.reflect), un ConcurrentHashMap
  como registro de beans, y un grafo para detectar ciclos.

  ¡Este desafío te hará entender IoC a un nivel profundo!
