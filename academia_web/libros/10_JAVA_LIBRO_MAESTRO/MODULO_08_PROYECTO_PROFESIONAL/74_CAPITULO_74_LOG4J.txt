-------------------------------------------------------------------------------
  74.1. ¿QUÉ ES LOG4J?
-------------------------------------------------------------------------------

Log4j es un framework de logging para Java, creado originalmente por
Ceki Gülcü en 1999 y mantenido por Apache Software Foundation.

  LOG4J 1.x (Legado - Fin de vida en 2015):
  - La versión original, ampliamente adoptada
  - Problemas de rendimiento y bugs críticos
  - NO USAR EN PROYECTOS NUEVOS

  LOG4J 2.x (Actual - Desde 2014):
  - Reescribo completo desde cero
  - Mucho más rápido (logging asíncrono)
  - Configuración flexible (XML, JSON, YAML, Properties)
  - Plugins personalizables
  - API limpia y moderna
  - Soporte para lambdas (evaluación perezosa)

  ALTERNATIVAS DE LOGGING:

  ┌─────────────────┬──────────────────┬──────────────────┐
  │ Framework       │ Ventajas         │ Desventajas      │
  ├─────────────────┼──────────────────┼──────────────────┤
  │ Log4j 2         │ Rápido, flexible │ Complejidad      │
  │                 │ m últiples conf  │                  │
  │ Logback         │ Nativo SLF4J     │ Menos flexible   │
  │ java.util.log   │ Integrado en JDK │ Limitado, lento  │
  │ .logging (JUL)  │                  │                  │
  │ SLF4J (fachada) │ Independiente    │ Requiere         │
  │                 │ de implementación│ implementación   │
  └─────────────────┴──────────────────┴──────────────────┘

-------------------------------------------------------------------------------
  74.2. ARQUITECTURA DE LOG4J 2
-------------------------------------------------------------------------------

Log4j 2 se compone de los siguientes componentes principales:

  ┌─────────────────────────────────────────────────────────────────┐
  │                     ARQUITECTURA LOG4J 2                        │
  │                                                                 │
  │   Logger          Appender         Layout                       │
  │   (Logger)        (Destino)        (Formato)                    │
  │                                                                 │
  │  ┌─────────┐     ┌──────────┐     ┌──────────┐                 │
  │  │Logger A │────►│ Console  │────►│ Pattern  │                 │
  │  │ (name=  │     │ Appender │     │ Layout   │                 │
  │  │ "com.   │     └──────────┘     └──────────┘                 │
  │  │ lucasega│                                                    │
  │  │ .App")  │     ┌──────────┐     ┌──────────┐                 │
  │  │         │────►│  File    │────►│ Pattern  │                 │
  │  └─────────┘     │ Appender │     │ Layout   │                 │
  │                  └──────────┘     └──────────┘                 │
  │                                                                 │
  │   Filter          Filter          Filter                        │
  │  (Pre-filtro)   (Post-filtro)  (En Appender)                   │
  │                                                                 │
  └─────────────────────────────────────────────────────────────────┘

  • Logger: Objeto que emite mensajes de log. Organizado jerárquicamente
    por nombre (basado en paquetes).

  • Appender: Destino del mensaje de log (consola, archivo, base de
    datos, red, etc.).

  • Layout: Formatea el mensaje de log antes de enviarlo al Appender.

  • Filter: Decide si un mensaje debe ser registrado o no, en diferentes
    puntos del pipeline.

-------------------------------------------------------------------------------
  74.3. NIVELES DE LOG
-------------------------------------------------------------------------------

Los niveles de log definen la severidad/importancia de un mensaje.
Log4j 2 define 6 niveles estándar, de menor a mayor severidad:

  ┌──────────┬───────┬────────────────────────────────────────────────┐
  │ Nivel    │ Valor │ Uso                                           │
  ├──────────┼───────┼────────────────────────────────────────────────┤
  │ TRACE    │   600 │ Depuración muy detallada. Flujo completo       │
  │          │       │ de ejecución. Solo en desarrollo.              │
  ├──────────┼───────┼────────────────────────────────────────────────┤
  │ DEBUG    │   500 │ Información de depuración. Valores de          │
  │          │       │ variables, estado de objetos.                  │
  ├──────────┼───────┼────────────────────────────────────────────────┤
  │ INFO     │   400 │ Eventos importantes del sistema. Inicio/fin    │
  │          │       │ de procesos, conexiones, transacciones.        │
  ├──────────┼───────┼────────────────────────────────────────────────┤
  │ WARN     │   300 │ Situaciones inesperadas pero no críticas.      │
  │          │       │ Configuración incorrecta, recurso bajo.        │
  ├──────────┼───────┼────────────────────────────────────────────────┤
  │ ERROR    │   200 │ Errores que requieren atención. Operación      │
  │          │       │ fallida, excepción atrapada.                   │
  ├──────────┼───────┼────────────────────────────────────────────────┤
  │ FATAL    │   100 │ Errores catastróficos. La aplicación no        │
  │          │       │ puede continuar. Pérdida de datos.             │
  └──────────┴───────┴────────────────────────────────────────────────┘

  REGLA DE ORO: El nivel de log configurado en producción determina
  qué mensajes se registran. Si configuras WARN, solo se registrarán
  mensajes WARN, ERROR y FATAL. Los mensajes TRACE, DEBUG e INFO se
  ignoran.

  74.5. CONFIGURACIÓN DE LOG4J 2
-------------------------------------------------------------------------------

Log4j 2 busca automáticamente archivos de configuración en el classpath
en el siguiente orden:
  1. log4j2-test.properties (para tests)
  2. log4j2-test.yaml
  3. log4j2-test.json
  4. log4j2-test.xml
  5. log4j2.properties
  6. log4j2.yaml
  7. log4j2.json
  8. log4j2.xml

  74.5.1. CONFIGURACIÓN EN XML

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!-- src/main/resources/log4j2.xml -->
<Configuration status="WARN" monitorInterval="30">
    <!--
        status: nivel de log para el propio Log4j
        monitorInterval: recargar configuración cada 30 segundos
    -->

    <!-- Propiedades reutilizables -->
    <Properties>
        <Property name="basePath">/var/log/mi-app</Property>
        <Property name="pattern">
            %d{yyyy-MM-dd HH:mm:ss.SSS} [%t] %-5level %logger{36} - %msg%n
        </Property>
        <Property name="patternConsola">
            %d{HH:mm:ss.SSS} %highlight{%-5level} %style{%logger{36}}{cyan} - %msg%n
        </Property>
    </Properties>

    <!-- Appenders (destinos de los logs) -->
    <Appenders>
        <!-- Consola (con colores) -->
        <Console name="Console" target="SYSTEM_OUT">
            <PatternLayout pattern="${patternConsola}"
                           disableAnsi="false"/>
        </Console>

        <!-- Archivo rotatorio por tamaño -->
        <RollingFile name="RollingFile"
                     fileName="${basePath}/app.log"
                     filePattern="${basePath}/app.%d{yyyy-MM-dd}-%i.log.gz">
            <PatternLayout pattern="${pattern}"/>
            <Policies>
                <!-- Rotar cuando el archivo llegue a 10 MB -->
                <SizeBasedTriggeringPolicy size="10 MB"/>
                <!-- Rotar diariamente -->
                <TimeBasedTriggeringPolicy interval="1"
                                           modulate="true"/>
            </Policies>
            <!-- Mantener máximo 30 archivos -->
            <DefaultRolloverStrategy max="30"/>
        </RollingFile>

        <!-- Archivo solo para errores -->
        <RollingFile name="ErrorFile"
                     fileName="${basePath}/error.log"
                     filePattern="${basePath}/error.%d{yyyy-MM-dd}-%i.log.gz">
            <PatternLayout pattern="${pattern}"/>
            <Policies>
                <SizeBasedTriggeringPolicy size="10 MB"/>
            </Policies>
            <DefaultRolloverStrategy max="10"/>
        </RollingFile>

        <!-- Archivo JSON estructurado -->
        <RollingFile name="JsonFile"
                     fileName="${basePath}/app.json"
                     filePattern="${basePath}/app.%d{yyyy-MM-dd}.json.gz">
            <JsonLayout compact="true" eventEol="true"
                       properties="true"/>
            <Policies>
                <TimeBasedTriggeringPolicy interval="1"
                                           modulate="true"/>
            </Policies>
        </RollingFile>
    </Appenders>

    <!-- Loggers -->
    <Loggers>
        <!-- Logger raíz -->
        <Root level="INFO">
            <AppenderRef ref="Console"/>
            <AppenderRef ref="RollingFile"/>
        </Root>

        <!-- Logger específico para paquete (nivel DEBUG) -->
        <Logger name="com.lucasega" level="DEBUG" additivity="false">
            <AppenderRef ref="Console"/>
            <AppenderRef ref="RollingFile"/>
            <AppenderRef ref="ErrorFile" level="ERROR"/>
        </Logger>

        <!-- Logger para Spring (nivel WARN para reducir ruido) -->
        <Logger name="org.springframework" level="WARN"/>

        <!-- Logger para Hibernate (nivel WARN) -->
        <Logger name="org.hibernate" level="WARN"/>

        <!-- Logger para SQL (nivel DEBUG para ver queries) -->
        <Logger name="org.hibernate.SQL" level="DEBUG"/>
    </Loggers>
</Configuration>
```

  74.6.2. MENSAJES CON TODOS LOS NIVELES

```java
package com.lucasega.app;

import org.apache.logging.log4j.LogManager;
import org.apache.logging.log4j.Logger;

public class EjemploLogging {

    private static final Logger log = LogManager.getLogger(EjemploLogging.class);

    public static void main(String[] args) {
        // Mensajes en todos los niveles
        log.trace("Mensaje TRACE: flujo detallado");
        log.debug("Mensaje DEBUG: valor de x = {}", calcularValor());
        log.info("Mensaje INFO: aplicación iniciada correctamente");
        log.warn("Mensaje WARN: memoria disponible baja: {} MB", 
                 Runtime.getRuntime().freeMemory() / 1024 / 1024);
        
        try {
            operacionRiesgosa();
        } catch (Exception e) {
            log.error("Mensaje ERROR: operación fallida", e);
        }

        if (hayErrorCritico()) {
            log.fatal("Mensaje FATAL: no se puede continuar. Cerrando aplicación.");
            System.exit(1);
        }
    }

    private static int calcularValor() {
        // Método costoso que solo se llama si DEBUG está activo
        return 42;
    }

    private static void operacionRiesgosa() throws Exception {
        throw new Exception("Error simulado");
    }

    private static boolean hayErrorCritico() {
        return false;
    }
}
```

  74.6.3. EVALUACIÓN PERZOSA CON LAMBDAS

```java
// Sin lambda (el mensaje se construye siempre, aunque DEBUG esté desactivado)
log.debug("El valor calculado es: " + metodoCostoso() + " y el otro es: " + otroMetodoCostoso());

// Con lambda (el mensaje solo se construye si DEBUG está activo)
log.debug("El valor calculado es: {}", () -> metodoCostoso());
log.debug("Valores: {} y {}", () -> metodoCostoso(), () -> otroMetodoCostoso());

// Mejor aún: verificar nivel antes de loguear
if (log.isDebugEnabled()) {
    log.debug("Valor complejo: {}", construirMensajeComplejo());
}
```

  74.6.4. LOGGER CON PARÁMETROS

```java
// Los placeholders {} evitan la concatenación de strings
log.info("Usuario '{}' realizó la operación '{}' en el recurso '{}'",
         usuario, operacion, recursoId);

// El logging parametrizado es más eficiente que la concatenación
// INCORRECTA (ineficiente):
log.info("Usuario " + usuario + " realizó " + operacion);

// CORRECTA (eficiente):
log.info("Usuario '{}' realizó '{}'", usuario, operacion);
```

-------------------------------------------------------------------------------
  74.7. PATTERN LAYOUT
-------------------------------------------------------------------------------

El Pattern Layout es el formato más común para los mensajes de log.
Entender sus patrones es esencial para diseñar logs efectivos.

  CONVERSORES PRINCIPALES:

  %d{patron}     → Fecha/hora. Ej: %d{yyyy-MM-dd HH:mm:ss.SSS}
  %p / %-5level  → Nivel del log (INFO, DEBUG, etc.) con padding
  %t             → Nombre del hilo (thread)
  %logger{len}   → Nombre del logger (clase). {len} limita longitud
  %m / %msg      → Mensaje del log
  %n             → Salto de línea (plataforma específica)
  %F             → Nombre del archivo fuente
  %L             → Número de línea (costoso, activar solo en debug)
  %M             → Nombre del método (costoso)
  %C             → Nombre completo de la clase
  %r             → Milisegundos desde inicio de la app
  %i             → Índice de rotación
  %X{key}        → Valor del MDC (Mapped Diagnostic Context)
  %%             → Signo de porcentaje literal

  PATRONES COMUNES:

```java
// Patrón estándar (producción)
PatternLayout pattern = "%d{yyyy-MM-dd HH:mm:ss.SSS} [%t] %-5level %logger{36} - %msg%n";

// Patrón con colores (consola)
PatternLayout pattern = "%d{HH:mm:ss.SSS} %highlight{%-5level} %style{%logger{36}}{cyan} - %msg%n";

// Patrón detallado (desarrollo)
PatternLayout pattern = "%d{yyyy-MM-dd HH:mm:ss.SSS} [%t] %-5level %logger.%M(%F:%L) - %msg%n";

// Patrón mínimo
PatternLayout pattern = "%-5level: %msg%n";

// Patrón con MDC (para trazabilidad de peticiones)
PatternLayout pattern = "%d{ISO8601} [%X{requestId}] [%t] %-5level %logger{36} - %msg%n";
```

  74.8.3. ROLLING FILE APPENDER (más usado)

```xml
<RollingFile name="RollingFile"
             fileName="${basePath}/app.log"
             filePattern="${basePath}/app-%d{yyyy-MM-dd}-%i.log.gz">
    <PatternLayout pattern="%d{yyyy-MM-dd HH:mm:ss.SSS} [%t] %-5level %logger{36} - %msg%n"/>
    <Policies>
        <SizeBasedTriggeringPolicy size="10 MB"/>
        <TimeBasedTriggeringPolicy interval="1" modulate="true"/>
    </Policies>
    <DefaultRolloverStrategy max="30"/>
</RollingFile>
```

  74.8.4. SOCKET APPENDER (logs remotos)


```java
package com.lucasega.db;

import javax.sql.DataSource;
import com.zaxxer.hikari.HikariConfig;
import com.zaxxer.hikari.HikariDataSource;
import java.sql.Connection;
import java.sql.SQLException;

public class LogConnectionFactory {

    private static final DataSource ds;

    static {
        HikariConfig config = new HikariConfig();
        config.setJdbcUrl("jdbc:postgresql://localhost:5432/logs");
        config.setUsername("loguser");
        config.setPassword(System.getenv("LOG_DB_PASSWORD"));
        config.setMaximumPoolSize(5);
        ds = new HikariDataSource(config);
    }

    public static Connection getConnection() throws SQLException {
        return ds.getConnection();
    }
}
```

  74.9. FILTROS
-------------------------------------------------------------------------------

Los filtros controlan qué mensajes pasan al Appender.

  74.9.1. THRESHOLD FILTER (por nivel)

```xml
<!-- Solo ERROR y FATAL -->
<ThresholdFilter level="ERROR" onMatch="ACCEPT" onMismatch="DENY"/>
```

  74.9.2. MARKER FILTER (por marcador)

```java
import org.apache.logging.log4j.Marker;
import org.apache.logging.log4j.MarkerManager;

Marker SEGURIDAD = MarkerManager.getMarker("SEGURIDAD");
```

-------------------------------------------------------------------------------
  74.10. LOGGING ASÍNCRONO
-------------------------------------------------------------------------------

El logging síncrono puede ralentizar la aplicación. Log4j 2 ofrece logging
asíncrono usando LMAX Disruptor, un ring buffer de alto rendimiento.

```xml
<!-- log4j2.xml con logging asíncrono -->
<Configuration status="WARN">
    <Appenders>
        <!-- Async Appender envuelve a un appender síncrono -->
        <Async name="AsyncConsole" bufferSize="1024">
            <AppenderRef ref="Console"/>
        </Async>
        <Async name="AsyncFile" bufferSize="1024">
            <AppenderRef ref="RollingFile"/>
        </Async>
    </Appenders>

    <Loggers>
        <Root level="INFO">
            <AppenderRef ref="AsyncConsole"/>
            <AppenderRef ref="AsyncFile"/>
        </Root>
    </Loggers>
</Configuration>
```

  También puedes usar el logger asíncrono completo (requiere Disruptor):

```xml
<Configuration status="WARN">
    <Appenders>
        <Console name="Console">
            <PatternLayout pattern="%d{HH:mm:ss.SSS} [%t] %-5level %logger{36} - %msg%n"/>
        </Console>
    </Appenders>

    <Loggers>
        <!-- Logger asíncrono (toda la jerarquía es asíncrona) -->
        <AsyncLogger name="com.lucasega" level="DEBUG" includeLocation="false">
            <AppenderRef ref="Console"/>
        </AsyncLogger>
        <AsyncRoot level="INFO" includeLocation="false">
            <AppenderRef ref="Console"/>
        </AsyncRoot>
    </Loggers>
</Configuration>
```

  BENEFICIOS DEL LOGGING ASÍNCRONO:
  - Latencia de hasta 10 microsegundos vs 1-10 milisegundos síncrono
  - El hilo principal no se bloquea mientras se escribe el log
  - Ideal para aplicaciones de alta concurrencia
  - Throughput mucho mayor (millones de mensajes por segundo)

  CONSIDERACIONES:
  - Si la aplicación se cae, se pierden los mensajes en el buffer
  - includeLocation=false para mejor rendimiento
  - Consume más memoria (buffer circular)

-------------------------------------------------------------------------------
  74.11. MDC (MAPPED DIAGNOSTIC CONTEXT)
-------------------------------------------------------------------------------

MDC permite añadir contexto a los mensajes de log, ideal para trazabilidad
en aplicaciones multi-hilo.

```java
package com.lucasega.app;

import org.apache.logging.log4j.LogManager;
import org.apache.logging.log4j.Logger;
import org.apache.logging.log4j.ThreadContext;

import java.util.UUID;

public class ServicioUsuarios {

    private static final Logger log = LogManager.getLogger(ServicioUsuarios.class);

    public void procesarPeticion(String usuario, String accion) {
        // Añadir contexto al hilo actual
        ThreadContext.put("requestId", UUID.randomUUID().toString());
        ThreadContext.put("usuario", usuario);
        ThreadContext.put("accion", accion);

        try {
            log.info("Iniciando procesamiento de petición");
            // ... lógica de negocio
            log.info("Petición procesada exitosamente");
        } catch (Exception e) {
            log.error("Error procesando petición", e);
        } finally {
            // Limpiar contexto (¡siempre en finally!)
            ThreadContext.clearAll();
        }
    }
}
```

```xml
<!-- Configuración que usa MDC en el patrón -->
<PatternLayout pattern="%d{ISO8601} [%X{requestId}] [%X{usuario}] [%t] %-5level %logger{36} - %msg%n"/>
```

  74.12. INTEGRACIÓN CON SLF4J
-------------------------------------------------------------------------------

SLF4J (Simple Logging Facade for Java) es una fachada que abstrae la
implementación de logging. Permite cambiar de implementación sin modificar
el código.

```java
package com.lucasega.app;

import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

public class AppConSLF4J {

    // Usando SLF4J (independiente de la implementación)
    private static final Logger log = LoggerFactory.getLogger(AppConSLF4J.class);

    public static void main(String[] args) {
        log.info("Aplicación iniciada con SLF4J + Log4j 2");
        log.debug("Valor de configuración: {}", System.getProperty("java.version"));

        try {
            // Simular error
            throw new RuntimeException("Error de prueba");
        } catch (Exception e) {
            log.error("Error crítico en la aplicación", e);
        }
    }
}
```

  DEPENDENCIAS NECESARIAS:

```xml
<dependencies>
    <!-- SLF4J API -->
    <dependency>
        <groupId>org.slf4j</groupId>
        <artifactId>slf4j-api</artifactId>
        <version>2.0.11</version>
    </dependency>

    <!-- Binding SLF4J → Log4j 2 -->
    <dependency>
        <groupId>org.apache.logging.log4j</groupId>
        <artifactId>log4j-slf4j2-impl</artifactId>
        <version>2.22.1</version>
    </dependency>

    <!-- Log4j 2 Core (implementación) -->
    <dependency>
        <groupId>org.apache.logging.log4j</groupId>
        <artifactId>log4j-core</artifactId>
        <version>2.22.1</version>
    </dependency>
</dependencies>
```

  ¿POR QUÉ USAR SLF4J?
  • Desacopla el código de la implementación de logging
  • Permite cambiar a Logback, JUL, etc. sin modificar código
  • Es el estándar de facto en la comunidad Java
  • Spring Boot lo usa por defecto

-------------------------------------------------------------------------------
  74.13. MEJORES PRÁCTICAS DE LOGGING
-------------------------------------------------------------------------------

  1. Usa el nivel adecuado
     TRACE: Depuración muy detallada (flujo de ejecución)
     DEBUG: Información de depuración general
     INFO: Eventos importantes del negocio
     WARN: Situaciones inesperadas no críticas
     ERROR: Errores que requieren intervención
     FATAL: Errores catastróficos

  2. Mensajes descriptivos y contextuales
     ✅ "Usuario 'juan' realizó transferencia de $500 a cuenta '9876'"
     ❌ "Transferencia realizada"

  3. Incluye suficiente contexto
     - ID de transacción
     - Usuario
     - Recurso afectado
     - Duración de la operación
     - Valores relevantes

  4. No loguees información sensible
     ❌ log.info("Login: usuario={}, password={}", user, pass);
     ❌ log.info("Tarjeta: {}", numeroTarjeta);
     ✅ log.info("Login: usuario={}", user);
     ✅ log.info("Pago procesado: tarjeta terminada en {}", ultimosDigitos);

  5. Usa parámetros, no concatenación
     ✅ log.info("Usuario {} logueado", usuario);
     ❌ log.info("Usuario " + usuario + " logueado");

  6. Siempre incluye la excepción completa
     ✅ log.error("Error al procesar pago", e);
     ❌ log.error("Error al procesar pago: " + e.getMessage());

  7. No loguees en catch y luego lances de nuevo
     ❌ catch (e) { log.error("Error", e); throw e; }  // Log duplicado
     ✅ catch (e) { throw new AppException("Error", e); } // Log en el nivel superior

  8. Configura niveles por paquete
     log4j.logger.com.lucasega=DEBUG
     log4j.logger.org.springframework=WARN
     log4j.logger.org.hibernate.SQL=DEBUG

  9. Usa MDC para contexto transversal
     - requestId, sessionId, userId, transactionId

  10. Archivos rotatorios
      - Por tamaño (10-100 MB)
      - Por tiempo (diario)
      - Compresión (gzip)
      - Retención (30 días)

  11. Logging asíncrono en producción
      - No bloquees el hilo principal
      - Usa Disruptor para alto rendimiento

  12. Monitoreo y alertas
      - Integra logs con sistemas de monitoreo
      - Alerta en ERROR/FATAL
      - Dashboards con métricas de logging

  74.14. SEGURIDAD EN LOG4J 2
-------------------------------------------------------------------------------

La vulnerabilidad Log4Shell (CVE-2021-44228) fue crítica porque permitía
ejecución remota de código mediante JNDI Lookup.

  MEDIDAS DE SEGURIDAD:

```xml
<!-- Configuración segura: deshabilitar JNDI -->
<Configuration status="WARN">
    <!-- Deshabilitar lookup JNDI -->
    <Properties>
        <Property name="log4j2.enableJndiLookup">false</Property>
        <Property name="log4j2.enableJndiJms">false</Property>
    </Properties>
    <!-- ... resto de la configuración -->
</Configuration>
```

```bash
# O desde la línea de comandos:
-Dlog4j2.enableJndiLookup=false
-Dlog4j2.enableJndiJms=false
-Dlog4j2.enableJndiContextSelector=false

# Usar la última versión de Log4j (2.17.0+ parchea Log4Shell)
# Verificar versión:
mvn versions:display-dependency-updates | grep log4j
```

  RECOMENDACIONES DE SEGURIDAD:
  1. Usa Log4j 2.17.0 o superior
  2. Deshabilita JNDI si no lo necesitas
  3. No loguees datos sensibles (contraseñas, tokens, tarjetas)
  4. Usa filtros para sanear mensajes
  5. Monitorea logs en busca de patrones sospechosos
  6. Rota y comprime logs regularmente
  7. Restringe acceso a archivos de log

  74.15. EJEMPLO COMPLETO: APLICACIÓN CON LOGGING PROFESIONAL
-------------------------------------------------------------------------------

```java
package com.lucasega.app;

import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
import org.slf4j.MDC;

import java.time.Duration;
import java.time.Instant;
import java.util.UUID;

public class ProcesadorPagos {

    private static final Logger log = LoggerFactory.getLogger(ProcesadorPagos.class);

    public ResultadoPago procesarPago(Pago pago) {
        // Generar ID de transacción y establecer contexto
        String transaccionId = UUID.randomUUID().toString();
        MDC.put("transaccionId", transaccionId);
        MDC.put("usuarioId", pago.getUsuarioId());

        Instant inicio = Instant.now();

        try {
            log.info("Iniciando procesamiento de pago: monto={}, moneda={}, metodo={}",
                     pago.getMonto(), pago.getMoneda(), pago.getMetodo());

            // Validar pago
            validarPago(pago);

            // Procesar con gateway
            log.debug("Enviando pago a gateway: {}", pago.getGatewayEndpoint());
            GatewayResponse respuesta = gateway.procesar(pago);

            // Verificar resultado
            if (respuesta.isExitosa()) {
                Duration duracion = Duration.between(inicio, Instant.now());
                log.info("Pago procesado exitosamente: transaccionId={}, duracionMs={}",
                         transaccionId, duracion.toMillis());

                // Loguear auditoría
                log.info("[AUDITORIA] Pago exitoso: usuario={}, monto={}, metodo={}",
                         pago.getUsuarioId(), pago.getMonto(), pago.getMetodo());

                return new ResultadoPago(true, transaccionId, respuesta.getCodigoAutorizacion());
            } else {
                log.warn("Pago rechazado por gateway: codigo={}, motivo={}",
                         respuesta.getCodigo(), respuesta.getMotivo());
                return new ResultadoPago(false, transaccionId, null);
            }

        } catch (PagoInvalidoException e) {
            log.warn("Pago inválido: {}", e.getMessage());
            return new ResultadoPago(false, transaccionId, null);

        } catch (GatewayTimeoutException e) {
            log.error("Timeout comunicándose con gateway: {}", pago.getGatewayEndpoint(), e);
            // Intentar reintentar
            return reintentarPago(pago, transaccionId);

        } catch (Exception e) {
            log.error("Error inesperado procesando pago: transaccionId={}", transaccionId, e);
            return new ResultadoPago(false, transaccionId, null);

        } finally {
            // Registrar tiempo total
            Duration total = Duration.between(inicio, Instant.now());
            log.debug("Procesamiento de pago completado en {}ms", total.toMillis());
            // Limpiar MDC
            MDC.clear();
        }
    }

    private ResultadoPago reintentarPago(Pago pago, String transaccionId) {
        log.info("Reintentando pago: transaccionId={}, intento=2", transaccionId);
        try {
            Thread.sleep(1000);
            GatewayResponse respuesta = gateway.procesar(pago);
            if (respuesta.isExitosa()) {
                return new ResultadoPago(true, transaccionId, respuesta.getCodigoAutorizacion());
            }
        } catch (Exception e) {
            log.error("Reintento fallido: transaccionId={}", transaccionId, e);
        }
        return new ResultadoPago(false, transaccionId, null);
    }

    private void validarPago(Pago pago) throws PagoInvalidoException {
        if (pago.getMonto() <= 0) {
            throw new PagoInvalidoException("Monto debe ser positivo");
        }
        if (pago.getMoneda() == null || pago.getMoneda().length() != 3) {
            throw new PagoInvalidoException("Moneda inválida");
        }
    }
}

// Clases auxiliares
class Pago {
    private String usuarioId;
    private double monto;
    private String moneda;
    private String metodo;
    private String gatewayEndpoint;
    // getters y setters
}

class GatewayResponse {
    private boolean exitosa;
    private String codigo;
    private String motivo;
    private String codigoAutorizacion;
    // getters y setters
}

class ResultadoPago {
    private final boolean exitosa;
    private final String transaccionId;
    private final String codigoAutorizacion;
    // constructor, getters
}

class PagoInvalidoException extends Exception {
    public PagoInvalidoException(String message) { super(message); }
}

class GatewayTimeoutException extends Exception {
    public GatewayTimeoutException(String message, Throwable cause) { super(message, cause); }
}
```

  RESUMEN DEL CAPÍTULO
-------------------------------------------------------------------------------

  • Log4j 2 es el framework de logging más potente y flexible del
    ecosistema Java.

  • Los niveles de log (TRACE, DEBUG, INFO, WARN, ERROR, FATAL)
    permiten categorizar la severidad de los mensajes.

  • Los Appenders definen el destino de los logs: consola, archivo,
    base de datos, socket, email, etc.

  • El Pattern Layout formatea los mensajes con conversores como
    %d, %p, %t, %logger, %msg.

  • Los archivos rotatorios (RollingFile) evitan que los logs
    crezcan indefinidamente.

  • Los filtros controlan qué mensajes se registran usando Threshold,
    Marker, Script, etc.

  • SLF4J es una fachada que abstrae la implementación de logging.
    Siempre úsalo en tus proyectos.

  • El MDC (Mapped Diagnostic Context) permite añadir contexto a los
    mensajes (requestId, usuario, etc.).

  • El logging asíncrono mejora drásticamente el rendimiento usando
    LMAX Disruptor.

  • La seguridad es crítica: usa Log4j 2.17+ y deshabilita JNDI.

  • Las mejores prácticas incluyen: mensajes descriptivos, parámetros
    en vez de concatenación, contexto suficiente, y limpieza de MDC
    en finally.

================================================================================
  PREGUNTAS FRECUENTES (FAQ)
================================================================================

  ❓ ¿SLF4J o Log4j 2 directamente?
  R: Usa SLF4J siempre. Es la fachada estándar y te permite cambiar de
     implementación sin modificar código. Log4j 2 es la implementación.

  ❓ ¿Cómo evito que los logs de librerías llenen mi consola?
  R: Configura el nivel de los loggers de esas librerías a WARN o ERROR.
     log4j.logger.org.springframework=WARN

  ❓ ¿Cuál es la diferencia entre log4j-api y log4j-core?
  R: log4j-api contiene las interfaces/API. log4j-core contiene la
     implementación. Solo necesitas log4j-core (incluye la API).

  ❓ ¿Cómo hago para que un archivo de log no crezca infinitamente?
  R: Usa RollingFile Appender con políticas de rotación por tamaño
     y tiempo. Configura DefaultRolloverStrategy con un max.

  ❓ ¿Puedo cambiar la configuración de logging sin reiniciar la app?
  R: Sí, configurando monitorInterval en el XML. Log4j recargará la
     configuración cada N segundos.

  ❓ ¿Cómo logueo en diferentes archivos según el nivel?
  R: Configura múltiples RollingFile Appenders con ThresholdFilter:
     uno para INFO, otro para ERROR.

  ❓ ¿Qué hago si el logging está ralentizando mi aplicación?
  R: Activa el logging asíncrono con LMAX Disruptor. Reduce el nivel
     de log en producción. Desactiva includeLocation.
