-------------------------------------------------------------------------------
  73.1. FUNDAMENTOS DE DEPENDENCIAS
-------------------------------------------------------------------------------

Una dependencia es una librería externa que tu proyecto necesita para
compilar, ejecutar o testear. En el mundo Java, las dependencias se
distribuyen como JARs (Java ARchives) que contienen clases compiladas,
recursos y metadatos.

  TIPOS DE DEPENDENCIAS:

  • Directas: Las que declaras explícitamente en tu POM
  • Transitivas: Las que necesitan tus dependencias directas
  • Opcionales: Dependencias que no se heredan transitivamente
  • Provided: Las que proporciona el contenedor en tiempo de ejecución
  • Test: Solo disponibles durante la fase de pruebas

  EL PROBLEMA DE LA "DIAMANTE":

       A
      / \
     B   C
      \ /
       D

  Si A depende de B y C, y ambos dependen de D pero en versiones
  diferentes, ¿qué versión de D usa A?

  Este es el problema clásico de resolución de dependencias, y Maven
  lo resuelve con reglas bien definidas.

-------------------------------------------------------------------------------
  73.2. EL MECANISMO DE RESOLUCIÓN DE MAVEN
-------------------------------------------------------------------------------

Maven utiliza un proceso de resolución basado en:

  73.2.1. REPOSITORIOS

  Orden de búsqueda de dependencias:
  1. Repositorio local (~/.m2/repository)
  2. Repositorio central (https://repo.maven.apache.org/maven2)
  3. Repositorios remotos configurados en el POM

  73.2.2. COORDENADAS GAV

  Cada dependencia se identifica por tres coordenadas:
  - groupId: Organización/grupo (com.google.guava)
  - artifactId: Nombre del artefacto (guava)
  - version: Versión específica (33.0.0-jre)
  
  Opcionalmente:
  - packaging: Tipo de empaquetado (jar, war, pom, test-jar)
  - classifier: Clasificador (sources, javadoc, tests)

  73.2.3. DEPENDENCY MEDIATION (MEDIADOR DE DEPENDENCIAS)

  Reglas que Maven usa para resolver conflictos de versiones:

  REGLA 1: RUTA MÁS CORTA
  Cuando una dependencia aparece en múltiples lugares del árbol
  con diferentes versiones, gana la que tenga la ruta más corta
  desde la raíz.

  Ejemplo:
    A → B → C → D v1.0  (profundidad 3)
    A → E → D v2.0       (profundidad 2)
    Resultado: D v2.0 (ruta más corta)

  REGLA 2: PRIMERA DECLARACIÓN
  Si dos dependencias están a la misma profundidad, gana la que
  se declare primero en el POM.

  Ejemplo:
    A → B → D v1.0  (profundidad 2, declarado 1ro)
    A → C → D v2.0  (profundidad 2, declarado 2do)
    Resultado: D v1.0 (primera declaración)

  REGLA 3: EXCLUSIÓN EXPLÍCITA
  Puedes forzar la exclusión de una dependencia transitiva
  independientemente de las reglas anteriores.

```java
// Programa para analizar dependencias programáticamente
// usando la API de Aether (Eclipse Aether)

package com.lucasega.deps;

import org.eclipse.aether.RepositorySystem;
import org.eclipse.aether.RepositorySystemSession;
import org.eclipse.aether.artifact.DefaultArtifact;
import org.eclipse.aether.collection.CollectRequest;
import org.eclipse.aether.collection.DependencyCollectionException;
import org.eclipse.aether.graph.Dependency;
import org.eclipse.aether.graph.DependencyNode;
import org.eclipse.aether.repository.RemoteRepository;
import org.eclipse.aether.resolution.DependencyRequest;
import org.eclipse.aether.resolution.DependencyResolutionException;
import org.eclipse.aether.resolution.DependencyResult;
import org.eclipse.aether.util.graph.visitor.PreorderNodeListGenerator;

import java.util.Arrays;
import java.util.List;

public class AnalizadorDependencias {

    public static void main(String[] args) {
        try {
            // Configurar sistema de repositorios
            RepositorySystem system = Booter.newRepositorySystem();
            RepositorySystemSession session = Booter.newRepositorySystemSession(system);

            // Repositorio central de Maven
            RemoteRepository central = new RemoteRepository.Builder(
                "central", "default", "https://repo.maven.apache.org/maven2"
            ).build();

            // Artefacto a analizar
            DefaultArtifact artifact = new DefaultArtifact(
                "com.google.guava:guava:33.0.0-jre"
            );

            // Solicitud de recolección de dependencias
            CollectRequest collectRequest = new CollectRequest();
            collectRequest.setRoot(new Dependency(artifact, "compile"));
            collectRequest.setRepositories(List.of(central));

            // Recolectar árbol de dependencias
            DependencyNode node = system.collectDependencies(
                session, collectRequest
            ).getRoot();

            // Resolver dependencias
            DependencyRequest dependencyRequest = new DependencyRequest(
                node, null
            );
            DependencyResult result = system.resolveDependencies(
                session, dependencyRequest
            );

            // Mostrar árbol
            PreorderNodeListGenerator nlg = new PreorderNodeListGenerator();
            node.accept(nlg);
            System.out.println("=== ÁRBOL DE DEPENDENCIAS ===");
            System.out.println(nlg);

            // Estadísticas
            System.out.println("=== ESTADÍSTICAS ===");
            System.out.println("Total nodos: " + nlg.getNodes().size());
            System.out.println("Clases: " + nlg.getClassCount());

        } catch (DependencyCollectionException | DependencyResolutionException e) {
            System.err.println("Error analizando dependencias: " + e.getMessage());
            e.printStackTrace();
        }
    }
}
```

-------------------------------------------------------------------------------
  73.4. EXCLUSIÓN DE DEPENDENCIAS TRANSITIVAS
-------------------------------------------------------------------------------

Cuando una dependencia trae consigo otras que no necesitas o que causan
conflictos, puedes excluirlas explícitamente.

  73.4.1. EXCLUSIÓN EN EL POM

```xml
<!-- Excluir dependencia transitiva no deseada -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
    <version>3.2.2</version>
    <exclusions>
        <!-- Excluir Tomcat (si vamos a usar Jetty) -->
        <exclusion>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-starter-tomcat</artifactId>
        </exclusion>
        <!-- Excluir logging Logback (si vamos a usar Log4j2) -->
        <exclusion>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-starter-logging</artifactId>
        </exclusion>
    </exclusions>
</dependency>

<!-- Agregar alternativa -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-jetty</artifactId>
    <version>3.2.2</version>
</dependency>

<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-log4j2</artifactId>
    <version>3.2.2</version>
</dependency>
```

  73.4.2. EXCLUSIÓN GLOBAL EN dependencyManagement

```xml
<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>com.google.guava</groupId>
            <artifactId>guava</artifactId>
            <version>33.0.0-jre</version>
        </dependency>
    </dependencies>
</dependencyManagement>
```

  73.4.3. EXCLUSIÓN POR REGLA EN EL POM

```xml
<build>
    <plugins>
        <plugin>
            <groupId>org.apache.maven.plugins</groupId>
            <artifactId>maven-enforcer-plugin</artifactId>
            <version>3.4.1</version>
            <executions>
                <execution>
                    <id>enforce-dependency-convergence</id>
                    <goals><goal>enforce</goal></goals>
                    <configuration>
                        <rules>
                            <dependencyConvergence/>
                            <bannedDependencies>
                                <excludes>
                                    <exclude>commons-logging:commons-logging</exclude>
                                    <exclude>log4j:log4j</exclude>
                                </excludes>
                            </bannedDependencies>
                        </rules>
                    </configuration>
                </execution>
            </executions>
        </plugin>
    </plugins>
</build>
```

  73.5. BOM (BILL OF MATERIALS)
-------------------------------------------------------------------------------

Un BOM es un POM especial que centraliza las versiones de un conjunto de
dependencias relacionadas. Es la forma más elegante de gestionar versiones
en proyectos grandes.

  73.5.1. BOM DE SPRING BOOT

```xml
<!-- Usando el BOM de Spring Boot -->
<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-dependencies</artifactId>
            <version>3.2.2</version>
            <type>pom</type>
            <scope>import</scope>
        </dependency>
    </dependencies>
</dependencyManagement>

<dependencies>
    <!-- Las versiones las gestiona el BOM -->
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-web</artifactId>
    </dependency>
    <dependency>
        <groupId>com.fasterxml.jackson.core</groupId>
        <artifactId>jackson-databind</artifactId>
    </dependency>
</dependencies>
```

  73.6. VERSIONADO SEMÁNTICO (SEMVER)
-------------------------------------------------------------------------------

El versionado semántico es un estándar para asignar y entender versiones.
Formato: MAJOR.MINOR.PATCH

  MAJOR: Cambios incompatibles en la API
  MINOR: Nuevas funcionalidades compatibles hacia atrás
  PATCH: Correcciones de bugs compatibles hacia atrás

  EJEMPLOS:
  1.0.0  → Versión inicial
  1.1.0  → Nuevas funcionalidades (compatible con 1.x)
  1.1.1  → Corrección de bug (compatible con 1.1.x)
  2.0.0  → Cambio radical en la API (incompatible con 1.x)

  VERSIONES ESPECIALES:
  • SNAPSHOT: Versión en desarrollo (1.0.0-SNAPSHOT)
  • RC: Release Candidate (2.0.0-RC1)
  • M: Milestone (3.0.0-M1)
  • ALPHA/BETA: Versiones pre-lanzamiento (4.0.0-alpha.1)

  RANGOS DE VERSIONES (útil en conflictos):

  [1.0.0,2.0.0)  → Cualquier versión desde 1.0.0 hasta antes de 2.0.0
  [1.0.0]        → Solo 1.0.0
  (,1.0.0]       → Cualquier versión hasta 1.0.0 inclusive
  [1.0.0,)       → Cualquier versión desde 1.0.0 inclusive

  73.7. SEGURIDAD EN DEPENDENCIAS
-------------------------------------------------------------------------------

Las dependencias pueden contener vulnerabilidades de seguridad. Es crítico
mantenerlas actualizadas y analizarlas regularmente.

  73.7.1. OWASP DEPENDENCY CHECK

```xml
<!-- Plugin de seguridad OWASP -->
<plugin>
    <groupId>org.owasp</groupId>
    <artifactId>dependency-check-maven</artifactId>
    <version>9.0.9</version>
    <configuration>
        <failBuildOnCVSS>7</failBuildOnCVSS>  <!-- Fallar si severidad >= 7 -->
        <formats>
            <format>HTML</format>
            <format>JSON</format>
        </formats>
        <suppressionFiles>
            <suppressionFile>dependency-check-suppressions.xml</suppressionFile>
        </suppressionFiles>
    </configuration>
    <executions>
        <execution>
            <goals>
                <goal>check</goal>
            </goals>
        </execution>
    </executions>
</plugin>
```

```bash
# Ejecutar análisis de seguridad
mvn dependency-check:check

# Verificar vulnerabilidades conocidas
mvn dependency-check:aggregate
```

  73.7.2. SBOM (Software Bill of Materials)

```xml
<plugin>
    <groupId>org.cyclonedx</groupId>
    <artifactId>cyclonedx-maven-plugin</artifactId>
    <version>2.7.10</version>
    <executions>
        <execution>
            <phase>package</phase>
            <goals>
                <goal>makeAggregateBom</goal>
            </goals>
        </execution>
    </executions>

```

```bash
# Generar SBOM en formato CycloneDX
mvn cyclonedx:makeAggregateBom

# El SBOM se genera en target/ bom.json
# Contiene todas las dependencias con sus versiones
```

  73.7.3. MAVEN ENFORCER PLUGIN

```xml
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-enforcer-plugin</artifactId>
    <version>3.4.1</version>
    <executions>
        <execution>
            <id>enforce-rules</id>
            <goals><goal>enforce</goal></goals>
            <configuration>
                <rules>
                    <!-- Requerir Java 17+ -->
                    <requireJavaVersion>
                        <version>[17,)</version>
                    </requireJavaVersion>
                    <!-- Requerir Maven 3.8+ -->
                    <requireMavenVersion>
                        <version>[3.8,)</version>
                    </requireMavenVersion>
                    <!-- No permitir SNAPSHOT en releases -->
                    <requireReleaseDeps>
                        <message>Release no puede tener dependencias SNAPSHOT</message>
                        <onlyWhenRelease>true</onlyWhenRelease>
                    </requireReleaseDeps>
                    <!-- Convergencia de dependencias -->
                    <dependencyConvergence/>
                    <!-- Prohibir dependencias específicas -->
                    <bannedDependencies>
                        <excludes>
                            <exclude>commons-logging:commons-logging</exclude>
                            <exclude>log4j:log4j</exclude>
                            <exclude>javax.servlet:servlet-api</exclude>
                        </excludes>
                    </bannedDependencies>
                </rules>
            </configuration>
        </execution>
    </executions>
</plugin>
```

-------------------------------------------------------------------------------
  73.8. DEPENDENCIAS OPTACIONALES
-------------------------------------------------------------------------------

Una dependencia opcional significa que cuando el proyecto A depende de B,
y B tiene una dependencia opcional C, A no hereda C automáticamente.

```xml
<!-- En el POM de la librería B -->
<dependency>
    <groupId>com.google.code.gson</groupId>
    <artifactId>gson</artifactId>
    <version>2.10.1</version>
    <optional>true</optional>
</dependency>
```

  ¿CUÁNDO USAR OPTIONAL?
  • Funcionalidades que no todos los usuarios necesitan
  • Drivers de bases de datos (MySQL, PostgreSQL, Oracle)
  • Implementaciones alternativas (Logback vs Log4j)
  • Versiones específicas de plataforma

  73.9. DEPENDENCIAS PROVIDED Y RUNTIME
-------------------------------------------------------------------------------

  73.9.1. PROVIDED

  Dependencias necesarias para compilar pero que el entorno de ejecución
  (contenedor web, JDK, etc.) proporciona.

```xml
<!-- Servlet API (la proporciona Tomcat/Jetty en runtime) -->
<dependency>
    <groupId>jakarta.servlet</groupId>
    <artifactId>jakarta.servlet-api</artifactId>
    <version>6.0.0</version>
    <scope>provided</scope>
</dependency>

<!-- Lombok (solo necesario en compilación) -->
<dependency>
    <groupId>org.projectlombok</groupId>
    <artifactId>lombok</artifactId>
    <version>1.18.30</version>
    <scope>provided</scope>
</dependency>
```

  73.9.2. RUNTIME

  Dependencias no necesarias para compilar pero sí para ejecutar.

```xml
<!-- Driver JDBC (solo necesario en runtime) -->
<dependency>
    <groupId>com.mysql</groupId>
    <artifactId>mysql-connector-j</artifactId>
    <version>8.3.0</version>
    <scope>runtime</scope>
</dependency>

<!-- Implementación de logging -->
<dependency>
    <groupId>org.apache.logging.log4j</groupId>
    <artifactId>log4j-core</artifactId>
    <version>2.22.1</version>
    <scope>runtime</scope>
</dependency>
```

-------------------------------------------------------------------------------
  73.10. PROYECTOS MULTI-MÓDULO
-------------------------------------------------------------------------------

En proyectos grandes, las dependencias entre módulos deben gestionarse
cuidadosamente.

  ESTRUCTURA:

  proyecto-padre/
  ├── pom.xml                  (packaging=pom)
  ├── modulo-common/
  │   ├── pom.xml              (dependencia: ninguna)
  │   └── src/
  ├── modulo-datos/
  │   ├── pom.xml              (dependencia: common)
  │   └── src/
  ├── modulo-servicios/
  │   ├── pom.xml              (dependencia: common, datos)
  │   └── src/
  └── modulo-web/

-------------------------------------------------------------------------------
  73.11. DEPENDENCIAS SYSTEM (NO RECOMENDADO)
-------------------------------------------------------------------------------

El scope system permite referenciar JARs locales. Está DESACONSEJADO
en proyectos profesionales porque rompe la portabilidad.

```xml
<!-- NO RECOMENDADO: usar solo como último recurso -->
<dependency>
    <groupId>com.antiguo</groupId>
    <artifactId>libreria-vieja</artifactId>
    <version>1.0</version>
    <scope>system</scope>
    <systemPath>${project.basedir}/lib/libreria-vieja.jar</systemPath>
</dependency>
```

  Alternativa correcta: Instalar el JAR en el repositorio local:

```bash
mvn install:install-file \
    -Dfile=libreria-vieja.jar \
    -DgroupId=com.antiguo \
    -DartifactId=libreria-vieja \
    -Dversion=1.0 \
    -Dpackaging=jar
```

  73.12. PLUGIN DEPENDENCY: ANALYZE
-------------------------------------------------------------------------------

Una herramienta fundamental para mantener las dependencias limpias:

```bash
# Analizar dependencias no usadas
mvn dependency:analyze

# Salida:
[WARNING] Unused declared dependencies found:
[WARNING]    com.google.guava:guava:jar:33.0.0-jre:compile
[WARNING]    org.apache.commons:commons-lang3:jar:3.14.0:compile
[WARNING] 
[WARNING] Used undeclared dependencies found:
[WARNING]    com.fasterxml.jackson.core:jackson-databind:jar:2.15.3:compile
```

  Esto indica:
  - Dependencias declaradas que no se usan (elimínalas)
  - Dependencias usadas pero no declaradas (decláralas explícitamente)

```xml
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-dependency-plugin</artifactId>
    <version>3.6.1</version>
    <executions>
        <execution>
            <id>analyze-dependencies</id>
            <phase>verify</phase>
            <goals>
                <goal>analyze</goal>
                <goal>analyze-duplicate</goal>
            </goals>
            <configuration>
                <failOnWarning>true</failOnWarning>
                <ignoredUnusedDeclaredDependencies>
                    <ignoredUnusedDeclaredDependency>
                        org.projectlombok:lombok
                    </ignoredUnusedDeclaredDependency>
                </ignoredUnusedDeclaredDependencies>
            </configuration>
        </execution>
    </executions>
</plugin>
```

-------------------------------------------------------------------------------
  73.14. CASOS PRÁCTICOS DE CONFLICTOS
-------------------------------------------------------------------------------

  73.14.1. CONFLICTO DE JACKSON

```bash
# Árbol mostrando conflicto
mvn dependency:tree -Dincludes=com.fasterxml.jackson

[INFO] com.lucasega:app:jar:1.0.0
[INFO] +- org.springframework.boot:spring-boot-starter-web:3.2.2
[INFO] |  \- com.fasterxml.jackson.core:jackson-databind:2.15.3
[INFO] \- com.lucasega:modulo-externo:1.0.0
[INFO]    \- com.fasterxml.jackson.core:jackson-databind:2.14.2 (conflicto)
```

  Solución: Forzar la versión más reciente en dependencyManagement:

```xml
<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>com.fasterxml.jackson.core</groupId>
            <artifactId>jackson-databind</artifactId>
            <version>2.15.3</version>
        </dependency>
    </dependencies>
</dependencyManagement>
```

  73.14.2. CONFLICTO DE LOGGING

  Problema común: múltiples frameworks de logging (Logback, Log4j,
  java.util.logging) generan advertencias y problemas en runtime.

  Solución: Usar SLF4J como fachada y una única implementación.

```xml
<!-- Excluir Logback -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
    <exclusions>
        <exclusion>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-starter-logging</artifactId>
        </exclusion>
    </exclusions>
</dependency>

<!-- Agregar Log4j2 -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-log4j2</artifactId>
</dependency>
```

  73.14.3. CONFLICTO DE SERVLET API

  Problema: javax.servlet vs jakarta.servlet (cambio de nombre en Java EE 8
  a Jakarta EE 9+).

  Solución: Usar la versión correcta según el contenedor:

```xml
<!-- Para Tomcat 10+ (Jakarta EE) -->
<dependency>
    <groupId>jakarta.servlet</groupId>
    <artifactId>jakarta.servlet-api</artifactId>
    <version>6.0.0</version>
    <scope>provided</scope>
</dependency>

<!-- Para Tomcat 9- (Java EE) -->
<dependency>
    <groupId>javax.servlet</groupId>
    <artifactId>javax.servlet-api</artifactId>
    <version>4.0.1</version>
    <scope>provided</scope>
</dependency>
```

-------------------------------------------------------------------------------
  73.15. BUENAS PRÁCTICAS
-------------------------------------------------------------------------------

  1. Versiona las dependencias en properties
     Facilita actualizaciones masivas y mantiene consistencia.

  2. Usa BOMs para librerías relacionadas
     Spring Boot BOM, Jackson BOM, etc. garantizan compatibilidad.

  3. Excluye dependencias no utilizadas
     Usa mvn dependency:analyze periódicamente.

  4. No uses SNAPSHOT en releases
     Las versiones SNAPSHOT son inestables y pueden cambiar sin aviso.

  5. Mantén las dependencias actualizadas
     Usa Dependabot/Renovate o mvn versions:display-dependency-updates.

  6. Documenta las exclusiones
     Cuando excluyas una dependencia, explica por qué.

  7. Verifica la seguridad
     Usa OWASP Dependency Check en tu pipeline CI/CD.

  8. Minimiza las dependencias
     Cada dependencia es un riesgo de seguridad y mantenimiento.

  9. Prefiere librerías maduras y mantenidas
     Revisa actividad en GitHub, fecha de última versión, número de
     mantenedores.

  10. Usa dependency convergence
      El maven-enforcer-plugin asegura que no haya conflictos.

  RESUMEN DEL CAPÍTULO
-------------------------------------------------------------------------------

  • Las dependencias son librerías externas necesarias para construir
    y ejecutar un proyecto Java.

  • Maven resuelve dependencias transitivas usando las reglas del
    Mediador de Dependencias: ruta más corta y primera declaración.

  • El comando mvn dependency:tree muestra el árbol completo de
    dependencias y es la principal herramienta de diagnóstico.

  • Las exclusiones permiten eliminar dependencias transitivas no
    deseadas que causan conflictos.

  • Los BOMs (Bill of Materials) centralizan la gestión de versiones
    para familias de librerías relacionadas.

  • El versionado semántico (MAJOR.MINOR.PATCH) es el estándar para
    comunicar cambios en las librerías.

  • La seguridad en dependencias es crítica: OWASP Dependency Check
    y SBOMs son herramientas esenciales.

  • Los scopes (compile, provided, runtime, test, optional) controlan
    en qué fases del ciclo de vida está disponible cada dependencia.

  • Los proyectos multi-módulo requieren una gestión cuidadosa de
    dependencias entre módulos.

  • Las dependencias system no son recomendadas; prefiere instalar
    JARs en el repositorio local.

  • La herramienta dependency:analyze identifica dependencias no
    usadas y no declaradas.

  • Mantener las dependencias actualizadas es una tarea continua
    que debe automatizarse con Dependabot/Renovate.

================================================================================
  PREGUNTAS FRECUENTES (FAQ)
================================================================================

  ❓ ¿Qué hago si dos dependencias requieren versiones diferentes de la
     misma librería y no puedo actualizar ninguna?
  R: Usa exclusiones para eliminar la versión no deseada de una de las
     dependencias, y fuerza una versión compatible en dependencyManagement.

  ❓ ¿Cómo sé si una dependencia es segura?
  R: Usa OWASP Dependency Check. También revisa el repositorio de la
     librería (actividad, issues, mantenedores).

  ❓ ¿Qué significa una dependencia con classifier "sources"?
  R: Es el JAR con el código fuente. Útil para depuración. No se incluye
     en el artefacto final a menos que se especifique.

  ❓ ¿Cómo comparto una librería interna entre proyectos?
  R: Instálala en un repositorio privado (Nexus, Artifactory) y declárala
     como dependencia en los proyectos que la necesiten.

  ❓ ¿Qué es el "fat JAR" o "uber JAR"?
  R: Es un JAR que incluye todas sus dependencias. Se crea con
     maven-assembly-plugin o spring-boot-maven-plugin.

  ❓ ¿Debo incluir la versión en el artifactId?
  R: No. La versión va en el elemento <version>. El artifactId debe ser
     el nombre lógico del proyecto.

  ❓ ¿Qué pasa si una dependencia no está en ningún repositorio?
  R: Puedes alojarla en un repositorio privado, o como último recurso,
     usar el scope system con la ruta local.

  🏆 SISTEMA DE GESTIÓN DE DEPENDENCIAS CONSCIENTE

  Diseña un sistema en Java que analice proactivamente las dependencias
  de un proyecto Maven y tome decisiones inteligentes:

  1. Escáner de vulnerabilidades en tiempo real:
     - Consulta la API de NVD (National Vulnerability Database)
     - Correlaciona CVEs con las dependencias del proyecto
     - Sugiere versiones parcheadas automáticamente

  2. Optimizador de dependencias:
     - Detecta dependencias redundantes (misma librería, múltiples
       versiones)
     - Sugiere consolidación de versiones
     - Calcula el "peso" de cada dependencia en el JAR final

  3. Generador de informes de salud:
     - Score de salud para cada dependencia (0-100)
     - Factores: antigüedad, actividad, vulnerabilidades, popularidad
     - Tablero visual HTML con recomendaciones

  4. Sistema de políticas automatizadas:
     - Define reglas: "No permitir dependencias con score < 50"
     - Bloquea builds que violen las políticas
     - Genera notificaciones para el equipo

  💡 PISTA: Usa las APIs de Maven Resolver (Aether) para el análisis
  programático. Integra con la API REST de NVD para datos de
  vulnerabilidades.
