BIENVENIDO

Bienvenido al mundo de los genéricos. Si has usado ArrayList<String>,
LinkedList<Integer> o HashMap<String, Persona>, ya has usado
genéricos sin saberlo. Ahora es momento de entender cómo funcionan
internamente y, más importante aún, cómo crear tus propias clases
y métodos genéricos.

Los genéricos fueron introducidos en Java 5 (2004) y representan
uno de los cambios más importantes del lenguaje. Permiten escribir
código que funciona con cualquier tipo de objeto, proporcionando
seguridad en tiempo de compilación y eliminando la necesidad de
casting explícito.

FRASE DEL DÍA

"Los genéricos convierten los errores de tiempo de ejecución en
errores de compilación. Esa es su verdadera magia."
— Gilad Bracha

OBJETIVOS DEL CAPÍTULO

• Comprender el problema que resuelven los genéricos
• Crear clases genéricas con parámetros de tipo
• Escribir métodos genéricos reutilizables
• Entender el type erasure y sus implicaciones
• Conocer las convenciones de nomenclatura de tipos
• Aplicar bounded type parameters (tipos acotados)
• Distinguir entre tipos raw y parametrizados

================================================================================
EL PROBLEMA ANTES DE LOS GENÉRICOS

Antes de Java 5, las colecciones trabajaban con Object:

```java
// Java 1.4 y anteriores (SIN genéricos)
List lista = new ArrayList();
lista.add("Hola");
lista.add(Integer.valueOf(42));
lista.add(new Persona("Ana"));

// Para obtener un elemento, había que hacer casting
String texto = (String) lista.get(0);  // Casting explícito
Integer num = (Integer) lista.get(1);

// ¡Peligro! Esto compila pero falla en ejecución
String error = (String) lista.get(1);  // ClassCastException
```

Problemas:
  1. Sin seguridad de tipos: puedes mezclar tipos sin error
  2. Casting explícito necesario al obtener elementos
  3. El error aparece en ejecución, no en compilación

Con genéricos:

```java
// Java 5+ (CON genéricos)
List<String> lista = new ArrayList<>();
lista.add("Hola");
lista.add("Mundo");
// lista.add(42);  // ERROR DE COMPILACIÓN: tipo incorrecto

// Sin casting necesario
String texto = lista.get(0);  // El compilador sabe que es String
```

Los genéricos desplazan la detección de errores de ejecución a
compilación, donde es mucho más barato y seguro corregirlos.

================================================================================
CLASES GENÉRICAS

Una clase genérica se define con uno o más parámetros de tipo
entre ángulos (< >). Los parámetros de tipo se colocan después
del nombre de la clase.

```java
// Clase genérica con un parámetro de tipo T
public class Caja<T> {
    private T contenido;

    public void guardar(T contenido) {
        this.contenido = contenido;
    }

    public T obtener() {
        return contenido;
    }

    public boolean estaVacia() {
        return contenido == null;
    }
}
```

Uso de la clase genérica:

```java
Caja<String> cajaString = new Caja<>();
cajaString.guardar("Hola Mundo");
String valor = cajaString.obtener();  // Sin casting

Caja<Integer> cajaInteger = new Caja<>();
cajaInteger.guardar(42);
Integer numero = cajaInteger.obtener();

// Caja<String> y Caja<Integer> son el mismo .class en runtime
// pero el compilador los trata como tipos diferentes
```

Clase genérica con múltiples parámetros:

```java
public class Par<K, V> {
    private K clave;
    private V valor;

    public Par(K clave, V valor) {
        this.clave = clave;
        this.valor = valor;
    }

    public K getClave() { return clave; }
    public V getValor() { return valor; }

    public void setClave(K clave) { this.clave = clave; }
    public void setValor(V valor) { this.valor = valor; }
}

// Uso
Par<String, Integer> persona = new Par<>("Ana", 25);
String nombre = persona.getClave();
Integer edad = persona.getValor();
```

================================================================================
CONVENCIONES DE NOMENCLATURA

Los nombres de parámetros de tipo son convencionales, aunque
puedes usar cualquier nombre válido:

  • E — Element (usado en colecciones: List<E>, Set<E>)
  • K — Key (clave en mapas: Map<K, V>)
  • V — Value (valor en mapas: Map<K, V>)
  • T — Type (tipo general)
  • R — Return (tipo de retorno en métodos)
  • N — Number (para tipos numéricos)
  • S, U, V — Segundos, terceros, cuartos tipos

```java
public interface List<E> extends Collection<E> { ... }
public interface Map<K, V> { ... }
public interface Comparator<T> { ... }
```

MÉTODOS GENÉRICOS

Un método genérico declara sus propios parámetros de tipo,
independientemente de si la clase es genérica o no.

```java
public class Utilidades {

    // Método genérico: declara <T> antes del tipo de retorno
    public static <T> T obtenerUltimo(List<T> lista) {
        if (lista == null || lista.isEmpty()) {
            return null;
        }
        return lista.get(lista.size() - 1);
    }

    // Método genérico con dos parámetros de tipo
    public static <K, V> void imprimirPar(K clave, V valor) {
        System.out.println(clave + " → " + valor);
    }

    // Método genérico que convierte array a lista
    public static <T> List<T> arrayALista(T[] array) {
        List<T> lista = new ArrayList<>();
        for (T elemento : array) {
            lista.add(elemento);
        }
        return lista;
    }
}

// Uso
List<String> palabras = List.of("A", "B", "C");
String ultima = Utilidades.obtenerUltimo(palabras);  // "C"

Utilidades.imprimirPar("Nombre", "Ana");

String[] nombres = {"Ana", "Luis", "Carlos"};
List<String> listaNombres = Utilidades.arrayALista(nombres);
```

Inferencia de tipos: el compilador deduce el tipo automáticamente:

```java
// El compilador infiere que T es String
String ultima = Utilidades.obtenerUltimo(palabras);

// También puedes especificarlo explícitamente (raro)
String ultima2 = Utilidades.<String>obtenerUltimo(palabras);
```

BOUNDED TYPE PARAMETERS (TIPOS ACOTADOS)

Puedes restringir los tipos que puede tomar un parámetro usando
la palabra clave extends (NO implements, incluso para interfaces).

```java
// T solo puede ser Number o sus subclases (Integer, Double, etc.)
public class Calculadora<T extends Number> {
    private T valor1;
    private T valor2;

    public Calculadora(T valor1, T valor2) {
        this.valor1 = valor1;
        this.valor2 = valor2;
    }

    public double sumar() {
        return valor1.doubleValue() + valor2.doubleValue();
    }

    public double restar() {
        return valor1.doubleValue() - valor2.doubleValue();
    }
}

// Uso
Calculadora<Integer> intCalc = new Calculadora<>(10, 5);
System.out.println(intCalc.sumar());  // 15.0

Calculadora<Double> doubleCalc = new Calculadora<>(3.5, 2.1);
System.out.println(doubleCalc.restar());  // 1.4

// Calculadora<String> error = ...;  // ERROR: String no extiende Number
```

Múltiples bound (solo una clase, múltiples interfaces):

```java
// T debe extender Number E implementar Comparable
public class Maximo<T extends Number & Comparable<T>> {
    public T mayor(T a, T b) {
        return a.compareTo(b) > 0 ? a : b;
    }
}
```

Ejemplo útil: método genérico con bound

```java
public static <T extends Comparable<T>> T maximo(T a, T b) {
    return a.compareTo(b) > 0 ? a : b;
}

// Uso
System.out.println(maximo(5, 10));       // 10 (Integer)
System.out.println(maximo("Ana", "Luis"));  // "Luis" (String)
```

================================================================================
TYPE ERASURE (BORRADO DE TIPO)

El type erasure es el proceso por el cual el compilador elimina
la información de tipos genéricos durante la compilación. Esto
significa que los genéricos existen solo en tiempo de compilación;
en tiempo de ejecución, no hay información genérica.

¿Qué pasa durante la compilación?

```java
// Código fuente (con genéricos)
public class Caja<T> {
    private T contenido;
    public T obtener() { return contenido; }
}

// Después de type erasure (código compilado)
public class Caja {
    private Object contenido;  // T se borra a Object (o su bound)
    public Object obtener() { return contenido; }
}
```

El compilador también añade casts implícitos:

```java
// Código fuente
Caja<String> caja = new Caja<>();
caja.guardar("Hola");
String valor = caja.obtener();

// Después de type erasure + inserción de casts
Caja caja = new Caja();
caja.guardar("Hola");
String valor = (String) caja.obtener();  // Cast insertado por el compilador
```

Con bounded type parameters:

```java
// Código fuente
public class Calculadora<T extends Number> {
    private T valor;
    public double convertir() { return valor.doubleValue(); }
}

// Después de type erasure
public class Calculadora {
    private Number valor;  // T se borra a Number (el bound)
    public double convertir() { return valor.doubleValue(); }
}
```

Implicaciones del type erasure:
  1. No puedes usar instanceof con tipos genéricos
  2. No puedes crear arrays de tipos genéricos
  3. No puedes crear instancias de T (new T())
  4. Todas las clases genéricas comparten la misma clase runtime
  5. No puedes tener sobrecarga que solo difiera en el tipo genérico

```java
// INCORRECTO: no compila
if (lista instanceof ArrayList<String>) { ... }  // Error

// CORRECTO: usa raw type o wildcard
if (lista instanceof ArrayList<?>) { ... }

// INCORRECTO: no compila
T obj = new T();  // Error: type parameter cannot be instantiated

// INCORRECTO: sobrecarga ilegal
void procesar(List<String> lista) { ... }
void procesar(List<Integer> lista) { ... }  // Misma firma borrada: procesar(List)
```

================================================================================
TIPOS RAW

Un tipo raw es una clase genérica usada SIN parámetros de tipo.
Se permite solo por compatibilidad con código legacy.

```java
// Tipo raw (SIN parámetro de tipo)
ArrayList lista = new ArrayList();  // Tipo raw

// Tipo parametrizado
ArrayList<String> listaSegura = new ArrayList<>();
```

Problemas de los tipos raw:

```java
ArrayList<String> strings = new ArrayList<>();
ArrayList raw = strings;  // OK (asignación de parametrizado a raw)
raw.add(42);              // ¡Solo warning, pero compila!

// Peligro: ahora strings tiene un Integer
String s = strings.get(0);  // ClassCastException en runtime
```

No uses tipos raw. Siempre parametriza tus genéricos. Los tipos
raw existen solo por compatibilidad con código pre-Java 5.

EJEMPLO: REPOSITORIO GENÉRICO

```java
import java.util.*;

public class Repositorio<T> {
    private Map<String, T> almacen = new HashMap<>();

    public void guardar(String id, T elemento) {
        almacen.put(id, elemento);
    }

    public T obtener(String id) {
        return almacen.get(id);
    }

    public T eliminar(String id) {
        return almacen.remove(id);
    }

    public List<T> listarTodos() {
        return new ArrayList<>(almacen.values());
    }

    public boolean existe(String id) {
        return almacen.containsKey(id);
    }

    public int total() {
        return almacen.size();
    }
}

// Uso
public class RepositorioDemo {
    public static void main(String[] args) {
        Repositorio<Persona> repo = new Repositorio<>();
        repo.guardar("001", new Persona("Ana", 25));
        repo.guardar("002", new Persona("Luis", 30));

        Persona p = repo.obtener("001");
        System.out.println(p);

        for (Persona persona : repo.listarTodos()) {
            System.out.println(persona);
        }
    }
}
```

EJEMPLO: INTERFAZ GENÉRICA

```java
// Interfaz genérica
public interface Transformador<T, R> {
    R transformar(T entrada);
}

// Implementación concreta
public class StringToInteger implements Transformador<String, Integer> {
    @Override
    public Integer transformar(String entrada) {
        return Integer.parseInt(entrada);
    }
}

// Implementación genérica
public class Duplicador<T extends Number> implements Transformador<T, Double> {
    @Override
    public Double transformar(T entrada) {
        return entrada.doubleValue() * 2;
    }
}

// Clase que usa la interfaz genérica
public class Procesador {
    public static <T, R> List<R> transformarLista(
            List<T> lista, Transformador<T, R> transformador) {
        List<R> resultado = new ArrayList<>();
        for (T elemento : lista) {
            resultado.add(transformador.transformar(elemento));
        }
        return resultado;
    }

    public static void main(String[] args) {
        List<String> numeros = List.of("1", "2", "3", "4", "5");

        List<Integer> enteros = transformarLista(
            numeros, new StringToInteger());
        System.out.println("Enteros: " + enteros);

        List<Double> duplicados = transformarLista(
            enteros, new Duplicador<>());
        System.out.println("Duplicados: " + duplicados);
    }
}
```

EJEMPLO: PAREJA ORDENADA CON COMPARABLE

```java
public class ParOrdenado<T extends Comparable<T>> {
    private T primero;
    private T segundo;

    public ParOrdenado(T primero, T segundo) {
        this.primero = primero;
        this.segundo = segundo;
    }

    public T getMenor() {
        return primero.compareTo(segundo) < 0 ? primero : segundo;
    }

    public T getMayor() {
        return primero.compareTo(segundo) > 0 ? primero : segundo;
    }

    public boolean estaOrdenado() {
        return primero.compareTo(segundo) <= 0;
    }

    public static <T extends Comparable<T>> ParOrdenado<T> of(T a, T b) {
        return new ParOrdenado<>(a, b);
    }
}

// Uso
ParOrdenado<Integer> numeros = ParOrdenado.of(10, 5);
System.out.println("Menor: " + numeros.getMenor());    // 5
System.out.println("Mayor: " + numeros.getMayor());    // 10
System.out.println("¿Ordenado?: " + numeros.estaOrdenado());  // false

ParOrdenado<String> nombres = ParOrdenado.of("Ana", "Luis");
System.out.println("Menor: " + nombres.getMenor());  // "Ana" (alfabético)
```

GENÉRICOS Y ARRAYS

Los arrays genéricos tienen restricciones importantes:

```java
// INCORRECTO: no puedes crear arrays de tipos genéricos
List<String>[] arrayDeListas = new List<String>[10];  // Error

// TRABAJANDO: array de raw type (con warning)
List<String>[] array = new List[10];  // Warning unchecked

// CORRECTO: usar ArrayList de listas
List<List<String>> listaDeListas = new ArrayList<>();

// CORRECTO: el tipo específico funciona
String[] arrayString = new String[10];
```

¿Por qué no se permiten arrays genéricos?
Porque los arrays son "reified" (conocen su tipo en runtime),
mientras que los genéricos son "erased" (borrados en runtime).
Esta incompatibilidad fundamental hace que la combinación sea
problemática.

```java
// Esto sería posible si los arrays genéricos existieran:
Object[] array = new ArrayList<String>[1];
array[0] = new ArrayList<Integer>();  // Esto debería fallar...
// Pero por type erasure, ArrayList<String> y ArrayList<Integer>
// son iguales en runtime. El array no podría detectarlo.
```

================================================================================
ERRORES COMUNES Y BUENAS PRÁCTICAS

Error #1: Usar tipos raw

```java
// MAL
List lista = new ArrayList();
lista.add("Hola");

// BIEN
List<String> lista = new ArrayList<>();
lista.add("Hola");
```

Error #2: Asumir que los genéricos existen en runtime

```java
// MAL: instanceof con tipo parametrizado
if (obj instanceof ArrayList<String>) { ... }  // No compila

// BIEN: wildcard
if (obj instanceof ArrayList<?>) { ... }
```

Error #3: Crear instancias de T

```java
public class Fabrica<T> {
    // MAL
    public T crear() {
        return new T();  // Error de compilación
    }

    // BIEN: pasar el Class<T> como parámetro
    public T crear(Class<T> clase) throws Exception {
        return clase.getDeclaredConstructor().newInstance();
    }
}
```

Buena práctica: usar diamante vacío

```java
// Java 7+: el compilador infiere el tipo
List<String> lista = new ArrayList<>();  // <> en lugar de <String>
Map<String, List<Integer>> mapa = new HashMap<>();
```

Buena práctica: nombrar claramente los parámetros de tipo

```java
// Confuso
public class C<A, B> { ... }

// Claro
public class Mapa<K, V> { ... }
public class Repositorio<T> { ... }
```

RESUMEN DEL CAPÍTULO

• Los genéricos proporcionan seguridad de tipos en tiempo de
  compilación, eliminando la necesidad de casting explícito.

• Las clases genéricas se parametrizan con <T> (o cualquier letra
  convencional) y pueden usarse con cualquier tipo de objeto.

• Los métodos genéricos declaran su propio parámetro de tipo
  antes del tipo de retorno.

• Los bounded type parameters (T extends Number) restringen los
  tipos permitidos y permiten usar métodos del bound.

• El type erasure elimina la información genérica en tiempo de
  compilación: List<String> y List<Integer> son solo List en
  runtime.

• Los tipos raw existen por compatibilidad pero no deben usarse.

• Los arrays genéricos no están permitidos por la incompatibilidad
  entre reified arrays y erased genéricos.

• Las convenciones de nomenclatura son: E, K, V, T, R, N.

PREGUNTAS FRECUENTES

P: ¿Por qué no puedo hacer new T()?
R: Por type erasure. En runtime, T es Object (o el bound), y
   Java no sabe cómo construir una instancia de Object que
   sea realmente un String, Integer, etc. La solución es
   pasar Class<T> como parámetro.

P: ¿Cuál es la diferencia entre List<String> y List<? extends String>?
R: List<String> solo acepta String. List<? extends String> acepta
   String y cualquier subtipo (aunque String es final, así que
   en este caso son equivalentes). Veremos más en el capítulo de
   wildcards.

P: ¿Puedo usar tipos primitivos con genéricos?
R: No. Debes usar los wrappers: Integer, Double, Boolean, etc.
   El auto-boxing convierte automáticamente.

P: ¿Por qué List<int> no compila?
R: Los parámetros de tipo deben ser tipos referencia. Los tipos
   primitivos no son objetos. Usa List<Integer> en su lugar.

P: ¿Los genéricos afectan el rendimiento?
R: No significativamente. El type erasure elimina los parámetros
   de tipo, y los casts insertados por el compilador son casi
   gratuitos. La ventaja en seguridad y claridad supera
   cualquier costo mínimo.

DESAFÍO DEL ARQUITECTO

FRAMEWORK DE PROCESAMIENTO DE DATOS GENÉRICO

Implementa un framework de procesamiento de datos genérico que
pueda trabajar con cualquier tipo de dato. El framework debe
incluir:

  • Pipeline<T, R>: una cadena de procesamiento donde cada etapa
    transforma los datos

  • Stage<I, O>: una etapa individual del pipeline que transforma
    entrada I a salida O. Interfaz funcional.

  • Filter<T>: Stage que filtra elementos según un Predicate<T>

  • Mapper<T, R>: Stage que transforma cada elemento

  • Reducer<T, R>: Stage que reduce una lista de T a un solo R

  • Collector<T>: Stage que acumula elementos en una colección

  El Pipeline debe soportar:
    - thenApply(Stage): añadir etapa al pipeline
    - execute(List<T>): ejecutar todo el pipeline
    - El resultado es el output de la última etapa

  Ejemplo de uso:
    Pipeline<String, Integer> pipeline = Pipeline.of(new Mapper<>(s -> s.length()))
        .thenApply(new Filter<>(n -> n > 3))
        .thenApply(new Reducer<>(Integer::sum));

    List<String> datos = List.of("Hola", "a", "todos", "mundos");
    Integer resultado = pipeline.execute(datos);  // 5 + 5 = 10

Requisitos adicionales:
  • Type safety en todas las etapas (el compilador debe detectar
    incompatibilidades)
  • Soporte para pipelines paralelos (usando streams paralelos)
  • Estadísticas de cada etapa (tiempo de procesamiento,
    elementos procesados)

Extra:
  • Implementar una etapa Branch<I, O> que bifurque según una
    condición
  • Implementar Merge<I, O> que fusione dos pipelines
  • Crear un DSL para definir pipelines de forma declarativa

Este desafío integra todos los conceptos de genéricos: clases
genéricas, métodos genéricos, bounded types, y diseño de APIs
type-safe.
