  BIENVENIDO

  Bienvenido al Módulo IV de Java Master Book. Hasta ahora has dominado los
  pilares fundamentales de la Programación Orientada a Objetos: encapsulación,
  herencia, polimorfismo y abstracción. Sin embargo, Java impone una restricción
  importante: una clase solo puede extender una superclase. Esto se conoce como
  herencia simple y fue una decisión de diseño deliberada para evitar la
  complejidad y ambigüedad del problema del diamante.

  Pero, ¿qué sucede cuando necesitamos comportamientos de múltiples fuentes?
  Java respondió con las interfaces, que desde Java 8 incorporan métodos
  default y static. Este capítulo te enseñará cómo implementar múltiples
  interfaces, cómo Java resuelve los conflictos de herencia múltiple a través
  de las interfaces, y qué es exactamente el "problema del diamante" que los
  lenguajes como C++ enfrentan y Java evitó con elegancia.

  Prepárate para dominar uno de los temas más debatidos y sofisticados de la
  POO: la herencia múltiple controlada a través de interfaces.

  CONTENIDO

  --- 1. EL PROBLEMA DEL DIAMANTE ---

  El problema del diamante surge cuando una clase hija hereda de dos clases
  padre que, a su vez, heredan de una misma clase abuela. Se forma una
  estructura en forma de diamante:

        A
       / \
      B   C
       \ /
        D

  Si A tiene un método m() y tanto B como C lo sobrescriben de forma
  diferente, ¿cuál de las dos versiones debe usar D? Esta ambigüedad es el
  corazón del problema.

  En C++, este problema se resuelve con herencia virtual, pero introduce
  complejidades. Java optó por una solución más limpia: herencia simple de
  clases + herencia múltiple de interfaces.

  Las interfaces en Java originalmente solo declaraban métodos abstractos
  (sin implementación). Como ninguna interfaz aportaba código, no había
  conflicto. Pero desde Java 8, las interfaces pueden tener métodos default
  con implementación, lo que reintroduce la posibilidad de conflicto. Java
  resuelve esto con reglas claras que veremos más adelante.

--------------------------------------------------------------------------------

  --- 2. SINTÁXIS DE INTERFACES MÚLTIPLES ---

  Una clase puede implementar varias interfaces usando la palabra clave
  implements seguida de una lista separada por comas:

  ```java
  public interface Volador {
      void volar();
  }

  public interface Nadador {
      void nadar();
  }

  public class Pato implements Volador, Nadador {
      @Override
      public void volar() {
          System.out.println("El pato vuela batiendo las alas.");
      }

      @Override
      public void nadar() {
          System.out.println("El pato nada usando las patas palmeadas.");
      }
  }
  ```

--------------------------------------------------------------------------------

  --- 3. MÉTODOS DEFAULT EN INTERFACES ---

  A partir de Java 8, las interfaces pueden contener métodos con
  implementación usando la palabra clave default. Esto permite añadir
  nuevos métodos a interfaces existentes sin romper las clases que las
  implementan.

  ```java
  public interface Vehiculo {
      void acelerar();
      void frenar();

      default void encenderAlarma() {
          System.out.println("Alarma activada: BEEP BEEP BEEP");
      }

      default void apagarAlarma() {
          System.out.println("Alarma desactivada.");
      }
  }

  public class Coche implements Vehiculo {
      @Override
      public void acelerar() {
          System.out.println("Coche acelerando...");
      }

      @Override
      public void frenar() {
          System.out.println("Coche frenando...");
      }
  }

  public class PruebaVehiculo {
      public static void main(String[] args) {
          Coche coche = new Coche();
          coche.acelerar();
          coche.frenar();
          coche.encenderAlarma();
          coche.apagarAlarma();
      }
  }
  ```

  Los métodos default pueden sobrescribirse en la clase que implementa la
  interfaz si se desea un comportamiento diferente.

--------------------------------------------------------------------------------

  --- 4. MÉTODOS STATIC EN INTERFACES ---

  Además de métodos default, Java 8 introdujo métodos static en interfaces.
  Estos métodos pertenecen a la interfaz y no a las instancias.

  ```java
  public interface Matematica {
      static int sumar(int a, int b) {
          return a + b;
      }

      static int multiplicar(int a, int b) {
          return a * b;
      }
  }

  public class PruebaMatematica {
      public static void main(String[] args) {
          int suma = Matematica.sumar(10, 20);
          int producto = Matematica.multiplicar(5, 6);
          System.out.println("Suma: " + suma);
          System.out.println("Producto: " + producto);
      }
  }
  ```

--------------------------------------------------------------------------------

  --- 5. CONFLICTOS DE MÉTODOS DEFAULT ---

  Cuando una clase implementa dos interfaces que definen el mismo método
  default, debe resolver el conflicto sobrescribiendo el método.

  ```java
  public interface A {
      default void saludar() {
          System.out.println("Hola desde interfaz A");
      }
  }

  public interface B {
      default void saludar() {
          System.out.println("Hola desde interfaz B");
      }
  }

  public class MiClase implements A, B {
      // Debe sobrescribir saludar() para resolver el conflicto
      @Override
      public void saludar() {
          System.out.println("Hola desde MiClase");
          // O invocar a una de las interfaces:
          A.super.saludar();
          B.super.saludar();
      }
  }

  public class PruebaConflicto {
      public static void main(String[] args) {
          MiClase obj = new MiClase();
          obj.saludar();
      }
  }
  ```

  Reglas de resolución de conflictos en Java:
  1. Los métodos de clase tienen prioridad sobre los métodos default de
     interfaz.
  2. Si dos interfaces proporcionan métodos default con la misma firma, la
     clase debe sobrescribir el método.
  3. Se puede usar InterfaceName.super.metodo() para invocar a un método
     default específico.

  ```java
  // Ejemplo completo de resolución de conflictos

  public interface Imprimible {
      default void mostrar() {
          System.out.println("Mostrando documento desde Imprimible");
      }
  }

  public interface Escaneable {
      default void mostrar() {
          System.out.println("Mostrando documento desde Escaneable");
      }
  }

  public class ImpresoraMultifuncional implements Imprimible, Escaneable {
      @Override
      public void mostrar() {
          // Resolvemos el conflicto combinando ambos comportamientos
          Imprimible.super.mostrar();
          Escaneable.super.mostrar();
          System.out.println("--- Fin del proceso multifuncional ---");
      }

      public void imprimir() {
          System.out.println("Imprimiendo página...");
      }

      public void escanear() {
          System.out.println("Escaneando documento...");
      }
  }

  public class Oficina {
      public static void main(String[] args) {
          ImpresoraMultifuncional imp = new ImpresoraMultifuncional();
          imp.mostrar();
          imp.imprimir();
          imp.escanear();
      }
  }
  ```

--------------------------------------------------------------------------------

  --- 6. PRIORIDAD DE MÉTODOS (REGLA DE JAVA) ---

  Java sigue estas reglas en orden de precedencia para determinar qué método
  se ejecuta:

  1. Métodos sobrescritos en la clase concreta (máxima prioridad).
  2. Métodos heredados de superclases (tienen prioridad sobre defaults).
  3. Métodos default de interfaces (prioridad más baja).

  ```java
  // Demostración completa de prioridad

  public interface Animal {
      default void sonido() {
          System.out.println("Sonido genérico de animal");
      }
  }

  public class Mamifero {
      public void sonido() {
          System.out.println("Sonido genérico de mamífero");
      }
  }

  // La clase Perro hereda sonido() de Mamifero (clase), no de Animal
  public class Perro extends Mamifero implements Animal {
      // No es necesario sobrescribir: hereda de Mamifero
  }

  public class PruebaPrioridad {
      public static void main(String[] args) {
          Perro perro = new Perro();
          perro.sonido();  // Imprime: "Sonido genérico de mamífero"
      }
  }
  ```

  Si la clase hija quiere usar el método default de la interfaz en lugar del
  de la superclase, debe sobrescribirlo explícitamente:

  ```java
  public class Gato extends Mamifero implements Animal {
      @Override
      public void sonido() {
          Animal.super.sonido();  // Invoca al método default de la interfaz
      }
  }
  ```

--------------------------------------------------------------------------------

  --- 7. INTERFACES FUNCIONALES Y HERENCIA MÚLTIPLE ---

  Una interfaz funcional es aquella que tiene exactamente un método abstracto.
  Puede tener múltiples métodos default y static.

  ```java
  @FunctionalInterface
  public interface Operacion {
      int ejecutar(int a, int b);

      default void mostrarOperacion(String nombre) {
          System.out.println("Operación: " + nombre);
      }

      static int identidad(int x) {
          return x;
      }
  }

  public class Calculadora implements Operacion {
      @Override
      public int ejecutar(int a, int b) {
          return a + b;
      }
  }
  ```

  La anotación @FunctionalInterface es opcional pero recomendada: el
  compilador lanzará un error si la interfaz tiene más de un método abstracto.

--------------------------------------------------------------------------------

  --- 8. BUENAS PRÁCTICAS ---

  1. Prefiere interfaces a clases abstractas cuando el comportamiento pueda
     ser compartido por clases no relacionadas.

  2. Usa métodos default con moderación. Los defaults están pensados para
     evolucionar APIs, no para diseñar jerarquías de herencia.

  3. Documenta los métodos default con @implSpec para especificar el contrato
     que deben cumplir las implementaciones.

  4. Evita "interfaces de marcador" vacías. Prefiere anotaciones.

  5. Mantén las interfaces pequeñas y cohesivas (Principio de Segregación de
     Interfaces - ISP).

  6. Cuando diseñes una jerarquía, piensa primero en interfaces y luego en
     clases abstractas.

  7. No uses herencia múltiple de interfaces para compartir estado. Las
     interfaces no pueden tener campos mutables (solo constantes static final).

  8. Si dos interfaces tienen métodos default con la misma firma, revisa el
     diseño: tal vez una interfaz debería extender a la otra.

  --- 9. EJEMPLO INTEGRADOR ---

  ```java
  // Sistema de dispositivos inteligentes con herencia múltiple de interfaces

  interface Conectable {
      void conectarWiFi();

      default void verificarRed() {
          System.out.println("Verificando red disponible...");
      }
  }

  interface Controlable {
      void encender();
      void apagar();

      default boolean estaEncendido() {
          return true;  // Valor por defecto
      }
  }

  interface Programable {
      void programarTarea(String tarea, int hora);

      default void mostrarProgramacion() {
          System.out.println("Mostrando programación actual...");
      }
  }

  // Dispositivo inteligente que hereda comportamiento de múltiples interfaces
  class TermostatoInteligente implements Conectable, Controlable, Programable {
      private boolean encendido = false;
      private int temperatura = 22;
      private String[] tareas = new String[24];

      @Override
      public void conectarWiFi() {
          System.out.println("Termostato conectado a WiFi.");
      }

      @Override
      public void encender() {
          encendido = true;
          System.out.println("Termostato encendido.");
      }

      @Override
      public void apagar() {
          encendido = false;
          System.out.println("Termostato apagado.");
      }

      @Override
      public boolean estaEncendido() {
          return encendido;
      }

      @Override
      public void programarTarea(String tarea, int hora) {
          if (hora >= 0 && hora < 24) {
              tareas[hora] = tarea;
              System.out.println("Tarea '" + tarea + "' programada a las " + hora + ":00");
          }
      }

      @Override
      public void mostrarProgramacion() {
          System.out.println("=== Programación del Termostato ===");
          for (int i = 0; i < tareas.length; i++) {
              if (tareas[i] != null) {
                  System.out.printf("%02d:00 - %s%n", i, tareas[i]);
              }
          }
      }

      public void setTemperatura(int temp) {
          this.temperatura = temp;
          System.out.println("Temperatura ajustada a " + temp + "°C");
      }
  }

  // Otra interfaz con método default que podría entrar en conflicto
  interface Monitoreable {
      default void mostrarEstado() {
          System.out.println("Estado: OK");
      }
  }

  interface Diagnosticable {
      default void mostrarEstado() {
          System.out.println("Diagnóstico: Sin errores");
      }
  }

  // Resolución de conflicto
  class SensorInteligente implements Conectable, Monitoreable, Diagnosticable {
      private String id;

      public SensorInteligente(String id) {
          this.id = id;
      }

      @Override
      public void conectarWiFi() {
          System.out.println("Sensor " + id + " conectado a WiFi.");
      }

      @Override
      public void mostrarEstado() {
          System.out.println("=== Estado del Sensor " + id + " ===");
          Monitoreable.super.mostrarEstado();
          Diagnosticable.super.mostrarEstado();
          System.out.println("Nivel de batería: 85%");
      }
  }

  public class HogarInteligente {
      public static void main(String[] args) {
          System.out.println("=== INICIALIZANDO HOGAR INTELIGENTE ===\n");

          TermostatoInteligente termostato = new TermostatoInteligente();
          termostato.conectarWiFi();
          termostato.encender();
          termostato.setTemperatura(24);
          termostato.programarTarea("Calentar casa", 6);
          termostato.programarTarea("Reducir temperatura", 22);
          termostato.mostrarProgramacion();
          termostato.apagar();

          System.out.println();

          SensorInteligente sensor = new SensorInteligente("S-001");
          sensor.conectarWiFi();
          sensor.mostrarEstado();

          System.out.println("\n=== SISTEMA INICIALIZADO CORRECTAMENTE ===");
      }
  }
  ```

  Salida del programa:

  ```
  === INICIALIZANDO HOGAR INTELIGENTE ===

  Termostato conectado a WiFi.
  Termostato encendido.
  Temperatura ajustada a 24°C
  Tarea 'Calentar casa' programada a las 6:00
  Tarea 'Reducir temperatura' programada a las 22:00
  === Programación del Termostato ===
  06:00 - Calentar casa
  22:00 - Reducir temperatura
  Termostato apagado.

  Sensor S-001 conectado a WiFi.
  === Estado del Sensor S-001 ===
  Estado: OK
  Diagnóstico: Sin errores
  Nivel de batería: 85%

  === SISTEMA INICIALIZADO CORRECTAMENTE ===
  ```

  --- 10. HERENCIA MÚLTIPLE VS COMPOSICIÓN ---

  A menudo, la composición es preferible a la herencia múltiple. En lugar de
  heredar comportamiento, puedes delegar en objetos internos.

  ```java
  // Enfoque con composición (preferible en muchos casos)

  interface Volador {
      void volar();
  }

  interface Nadador {
      void nadar();
  }

  class Ala {
      void batir() {
          System.out.println("Alas batiendo...");
      }
  }

  class Aleta {
      void impulsar() {
          System.out.println("Aletas impulsando...");
      }
  }

  class AveAcuatica implements Volador, Nadador {
      private Ala ala = new Ala();        // Composición
      private Aleta aleta = new Aleta();  // Composición

      @Override
      public void volar() {
          ala.batir();
          System.out.println("¡Volando!");
      }

      @Override
      public void nadar() {
          aleta.impulsar();
          System.out.println("¡Nadando!");
      }
  }
  ```

  Ventajas de la composición sobre la herencia múltiple:
  - Mayor flexibilidad (puedes cambiar componentes en tiempo de ejecución).
  - Menor acoplamiento.
  - Más fácil de probar con mocks.
  - Evita jerarquías rígidas.

  PREGUNTAS FRECUENTES (FAQ)

  1. ¿Por qué Java no permite herencia múltiple de clases?
     Para evitar el problema del diamante y la complejidad que introduce
     en la resolución de métodos y en la gestión de la memoria.

  2. ¿Las interfaces pueden tener atributos?
     Sí, pero solo constantes (public static final). No pueden tener
     atributos de instancia porque las interfaces no tienen estado.

  3. ¿Puedo crear un objeto de una interfaz?
     No directamente. Las interfaces no se instancian. Debes crear un objeto
     de una clase que implemente la interfaz.

  4. ¿Qué ocurre si implemento dos interfaces con el mismo método default?
     La clase debe sobrescribir el método para resolver el conflicto,
     usando InterfaceName.super.metodo() si desea invocar a un default.

  5. ¿Los métodos default reemplazan a las clases abstractas?
     No. Las clases abstractas pueden tener estado (campos), constructores
     y métodos con cualquier visibilidad. Las interfaces no.

  6. ¿Puedo tener herencia múltiple de interfaces (interface extends interface)?
     Sí. Una interfaz puede extender múltiples interfaces:
     interface C extends A, B { }

  7. ¿Qué es @FunctionalInterface?
     Una anotación que indica que la interfaz tiene exactamente un método
     abstracto, permitiendo su uso con expresiones lambda.

  8. ¿Los métodos static en interfaces se heredan?
     No. Los métodos static en interfaces no se heredan a las clases que
     implementan la interfaz, a diferencia de los métodos static en clases.

  RESUMEN DEL CAPÍTULO

  En este capítulo has aprendido:

  • El problema del diamante es una ambigüedad que surge en la herencia
    múltiple cuando dos clases padre heredan de una misma clase abuela.

  • Java evitó este problema usando herencia simple de clases + herencia
    múltiple de interfaces.

  • Una clase puede implementar múltiples interfaces separadas por comas.

  • Los métodos default (Java 8+) permiten agregar implementación en
    interfaces para evolucionar APIs sin romper código existente.

  • Cuando dos interfaces tienen métodos default con la misma firma, la
    clase que las implementa debe sobrescribir el método.

  • La prioridad de métodos es: clase concreta > superclase > interfaz.

  • Las interfaces funcionales tienen un solo método abstracto y se
    anotan con @FunctionalInterface.

  • La composición es frecuentemente una mejor alternativa que la herencia
    múltiple.

  • Las interfaces no pueden tener estado mutable, solo constantes, métodos
    abstractos, default y static.

  - Crea una fachada ProcesadorPagos que centralice la lógica.
