--- 1. QUÉ ES UN MÉTODO EN GO ------------------------------------------------

En Go, un método es una función que tiene un parámetro especial llamado
receptor (receiver) que aparece entre la palabra clave func y el nombre del
método.

Sintaxis general:

    func (receptor Tipo) NombreMetodo(parametros) TipoRetorno {
        // cuerpo
    }

El receptor es como el "this" de Java o el "self" de Python, pero se pasa
de forma explícita y puedes llamarlo como quieras (por convención, una letra
o abreviatura del tipo).

Ejemplo mínimo:

--------------------------------------------------------------------------------
package main

import "fmt"

type Usuario struct {
    Nombre string
    Edad   int
}

// Método con receptor por valor
func (u Usuario) Saludar() {
    fmt.Printf("Hola, soy %s y tengo %d años.\n", u.Nombre, u.Edad)
}

func main() {
    user := Usuario{Nombre: "Ada", Edad: 29}
    user.Saludar() // Llamada al método
}
--------------------------------------------------------------------------------

Salida:
    Hola, soy Ada y tengo 29 años.

--- 2. DIFERENCIA ENTRE FUNCIÓN Y MÉTODO -------------------------------------

La diferencia clave es QUÉ invoca la lógica:

    • Función:  paquete.Nombre(args)
    • Método:   variable.Nombre(args)

El método está ligado a una variable concreta de un tipo. Eso permite:

  - Leer campos del receptor.
  - Mantener una sintaxis natural: usuario.Saludar().
  - Satisfacer interfaces (lo veremos en capítulo 33).

Mira esta comparación:

--------------------------------------------------------------------------------
package main

import "fmt"

type Punto struct {
    X, Y float64
}

// Función tradicional
func DistanciaOrigenFuncion(p Punto) float64 {
    return sqrt(p.X*p.X + p.Y*p.Y)
}

// Método sobre Punto
func (p Punto) DistanciaOrigen() float64 {
    return sqrt(p.X*p.X + p.Y*p.Y)
}

func sqrt(x float64) float64 { return x * 0.5 } // placeholder

func main() {
    p := Punto{3, 4}
    fmt.Println(DistanciaOrigenFuncion(p)) // función
    fmt.Println(p.DistanciaOrigen())       // método
}
--------------------------------------------------------------------------------

Ambos hacen lo mismo, pero el método expresa mejor la intención:
"pregúntale al punto su distancia al origen".

--- 3. RECEPTOR POR VALOR (VALUE RECEIVER) -----------------------------------

Cuando declaras:

    func (u Usuario) Saludar() { ... }

El receptor u es una COPIA del valor original. Cualquier modificación que
hagas a u dentro del método NO afecta a la variable original.

Esto es intencional: el receptor por valor garantiza inmutabilidad.

--------------------------------------------------------------------------------
package main

import "fmt"

type Contador struct {
    Valor int
}

func (c Contador) Incrementar() {
    c.Valor++ // modifica solo la copia
}

func (c Contador) ObtenerValor() int {
    return c.Valor
}

func main() {
    c := Contador{Valor: 10}
    c.Incrementar()
    fmt.Println(c.ObtenerValor()) // 10, no 11
}
--------------------------------------------------------------------------------

Como se ve, Incrementar no logra cambiar el original. Para eso necesitamos
un receptor por puntero (lo veremos en el capítulo 32).

¿Cuándo usar receptor por valor?

  • Cuando el método no necesita modificar el receptor.
  • Cuando el tipo es pequeño (int, string, structs de pocos campos).
  • Cuando quieres comportamiento de solo lectura.
  • Por claridad: el receptor no se muta.

--- 4. MÉTODOS SOBRE CUALQUIER TIPO ------------------------------------------

Una de las características más potentes de Go es que puedes definir métodos
sobre CUALQUIER tipo definido dentro de tu paquete, no solo structs.

Esto incluye:

  • Tipos alias: type Edad int
  • Tipos compuestos: type Lista []string
  • Tipos mapa: type Mapa map[string]int
  • Incluso tipos función: type Operacion func(int, int) int

Ejemplo con tipo alias:

--------------------------------------------------------------------------------
package main

import "fmt"

type Celsius float64
type Fahrenheit float64

func (c Celsius) AFahrenheit() Fahrenheit {
    return Fahrenheit(c*9/5 + 32)
}

func (f Fahrenheit) ACelsius() Celsius {
    return Celsius((f - 32) * 5 / 9)
}

func main() {
    var temp Celsius = 100
    fmt.Println(temp.AFahrenheit()) // 212

    var f Fahrenheit = 212
    fmt.Println(f.ACelsius()) // 100
}
--------------------------------------------------------------------------------

Ejemplo con tipo slice:

--------------------------------------------------------------------------------
package main

import "fmt"

type Nombres []string

func (n Nombres) SaludarTodos() {
    for _, nombre := range n {
        fmt.Printf("¡Hola, %s!\n", nombre)
    }
}

func main() {
    equipo := Nombres{"Ada", "Barbara", "Carlos"}
    equipo.SaludarTodos()
}
--------------------------------------------------------------------------------

Ejemplo con tipo mapa:

--------------------------------------------------------------------------------
package main

import "fmt"

type Inventario map[string]int

func (inv Inventario) Agregar(producto string, cantidad int) {
    inv[producto] += cantidad
}

func (inv Inventario) Total() int {
    total := 0
    for _, c := range inv {
        total += c
    }
    return total
}

func main() {
    stock := Inventario{"manzanas": 10, "peras": 5}
    stock.Agregar("manzanas", 3)
    fmt.Println("Total unidades:", stock.Total()) // 18
}
--------------------------------------------------------------------------------

Ejemplo con tipo función:

--------------------------------------------------------------------------------
package main

import "fmt"

type Transformador func(string) string

func (t Transformador) Aplicar(texto string) string {
    return t(texto)
}

func main() {
    mayus := Transformador(func(s string) string {
        return strings.ToUpper(s)
    })
    fmt.Println(mayus.Aplicar("hola")) // HOLA
}
--------------------------------------------------------------------------------

Esto último puede parecer extraño, pero abre la puerta a patrones muy
interesantes de decoración y middleware.

--- 5. METHOD SET (CONJUNTO DE MÉTODOS) --------------------------------------

El method set de un tipo T es el conjunto de todos los métodos declarados
con receptor T.

El method set de un tipo *T (puntero a T) incluye los métodos de T y los
de *T.

¿Por qué es importante? Porque para satisfacer una interfaz, un tipo debe
tener TODOS los métodos que la interfaz exige en su method set.

Ejemplo:

--------------------------------------------------------------------------------
package main

import "fmt"

type Animal struct {
    Nombre string
}

func (a Animal) Hablar() string {
    return "..." // método base
}

// El method set de Animal contiene: {Hablar}
// El method set de *Animal contiene: {Hablar} (lo hereda)

type Mascota interface {
    Hablar() string
}

func Presentar(m Mascota) {
    fmt.Println("La mascota dice:", m.Hablar())
}

type Perro struct {
    Animal
}

func (p Perro) Hablar() string {
    return "¡Guau!"
}

func main() {
    perro := Perro{}
    Presentar(perro) // ok
    // Presentar(&perro) // también ok
}
--------------------------------------------------------------------------------

Regla clave:

  • Las variables de tipo T solo tienen acceso a los métodos del method set de T.
  • Las variables de tipo *T tienen acceso a los métodos de T y *T.
  • Las interfaces se satisfacen según el method set completo.

Más reglas:

  1. Si un método tiene receptor por valor, tanto T como *T lo incluyen.
  2. Si un método tiene receptor por puntero, SOLO *T lo incluye en su
     method set (no T).

Esto lo veremos en detalle en el capítulo 32.

--- 6. MÉTODOS CON PARÁMETROS Y VALORES DE RETORNO ---------------------------

Los métodos pueden tener tantos parámetros y valores de retorno como
necesites, igual que las funciones:

--------------------------------------------------------------------------------
package main

import "fmt"

type Rectangulo struct {
    Ancho, Alto float64
}

func (r Rectangulo) Area() float64 {
    return r.Ancho * r.Alto
}

func (r Rectangulo) Escalar(factor float64) Rectangulo {
    return Rectangulo{Ancho: r.Ancho * factor, Alto: r.Alto * factor}
}

func (r Rectangulo) EsCuadrado() bool {
    return r.Ancho == r.Alto
}

func main() {
    r := Rectangulo{10, 5}
    fmt.Println("Area:", r.Area())
    fmt.Println("Es cuadrado?", r.EsCuadrado())

    r2 := r.Escalar(2)
    fmt.Println("Area escalada:", r2.Area())
}
--------------------------------------------------------------------------------

--- 7. MÉTODOS CON NOMBRES DE RECEPTOR ---------------------------------------

Por convención, el nombre del receptor debe ser de una o dos letras,
preferiblemente la inicial del tipo:

    type Usuario struct { ... }
    func (u Usuario) Metodo() {}

    type ClientePremium struct { ... }
    func (cp ClientePremium) Metodo() {}

No uses nombres como this, self, me. En Go se usa una abreviatura breve.

--- 8. MÉTODOS NO RECOMENDADOS -----------------------------------------------

Hay algunos patrones que debes evitar:

  a) Métodos sin usar el receptor:

    type Ejemplo struct {}
    func (e Ejemplo) Hola() { fmt.Println("hola") } // e no se usa

    Si no usas el receptor, considera una función.

  b) Nombres de métodos confusos:

    type Persona struct { Nombre string }
    func (p Persona) GetNombre() string { return p.Nombre }

    En Go, los getters no llevan "Get". Se llaman simplemente Nombre().

  c) Métodos que llaman a otros métodos del mismo receptor innecesariamente:

    Prefiere claridad, pero no crees acoplamientos raros.

--- 9. EJEMPLO COMPLETO: SISTEMA DE CUENTAS BANCARIAS ------------------------

Aquí tienes un ejemplo real que integra todo lo aprendido:

--------------------------------------------------------------------------------
package main

import (
    "fmt"
    "strings"
)

type Moneda string

func (m Moneda) Simbolo() string {
    switch m {
    case "USD":
        return "$"
    case "EUR":
        return "€"
    case "GBP":
        return "£"
    case "MXN":
        return "Mex$"
    default:
        return string(m)
    }
}

type CuentaBancaria struct {
    Titular string
    Saldo   float64
    Moneda  Moneda
}

func (c CuentaBancaria) TitularMayusculas() string {
    return strings.ToUpper(c.Titular)
}

func (c CuentaBancaria) FormatearSaldo() string {
    return fmt.Sprintf("%s%.2f", c.Moneda.Simbolo(), c.Saldo)
}

func (c CuentaBancaria) PuedeRetirar(monto float64) bool {
    return c.Saldo >= monto
}

func (c CuentaBancaria) Resumen() string {
    return fmt.Sprintf("Titular: %s | Saldo: %s",
        c.TitularMayusculas(), c.FormatearSaldo())
}

type Transaccion struct {
    Descripcion string
    Monto       float64
}

type Historial []Transaccion

func (h Historial) Agregar(desc string, monto float64) Historial {
    return append(h, Transaccion{desc, monto})
}

func (h Historial) Total() float64 {
    total := 0.0
    for _, t := range h {
        total += t.Monto
    }
    return total
}

func (h Historial) Resumen() string {
    lineas := make([]string, len(h))
    for i, t := range h {
        lineas[i] = fmt.Sprintf("  %d. %s: $%.2f", i+1, t.Descripcion, t.Monto)
    }
    return strings.Join(lineas, "\n")
}

func main() {
    usd := Moneda("USD")
    cuenta := CuentaBancaria{
        Titular: "Lucas EGA",
        Saldo:   15000.50,
        Moneda:  usd,
    }

    hist := Historial{
        {Descripcion: "Depósito inicial", Monto: 10000},
    }
    hist = hist.Agregar("Venta de curso", 5000)
    hist = hist.Agregar("Suscripción monthly", -500.50)

    fmt.Println(cuenta.Resumen())
    fmt.Println("Historial de transacciones:")
    fmt.Println(hist.Resumen())
    fmt.Printf("Balance del historial: $%.2f\n", hist.Total())
    fmt.Println("¿Puede retirar $20000?", cuenta.PuedeRetirar(20000))
}
--------------------------------------------------------------------------------

--- 10. DEBUGGING CON MÉTODOS ------------------------------------------------

Un patrón útil es tener un método Debug que imprima el estado interno:

--------------------------------------------------------------------------------
package main

import "fmt"

type Servidor struct {
    Host    string
    Puerto  int
    Activo  bool
    Conexiones int
}

func (s Servidor) Debug() string {
    estado := "inactivo"
    if s.Activo {
        estado = "activo"
    }
    return fmt.Sprintf(
        "Servidor{host:%s, puerto:%d, estado:%s, conexiones:%d}",
        s.Host, s.Puerto, estado, s.Conexiones,
    )
}

func main() {
    s := Servidor{Host: "localhost", Puerto: 8080, Activo: true, Conexiones: 42}
    fmt.Println(s.Debug())
}

--- 11. MÉTODOS PRIVADOS Y EXPORTADOS ----------------------------------------

Al igual que las funciones, los métodos pueden ser exportados (mayúscula)
o no exportados (minúscula):

    type Usuario struct {
        nombre string // campo privado
    }

    func (u Usuario) nombreCompleto() string { ... } // privado
    func (u Usuario) NombreCompleto() string { ... } // exportado

Esto te permite controlar la API pública de tus tipos.

--- 12. MÉTODOS EN VARIABLES ANÓNIMAS (LITERALES) ----------------------------

Puedes llamar métodos directamente sobre literales:

--------------------------------------------------------------------------------
package main

import "fmt"

type Punto struct{ X, Y int }

func (p Punto) Mover(dx, dy int) Punto {
    return Punto{X: p.X + dx, Y: p.Y + dy}
}

func main() {
    p := Punto{1, 2}.Mover(3, 4)
    fmt.Println(p) // {4, 6}
}
--------------------------------------------------------------------------------

--- 13. MÉTODOS VS FUNCIONES: ¿CUÁNDO USAR CADA UNO? ------------------------

Usa MÉTODO cuando:
  • La operación actúa sobre datos de un tipo específico.
  • Quieres sintaxis natural: objeto.operacion().
  • Necesitas satisfacer una interfaz.
  • Tiene sentido semánticamente: "el usuario se presenta".

Usa FUNCIÓN cuando:
  • La operación no pertenece naturalmente a un tipo.
  • Trabajas con valores primitivos sin alias.
  • Necesitas múltiples tipos de entrada y ninguno es "principal".
  • Es una utilidad genérica: math.Sin, sort.Slice.

--- 14. RANGOS Y MÉTODOS SOBRE SLICE HEADERS ---------------------------------

Cuando defines un método sobre un tipo slice, recuerda que el slice header
(capacidad, longitud, puntero al array subyacente) se copia, pero el array
subyacente NO. Por eso métodos como append() deben retornar el nuevo slice:

--------------------------------------------------------------------------------
package main

import "fmt"

type Pila struct {
    datos []int
}

func (p Pila) Push(valor int) Pila {
    return Pila{datos: append(p.datos, valor)}
}

func (p Pila) Pop() (Pila, int) {
    if len(p.datos) == 0 {
        return p, 0
    }
    return Pila{datos: p.datos[:len(p.datos)-1]}, p.datos[len(p.datos)-1]
}

func (p Pila) Vacia() bool {
    return len(p.datos) == 0
}

func main() {
    p := Pila{}
    p = p.Push(10)
    p = p.Push(20)
    p = p.Push(30)

    for !p.Vacia() {
        var v int
        p, v = p.Pop()
        fmt.Println("Sacado:", v)
    }
}

--- 15. COMPORTAMIENTO ESPECIAL CON STRING() ---------------------------------

Al igual que en Java toString(), en Go el método String() tiene un
significado especial (lo veremos en capítulo 37). Si defines:

    func (t Tipo) String() string { ... }

Entonces fmt.Println, fmt.Sprintf, etc. usarán tu representación.

--------------------------------------------------------------------------------
package main

import "fmt"

type Persona struct {
    Nombre string
    Edad   int
}

func (p Persona) String() string {
    return fmt.Sprintf("%s (%d años)", p.Nombre, p.Edad)
}

func main() {
    p := Persona{"Ana", 30}
    fmt.Println(p)          // Ana (30 años)
    fmt.Printf("%s\n", p)   // Ana (30 años)
    fmt.Printf("%v\n", p)   // Ana (30 años)
}
--------------------------------------------------------------------------------

--------------------------------------------------------------------------------
  RESUMEN DEL CAPÍTULO
--------------------------------------------------------------------------------

  • Un método es una función con un receptor explícito.
  • Sintaxis: func (t Tipo) Nombre() { }.
  • El receptor por valor (value receiver) COPIA el valor original.
  • Las modificaciones al receptor por valor no afectan al original.
  • Los métodos se pueden definir sobre CUALQUIER tipo del mismo paquete:
    struct, alias numérico, slice, mapa, incluso tipo función.
  • El method set de un tipo T incluye todos los métodos con receptor T.
  • El method set de *T incluye los métodos de T y de *T.
  • Los métodos exportados empiezan con mayúscula; los privados con minúscula.
  • Por convención, el receptor se nombra con 1 o 2 letras (inicial del tipo).
  • Value receiver es ideal para lecturas, inmutabilidad y tipos pequeños.

--------------------------------------------------------------------------------
  FAQ — PREGUNTAS FRECUENTES
--------------------------------------------------------------------------------

P: ¿Puedo definir métodos fuera del paquete del tipo?
R: No. Solo puedes definir métodos en el mismo paquete donde se declara el
   tipo. Esto evita la "monkey patching" y mantiene el código predecible.

P: ¿Qué pasa si no uso el receptor en el método?
R: Compila, pero es mala práctica. Si no necesitas el receptor, mejor usa
   una función normal.

P: ¿Los métodos sobreescriben (override)?
R: No. Go no tiene herencia ni override. Si un struct embebido tiene un
   método y el struct contenedor también, el método del contenedor
   "oculta" al del embebido, pero no hay polimorfismo de herencia.

P: ¿Puedo tener dos métodos con el mismo nombre pero diferente receptor?
R: No. Go no permite sobrecarga de métodos. El nombre debe ser único
   dentro del paquete (con algunas excepciones por tipo).

P: ¿Se pueden pasar métodos como argumentos?
R: No directamente, pero puedes pasar una función que llame al método, o
   usar method values (expresiones como obj.Metodo).

P: ¿Qué es más eficiente, value o pointer receiver?
R: Para tipos grandes (> 64 bytes), pointer receiver es más eficiente.
   Para tipos pequeños (int, bool, structs pequeños), value receiver es
   igual de eficiente y más predecible.

P: ¿Puedo llamar a un método sobre un nil?
R: Depende. Si el receptor es un puntero y el método está definido sobre
   *T, entonces SÍ puedes llamarlo incluso si el puntero es nil (pero
   debes manejarlo dentro del método). Si es value receiver, no hay nil
   posible porque es un valor concreto.

Has completado el capítulo 31. Ahora sabes que los métodos en Go son
funciones con un contexto explícito: el receptor.

Domina el value receiver: es la base para construir tipos inmutables,
predecibles y fáciles de razonar. En el próximo capítulo veremos el
receptor por puntero, que te permitirá mutar el estado interno y mejorar
el rendimiento en tipos grandes.

Cada método que escribas es un mensaje que tu tipo entiende. Diseña bien
esos mensajes.
