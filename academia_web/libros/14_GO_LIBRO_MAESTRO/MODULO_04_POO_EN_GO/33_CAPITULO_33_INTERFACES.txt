Bienvenido al capítulo 33. Aquí llegamos al corazón del polimorfismo en Go.

Si en Java o C++ las interfaces son contratos explícitos que una clase
declara implementar, en Go las interfaces son conjuntos de métodos que
un tipo satisface de forma implícita — sin necesidad de declararlo.
Esto se conoce como duck typing: si algo habla como pato y nada como
pato, entonces es un pato.

Este diseño permite un desacoplamiento extraordinario: puedes definir
interfaces después de los tipos, crear abstracciones sin jerarquías,
y escribir código genérico sin herencia.

Verás cómo io.Reader, io.Writer, fmt.Stringer y otras interfaces
de la stdlib son la columna vertebral del ecosistema Go.

¡Es hora de volar con interfaces!

--- 1. QUÉ ES UNA INTERFAZ EN GO ---------------------------------------------

Una interfaz en Go es un TIPO que define un CONJUNTO DE MÉTODOS.
Cualquier tipo que implemente todos esos métodos SATISFACE la interfaz
automáticamente (sin declaración explícita).

Definición:

    type Nombre interface {
        Metodo1(parametros) (retornos)
        Metodo2(parametros) (retornos)
    }

Ejemplo mínimo:

--------------------------------------------------------------------------------
package main

import "fmt"

// Definición de la interfaz
type Saludador interface {
    Saludar() string
}

// Tipo que implementa Saludador implícitamente
type Persona struct {
    Nombre string
}

func (p Persona) Saludar() string {
    return "¡Hola! Soy " + p.Nombre
}

// Otro tipo que también implementa Saludador
type Perro struct {
    Nombre string
}

func (p Perro) Saludar() string {
    return "¡Guau guau! Soy " + p.Nombre
}

// Función que acepta cualquier Saludador
func Presentar(s Saludador) {
    fmt.Println(s.Saludar())
}

func main() {
    p := Persona{Nombre: "Lucas"}
    perro := Perro{Nombre: "Rex"}

    Presentar(p)     // ¡Hola! Soy Lucas
    Presentar(perro) // ¡Guau guau! Soy Rex
}

--- 2. IMPLEMENTACIÓN IMPLÍCITA (DUCK TYPING) --------------------------------

Go no requiere la palabra clave implements ni una declaración explícita.
Si un tipo tiene los métodos que la interfaz requiere, automáticamente
la satisface.

Esto permite:

  • Definir interfaces DESPUÉS de los tipos (incluso en otro paquete).
  • Tipos de bibliotecas externas pueden satisfacer tus interfaces sin
    modificarlos.
  • Mayor desacoplamiento: los tipos no conocen las interfaces que
    satisfacen.

Ejemplo de interfaz definida después del tipo:

--------------------------------------------------------------------------------
package main

import "fmt"

// Primero definimos el tipo
type Rectangulo struct {
    Ancho, Alto float64
}

func (r Rectangulo) Area() float64 {
    return r.Ancho * r.Alto
}

func (r Rectangulo) Perimetro() float64 {
    return 2 * (r.Ancho + r.Alto)
}

// Después definimos la interfaz que Rectangulo satisface
type FiguraPlana interface {
    Area() float64
    Perimetro() float64
}

func MostrarFigura(f FiguraPlana) {
    fmt.Printf("Area: %.2f, Perímetro: %.2f\n", f.Area(), f.Perimetro())
}

func main() {
    r := Rectangulo{10, 5}
    MostrarFigura(r) // funciona aunque Rectangulo no sabe que existe FiguraPlana
}

--- 3. POLIMORFISMO CON INTERFACES -------------------------------------------

Una variable de tipo interfaz puede contener cualquier valor concreto
que satisfaga la interfaz. Esto es polimorfismo en acción.

--------------------------------------------------------------------------------
package main

import "fmt"

type Animal interface {
    Sonido() string
    Nombre() string
}

type Gato struct{ name string }
func (g Gato) Sonido() string { return "Miau" }
func (g Gato) Nombre() string { return g.name }

type Vaca struct{ name string }
func (v Vaca) Sonido() string { return "Muuu" }
func (v Vaca) Nombre() string { return v.name }

type Robot struct{ serial string }
func (r Robot) Sonido() string { return "Beep boop" }
func (r Robot) Nombre() string { return "Robot-" + r.serial }

func HacerSonido(a Animal) {
    fmt.Printf("%s dice: %s\n", a.Nombre(), a.Sonido())
}

func main() {
    animales := []Animal{
        Gato{"Misi"},
        Vaca{"Lola"},
        Robot{"X-100"},
    }

    for _, a := range animales {
        HacerSonido(a)
    }
}
--------------------------------------------------------------------------------

--- 4. INTERFACE VALUE: CONCRETE TYPE + VALUE ---------------------------------

Una variable de interfaz almacena un par: (tipo concreto, valor concreto).

    var a Animal = Gato{"Misi"}
    // a = (Gato, {name: "Misi"})

Esto permite que una misma variable de interfaz pueda contener diferentes
tipos en tiempo de ejecución.

Comprobación:

--------------------------------------------------------------------------------
package main

import (
    "fmt"
    "reflect"
)

type Animal interface {
    Sonido() string
}

type Gato struct{ name string }
func (g Gato) Sonido() string { return "Miau" }

type Perro struct{ name string }
func (p Perro) Sonido() string { return "Guau" }

func main() {
    var a Animal

    a = Gato{"Misi"}
    fmt.Printf("Tipo: %T, Valor: %v\n", a, a)

    a = Perro{"Rex"}
    fmt.Printf("Tipo: %T, Valor: %v\n", a, a)

    fmt.Println(reflect.TypeOf(a)) // main.Perro
}
--------------------------------------------------------------------------------

--- 5. io.Reader Y io.Writer -------------------------------------------------

Dos de las interfaces más importantes de la stdlib:

    type Reader interface {
        Read(p []byte) (n int, err error)
    }

    type Writer interface {
        Write(p []byte) (n int, err error)
    }

Estas interfaces permiten leer/escribir de forma uniforme desde:
archivos, redes, buffers, compresión, cifrado, etc.

Ejemplo con bytes.Buffer (que implementa ambas):

--------------------------------------------------------------------------------
package main

import (
    "bytes"
    "fmt"
    "io"
    "os"
)

func main() {
    var buf bytes.Buffer

    // bytes.Buffer implementa io.Writer
    buf.WriteString("Hola, ")
    buf.WriteString("Mundo!")

    // bytes.Buffer implementa io.Reader
    datos := make([]byte, buf.Len())
    n, err := buf.Read(datos)
    if err != nil && err != io.EOF {
        fmt.Println("Error:", err)
        return
    }
    fmt.Printf("Leídos %d bytes: %s\n", n, string(datos[:n]))

    // io.MultiWriter: escribe a múltiples writers
    archivo, _ := os.Create("/tmp/ejemplo.txt")
    defer archivo.Close()

    multi := io.MultiWriter(archivo, os.Stdout)
    multi.Write([]byte("Escribiendo a archivo y pantalla\n"))
}
--------------------------------------------------------------------------------

Ejemplo con io.Copy:

--------------------------------------------------------------------------------
package main

import (
    "bytes"
    "fmt"
    "io"
    "os"
    "strings"
)

func main() {
    // io.Copy copia de Reader a Writer
    r := strings.NewReader("Hola desde un Reader")
    // Copiar a stdout
    n, err := io.Copy(os.Stdout, r)
    if err != nil {
        fmt.Println("Error:", err)
        return
    }
    fmt.Printf("\nCopiados %d bytes\n", n)

    // Copiar a buffer
    var buf bytes.Buffer
    r2 := strings.NewReader("Datos para el buffer")
    io.Copy(&buf, r2)
    fmt.Println("Buffer contiene:", buf.String())
}
--------------------------------------------------------------------------------

--- 6. fmt.Stringer ----------------------------------------------------------

Una interfaz definida en el paquete fmt:

    type Stringer interface {
        String() string
    }

Cualquier tipo que implemente String() obtiene representación textual
automática con fmt.Println, %s, %v, etc.

--------------------------------------------------------------------------------
package main

import "fmt"

type Persona struct {
    Nombre string
    Edad   int
}

func (p Persona) String() string {
    return fmt.Sprintf("Persona(%s, %d años)", p.Nombre, p.Edad)
}

type Direccion struct {
    Calle string
    Numero int
}

func (d Direccion) String() string {
    return fmt.Sprintf("%s %d", d.Calle, d.Numero)
}

func main() {
    p := Persona{"Ana", 30}
    d := Direccion{"Av. Siempre Viva", 742}

    fmt.Println(p) // Persona(Ana, 30 años)
    fmt.Println(d) // Av. Siempre Viva 742
    fmt.Printf("%s vive en %s\n", p, d)
}
--------------------------------------------------------------------------------

--- 7. INTERFACES EN LA BIBLIOTECA ESTÁNDAR -----------------------------------

La stdlib de Go está llena de interfaces pequeñas y poderosas:

  • io.Reader / io.Writer — E/S universal
  • fmt.Stringer — Representación textual
  • error — type error interface { Error() string }
  • sort.Interface — type Interface interface { Len(); Less(i,j int) bool; Swap(i,j int) }
  • http.Handler — type Handler interface { ServeHTTP(ResponseWriter, *Request) }
  • json.Marshaler — type Marshaler interface { MarshalJSON() ([]byte, error) }
  • net.Listener — type Listener interface { Accept() (Conn, error); Close() error }

Principio: interfaces pequeñas (1-3 métodos) son más fáciles de
implementar y más reutilizables.

--- 8. INTERFAZ COMO CONTRATO MÍNIMO -----------------------------------------

Una interfaz debe definir lo mínimo necesario para que funcione.
No pidas más métodos de los que realmente necesitas.

Mal diseño:

    type Vehiculo interface {
        Conducir()
        Lavar()
        Reparar()
        Vender()
        ObtenerPrecio() float64
        ObtenerMarca() string
        ObtenerModelo() string
    }

Buen diseño (interfaces pequeñas):

    type Conductor interface {
        Conducir()
    }

    type Vendedor interface {
        Vender() float64
    }

--- 9. COMPOSICIÓN DE INTERFACES ---------------------------------------------

Puedes componer interfaces a partir de otras:

--------------------------------------------------------------------------------
package main

import (
    "fmt"
    "io"
    "os"
)

// Composición de interfaces
type ReadWriter interface {
    io.Reader
    io.Writer
}

// También puedes añadir tus propios métodos
type ReadWriteCloser interface {
    io.Reader
    io.Writer
    io.Closer
}

func main() {
    // os.File implementa io.Reader, io.Writer, io.Closer
    var rwc ReadWriteCloser
    archivo, _ := os.OpenFile("/tmp/test.txt", os.O_CREATE|os.O_RDWR, 0644)
    defer archivo.Close()

    rwc = archivo // ok, satisface las 3 interfaces

    rwc.Write([]byte("Hola composición"))
    fmt.Println("Escrito correctamente")
}
--------------------------------------------------------------------------------

--- 10. INTERFAZ COMO DEPENDENCIA ---------------------------------------------

Uno de los patrones más potentes: usar interfaces como parámetros en
lugar de tipos concretos. Esto permite inyectar dependencias.

--------------------------------------------------------------------------------
package main

import (
    "fmt"
    "strings"
)

type Almacenamiento interface {
    Guardar(datos string) error
}

type ArchivoAlmacen struct {
    ruta string
}

func (a *ArchivoAlmacen) Guardar(datos string) error {
    fmt.Printf("Guardando en archivo %s: %s\n", a.ruta, datos)
    return nil
}

type MemoriaAlmacen struct {
    datos []string
}

func (m *MemoriaAlmacen) Guardar(datos string) error {
    m.datos = append(m.datos, datos)
    fmt.Printf("Guardado en memoria: %s\n", datos)
    return nil
}

type Logger struct {
    almacen Almacenamiento // depende de la interfaz, no del concreto
}

func (l *Logger) Log(mensaje string) {
    l.almacen.Guardar(fmt.Sprintf("[LOG] %s", mensaje))
}

func main() {
    // Con archivo
    logger1 := Logger{almacen: &ArchivoAlmacen{ruta: "/tmp/log.txt"}}
    logger1.Log("Inicio del sistema")

    // Con memoria
    logger2 := Logger{almacen: &MemoriaAlmacen{}}
    logger2.Log("Evento en memoria")
}
--------------------------------------------------------------------------------

--- 11. SATISFACER INTERFACES DE OTROS PAQUETES ------------------------------

Puedes hacer que tus tipos satisfagan interfaces definidas en otros
paquetes sin modificarlos.

Ejemplo: hacer que un tipo satisfaga sort.Interface:

--------------------------------------------------------------------------------
package main

import (
    "fmt"
    "sort"
)

type Persona struct {
    Nombre string
    Edad   int
}

type PorEdad []Persona

func (p PorEdad) Len() int           { return len(p) }
func (p PorEdad) Less(i, j int) bool { return p[i].Edad < p[j].Edad }
func (p PorEdad) Swap(i, j int)      { p[i], p[j] = p[j], p[i] }

type PorNombre []Persona

func (p PorNombre) Len() int           { return len(p) }
func (p PorNombre) Less(i, j int) bool { return p[i].Nombre < p[j].Nombre }
func (p PorNombre) Swap(i, j int)      { p[i], p[j] = p[j], p[i] }

func main() {
    personas := []Persona{
        {"Carlos", 30},
        {"Ana", 25},
        {"Beto", 35},
    }

    fmt.Println("Original:", personas)

    sort.Sort(PorEdad(personas))
    fmt.Println("Por edad:", personas)

    sort.Sort(PorNombre(personas))
    fmt.Println("Por nombre:", personas)
}

--- 12. COMPARACIÓN CON INTERFACES -------------------------------------------

No puedes comparar interfaces con == a menos que el valor concreto
subyacente sea comparable. Si el tipo dinámico no es comparable
(como slice, map, o función), el programa paniquea.

--------------------------------------------------------------------------------
package main

import "fmt"

type Animal interface {
    Sonido() string
}

type Gato struct{ nombre string }
func (g Gato) Sonido() string { return "Miau" }

type Perro struct{ nombre string }
func (p Perro) Sonido() string { return "Guau" }

func main() {
    var a, b Animal

    a = Gato{"Misi"}
    b = Gato{"Misi"}
    fmt.Println(a == b) // true (structs comparables)

    a = Gato{"Misi"}
    b = Gato{"Otro"}
    fmt.Println(a == b) // false

    // a = &Gato{"Misi"}
    // b = &Gato{"Misi"}
    // fmt.Println(a == b) // false (punteros diferentes)
}
--------------------------------------------------------------------------------

--- 13. INTERFAZ NIL ----------------------------------------------------------

Una variable de tipo interfaz es nil solo cuando tanto su tipo concreto
como su valor son nil.

Esto es fuente de confusión:

--------------------------------------------------------------------------------
package main

import "fmt"

type Animal interface {
    Sonido() string
}

type Gato struct{}

func (g *Gato) Sonido() string { return "Miau" }

func ObtenerGato() Animal {
    var g *Gato = nil
    return g // retorna (Animal) con tipo *Gato y valor nil
}

func main() {
    var a Animal = nil
    fmt.Println(a == nil) // true

    a = ObtenerGato()
    fmt.Println(a == nil)       // false! porque el tipo dinámico es *Gato
    fmt.Printf("Tipo: %T\n", a) // *main.Gato
    fmt.Println(a.Sonido())     // Miau (funciona porque el método maneja nil)
}
--------------------------------------------------------------------------------

Para verificar si la interfaz es realmente nil, debes comprobar tanto
el tipo como el valor:

    if a == nil { ... } // solo funciona si ambos son nil

--- 14. INTERFACES CON PUNTERO VS VALOR ---------------------------------------

¿Cuándo un tipo *T satisface una interfaz pero T no?

Si la interfaz exige un método con receptor de puntero, SOLO *T satisface
la interfaz (no T).

--------------------------------------------------------------------------------
package main

import "fmt"

type Mutador interface {
    Mutar()
}

type Dato struct {
    Valor int
}

func (d *Dato) Mutar() {
    d.Valor = 100
}

func main() {
    var m Mutador

    // m = Dato{10} // ERROR: Dato no implementa Mutador
    m = &Dato{10} // OK: *Dato implementa Mutador

    m.Mutar()
    fmt.Println(m)
}
--------------------------------------------------------------------------------

--- 15. PATRÓN: INTERFACES COMO CONTRATO DE SERVICIO -------------------------

Este es un patrón arquitectónico común en Go:

1. Define una interfaz en el paquete que la necesita (cliente).
2. El paquete servidor la implementa sin conocer al cliente.
3. Esto elimina dependencias circulares.

--------------------------------------------------------------------------------
package main

import "fmt"

// Paquete "servicio"
type Notificador interface {
    Enviar(mensaje string) error
}

type EmailNotificador struct {
    Email string
}

func (e *EmailNotificador) Enviar(mensaje string) error {
    fmt.Printf("Enviando email a %s: %s\n", e.Email, mensaje)
    return nil
}

// Paquete "usuario"
type Usuario struct {
    Nombre      string
    Notificador Notificador
}

func (u *Usuario) Notificar(mensaje string) {
    u.Notificador.Enviar(fmt.Sprintf("Hola %s: %s", u.Nombre, mensaje))
}

func main() {
    user := Usuario{
        Nombre: "Lucas",
        Notificador: &EmailNotificador{Email: "lucas@example.com"},
    }
    user.Notificar("Bienvenido al sistema")
}

--- 16. EJEMPLO COMPLETO: SISTEMA DE PAGOS ------------------------------------

--------------------------------------------------------------------------------
package main

import (
    "fmt"
    "strings"
    "time"
)

type Pago interface {
    Pagar(monto float64) string
    ObtenerMetodo() string
}

type TarjetaCredito struct {
    Numero    string
    Titular   string
    Vencimiento string
}

func (t *TarjetaCredito) Pagar(monto float64) string {
    ultimos := t.Numero[len(t.Numero)-4:]
    return fmt.Sprintf("Pagado $%.2f con TC ****%s (titular: %s)", monto, ultimos, t.Titular)
}

func (t *TarjetaCredito) ObtenerMetodo() string {
    return "Tarjeta de Crédito"
}

type PayPal struct {
    Email string
}

func (p *PayPal) Pagar(monto float64) string {
    return fmt.Sprintf("Pagado $%.2f con PayPal (%s)", monto, p.Email)
}

func (p *PayPal) ObtenerMetodo() string {
    return "PayPal"
}

type Transferencia struct {
    Banco     string
    Cuenta    string
    Titular   string
}

func (t *Transferencia) Pagar(monto float64) string {
    return fmt.Sprintf("Pagado $%.2f por transferencia %s - cta: %s", monto, t.Banco, t.Cuenta)
}

func (t *Transferencia) ObtenerMetodo() string {
    return "Transferencia Bancaria"
}

type Factura struct {
    ID          string
    Concepto    string
    Monto       float64
    Fecha       time.Time
}

func (f *Factura) ProcesarPago(p Pago) {
    fmt.Println("===================================")
    fmt.Printf("Factura: %s\n", f.ID)
    fmt.Printf("Concepto: %s\n", f.Concepto)
    fmt.Printf("Monto: $%.2f\n", f.Monto)
    fmt.Printf("Método: %s\n", p.ObtenerMetodo())
    fmt.Println("---")
    fmt.Println(p.Pagar(f.Monto))
    fmt.Println("===================================")
}

type Caja struct {
    Ingresos float64
    Transacciones []string
}

func (c *Caja) Registrar(monto float64, descripcion string) {
    c.Ingresos += monto
    c.Transacciones = append(c.Transacciones, descripcion)
}

func (c *Caja) Resumen() string {
    var sb strings.Builder
    sb.WriteString(fmt.Sprintf("Total ingresos: $%.2f\n", c.Ingresos))
    sb.WriteString("Transacciones:\n")
    for _, t := range c.Transacciones {
        sb.WriteString(fmt.Sprintf("  - %s\n", t))
    }
    return sb.String()
}

func main() {
    caja := &Caja{}

    facturas := []*Factura{
        {ID: "F-001", Concepto: "Curso de Go", Monto: 99.99, Fecha: time.Now()},
        {ID: "F-002", Concepto: "Membresía Premium", Monto: 29.99, Fecha: time.Now()},
        {ID: "F-003", Concepto: "Consultoría", Monto: 500.00, Fecha: time.Now()},
    }

    metodos := []Pago{
        &TarjetaCredito{Numero: "1234567890123456", Titular: "Lucas EGA", Vencimiento: "12/28"},
        &PayPal{Email: "lucas@example.com"},
        &Transferencia{Banco: "Banco Nacional", Cuenta: "123-456-789", Titular: "Lucas EGA"},
    }

    for i, f := range facturas {
        f.ProcesarPago(metodos[i])
        caja.Registrar(f.Monto, fmt.Sprintf("%s - %s", f.ID, metodos[i].ObtenerMetodo()))
    }

    fmt.Println("\nResumen de Caja:")
    fmt.Println(caja.Resumen())
}

--- 17. VERIFICACIÓN EN TIEMPO DE COMPILACIÓN --------------------------------

Puedes verificar en tiempo de compilación que un tipo satisface una
interfaz:

    var _ Interfaz = Tipo{} // error si Tipo no implementa Interfaz

Esto es útil para asegurarte de que no rompes la implementación:

--------------------------------------------------------------------------------
package main

import "fmt"

type Stringer interface {
    String() string
}

type Punto struct {
    X, Y int
}

// Esta línea falla en compilación si Punto no implementa Stringer
var _ Stringer = Punto{}

func (p Punto) String() string {
    return fmt.Sprintf("(%d, %d)", p.X, p.Y)
}

func main() {
    p := Punto{3, 4}
    fmt.Println(p) // (3, 4)
}

--- 18. PRUEBAS CON INTERFACES -----------------------------------------------

Las interfaces facilitan el testing: puedes crear mocks que implementen
la interfaz para simular comportamientos.

--------------------------------------------------------------------------------
package main

import (
    "fmt"
    "strings"
)

type BaseDatos interface {
    ObtenerUsuario(id int) (string, bool)
}

type Servicio struct {
    db BaseDatos
}

func (s *Servicio) SaludarUsuario(id int) string {
    nombre, ok := s.db.ObtenerUsuario(id)
    if !ok {
        return "Usuario no encontrado"
    }
    return fmt.Sprintf("Hola, %s!", nombre)
}

// Mock para pruebas
type MockDB struct {
    datos map[int]string
}

func (m *MockDB) ObtenerUsuario(id int) (string, bool) {
    nombre, ok := m.datos[id]
    return nombre, ok
}

func testSaludarUsuario() {
    mock := &MockDB{datos: map[int]string{
        1: "Lucas",
        2: "Ana",
    }}
    svc := &Servicio{db: mock}

    tests := []struct {
        id       int
        esperado string
    }{
        {1, "Hola, Lucas!"},
        {2, "Hola, Ana!"},
        {99, "Usuario no encontrado"},
    }

    for _, t := range tests {
        resultado := svc.SaludarUsuario(t.id)
        estado := "PASÓ"
        if resultado != t.esperado {
            estado = "FALLÓ"
        }
        fmt.Printf("Test ID %d: %s (obtuvo: %q)\n", t.id, estado, resultado)
    }
}

func main() {
    testSaludarUsuario()
}

--- 19. BUENAS PRÁCTICAS CON INTERFACES ---------------------------------------

1. Interfaces pequeñas: 1 a 3 métodos. Prefiere interfaces de 1 método
   (ej: io.Reader).

2. Nombres con sufijo "-er": para interfaces de 1 método usa el sufijo
   -er: Reader, Writer, Stringer, Closer, Payer, Notifier.

3. Acepta interfaces, retorna structs: las funciones deberían aceptar
   interfaces (flexibilidad) y retornar tipos concretos (simplicidad).

4. No definas interfaces antes de usarlas: en Go, las interfaces se
   definen donde se necesitan, no donde se implementan.

5. Satisface interfaces de la stdlib: implementa io.Reader, fmt.Stringer,
   etc. cuando tenga sentido.

6. Usa composición de interfaces: en lugar de interfaces grandes,
   compón interfaces pequeñas.

--------------------------------------------------------------------------------
  RESUMEN DEL CAPÍTULO
--------------------------------------------------------------------------------

  • Una interfaz define un conjunto de métodos que un tipo debe implementar.
  • La implementación es IMPLÍCITA (duck typing) — no se declara.
  • Cualquier tipo que tenga todos los métodos requeridos satisface la interfaz.
  • Una variable de interfaz almacena (tipo concreto, valor concreto).
  • io.Reader y io.Writer son las interfaces más importantes de Go.
  • fmt.Stringer da representación textual automática con %s y %v.
  • Las interfaces permiten polimorfismo, inyección de dependencias y testing.
  • Interfaces pequeñas y enfocadas son mejor que interfaces grandes.
  • Composición de interfaces: type ReadWriter interface { Reader; Writer }.

--------------------------------------------------------------------------------
  FAQ — PREGUNTAS FRECUENTES
--------------------------------------------------------------------------------

P: ¿Por qué Go no usa implements explícito?
R: Para permitir implementación retroactiva: un tipo de una librería
   externa puede satisfacer tus interfaces sin modificarlo.

P: ¿Cómo sé si un tipo satisface una interfaz?
R: Si tiene todos los métodos declarados en la interfaz, la satisface.
   El compilador lo verifica automáticamente al asignar.

P: ¿Una interfaz puede contener campos de datos?
R: No. Las interfaces solo declaran métodos, no campos.

P: ¿Puedo definir una interfaz con métodos privados?
R: No. Todos los métodos de una interfaz son exportados (públicos) por
   definición. No existen métodos no exportados en interfaces.

P: ¿Cuál es la diferencia entre interfaz y abstract class?
R: Go no tiene clases abstractas. Las interfaces solo definen
   comportamiento (métodos), no estado (campos). Además, la
   implementación es implícita, no heredada.

P: ¿Se puede hacer casting de interfaz a tipo concreto?
R: Sí, mediante type assertion (capítulo 35): x.(TipoConcreto).

P: ¿Una interfaz puede embeder otra interfaz?
R: Sí. Composición de interfaces: type A interface { B; C; MetodoPropio() }.

P: ¿Qué es el principio de "accept interfaces, return structs"?
R: Significa que las funciones deben aceptar parámetros de tipo interfaz
   (flexibilidad) pero retornar tipos concretos (simplicidad y rendimiento).

Sistema de plugin de Procesamiento de Datos:

Diseña e implementa un sistema de plugins para procesar datos usando
interfaces. El sistema debe ser extensible sin modificar el código
principal.

  Interfaz base:
    type Procesador interface {
        Nombre() string
        Procesar(datos []byte) ([]byte, error)
    }

  Procesadores a implementar:
    1. CompresorGZip — comprime datos con gzip
    2. EncriptadorAES — encripta datos (simulado con ROT13)
    3. ValidadorJSON — verifica que los datos sean JSON válido
    4. TransformadorMayusculas — convierte a mayúsculas

  Sistema:
    type Pipeline struct {
        pasos []Procesador
    }

    Métodos de Pipeline:
      - Agregar(p Procesador) *Pipeline
      - Ejecutar(datos []byte) ([]byte, error)
      - ListarProcesadores() []string

  Requisitos:
    1. Los procesadores deben poder encadenarse en un pipeline.
    2. Cada procesador recibe la salida del anterior.
    3. Pipeline debe ser extensible: nuevos procesadores sin modificar Pipeline.
    4. Cada procesador debe tener un nombre único.
    5. Implementa un procesador Compuesto que ejecute varios procesadores internos.

  Extra: Crea un LoggerMiddleware que envuelva cualquier Procesador y loguee
  el tiempo de ejecución.

Este desafío integra interfaces, composición, pipeline pattern, y
diseño extensible.
