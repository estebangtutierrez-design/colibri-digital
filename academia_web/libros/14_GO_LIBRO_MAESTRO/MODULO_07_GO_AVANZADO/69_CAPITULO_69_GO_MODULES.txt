  BIENVENIDA
  ──────────
  Bienvenido al Capítulo 69. Go Modules es el sistema oficial de gestión
  de dependencias desde Go 1.11 (estable en 1.16). Reemplazó a GOPATH y
  a herramientas como dep, glide, y govendor. Un dominio sólido de go
  mod y go.mod es esencial para cualquier proyecto Go profesional.

  ── 1. INTRODUCCIÓN A GO MODULES ──

  Un módulo es una colección de paquetes Go con un archivo go.mod en su
  raíz. go.mod define:

  ● El module path (ruta única del módulo).
  ● La versión de Go requerida.
  ● Las dependencias con sus versiones.
  ● Directivas especiales (replace, exclude, retract).

  Ventajas sobre GOPATH:
  ● El proyecto puede estar en cualquier directorio.
  ● Cada proyecto tiene su propia versión de dependencias.
  ● Versionado semántico obligatorio.
  ● Reproducibilidad garantizada con go.sum.

  ── 2. go mod init ──

  Crea un nuevo módulo:

  mkdir mi-proyecto
  cd mi-proyecto
  go mod init github.com/usuario/mi-proyecto

  Resultado: archivo go.mod con:

  module github.com/usuario/mi-proyecto

  go 1.22

  El module path debe identificar el módulo de forma única.
  Convención: URL del repositorio (github.com/usuario/repo).

  Para proyectos que no se publicarán:
  go mod init mi-proyecto
  go mod init ejemplo.com/app

  ── 3. go mod tidy ──

  El comando más importante para mantener dependencias:

  go mod tidy

  Hace tres cosas:
  1. Agrega dependencias que están en el código pero no en go.mod.
  2. Elimina dependencias que están en go.mod pero no en el código.
  3. Actualiza go.sum con los checksums correctos.

  Deberías ejecutarlo:
  ● Después de cada cambio de dependencias.
  ● Antes de commit.
  ● En CI (para verificar que go.mod y go.sum están limpios).

  ── 4. go get ──

  Agrega o actualiza dependencias:

  # Agregar una dependencia
  go get github.com/gorilla/mux

  # Versión específica
  go get github.com/gorilla/mux@v1.8.1

  # Última versión disponible
  go get -u github.com/gorilla/mux

  # Actualizar todas las dependencias
  go get -u ./...

  # Actualizar a versiones patch (no minor, no major)
  go get -u=patch ./...

  # Eliminar una dependencia (y luego go mod tidy)
  go get github.com/old/dep@none

  ── 5. VERSIONADO SEMÁNTICO ──

  Go Modules usa Semantic Versioning 2.0.0:

  vMAJOR.MINOR.PATCH

  v1.8.1
  │ │ └─ Patch: bug fixes (compatible hacia atrás)
  │ └── Minor: new features (compatible hacia atrás)
  └──── Major: breaking changes (incompatible)

  Versiones especiales:
  ● v0.x.x: inestable, no hay promesa de compatibilidad.
  ● v1.x.x: primera versión estable (implícita, sin /v1 en la ruta).
  ● v2+: breaking changes (EXIGE /v2 en el module path).
  ● Pre-release: v1.0.0-alpha.1, v1.0.0-beta.2, v1.0.0-rc.1.
  ● Pseudo-versions: v0.0.0-20240601123456-abcdef123456.

  ── 6. SEMANTIC IMPORT VERSIONING (SIV) ──

  Para v2+, el module path debe incluir el major version:

  // go.mod v1 (no lleva /v1)
  module github.com/user/lib

  // go.mod v2
  module github.com/user/lib/v2

  // go.mod v3
  module github.com/user/lib/v3

  Esto permite importar múltiples versiones mayores:

  import (
      "github.com/user/lib"         // v1
      libv2 "github.com/user/lib/v2" // v2
      libv3 "github.com/user/lib/v3" // v3
  )

  El versionado semántico en la ruta es OBLIGATORIO en Go. Si publicas
  v2 sin /v2, go get no funcionará correctamente.

  ── 7. LA DIRECTIVA replace ──

  replace sustituye una dependencia por otra:

  // Local (desarrollo)
  replace github.com/user/mi-lib => ../mi-lib

  // Otra URL
  replace github.com/old/pkg => github.com/new/pkg v1.0.0

  // Versión diferente
  replace github.com/user/pkg v1.0.0 => github.com/user/pkg v1.0.1

  // Reemplazar por un fork
  replace github.com/original/pkg => github.com/mi-fork/pkg v1.0.0

  replace es ideal para:
  ● Desarrollo local de múltiples módulos.
  ● Parchear dependencias con bugs.
  ● Usar forks temporales.

  Advertencia: replace NO debe llegar a producción. Se puede usar
  Go Workspace para desarrollo en equipo.

  ── 8. LA DIRECTIVA exclude ──

  exclude impide que una versión se use:

  // go.mod
  exclude (
      github.com/example/pkg v1.2.3
      github.com/example/pkg v1.2.4
  )

  Si una dependencia requiere una versión excluida, Go busca la
  siguiente versión más alta que no esté excluida.

  ── 9. LA DIRECTIVA retract ──

  retract (Go 1.16+) marca versiones como retiradas:

  // go.mod
  retract (
      v1.5.0 // Error de seguridad crítico
      v1.4.0 // Dependencia rota
      v1.2.0 // No compila en Go 1.22
  )

  Cuando alguien intenta usar go get para obtener una versión retirada:

  $ go get github.com/user/pkg@v1.5.0
  go: warning: github.com/user/pkg@v1.5.0: retracted by module author:
        Error de seguridad crítico

  Útil para retirar versiones problemáticas sin eliminarlas (porque
  otros podrían depender de ellas).

  ── 10. go.sum ──

  go.sum contiene los hashes de las dependencias:

  github.com/gorilla/mux v1.8.1 h1:TuMoUvkRETdXqUhr3DmN3cWgH8R+5Lq1Nmo3s4o=
  github.com/gorilla/mux v1.8.1/go.mod h1:AKf9I4AEqPTmMytc0McKkN0ouC66V3BtZ4qD5fmWSiMQ=

  Formato:
  módulo versión h1:base64hash
  módulo versión/go.mod h1:base64hash

  go.sum:
  ● Es generado automáticamente (no editar manualmente).
  ● Debe estar en el repositorio (es parte del contrato de
    reproducibilidad).
  ● Se verifica con `go mod verify`.
  ● Si falla la verificación, el build falla (protección contra
    modificaciones maliciosas).

  ── 11. MINIMUM VERSION SELECTION (MVS) ──

  Go usa MVS para resolver dependencias:

  ● Si A requiere B v1.0.0 y C requiere B v1.5.0, Go usa B v1.5.0.
  ● Si A requiere B v1.0.0 y C requiere B v2.0.0, no hay conflicto
    (son módulos diferentes por SIV).
  ● MVS siempre elige la versión MÍNIMA que satisface todos los
    requisitos.

  MVS garantiza:
  ● Reproducibilidad (misma entrada = misma salida).
  ● Sin conflictos de dependencias (como diamond problem).
  ● La versión más baja posible (evita breaking changes innecesarios).

  ── 12. GO WORKSPACE (go.work) ──

  Go 1.18+ introdujo workspace mode para trabajar con múltiples módulos
  simultáneamente:

  // go.work
  go 1.22

  use (
      ./services/auth
      ./services/api
      ./libs/common
  )

  replace github.com/user/common => ./libs/common

  Comandos:
  go work init ./mod1 ./mod2   # Crear workspace
  go work use ./mod3           # Agregar módulo al workspace
  go work sync                 # Sincronizar dependencias

  Beneficios:
  ● Desarrollar múltiples módulos localmente sin go.mod replace.
  ● Compartir el setup con el equipo (go.work se ignora en CI típicamente).
  ● Resolver dependencias entre módulos del workspace automáticamente.

  ── 13. PUBLICAR UN MÓDULO ──

  Pasos para publicar un módulo Go:

  1. Crear el repositorio (ej. github.com/user/mi-lib).
  2. Inicializar módulo:
     go mod init github.com/user/mi-lib

  3. Escribir el código y tests.

  4. Crear un tag de versión:
     git tag v1.0.0
     git push origin v1.0.0

  5. Para v2+:
     git tag v2.0.0
     git push origin v2.0.0

  El módulo está disponible inmediatamente para quien haga:
  go get github.com/user/mi-lib@v1.0.0

  Para retirar una versión, agrega retract a go.mod:

  // go.mod
  retract v1.0.0 // Bug crítico

  Luego tag v1.0.1 y push.

  ── 14. PRÓGELO: DEPENDENCIAS PRIVADAS ──

  Para módulos en repositorios privados:

  # Configurar GOPRIVATE
  go env -w GOPRIVATE=github.com/empresa/*

  # O usar GOPROXY con autenticación
  go env -w GOPROXY=https://proxy.empresa.com,direct

  # Para git con SSH
  git config --global url."git@github.com:".insteadOf "https://github.com/"

  # Usar .netrc para autenticación
  echo "machine github.com login $USER password $TOKEN" > ~/.netrc

  GOPRIVATE evita que Go intente usar el proxy público para esos
  módulos.

  ── 15. COMANDOS ÚTILES DE GO MOD ──

  go mod init <path>     # Crear nuevo módulo
  go mod tidy             # Limpiar y descargar dependencias
  go mod download         # Descargar dependencias sin modificar código
  go mod verify           # Verificar integridad de dependencias
  go mod why -m <pkg>     # Explicar por qué se necesita un paquete
  go mod graph            # Mostrar grafo de dependencias
  go mod vendor           # Crear directorio vendor/
  go mod edit             # Editar go.mod programáticamente

  Ejemplos:

  # Ver por qué se necesita una dependencia
  go mod why -m golang.org/x/text

  # Mostrar grafo
  go mod graph | grep gorilla

  # Editar go.mod con reemplazo
  go mod edit -replace github.com/old/pkg@v1.0.0=github.com/new/pkg@v1.0.0

  ── 16. EJEMPLO COMPLETO: CREAR Y PUBLICAR UN MÓDULO ──

  # --- Paso 1: Crear el módulo ---
  mkdir go-utils && cd go-utils
  go mod init github.com/user/go-utils

  # --- Paso 2: Escribir código ---
  // --- math.go ---
  package go_utils

  func Sum(nums ...int) int {
      total := 0
      for _, n := range nums {
          total += n
      }
      return total
  }

  func Average(nums ...int) float64 {
      if len(nums) == 0 {
          return 0
      }
      return float64(Sum(nums...)) / float64(len(nums))
  }

  // --- math_test.go ---
  package go_utils

  import "testing"

  func TestSum(t *testing.T) {
      result := Sum(1, 2, 3)
      if result != 6 {
          t.Errorf("Sum(1,2,3) = %d; want 6", result)
      }
  }

  func TestAverage(t *testing.T) {
      result := Average(2, 4, 6)
      if result != 4.0 {
          t.Errorf("Average(2,4,6) = %f; want 4.0", result)
      }
  }

  # --- Paso 3: Agregar dependencia externa ---
  go get github.com/google/uuid

  # --- go.mod resultante ---
  module github.com/user/go-utils

  go 1.22

  require (
      github.com/google/uuid v1.6.0
  )

  # --- Paso 4: Publicar ---
  git init && git add . && git commit -m "Initial release"
  git remote add origin https://github.com/user/go-utils.git
  git push origin main
  git tag v1.0.0 && git push origin v1.0.0

  # --- Paso 5: Usar el módulo en otro proyecto ---
  cd /otro/proyecto
  go get github.com/user/go-utils@v1.0.0

  // main.go
  import "github.com/user/go-utils"

  func main() {
      fmt.Println(go_utils.Sum(1, 2, 3)) // 6
  }

  ── 17. MODULE COMPATIBILITY ──

  Cuando actualizas Go, también debes actualizar go.mod:

  go mod tidy -go=1.22   # Actualizar a Go 1.22

  Para verificar compatibilidad:

  go build ./...
  go vet ./...

  Si una dependencia no soporta la nueva versión de Go, puedes:
  1. Buscar una alternativa.
  2. Usar replace con un fork compatible.
  3. Mantener la versión anterior de Go.

  ── 18. DEPENDENCIAS CÍCLICAS ──

  Go Modules no permite dependencias cíclicas entre módulos:

  Módulo A → depende de Módulo B
  Módulo B → depende de Módulo A  // ❌ Ciclo

  Si detectas un ciclo:
  1. Extrae la interfaz común a un tercer módulo C.
  2. Fusiona A y B en un solo módulo.
  3. Invierte la dirección de la dependencia.

  ── 19. DIAGNÓSTICO DE PROBLEMAS ──

  go env GOPATH              # Ver GOPATH
  go env GOMODCACHE          # Ver caché de módulos
  go env GOPROXY             # Ver proxy usado
  go env GONOSUMCHECK        # Módulos sin verificación
  go env GONOSUMDB           # Bases de datos de checksums omitidas
  go env GOFLAGS             # Flags globales

  # Limpiar caché de módulos
  go clean -modcache

  # Ver todos los módulos disponibles
  go list -m all

  # Ver módulos en el build
  go list -m -json all

  ── 20. BUENAS PRÁCTICAS ──

  1. Siempre usa `go mod tidy` antes de commit.
  2. go.sum debe estar versionado en el repositorio.
  3. Usa versionado semántico rigurosamente.
  4. Para v2+, incluye /v2 en el module path.
  5. Evita replace en producción; usa workspace para desarrollo local.
  6. Publica versiones estables (v1+) solo cuando la API sea estable.
  7. Documenta breaking changes en las release notes.
  8. Retira versiones problemáticas con retract, no las elimines.
  9. Usa GOPRIVATE para módulos privados.
  10. Revisa dependencias no usadas con go mod tidy regularmente.

  ── 21. EJEMPLO: MIGRAR DE V1 A V2 ──

  // --- go.mod v1 ---
  module github.com/user/mi-lib

  go 1.22

  // --- v1: maths.go ---
  package mi_lib

  func Sum(a, b int) int { return a + b }

  // --- Migrar a v2 con breaking changes ---

  // 1. Crear directorio v2/
  mkdir -p v2

  // 2. Nueva carpeta con su propio go.mod
  // --- v2/go.mod ---
  module github.com/user/mi-lib/v2

  go 1.22

  // --- v2/maths.go ---
  package mi_lib

  func Sum(nums ...int) int { // Breaking: de binaria a variádica
      total := 0
      for _, n := range nums {
          total += n
      }
      return total
  }

  // 3. Publicar
  git tag v2.0.0 && git push origin v2.0.0

  // 4. Usuarios pueden importar ambas:
  import (
      "github.com/user/mi-lib"         // v1
      v2 "github.com/user/mi-lib/v2"   // v2
  )

  ── 22. WORKSPACE EN ACCIÓN ──

  // --- Estructura del workspace ---
  mi-workspace/
  ├── go.work
  ├── servicios/
  │   └── api/
  │       ├── go.mod        // module github.com/team/api
  │       └── main.go       // importa "github.com/team/libs/common"
  └── libs/
      └── common/
          ├── go.mod        // module github.com/team/libs/common
          └── util.go

  // --- go.work ---
  go 1.22

  use (
      ./servicios/api
      ./libs/common
  )

  // Beneficio: api puede importar common sin replace
  // y los cambios en common se reflejan inmediatamente.

  // En CI, puedes ignorar go.work y compilar cada módulo por separado:
  cd servicios/api && go build ./...
  cd libs/common && go build ./...

  RESUMEN DEL CAPÍTULO
  ────────────────────
  ● go mod init crea un módulo; go mod tidy mantiene dependencias.
  ● go get agrega/actualiza dependencias con versionado semántico.
  ● SIV exige /v2+ en el module path para versiones mayores a 1.
  ● replace redirige dependencias (útil en desarrollo local).
  ● exclude bloquea versiones; retract retira versiones publicadas.
  ● go.sum verifica integridad de dependencias.
  ● MVS (Minimum Version Selection) resuelve dependencias sin
    conflictos.
  ● Go Workspace (go.work) permite desarrollar múltiples módulos.
  ● GOPRIVATE gestiona módulos privados.

  PREGUNTAS FRECUENTES (FAQ)
  ──────────────────────────
  P: ¿Qué hago si mi módulo no está en GitHub?
  R: Puedes usar cualquier URL: `ejemplo.com/usuario/mod`. La URL no
     necesita ser accesible (Go no hace HTTP a la raíz del módulo).

  P: ¿Cómo actualizo todas las dependencias a la última versión?
  R: `go get -u ./...` seguido de `go mod tidy`.

  P: ¿Qué significa "verifying module" cuando ejecuto go build?
  R: Go está descargando y verificando los checksums de go.sum. Es
     normal en la primera compilación.

  P: ¿Puedo tener múltiples go.mod en un repositorio?
  R: Sí. Es común en monorepos tener un go.mod por cada servicio o
     biblioteca independiente.

  P: ¿Debo usar go.work en producción?
  R: No. go.work es para desarrollo local. En CI compila cada módulo
     independientemente.

  DESAFÍO DEL ARQUITECTO
  ──────────────────────
  Diseña la arquitectura de módulos para un monorepo con estas
  características:

  1. Tres servicios (auth, api, worker) con dependencias compartidas.
  2. Una biblioteca común (common) usada por todos los servicios.
  3. Una biblioteca de proto/gRPC (protos) con dependencia de common.
  4. Versiones internas de las bibliotecas (v1, v2 coexistiendo).
  5. Dependencias externas compartidas.
  6. Workspace para desarrollo local.
  7. CI/CD que compile y testee cada servicio por separado.
  8. El desafío: implementa un pipeline de CI que:
     - Detecte qué módulos cambiaron.
     - Compile y testee solo los módulos afectados y sus dependientes.
     - Publique nuevas versiones de las bibliotecas automáticamente.
     - Use versionado semántico para las bibliotecas compartidas.
     - Gestione retract de versiones problemáticas.
