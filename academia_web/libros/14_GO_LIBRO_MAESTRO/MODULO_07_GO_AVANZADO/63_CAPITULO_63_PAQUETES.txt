  BIENVENIDA
  ──────────
  Bienvenido al Capítulo 63. Los paquetes y módulos son la unidad
  fundamental de organización en Go. Un proyecto Go bien estructurado
  refleja experiencia y previsión arquitectónica. En este capítulo
  exploraremos desde la declaración package hasta go.mod, go.sum,
  y el versionado semántico.

  FRASE DEL CAPÍTULO
  ──────────────────
  "El código se organiza para las personas, no para la máquina. Los
  paquetes son la forma en que Go nos obliga a pensar en el diseño."
  — Russ Cox

  4. Usar go.sum para la verificación de dependencias.
  5. Aplicar Semantic Import Versioning (SIV).
  6. Manejar dependencias internas y externas.

  ── 1. LA CLÁUSULA PACKAGE ──

  Todo archivo .go comienza con la declaración package:

  package mi_paquete

  Reglas:
  ● El nombre del paquete debe coincidir con el nombre del directorio
    que lo contiene (salvo excepciones como main).
  ● Todos los archivos .go en un mismo directorio deben pertenecer al
    mismo paquete.
  ● El paquete `main` genera un ejecutable; cualquier otro genera una
    biblioteca.

  Ejemplo de estructura:

  mi-proyecto/
  ├── main.go           // package main
  ├── go.mod
  ├── matematica/
  │   ├── suma.go       // package matematica
  │   └── resta.go      // package matematica
  └── texto/
      └── formato.go    // package texto

  ── 2. EL ARCHIVO go.mod ──

  go.mod define el módulo: su ruta, versión de Go, y dependencias.

  // go.mod típico
  module github.com/usuario/mi-proyecto

  go 1.22

  require (
      github.com/gorilla/mux v1.8.1
      golang.org/x/exp v0.0.0-20240613220409-f7d5f5e3e6f1
  )

  Campos principales:
  ● module: ruta del módulo (coincide con el repositorio generalmente).
  ● go: versión mínima de Go requerida.
  ● require: dependencias con versiones.
  ● exclude: módulos excluidos.
  ● replace: redirección de módulos (útil para desarrollo local).
  ● retract: versiones retiradas.

  ── 3. MODULE PATH ──

  El module path es el nombre único del módulo. Tradicionalmente coincide
  con la URL del repositorio:

  module github.com/gorilla/mux

  Se usa como prefijo para las rutas de importación internas:

  // main.go
  import (
      "github.com/gorilla/mux"
      "github.com/usuario/mi-proyecto/matematica"
  )

  La ruta de importación es: module path + subdirectorio.

  ── 4. EL ARCHIVO go.sum ──

  go.sum contiene los checksums (hashes) de las dependencias para
  verificar integridad y autenticidad. Es generado automáticamente por
  `go mod tidy`.

  // go.sum (ejemplo)
  github.com/gorilla/mux v1.8.1 h1:TuMoUvkRETdXqUhr3DmN3cWgH8R+5Lq1Nmo3s4o=
  github.com/gorilla/mux v1.8.1/go.mod h1:AKf9I4AEqPTmMytcMc0KkNouC66V3BtZ4qD5fmWSiMQ=

  Cada dependencia genera dos líneas:
  ● Una para el contenido del módulo.
  ● Una para su go.mod.

  go.sum NO debe editarse manualmente. Siempre usa `go mod tidy`.

  ── 5. INICIALIZAR UN MÓDULO ──

  # Crear un nuevo módulo
  mkdir mi-proyecto && cd mi-proyecto
  go mod init github.com/usuario/mi-proyecto

  Esto crea go.mod con:

  module github.com/usuario/mi-proyecto

  go 1.22

  Luego, al agregar código que importe paquetes externos, ejecuta:

  go mod tidy

  Este comando:
  ● Descarga las dependencias faltantes.
  ● Elimina dependencias no usadas.
  ● Actualiza go.sum.
  ● Actualiza go.mod con las versiones correctas.

  ── 6. ESTRUCTURA DE UN PAQUETE ──

  Dentro de un módulo, cada subdirectorio (excepto los ignorados) es un
  paquete independiente:

  mi-proyecto/
  ├── cmd/
  │   └── servidor/
  │       └── main.go          // package main
  ├── internal/
  │   └── auth/
  │       ├── auth.go          // package auth
  │       └── auth_test.go     // package auth
  ├── pkg/
  │   ├── api/
  │   │   └── handlers.go     // package api
  │   └── models/
  │       └── user.go         // package models
  ├── go.mod
  └── go.sum

  Convenciones:
  ● `cmd/`: ejecutables (package main).
  ● `internal/`: código privado del módulo.
  ● `pkg/`: código público reutilizable.
  ● `config/`, `docs/`, `scripts/`: directorios auxiliares.

  ── 7. SEMANTIC IMPORT VERSIONING (SIV) ──

  Go usa versionado semántico (vX.Y.Z) y lo integra en la ruta del
  módulo para versiones mayores > 1:

  // Módulo versión 1.x (la v1 es implícita)
  module github.com/user/mod

  // Módulo versión 2.x (v2 explícita en la ruta)
  module github.com/user/mod/v2

  // Módulo versión 3.x
  module github.com/user/mod/v3

  Esto permite que múltiples versiones mayores coexistan:

  import (
      "github.com/user/mod/v2/pkg"
      v3 "github.com/user/mod/v3/pkg"
  )

  Reglas SIV en Go:
  ● v0.x.x: versión inestable, no hay compromiso de compatibilidad.
  ● v1.x.x: API estable. La ruta NO lleva /v1.
  ● v2+: la ruta DEBE llevar /vN.

  ── 8. DEPENDENCIAS INTERNAS ──

  Las importaciones dentro del mismo módulo usan el module path como
  prefijo:

  // go.mod
  module github.com/empresa/mi-app

  // cmd/servidor/main.go
  package main

  import (
      "github.com/empresa/mi-app/internal/auth"
      "github.com/empresa/mi-app/pkg/models"
  )

  func main() {
      u := models.User{Name: "Alice"}
      auth.Login(u)
  }

  No uses `../` en importaciones. La ruta del módulo es la referencia
  absoluta.

  ── 9. DEPENDENCIAS EXTERNAS ──

  Para agregar una dependencia externa:

  # Método 1: importar y luego go mod tidy
  # Edita tu código para importar el paquete, luego:
  go mod tidy

  # Método 2: go get directo
  go get github.com/gorilla/mux@v1.8.1

  # Método 3: actualizar a la última versión
  go get -u github.com/gorilla/mux

  go get modifica go.mod y go.sum automáticamente.

  ── 10. EL COMANDO go mod ──

  go mod tidy        # Limpia y descarga dependencias
  go mod download    # Descarga dependencias sin modificar código
  go mod verify      # Verifica integridad de dependencias
  go mod why         # Explica por qué se necesita una dependencia
  go mod graph       # Muestra el grafo de dependencias
  go mod vendor      # Crea directorio vendor/
  go mod edit        # Edita go.mod programáticamente

  Ejemplo de go mod why:

  $ go mod why -m golang.org/x/text
  # golang.org/x/text
  github.com/user/project
  github.com/user/project/pkg/handler
  github.com/gorilla/mux
  golang.org/x/text

  ── 11. LA DIRECTIVA REPLACE ──

  replace permite sustituir una dependencia por otra ruta (local o
  remota). Es útil para desarrollo local:

  // go.mod
  module github.com/user/project

  go 1.22

  require (
      github.com/user/mi-lib v0.1.0
  )

  // Reemplazar con versión local
  replace github.com/user/mi-lib => ../mi-lib

  // También puedes reemplazar por una URL diferente
  replace github.com/old/pkg => github.com/new/pkg v1.0.0

  replace SOLO debe usarse en desarrollo, no en producción. Se puede
  eliminar el bloque replace antes de publicar.

  ── 12. LA DIRECTIVA EXCLUDE ──

  exclude impide que una versión específica sea usada:

  // go.mod
  exclude github.com/example/pkg v1.2.3

  Útil cuando una versión tiene un bug crítico. El módulo usará la
  versión anterior o deberá actualizar a una más reciente.

  ── 13. LA DIRECTIVA RETRACT ──

  retract (Go 1.16+) permite marcar versiones como retiradas:

  // go.mod
  retract (
      v1.2.0 // Error crítico en la función CreateUser
      v1.1.0 // Problema de seguridad
  )

  Cuando un usuario intente usar una versión retirada, go get mostrará
  un mensaje de advertencia.

  ── 14. VENDORING ──

  El directorio vendor/ contiene copias de todas las dependencias:

  # Crear vendor/
  go mod vendor

  # Compilar usando vendor/ (sin descargar de internet)
  go build -mod=vendor

  # Verificar que vendor/ está actualizado
  go mod vendor && go build -mod=vendor ./...

  Ventajas del vendoring:
  ● Reproducibilidad total (incluso si la fuente remota desaparece).
  ● Control de cambios en las dependencias (puedes commitear vendor/).
  ● Compilación offline.

  Desventajas:
  ● Aumenta el tamaño del repositorio.
  ● Requiere mantenimiento extra.

  ── 15. EJEMPLO COMPLETO: MÓDULO MULTI-PAQUETE ──

  // --- go.mod ---
  module github.com/ejemplo/calculadora

  go 1.22

  require (
      github.com/ejemplo/matematica v0.1.0
  )

  // --- cmd/calc/main.go ---
  package main

  import (
      "fmt"
      "os"
      "strconv"

      "github.com/ejemplo/calculadora/internal/operaciones"
      "github.com/ejemplo/calculadora/pkg/formateo"
  )

  func main() {
      if len(os.Args) < 4 {
          fmt.Println("Uso: calc <op> <a> <b>")
          os.Exit(1)
      }
      op := os.Args[1]
      a, _ := strconv.Atoi(os.Args[2])
      b, _ := strconv.Atoi(os.Args[3])

      var result int
      switch op {
      case "suma":
          result = operaciones.Sumar(a, b)
      case "resta":
          result = operaciones.Restar(a, b)
      case "mult":
          result = operaciones.Multiplicar(a, b)
      default:
          fmt.Println("Operación no soportada")
          os.Exit(1)
      }

      fmt.Println(formateo.FormatResult(result))
  }

  // --- internal/operaciones/operaciones.go ---
  package operaciones

  func Sumar(a, b int) int {
      return a + b
  }

  func Restar(a, b int) int {
      return a - b
  }

  func Multiplicar(a, b int) int {
      return a * b
  }

  // --- pkg/formateo/formateo.go ---
  package formateo

  import "fmt"

  func FormatResult(result int) string {
      return fmt.Sprintf("Resultado: %d", result)
  }

  ── 16. CICLOS DE IMPORTACIÓN ──

  Go NO permite ciclos de importación. Si A importa B y B importa A,
  el compilador lanza error:

  // ❌ Ciclo
  // paquete A: import "B"
  // paquete B: import "A"

  Soluciones:
  1. Extraer la dependencia común a un tercer paquete C.
  2. Usar una interface en lugar de un tipo concreto.
  3. Reestructurar los paquetes.

  ── 17. EL PAQUETE INTERNAL ──

  El directorio `internal/` tiene un tratamiento especial: solo los
  paquetes dentro del mismo árbol (mismo módulo raíz) pueden importarlo:

  proyecto/
  ├── internal/
  │   └── secreto/
  │       └── token.go    // package secreto
  ├── cmd/
  │   └── app/
  │       └── main.go     // puede importar internal/secreto
  └── go.mod

  // Otro módulo externo NO puede importar:
  // import "github.com/usuario/proyecto/internal/secreto"

  El compilador Go impone esta restricción. Es el mecanismo de
  encapsulación más fuerte en Go.

  ── 18. INIT FUNCTIONS ──

  Cada paquete puede tener una o más funciones init() que se ejecutan
  automáticamente al importar el paquete:

  package db

  import "log"

  var Conexion *sql.DB

  func init() {
      log.Println("Inicializando conexión a BD...")
      // Conexion = abrirConexion()
  }

  func init() {
      log.Println("Segunda inicialización...")
  }

  Orden de ejecución:
  1. init() de dependencias importadas (en profundidad).
  2. init() del paquete actual (en orden de aparición).
  3. main() del paquete main.

  ── 19. BLANK IDENTIFIER EN IMPORTACIONES ──

  Usa `_` para importar un paquete solo por sus efectos secundarios
  (como init()):

  import (
      _ "github.com/lib/pq"   // Registra driver PostgreSQL
      _ "image/png"            // Registra decodificador PNG
  )

  Así no obtienes errores de "imported and not used", pero el init()
  del paquete se ejecuta.

  ── 20. ALIAS DE IMPORTACIÓN ──

  Puedes renombrar paquetes importados para evitar colisiones o acortar
  nombres:

  import (
      "crypto/rand"
      mrand "math/rand"
  )

  func main() {
      // crypto/rand
      b := make([]byte, 16)
      rand.Read(b)

      // math/rand (con alias)
      r := mrand.Intn(100)
      fmt.Println(r)
  }

  También puedes usar el punto (.) para importar nombres al scope
  global (no recomendado en producción):

  import (
      . "fmt"
  )

  func main() {
      Println("Hola") // Sin prefijo fmt.
  }

  ── 21. EJEMPLO AVANZADO: PLUGIN ARCHITECTURE ──

  // --- pkg/plugin/plugin.go ---
  package plugin

  type Plugin interface {
      Name() string
      Execute(data interface{}) (interface{}, error)
  }

  type Registry struct {
      plugins map[string]Plugin
  }

  func NewRegistry() *Registry {
      return &Registry{plugins: make(map[string]Plugin)}
  }

  func (r *Registry) Register(p Plugin) {
      r.plugins[p.Name()] = p
  }

  func (r *Registry) Get(name string) Plugin {
      return r.plugins[name]
  }

  func (r *Registry) List() []string {
      names := make([]string, 0, len(r.plugins))
      for name := range r.plugins {
          names = append(names, name)
      }
      return names
  }

  // --- internal/plugins/upper.go ---
  package plugins

  import "strings"

  type UpperPlugin struct{}

  func (p UpperPlugin) Name() string { return "upper" }
  func (p UpperPlugin) Execute(data interface{}) (interface{}, error) {
      s, ok := data.(string)
      if !ok {
          return nil, fmt.Errorf("expected string, got %T", data)
      }
      return strings.ToUpper(s), nil
  }

  // --- cmd/server/main.go ---
  package main

  import (
      "fmt"
      "github.com/ejemplo/app/internal/plugins"
      "github.com/ejemplo/app/pkg/plugin"
  )

  func main() {
      reg := plugin.NewRegistry()
      reg.Register(plugins.UpperPlugin{})

      for _, name := range reg.List() {
          p := reg.Get(name)
          result, _ := p.Execute("hello")
          fmt.Printf("%s: %v\n", name, result)
      }
  }

  ── 22. BUENAS PRÁCTICAS ──

  1. Mantén los paquetes pequeños y enfocados (SRP).
  2. Nombra los paquetes con sustantivos simples en minúscula.
  3. Evita nombres genéricos como `utils`, `common`, `helpers`.
  4. Un paquete no debe exportar más de lo necesario.
  5. Usa `internal/` para proteger implementaciones privadas.
  6. No uses `init()` para efectos secundarios ocultos; prefiere
     inicialización explícita.
  7. Mantén actualizado go.mod con `go mod tidy`.
  8. Versiona semánticamente y sigue SIV para v2+.
  9. No comitees vendor/ a menos que sea estrictamente necesario.
  10. Usa `replace` solo en desarrollo local; elimínalo en CI/CD.

  RESUMEN DEL CAPÍTULO
  ────────────────────
  ● La cláusula package define el paquete de cada archivo .go; todos los
    archivos en un directorio deben tener el mismo package.
  ● go.mod define el módulo, su ruta, versión de Go y dependencias.
  ● go.sum contiene checksums para verificar integridad de dependencias.
  ● SIV (Semantic Import Versioning) requiere /v2+ en la ruta para
    versiones mayores a 1.
  ● `internal/` restringe importaciones al árbol del módulo.
  ● `replace` permite redirigir dependencias en desarrollo.

  EJERCICIOS PROPUESTOS

     - `pkg/logger` (público)
     - `internal/formatter` (privado)
     - `cmd/cli` (ejecutable)
  2. Agrega una dependencia externa (ej. `github.com/fatih/color`) y usa
     `go mod tidy`.
  3. Simula un ciclo de importación y luego resuélvelo extrayendo un
     tercer paquete.
  4. Usa `replace` para apuntar un módulo a un directorio local.
  5. Crea una versión v2 de tu módulo (con /v2 en el module path).
  6. Usa `go mod graph` para visualizar el grafo de dependencias.
  7. Implementa `go mod vendor` y compila con `-mod=vendor`.
  8. Añade una directiva `retract` a una versión y ejecuta `go list -m
     -retracted`.
  9. Crea un paquete que use blank import para registrar un driver.
  10. Documenta el uso de init() en tus paquetes y luego refactoriza
      para usar inicialización explícita.

  PREGUNTAS FRECUENTES (FAQ)
  ──────────────────────────
  P: ¿Qué pasa si dos dependencias requieren versiones diferentes del
      mismo módulo?
  R: Go resuelve automáticamente usando Minimum Version Selection (MVS):
     elige la versión mínima que satisface todos los require.

  P: ¿Debo comitear go.sum?
  R: Sí. go.sum debe estar en el repositorio para garantizar
     reproducibilidad y seguridad.

  P: ¿Cómo actualizo todas las dependencias a la última versión?
  R: `go get -u ./...` actualiza todas las dependencias. Luego
     `go mod tidy` para limpiar.

  P: ¿Qué hago si una dependencia se elimina de internet?
  R: Si tienes vendor/, puedes compilar offline. Si no, busca un mirror
     o reemplaza el módulo con replace.

  P: ¿Puedo tener múltiples go.mod en un mismo repositorio?
  R: Sí. Es común tener un go.mod por cada aplicación o biblioteca
     independiente dentro del mismo repo (monorepo).

  DESAFÍO DEL ARQUITECTO
  ──────────────────────
  Diseña un monorepo con esta estructura:

  monorepo/
  ├── go.work              // Workspace global
  ├── services/
  │   ├── auth/
  │   │   ├── go.mod
  │   │   └── cmd/auth/main.go
  │   └── api/
  │       ├── go.mod
  │       └── cmd/api/main.go
  ├── libs/
  │   ├── common/
  │   │   ├── go.mod
  │   │   └── pkg/...
  │   └── proto/
  │       └── ...
  └── tools/
      └── ...

  Implementa:
  1. Un workspace (go.work) que conecte services/ y libs/.
  2. El servicio auth usa common como dependencia local.
  3. El servicio api usa ambos auth y common.
  4. Usa replace para desarrollo local.
  5. La biblioteca common tiene una versión v2 con breaking changes.
  6. El desafío: gestionar versiones compartidas sin romper nada.
