================================================================================
  19.1 INTRODUCCIÓN TEÓRICA
================================================================================

  Scope (ámbito) se refiere a la región del código donde una variable
  es visible y accesible. En Go, el scope sigue reglas simples y
  predecibles.

  Tipos de scope en Go:

  1. Scope de bloque: las variables declaradas dentro de un bloque
     { } solo son visibles dentro de ese bloque y sus sub-bloques.
     Esto incluye: funciones, if, for, switch, etc.

  2. Scope de paquete: las variables declaradas fuera de cualquier
     función son accesibles desde cualquier archivo del mismo paquete.

  3. Scope exportado: las variables/funciones que empiezan con
     mayúscula son accesibles desde otros paquetes.

  Un paquete en Go es una colección de archivos .go en el mismo
  directorio que comparten el mismo nombre de paquete (package).

  Cada archivo .go comienza con:
    package nombrepaquete

  El paquete main es especial: genera un ejecutable. Los demás
  paquetes son librerías que se importan.

  Los Go modules (go.mod) son la forma moderna de gestionar
  dependencias en Go. Un módulo agrupa uno o más paquetes.

================================================================================
  19.2 SCOPE LOCAL (DENTRO DE FUNCIÓN/BLOQUE)
================================================================================

  Las variables declaradas dentro de una función o bloque solo
  existen dentro de ese bloque.

  ```go
  package main

  import "fmt"

  func main() {
      x := 10 // scope: toda la función main
      fmt.Println(x)

      if x > 5 {
          y := 20 // scope: solo dentro del if
          fmt.Println(x + y) // 30 (x es visible aquí)
      }

      // fmt.Println(y) // ERROR: y no está definido aquí
  }
  ```

  Reglas de scope de bloque:

  1. Una variable es visible desde su declaración hasta el final
     del bloque donde fue declarada.

  2. Los sub-bloques (bloques anidados) pueden acceder a variables
     de bloques exteriores.

  3. Un bloque no puede acceder a variables de bloques hermanos.

  ```go
  package main

  import "fmt"

  func main() {
      a := 1

      {
          b := 2
          fmt.Println(a) // 1 (visible del exterior)
          fmt.Println(b) // 2 (visible localmente)

          {
              c := 3
              fmt.Println(a, b, c) // 1, 2, 3 (todos visibles)
          }

          // fmt.Println(c) // ERROR: c no está definido aquí
      }

      // fmt.Println(b) // ERROR: b no está definido aquí
  }
  ```

  Short statement en if, for, switch:

  ```go
  if v := calcular(); v > 10 {
      fmt.Println(v) // v visible aquí
  } else {
      fmt.Println(v) // v también visible aquí
  }
  // fmt.Println(v) // ERROR: v no está definido aquí
  ```

================================================================================
  19.3 SCOPE DE PAQUETE (PACKAGE-LEVEL)
================================================================================

  Las variables declaradas fuera de cualquier función tienen scope
  de paquete. Son accesibles desde cualquier archivo del mismo
  paquete.

  archivo: main.go
  ```go
  package main

  import "fmt"

  var version string = "1.0.0" // scope de paquete

  func main() {
      fmt.Println("Versión:", version)
      mostrarVersion()
  }

  func mostrarVersion() {
      fmt.Println("Versión desde otra función:", version)
  }
  ```

  archivo: helpers.go (mismo paquete main)
  ```go
  package main

  import "fmt"

  func init() {
      version = "1.0.1" // Accede a version del paquete
  }

  func ayuda() {
      fmt.Println("Ayuda - Versión:", version)
  }
  ```

  Las variables de paquete se inicializan antes de que main se
  ejecute. Puedes usar init() para inicialización adicional.

  Orden de inicialización:
  1. Variables de paquete (en orden de declaración).
  2. Funciones init() (en orden de aparición en los archivos).
  3. main().

  ```go
  package main

  import "fmt"

  var a = b + 1 // 3 (se evalúa después de b)
  var b = 2

  func init() {
      fmt.Println("init: a=", a, "b=", b)
  }

  func init() {
      fmt.Println("init2: orden de init por archivo")
  }

  func main() {
      fmt.Println("main: a=", a, "b=", b)
  }
  ```

  Output:
    init: a= 3 b= 2
    init2: orden de init por archivo
    main: a= 3 b= 2

================================================================================
  19.4 SHADOWING DE VARIABLES
================================================================================

  Shadowing (sombreado) ocurre cuando declaras una variable con el
  mismo nombre que otra de un scope exterior. La variable interna
  "sombrea" a la externa dentro de su bloque.

  ```go
  package main

  import "fmt"

  var x = 10 // package-level

  func main() {
      x := 20 // shadowing: esta x sombrea a la de paquete
      fmt.Println(x) // 20

      if true {
          x := 30 // shadowing dentro del if
          fmt.Println(x) // 30
      }

      fmt.Println(x) // 20
  }
  ```

  Output:
    20
    30
    20

  El shadowing puede causar bugs sutiles:

  ```go
  package main

  import (
      "fmt"
      "os"
  )

  func main() {
      var err error

      // Incorrecto: := crea una nueva err (shadowing)
      // archivo, err := os.Open("archivo.txt")

      // Correcto: = reusa la err existente
      archivo, err := os.Open("archivo.txt")
      // Pero := con err ya declarada... ¿qué pasa?

      // En Go, := re-declara si al menos una variable es nueva
      // archivo es nueva, err ya existe → se reusa
      if err != nil {
          fmt.Println("Error:", err)
          return
      }
      defer archivo.Close()
  }
  ```

  Shadowing con short statement en if:

  ```go
  x := "original"
  if x := "shadow"; x != "" {
      fmt.Println(x) // "shadow"
  }
  fmt.Println(x) // "original"
  ```

  Para evitar shadowing accidental, algunos equipos usan
  herramientas como `go vet` con `-shadow` flag.

================================================================================
  19.5 EXPORTADO VS NO EXPORTADO
================================================================================

  En Go, la visibilidad fuera del paquete se determina por la
  primera letra del nombre:

  - MAYÚSCULA: exportado (visible desde otros paquetes).
  - minúscula: no exportado (visible solo dentro del paquete).

  ```go
  package geometries

  import "math"

  // Exportada (mayúscula) - visible fuera del paquete
  type Circulo struct {
      Radio float64 // Exportado (mayúscula)
  }

  // Exportada
  func (c Circulo) Area() float64 {
      return c.radioAlCuadrado() * math.Pi
  }

  // No exportada (minúscula) - solo visible dentro del paquete
  func (c Circulo) radioAlCuadrado() float64 {
      return c.Radio * c.Radio
  }

  // No exportada
  var piAprox = 3.1416

  // Exportada
  const PI_EXACTO = 3.14159265359
  ```

  Desde otro paquete:

  ```go
  package main

  import "tuproyecto/geometries"

  func main() {
      c := geometries.Circulo{Radio: 5}
      area := c.Area()
      // c.radioAlCuadrado() // ERROR: no exportada

      // fmt.Println(geometries.piAprox) // ERROR: no exportada
      fmt.Println(geometries.PI_EXACTO) // OK
  }
  ```

  Reglas:
  - Tipos, funciones, variables, constantes: siguen la misma regla.
  - Campos de struct: si el campo empieza con minúscula, no es
    accesible desde otros paquetes.
  - Métodos: si el método empieza con minúscula, no es accesible.
  - Las interfaces pueden tener métodos no exportados, pero no
    podrán implementarse fuera del paquete.

  Esta es una de las características de diseño más importantes de Go.
  No hay public/private/protected como en otros lenguajes. Solo
  exportado (mayúscula) y no exportado (minúscula).

================================================================================
  19.6 PAQUETES EN GO
================================================================================

  Un paquete es una colección de archivos .go en un mismo directorio
  que comparten el mismo package declaration.

  Estructura de directorios típica:

    mi-proyecto/
      go.mod
      main.go               (package main)
      auth/
        auth.go             (package auth)
        tokens.go           (package auth)
      db/
        postgres.go         (package db)
        mysql.go            (package db)
      api/
        handlers.go         (package api)
        middleware.go       (package api)
        models/
          user.go           (package models)
          product.go        (package models)

  Reglas de paquetes:
  1. Todos los archivos en un directorio deben tener el mismo
     package name.
  2. El nombre del paquete generalmente coincide con el nombre
     del directorio, pero no es obligatorio.
  3. No puedes tener múltiples paquetes en el mismo directorio.

  archivo: auth/auth.go
  ```go
  package auth

  import "fmt"

  type Usuario struct {
      Nombre string
      Rol    string
  }

  func Login(nombre, password string) (*Usuario, error) {
      if nombre == "" || password == "" {
          return nil, fmt.Errorf("credenciales requeridas")
      }
      return &Usuario{Nombre: nombre, Rol: "user"}, nil
  }

  func (u *Usuario) EsAdmin() bool {
      return u.Rol == "admin"
  }
  ```

  archivo: main.go
  ```go
  package main

  import (
      "fmt"
      "mi-proyecto/auth"
  )

  func main() {
      usuario, err := auth.Login("ana", "pass123")
      if err != nil {
          fmt.Println("Error:", err)
          return
      }
      fmt.Println("Usuario:", usuario.Nombre)
  }
  ```

================================================================================
  19.7 IMPORTACIÓN DE PAQUETES
================================================================================

  Para usar un paquete, debes importarlo:

  ```go
  import "fmt"
  import "math/rand"
  import "mi-proyecto/auth"
  ```

  O agruparlos:

  ```go
  import (
      "fmt"
      "math/rand"
      "mi-proyecto/auth"
  )
  ```

  Reglas de importación:
  - Cada ruta de importación es única.
  - El nombre usado para referenciar el paquete es el último
    elemento de la ruta (auth para "mi-proyecto/auth").
  - Si importas un paquete y no lo usas, el código no compila.
  - Puedes usar _ para importar un paquete solo por sus efectos
    secundarios (init functions).

  ```go
  import (
      "fmt"
      _ "image/png" // Solo ejecuta init() de image/png
  )
  ```

  Sub-importaciones:

  ```go
  import (
      "math/rand"   // rand.Intn()
      "net/http"    // http.Get()
  )
  ```

  El paquete standard library (stdlib) se importa solo con el
  nombre del paquete. Los paquetes de terceros o propios necesitan
  la ruta completa del módulo.

  Importación para side effects:

  ```go
  import (
      "database/sql"
      _ "github.com/lib/pq" // Driver PostgreSQL, solo init()
  )
  ```

================================================================================
  19.8 ALIAS DE PAQUETES
================================================================================

  Puedes asignar un alias a un paquete importado para evitar
  conflictos de nombres o acortar referencias:

  ```go
  import (
      "crypto/rand"
      mrand "math/rand" // alias
  )

  func main() {
      // crypto/rand
      c := rand.Reader

      // math/rand (con alias)
      r := mrand.Intn(100)
  }
  ```

  También puedes usar un alias cuando el nombre del paquete no es
  descriptivo o cuando hay conflicto:

  ```go
  import (
      "encoding/json"
      "encoding/xml"

      "mi-app/models"
      admin "mi-app/models/admin"
      user "mi-app/models/user"
  )

  func main() {
      // Sin alias
      var u models.Usuario
      var a admin.Admin
      var us user.Usuario
  }
  ```

  El alias puede ser cualquier identificador válido:

  ```go
  import (
      f "fmt"
      t "time"
  )

  func main() {
      f.Println("Hora:", t.Now())
  }
  ```

  El alias con punto (.) importa todos los nombres exportados al
  scope actual (no recomendado, causa confusión):

  ```go
  import . "fmt"

  func main() {
      Println("Hola") // No necesita fmt.Println
  }
  ```

================================================================================
  19.9 GO MODULES
================================================================================

  Desde Go 1.11, los Go modules son el sistema oficial de gestión
  de dependencias.

  Un módulo se define con un archivo go.mod en la raíz del proyecto:

    module mi-proyecto

    go 1.21

    require (
        github.com/gin-gonic/gin v1.9.1
        github.com/lib/pq v1.10.9
    )

  Comandos útiles:

    go mod init <nombre>    # Inicializa un módulo
    go mod tidy             # Limpia y descarga dependencias
    go mod download         # Descarga dependencias
    go mod vendor           # Copia dependencias a vendor/
    go get <paquete>        # Agrega dependencia
    go list -m all          # Lista todos los módulos

  Ejemplo de ciclo de trabajo:

    # Iniciar proyecto
    mkdir mi-api
    cd mi-api
    go mod init github.com/usuario/mi-api

    # Agregar dependencia
    go get github.com/gin-gonic/gin

    # Estructura del proyecto:
    # mi-api/
    #   go.mod
    #   go.sum
    #   main.go
    #   handlers/
    #     usuarios.go
    #   models/
    #     usuario.go

  archivo: go.mod
  ```
  module github.com/usuario/mi-api

  go 1.21

  require (
      github.com/gin-gonic/gin v1.9.1
      github.com/lib/pq v1.10.9
  )
  ```

  archivo: main.go
  ```go
  package main

  import (
      "github.com/gin-gonic/gin"
      "github.com/usuario/mi-api/handlers"
  )

  func main() {
      r := gin.Default()
      r.GET("/usuarios", handlers.ListarUsuarios)
      r.Run()
  }
  ```

================================================================================
  19.10 ORGANIZACIÓN DEL CÓDIGO
================================================================================

  Estructura recomendada para proyectos Go:

    mi-proyecto/
      go.mod                    # Definición del módulo
      go.sum                    # Checksums de dependencias
      main.go                   # Punto de entrada (package main)

      internal/                 # Código no exportado (no accesible
        auth/                   # desde fuera del módulo)
        db/
        config/

      pkg/                      # Código reutilizable exportado
        api/
        models/
        utils/

      cmd/                      # Ejecutables (varios mains)
        server/
          main.go
        cli/
          main.go

      api/                      # Definiciones de API
        handlers/
        middleware/

      scripts/                  # Scripts de build/deploy

  El directorio internal/ es especial: Go no permite importar
  paquetes internal/ desde fuera del módulo.

  Consejos de organización:
  - Un paquete debe tener un propósito claro y único.
  - No mezcles responsabilidades en un mismo paquete.
  - Los nombres de paquetes deben ser cortos y descriptivos.
  - Evita nombres genéricos como "utils" o "common".
  - Prefiere nombres concretos: "auth", "db", "api", "models".

================================================================================
  19.11 CICLO DE IMPORTACIÓN
================================================================================

  Go no permite ciclos de importación. Si el paquete A importa a B,
  y B importa a A (directa o indirectamente), el código no compila.

  ```go
  // package users
  // import "proyecto/orders"
  // users.go

  // package orders  ← ESTO CREA UN CICLO
  // import "proyecto/users"
  // orders.go
  ```

  Soluciones:
  1. Extraer tipos compartidos a un tercer paquete.
  2. Usar interfaces en lugar de tipos concretos.
  3. Reorganizar la arquitectura.

  Ejemplo de solución:

  ```go
  // package models (tipos compartidos)
  type Usuario struct { ID int }
  type Orden struct { ID int; UsuarioID int }

  // package users
  // import "proyecto/models"
  // No importa "proyecto/orders"

  // package orders
  // import "proyecto/models"
  // No importa "proyecto/users"
  ```

  Los ciclos de importación son una señal de diseño incorrecto. Si
  los encuentras, es momento de repensar la arquitectura.

================================================================================
  19.12 BUENAS PRÁCTICAS
================================================================================

  1. Nombra los paquetes con nombres cortos y en minúscula.
     Preferiblemente una sola palabra.

  2. No uses nombres genéricos como "utils", "common", "lib".
     Usa nombres descriptivos: "auth", "db", "config".

  3. Exporta solo lo necesario. Mantén la API pública pequeña.

  4. No uses variables de paquete a menos que sea absolutamente
     necesario. Dificultan las pruebas y la razón sobre el código.

  5. Prefiere pasar dependencias explícitamente (inyección) en
     lugar de usar variables globales.

  6. Usa internal/ para ocultar implementación interna.

  7. No abuses de init(). Son difíciles de testear y entender.

  8. Organiza el código por funcionalidad, no por tipo de archivo.
     Ejemplo: auth/ en lugar de models/auth.go, handlers/auth.go.

  9. Los nombres de archivo deben ser descriptivos:
     "user_handler.go" en lugar de "handler1.go".

  10. Usa go mod tidy regularmente para mantener el go.sum limpio.

================================================================================
  19.13 ERRORES COMUNES
================================================================================

  Error 1: Ciclo de importación
  ```go
  // package A importa B, y B importa A → no compila
  ```

  Error 2: Import no usado
  ```go
  import "fmt" // Error si no usas fmt en el archivo
  ```

  Error 3: Confundir exportado con no exportado
  ```go
  // package auth
  func login() {} // No exportada

  // package main
  // import "proyecto/auth"
  // auth.login() // Error: login no exportada
  ```

  Error 4: Dos paquetes en el mismo directorio
  ```go
  // directorio: db/
  // archivo1.go: package db
  // archivo2.go: package database // Error: conflicto
  ```

  Error 5: Shadowing no intencional
  ```go
  x := 10
  if true {
      x := 20 // Shadowing: esta x es diferente
      fmt.Println(x) // 20
  }
  fmt.Println(x) // 10 (fácil de olvidar)
  ```

  Error 6: Usar variables de paquete como "globales" mutables
  ```go
  var db *sql.DB // Mutable, difícil de testear

  func InitDB() { db = ... }
  func GetDB() *sql.DB { return db }
  // Prefiere inyección de dependencias
  ```

  Error 7: Olvidar go.mod
  ```go
  // Sin go.mod, no puedes importar paquetes locales
  ```

  19.14 CÓDIGO PROFESIONAL: EJEMPLOS COMPLETOS
================================================================================

  Ejemplo 1: Estructura de proyecto con múltiples paquetes

  Estructura:
    tienda/
      go.mod
      main.go
      internal/
        auth/
          auth.go
          auth_test.go

          db.go
      pkg/
        models/
          producto.go
          usuario.go
        api/
          handlers.go

  archivo: go.mod
  ```
  module github.com/usuario/tienda

  go 1.21
  ```

  archivo: pkg/models/producto.go
  ```go
  package models

  type Producto struct {
      ID       int
      Nombre   string
      Precio   float64
      Stock    int
      Categoria string
  }

  func (p Producto) ValorStock() float64 {
      return float64(p.Stock) * p.Precio
  }
  ```

  archivo: pkg/models/usuario.go
  ```go
  package models

  import "time"

  type Usuario struct {
      ID         int
      Nombre     string
      Email      string
      Rol        string
      CreatedAt  time.Time
  }

  type UsuarioPublico struct {
      Nombre string
      Email  string
      Rol    string
  }

  func (u Usuario) ToPublico() UsuarioPublico {
      return UsuarioPublico{
          Nombre: u.Nombre,
          Email:  u.Email,
          Rol:    u.Rol,
      }
  }
  ```

  archivo: internal/auth/auth.go
  ```go
  package auth

  import (
      "errors"
      "github.com/usuario/tienda/pkg/models"
  )

  var (
      ErrCredencialesInvalidas = errors.New("credenciales inválidas")
      ErrUsuarioBloqueado      = errors.New("usuario bloqueado")
  )

  tipoUsuario struct {
      Nombre   string
      Password string
      Rol      string
  }

  var usuariosDB = map[string]tipoUsuario{
      "admin": {"Admin", "admin123", "admin"},
      "user":  {"Usuario", "user123", "user"},
  }

  func Login(nombre, password string) (*models.Usuario, error) {
      u, ok := usuariosDB[nombre]
      if !ok {
          return nil, ErrCredencialesInvalidas
      }
      if u.Password != password {
          return nil, ErrCredencialesInvalidas
      }

      return &models.Usuario{
          Nombre: nombre,
          Email:  nombre + "@tienda.com",
          Rol:    u.Rol,
      }, nil
  }

  func Autenticar(token string) (*models.Usuario, error) {
      if token == "" {
          return nil, errors.New("token requerido")
      }
      // Simular autenticación
      return &models.Usuario{Nombre: "anonimo", Rol: "guest"}, nil
  }
  ```

  archivo: pkg/api/handlers.go
  ```go
  package api

  import (
      "encoding/json"
      "net/http"

      "github.com/usuario/tienda/internal/auth"
      "github.com/usuario/tienda/pkg/models"
  )

  type Handler struct {
      Autenticador *auth.Autenticador
  }

  func (h *Handler) Login(w http.ResponseWriter, r *http.Request) {
      var creds struct {
          Nombre   string `json:"nombre"`
          Password string `json:"password"`
      }
      json.NewDecoder(r.Body).Decode(&creds)

      usuario, err := auth.Login(creds.Nombre, creds.Password)
      if err != nil {
          http.Error(w, err.Error(), http.StatusUnauthorized)
          return
      }

      json.NewEncoder(w).Encode(usuario.ToPublico())
  }

  func (h *Handler) ListarProductos(w http.ResponseWriter, r *http.Request) {
      productos := []models.Producto{
          {ID: 1, Nombre: "Laptop", Precio: 1200, Stock: 10, Categoria: "electrónica"},
          {ID: 2, Nombre: "Mouse", Precio: 25, Stock: 50, Categoria: "electrónica"},
      }

      json.NewEncoder(w).Encode(productos)
  }
  ```

      "crypto/rand"
      "fmt"
      "math/big"
      mrand "math/rand"
      "time"
  )

  func main() {
      // crypto/rand (seguro para criptografía)
      n, _ := rand.Int(rand.Reader, big.NewInt(100))
      fmt.Println("Número criptográfico:", n)

      // math/rand (rápido, no criptográfico)

      fmt.Println("Número pseudo-aleatorio:", mrand.Intn(100))
  }

================================================================================
  19.17 PREGUNTAS FRECUENTES (FAQ)
================================================================================

  P: ¿Puedo tener dos archivos con diferente package en el mismo
     directorio?
  R: No. Todos los archivos .go en un directorio deben pertenecer
     al mismo paquete.

  P: ¿Cómo decido si algo debe ser exportado o no?
  R: Exporta solo lo que otros paquetes necesitan usar. Mantén
     todo lo demás no exportado.

  P: ¿Qué son las funciones init()?
  R: Funciones que se ejecutan automáticamente antes de main(),
     en el orden en que aparecen en los archivos. Se usan para
     inicialización.

  P: ¿Puedo tener múltiples funciones init() en el mismo archivo?
  R: Sí, se ejecutan en el orden de declaración. Pero no es
     recomendable tener más de una.

  P: ¿Cómo importo un paquete local sin go.mod?
  R: No puedes. Desde Go 1.16, go.mod es obligatorio incluso para
     proyectos simples.

  P: ¿Qué es internal/?
  R: Un directorio especial. Los paquetes dentro de internal/ solo
     pueden ser importados por código dentro del mismo módulo.

  P: ¿Cómo evito ciclos de importación?
  R: Extrayendo tipos compartidos a un tercer paquete, usando
     interfaces, o rediseñando la arquitectura.

  P: ¿Cuándo debo usar puntero vs valor como receptor?
  R: Usa puntero si el método modifica el receptor. Usa valor si
     no lo modifica. Sé consistente en todo el tipo.
