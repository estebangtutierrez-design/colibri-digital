1. INSTALACIÓN DE DELVE
───────────────────────

Delve se instala como cualquier herramienta Go:

```bash
# Instalación estándar
go install github.com/go-delve/delve/cmd/dlv@latest

# Verificar instalación
dlv version

# Ayuda general
dlv help
```

Delve tiene varios modos de operación:

```bash
# Depurar un programa directamente
dlv debug ./cmd/server/ -- -port=9090

# Ejecutar y adjuntar a proceso existente
dlv exec ./mybinary -- --config=prod.yaml

# Adjuntarse a un proceso en ejecución
dlv attach <PID>

# Modo headless para integración con IDEs
dlv debug --headless --listen=:2345 --api-version=2 ./cmd/server/
```

2. COMANDOS ESENCIALES DE DELVE
─────────────────────────────────

Dentro de la sesión interactiva de Delve:

```
(dlv) help                    # Lista completa de comandos
(dlv) break main.main         # Breakpoint en función
(dlv) break main.go:25        # Breakpoint en archivo:línea
(dlv) cond 1 i == 5           # Breakpoint condicional en #1
(dlv) continue                # Continuar ejecución
(dlv) next                    # Siguiente línea (sin entrar)
(dlv) step                    # Entrar a la función
(dlv) stepout                 # Salir de la función actual
(dlv) print variable          # Ver valor de variable
(dlv) locals                  # Ver variables locales
(dlv) args                    # Ver argumentos
(dlv) goroutines              # Listar goroutines
(dlv) goroutine 5             # Cambiar a goroutine 5
(dlv) stack                   # Ver stack trace
(dlv) list                    # Ver código fuente
(dlv) clear 1                 # Eliminar breakpoint #1
(dlv) clearall                # Eliminar todos los breakpoints
(dlv) restart                 # Reiniciar programa
(dlv) exit                    # Salir
```

3. PROGRAMA DE EJEMPLO PARA DEBUG
──────────────────────────────────

```go
package main

import (
    "fmt"
    "math/rand"
    "sync"
    "time"
)

type Account struct {
    ID      int
    Balance float64
    mu      sync.Mutex
}

func (a *Account) Deposit(amount float64) {
    a.mu.Lock()
    defer a.mu.Unlock()
    a.Balance += amount
}

func (a *Account) Withdraw(amount float64) error {
    a.mu.Lock()
    defer a.mu.Unlock()
    if a.Balance < amount {
        return fmt.Errorf("saldo insuficiente: %.2f < %.2f", a.Balance, amount)
    }
    a.Balance -= amount
    return nil
}

type TransferService struct {
    accounts map[int]*Account
}

func NewTransferService() *TransferService {
    return &TransferService{
        accounts: map[int]*Account{
            1: {ID: 1, Balance: 1000.0},
            2: {ID: 2, Balance: 500.0},
            3: {ID: 3, Balance: 250.0},
        },
    }
}

func (ts *TransferService) Transfer(from, to int, amount float64) error {
    if from == to {
        return fmt.Errorf("no puedes transferirte a ti mismo")
    }
    sender, ok := ts.accounts[from]
    if !ok {
        return fmt.Errorf("cuenta origen %d no encontrada", from)
    }
    receiver, ok := ts.accounts[to]
    if !ok {
        return fmt.Errorf("cuenta destino %d no encontrada", to)
    }
    // Deadlock potencial: orden de locks inconsistente
    if from < to {
        sender.mu.Lock()
        receiver.mu.Lock()
    } else {
        receiver.mu.Lock()
        sender.mu.Lock()
    }
    defer sender.mu.Unlock()
    defer receiver.mu.Unlock()
    if sender.Balance < amount {
        return fmt.Errorf("saldo insuficiente en cuenta %d", from)
    }
    sender.Balance -= amount
    receiver.Balance += amount
    return nil
}

func (ts *TransferService) GetBalance(id int) float64 {
    if acc, ok := ts.accounts[id]; ok {
        acc.mu.Lock()
        defer acc.mu.Unlock()
        return acc.Balance
    }
    return 0
}

func main() {
    svc := NewTransferService()
    var wg sync.WaitGroup

    // Simular transferencias concurrentes
    for i := 0; i < 10; i++ {
        wg.Add(1)
        go func(id int) {
            defer wg.Done()
            from := rand.Intn(3) + 1
            to := rand.Intn(3) + 1
            amount := float64(rand.Intn(200) + 10)
            err := svc.Transfer(from, to, amount)
            if err != nil {
                fmt.Printf("[%d] Error: %v\n", id, err)
            } else {
                fmt.Printf("[%d] Transferencia %.2f de %d a %d OK\n",
                    id, amount, from, to)
            }
        }(i)
    }

    wg.Wait()
    fmt.Println("Saldos finales:")
    for id := 1; id <= 3; id++ {
        fmt.Printf("  Cuenta %d: %.2f\n", id, svc.GetBalance(id))
    }
}
```

5. DEBUGGING DE CONCURRENCIA — DEADLOCKS
─────────────────────────────────────────

El programa de ejemplo tiene un potencial deadlock. Veamos cómo
detectarlo con Delve.

Si el programa se queda colgado, podemos:

1. Pausar con Ctrl+C.
2. Inspeccionar goroutines.

```
(dlv) goroutines -t
* Goroutine 1 - runtime.gopark (sync.Mutex.Lock)
  Goroutine 5 - runtime.gopark (sync.Mutex.Lock)
  Goroutine 6 - runtime.gopark (sync.Mutex.Lock)
```

Ver dónde está bloqueada cada goroutine:

```
(dlv) goroutine 5
(dlv) stack
0  0x0000000000457a1c in runtime.gopark
    at /usr/local/go/src/runtime/proc.go:382
1  0x0000000000457b9e in runtime.goparkunlock
    at /usr/local/go/src/runtime/proc.go:387
2  0x0000000000458a45 in runtime.semacquire1
    at /usr/local/go/src/runtime/sema.go:147
3  0x0000000000469b32 in sync.runtime_SemacquireMutex
    at /usr/local/go/src/runtime/sema.go:71
4  0x000000000046a2b0 in sync.(*Mutex).lockSlow
    at /usr/local/go/src/sync/mutex.go:138
5  0x000000000046a150 in sync.(*Mutex).Lock
    at /usr/local/go/src/sync/mutex.go:81
6  0x00000000004a2c15 in main.(*TransferService).Transfer
    at ./main.go:57
```

¡La línea 57 es sender.mu.Lock()! Esto indica que las goroutines
están esperando locks en orden inverso.

Solución: siempre adquirir locks en el mismo orden global.

7. DEBUGGING REMOTO
────────────────────

Delve puede operar en modo headless para depurar contenedores o
servidores remotos.

En el servidor (o contenedor):

```bash
# Compilar sin optimizaciones para mejor debugging
go build -gcflags='all=-N -l' -o /app/server ./cmd/server/

# Iniciar Delve en modo headless
dlv exec --headless --listen=:2345 --api-version=2 --accept-multiclient ./app/server
```

En la máquina local:

```bash
# Conectar al servidor remoto
dlv connect remote-host:2345
```

O desde VS Code, configurar launch.json:

```json
{
    "version": "0.2.0",
    "configurations": [
        {
            "name": "Remote Debug",
            "type": "go",
            "request": "attach",
            "mode": "remote",
            "remotePath": "/app",
            "port": 2345,
            "host": "192.168.1.100"
        }
    ]
}
```

9. TRACE — DEBUGGING SIN DEPURADOR
───────────────────────────────────

A veces no puedes usar Delve (producción). Usa la función
runtime/trace y el paquete trace:

```go
package main

import (
    "log"
    "os"
    "runtime/trace"
    "sync"
)

func main() {
    f, err := os.Create("trace.out")
    if err != nil {
        log.Fatal(err)
    }
    defer f.Close()

    if err := trace.Start(f); err != nil {
        log.Fatal(err)
    }
    defer trace.Stop()

    var wg sync.WaitGroup
    for i := 0; i < 10; i++ {
        wg.Add(1)
        go func(n int) {
            defer wg.Done()
            // Simular trabajo
            sum := 0
            for j := 0; j < 1000000; j++ {
                sum += j
            }
            _ = sum
        }(i)
    }
    wg.Wait()
}
```

Ejecutar:

```bash
go run main.go
go tool trace trace.out
```

Esto abre un navegador con análisis visual de goroutines, eventos,
y tiempo de ejecución.

11. DEBUGGING DE PÁNICOS
────────────────────────

Cuando un programa entra en pánico, podemos capturar el stack trace:

```go
func safeCall(fn func()) {
    defer func() {
        if r := recover(); r != nil {
            debug.PrintStack()
            log.Printf("pánico recuperado: %v", r)
        }
    }()
    fn()
}
```

Con Delve, podemos interrumpir antes del pánico:

```
(dlv) break runtime.gopanic
(dlv) continue
```

12. DEBUGGING DE FUGA DE MEMORIA
─────────────────────────────────

Programa con fuga de memoria (goroutines que nunca terminan):

```go
package main

import (
    "fmt"
    "net/http"
    _ "net/http/pprof"
    "time"
)

func leakyFunction() {
    ch := make(chan int)
    go func() {
        // Esta goroutine nunca termina porque nadie escribe en ch
        val := <-ch
        fmt.Println(val)
    }()
}

func main() {
    go func() {
        http.ListenAndServe(":6060", nil)
    }()

    for i := 0; i < 100; i++ {
        leakyFunction()
    }

    time.Sleep(5 * time.Second)
    fmt.Println("Revisa http://localhost:6060/debug/pprof/goroutine")
    select {}
}
```

Con Delve:

```
(dlv) break leakyFunction
(dlv) continue
(dlv) goroutines
(dlv) goroutine 5
(dlv) stack
```

```go
// debug_script.go
package main

import (
    "fmt"
    "os"
    "os/exec"
)

func runDelveScript() {
    commands := []string{
        "break main.main",
        "break main.(*TransferService).Transfer",
        "cond 2 amount > 100",
        "continue",
        "print from",
        "print to",
        "print amount",
        "continue",
        "exit",
    }

    cmd := exec.Command("dlv", "debug", "./cmd/server/")
    stdin, _ := cmd.StdinPipe()
    stdout, _ := cmd.StdoutPipe()
    cmd.Start()

    for _, c := range commands {
        fmt.Fprintln(stdin, c)
    }

    output := make([]byte, 1024)
    stdout.Read(output)
    fmt.Println(string(output))
    cmd.Wait()
}
```

17. DEBUGGING DE BASES DE DATOS
────────────────────────────────

```go
package repository

import (
    "context"
    "database/sql"
    "fmt"
    "log"
    "time"
)

type QueryLogger struct {
    db *sql.DB
}

func NewQueryLogger(db *sql.DB) *QueryLogger {
    return &QueryLogger{db: db}
}

func (ql *QueryLogger) QueryContext(ctx context.Context, query string, args ...interface{}) (*sql.Rows, error) {
    start := time.Now()
    rows, err := ql.db.QueryContext(ctx, query, args...)
    duration := time.Since(start)
    log.Printf("[DB] %s | args=%v | duration=%s | err=%v",
        query, args, duration, err)
    return rows, err
}

func (ql *QueryLogger) ExecContext(ctx context.Context, query string, args ...interface{}) (sql.Result, error) {
    start := time.Now()
    result, err := ql.db.ExecContext(ctx, query, args...)
    duration := time.Since(start)
    log.Printf("[DB] %s | args=%v | duration=%s | err=%v",
        query, args, duration, err)
    return result, err
}
```

19. LOGGING ESTRATÉGICO PARA DEBUG
──────────────────────────────────

```go
package debug

import (
    "fmt"
    "log"
    "runtime"
    "strings"
)

func Trace() {
    pc, file, line, ok := runtime.Caller(1)
    if !ok {
        return
    }
    fn := runtime.FuncForPC(pc)
    funcName := fn.Name()
    parts := strings.Split(funcName, ".")
    shortName := parts[len(parts)-1]
    log.Printf("[TRACE] %s:%d %s", file, line, shortName)
}

func Tracef(format string, args ...interface{}) {
    pc, file, line, ok := runtime.Caller(1)
    if !ok {
        return
    }
    fn := runtime.FuncForPC(pc)
    funcName := fn.Name()
    parts := strings.Split(funcName, ".")
    shortName := parts[len(parts)-1]
    msg := fmt.Sprintf(format, args...)
    log.Printf("[TRACE] %s:%d %s: %s", file, line, shortName, msg)
}
```

20. DEBUG PROFESIONAL — CHECKLIST
──────────────────────────────────

Cuando debuggees un problema en producción:

[ ] 1. Reproducir el problema en entorno controlado.
[ ] 2. Revisar logs en busca de patrones.
[ ] 3. Usar pprof para identificar cuellos de botella.
[ ] 4. Verificar goroutines colgadas con Delve.
[ ] 5. Revisar data races con -race.
[ ] 6. Aislar el componente sospechoso.
[ ] 7. Escribir un test que reproduzca el bug.
[ ] 8. Aplicar fix y verificar que el test pasa.
[ ] 9. Monitorear en producción después del deploy.
[ ] 10. Documentar la causa raíz y la solución.

PREGUNTAS FRECUENTES (FAQ)
──────────────────────────

P: ¿Delve funciona con Go 1.22+?
R: Sí, Delve se actualiza constantemente. Usa la última versión.

P: ¿Puedo debuggear un binario compilado sin -gcflags?
R: Sí, pero algunas optimizaciones pueden dificultar el debugging.
   Las variables pueden estar optimizadas y no ser visibles.

P: ¿Cómo debuggeo un programa que solo falla en producción?
R: Usa dlv exec en el binario de producción o configura Delve
   headless en el contenedor. Alternativamente, usa logs
   estratégicos y pprof.

P: ¿Delve funciona con Go modules?
R: Sí, Delve soporta Go modules nativamente.

P: ¿Cómo debuggeo un programa que usa cgo?
R: Delve tiene soporte limitado para cgo. Puedes depurar la parte
   Go, pero el código C debes depurarlo con GDB.

Construye un pequeño servidor web concurrente que sirva endpoints
con las siguientes características:

1. Endpoint /api/process que recibe un JSON y lanza N goroutines
   para procesarlo en paralelo.

b) Explicar la causa raíz con el stack trace.
c) Proponer la solución con código.
d) Verificar que la solución funciona ejecutando con -race.
