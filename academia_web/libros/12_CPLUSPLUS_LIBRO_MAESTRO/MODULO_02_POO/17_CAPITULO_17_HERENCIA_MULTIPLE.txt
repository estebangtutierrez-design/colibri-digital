  Capitulo 17: HERENCIA MULTIPLE
================================================================================

  "La herencia multiple no es un lujo: es una responsabilidad.
   Con grandes poderes vienen grandes responsabilidades."

================================================================================
  BIENVENIDA
================================================================================

  Bienvenido al Capitulo 17 de C++ Master Book. A diferencia de lenguajes
  como Java o C# (que solo permiten herencia simple con interfaces), C++
  permite que una clase herede de multiples clases base simultaneamente.

  La herencia multiple es poderosa pero peligrosa. Permite combinar
  comportamientos de varias clases, pero introduce complejidades como
  el problema del diamante, conflictos de nombres y mayor complejidad
  en la construccion/destruccion.

  En este capitulo aprenderemos a usar la herencia multiple de forma
  segura, a resolver el problema del diamante con herencia virtual,
  y a disenar jerarquias multiples coherentes.

================================================================================
  FRASE DEL CAPÍTULO
================================================================================

  "La herencia multiple es como el fuego: util cuando se controla,
   devastadora cuando se descuida."

================================================================================
  OBJETIVOS DEL CAPÍTULO
================================================================================

  Comprender y aplicar la herencia multiple en C++.
  Identificar y resolver el problema del diamante.
  Usar herencia virtual para compartir una unica instancia base.
  Conocer el orden de construccion con herencia multiple.
  Manejar conflictos de nombres entre bases.
  Usar upcasting con herencia multiple.
  Entender cuando usar herencia multiple vs composicion.

  ------------------------------------------------------------------
  1. SINTAXIS DE HERENCIA MULTIPLE
  ------------------------------------------------------------------

  class Derivada : public Base1, public Base2, ... {
      // ...
  };

  Ejemplo:

  class Printable {
  public:
      virtual void imprimir() const = 0;
      virtual ~Printable() = default;
  };

  class Serializable {
  public:
      virtual std::string serializar() const = 0;
      virtual ~Serializable() = default;
  };

  class Documento : public Printable, public Serializable {
  private:
      std::string contenido_;
  public:
      Documento(const std::string& c) : contenido_(c) {}
      void imprimir() const override {
          std::cout << contenido_ << std::endl;
      }
      std::string serializar() const override {
          return "{ \"contenido\": \"" + contenido_ + "\" }";
      }
  };

  Documento doc("Hola Mundo");
  doc.imprimir();
  std::cout << doc.serializar() << std::endl;

  ------------------------------------------------------------------
  2. PROBLEMA DEL DIAMANTE
  ------------------------------------------------------------------

  Ocurre cuando una clase hereda de dos clases que a su vez heredan
  de una misma clase base. La jerarquia forma un diamante:

        Base
       /    \
    DerA    DerB
       \    /
       DerC

  Sin herencia virtual, DerC contiene DOS copias de Base, lo que
  causa ambiguedad.

  class Base {
  public:
      int valor_;
  };

  class DerA : public Base { };
  class DerB : public Base { };

  class DerC : public DerA, public DerB { };

  DerC obj;
  // obj.valor_ = 5;           // ERROR: ambiguo
  obj.DerA::valor_ = 5;         // OK, pero hay dos copias
  obj.DerB::valor_ = 10;        // otra copia independiente

  sizeof(DerC) incluye dos copias de Base (mas los miembros).

  ------------------------------------------------------------------
  3. HERENCIA VIRTUAL
  ------------------------------------------------------------------

  La herencia virtual resuelve el problema del diamante, asegurando
  que solo haya UNA copia de la clase base compartida.

  class Base {
  public:
      int valor_;
  };

  class DerA : virtual public Base { };
  class DerB : virtual public Base { };

  class DerC : public DerA, public DerB { };

  DerC obj;
  obj.valor_ = 5;   // OK: solo una copia
  obj.DerA::valor_ = 10;  // misma copia
  obj.DerB::valor_ = 15;  // misma copia

  // obj.valor_ = 15 (el ultimo asignado)

  ------------------------------------------------------------------
  4. ORDEN DE CONSTRUCCION CON HERENCIA VIRTUAL
  ------------------------------------------------------------------

  Con herencia virtual, el orden de construccion es:
  1. Clases base virtuales (en orden de izquierda a derecha, en
     profundidad primero).
  2. Clases base no virtuales.
  3. Miembros.
  4. Cuerpo del constructor.

  class Base {
  public:
      Base(int v) : valor_(v) {
          std::cout << "Base(" << v << ")\n";
      }
      int valor_;
  };

  class DerA : virtual public Base {
  public:
      DerA() : Base(1) {
          std::cout << "DerA()\n";
      }
  };

  class DerB : virtual public Base {
  public:
      DerB() : Base(2) {
          std::cout << "DerB()\n";
      }
  };

  class DerC : public DerA, public DerB {
  public:
      // La clase mas derivada inicializa la base virtual
      DerC() : Base(3), DerA(), DerB() {
          std::cout << "DerC()\n";
      }
  };

  // Salida: Base(3) DerA() DerB() DerC()

  REGLA IMPORTANTE: la clase mas derivada es responsable de
  inicializar las bases virtuales, NO las clases intermedias.

  ------------------------------------------------------------------
  5. CONFLICTOS DE NOMBRES
  ------------------------------------------------------------------

  Cuando dos clases base tienen funciones o tipos con el mismo nombre:

  class InterfaceA {
  public:
      void ejecutar() { std::cout << "A ejecutando\n"; }
  };

  class InterfaceB {
  public:
      void ejecutar() { std::cout << "B ejecutando\n"; }
  };

  class Implementacion : public InterfaceA, public InterfaceB {
  public:
      // Resolucion explicita
      void ejecutarA() { InterfaceA::ejecutar(); }
      void ejecutarB() { InterfaceB::ejecutar(); }

      // O redefinir:
      void ejecutar() override {
          InterfaceA::ejecutar();
          InterfaceB::ejecutar();
      }
  };

  Implementacion impl;
  // impl.ejecutar();  // ERROR: ambiguo
  impl.InterfaceA::ejecutar();  // OK
  impl.InterfaceB::ejecutar();  // OK
  impl.ejecutarA();
  impl.ejecutarB();

  ------------------------------------------------------------------
  6. UPCASTING CON HERENCIA MULTIPLE
  ------------------------------------------------------------------

  El upcasting a cualquiera de las bases funciona:

  class BaseA { public: virtual ~BaseA() = default; };
  class BaseB { public: virtual ~BaseB() = default; };
  class Derivada : public BaseA, public BaseB { };

  Derivada d;
  BaseA* ptrA = &d;  // OK
  BaseB* ptrB = &d;  // OK (el puntero se ajusta automaticamente)

  static_cast<BaseB*>(&d);   // OK, ajusta puntero
  dynamic_cast<BaseB*>(&d);  // OK con RTTI

  El ajuste de puntero es necesario porque las subobjetos BaseA y
  BaseB estan en diferentes direcciones dentro del objeto Derivada.

  ------------------------------------------------------------------
  7. DESTRUCTORES EN HERENCIA MULTIPLE
  ------------------------------------------------------------------

  Todos los destructores de las bases deben ser virtuales si se usa
  polimorfismo.

  class BaseA {
  public:
      virtual ~BaseA() { std::cout << "~BaseA()\n"; }
  };

  class BaseB {
  public:
      virtual ~BaseB() { std::cout << "~BaseB()\n"; }
  };

  class Derivada : public BaseA, public BaseB {
  public:
      ~Derivada() override { std::cout << "~Derivada()\n"; }
  };

  // Orden de destruccion: ~Derivada() -> ~BaseB() -> ~BaseA()
  // (orden inverso a la declaracion de herencia)

  ------------------------------------------------------------------
  8. CUANDO USAR HERENCIA MULTIPLE
  ------------------------------------------------------------------

  Casos de uso apropiados:
  - Mixins: pequenas clases que anaden comportamiento reutilizable
    (ej: Printable, Serializable, Clonable, Comparable).
  - Separacion de interfaces: una clase puede implementar multiples
    interfaces (como en Java).
  - Combinacion de funcionalidades independientes.

  Cuando NO usar:
  - Cuando las bases comparten estado o tienen logica conflictiva.
  - Cuando se puede lograr con composicion.
  - Cuando la relacion no es claramente "es-un" para cada base.

  ------------------------------------------------------------------
  9. MIXINS CON HERENCIA MULTIPLE
  ------------------------------------------------------------------

  Los mixins son clases pequenas que anaden una funcionalidad
  especifica a traves de herencia multiple.

  class Dibujable {
  public:
      virtual void dibujar() const = 0;
      virtual ~Dibujable() = default;
  };

  class Redimensionable {
  public:
      virtual void redimensionar(double factor) = 0;
      virtual ~Redimensionable() = default;
  };

  class Coloreable {
  public:
      virtual void setColor(const std::string& color) = 0;
      virtual std::string getColor() const = 0;
      virtual ~Coloreable() = default;
  };

  class Boton : public Dibujable, public Redimensionable,
                public Coloreable {
  private:
      std::string color_;
      double ancho_, alto_;
  public:
      Boton() : color_("gris"), ancho_(100), alto_(50) {}

      void dibujar() const override {
          std::cout << "Dibujando boton " << color_
                    << " de " << ancho_ << "x" << alto_ << "\n";
      }

      void redimensionar(double factor) override {
          ancho_ *= factor;
          alto_ *= factor;
      }

      void setColor(const std::string& c) override { color_ = c; }
      std::string getColor() const override { return color_; }
  };

  ------------------------------------------------------------------
  10. HERENCIA MULTIPLE VS COMPOSICION
  ------------------------------------------------------------------

  A veces es mejor usar composicion interna de objetos que heredar:

  // Herencia multiple (si es "es-un" para cada uno):
  class Coche : public Vehiculo, public Asegurable { };

  // Composicion (si es "tiene-un"):
  class Coche {
  private:
      Vehiculo vehiculo_;
      PolizaSeguro seguro_;
  };

  La regla es: si la relacion es "es-un" para CADA base y las bases
  no comparten estado, la herencia multiple puede ser adecuada.

  ------------------------------------------------------------------
  11. USANDO USING CON HERENCIA MULTIPLE
  ------------------------------------------------------------------

  using puede resolver conflictos trayendo funciones especificas al
  ambito de la clase derivada.

  class A {
  public:
      void func(int) { std::cout << "A::func(int)\n"; }
      void func(double) { std::cout << "A::func(double)\n"; }
  };

  class B {
  public:
      void func(const std::string&) {
          std::cout << "B::func(string)\n";
      }
  };

  class C : public A, public B {
  public:
      using A::func;  // trae las versiones de A
      using B::func;  // trae las versiones de B

      void func() {  // propia de C
          std::cout << "C::func()\n";
      }
  };

  C obj;
  obj.func(5);          // A::func(int)
  obj.func(3.14);       // A::func(double)
  obj.func("texto");    // B::func(string)
  obj.func();           // C::func()

  12. DIAMANTE CON DATOS MIEMBRO
  ------------------------------------------------------------------

  class Persona {
  protected:
      std::string nombre_;
  public:
      Persona(const std::string& n) : nombre_(n) {}
      virtual void presentar() const {
          std::cout << "Soy " << nombre_ << "\n";
      }
  };

  class Estudiante : virtual public Persona {
  public:
      Estudiante(const std::string& n) : Persona(n) {}
  };

  class Trabajador : virtual public Persona {
  public:
      Trabajador(const std::string& n) : Persona(n) {}
  };

  class Becario : public Estudiante, public Trabajador {
  public:
      Becario(const std::string& n)
          : Persona(n), Estudiante(n), Trabajador(n) {}
      // ^^ La mas derivada inicializa la base virtual
  };

------------[ archivo: informe.cpp ]------------
#include "informe.hpp"

Informe::Informe(const std::string& titulo) : titulo_(titulo) {}

Informe::Informe(const Informe& otro)
    : titulo_(otro.titulo_), secciones_(otro.secciones_) {}

void Informe::agregarSeccion(const std::string& seccion) {
    secciones_.push_back(seccion);
}

void Informe::setTitulo(const std::string& titulo) { titulo_ = titulo; }

void Informe::imprimir() const {
    std::cout << "=== INFORME: " << titulo_ << " ===\n";
    for (size_t i = 0; i < secciones_.size(); ++i) {
        std::cout << "  " << (i + 1) << ". " << secciones_[i] << "\n";
    }
    std::cout << "========================\n";
}

std::string Informe::exportar() const {
    std::string resultado = "TITULO: " + titulo_ + "\n";
    for (const auto& s : secciones_) {
        resultado += "SECCION: " + s + "\n";
    }
    return resultado;
}

void Informe::guardar(const std::string& archivo) const {
    std::ofstream out(archivo);
    if (out.is_open()) {
        out << exportar();
        out.close();
        std::cout << "Informe guardado en " << archivo << "\n";
    }
}

void Informe::cargar(const std::string& archivo) {
    std::ifstream in(archivo);
    if (in.is_open()) {
        secciones_.clear();
        std::string linea;
        while (std::getline(in, linea)) {
            if (linea.substr(0, 8) == "SECCION: ") {
                secciones_.push_back(linea.substr(8));
            }
        }
        in.close();
        std::cout << "Informe cargado desde " << archivo << "\n";
    }
}

#ifndef DIAMANTE_HPP
#define DIAMANTE_HPP

#include <iostream>
#include <string>

class Animal {
public:
    Animal(const std::string& nombre) : nombre_(nombre) {
        std::cout << "Animal(" << nombre_ << ")\n";
    }
    virtual void sonido() const = 0;
    std::string getNombre() const { return nombre_; }
    virtual ~Animal() = default;
protected:
    std::string nombre_;
};

class Volador : virtual public Animal {
public:
    Volador(const std::string& nombre) : Animal(nombre) {
        std::cout << "Volador(" << nombre_ << ")\n";
    }
    virtual void volar() const {
        std::cout << nombre_ << " esta volando\n";
    }
};

class Nadador : virtual public Animal {
public:
    Nadador(const std::string& nombre) : Animal(nombre) {
        std::cout << "Nadador(" << nombre_ << ")\n";
    }
    virtual void nadar() const {
        std::cout << nombre_ << " esta nadando\n";
    }
};

class Pato : public Volador, public Nadador {
public:
    Pato(const std::string& nombre)
        : Animal(nombre), Volador(nombre), Nadador(nombre) {
        std::cout << "Pato(" << nombre_ << ")\n";
    }

    void sonido() const override {
        std::cout << nombre_ << " dice: Cuac!\n";
    }
};

#endif

    std::cout << "\nComo Exportable:\n";
    Exportable* exp = &informe;
    std::cout << exp->exportar() << std::endl;

    std::cout << "\nComo Almacenable:\n";
    Almacenable* alm = &informe;
    alm->guardar("informe.txt");

    std::cout << "\n=== PROBLEMA DEL DIAMANTE RESUELTO ===\n";
    Pato donald("Donald");
    donald.sonido();
    donald.volar();
    donald.nadar();

    std::cout << "\nUpcasting multiple:\n";
    Animal* ptrAnimal = &donald;
    ptrAnimal->sonido();

    Volador* ptrVolador = &donald;
    ptrVolador->volar();

    Nadador* ptrNadador = &donald;
    ptrNadador->nadar();

    std::cout << "\nUnica instancia de Animal (nombre): "
              << donald.getNombre() << std::endl;

    // Verificar que hay una sola copia
    std::cout << "Direccion via Volador: "
              << static_cast<void*>(static_cast<Volador*>(&donald)) << std::endl;
    std::cout << "Direccion via Nadador: "
              << static_cast<void*>(static_cast<Nadador*>(&donald)) << std::endl;

    std::cout << "\n=== FIN ===\n";
    return 0;

==========================================================================
  RESUMEN DEL CAPÍTULO
==========================================================================

  Herencia multiple: una clase deriva de varias bases.
  Sintaxis: class D : public A, public B, ... { };
  Problema del diamante: dos copias de la base compartida.
  Herencia virtual (virtual public): una sola copia de la base.
  La clase mas derivada inicializa las bases virtuales.
  Conflictos de nombres: resolver con NombreBase::miembro o using.
  Upcasting a cualquier base funciona (ajuste de puntero automatico).
  Usar herencia multiple para mixins e interfaces.
  Si se puede, preferir composicion sobre herencia multiple.

==========================================================================
  PREGUNTAS FRECUENTES (FAQ)
==========================================================================

  P: Java y C# no tienen herencia multiple. Por que C++ si?
  R: C++ busca eficiencia y flexibilidad maximas. La herencia multiple
     permite mixins y composicion de comportamientos sin overhead de
     interfaces (como en Java). Pero requiere mas cuidado del
     programador.

  P: Cual es el costo de la herencia virtual?
  R: Overhead en memoria (vtable pointer adicional) y tiempo de
     ejecucion (pequeno). Es aceptable para la mayoria de casos.

  P: Como se cual clase base virtual inicializar?
  R: La clase mas derivada inicializa todas las bases virtuales. Las
     clases intermedias pueden tener inicializaciones que seran
     ignoradas.

  P: Puedo tener herencia virtual y no-virtual de la misma clase?
  R: Si, pero es una pesadilla. Evitalo. Si una clase es base virtual,
     todas las herencias deben ser virtuales.

  P: Cual es la alternativa a herencia multiple?
  R: Composicion y patrones de diseno (Strategy, Decorator, etc.)
     pueden lograr resultados similares sin la complejidad.

  P: Que pasa con sizeof() en herencia multiple?
  R: El objeto contiene subobjetos para cada base. Con herencia
     virtual, hay un puntero adicional (vptr o similar) para acceder
     a la base compartida.

  CIERRE DEL CAPÍTULO
==========================================================================

  La herencia multiple es una de las caracteristicas mas polemicas de
  C++. Bien usada, permite disenos elegantes con mixins e interfaces.
  Mal usada, produce jerarquias confusas y fragiles.

  Recuerda las reglas de oro: herencia virtual para resolver el diamante,
  la clase mas derivada inicializa las bases virtuales, y prefiere
  composicion cuando la relacion no sea claramente "es-un" para cada
  base.

  fundamental que permite a los objetos referenciarse a si mismos.
