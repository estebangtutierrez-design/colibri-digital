► 1. ¿QUÉ ES EL ANÁLISIS ESTÁTICO?

  El análisis estático examina el código fuente sin ejecutarlo,
  buscando patrones que indican posibles bugs, malas prácticas
  o violaciones de estilo.

  TIPOS DE ANÁLISIS:
  • Sintáctico: Errores de sintaxis (el compilador ya lo hace).
  • Semántico: Uso incorrecto del lenguaje (variables sin init).
  • De flujo: Valores que pueden llevar a comportamientos indefinidos.
  • De estilo: Violaciones de guías de estilo (Google, LLVM).
  • De seguridad: Vulnerabilidades potenciales (buffer overflow).

  vs. ANÁLISIS DINÁMICO (Valgrind, Sanitizers):
  • Estático: Antes de ejecutar, cubre todos los paths.
  • Dinámico: Durante la ejecución, solo cubre paths recorridos.

  IDEAL: Ambas herramientas combinadas.

► 2. WARNINGS DEL COMPILADOR — LA PRIMERA LÍNEA DE DEFENSA

  Los warnings del compilador son la forma más básica (y gratuita)
  de análisis estático. Gcc y Clang tienen décadas de experiencia
  en detectar código sospechoso.

*/

/*
  // Compilación con TODOS los warnings
  g++ -Wall -Wextra -Wpedantic -Werror -Wshadow
      -Wnon-virtual-dtor -Wold-style-cast -Wcast-align
      -Wunused -Woverloaded-virtual -Wconversion
      -Wsign-conversion -Wmisleading-indentation
      -Wduplicated-cond -Wduplicated-branches
      -Wlogical-op -Wnull-dereference -Wuseless-cast
      -Wformat=2 -Wno-unused-parameter

  FLAGS ESENCIALES:
  • -Wall          → Activa warnings comunes.
  • -Wextra        → Más warnings (no todos, el nombre engaña).
  • -Wpedantic     → Rechaza extensiones no estándar.
  • -Werror        → Trata warnings como errores (obligatorio en CI).
  • -Wshadow       → Variable shadows otra en ámbito externo.
  • -Wconversion   → Pérdida de datos en conversiones implícitas.
*/

// Ejemplos de código que activa warnings:

void warning_shadow() {
    int x = 10;
    for (int i = 0; i < 5; ++i) {
        int x = 20;  // -Wshadow: x oculta la variable externa
        std::cout << x;
    }
}

void warning_conversion() {
    int grande = 100000;
    short chico = grande;  // -Wconversion: posible pérdida de datos

    double d = 3.14159;
    int truncado = d;  // -Wconversion: truncamiento

    // Corrección: cast explícito
    short seguro = static_cast<short>(grande);
}

void warning_no_virtual_dtor() {
    // Base sin destructor virtual
    struct Base {
        ~Base() {}  // -Wnon-virtual-dtor si se usa polimórficamente
    };
    struct Derivada : Base {
        ~Derivada() {}
    };
    Base* ptr = new Derivada();
    delete ptr;  // Undefined behavior: no llama ~Derivada()
}

//==============================================================================

/*
  cppcheck es un analizador estático específico para C/C++ que
  detecta bugs que el compilador no ve.

  INSTALACIÓN:
  sudo apt install cppcheck        # Ubuntu/Debian
  sudo pacman -S cppcheck          # Arch
  brew install cppcheck            # macOS
  winget install cppcheck          # Windows

  USO BÁSICO:
  cppcheck --enable=all --suppress=missingIncludeSystem .
  cppcheck --enable=warning --error-exitcode=1 src/
  cppcheck --enable=performance --enable=portability src/
  cppcheck --enable=style --suppress=*:test/* src/

  NIVELES DE GRAVEDAD:
  • error        → Bug definitivo (buffer overflow, null pointer)
  • warning      → Bug probable (memory leak, recurso no liberado)
  • style        → Código no idiomático (funciones muy largas)
  • performance  → Código lento (copias innecesarias)
  • portability  → Problemas entre plataformas (tamaño de tipos)
  • information  → Observaciones generales
*/

// Código con bugs que cppcheck detectaría:

class GestorRecursos {
public:
    GestorRecursos() {
        // cppcheck detecta: asignación en constructor sin init list
        archivo_ = fopen("datos.txt", "r");
        buffer_ = new char[1024];
        // cppcheck: recurso adquirido pero no liberado en destructor
    }

    // Error: Falta destructor ~GestorRecursos()
    // cppcheck: class 'GestorRecursos' does not have a destructor

    void leer() {
        if (archivo_) {
            // cppcheck: no verificar fgets return
            fgets(buffer_, 1024, archivo_);
        }
    }

    // Error: Se permite copia pero hay recursos raw
    // cppcheck: class has copy constructor but no operator=

private:
    FILE* archivo_;
    char* buffer_;
};

// Corregido con RAII:

class GestorRecursosRAII {
public:
    GestorRecursosRAII()
        : archivo_(fopen("datos.txt", "r"))
        , buffer_(std::make_unique<char[]>(1024)) {
        if (!archivo_) {
            throw std::runtime_error("No se pudo abrir archivo");
        }
    }

    ~GestorRecursosRAII() {
        if (archivo_) fclose(archivo_);
    }

    // No copiable (fstream no se copia)
    GestorRecursosRAII(const GestorRecursosRAII&) = delete;
    GestorRecursosRAII& operator=(const GestorRecursosRAII&) = delete;

    // Movible
    GestorRecursosRAII(GestorRecursosRAII&&) noexcept = default;
    GestorRecursosRAII& operator=(GestorRecursosRAII&&) noexcept = default;

    void leer() {
        if (archivo_ && buffer_) {
            if (!fgets(buffer_.get(), 1024, archivo_)) {
                throw std::runtime_error("Error de lectura");
            }
        }
    }

private:
    FILE* archivo_;
    std::unique_ptr<char[]> buffer_;
};

// --- Buffer overflow ---
void cppcheck_buffer_overflow() {
    char destino[10];
    strcpy(destino, "Este texto es demasiado largo"); // cppcheck: buffer overflow
    // Corrección: strncpy(destino, "texto", sizeof(destino)-1);
}

// --- Null pointer dereference ---
void cppcheck_null_ptr(int* ptr) {
    if (ptr == nullptr) {
        *ptr = 42; // cppcheck: null pointer dereference
    }
}

// --- Memory leak ---
void cppcheck_memory_leak() {
    int* datos = new int[100];
    if (datos[0] == 0) {
        return; // cppcheck: memory leak, no se libera datos
    }
    delete[] datos;
}

// --- Use after free ---
void cppcheck_use_after_free() {
    int* ptr = new int(42);
    delete ptr;
    *ptr = 10; // cppcheck: using pointer after free
}

// --- Variable no inicializada ---
void cppcheck_uninit() {
    int x;
    if (x == 0) { // cppcheck: uninitialized variable
        std::cout << "cero\n";
    }
}

// --- Doble liberación ---
void cppcheck_double_free() {
    int* ptr = new int(10);
    delete ptr;
    delete ptr; // cppcheck: double free
}

// --- División por cero ---
int cppcheck_div_cero(int a, int b) {
    if (b == 0) {
        return a / b; // cppcheck: division by zero (ya en el if!)
    }
    return a / b;
}

// --- Asignación en condicional ---
void cppcheck_assign_in_cond() {
    int x;
    if (x = 5) { // cppcheck: assignment in condition, probablemente ==
        std::cout << "x es 5\n";
    }
}

// --- Stray semicolon ---
void cppcheck_stray_semicolon() {
    for (int i = 0; i < 10; ++i); // cppcheck: stray semicolon
    {
        std::cout << "Esto se ejecuta una vez!\n";
    }
}

    modernize-use-auto: true
    modernize-use-nullptr: true
    modernize-use-override: true
    performance-unnecessary-value-param: true

// Código que clang-tidy mejoraría:

class mal_estilo {
public:
    int valor;          // clang-tidy: member sin sufijo _
    int getValor() {    // clang-tidy: función debería ser const
        return valor;
    }

    // clang-tidy: usar nullptr en lugar de NULL
    mal_estilo() : ptr(NULL) {}

    // clang-tidy: usar override
    virtual ~mal_estilo() {}

    // clang-tidy: parámetro por copia innecesario
    void setNombre(std::string nombre) {
        nombre_ = nombre;
    }

private:
    std::string nombre_;
    int* ptr;
};

// Versión mejorada:

class BuenEstilo {
public:
    explicit BuenEstilo(int valor = 0)
        : valor_(valor), ptr_(nullptr) {}

    int getValor() const { return valor_; }

    void setNombre(const std::string& nombre) {
        nombre_ = nombre;
    }

    virtual ~BuenEstilo() = default;

private:
    int valor_;
    std::string nombre_;
    std::unique_ptr<int> ptr_;
};

// EJEMPLO 6: CHECKS ESENCIALES DE CLANG-TIDY
//==============================================================================

/*
  modernize-*
  -----------
  • modernize-use-auto: Reemplazar tipos verbosos con auto.
  • modernize-use-nullptr: Reemplazar NULL/0 con nullptr.
  • modernize-use-override: Añadir override a funciones virtuales.
  • modernize-use-default-member-init: Inicializar miembros in-class.
  • modernize-use-emplace: Usar emplace_back en lugar de push_back.
  • modernize-loop-convert: Convertir for tradicional a range-based.
  • modernize-raw-string-literal: Usar raw strings para regex.
  • modernize-return-braced-init-list: Devolver {} en lugar de make_pair.

  performance-*
  ------------
  • performance-unnecessary-value-param: Parámetros que deberían ser const&.
  • performance-move-const-arg: std::move en objeto const.
  • performance-for-range-copy: Copia innecesaria en range-for.
  • performance-implicit-conversion-in-loop: Conversiones en loops.

  bugprone-*
  ----------
  • bugprone-suspicious-missing-comma: Saltos de línea sospechosos.
  • bugprone-too-small-loop-variable: Loop variable muy pequeña.
  • bugprone-integer-division: División entera asignada a float.
  • bugprone-unused-raii: Objeto RAII no usado.
  • bugprone-narrowing-conversions: Conversiones que pierden datos.

  readability-*
  ------------
  • readability-identifier-naming: Forzar convención de nombres.
  • readability-braces-around-statements: Exigir llaves siempre.
  • readability-const-return-type: Devolver const no tiene sentido.
  • readability-implicit-bool-conversion: Conversiones implícitas a bool.
*/

// --- modernize-use-auto ---
void check_modernize_auto() {
    // Antes:
    std::vector<int> vec = {1, 2, 3};
    std::vector<int>::iterator it = vec.begin();

    // Después (clang-tidy --fix):
    auto vec2 = std::vector{1, 2, 3};
    auto it2 = vec2.begin();
}

// --- modernize-use-emplace ---
void check_modernize_emplace() {
    std::vector<std::pair<int, std::string>> v;

    // Antes:
    v.push_back(std::make_pair(1, "uno"));
    v.push_back(std::pair<int, std::string>(2, "dos"));

    // Después:
    v.emplace_back(1, "uno");
    v.emplace_back(2, "dos");
}

// --- performance-unnecessary-value-param ---
void check_perf_value_param(std::string s) { // Warning: debería ser const std::string&
    std::cout << s;
}

void check_perf_value_param_ok(const std::string& s) {
    std::cout << s;
}

// --- bugprone-integer-division ---
void check_bugprone_int_div() {
    int a = 5, b = 2;
    double resultado = a / b; // bugprone: división entera, resultado = 2.0
    // Debería ser: double resultado = static_cast<double>(a) / b;
}

/*
  Los sanitizers son análisis DINÁMICO (en ejecución), pero
  complementan perfectamente al análisis estático.

  AddressSanitizer (ASan):
  g++ -fsanitize=address -g -O1 programa.cpp

  UndefinedBehaviorSanitizer (UBSan):
  g++ -fsanitize=undefined -g -O1 programa.cpp

  ThreadSanitizer (TSan):
  g++ -fsanitize=thread -g -O1 programa.cpp

  MemorySanitizer (MSan):
  g++ -fsanitize=memory -g -O1 programa.cpp

  LeakSanitizer (LSan):
  g++ -fsanitize=leak -g -O1 programa.cpp

  COMBINACIÓN RECOMENDADA (más común):
  g++ -g -O1 -fsanitize=address,undefined -fno-omit-frame-pointer
*/

class ObjetoConBug {
public:
    ObjetoConBug() : datos(new int[100]) {}

    // BUG: Falta destructor, copy constructor, copy assignment
    // ASan detectará memory leak
    // UBSan no dirá nada aquí

    void escribir(int idx, int valor) {
        datos[idx] = valor; // ASan detecta si idx < 0 o idx >= 100
    }

private:
    int* datos;
};

  │                                                                     │
  │  5. Sanitizers (tests):                                            │
  │     g++ -fsanitize=address,undefined programa.cpp                   │
  │                                                                     │
  │  6. CI/CD: ERROR si hay warnings:                                  │
  │     cppcheck --error-exitcode=1 ...                                │
  │     clang-tidy --warnings-as-errors='*' ...                        │
  │                                                                     │
  │  7. Suprime solo cuando es necesario:                              │
  │     // NOLINTNEXTLINE                                               │
  │     // cppcheck-suppress nombre                                     │
  └─────────────────────────────────────────────────────────────────────┘
*/

/*
  P: ¿cppcheck o clang-tidy? ¿Cuál es mejor?
  R: Ambos. cppcheck es más rápido y detecta bugs clásicos.
     clang-tidy es más moderno y configurable. Úsalos juntos.

  P: ¿Debo usar -Werror en producción?
  R: Sí, si tu base de código está limpia. Si hay warnings que no
     puedes arreglar, suprímelos específicamente, no quites -Werror.

  P: Los sanitizers ya detectan bugs, ¿necesito análisis estático?
  R: Sí. El análisis estático cubre TODOS los paths, los sanitizers
     solo los que se ejecutan. Se complementan.

  P: ¿Cómo integro esto en un proyecto existente con miles de warnings?
  R: Gradualmente. Empieza con --suppress para ignorar los existentes
     y --error-exitcode=1 para que los NUEVOS fallen. Limpia de a poco.

  P: clang-tidy es muy lento, ¿cómo acelero?
  R: Usa -p build/ para pasar el compile_commands.json. Limita los
     checks a los que necesitas. No analices archivos generados.

  P: ¿Qué hago con los falsos positivos?
  R: Suprímelos LOCALMENTE con // NOLINT o // cppcheck-suppress.
     Nunca desactives un check globalmente por un falso positivo.

  P: ¿Funciona con C++20 modules?
  R: cppcheck tiene soporte experimental. clang-tidy funciona mejor
     con módulos de Clang. Es un área en evolución.

  P: ¿Qué es MISRA y cómo se relaciona?
  R: MISRA es un conjunto de reglas para C++ crítico (automoción,
     aviación). cppcheck y clang-tidy pueden verificar subsets de MISRA.
*/
