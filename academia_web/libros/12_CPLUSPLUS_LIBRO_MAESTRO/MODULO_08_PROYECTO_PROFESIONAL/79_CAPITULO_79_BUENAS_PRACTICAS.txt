► 1. GUÍAS DE ESTILO — ¿CUÁL ELEGIR?

  Las principales guías de estilo en C++:

  • Google C++ Style Guide: La más usada en la industria.
    Prohibe excepciones, usa raw pointers con propiedad marcada.
    https://google.github.io/styleguide/cppguide.html

  • LLVM Coding Standards: Usada en el ecosistema Clang/LLVM.
    Similar a Google. camelBack para variables, CamelCase para clases.
    https://llvm.org/docs/CodingStandards.html

  • C++ Core Guidelines: Del propio Bjarne Stroustrup y Herb Sutter.
    No es tanto un style guide como un conjunto de reglas de buen uso.
    https://isocpp.github.io/CppCoreGuidelines/

  • Qt Coding Style: Usada en proyectos Qt. Diferente: archivos .h/.cpp
    separados, naming específico.

  IMPORTANTE: CUALQUIER guía consistente es mejor que ninguna.
  La peor convención es la que cambia cada 100 líneas.

*/

/*
  Google Style (usaremos esta para el libro):

  CLASES:          CamelCase     → class UsuarioService { };
  FUNCIONES:       camelBack     → void obtener_datos();
  VARIABLES:       snake_case    → int total_usuarios;
  MIEMBROS:        snake_case_   → int id_; (sufijo _)
  CONSTANTES:      kCamelCase    → const int kMaximoIntentos = 3;
  ENUMS:           kCamelCase    → enum class Color { kRojo, kVerde };
  MACROS:          UPPER_CASE    → #define PI 3.14159
  NAMESPACES:      snake_case    → namespace modulo_autenticacion {}
  ARCHIVOS:        snake_case    → usuario_service.h, usuario_service.cpp

  LLVM Style (alternativa):

  CLASES:          CamelCase     → igual
  FUNCIONES:       camelBack     → similar pero puede diferir
  VARIABLES:       camelBack     → totalUsuarios (¡diferente!)
  MIEMBROS:     camelBack_       → id_ (¡casi igual!)
*/

// Google Style:
class UsuarioService {
public:
    static constexpr int kMaxIntentos = 5;

    UsuarioService() : id_(next_id_++) {}

    int id() const { return id_; }
    void set_nombre(const std::string& nombre) {
        nombre_ = nombre;
    }
    const std::string& nombre() const { return nombre_; }

    bool autenticar(const std::string& password) {
        intentos_++;
        return password == password_hash_;
    }

    bool bloqueado() const {
        return intentos_ >= kMaxIntentos;
    }

private:
    int id_;
    std::string nombre_;
    std::string password_hash_;
    int intentos_ = 0;
    static int next_id_;
};

int UsuarioService::next_id_ = 1;

// EJEMPLO 2: CONST-CORRECTNESS
//==============================================================================

/*
  const-correctness es la disciplina de marcar como const todo
  aquello que no cambia. Esto permite:

  1. Que el compilador te avise si modificas algo por error.
  2. Documentar la intención de la función.
  3. Habilitar optimizaciones del compilador.
  4. Usar objetos const en contextos que lo requieren.

  REGLAS:
  • Miembros que no modifican el objeto → const.
  • Parámetros que no modificas → const&.
  • Variables que no cambian → const.
  • Punteros que no modifican el apuntado → const*.
  • Devoluciones que no deben modificarse → const.
*/

class Punto {
public:
    constexpr Punto(double x, double y) noexcept
        : x_(x), y_(y) {}

    // Getters const (no modifican el objeto)
    constexpr double x() const noexcept { return x_; }
    constexpr double y() const noexcept { return y_; }

    // Setters (modifican)
    constexpr void set_x(double x) noexcept { x_ = x; }
    constexpr void set_y(double y) noexcept { y_ = y; }

    // Función const que devuelve un nuevo objeto (no modifica this)
    constexpr Punto operator+(const Punto& otro) const noexcept {
        return {x_ + otro.x_, y_ + otro.y_};
    }

    // Función const que computa algo
    constexpr double distancia_al_origen() const noexcept {
        return x_ * x_ + y_ * y_;
    }

private:
    double x_, y_;
};

void ejemplo_const_correctness() {
    // Objeto mutable
    Punto p1(3, 4);
    p1.set_x(10);  // OK: p1 no es const

    // Objeto const
    const Punto origen(0, 0);
    // origen.set_x(5);  // Error: no puedes modificar objeto const
    double dist = origen.distancia_al_origen();  // OK: es const

    // Parámetros
    auto imprimir_punto = [](const Punto& p) {  // const ref para no copiar
        std::cout << "(" << p.x() << ", " << p.y() << ")\n";
        // p.set_x(10);  // Error: p es const ref
    };
    imprimir_punto(p1);
}

 * @brief Servicio de gestión de usuarios del sistema
 * @author Arquitecto C++

/**
 * @brief Representa un usuario del sistema
 *
 * Almacena la información básica de un usuario incluyendo
 * credenciales y estado de actividad.
 *
 * @note Esta clase es thread-safe solo si se usa con mutex externo
 * @see UsuarioService para la gestión de usuarios
 */
class Usuario {
public:
    /**
     * @brief Constructor principal
     * @param id Identificador único del usuario
     * @param nombre Nombre completo del usuario
     * @param email Correo electrónico (debe ser válido)
     *
     * El email se valida internamente; si no es válido,
     * se lanza std::invalid_argument.
     */
    Usuario(int id,
            std::string nombre,
            std::string email)
        : id_(id)
        , nombre_(std::move(nombre))
        , email_(std::move(email)) {
        if (!validar_email(email_)) {
            throw std::invalid_argument(
                "Email inválido: " + email_);
        }
    }

    /// @brief Obtiene el ID del usuario
    /// @return int ID único
    int id() const noexcept { return id_; }

    /// @brief Obtiene el nombre del usuario
    /// @return const std::string& Referencia al nombre (no copia)
    const std::string& nombre() const noexcept { return nombre_; }

    /**
     * @brief Actualiza el nombre del usuario
     * @param nombre Nuevo nombre (debe tener al menos 2 caracteres)
     * @return true si se actualizó correctamente
     * @return false si el nombre es demasiado corto
     */
    bool set_nombre(const std::string& nombre) {
        if (nombre.size() < 2) return false;
        nombre_ = nombre;
        return true;
    }

    /**
     * @brief Verifica si el usuario está activo
     * @return true si el usuario está activo
     *
     * Un usuario se considera inactivo si no ha iniciado sesión
     * en los últimos @ref kDiasInactividad días.
     */
    bool activo() const {
        using namespace std::chrono;
        auto ahora = system_clock::now();
        auto diff = ahora - ultimo_acceso_;
        return diff < days(kDiasInactividad);
    }

private:
    static constexpr int kDiasInactividad = 90;

    /**
     * @brief Valida un email
     * @param email String a validar
     * @return true si el email tiene formato válido
     *
     * Validación básica: debe contener @ y un punto después.
     * No es una validación exhaustiva RFC 5322.
     */
    static bool validar_email(const std::string& email) {
        auto arroba = email.find('@');
        if (arroba == std::string::npos) return false;
        auto punto = email.find('.', arroba);
        if (punto == std::string::npos) return false;
        return punto < email.size() - 1;
    }

    int id_;
    std::string nombre_;
    std::string email_;
    std::chrono::system_clock::time_point ultimo_acceso_;
};

/*
  RECONOCE Y EVITA:

  • GOD CLASS: Clase que hace de todo. -> SRP
  • SPAGHETTI CODE: Flujo de control enredado. -> Refactor
  • COPY-PASTA: Código duplicado. -> DRY (Don't Repeat Yourself)
  • MAGIC NUMBERS: Números sin nombre. -> constexpr
  • LONG PARAMETER LIST: >3 parámetros. -> Struct de parámetros
  • SHOTGUN SURGERY: Cambio requiere modificar muchos archivos.
  • FEATURE ENVY: Una clase usa mucho de otra. -> Mover método
  • SWITCH STATEMENTS: Switches grandes. -> Polimorfismo
  • PRIMITIVE OBSESSION: Usar primitivos donde va un objeto.
  • NULL CHECKS: if (ptr) repetidos. -> std::optional
  • NAKED NEW/DELETE: new sin smart pointers. -> unique_ptr
  • C-STYLE CASTS: (int) en lugar de static_cast.
*/

// ANTI-PATRÓN: Magic numbers

// ERROR: Olvidar virtual destructor
class Base {
public:
    ~Base() { std::cout << "~Base()\n"; }  // No virtual!
};
class Derivada : public Base {
    int* datos = new int[100];
public:
    ~Derivada() { delete[] datos; std::cout << "~Derivada()\n"; }
};
void error_virtual_dtor() {
    Base* b = new Derivada();
    delete b;  // Sólo llama ~Base(), fuga de memoria!
}

// ERROR: Copia superficial (shallow copy)
class BufferMal {
public:
    BufferMal(size_t n) : size_(n), data_(new int[n]) {}
    // Error: usa el compilador copy constructor (shallow copy)
    ~BufferMal() { delete[] data_; }  // Doble delete si se copia
private:
    size_t size_;
    int* data_;
};

// ERROR: Orden de inicialización en constructor
class OrdenMal {
    std::vector<int> datos_;  // Se inicializa PRIMERO (orden declaración)
    int capacidad_;           // Se inicializa SEGUNDO
public:
    // Error: capacidad_ se usa ANTES de inicializarse
    OrdenMal(int n) : capacidad_(n), datos_(capacidad_) {}
};

// CORRECTO:
class OrdenBien {
    std::vector<int> datos_;  // Primero (orden declaración)
    int capacidad_;           // Segundo
public:
    OrdenBien(int n) : datos_(n), capacidad_(n) {}  // OK
};

// CÓDIGO LEGACY (C++98 style):
class LegacyClass {
public:
    LegacyClass() {
        data = new int[100];
        name = new char[256];
        strcpy(name, "default");
    }
    ~LegacyClass() {
        delete[] data;
        delete[] name;
    }
    LegacyClass(const LegacyClass& other) {
        data = new int[100];
        memcpy(data, other.data, 100 * sizeof(int));
        name = new char[256];
        strcpy(name, other.name);
    }
    LegacyClass& operator=(const LegacyClass& other) {
        if (this != &other) {
            memcpy(data, other.data, 100 * sizeof(int));
            strcpy(name, other.name);
        }
        return *this;
    }
    void set_name(const char* n) {
        strcpy(name, n);  // Buffer overflow si n > 255
    }
    const char* get_name() const { return name; }
    int* data;
    char* name;
};

// CÓDIGO MODERNO (C++20):
class ModernClass {
public:
    ModernClass() : data_(100), name_("default") {}

    void set_name(const std::string& name) { name_ = name; }
    const std::string& get_name() const noexcept { return name_; }

    // RAII: sin destructor manual, sin copy/move boilerplate
private:
    std::vector<int> data_;
    std::string name_;
};

// MAL: #define para constantes
#define PI 3.14159
#define MAX(x, y) ((x) > (y) ? (x) : (y))

// BIEN: constexpr
constexpr double kPi = 3.14159265358979323846;

template<typename T>
constexpr T maximo(const T& a, const T& b) {
    return a > b ? a : b;
}

// MAL: #ifdef para configuración
// #ifdef DEBUG
//     #define LOG(x) std::cout << x
// #else
//     #define LOG(x)
// #endif

// BIEN: constexpr booleano
constexpr bool kDebugMode = true;

template<typename... Args>
void log(Args&&... args) {
    if constexpr (kDebugMode) {
        (std::cout << ... << args) << "\n";
    }
}

/*
  Un test unitario debe:

  1. Aislar la unidad bajo prueba (sin dependencias externas).
  2. Ser rápido (< 1ms por test).
  3. Ser determinista (misma entrada = misma salida).
  4. Probar UN solo comportamiento.
  5. Tener nombre descriptivo.

  ESTRUCTURA ARRANGE-ACT-ASSERT (AAA):
  // Arrange: Preparar datos y mocks
  // Act: Ejecutar la función bajo prueba
  // Assert: Verificar resultado
*/

void test_sumar() {
    // Arrange
    int a = 5, b = 3;

    // Act
    int resultado = a + b;

    // Assert
    assert(resultado == 8);
}

/*
  P: ¿Realmente importa el estilo? ¿No es perder el tiempo?
  R: En proyectos individuales, menos. En equipo, es crítico.
     La consistencia reduce la fricción cognitiva al leer código.
     Un equipo sin guía de estilo gasta el 30% del tiempo en
     discusiones de formato.

  P: ¿Google Style prohíbe excepciones. ¿Debo seguirlo al pie?
  R: Depende del contexto. En sistemas críticos donde las
     excepciones no son aceptables (embedded, game engines),
     sí. En aplicaciones de escritorio, las excepciones son
     útiles. Adapta la guía a tu proyecto.

  P: ¿Todos los comentarios deben ser Doxygen?
  R: Las APIs públicas sí. El código interno puede tener
     comentarios explicativos cuando la lógica no es obvia.
     "Código autodocumentado" es el ideal; los comentarios
     explican el POR QUÉ, no el QUÉ.

  P: ¿Cuándo usar auto?
  R: auto es obligatorio para lambdas y tipos no escribibles.
     Es opcional para tipos largos (std::vector::iterator).
     No uses auto cuando el tipo no es obvio (auto x = funcion();
     ¿qué es x?).

  P: ¿struct o class?
  R: struct para POD (Plain Old Data) sin métodos complejos.
     class para objetos con invariantes y encapsulación.
     La convención es: struct para datos, class para comportamiento.

  P: ¿Debo usar #pragma once o #ifndef?
  R: #pragma once es más rápido, menos propenso a errores y
     más limpio. #ifndef es más portable (proyectos muy antiguos).
     Hoy #pragma once es soportado por todos los compiladores.

  P: ¿Snake_case o camelCase para variables?
  R: Google: snake_case. LLVM: camelBack. Qt: camelCase.
     Elige UNA y sé consistente. Lo peor es mezclar.

  P: ¿Cómo manejo errores sin excepciones?
  R: std::optional, std::expected (C++23), error codes,
     out parameters. Google Style usa error codes.
*/
