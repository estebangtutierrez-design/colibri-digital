► PARTE I: FUNDAMENTOS (20 preguntas, 1 pt c/u)

  1. ¿Qué diferencia hay entre int* p, int *p y int * p?
     RESPUESTA: Las tres son equivalentes. El espacio puede ir
     en cualquier lugar. La convención común es int* p (énfasis
     en el tipo) o int *p (énfasis en la sintaxis).

  2. Explica la diferencia entre:
     int arr[10];
     std::array<int, 10> arr;
     std::vector<int> arr(10);

     RESPUESTA:
     - int arr[10]: Array de C en el stack. Sin bounds checking.
       No tiene métodos. Decae a puntero.
     - std::array: Wrapper de C++ sobre array de C. Tamaño fijo
       en compilación. Interfaz similar a vector. Sin overhead.
     - std::vector: Heap allocation. Tamaño dinámico. Más pesado
       pero flexible.

  3. ¿Cuándo se llama al destructor de un objeto?
     RESPUESTA: Cuando el objeto sale de ámbito (stack), cuando
     se hace delete (heap), o cuando el último shared_ptr se
     destruye. También durante stack unwinding por excepciones.

  4. ¿Qué es la elisión de copia (copy elision)?
     RESPUESTA: Optimización del compilador que elimina copias
     innecesarias, especialmente en return by value. Desde C++17,
     la elisión de copia es obligatoria en ciertos contextos (RVO).

  5. Explica la diferencia entre:
     void f(int& x);
     void f(const int& x);
     void f(int&& x);

     RESPUESTA:
     - int& x: Referencia mutable — puede modificar el original.
     - const int& x: Referencia const — no puede modificar, acepta
       temporales y lvalues.
     - int&& x: Rvalue reference — solo acepta temporales, usado
       para move semantics.

  6. ¿Qué es el type erasure? Da un ejemplo.
     RESPUESTA: Técnica que oculta el tipo concreto detrás de una
     interfaz. std::function es el ejemplo clásico: puede almacenar
     cualquier callable independientemente de su tipo real.

  7. ¿Qué imprime este código?
     int x = 5;
     int y = x++ + ++x;
     // Respuesta: Comportamiento indefinido (UB) entre C++11 y C++16.
     // x++ y ++x modifican x sin secuenciación entre sí.

  8. ¿Qué hace std::move realmente?
     RESPUESTA: std::move no mueve nada. Es un static_cast a
     rvalue reference (T&&). El movimiento real ocurre en el
     constructor/assignment que recibe el rvalue.

  9. Explica RAII con un ejemplo.
     RESPUESTA: Resource Acquisition Is Initialization. El recurso
     se adquiere en el constructor y se libera en el destructor.
     Ejemplo: std::ifstream abre el archivo en el constructor y
     lo cierra en el destructor automáticamente.

  10. ¿Qué diferencia hay entre struct y class?
      RESPUESTA: Por defecto, los miembros de struct son public,
      los de class son private. Por defecto, la herencia de struct
      es public, la de class es private. Son idénticos en todo lo
      demás.

  11. ¿Qué es un virtual destructor y cuándo es necesario?
      RESPUESTA: Un destructor virtual asegura que se llame al
      destructor de la clase derivada cuando se elimina un objeto
      a través de un puntero a la base. Necesario en jerarquías
      polimórficas.

  12. Explica el propósito de std::optional.
      RESPUESTA: Representa un valor que puede estar presente o no.
      Alternativa segura a punteros nulos o valores centinela.
      Similar a Option en Rust o Maybe en Haskell.

  13. ¿Qué diferencia hay entre new/malloc?
      RESPUESTA: new llama al constructor, malloc solo asigna
      memoria. new lanza std::bad_alloc si falla, malloc devuelve
      nullptr. new es type-safe, malloc devuelve void*.

  14. ¿Qué es SFINAE?
      RESPUESTA: Substitution Failure Is Not An Error. Si durante
      la sustitución de template un tipo da error, no es un error
      de compilación, simplemente se descarta esa sobrecarga.

  15. Explica la regla de los 0/3/5.
      RESPUESTA: Regla de 0: si no gestionas recursos, no declares
      nada (el compilador genera todo). Regla de 3: si declaras
      destructor, copy constructor o copy assignment, declara los
      tres. Regla de 5: añade move constructor y move assignment.

  16. ¿Qué es un lambda y cuál es su tipo?
      RESPUESTA: Lambda es una función anónima. Su tipo es único
      y no escribible (solo con auto o std::function). Cada lambda
      tiene un tipo diferente aunque tenga la misma firma.

  17. ¿Cómo funciona std::variant?
      RESPUESTA: Union type-safe (similar a sum types). Almacena
      un valor de uno de los tipos especificados. Se accede con
      std::visit o std::get. No hay acceso incorrecto sin excepción.

  18. ¿Qué es el comportamiento indefinido (UB)?
      RESPUESTA: Cualquier comportamiento no especificado por el
      estándar. El compilador puede hacer cualquier cosa (incluyendo
      generar código que parezca funcionar, o que explote). Ejemplos:
      overflow de signed, desreferencia nullptr, buffer overflow.

  19. ¿Qué son los módulos de C++20?
      RESPUESTA: Sistema de módulos que reemplaza #include.
      Ofrece mejor encapsulación, compilación más rápida y sin
      macros. Usa export e import en lugar de #include.

  20. Explica la diferencia entre:
      template<typename T>
      template<class T>
      template<typename T = int>

      RESPUESTA: typename y class son equivalentes en templates.
      La tercera versión da un valor por defecto (int) si no se
      especifica el argumento.

► PARTE II: PROGRAMACIÓN AVANZADA (10 preguntas, 3 pts c/u)

  1. Escribe una función constexpr que calcule el n-ésimo número
     de Fibonacci y verifica con static_assert que Fib(20) == 6765.

     RESPUESTA:
     constexpr int fibonacci(int n) {
         if (n <= 1) return n;
         int a = 0, b = 1;
         for (int i = 2; i <= n; ++i) {
             int c = a + b;
             a = b;
             b = c;
         }
         return b;
     }
     static_assert(fibonacci(20) == 6765);

  2. Implementa una función que reciba un vector y devuelva
     otro vector con los elementos únicos, SIN modificar el
     original. Hazlo eficiente (O(n) promedio).

     RESPUESTA:
     template<typename T>
     std::vector<T> unicos(const std::vector<T>& v) {
         std::unordered_set<T> vistos;
         std::vector<T> resultado;
         resultado.reserve(v.size());
         for (const auto& elem : v) {
             if (vistos.insert(elem).second) {
                 resultado.push_back(elem);
             }
         }
         return resultado;
     }

  3. Explica el output de este código y por qué:

     std::string s1 = "Hello";
     std::string s2 = std::move(s1);
     std::cout << s1;

     RESPUESTA: s1 está en estado válido pero no especificado
     después del move. Podría imprimir "" (vacío) o "Hello"
     dependiendo de la implementación de std::string. No es
     confiable. Después de std::move, solo se debe destruir o
     reasignar s1.

  4. Implementa una clase Matrix<T> que:
     • Almacene datos en un único vector (no vector de vectores).
     • Soporte operator()(i, j) para acceso.
     • Sea eficiente (cache-friendly).

     RESPUESTA:
     template<typename T>
     class Matrix {
         size_t rows_, cols_;
         std::vector<T> data_;
     public:
         Matrix(size_t rows, size_t cols, T init = T{})
             : rows_(rows), cols_(cols), data_(rows * cols, init) {}

         T& operator()(size_t i, size_t j) {
             return data_[i * cols_ + j];
         }
         const T& operator()(size_t i, size_t j) const {
             return data_[i * cols_ + j];
         }
         size_t rows() const { return rows_; }
         size_t cols() const { return cols_; }
     };

  5. Escribe una implementación segura de una pila (stack)
     usando std::vector como contenedor subyacente. Hazla
     thread-safe.

     RESPUESTA:
     template<typename T>
     class ThreadSafeStack {
         std::vector<T> data_;
         mutable std::mutex mtx_;
     public:
         void push(const T& value) {
             std::lock_guard<std::mutex> lock(mtx_);
             data_.push_back(value);
         }
         std::optional<T> pop() {
             std::lock_guard<std::mutex> lock(mtx_);
             if (data_.empty()) return std::nullopt;
             T value = std::move(data_.back());
             data_.pop_back();
             return value;
         }
         bool empty() const {
             std::lock_guard<std::mutex> lock(mtx_);
             return data_.empty();
         }
     };

  6. Explica cómo implementarías un pool de hilos (thread pool)
     en C++ moderno. Menciona las clases y mecanismos clave.

     RESPUESTA:
     - std::vector<std::thread> para los hilos de trabajo.
     - std::queue<std::function<void()>> para las tareas.
     - std::mutex + std::condition_variable para sincronización.
     - std::atomic<bool> para señal de parada.
     - Los hilos esperan en la condition variable hasta que hay
       tareas o se les pide terminar.

  7. Implementa una función split que divida un string por un
     delimitador y devuelva un vector de strings. Hazla eficiente.

     RESPUESTA:
     std::vector<std::string> split(
         const std::string& s, char delim) {
         std::vector<std::string> resultado;
         size_t inicio = 0;
         for (size_t i = 0; i <= s.size(); ++i) {
             if (i == s.size() || s[i] == delim) {
                 resultado.push_back(s.substr(inicio, i - inicio));
                 inicio = i + 1;
             }
         }
         return resultado;
     }

  8. ¿Qué es un "fold expression" en C++17? Da un ejemplo.

     RESPUESTA: Los fold expressions permiten aplicar un operador
     sobre un pack de parámetros variádicos.

     template<typename... Args>
     auto sumar(Args... args) {
         return (... + args);  // Unario left fold
     }
     // sumar(1, 2, 3) = ((1 + 2) + 3) = 6

     template<typename... Args>
     void print(Args&&... args) {
         ((std::cout << args << " "), ...);  // Comma fold
     }

  9. Escribe un wrapper que mida el tiempo de ejecución de
     cualquier función (decorator de tiempo).

     RESPUESTA:
     template<typename Func, typename... Args>
     auto medir_tiempo(Func&& func, Args&&... args) {
         auto inicio = std::chrono::steady_clock::now();
         auto resultado = std::forward<Func>(func)(
             std::forward<Args>(args)...);
         auto fin = std::chrono::steady_clock::now();
         auto duracion = std::chrono::duration_cast<
             std::chrono::microseconds>(fin - inicio);
         std::cout << "Tiempo: " << duracion.count() << "μs\n";
         return resultado;
     }

  10. Explica el concepto de "continuation passing style"
      aplicado a C++ con lambdas o std::function.

      RESPUESTA: En lugar de devolver un valor, se pasa una
      función (continuation) que recibe el resultado:

      void procesar(int x, std::function<void(int)> cont) {
          int resultado = x * 2;
          cont(resultado);
      }
      // Uso:
      procesar(5, [](int r) {
          std::cout << "Resultado: " << r << "\n";
      });

► PARTE III: DISEÑO Y ARQUITECTURA (5 preguntas, 4 pts c/u)

  1. Diseña un sistema de plugin para C++ donde los módulos
     se puedan cargar en tiempo de ejecución (DLL/.so). Especifica:
     • Interfaz del plugin.
     • Cómo se registran los plugins.
     • Cómo se gestiona el ciclo de vida.

     RESPUESTA ESPERADA:
     - Interfaz base class Plugin con métodos virtuales init(),
       execute(), shutdown().
     - Cada plugin se exporta con extern "C" Plugin* create().
     - Se cargan con dlopen()/dlsym().
     - Se gestionan con shared_ptr y un PluginManager singleton.
     - Los plugins se registran en un mapa<string, PluginPtr>.

  2. Compara y contrasta:
     • Arquitectura por capas.
     • Arquitectura hexagonal (puertos y adaptadores).
     • Arquitectura limpia (Clean Architecture).
     ¿Cuándo usarías cada una?

     RESPUESTA ESPERADA:
     - Capas: Proyectos pequeños-medianos donde el dominio no
       es complejo. Fácil de entender pero acopla capas.
     - Hexagonal: Aplicaciones donde el dominio es el core del
       negocio. Aísla completamente el dominio. Ideal para DDD.
     - Clean Architecture: Similar a hexagonal pero más formal.
       Útil en equipos grandes donde se necesita mucha disciplina.
     Todas dependen de Inversión de Dependencias (DIP).

  3. Implementa un contenedor de inyección de dependencias
     básico que soporte:
     • Registro por interfaz.
     • Resolución automática de dependencias.
     • Singleton vs transient.
     • Inyección por constructor.

     RESPUESTA ESPERADA:
     class DIContainer {
         std::unordered_map<std::type_index,
             std::function<std::shared_ptr<void>()>> factories;
         std::unordered_map<std::type_index,
             std::shared_ptr<void>> singletons;
     public:
         template<typename T, typename Impl>
         void register_singleton() {
             factories[typeid(T)] = [this]() {
                 auto ptr = std::make_shared<Impl>();
                 singletons[typeid(T)] = ptr;
                 return ptr;
             };
         }
         template<typename T>
         std::shared_ptr<T> resolve() { /* ... */ }
     };

  4. Evalúa este diseño. ¿Qué patrones reconoce? ¿Qué mejoras
     sugerirías?

     class ReportGenerator {
         Database db;
         EmailService email;
         PDFConverter pdf;
     public:
         void generateAndSend(int reportId) {
             auto data = db.query(reportId);
             auto doc = pdf.convert(data);
             email.send(doc);
         }
     };

     RESPUESTA ESPERADA:
     - Problemas: Alto acoplamiento, viola DIP, no testeable,
       viola SRP (genera, convierte y envía).
     - Patrones a aplicar: DI (inyectar dependencias), Strategy
       (diferentes formatos de salida), Observer (notificar evento).
     - Mejora: Inyectar Database, EmailService, PDFConverter como
       interfaces. Separar en ReportService, ReportFormatter,
       ReportSender.

  5. Diseña un caché LRU (Least Recently Used) thread-safe
     usando C++ moderno.

     RESPUESTA ESPERADA:
     template<typename K, typename V>
     class LRUCache {
         size_t capacity_;
         std::list<std::pair<K, V>> items_;
         std::unordered_map<K, typename decltype(items_)::iterator> map_;
         mutable std::mutex mtx_;
     public:
         explicit LRUCache(size_t cap) : capacity_(cap) {}
         V get(const K& key) {
             std::lock_guard lock(mtx_);
             auto it = map_.find(key);
             if (it == map_.end()) throw std::out_of_range("key");
             items_.splice(items_.begin(), items_, it->second);
             return it->second->second;
         }
         void put(const K& key, const V& value) {
             std::lock_guard lock(mtx_);
             if (map_.find(key) != map_.end()) {
                 items_.erase(map_[key]);
             }
             items_.emplace_front(key, value);
             map_[key] = items_.begin();
             if (map_.size() > capacity_) {
                 map_.erase(items_.back().first);
                 items_.pop_back();
             }
         }
     };

► PARTE IV: PROBLEMAS PRÁCTICOS (3 problemas, varía puntaje)

  PROBLEMA 1: ANALIZADOR DE LOGS (8 pts)
  ---------------------------------------

  Escribe un programa que lea un archivo de log con formato:
  [NIVEL] [FECHA] [COMPONENTE] Mensaje

  y produzca un resumen con:
  • Número de líneas por nivel (INFO, WARN, ERROR).
  • Los 3 componentes con más errores.
  • Tiempo entre el primer y último evento de cada nivel.

  Solución esperada: Uso de std::regex, std::map, manejo
  eficiente de strings, flujo (ifstream), sort parcial.

  PROBLEMA 2: COMPRESOR RLE (6 pts)
  ----------------------------------

  Implementa Run-Length Encoding (RLE):
  • Comprimir: "AAAABBBCCDAA" → "A4B3C2D1A2"
  • Descomprimir: "A4B3C2D1A2" → "AAAABBBCCDAA"

  Requisitos:
  • Funciones constexpr (si es posible).
  • Manejar caracteres que se repiten más de 9 veces.
  • Tests con assert.

  Solución esperada: Templates, constexpr, string_view,
  range-based for, std::to_string o stringstream.

  PROBLEMA 3: SISTEMA DE EVENTOS (6 pts)
  ---------------------------------------

  Implementa un sistema de eventos donde:
  • Se puedan registrar oyentes para eventos por nombre.
  • Los oyentes sean std::function<void(const Event&)>.
  • Se puedan emitir eventos con datos asociados.
  • Soporte prioridad de oyentes.
  • Sea thread-safe para emitir desde cualquier hilo.

  Solución esperada: Signal/Slot, variadic templates,
  std::map con prioridad, mutex compartido (shared_mutex).

  ¿Tiene futuro C++ en este campo?

  --------------------------------------
