► 1. FLAGS DEL COMPILADOR — TU PRIMER AJUSTE

  El compilador GCC/Clang ofrece múltiples niveles de optimización:

  • -O0: Sin optimización. Compilación rápida, ejecución lenta.
         Ideal para debugging.
  • -O1: Optimizaciones básicas. Sin crecimiento de código.
         Balance entre compilación y ejecución.
  • -O2: Optimizaciones fuertes. Estándar para releases.
         Incluye inlining, loop unrolling, etc.
  • -O3: Optimizaciones agresivas. Incluye vectorización.
         Puede aumentar tamaño del binario.
  • -Os: Optimizar por tamaño. Útil para sistemas embebidos.
  • -Ofast: -O3 + optimizaciones que violan el estándar.
            No recomendado para código portable.
  • -Og: Optimizar para debugging. Mejor que -O0 pero debuggeable.

  FLAGS ESPECÍFICOS:
  • -march=native: Optimiza para la CPU actual.
  • -mtune=native: Ajusta para la CPU actual.
  • -flto: Link Time Optimization — optimiza entre unidades de compilación.
  • -funroll-loops: Desenrolla bucles (puede hinchar el código).
  • -fomit-frame-pointer: No guarda frame pointer (más registros).
  • -ffast-math: Matemáticas rápidas (peligroso para precisión).
  • -fprofile-generate / -fprofile-use: PGO (Profile Guided Optimization).

// EJEMPLO 1: CONSTEXPR — CÓMPUTO EN TIEMPO DE COMPILACIÓN
//==============================================================================

// constexpr: la función se evalúa en compilación si los args son constantes
constexpr int factorial(int n) {
    if (n <= 1) return 1;
    return n * factorial(n - 1);
}

// C++20: consteval — OBLIGA a evaluar en compilación
consteval int cuadrado(int n) {
    return n * n;
}

// C++20: constinit — asegura inicialización en tiempo de compilación
constinit int valor_global = factorial(6); // 720

// constexpr con structs
struct Punto {
    constexpr Punto(double x, double y) : x_(x), y_(y) {}

    constexpr double distancia_al_origen() const {
        return x_ * x_ + y_ * y_;
    }

    constexpr static Punto origen() { return {0, 0}; }

    double x_, y_;
};

void ejemplo_constexpr() {
    // En compilación:
    constexpr int fact_10 = factorial(10);
    static_assert(fact_10 == 3628800);

    // constexpr con array:
    constexpr std::array<int, 5> cuadrados = {
        cuadrado(1), cuadrado(2), cuadrado(3),
        cuadrado(4), cuadrado(5)
    };

    // constexpr con struct:
    constexpr Punto p(3.0, 4.0);
    constexpr double dist = p.distancia_al_origen();
    static_assert(dist == 25.0);

    std::cout << "factorial(10) = " << fact_10 << "\n";
    std::cout << "distancia = " << dist << "\n";
}

// EJEMPLO 2: MOVE SEMANTICS — ELIMINANDO COPIAS
//==============================================================================

/*
  Las move semantics (C++11) permiten transferir recursos de un
  objeto a otro sin copiar. Esencial para evitar copias costosas.

  REGLAS DE ORO:
  1. std::move no mueve nada — solo convierte a rvalue reference.
  2. El movimiento real lo hace el constructor/move assignment.
  3. Después de mover, el origen está en estado válido pero
     no especificado (puedes llamar a destructor y asignación).
*/

class Buffer {
public:
    // Constructor
    explicit Buffer(size_t size)
        : size_(size), datos_(new int[size]) {
        std::cout << "Constructor " << size << " elementos\n";
    }

    // Copy constructor (costoso)
    Buffer(const Buffer& other)
        : size_(other.size_), datos_(new int[other.size_]) {
        std::cout << "Copy constructor (" << size_ << ")\n";
        std::copy(other.datos_, other.datos_ + size_, datos_);
    }

    // Move constructor (barato)
    Buffer(Buffer&& other) noexcept
        : size_(other.size_), datos_(other.datos_) {
        std::cout << "Move constructor (" << size_ << ")\n";
        other.size_ = 0;
        other.datos_ = nullptr;
    }

    // Copy assignment
    Buffer& operator=(const Buffer& other) {
        std::cout << "Copy assignment\n";
        if (this != &other) {
            delete[] datos_;
            size_ = other.size_;
            datos_ = new int[size_];
            std::copy(other.datos_, other.datos_ + size_, datos_);
        }
        return *this;
    }

    // Move assignment
    Buffer& operator=(Buffer&& other) noexcept {
        std::cout << "Move assignment\n";
        if (this != &other) {
            delete[] datos_;
            size_ = other.size_;
            datos_ = other.datos_;
            other.size_ = 0;
            other.datos_ = nullptr;
        }
        return *this;
    }

    ~Buffer() {
        delete[] datos_;
    }

private:
    size_t size_;
    int* datos_;
};

void ejemplo_move_semantics() {
    Buffer b1(1000);           // Constructor
    Buffer b2 = b1;            // Copy constructor

    Buffer b3(std::move(b1));  // Move constructor

    Buffer b4(500);
    b4 = std::move(b2);        // Move assignment

    // std::move en contenedores:
    std::vector<Buffer> buffers;
    buffers.reserve(3);

    Buffer temp(2000);
    buffers.push_back(std::move(temp));

    // temp ya no tiene recursos, pero se puede destruir o reasignar
}

// MAL: Copia el string completo
void procesar_nombre_mal(std::string nombre) {
    std::cout << "Procesando: " << nombre << "\n";
}

// BIEN: Referencia const (no copia)
void procesar_nombre_bien(const std::string& nombre) {
    std::cout << "Procesando: " << nombre << "\n";
}

// PERFECTO: Para almacenar, move si es temporal
class Persona {
public:
    // Versión 1: const ref + copia
    void set_nombre(const std::string& n) {
        nombre_ = n; // Copia desde const ref
    }

    // Versión 2: rvalue ref + move
    void set_nombre(std::string&& n) noexcept {
        nombre_ = std::move(n); // Mueve
    }

    // Versión 3 (C++ moderno): por valor + move
    void set_nombre_moderno(std::string n) {
        nombre_ = std::move(n); // Copia si lvalue, move si rvalue
    }

private:
    std::string nombre_;
};

void ejemplo_pass_by_ref() {
    std::string nombre_largo = "Juan Carlos Martínez López Fernández";

    // Copia: 30+ caracteres copiados
    procesar_nombre_mal(nombre_largo);

    // Sin copia
    procesar_nombre_bien(nombre_largo);

    // Move (rápido)
    procesar_nombre_mal(std::move(nombre_largo));
    // nombre_largo ahora está vacío
}

/*
  La jerarquía de memoria es:
  • L1 Cache: ~4 ciclos, 32KB
  • L2 Cache: ~10 ciclos, 256KB
  • L3 Cache: ~40 ciclos, 8-32MB
  • RAM: ~100-200 ciclos, GBs
  • SSD/HDD: millones de ciclos

  Si tus datos no están en caché, la CPU espera.
*/

// MAL: Array de punteros (fragmentación)
struct Nodo {
    int datos[16];
    Nodo* siguiente;
};

// Para acceder a todos los datos, la CPU salta por la memoria
struct NodoMalo {
    int x, y;
};

// BIEN: Array de estructuras (SoA vs AoS)

// Array of Structures (AoS) — MAL para acceso secuencial
struct ParticulaAoS {
    float x, y, z;
    float vx, vy, vz;
    float masa;
    int tipo;
};

// Structure of Arrays (SoA) — BIEN para acceso secuencial
struct ParticulasSoA {
    std::vector<float> x, y, z;
    std::vector<float> vx, vy, vz;
    std::vector<float> masa;
    std::vector<int> tipo;
};

// Process AoS vs SoA
void ejemplo_cache() {
    const int N = 100000;

    // AoS: datos mezclados en memoria
    std::vector<ParticulaAoS> aos(N);
    // SoA: datos contiguos por campo

    // Para sumar todas las masas:
    // AoS: salta sizeof(ParticulaAoS) cada vez
    double suma_aos = 0;
    for (const auto& p : aos) {
        suma_aos += p.masa;
    }

    // SoA con acceso directo:
    ParticulasSoA soa;
    soa.masa.resize(N);
    double suma_soa = std::accumulate(
        soa.masa.begin(), soa.masa.end(), 0.0);
}

/*
  "Mide, no asumas." — Tautología del ingeniero de performance.

  HERRAMIENTAS DE PROFILING:
  • perf (Linux): Muestreo de CPU, caché misses, branches.
  • Valgrind --tool=callgrind: Simulación detallada.
  • gperftools (Google): CPU profiler y heap profiler.
  • Intel VTune: Completo pero comercial.
  • std::chrono: Medición manual de tiempo.
*/

class Temporizador {
public:
    Temporizador(const std::string& nombre)
        : nombre_(nombre)
        , inicio_(std::chrono::steady_clock::now()) {}

    ~Temporizador() {
        auto fin = std::chrono::steady_clock::now();
        auto duracion = std::chrono::duration_cast<
            std::chrono::microseconds>(fin - inicio_);
        std::cout << nombre_ << ": " << duracion.count()
                  << " μs\n";
    }

private:
    std::string nombre_;
    std::chrono::steady_clock::time_point inicio_;
};

// Benchmark de diferentes formas de sumar
int sumar_for(const std::vector<int>& v) {
    int total = 0;
    for (size_t i = 0; i < v.size(); ++i) {
        total += v[i];
    }
    return total;
}

int sumar_range_for(const std::vector<int>& v) {
    int total = 0;
    for (int x : v) {
        total += x;
    }
    return total;
}

int sumar_accumulate(const std::vector<int>& v) {
    return std::accumulate(v.begin(), v.end(), 0);
}

#ifdef __cpp_lib_parallel_algorithm
int sumar_parallel(const std::vector<int>& v) {
    return std::reduce(std::execution::par, v.begin(), v.end(), 0);
}
#endif

void ejemplo_benchmark() {
    std::vector<int> datos(1000000);
    std::iota(datos.begin(), datos.end(), 1);

    {
        Temporizador t("for loop");
        volatile int r = sumar_for(datos);
    }

    {
        Temporizador t("range-for");
        volatile int r = sumar_range_for(datos);
    }

    {
        Temporizador t("accumulate");
        volatile int r = sumar_accumulate(datos);
    }
}

/*
  std::string típicamente tiene SSO: para strings pequeños

void concatenar_mal(const std::vector<std::string>& palabras) {
    std::string resultado;
    for (const auto& p : palabras) {
        resultado += p; // Puede realocar varias veces
    }
}

// BIEN: Reservar espacio de antemano
void concatenar_bien(const std::vector<std::string>& palabras) {
    std::string resultado;
    size_t total = 0;
    for (const auto& p : palabras) {
        total += p.size();
    }
    resultado.reserve(total);
    for (const auto& p : palabras) {
        resultado += p;
    }
}

    std::ostringstream oss;
    for (const auto& p : palabras) {
        oss << p;
    }
    std::string resultado = oss.str();
}

// Loop interchange: cambiar orden para mejor localidad
void loop_interchange_mal(std::vector<std::vector<int>>& matriz) {
    // MAL: columnas en inner loop (salta en memoria)
    for (size_t i = 0; i < matriz.size(); ++i) {
        for (size_t j = 0; j < matriz[0].size(); ++j) {
            matriz[i][j] = 0;
        }
    }
}

void loop_interchange_bien(std::vector<std::vector<int>>& matriz) {
    // BIEN: filas en inner loop (acceso secuencial en memoria)
    for (size_t i = 0; i < matriz.size(); ++i) {
        int* fila = matriz[i].data();
        for (size_t j = 0; j < matriz[0].size(); ++j) {
            fila[j] = 0;
        }
    }
}

// EJEMPLO 11: VECTORIZACIÓN AUTOMÁTICA
//==============================================================================

/*
  El compilador puede vectorizar bucles (usar SIMD: Single Instruction
  Multiple Data). Condiciones para vectorización:

  1. Bucle con número fijo de iteraciones.
  2. Sin dependencias entre iteraciones.
  3. Acceso a memoria contiguo.
  4. Sin branches (o predecibles).

  g++ -O3 -march=native -ftree-vectorize programa.cpp
*/

// Vectorizable:
void sumar_arrays(const float* a, const float* b, float* c, int n) {
    for (int i = 0; i < n; ++i) {
        c[i] = a[i] + b[i]; // Contiguo, sin dependencias
    }
}

// NO vectorizable (dependencia):
void suma_acumulada(float* a, int n) {
    for (int i = 1; i < n; ++i) {
        a[i] += a[i - 1]; // Dependencia: a[i] depende de a[i-1]
    }
}

// Manual con SIMD intrinsics (x86):
void sumar_simd(const float* a, const float* b, float* c, int n) {
    int i = 0;
    // Procesar de a 4 floats (128 bits SSE) (o 8 con AVX)
    for (; i + 4 <= n; i += 4) {
        __m128 va = _mm_loadu_ps(a + i);
        __m128 vb = _mm_loadu_ps(b + i);
        __m128 vc = _mm_add_ps(va, vb);
        _mm_storeu_ps(c + i, vc);
    }
    // Procesar el resto
    for (; i < n; ++i) {
        c[i] = a[i] + b[i];
    }
}

/*
  Costo de cada mecanismo de polimorfismo:

  • Virtual (vtable): 1 desreferencia de puntero + indirección.
    ~2-5 ns de overhead por llamada.

  • CRTP (static polymorphism): 0 overhead — todo en línea.
    ~0 ns de overhead.

  • std::function: Type erasure + posible heap allocation.
    ~10-30 ns de overhead por llamada.

  • Function pointer: Indirección simple.
    ~1-3 ns de overhead.
*/

// Virtual
struct Animal {
    virtual void sonido() const = 0;
    virtual ~Animal() = default;
};

struct Perro : Animal {
    void sonido() const override { std::cout << "Guau\n"; }
};

// CRTP (ver capítulo 72)
template<typename T>
struct AnimalCRTP {
    void sonido() const {
        static_cast<const T*>(this)->sonido_impl();
    }
};

struct PerroCRTP : AnimalCRTP<PerroCRTP> {
    void sonido_impl() const { std::cout << "Guau\n"; }
};

// std::function
using AnimalFunc = std::function<void()>;

/*
  P: ¿Debo usar -O3 o -O2 en producción?
  R: Generalmente -O2 es suficiente y más seguro. -O3 puede
     aumentar el tamaño del binario y en raras ocasiones hacer
     más lento por code bloat. Mide ambos.

  P: ¿Move semantics realmente mejora el rendimiento?
  R: Sí, especialmente con contenedores grandes (vector, string,
     map). Moves transfieren recursos sin copiar. Hasta 10-100x
     más rápido para datos grandes.

  P: ¿Cómo sé si mi código es cache-friendly?
  R: Con perf stat -e cache-misses ./programa. Si hay muchos
     cache misses, revisa el patrón de acceso a memoria.

  P: ¿Debo optimizar cada función?
  R: No. El 90% del tiempo de ejecución suele estar en el 10%
     del código. Usa un profiler para encontrar los hotspots.

  P: ¿Qué es PGO (Profile Guided Optimization)?
  R: Compilas dos veces: la primera genera un perfil, la segunda
     usa ese perfil para optimizar. g++ -fprofile-generate run
     → g++ -fprofile-use. Puede dar 10-20% de mejora adicional.

  P: ¿std::vector<bool> es eficiente?
  R: std::vector<bool> es una especialización que usa bits.
     Es eficiente en espacio pero lento en acceso. Para velocidad
     usa std::vector<char> o std::bitset.

  R: Unique_ptr vs raw pointer para rendimiento?
  R: Unique_ptr tiene CERO overhead en tiempo de ejecución.
     Es idéntico a un puntero raw. No hay excusa para no usarlo.

  P: ¿Templates hacen el código más rápido?
  R: Sí, porque todo se resuelve en compilación y el compilador
     puede inlinear y optimizar agresivamente. Sin overhead de
     vtable o type erasure.
*/

  Contexto:
  Eres el ingeniero de performance de un motor de búsqueda en
  tiempo real. El sistema debe procesar 1 millón de consultas

  Código base (simplificado):

  struct Documento {
      int id;
      std::string titulo;
      std::string contenido;
      double relevancia;
  };

  class MotorBusqueda {
      std::vector<Documento> docs_;
  public:
      void agregar(Documento d) { docs_.push_back(std::move(d)); }

      std::vector<int> buscar(const std::string& query) const {
          std::vector<int> resultados;
          for (const auto& doc : docs_) {
              if (doc.contenido.find(query) != std::string::npos) {
                  resultados.push_back(doc.id);
              }
          }
          return resultados;
      }
  };

  Problemas a resolver:
  1. La búsqueda es O(n) y copia strings.
  2. No hay caché de resultados frecuentes.
  3. Las inserciones pueden realocar.
  4. No aprovecha SIMD ni paralelismo.

  Tu misión:
  1. Mide la performance actual (crea un benchmark).
  2. Identifica los 3 cuellos de botella principales.
  3. Implementa optimizaciones (al menos 5).
  4. Mide la mejora y documenta cada cambio.

  Criterios:
  • Mejora de al menos 10x (40 pts)
  • Código optimizado pero legible (30 pts)
  • Benchmark riguroso (30 pts)
