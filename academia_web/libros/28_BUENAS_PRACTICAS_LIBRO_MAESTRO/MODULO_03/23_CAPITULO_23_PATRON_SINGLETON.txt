1. SINGLETON

Garantiza que una clase tenga solo una instancia y proporciona punto de

acceso global. Controversial: a menudo es anti-patrón. Dificulta testing.

Crear dependencia global oculta.



2. IMPLEMENTACIÓN

Constructor privado. Método estático getInstance(). Variable de clase para

instancia única. Thread-safe con doble verificación o __init__ una vez.



3. CRÍTICAS

State global: difícil de testear (mocks). Acoplamiento: todas las clases

dependen del singleton. Oculto: dependencia no explícita. Violación SRP.

Alternativas: inyección de dependencias, factories.



4. CUÁNDO USAR (CON CUIDADO)

Logger (aceptado). Gestor de configuración (usar DI mejor). Pool de

conexiones (usar DI). Cache global. Contadores globales.

EJEMPLO DE CÓDIGO
----------------------------------------

# Singleton en Python
class Singleton:
  _instance = None
  
  def __new__(cls):
    if cls._instance is None:
      cls._instance = super().__new__(cls)
    return cls._instance

# Singleton con metaclase
class SingletonMeta(type):
  _instances = {}
  def __call__(cls, *args, **kwargs):
    if cls not in cls._instances:
      cls._instances[cls] = super().__call__(*args, **kwargs)
    return cls._instances[cls]

class Database(metaclass=SingletonMeta):
  def __init__(self):
    self.connection = create_connection()

# Alternativa: Module como singleton (Python)
# database.py
connection = None
def get_connection():
  global connection
  if connection is None:
    connection = create_connection()
  return connection

# Mejor: Inyeccion de dependencias
class UserService:
  def __init__(self, db):
    self.db = db  # No singleton, inyectado
