CONTENIDO DEL CAPÍTULO
===============================================================================

1. CICLO TDD

RED: escribe test que falla (define comportamiento deseado). GREEN: escribe

código mínimo para pasar test. REFACTOR: mejora código sin romper tests.

Ciclo corto: 1-5 minutos por iteración. Código siempre testeado.



2. BENEFICIOS

Diseño emerge del testing. Código siempre testeable. Documentación viva.

Confianza para refactorizar. Menos bugs. Menos debugging. Ritmo sostenible.



3. TDD NO ES SOLO TESTING

TDD es una técnica de diseño, no solo de testing. Obliga a pensar en API

antes de implementar. Interfaces pequeñas y enfocadas. Mejor diseño.



4. CUÁNDO NO USAR TDD

Prototipos exploratorios. Código de UI compleja. Algoritmos inciertos.

Legacy code sin tests (primero caracterización).

EJEMPLO DE CÓDIGO
----------------------------------------

# TDD en accion: FizzBuzz

# 1. RED: primer test
def test_fizzbuzz_1_returns_1():
  assert fizzbuzz(1) == "1"  # falla (funcion no existe)

# 2. GREEN: codigo minimo
def fizzbuzz(n):
  return "1"

# 3. RED: segundo test
def test_fizzbuzz_2_returns_2():
  assert fizzbuzz(2) == "2"  # falla (siempre retorna "1")

# 4. GREEN: generalizar
def fizzbuzz(n):
  return str(n)

# 5. RED: fizz
def test_fizzbuzz_3_returns_fizz():
  assert fizzbuzz(3) == "Fizz"  # falla

# 6. GREEN
def fizzbuzz(n):
  if n % 3 == 0: return "Fizz"
  return str(n)

# Continuar... hasta completar FizzBuzz completo
