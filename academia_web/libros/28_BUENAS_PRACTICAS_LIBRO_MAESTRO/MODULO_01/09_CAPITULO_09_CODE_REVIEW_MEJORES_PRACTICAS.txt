CONTENIDO DEL CAPÍTULO
===============================================================================

1. ¿POR QUÉ CODE REVIEW?

Encuentra bugs temprano. Comparte conocimiento del código. Mejora calidad.

Consistencia en el equipo. Aprendizaje continuo. Responsabilidad colectiva.



2. QUÉ REVISAR

Lógica: ¿es correcta? Diseño: ¿sigue patrones? Rendimiento: ¿problemas?

Seguridad: ¿vulnerabilidades? Tests: ¿suficientes? Documentación: ¿clara?

Nombres: ¿significativos? Formato: ¿sigue estándares?



3. BUENAS PRÁCTICAS

PRs pequeños (200-400 líneas máximo). Proporcionar contexto en descripción.

Feedback constructivo: 'Sugiero... porque...' No personal: código, no persona.

Enfocarse en lo importante. Usar checklist. Celebrar código bueno.



4. CONDUCTA EN REVIEW

Revisar pronto (mismo día). No aprobar sin entender. Hacer preguntas.

No asumir malas intenciones. Agradecer feedback. No defender código.

Separar estilo (automático con formateador) de lógica.

EJEMPLO DE CÓDIGO
----------------------------------------

# Checklist de code review
# [ ] La logica es correcta?
# [ ] Hay tests?
# [ ] Cumple con los requerimientos?
# [ ] Sigue las guias de estilo?
# [ ] Tiene nombres claros?
# [ ] Esta documentado?
# [ ] Sin vulnerabilidades de seguridad?
# [ ] Rendimiento aceptable?
# [ ] Manejo de errores adecuado?
# [ ] Sin codigo duplicado?

# Feedback constructivo
# Mal: "Esto esta mal"
# Bien: "Sugiero usar async/await aqui para mejorar legibilidad"
# Mal: "Tu codigo es confuso"
# Bien: "Podemos simplificar esta funcion? Tiene muchas responsabilidades"
