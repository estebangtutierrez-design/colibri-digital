CONTENIDO DEL CAPÍTULO
===============================================================================

1. ¿QUÉ DOCUMENTAR?

API pública (usuarios de tu código). Arquitectura (nueva gente en equipo).

Setup (cómo empezar). Decisiones (ADRs: Architecture Decision Records).

No documentar lo obvio. El código fuente es la documentación viva.



2. README IMPRESCINDIBLE

Qué hace el proyecto. Cómo instalarlo. Cómo ejecutarlo. Cómo testearlo.

Dependencias. Variables de entorno. Ejemplos de uso. Enlaces a más docs.

Badges: build status, coverage, version, license.



3. ADRs (ARCHITECTURE DECISION RECORDS)

Formato: título, status, contexto, decisión, consecuencias. Fechados.

Versionados en Git. Explícitos. Reversibles. Ejemplo: usar PostgreSQL

versus MongoDB. Motivos y trade-offs documentados.



4. HERRAMIENTAS

Markdown para documentación general. JSDoc/Python docstrings para API.

Swagger/OpenAPI para APIs REST. Storybook para componentes. PlantUML

para diagramas. C4 model para arquitectura.
