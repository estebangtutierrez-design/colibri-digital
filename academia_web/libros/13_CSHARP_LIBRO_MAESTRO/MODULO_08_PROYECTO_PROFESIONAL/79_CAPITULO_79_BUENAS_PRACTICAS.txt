  1. CONVENCIONES DE NOMENCLATURA (MICROSOFT)
  ---------------------------------------------
  Microsoft define guías de nomenclatura para .NET:

    • PascalCase: clases, métodos, propiedades, eventos, namespaces.
      Ej: ProductoService, ObtenerPorId, NombreCompleto.
    • camelCase: parámetros, variables locales, campos privados.
      Ej: nombreCompleto, totalPrecio.
    • _camelCase: campos privados (con guion bajo).
      Ej: _context, _repositorio.
    • MAYUSCULAS: constantes (opcional, no obligatorio).
      Ej: const int MAXIMO_INTENTOS = 3.
    • Interfaces: prefijo I.
      Ej: IProductoRepositorio, IEmailService.
    • Tipos genéricos: T, TEntity, TKey.
    • Async methods: sufijo Async.
      Ej: GetByIdAsync, SaveChangesAsync.

  2. PRINCIPIOS SOLID
  --------------------
    S — Single Responsibility: una clase, una razón de cambio.
    O — Open/Closed: abierto a extensión, cerrado a modificación.
    L — Liskov Substitution: subtipos deben ser sustituibles.
    I — Interface Segregation: interfaces pequeñas y específicas.
    D — Dependency Inversion: depender de abstracciones.

  3. PATRONES DE DISEÑO ESENCIALES
  -----------------------------------
    • Repository: abstrae el acceso a datos.
    • Unit of Work: agrupa operaciones en una transacción.
    • Singleton: una instancia global.
    • Factory: crea objetos sin especificar la clase concreta.
    • Strategy: selecciona un algoritmo en tiempo de ejecución.
    • Observer: notifica cambios a múltiples objetos.
    • Decorator: añade responsabilidades dinámicamente.

  4. CODE ANALYSIS
  ------------------
  .NET tiene analizadores integrados (Roslyn analyzers) que detectan:
    • Posibles NullReferenceException.
    • Async methods sin await.
    • Disposición incorrecta de recursos.
    • Convenciones de nomenclatura violadas.

  Paquetes recomendados:
    • Microsoft.CodeAnalysis.NetAnalyzers
    • StyleCop.Analyzers
    • SonarAnalyzer.CSharp

  5. XML COMMENTS

  // O — Open/Closed
  public abstract class DescuentoStrategy
  {
      public abstract decimal Calcular(decimal monto);
  }

  public class DescuentoNormal : DescuentoStrategy
  {
      public override decimal Calcular(decimal monto) => monto * 0.05m;
  }

  public class DescuentoPremium : DescuentoStrategy
  {
      public override decimal Calcular(decimal monto) => monto * 0.15m;
  }

  public class CalculadorPrecio
  {
      private readonly DescuentoStrategy _strategy;
      public CalculadorPrecio(DescuentoStrategy strategy) => _strategy = strategy;

      public decimal CalcularPrecioFinal(decimal precioBase)
      {
          return precioBase - _strategy.Calcular(precioBase);
      }
  }

  // L — Liskov Substitution
  public class Rectangulo
  {
      public virtual int Alto { get; set; }
      public virtual int Ancho { get; set; }

  Ejemplo 5: Null safety y pattern matching moderno
  ------------------------------------------------------------------------------

  using System;

  public class NullSafetyEjemplos
  {
      // Null-conditional operator
      public string? ObtenerNombreProducto(Producto? p)
          => p?.Nombre?.ToUpper();

      // Null-coalescing operator
      public string ObtenerNombreSeguro(Producto? p)
          => p?.Nombre ?? "Producto sin nombre";

      // Null-coalescing assignment
      private List<Producto>? _cache;
      public List<Producto> Cache => _cache ??= ObtenerProductos();

      // Pattern matching
      public string DescribirProducto(object obj) => obj switch
      {
          Producto p when p.Precio > 1000 => $"Producto premium: {p.Nombre}",
          Producto p => $"Producto estándar: {p.Nombre}",
          null => "Objeto nulo",
          _ => "Tipo desconocido"
      };

      // Property patterns
      public bool EsProductoElegible(Producto p)
          => p is { Activo: true, Stock: > 0, Precio: > 0 };

      // List patterns (.NET 8+)
      public string EvaluarLista(int[] numeros) => numeros switch
      {
          [] => "Lista vacía",
          [1, 2, ..] => "Empieza con 1, 2",
          [.., 0] => "Termina en 0",
          _ => "Otro patrón"
      };

      private List<Producto> ObtenerProductos() => new();
  }

  Ejemplo 6: Gestión de errores consistente
  ------------------------------------------------------------------------------

  using System.Net;
  using System.Text.Json;

  public class ErrorResponse
  {
      public int StatusCode { get; set; }
      public string Message { get; set; } = string.Empty;
      public string? Detail { get; set; }
      public List<string> Errors { get; set; } = new();
      public DateTime Timestamp { get; set; } = DateTime.UtcNow;
  }

  public class ExceptionMiddleware
  {
      private readonly RequestDelegate _next;
      private readonly ILogger<ExceptionMiddleware> _logger;

      public ExceptionMiddleware(RequestDelegate next, ILogger<ExceptionMiddleware> logger)
      {
          _next = next;
          _logger = logger;
      }

      public async Task InvokeAsync(HttpContext context)
      {
          try
          {
              await _next(context);
          }
          catch (KeyNotFoundException ex)
          {
              _logger.LogWarning(ex, "Recurso no encontrado");
              await HandleExceptionAsync(context, HttpStatusCode.NotFound, ex);
          }
          catch (ArgumentException ex)
          {
              _logger.LogWarning(ex, "Argumento inválido");
              await HandleExceptionAsync(context, HttpStatusCode.BadRequest, ex);
          }
          catch (UnauthorizedAccessException ex)
          {
              _logger.LogWarning(ex, "Acceso no autorizado");
              await HandleExceptionAsync(context, HttpStatusCode.Forbidden, ex);
          }
          catch (Exception ex)
          {
              _logger.LogError(ex, "Error interno");
              await HandleExceptionAsync(context, HttpStatusCode.InternalServerError, ex);
          }
      }

      private static async Task HandleExceptionAsync(
          HttpContext context, HttpStatusCode statusCode, Exception ex)
      {
          var response = new ErrorResponse
          {
              StatusCode = (int)statusCode,
              Message = statusCode == HttpStatusCode.InternalServerError
                  ? "Ocurrió un error interno. Intente nuevamente."
                  : ex.Message
          };

          context.Response.ContentType = "application/json";
          context.Response.StatusCode = (int)statusCode;
          await context.Response.WriteAsync(JsonSerializer.Serialize(response));
      }
  }

  Ejemplo 9: Result pattern (evitar excepciones para control de flujo)
  ------------------------------------------------------------------------------

  public class Result<T>
  {
      public bool IsSuccess { get; }
      public bool IsFailure => !IsSuccess;
      public T? Value { get; }
      public string? ErrorMessage { get; }

      private Result(T value) { IsSuccess = true; Value = value; }
      private Result(string errorMessage) { IsSuccess = false; ErrorMessage = errorMessage; }

      public static Result<T> Success(T value) => new(value);
      public static Result<T> Failure(string error) => new(error);

      public TResult Match<TResult>(Func<T, TResult> onSuccess, Func<string, TResult> onFailure)
          => IsSuccess ? onSuccess(Value!) : onFailure(ErrorMessage!);
  }

  // Uso:
  public class ProductoServiceConResult
  {
      public async Task<Result<ProductoDTO>> ObtenerPorIdAsync(int id)
      {
          if (id <= 0)
              return Result<ProductoDTO>.Failure("El ID debe ser mayor a 0");

          var producto = await _repo.GetByIdAsync(id);
          if (producto == null)
              return Result<ProductoDTO>.Failure("Producto no encontrado");

          return Result<ProductoDTO>.Success(MapToDTO(producto));
      }
  }

================================================================================
  [PREGUNTAS FRECUENTES — FAQ]
================================================================================

  P: ¿Debo documentar todo con XML Comments?
  R: Documenta siempre las APIs públicas. El código interno bien escrito
     se documenta solo (nombres claros, métodos pequeños).

  P: ¿Qué hago si heredo código que no sigue las convenciones?
  R: Refactoriza gradualmente. No reescribas todo de golpe. Sigue la regla
     del boy scout: "deja el campamento más limpio de como lo encontraste".

  P: ¿Son obligatorios los patrones de diseño?
  R: No. Usa patrones solo cuando resuelvan un problema. Forzar patrones
     innecesariamente empeora el código.

  P: ¿Qué nivel de cobertura de código es aceptable?
  R: > 70% es bueno, > 80% es excelente. Pero no sacrifiques calidad de
     prueba por cobertura: prefiere pruebas significativas.

  P: ¿Cómo manejar versiones de NuGet?
  R: Usa Central Package Management y fija versiones. Revisa periódicamente
     con dotnet list package --outdated.
