================================================================================
  BIENVENIDA
================================================================================

Imagina que conduces un coche. No necesitas saber cómo funciona el motor
para acelerar; solo pisas el pedal. El motor está encapsulado debajo del
capó. El fabricante decidió qué exponer (el pedal, el volante) y qué ocultar
(los cilindros, la inyección de combustible).

De igual forma, en POO la ENCAPSULACIÓN es el mecanismo que oculta los
detalles internos de una clase y expone solo lo necesario mediante una
interfaz pública controlada.

La encapsulación nos permite:
  • Proteger la integridad de los datos.
  • Cambiar la implementación interna sin afectar a los usuarios de la clase.
  • Reducir la complejidad al exponer solo lo relevante.
  • Aumentar la mantenibilidad y la seguridad del código.

En este capítulo aprenderemos los modificadores de acceso de C#, cómo
encapsular campos, y las buenas prácticas para diseñar clases con una
frontera clara entre lo público y lo privado.

--------------------------------------------------------------------
  1. ¿QUÉ ES LA ENCAPSULACIÓN?
--------------------------------------------------------------------

La encapsulación es el pilar de la POO que consiste en:

  • OCULTAR el estado interno (datos) de un objeto.
  • EXPONER una interfaz pública controlada (métodos, propiedades).
  • PROTEGER la integridad de los datos mediante validación.
  • PERMITIR cambios internos sin afectar el código externo.

Ventajas concretas:

  ✓ Control de acceso:   No todo el mundo puede modificar todo.
  ✓ Validación:          Los datos se validan antes de asignarse.
  ✓ Flexibilidad:        Puedes cambiar la implementación interna
                          sin romper el código cliente.
  ✓ Seguridad:           Datos sensibles no se exponen innecesariamente.

```csharp
// SIN encapsulación (campo público)
public class CuentaInsegura
{
    public double saldo;
}

// CON encapsulación (campo privado + método público)
public class CuentaSegura
{
    private double _saldo;

    public void Depositar(double monto)
    {
        if (monto <= 0)
            throw new ArgumentException("El monto debe ser positivo.");
        _saldo += monto;
    }

    public bool Retirar(double monto)
    {
        if (monto <= 0 || monto > _saldo)
            return false;
        _saldo -= monto;
        return true;
    }

    public double ObtenerSaldo()
    {
        return _saldo;
    }
}
```

--------------------------------------------------------------------
  2. MODIFICADORES DE ACCESO EN C#
--------------------------------------------------------------------

C# proporciona 6 modificadores de acceso:

┌─────────────────────┬─────────────────────────────────────────────┐
│   MODIFICADOR       │               ACCESO                        │
├─────────────────────┼─────────────────────────────────────────────┤
│ public              │ Accesible desde cualquier parte.             │
├─────────────────────┼─────────────────────────────────────────────┤
│ private             │ Accesible solo desde la misma clase.        │
├─────────────────────┼─────────────────────────────────────────────┤
│ protected           │ Accesible desde la misma clase y clases     │
│                     │ derivadas.                                   │
├─────────────────────┼─────────────────────────────────────────────┤
│ internal            │ Accesible desde cualquier clase del mismo   │
│                     │ ensamblado (.dll/.exe).                      │
├─────────────────────┼─────────────────────────────────────────────┤
│ protected internal  │ Accesible desde clases derivadas O desde   │
│                     │ cualquier clase del mismo ensamblado.       │
├─────────────────────┼─────────────────────────────────────────────┤
│ private protected   │ Accesible solo desde clases derivadas      │
│                     │ DENTRO del mismo ensamblado (C# 7.2+).      │
└─────────────────────┴─────────────────────────────────────────────┘

Jerarquía de accesibilidad (de menos a más restrictivo):

  public > protected internal > internal > protected > private protected > private

```csharp
public class DemoModificadores
{
    public int Publico;
    private int Privado;
    protected int Protegido;
    internal int Interno;
    protected internal int ProtegidoInterno;
    private protected int PrivadoProtegido;

    public void Demo()
    {
        Publico = 1;
        Privado = 2;
        Protegido = 3;
        Interno = 4;
        ProtegidoInterno = 5;
        PrivadoProtegido = 6;
    }
}

public class OtraClase
{
    public void Acceso()
    {
        DemoModificadores demo = new DemoModificadores();
        demo.Publico = 1;
        // demo.Privado = 2;        // ERROR
        // demo.Protegido = 3;      // ERROR
        demo.Interno = 4;
        demo.ProtegidoInterno = 5;
        // demo.PrivadoProtegido = 6; // ERROR
    }
}

public class Derivada : DemoModificadores
{
    public void AccesoDerivado()
    {
        Publico = 1;
        // Privado = 2;         // ERROR
        Protegido = 3;
        Interno = 4;
        ProtegidoInterno = 5;
        PrivadoProtegido = 6;
    }
}
```

--------------------------------------------------------------------
  3. MODIFICADOR PREDETERMINADO
--------------------------------------------------------------------

Si no se especifica un modificador para un miembro de clase, el
predeterminado es `private`.

```csharp
public class Ejemplo
{
    int campo1;          // private
    private int campo2;  // private explícito
    public int campo3;   // public
}

// Para clases de nivel superior, el predeterminado es internal:
class ClaseInternal { }       // internal
public class ClasePublic { }  // public explícito
```

--------------------------------------------------------------------
  4. ENCAPSULACIÓN CON PROPIEDADES
--------------------------------------------------------------------

La forma más común y recomendada de encapsular campos en C# es mediante
PROPIEDADES (getters y setters).

En lugar de exponer un campo público, exponemos una propiedad que controla
el acceso con validación:

```csharp
// MAL: campo público
public class Producto
{
    public decimal precio;
}

// BIEN: encapsulación con propiedad
public class Producto
{
    private decimal _precio;

    public decimal Precio
    {
        get { return _precio; }
        set
        {
            if (value < 0)
                throw new ArgumentException("El precio no puede ser negativo.");
            _precio = value;
        }
    }
}
```

Las propiedades pueden tener distintos modificadores en get y set:

```csharp
public class Cuenta
{
    private decimal _saldo;

    public decimal Saldo
    {
        get { return _saldo; }
        private set { _saldo = value; }
    }

    public void Depositar(decimal monto)
    {
        if (monto <= 0) throw new ArgumentException("...");
        _saldo += monto;
    }
}

// Uso:
Cuenta c = new Cuenta();
c.Depositar(100);
decimal s = c.Saldo;       // OK
// c.Saldo = 50;           // ERROR: set privado
```

--------------------------------------------------------------------
  5. PRINCIPIO DE MÍNIMA EXPOSICIÓN
--------------------------------------------------------------------

"Expón lo mínimo necesario, oculta todo lo demás."

Reglas prácticas:

  • Campos: siempre privados.
  • Propiedades: públicas solo si es necesario.
  • Métodos: públicos solo si son parte de la API.
  • Ayudantes internos: privados o protected.
  • Constantes: pueden ser públicas (son inmutables).

```csharp
public class Calculadora
{
    public const double PI = 3.1415926535;

    private double _ultimoResultado;

    public double UltimoResultado => _ultimoResultado;

    public double Sumar(double a, double b)
    {
        _ultimoResultado = a + b;
        return _ultimoResultado;
    }

    private void RegistrarOperacion(string operacion)
    {
        Console.WriteLine($"[LOG] {operacion} = {_ultimoResultado}");
    }
}
```

--------------------------------------------------------------------
  6. INTERNAL Y ENSAMBLADOS
--------------------------------------------------------------------

`internal` permite acceso dentro del mismo proyecto/ensamblado pero no
desde proyectos externos.

```csharp
// Solo visible dentro de este proyecto
internal class ProcesadorInterno
{
    public void Procesar()
    {
        Console.WriteLine("Procesando...");
    }
}

public class ServicioPublico
{
    public void Ejecutar()
    {
        var procesador = new ProcesadorInterno();
        procesador.Procesar();
    }
}
```

Para exponer a proyectos de testing:
  [assembly: InternalsVisibleTo("MiProyecto.Tests")]

--------------------------------------------------------------------
  7. PROTECTED EN HERENCIA
--------------------------------------------------------------------

`protected` permite que las clases derivadas accedan a miembros que el
resto del mundo no puede ver.

```csharp
public class Animal
{
    private string _nombre;
    protected int _edad;
    public string Especie { get; set; }

    protected void Dormir() { }
    public void Comer() { }
}

public class Perro : Animal
{
    public void Ladrar()
    {
        // _nombre = "Fido";    // ERROR
        _edad = 5;              // OK
        Especie = "Canino";     // OK
        Dormir();               // OK
        Comer();                // OK
    }
}
```

--------------------------------------------------------------------
  8. PROTECTED INTERNAL vs PRIVATE PROTECTED
--------------------------------------------------------------------

  • protected internal: clases derivadas (cualquier ensamblado)
    O cualquier clase del mismo ensamblado.

  • private protected: solo clases derivadas del mismo ensamblado
    (C# 7.2+).

```csharp
// Ensamblado A:
public class Base
{
    protected internal int Pi;
    private protected int Pp;
}

// Ensamblado A, clase NO derivada:
public class Otra
{
    void Test()
    {
        Base b = new Base();
        b.Pi = 1;     // OK (mismo ensamblado)
        // b.Pp = 2;  // ERROR
    }
}

// Ensamblado A, clase derivada:
public class DerivadaA : Base
{
    void Test()
    {
        Pi = 1;  // OK
        Pp = 2;  // OK (mismo ensamblado, derivada)
    }
}

// Ensamblado B, clase derivada:
public class DerivadaB : Base
{
    void Test()
    {
        Pi = 1;  // OK (derivada, cualquier ensamblado)
        // Pp = 2;  // ERROR (diferente ensamblado)
    }
}
```

--------------------------------------------------------------------
  9. VALIDACIÓN EN PROPIEDADES
--------------------------------------------------------------------

Las propiedades permiten validar los valores antes de asignarlos:

```csharp
public class Persona
{
    private string _nombre;
    private int _edad;

    public string Nombre
    {
        get => _nombre;
        set
        {
            if (string.IsNullOrWhiteSpace(value))
                throw new ArgumentException("El nombre no puede estar vacio.");
            if (value.Length < 2)
                throw new ArgumentException("El nombre debe tener al menos 2 caracteres.");
            _nombre = value;
        }
    }

    public int Edad
    {
        get => _edad;
        set
        {
            if (value < 0 || value > 150)
                throw new ArgumentOutOfRangeException(nameof(Edad),
                    "La edad debe estar entre 0 y 150.");
            _edad = value;
        }
    }

    public string Email { get; set; }
}
```

--------------------------------------------------------------------
  10. READONLY Y ENCAPSULACIÓN
--------------------------------------------------------------------

El modificador `readonly` permite que un campo solo se asigne en la
declaración o en el constructor. Es una forma de encapsulación que
garantiza inmutabilidad.

```csharp
public class Configuracion
{
    public readonly string Servidor;
    public readonly int Puerto;

    private readonly DateTime _fechaCreacion;

    public Configuracion(string servidor, int puerto)
    {
        Servidor = servidor;
        Puerto = puerto;
        _fechaCreacion = DateTime.Now;
    }

    public DateTime FechaCreacion => _fechaCreacion;
}
```

--------------------------------------------------------------------
  11. EJEMPLO COMPLETO: SISTEMA DE CUENTAS BANCARIAS
--------------------------------------------------------------------

```csharp
using System;

namespace SistemaBancario
{
    public class CuentaBancaria
    {
        private static int _contador = 1000;
        private string _titular;
        private decimal _saldo;
        private string _tipo;
        private bool _activa;
        private readonly int _numeroCuenta;
        private readonly DateTime _fechaCreacion;

        public CuentaBancaria(string titular, decimal saldoInicial, string tipo)
        {
            if (string.IsNullOrWhiteSpace(titular))
                throw new ArgumentException("El titular es obligatorio.");
            if (saldoInicial < 0)
                throw new ArgumentException("El saldo inicial no puede ser negativo.");
            if (tipo != "Ahorros" && tipo != "Corriente")
                throw new ArgumentException("Tipo debe ser Ahorros o Corriente.");

            _titular = titular;
            _saldo = saldoInicial;
            _tipo = tipo;
            _activa = true;
            _numeroCuenta = ++_contador;
            _fechaCreacion = DateTime.Now;
        }

        public int NumeroCuenta => _numeroCuenta;
        public string Titular
        {
            get => _titular;
            set
            {
                if (string.IsNullOrWhiteSpace(value))
                    throw new ArgumentException("El titular no puede estar vacio.");
                _titular = value;
            }
        }

        public decimal Saldo => _saldo;
        public string Tipo => _tipo;
        public DateTime FechaCreacion => _fechaCreacion;
        public bool Activa => _activa;

        public string Estado
        {
            get
            {
                if (!_activa) return "Inactiva";
                if (_saldo < 0) return "Sobregiro";
                if (_saldo == 0) return "Cero";
                return "Activa";
            }
        }

        public void Depositar(decimal monto)
        {
            if (!_activa)
                throw new InvalidOperationException("Cuenta inactiva.");
            if (monto <= 0)
                throw new ArgumentException("El monto a depositar debe ser positivo.");

            _saldo += monto;
            Console.WriteLine($"Deposito de ${monto:F2}. Nuevo saldo: ${_saldo:F2}");
        }

        public bool Retirar(decimal monto)
        {
            if (!_activa)
                throw new InvalidOperationException("Cuenta inactiva.");
            if (monto <= 0)
                throw new ArgumentException("El monto a retirar debe ser positivo.");

            if (monto > _saldo)
            {
                Console.WriteLine($"Fondos insuficientes. Saldo: ${_saldo:F2}");
                return false;
            }

            _saldo -= monto;
            Console.WriteLine($"Retiro de ${monto:F2}. Nuevo saldo: ${_saldo:F2}");
            return true;
        }

        public void Desactivar()
        {
            _activa = false;
            Console.WriteLine("Cuenta desactivada.");
        }

        public void MostrarInfo()
        {
            Console.WriteLine("=================================");
            Console.WriteLine($"Cuenta #{_numeroCuenta}");
            Console.WriteLine($"Titular: {_titular}");
            Console.WriteLine($"Tipo: {_tipo}");
            Console.WriteLine($"Saldo: ${_saldo:F2}");
            Console.WriteLine($"Estado: {Estado}");
            Console.WriteLine($"Creada: {_fechaCreacion:dd/MM/yyyy}");
            Console.WriteLine("=================================");
        }
    }

    public class Programa
    {
        public static void Main()
        {
            try
            {
                CuentaBancaria c1 = new CuentaBancaria("Ana Lopez", 5000, "Ahorros");
                CuentaBancaria c2 = new CuentaBancaria("Carlos Ruiz", 10000, "Corriente");
                CuentaBancaria c3 = new CuentaBancaria("Maria Garcia", 0, "Ahorros");

                c1.MostrarInfo();
                c2.MostrarInfo();

                c1.Depositar(1500);
                c1.Retirar(2000);
                c1.Retirar(10000);

                c2.Titular = "Carlos Ruiz M.";
                Console.WriteLine($"Nuevo titular: {c2.Titular}");

                Console.WriteLine($"Estado de c3: {c3.Estado}");
                c3.Desactivar();
                Console.WriteLine($"Estado de c3: {c3.Estado}");
            }
            catch (Exception ex)
            {
                Console.WriteLine($"Error: {ex.Message}");
            }
        }
    }
}
```

  12. COMPARATIVA: CAMPOS vs PROPIEDADES
--------------------------------------------------------------------

┌─────────────────────────┬──────────────────────┬─────────────────────┐
│      CARACTERÍSTICA      │       CAMPO           │     PROPIEDAD       │
├─────────────────────────┼──────────────────────┼─────────────────────┤
│ Encapsulación            │ No (público)          │ Sí (get/set)        │
│ Validación               │ No                    │ Sí (en set)         │
│ Control de acceso        │ Un solo nivel         │ Get y set distintos │
│ Sintaxis                 │ tipo nombre;          │ tipo Nombre {get;set}│
│ Puede ser virtual        │ No                    │ Sí                  │
│ Puede ser abstract       │ No                    │ Sí (en interfaces)  │
│ Puede tener lógica       │ No                    │ Sí                  │
│ Inicializadores          │ Sí                    │ Sí (auto-prop)      │
│ Binding (WPF/WinForms)   │ No                    │ Sí                  │
└─────────────────────────┴──────────────────────┴─────────────────────┘

--------------------------------------------------------------------
  13. BUENAS PRÁCTICAS DE ENCAPSULACIÓN
--------------------------------------------------------------------

  ✓ Usa siempre propiedades en lugar de campos públicos.
  ✓ Valida en el setter de las propiedades.
  ✓ Usa getter privado + setter público (o viceversa) según necesidad.
  ✓ Prefiere inmutabilidad: get-only + constructor.
  ✓ Usa readonly para campos que no deben cambiar.
  ✓ No expongas colecciones como arrays o List (usa IEnumerable, IReadOnlyList).
  ✓ Mantén la clase "cerrada" por defecto; solo abre lo necesario.

```csharp
public class Pedido
{
    private readonly List<string> _productos = new List<string>();

    public string Cliente { get; }
    public IReadOnlyList<string> Productos => _productos.AsReadOnly();

    public Pedido(string cliente)
    {
        Cliente = cliente ?? throw new ArgumentNullException(nameof(cliente));
    }

    public void AgregarProducto(string producto)
    {
        if (string.IsNullOrWhiteSpace(producto))
            throw new ArgumentException("Producto invalido.");
        _productos.Add(producto);
    }
}
```

--------------------------------------------------------------------
  14. INMUTABILIDAD Y ENCAPSULACIÓN
--------------------------------------------------------------------

La inmutabilidad es una forma extrema de encapsulación: el objeto no puede
cambiar después de creado.

```csharp
public class PuntoInmutable
{
    public double X { get; }
    public double Y { get; }

    public PuntoInmutable(double x, double y)
    {
        X = x;
        Y = y;
    }

    public PuntoInmutable Desplazar(double dx, double dy)
    {
        return new PuntoInmutable(X + dx, Y + dy);
    }

    public override string ToString() => $"({X}, {Y})";
}

// Uso:
PuntoInmutable p1 = new PuntoInmutable(3, 4);
PuntoInmutable p2 = p1.Desplazar(1, 2);
Console.WriteLine(p1);  // (3, 4)
Console.WriteLine(p2);  // (4, 6)
```

  RESUMEN DEL CAPÍTULO
================================================================================

  • La ENCAPSULACIÓN oculta los detalles internos de una clase y expone
    solo lo necesario mediante una interfaz pública controlada.

  • Los MODIFICADORES DE ACCESO en C# son: public, private, protected,
    internal, protected internal, private protected.

  • El modificador PREDETERMINADO para miembros es `private`.

  • Las PROPIEDADES son la forma correcta de encapsular campos, permitiendo
    validación y control de acceso diferenciado (get/set).

  • El PRINCIPIO DE MÍNIMA EXPOSICIÓN dice: expón lo mínimo necesario.

  • `internal` restringe el acceso al mismo ensamblado.

  • `readonly` garantiza que un campo solo se asigne en el constructor.

  • La inmutabilidad (propiedades get-only) es una forma de encapsulación
    que impide cambios después de la creación.

================================================================================
  PREGUNTAS FRECUENTES (FAQ)
================================================================================

  ───
  ❓ ¿Por qué no debo usar campos públicos?
  ───
  Los campos públicos rompen la encapsulación. Cualquier código puede
  modificar el estado interno sin validación. Las propiedades permiten
  control y flexibilidad futura.

  ───
  ❓ ¿Puedo tener una propiedad sin setter?
  ───
  Sí. Una propiedad get-only se usa para datos que solo se asignan en
  el constructor (inmutabilidad).

  ───
  ❓ ¿Qué diferencia hay entre readonly y get-only?
  ───
  readonly es para campos (private readonly int _x). Get-only es para
  propiedades (public int X { get; }). Son conceptos distintos pero
  relacionados: readonly garantiza que el campo solo se asigne en el
  constructor.

  ───
  ❓ ¿Cuándo usar protected vs internal?
  ───
  protected: cuando quieres que las clases derivadas accedan al miembro
  (incluso desde otros ensamblados). internal: cuando quieres compartir
  código dentro del mismo proyecto pero no exponerlo externamente.

  ───
  ❓ ¿Qué es la "propiedad auto-implementada"?
  ───
  Es una sintaxis abreviada: public string Nombre { get; set; }. El
  compilador genera automáticamente un campo privado de respaldo.

  ───
  ❓ ¿Puedo tener set público y get privado?
  ───
  Sí, aunque es poco común: public string Nombre { private get; set; }.
  Generalmente se usa get público y set privado.

  ───
  ❓ ¿La encapsulación afecta el rendimiento?
  ───
  Las propiedades se compilan como métodos (get_Nombre, set_Nombre) y
  el JIT puede inlinearlos. El impacto es mínimo o nulo.
