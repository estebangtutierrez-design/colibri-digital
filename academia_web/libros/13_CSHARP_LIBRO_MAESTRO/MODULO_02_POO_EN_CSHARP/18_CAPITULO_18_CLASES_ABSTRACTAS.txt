================================================================================
  BIENVENIDA
================================================================================

Imagina que eres un arquitecto y diseñas el plano de una "casa". Pero no
especificas los materiales ni los acabados: solo dices que toda casa debe
tener cimientos, paredes y techo. Eso es una clase abstracta: una clase
que define QUÉ hacer pero no CÓMO hacerlo en todos los detalles.

Las clases abstractas son clases que NO pueden instanciarse directamente.
Sirven como BASE para otras clases, estableciendo un contrato parcial:
algunos métodos tienen implementación y otros se declaran abstractos
(sin implementación) para que las clases derivadas los completen.

En este capítulo exploraremos:

  • Qué son las clases abstractas y por qué existen.
  • Métodos abstractos: declaración sin cuerpo.
  • Diferencias entre abstract y virtual.
  • Cuándo usar clase abstracta vs interfaz vs clase concreta.
  • Patrones de diseño con clases abstractas.
  • Buenas prácticas.

Las clases abstractas son fundamentales para diseñar jerarquías robustas
y extensibles. Dominarlas es señal de un diseñador maduro.

--------------------------------------------------------------------
  1. ¿QUÉ ES UNA CLASE ABSTRACTA?
--------------------------------------------------------------------

Una clase abstracta es una clase que:
  • Se declara con la palabra clave `abstract`.
  • NO se puede instanciar (no puedes hacer `new` de ella).
  • PUEDE contener métodos abstractos (sin implementación).
  • PUEDE contener métodos concretos (con implementación).
  • PUEDE contener campos, propiedades, constructores, etc.
  • SIRVE como base para clases derivadas.

```csharp
public abstract class Forma
{
    // Metodo abstracto (sin implementacion)
    public abstract double CalcularArea();

    // Metodo concreto (con implementacion)
    public void MostrarInfo()
    {
        Console.WriteLine($"Area: {CalcularArea():F2}");
    }
}

// Error: no se puede instanciar una clase abstracta
// Forma f = new Forma();  // ERROR: abstract

// Debe heredarse:
public class Circulo : Forma
{
    public double Radio { get; set; }

    public Circulo(double radio) => Radio = radio;

    public override double CalcularArea()
    {
        return Math.PI * Radio * Radio;
    }
}

// Ahora si podemos:
Circulo c = new Circulo(5);
c.MostrarInfo();  // Area: 78.54
```

--------------------------------------------------------------------
  2. MÉTODOS ABSTRACTOS
--------------------------------------------------------------------

Un método abstracto:
  • Se declara con `abstract`.
  • NO tiene cuerpo (termina en `;`).
  • DEBE ser implementado (override) en todas las clases derivadas
    NO abstractas.
  • Es implícitamente virtual (no se puede combinar abstract y virtual).

```csharp
public abstract class Animal
{
    // Metodo abstracto: las derivadas DEBEN implementarlo
    public abstract void HacerSonido();

    // Propiedad abstracta
    public abstract string Nombre { get; }

    // Metodo concreto
    public void Dormir()
    {
        Console.WriteLine($"{Nombre} esta durmiendo.");
    }
}

public class Perro : Animal
{
    public override string Nombre => "Perro";

    public override void HacerSonido()
    {
        Console.WriteLine("¡Guau!");
    }
}

public class Gato : Animal
{
    public override string Nombre => "Gato";

    public override void HacerSonido()
    {
        Console.WriteLine("¡Miau!");
    }
}
```

Reglas de métodos abstractos:

  ✓ Solo pueden declararse en clases abstractas.
  ✓ No tienen implementación (no hay cuerpo `{}`).
  ✓ Deben ser override en clases concretas.
  ✓ No pueden ser private, static o sealed.
  ✓ Las propiedades también pueden ser abstractas.

--------------------------------------------------------------------
  3. CLASES ABSTRACTAS CON MIEMBROS CONCRETOS
--------------------------------------------------------------------

Las clases abstractas pueden (y suelen) tener métodos concretos que las
derivadas heredan sin necesidad de sobrescribir.

```csharp
public abstract class Vehiculo
{
    // Campos
    protected string _marca;
    protected int _velocidad;

    // Constructor (puede llamarse desde derivadas con : base())
    public Vehiculo(string marca)
    {
        _marca = marca;
        _velocidad = 0;
    }

    // Metodos abstractos
    public abstract void Acelerar(int incremento);
    public abstract void Frenar(int decremento);

    // Metodo concreto (compartido por todos)
    public void MostrarEstado()
    {
        Console.WriteLine($"{_marca}: {_velocidad} km/h");
    }
}

public class Coche : Vehiculo
{
    public Coche(string marca) : base(marca) { }

    public override void Acelerar(int incremento)
    {
        _velocidad += incremento * 2;
        Console.WriteLine($"{_marca}: acelerando a {_velocidad} km/h");
    }

    public override void Frenar(int decremento)
    {
        _velocidad = Math.Max(0, _velocidad - decremento);
        Console.WriteLine($"{_marca}: frenando a {_velocidad} km/h");
    }
}

public class Bicicleta : Vehiculo
{
    public Bicicleta(string marca) : base(marca) { }

    public override void Acelerar(int incremento)
    {
        _velocidad += incremento;
        Console.WriteLine($"{_marca}: pedaleando a {_velocidad} km/h");
    }

    public override void Frenar(int decremento)
    {
        _velocidad = Math.Max(0, _velocidad - decremento);
        Console.WriteLine($"{_marca}: frenando a {_velocidad} km/h");
    }
}
```

--------------------------------------------------------------------
  4. ABSTRACT vs VIRTUAL: COMPARATIVA
--------------------------------------------------------------------

┌─────────────────────────┬────────────────────┬────────────────────────┐
│      CARACTERÍSTICA      │     ABSTRACT        │        VIRTUAL        │
├─────────────────────────┼────────────────────┼────────────────────────┤
│ Tiene implementación     │ No                 │ Sí                    │
│ Obliga a sobrescribir    │ Sí                 │ No                    │
│ Puede estar en clase     │ Solo abstracta     │ Cualquier clase       │
│ concreta                 │                    │                        │
│ Palabra clave            │ abstract           │ virtual               │
│ En derivada se usa       │ override           │ override (opcional)   │
│ Propiedades              │ Sí                 │ Sí                    │
│ Eventos                  │ Sí                 │ Sí                    │
└─────────────────────────┴────────────────────┴────────────────────────┘

```csharp
public abstract class Base
{
    // Abstract: obligatorio implementar
    public abstract void MetodoObligatorio();

    // Virtual: opcional sobrescribir
    public virtual void MetodoOpcional()
    {
        Console.WriteLine("Implementacion por defecto");
    }
}

public class Derivada : Base
{
    // Obligatorio
    public override void MetodoObligatorio()
    {
        Console.WriteLine("Implementacion obligatoria");
    }

    // Opcional (podemos no sobrescribirlo)
    public override void MetodoOpcional()
    {
        Console.WriteLine("Implementacion personalizada");
    }
}
```

¿Cuándo usar abstract vs virtual?

  • ABSTRACT: cuando NO hay una implementación base sensible y cada
    derivada DEBE proporcionar la suya.

  • VIRTUAL: cuando hay una implementación base RAZONABLE que las
    derivadas PUEDEN cambiar si lo necesitan.

--------------------------------------------------------------------
  5. CONSTRUCTORES EN CLASES ABSTRACTAS
--------------------------------------------------------------------

Las clases abstractas pueden tener constructores. Se llaman desde las
clases derivadas con `: base()`.

```csharp
public abstract class Persona
{
    public string Nombre { get; }
    public int Edad { get; }

    public Persona(string nombre, int edad)
    {
        Nombre = nombre ?? throw new ArgumentNullException(nameof(nombre));
        Edad = edad > 0 ? edad : throw new ArgumentException("Edad invalida");
    }

    public abstract string ObtenerDescripcion();
}

public class Empleado : Persona
{
    public string Puesto { get; }

    public Empleado(string nombre, int edad, string puesto)
        : base(nombre, edad)
    {
        Puesto = puesto;
    }

    public override string ObtenerDescripcion()
    {
        return $"{Nombre}, {Edad} años, {Puesto}";
    }
}
```

Los constructores de clases abstractas suelen ser `protected` (no tiene
sentido que sean públicos, ya que no se puede hacer new).

--------------------------------------------------------------------
  6. PROPIEDADES ABSTRACTAS
--------------------------------------------------------------------

Las propiedades también pueden ser abstractas, forzando a las derivadas
a implementar getters y/o setters.

```csharp
public abstract class Figura
{
    public abstract double Area { get; }
    public abstract double Perimetro { get; }
    public abstract string Nombre { get; }

    public void Mostrar()
    {
        Console.WriteLine($"{Nombre}: Area={Area:F2}, Perimetro={Perimetro:F2}");
    }
}

public class Circulo : Figura
{
    public double Radio { get; }

    public Circulo(double radio) => Radio = radio;

    public override double Area => Math.PI * Radio * Radio;
    public override double Perimetro => 2 * Math.PI * Radio;
    public override string Nombre => $"Circulo(R={Radio})";
}
```

--------------------------------------------------------------------
  7. CLASE ABSTRACTA vs INTERFAZ
--------------------------------------------------------------------

Esta es una de las decisiones de diseño más importantes en POO.

┌─────────────────────────┬──────────────────────┬─────────────────────────┐
│      CARACTERÍSTICA      │   CLASE ABSTRACTA     │       INTERFAZ         │
├─────────────────────────┼──────────────────────┼─────────────────────────┤
│ Instanciable             │ No                   │ No                     │
│ Herencia múltiple        │ No (solo una base)   │ Sí (múltiples)         │
│ Métodos con impl.        │ Sí                   │ Sí (C# 8+ default)     │
│ Campos                   │ Sí                   │ No                     │
│ Constructores            │ Sí                   │ No                     │
│ Modificadores de acceso  │ Todos                │ public (implícito)     │
│ Constantes               │ Sí                   │ No                     │
│ Propiedades con respaldo │ Sí                   │ No (solo declaración)  │
│ Indica                   │ "Es-un" con estado   │ "Puede-hacer" (capacid)│
└─────────────────────────┴──────────────────────┴─────────────────────────┘

Regla práctica:
  • Usa CLASE ABSTRACTA cuando las clases derivadas COMPARTAN estado
    (campos) y comportamiento base.
  • Usa INTERFAZ cuando quieras definir una CAPACIDAD o CONTRATO que
    clases NO RELACIONADAS puedan implementar.

```csharp
// Clase abstracta: relacion "es-un" con estado compartido
public abstract class Animal
{
    public string Nombre { get; set; }
    public int Edad { get; set; }
    public abstract void HacerSonido();
}

// Interfaz: capacidad que cualquier clase puede tener
public interface IVolador
{
    void Volar();
}

// Pez no puede heredar de Animal Volador, pero una interfaz lo permite
public class Pez : Animal
{
    public override void HacerSonido() => Console.WriteLine("Glub glub");
}

public class Murcielago : Animal, IVolador
{
    public override void HacerSonido() => Console.WriteLine("Chillido");
    public void Volar() => Console.WriteLine("Volando...");
}
```

--------------------------------------------------------------------
  8. EJEMPLO COMPLETO: SISTEMA DE NOTIFICACIONES
--------------------------------------------------------------------

```csharp
using System;

namespace SistemaNotificaciones
{
    // Clase abstracta base
    public abstract class Notificacion
    {
        public string Destinatario { get; }
        public string Mensaje { get; }
        public DateTime FechaEnvio { get; protected set; }
        public bool Enviada { get; protected set; }

        protected Notificacion(string destinatario, string mensaje)
        {
            Destinatario = destinatario ?? throw new ArgumentNullException(nameof(destinatario));
            Mensaje = mensaje ?? throw new ArgumentNullException(nameof(mensaje));
            FechaEnvio = DateTime.MinValue;
            Enviada = false;
        }

        // Metodos abstractos
        public abstract bool ValidarDestinatario();
        public abstract bool Enviar();
        public abstract string ObtenerResumen();

        // Metodo concreto
        public void RegistrarIntento(bool exitoso)
        {
            if (exitoso)
            {
                Enviada = true;
                FechaEnvio = DateTime.Now;
                Console.WriteLine($"[OK] Notificacion enviada a {Destinatario}");
            }
            else
            {
                Console.WriteLine($"[FALLO] Error al enviar a {Destinatario}");
            }
        }
    }

    public class NotificacionEmail : Notificacion
    {
        public string Asunto { get; }

        public NotificacionEmail(string email, string asunto, string mensaje)
            : base(email, mensaje)
        {
            Asunto = asunto;
        }

        public override bool ValidarDestinatario()
        {
            return Destinatario.Contains("@") && Destinatario.Contains(".");
        }

        public override bool Enviar()
        {
            if (!ValidarDestinatario())
            {
                RegistrarIntento(false);
                return false;
            }

            Console.WriteLine($"Enviando email a {Destinatario}");
            Console.WriteLine($"Asunto: {Asunto}");
            Console.WriteLine($"Cuerpo: {Mensaje}");
            System.Threading.Thread.Sleep(100); // Simular envio

            RegistrarIntento(true);
            return true;
        }

        public override string ObtenerResumen()
        {
            return $"Email -> {Destinatario} | Asunto: {Asunto} | {(Enviada ? $"Enviado {FechaEnvio:dd/MM/yyyy HH:mm}" : "Pendiente")}";
        }
    }

    public class NotificacionSMS : Notificacion
    {
        public NotificacionSMS(string telefono, string mensaje)
            : base(telefono, mensaje)
        {
        }

        public override bool ValidarDestinatario()
        {
            return Destinatario.Length >= 10 && Destinatario.All(char.IsDigit);
        }

        public override bool Enviar()
        {
            if (!ValidarDestinatario())
            {
                RegistrarIntento(false);
                return false;
            }

            Console.WriteLine($"Enviando SMS a {Destinatario}: {Mensaje}");
            System.Threading.Thread.Sleep(50);

            RegistrarIntento(true);
            return true;
        }

        public override string ObtenerResumen()
        {
            return $"SMS -> {Destinatario} | {(Enviada ? $"Enviado {FechaEnvio:dd/MM/yyyy HH:mm}" : "Pendiente")}";
        }
    }

    public class NotificacionPush : Notificacion
    {
        public string DispositivoId { get; }

        public NotificacionPush(string dispositivoId, string mensaje)
            : base(dispositivoId, mensaje)
        {
            DispositivoId = dispositivoId;
        }

        public override bool ValidarDestinatario()
        {
            return !string.IsNullOrWhiteSpace(DispositivoId) && DispositivoId.Length > 5;
        }

        public override bool Enviar()
        {
            if (!ValidarDestinatario())
            {
                RegistrarIntento(false);
                return false;
            }

            Console.WriteLine($"Enviando Push a dispositivo {DispositivoId}: {Mensaje}");
            System.Threading.Thread.Sleep(30);

            RegistrarIntento(true);
            return true;
        }

        public override string ObtenerResumen()
        {
            return $"Push -> {DispositivoId} | {(Enviada ? $"Enviado {FechaEnvio:dd/MM/yyyy HH:mm}" : "Pendiente")}";
        }
    }

    public class Programa
    {
        public static void Main()
        {
            Notificacion[] notificaciones = {
                new NotificacionEmail("ana@email.com", "Bienvenida", "Gracias por registrarte!"),
                new NotificacionSMS("51999887766", "Tu codigo es 1234"),
                new NotificacionPush("DEVICE-ABC-123", "Nuevo mensaje recibido"),
                new NotificacionEmail("invalido", "Test", "No deberia enviarse"),
                new NotificacionSMS("123", "Corto")
            };

            int enviados = 0, fallidos = 0;

            foreach (var notif in notificaciones)
            {
                Console.WriteLine($"\n--- {notif.GetType().Name} ---");
                Console.WriteLine($"Destinatario: {notif.Destinatario}");
                Console.WriteLine($"Valido: {notif.ValidarDestinatario()}");

                if (notif.Enviar())
                    enviados++;
                else
                    fallidos++;

                Console.WriteLine(notif.ObtenerResumen());
            }

            Console.WriteLine($"\n=== RESUMEN ===");
            Console.WriteLine($"Enviadas: {enviados}, Fallidas: {fallidos}");
        }
    }
}
```

  9. CLASES ABSTRACTAS Y EL PATRON TEMPLATE METHOD
--------------------------------------------------------------------

Un patrón clásico que usa clases abstractas es el Template Method:
define el esqueleto de un algoritmo en la clase base (concreto) y
deja que las derivadas implementen pasos específicos (abstractos).

```csharp
public abstract class ReporteGenerator
{
    // Template Method (define el esqueleto)
    public void GenerarReporte()
    {
        Console.WriteLine("=== INICIO REPORTE ===");
        Cabecera();
        Cuerpo();
        Pie();
        Console.WriteLine("=== FIN REPORTE ===");
    }

    // Pasos abstractos (las derivadas los implementan)
    protected abstract void Cabecera();
    protected abstract void Cuerpo();
    protected abstract void Pie();
}

public class ReportePDF : ReporteGenerator
{
    protected override void Cabecera()
    {
        Console.WriteLine("[PDF] Cabecera del reporte");
    }

    protected override void Cuerpo()
    {
        Console.WriteLine("[PDF] Contenido del reporte");
    }

    protected override void Pie()
    {
        Console.WriteLine("[PDF] Pie de pagina");
    }
}

public class ReporteHTML : ReporteGenerator
{
    protected override void Cabecera()
    {
        Console.WriteLine("<header>Cabecera HTML</header>");
    }

    protected override void Cuerpo()
    {
        Console.WriteLine("<main>Contenido HTML</main>");
    }

    protected override void Pie()
    {
        Console.WriteLine("<footer>Pie HTML</footer>");
    }
}
```

  10. BUENAS PRÁCTICAS
--------------------------------------------------------------------

  ✓ Usa clases abstractas cuando hay una relación "es-un" clara y las
    derivadas comparten estado o comportamiento.

  ✓ Diseña la clase abstracta pensando en cómo será extendida.

  ✓ Documenta qué métodos abstractos deben implementar las derivadas
    y qué se espera de ellos.

  ✓ Los constructores de clases abstractas deben ser protected.

  ✓ No hagas una clase abstracta si solo tiene métodos abstractos
    (en ese caso, prefiere una interfaz).

  ✓ Limita el número de métodos abstractos (3-5 máximo) para no
    sobrecargar a las derivadas.

  ✓ Usa el patrón Template Method para algoritmos reutilizables.

  ✓ No instancies clases abstractas (el compilador lo impide, pero
    tenlo claro conceptualmente).

```csharp
// DISENO DE CLASE ABSTRACTA BIEN DOCUMENTADA
/// <summary>
/// Representa una fuente de datos que puede ser leida.
/// Implemente GetData() para proporcionar los datos especificos.
/// </summary>
public abstract class DataSource
{
    /// <summary>
    /// Lee los datos de la fuente. Debe ser implementado por
    /// cada tipo concreto de fuente.
    /// </summary>
    public abstract string[] GetData();

    /// <summary>
    /// Procesa los datos leidos. Las derivadas pueden sobrescribir
    /// para cambiar el comportamiento de procesamiento.
    /// </summary>
    public virtual void ProcessData()
    {
        var data = GetData();
        foreach (var item in data)
            Console.WriteLine(item);
    }
}
```

  11. CLASES ABSTRACTAS EN LA VIDA REAL (.NET)
--------------------------------------------------------------------

Muchas clases en .NET son abstractas:

  • `System.IO.Stream`:      Base para streams (FileStream, MemoryStream).
  • `System.DbConnection`:   Base para conexiones (SqlConnection, OracleConnection).
  • `System.Text.Encoding`:  Base para codificaciones (UTF8, ASCII).
  • `System.Xml.XmlReader`:  Base para lectores XML.
  • `System.IO.TextReader`:  Base para lectores de texto (StreamReader, StringReader).

```csharp
public class EjemploStream
{
    public void Demo()
    {
        // Stream es abstracto:
        // Stream s = new Stream();  // ERROR

        // Pero podemos usar sus derivadas:
        Stream archivo = new FileStream("datos.txt", FileMode.Open);
        Stream memoria = new MemoryStream();
        Stream red = new System.Net.Sockets.NetworkStream(null);

        // Todas se usan polimorficamente a traves de Stream:
        void ProcesarStream(Stream stream)
        {
            byte[] buffer = new byte[1024];
            int bytes = stream.Read(buffer, 0, buffer.Length);
            Console.WriteLine($"Leidos {bytes} bytes");
        }

        ProcesarStream(archivo);
        ProcesarStream(memoria);
    }
}
```

================================================================================
  RESUMEN DEL CAPÍTULO
================================================================================

  • Una CLASE ABSTRACTA se declara con `abstract` y NO puede
    instanciarse.

  • Los MÉTODOS ABSTRACTOS se declaran sin implementación y DEBEN
    ser implementados por las clases derivadas concretas.

  • Las clases abstractas pueden tener métodos CONCRETOS (con
    implementación), campos, propiedades y constructores.

  • ABSTRACT obliga a implementar; VIRTUAL permite sobrescribir
    opcionalmente.

  • Las clases abstractas representan la relación "es-un" con estado
    compartido.

  • Las INTERFACES representan capacidades ("puede-hacer") sin estado.

  • El patrón TEMPLATE METHOD usa clases abstractas para definir el
    esqueleto de un algoritmo con pasos personalizables.

  • Muchas clases del framework .NET son abstractas (Stream, DbConnection).

================================================================================
  PREGUNTAS FRECUENTES (FAQ)
================================================================================

  ───
  ❓ ¿Puedo tener una clase abstracta sin métodos abstractos?
  ───
  Sí. Puedes tener una clase abstracta solo con métodos concretos.
  Sirve para evitar que se instancie directamente.

  ───
  ❓ ¿Puedo tener un método abstracto en una clase no abstracta?
  ───
  No. Los métodos abstractos solo pueden existir en clases abstractas.

  ───
  ❓ ¿Puedo sellar (sealed) una clase abstracta?
  ───
  No. sealed y abstract son excluyentes. Una clase sellada no puede
  ser heredada, y una clase abstracta debe ser heredada.

  ───
  ❓ ¿Puedo tener propiedades abstractas?
  ───
  Sí. Las propiedades pueden ser abstractas, forzando a las derivadas
  a implementar los accesores.

  ───
  ❓ ¿Puedo tener un constructor público en una clase abstracta?
  ───
  Técnicamente sí, pero no tiene sentido porque no se puede hacer new.
  Normalmente se usa protected.

  ───
  ❓ ¿Cuándo debo elegir clase abstracta vs interfaz?
  ───
  Abstracta: cuando hay estado compartido (campos) y una relación
  "es-un". Interfaz: cuando defines capacidades que clases no
  relacionadas pueden implementar.

  ───
  ❓ ¿Puedo heredar de una clase abstracta y una interfaz?
  ───
  Sí. Puedes heredar de UNA clase abstracta e implementar VARIAS
  interfaces: public class X : ClaseAbstracta, IInterfaz1, IInterfaz2

  SISTEMA DE PROCESAMIENTO DE DOCUMENTOS

  Diseña un sistema de procesamiento de documentos usando clases
  abstractas y el patrón Template Method.

  Clase abstracta `Documento`:
    • Titulo, Autor, FechaCreacion (readonly)
    • Campos protegidos: _contenido (string)
    • Constructor con parametros
    • Metodos abstractos:
      - CargarContenido(): string (simular lectura)
      - Validar(): bool
      - Exportar(string formato): string
    • Metodos virtuales:
      - MostrarResumen(): void (default: titulo + autor)
      - ObtenerTamanio(): int (default: _contenido.Length)
    • Template Method Procesar():
      1. CargarContenido()
      2. Validar()
      3. Si valido, MostrarResumen()
      4. Exportar("txt")
      5. RegistrarProcesamiento() (privado)

  Clases derivadas:
    • DocumentoPDF:
      - Cargar: simular extraer texto de PDF
      - Validar: que el titulo no este vacio
      - Exportar: devuelve "[PDF]" + contenido
      - Sobrescribe ObtenerTamanio como contenido.Length * 2

    • DocumentoWord:
      - Cargar: simular extraer texto de Word
      - Validar: que tenga al menos 10 caracteres
      - Exportar: devuelve "[DOCX]" + contenido

    • HojaCalculo:
      - Cargar: simular extraer de Excel
      - Validar: que contenga "Tabla:" al inicio
      - Exportar: devuelve contenido en formato CSV
      - Tiene propiedad Filas, Columnas

  Clase `Programa`:
    • Crea documentos de cada tipo
    • Usa polimorfismo a traves de Documento
    • Llama al Template Method Procesar() para cada uno
    • Agrega un 4to tipo de documento sin modificar el
      procesamiento (Open/Closed)

Las clases abstractas son una herramienta de diseño fundamental. Permiten
definir contratos parciales con comportamiento compartido, dejando
flexibilidad para que las clases derivadas completen los detalles.

Dominar cuándo usar abstract, virtual o interfaz es lo que diferencia un
diseño mediocre de uno elegante y mantenible.
