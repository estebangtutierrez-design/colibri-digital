21.1 — TIPOS DE VALOR VS TIPOS DE REFERENCIA
----------------------------------------------

En C#, los tipos se dividen en dos categorías:

- Tipos de valor (value types): Se almacenan directamente en la pila
  (stack) o embebidos en el heap. Se copian por valor al asignarse.
  Incluyen: tipos numéricos (int, double, etc.), bool, char, enum, struct.

- Tipos de referencia (reference types): Se almacenan en el heap y la
  variable guarda una referencia (puntero) hacia esa ubicación. Al
  asignarse, se copia la referencia, no el objeto.
  Incluyen: class, string, array, delegate, interface, record (class).

Ejemplo conceptual:

```csharp
// Tipo de valor: copia independiente
int a = 10;
int b = a;
b = 20;
Console.WriteLine(a); // 10 — a no se ve afectado

// Tipo de referencia: comparten el objeto
var lista1 = new List<int> { 1, 2, 3 };
var lista2 = lista1;
lista2.Add(4);
Console.WriteLine(lista1.Count); // 4 — ambas referencias apuntan al mismo objeto
```

21.2 — DECLARACIÓN DE UN STRUCT
--------------------------------

Un struct se declara con la palabra clave struct. Puede contener
campos, propiedades, métodos, eventos e implementar interfaces.

```csharp
public struct Punto
{
    public int X;
    public int Y;

    public Punto(int x, int y)
    {
        X = x;
        Y = y;
    }

    public double DistanciaAlOrigen()
    {
        return Math.Sqrt(X * X + Y * Y);
    }
}
```

Uso:

```csharp
Punto p1 = new Punto(3, 4);
Console.WriteLine(p1.DistanciaAlOrigen()); // 5

Punto p2 = p1; // Copia independiente
p2.X = 10;
Console.WriteLine(p1.X); // 3 — p1 no cambia
```

21.3 — CONSTRUCTORES EN STRUCT
-------------------------------

Los structs tienen reglas especiales para constructores:

- No puedes definir un constructor sin parámetros (por defecto, el
  compilador genera uno que inicializa todos los campos a sus valores
  predeterminados).

- Todo constructor con parámetros debe asignar valor a TODOS los campos
  del struct.

- A partir de C# 10, puedes definir constructores sin parámetros, pero
  deben inicializar todos los campos.

```csharp
public struct Rectangulo
{
    public int Ancho;
    public int Alto;

    // Constructor obliga a inicializar TODOS los campos
    public Rectangulo(int ancho, int alto)
    {
        Ancho = ancho;
        Alto = alto;
    }
}
```

Ejemplo con this:

```csharp
public struct ColorRGB
{
    public byte R;
    public byte G;
    public byte B;

    public ColorRGB(byte r, byte g, byte b)
    {
        R = r;
        G = g;
        B = b;
    }

    // Constructor con parámetro por defecto
    public ColorRGB(byte gris) : this(gris, gris, gris) { }
}
```

21.4 — STRUCT READONLY
-----------------------

Un struct readonly garantiza que ninguna instancia puede ser modificada.
Todos los campos deben ser readonly y las propiedades automáticas deben ser
de solo lectura (get-only).

```csharp
public readonly struct Vector3D
{
    public double X { get; }
    public double Y { get; }
    public double Z { get; }

    public Vector3D(double x, double y, double z)
    {
        X = x;
        Y = y;
        Z = z;
    }

    public double Magnitud => Math.Sqrt(X * X + Y * Y + Z * Z);

    // Los métodos no modifican el estado
    public Vector3D Normalizar()
    {
        double mag = Magnitud;
        return new Vector3D(X / mag, Y / mag, Z / mag);
    }
}
```

21.5 — MODIFICADOR READONLY EN MIEMBROS
----------------------------------------

A partir de C# 8, puedes marcar miembros individuales como readonly
dentro de un struct no-readonly. Esto evita que el compilador haga
copias defensivas.

```csharp
public struct Punto2D
{
    public int X;
    public int Y;

    public Punto2D(int x, int y)
    {
        X = x;
        Y = y;
    }

    // Este método no modifica el estado
    public readonly double DistanciaAlOrigen()
    {
        return Math.Sqrt(X * X + Y * Y);
    }

    // Este método SÍ modifica el estado
    public void Desplazar(int dx, int dy)
    {
        X += dx;
        Y += dy;
    }
}
```

21.6 — RECORD STRUCT (C# 10)
-------------------------------

Los record struct combinan la semántica de valor de los structs con
la igualdad estructural de los records.

```csharp
public readonly record struct Coordenada(double Latitud, double Longitud);

// Uso
var c1 = new Coordenada(40.4168, -3.7038);
var c2 = new Coordenada(40.4168, -3.7038);
Console.WriteLine(c1 == c2); // True — igualdad estructural

// with expression (immutabilidad)
var c3 = c1 with { Longitud = -0.1276 };
Console.WriteLine(c3); // Coordenada { Latitud = 40.4168, Longitud = -0.1276 }
```

Record struct no-readonly (mutable):

```csharp
public record struct Puntuacion
{
    public string Jugador { get; set; }
    public int Puntos { get; set; }

    public Puntuacion(string jugador, int puntos)
    {
        Jugador = jugador;
        Puntos = puntos;
    }
}
```

21.7 — STRUCT VS CLASS — ¿CUÁNDO USAR CADA UNO?
--------------------------------------------------

Usa struct CUANDO:
- El tipo representa un único valor primitivo (Point, Color, Complex).
- La instancia es pequeña (< 16-24 bytes aproximadamente).
- El tipo es inmutable (preferiblemente).
- No necesitas herencia ni identidad referencial.
- Habrá muchas instancias en colecciones (arrays de structs son contiguos).

Usa class CUANDO:
- El tipo tiene más de 24 bytes.
- Necesitas herencia o polimorfismo.
- Necesitas identidad referencial (dos objetos distintos aunque tengan
  el mismo contenido).
- El tipo necesita valores null (los structs no son null por defecto).
- Necesitas un patrón de objeto compartido (varias referencias al mismo).

21.8 — COMPORTAMIENTO EN MEMORIA
---------------------------------

Los structs pueden vivir en tres lugares:

1. Stack: variable local de tipo struct.
2. Heap (embebido): cuando el struct es campo de una clase.
3. Heap (boxing): cuando se convierte a object o a una interfaz.

```csharp
// Stack
int numero = 42;

// Heap (embebido) — el struct está dentro del objeto en el heap
object contenedor = new Punto(10, 20);

// Boxing — el struct se envuelve en el heap
int x = 100;
object obj = x; // boxing
int y = (int)obj; // unboxing
```

El boxing/unboxing tiene costo de rendimiento. Evitarlo es clave.

21.9 — STRUCT E INTERFACES
---------------------------

Los structs pueden implementar interfaces, pero cuidado con el boxing.

```csharp
public interface IDesplazable
{
    void Desplazar(int dx, int dy);
}

public struct Punto : IDesplazable
{
    public int X, Y;
    public Punto(int x, int y) { X = x; Y = y; }
    public void Desplazar(int dx, int dy) { X += dx; Y += dy; }
}

// Uso — SIN boxing cuando se llama directamente
var p = new Punto(1, 2);
p.Desplazar(1, 1);

// CON boxing cuando se usa la interfaz
IDesplazable d = p; // boxing
d.Desplazar(1, 1);
```

21.10 — ARRAYS Y RENDIMIENTO
------------------------------

Los arrays de structs son contiguos en memoria, lo que mejora la
localidad de la caché.

```csharp
// Array de structs — contiguo, rápido para iterar
var puntos = new Punto[1000];
for (int i = 0; i < puntos.Length; i++)
    puntos[i] = new Punto(i, i * 2);

// Array de clases — cada objeto en el heap, referencias dispersas
var puntosRef = new PuntoRef[1000];
for (int i = 0; i < puntosRef.Length; i++)
    puntosRef[i] = new PuntoRef(i, i * 2);
```

21.11 — PARÁMETROS Y STRUCTS
------------------------------

Los structs se pasan por valor (copia) a los métodos. Para evitar
copias grandes, usa ref, in o ref readonly.

```csharp
public struct Matriz3x3
{
    public double M11, M12, M13;
    public double M21, M22, M23;
    public double M31, M32, M33;
}

// Paso por valor — copia cara
public static double Determinante(Matriz3x3 m) { ... }

// Paso por referencia de solo lectura — sin copia
public static double DeterminanteFast(in Matriz3x3 m) { ... }

// Paso por referencia (mutable)
public static void Escalar(ref Matriz3x3 m, double factor)
{
    m.M11 *= factor;
    // ...
}
```

21.13 — COMPARACIÓN DE IGUALDAD
---------------------------------

Por defecto, los structs usan ValueType.Equals que hace reflexión.
Siempre es mejor sobrecargar Equals e implementar IEquatable<T>.

```csharp
public struct Fraccion : IEquatable<Fraccion>
{
    public int Numerador { get; }
    public int Denominador { get; }

    public Fraccion(int num, int den)
    {
        if (den == 0)
            throw new ArgumentException("Denominador no puede ser cero");
        Numerador = num;
        Denominador = den;
    }

    public override bool Equals(object? obj) =>
        obj is Fraccion other && Equals(other);

    public bool Equals(Fraccion other) =>
        Numerador == other.Numerador && Denominador == other.Denominador;

    public override int GetHashCode() => HashCode.Combine(Numerador, Denominador);

    public static bool operator ==(Fraccion a, Fraccion b) => a.Equals(b);
    public static bool operator !=(Fraccion a, Fraccion b) => !a.Equals(b);

    public override string ToString() => $"{Numerador}/{Denominador}";
}
```

21.14 — BUENAS PRÁCTICAS
--------------------------

- Prefiere structs inmutables (readonly).
- No expongas campos públicos mutables (usa propiedades).
- Mantén los structs pequeños (< 24 bytes).
- No heredes de un struct (no es posible).
- Implementa IEquatable<T> para evitar boxing.
- Usa ref/in para parámetros grandes.
- No crees structs que contengan campos de tipo referencia mutable.
- Marca miembros que no modifiquen el estado como readonly.

21.15 — EJEMPLO COMPLETO: SISTEMA DE COORDENADAS
--------------------------------------------------

```csharp
using System;
using System.Collections.Generic;

public readonly struct Coordenadas
{
    public double Lat { get; }
    public double Lon { get; }

    public Coordenadas(double lat, double lon)
    {
        Lat = lat;
        Lon = lon;
    }

    public double DistanciaHasta(Coordenadas otra)
    {
        const double R = 6371; // Radio terrestre en km
        double dLat = (otra.Lat - Lat) * Math.PI / 180;
        double dLon = (otra.Lon - Lon) * Math.PI / 180;
        double a = Math.Sin(dLat / 2) * Math.Sin(dLat / 2) +
                   Math.Cos(Lat * Math.PI / 180) * Math.Cos(otra.Lat * Math.PI / 180) *
                   Math.Sin(dLon / 2) * Math.Sin(dLon / 2);
        double c = 2 * Math.Atan2(Math.Sqrt(a), Math.Sqrt(1 - a));
        return R * c;
    }

    public override string ToString() => $"({Lat:F4}, {Lon:F4})";
}

public class Programa
{
    public static void Main()
    {
        var madrid = new Coordenadas(40.4168, -3.7038);
        var buenosAires = new Coordenadas(-34.6037, -58.3816);
        var londres = new Coordenadas(51.5074, -0.1278);
        var tokio = new Coordenadas(35.6762, 139.6503);
        var nuevaYork = new Coordenadas(40.7128, -74.0060);

        var ciudades = new Dictionary<string, Coordenadas>
        {
            ["Madrid"] = madrid,
            ["Buenos Aires"] = buenosAires,
            ["Londres"] = londres,
            ["Tokio"] = tokio,
            ["Nueva York"] = nuevaYork
        };

        Console.WriteLine("=== DISTANCIAS ENTRE CIUDADES ===\n");

        foreach (var c1 in ciudades)
        {
            foreach (var c2 in ciudades)
            {
                if (c1.Key == c2.Key) continue;
                double d = c1.Value.DistanciaHasta(c2.Value);
                Console.WriteLine($"{c1.Key,-15} -> {c2.Key,-15}: {d,8:F0} km");
            }
        }

        Console.WriteLine($"\nTotal de ciudades: {ciudades.Count}");
        Console.WriteLine($"Tamaño del struct Coordenadas: {System.Runtime.InteropServices.Marshal.SizeOf<Coordenadas>()} bytes");
    }
}
```

21.16 — BOXING Y UNBOXING — DETALLE TÉCNICO
----------------------------------------------

```csharp
int valor = 123;
object obj = valor;   // boxing: crea un objeto en el heap con el valor
int devuelto = (int)obj; // unboxing: extrae el valor del heap

// Evitar boxing con genéricos
List<int> numeros = new List<int>(); // SIN boxing
ArrayList listaVieja = new ArrayList(); // CON boxing (cada int se boxea)
```

21.17 — REF STRUCT (C# 7.2)
-----------------------------

Los ref struct solo pueden vivir en la pila, nunca en el heap.
Son útiles para interoperabilidad y escenarios de alto rendimiento.

```csharp
public ref struct SpanBuffer
{
    private Span<byte> _buffer;

    public SpanBuffer(Span<byte> buffer)
    {
        _buffer = buffer;
    }

    public void Escribir(int offset, byte valor)
    {
        _buffer[offset] = valor;
    }

    public byte Leer(int offset) => _buffer[offset];
}

// Limitaciones:
// - No puede ser campo de una clase
// - No puede ser usado en tipos genéricos
// - No puede ser boxeado
// - No puede ser capturado por una lambda
// - No puede implementar interfaces
```

21.18 — STRUCT Y DISPOSE
--------------------------

Los structs pueden implementar IDisposable, aunque no pueden tener
finalizadores.

```csharp
public ref struct ResourceGuard
{
    private IntPtr _resource;

    public ResourceGuard(IntPtr resource)
    {
        _resource = resource;
    }

    public void Dispose()
    {
        if (_resource != IntPtr.Zero)
        {
            // Liberar recurso
            _resource = IntPtr.Zero;
        }
    }
}
```

================================================================================
  FAQ — PREGUNTAS FRECUENTES
================================================================================

P: ¿Puedo hacer boxeo de un struct nullable?
R: Sí, int? es Nullable<int> que es un struct. Al boxearlo, si tiene
   valor se boxea el valor subyacente; si es null, se boxea a null.

P: ¿Por qué mi struct no me deja tener constructor sin parámetros?
R: Antes de C# 10, los structs no permitían constructores sin parámetros.
   A partir de C# 10 sí, pero deben inicializar todos los campos.

P: ¿Los structs tienen rendimiento siempre mejor que las clases?
R: No. Si el struct es grande (> 24 bytes) o se copia frecuentemente,
   el costo de copia puede superar el costo de asignación en el heap.

P: ¿Puedo heredar de un struct?
R: No. Los structs sellan implícitamente. Solo pueden implementar interfaces.

P: ¿Qué pasa si asigno un struct a una variable dynamic?
R: Habrá boxing, pues dynamic trata el valor como object.

P: ¿Cuándo debo usar record struct en lugar de struct?
R: Cuando necesitas igualdad estructural automática, with expressions,
   y una representación ToString útil sin escribir código adicional.

P: ¿Los structs pueden tener eventos?
R: Sí, pero no es recomendable porque el evento se copiaría con el struct.

P: ¿Qué es una copia defensiva?
R: Cuando el compilador copia un struct antes de llamar a un método no-readonly
   para evitar que el método modifique el original. Se evita marcando el método
   como readonly.
