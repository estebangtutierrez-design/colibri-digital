=== 1. INTRODUCCIÓN A LA ARQUITECTURA EN C ===

La arquitectura de software en C difiere de otros lenguajes porque C carece
de soporte nativo para orientación a objetos, namespaces o módulos formales.
Sin embargo, esta carencia nos obliga a ser disciplinados y a usar las
herramientas que el lenguaje nos proporciona de manera estructurada.

Los pilares de la arquitectura en C son:

  1. Separación en módulos (archivos .h y .c)
  2. Abstracción mediante punteros a función y structs
  3. Información oculta (static y forward declaraciones)
  4. Interfaces bien definidas
  5. Gestión explícita de recursos

Un arquitecto de software en C debe pensar en términos de:
  - ¿Cómo se comunican los módulos?
  - ¿Quién es responsable de cada recurso?
  - ¿Cómo se manejan los errores?
  - ¿Cómo se prueba cada componente?

La diferencia entre un programador novato y un arquitecto es que el novato
piensa en funciones, mientras que el arquitecto piensa en sistemas completos
y en cómo las partes interactúan a lo largo del tiempo.

=== 2. ORGANIZACIÓN DE PROYECTOS ===

La estructura típica de un proyecto profesional en C sigue un patrón
estándar que facilita la navegación, compilación y distribución.

Estructura recomendada:

```
proyecto/
├── src/              # Código fuente
│   ├── main.c
│   ├── modulo1/
│   │   ├── modulo1.h
│   │   └── modulo1.c
│   └── modulo2/
│       ├── modulo2.h
│       └── modulo2.c
├── include/          # Headers públicos
│   └── proyecto/
│       ├── api.h
│       └── tipos.h
├── lib/              # Bibliotecas externas
├── build/            # Archivos objeto
├── bin/              # Ejecutables
├── test/             # Pruebas unitarias
├── docs/             # Documentación
├── Makefile          # Sistema de compilación
└── README.md         # Descripción del proyecto
```

Cada módulo debe tener una responsabilidad única y bien definida. Si un
módulo hace demasiadas cosas, es momento de dividirlo.

La organización no es solo estética: una buena estructura permite que
varios desarrolladores trabajen en paralelo sin pisarse los unos a los
otros. También facilita la integración continua y el despliegue.

=== 3. DISEÑO DE INTERFACES ===

Una interfaz en C es un archivo de cabecera (.h) que expone:

  - Tipos de datos públicos
  - Constantes y macros públicas
  - Prototipos de funciones públicas
  - Documentación de uso

Reglas de oro para diseñar interfaces:

  1. Una interfaz debe ser completa pero minimalista.
  2. Cada función debe hacer una sola cosa.
  3. Los nombres deben ser descriptivos y consistentes.
  4. Los parámetros deben tener un orden lógico (entrada, salida).
  5. Documenta el contrato de cada función (precondiciones,
     postcondiciones).

Ejemplo de una interfaz bien diseñada:

```c
#ifndef COLAS_H
#define COLAS_H

#include <stddef.h>
#include <stdbool.h>

typedef struct Cola Cola;

Cola* cola_crear(size_t capacidad);
void cola_destruir(Cola* cola);
bool cola_encolar(Cola* cola, void* dato);
void* cola_desencolar(Cola* cola);
void* cola_frente(const Cola* cola);
bool cola_vacia(const Cola* cola);
size_t cola_tamano(const Cola* cola);

#endif /* COLAS_H */
```

La implementación oculta el detalle interno:

```c
#include "cola.h"
#include <stdlib.h>

struct Cola {
    void** elementos;
    size_t capacidad;
    size_t frente;
    size_t final;
    size_t count;
};

Cola* cola_crear(size_t capacidad) {
    Cola* c = malloc(sizeof(Cola));
    if (!c) return NULL;
    c->elementos = calloc(capacidad, sizeof(void*));
    if (!c->elementos) { free(c); return NULL; }
    c->capacidad = capacidad;
    c->frente = 0;
    c->final = 0;
    c->count = 0;
    return c;
}

bool cola_encolar(Cola* c, void* dato) {
    if (!c || c->count >= c->capacidad) return false;
    c->elementos[c->final] = dato;
    c->final = (c->final + 1) % c->capacidad;
    c->count++;
    return true;
}

void* cola_desencolar(Cola* c) {
    if (!c || c->count == 0) return NULL;
    void* dato = c->elementos[c->frente];
    c->frente = (c->frente + 1) % c->capacidad;
    c->count--;
    return dato;
}

void* cola_frente(const Cola* c) {
    if (!c || c->count == 0) return NULL;
    return c->elementos[c->frente];
}

bool cola_vacia(const Cola* c) {
    return !c || c->count == 0;
}

size_t cola_tamano(const Cola* c) {
    return c ? c->count : 0;
}

void cola_destruir(Cola* c) {
    if (!c) return;
    free(c->elementos);
    free(c);
}
```

=== 4. MÓDULOS Y SEPARACIÓN DE RESPONSABILIDADES ===

Cada módulo debe tener una única responsabilidad. Para lograr esto, podemos
seguir el principio de "Alta cohesión, bajo acoplamiento".

  - Alta cohesión: Las funciones dentro de un módulo están estrechamente
    relacionadas y trabajan juntas para un propósito común.
  - Bajo acoplamiento: Los módulos dependen poco unos de otros; se comunican
    a través de interfaces bien definidas.

Ejemplo de un módulo bien diseñado:

```c
// logger.h
#ifndef LOGGER_H
#define LOGGER_H

#include <stdio.h>
#include <stdarg.h>

typedef enum {
    LOG_TRACE, LOG_DEBUG, LOG_INFO,
    LOG_WARN, LOG_ERROR, LOG_FATAL
} LogLevel;

typedef struct Logger Logger;

Logger* logger_crear(const char* archivo, LogLevel nivel);
void   logger_destruir(Logger* logger);
void   logger_log(Logger* logger, LogLevel nivel, const char* fmt, ...);
void   logger_set_nivel(Logger* logger, LogLevel nivel);

#endif
```

```c
// logger.c
#include "logger.h"
#include <stdlib.h>
#include <string.h>
#include <time.h>

static const char* NIVELES[] = {
    "TRACE", "DEBUG", "INFO", "WARN", "ERROR", "FATAL"
};

struct Logger {
    FILE*    archivo;
    LogLevel nivel;
};

Logger* logger_crear(const char* archivo, LogLevel nivel) {
    Logger* logger = malloc(sizeof(Logger));
    if (!logger) return NULL;
    logger->archivo = fopen(archivo, "a");
    if (!logger->archivo) { free(logger); return NULL; }
    logger->nivel = nivel;
    return logger;
}

void logger_destruir(Logger* logger) {
    if (!logger) return;
    if (logger->archivo) fclose(logger->archivo);
    free(logger);
}

static void timestamp(char* buf, size_t tam) {
    time_t t = time(NULL);
    struct tm* tm = localtime(&t);
    strftime(buf, tam, "%Y-%m-%d %H:%M:%S", tm);
}

void logger_log(Logger* logger, LogLevel nivel, const char* fmt, ...) {
    if (!logger || nivel < logger->nivel) return;
    char ts[32];
    timestamp(ts, sizeof(ts));
    fprintf(logger->archivo, "[%s] %s: ", ts, NIVELES[nivel]);
    va_list args;
    va_start(args, fmt);
    vfprintf(logger->archivo, fmt, args);
    va_end(args);
    fprintf(logger->archivo, "\n");
    fflush(logger->archivo);
}

void logger_set_nivel(Logger* logger, LogLevel nivel) {
    if (logger) logger->nivel = nivel;
}
```

=== 5. PATRONES ARQUITECTÓNICOS EN C ===

Aunque C no es un lenguaje orientado a objetos, podemos implementar patrones
de diseño clásicos usando structs y punteros a función.

5.1. PATRÓN MÓDULO (SIMULACIÓN DE CLASE)

```c
typedef struct {
    void (*iniciar)(void);
    void (*procesar)(int dato);
    void (*finalizar)(void);
    int   estado;
} Maquina;

static void maquina_iniciar_impl(void) {
    printf("Máquina iniciada.\n");
}

static void maquina_procesar_impl(int dato) {
    printf("Procesando: %d\n", dato);
}

static void maquina_finalizar_impl(void) {
    printf("Máquina finalizada.\n");
}

void maquina_crear(Maquina* m) {
    m->iniciar   = maquina_iniciar_impl;
    m->procesar  = maquina_procesar_impl;
    m->finalizar = maquina_finalizar_impl;
    m->estado    = 0;
}

int main(void) {
    Maquina m;
    maquina_crear(&m);
    m.iniciar();
    m.procesar(42);
    m.finalizar();
    return 0;
}
```

5.2. PATRÓN OBSERVADOR

```c
#include <stdio.h>
#include <stdlib.h>
#include <stdbool.h>

typedef void (*Observador)(int evento, void* contexto);

typedef struct {
    Observador* slots;
    void**      contextos;
    size_t      capacidad;
    size_t      count;
} GestorEventos;

void gestor_init(GestorEventos* g, size_t max) {
    g->slots = calloc(max, sizeof(Observador));
    g->contextos = calloc(max, sizeof(void*));
    g->capacidad = max;
    g->count = 0;
}

bool gestor_registrar(GestorEventos* g, Observador obs, void* ctx) {
    if (g->count >= g->capacidad) return false;
    g->slots[g->count] = obs;
    g->contextos[g->count] = ctx;
    g->count++;
    return true;
}

void gestor_notificar(GestorEventos* g, int evento) {
    for (size_t i = 0; i < g->count; i++)
        g->slots[i](evento, g->contextos[i]);
}

void gestor_destruir(GestorEventos* g) {
    free(g->slots);
    free(g->contextos);
}

// Uso
void on_click(int evento, void* ctx) {
    printf("Click en %s\n", (const char*)ctx);
}

int main(void) {
    GestorEventos g;
    gestor_init(&g, 10);
    gestor_registrar(&g, on_click, "boton1");
    gestor_registrar(&g, on_click, "boton2");
    gestor_notificar(&g, 1);
    gestor_destruir(&g);
    return 0;
}
```

5.3. PATRÓN ESTRATEGIA

```c
#include <stdio.h>
#include <string.h>

typedef int (*Estrategia)(int* arr, size_t n);

typedef struct {
    Estrategia ordenar;
    char       nombre[32];
} ContextoOrdenamiento;

static int bubble_sort(int* arr, size_t n) {
    for (size_t i = 0; i < n - 1; i++)
        for (size_t j = 0; j < n - i - 1; j++)
            if (arr[j] > arr[j + 1]) {
                int tmp = arr[j];
                arr[j] = arr[j + 1];
                arr[j + 1] = tmp;
            }
    return 0;
}

static int comparar_int(const void* a, const void* b) {
    return (*(int*)a - *(int*)b);
}

static int qsort_estrategia(int* arr, size_t n) {
    qsort(arr, n, sizeof(int), comparar_int);
    return 0;
}

void contexto_usar(ContextoOrdenamiento* ctx,
                   Estrategia est, const char* nombre) {
    ctx->ordenar = est;
    strncpy(ctx->nombre, nombre, sizeof(ctx->nombre) - 1);
    ctx->nombre[sizeof(ctx->nombre) - 1] = '\0';
}

int contexto_ejecutar(ContextoOrdenamiento* ctx, int* arr, size_t n) {
    printf("Usando estrategia: %s\n", ctx->nombre);
    return ctx->ordenar(arr, n);
}

int main(void) {
    int datos[] = {5, 3, 1, 4, 2};
    size_t n = 5;
    ContextoOrdenamiento ctx;
    contexto_usar(&ctx, bubble_sort, "Bubble Sort");
    contexto_ejecutar(&ctx, datos, n);
    for (size_t i = 0; i < n; i++) printf("%d ", datos[i]);
    printf("\n");
    return 0;
}
```

=== 6. MANEJO DE ERRORES ARQUITECTÓNICO ===

Un aspecto fundamental de la arquitectura en C es cómo se manejan los errores.
Las estrategias comunes incluyen:

  1. Códigos de retorno: La función devuelve un entero indicando éxito/error.
  2. Parámetros de salida: Se pasa un puntero donde escribir el resultado.
  3. errno global: Variable global que indica el último error.
  4. Callbacks de error: Función llamada cuando ocurre un error.

La convención más profesional es combinar códigos de retorno con parámetros
de salida:

```c
#include <stdio.h>
#include <string.h>
#include <errno.h>

typedef enum {
    ERR_OK      = 0,
    ERR_MEMORIA = -1,
    ERR_IO      = -2,
    ERR_PARAM   = -3,
    ERR_ESTADO  = -4
} Error;

typedef struct {
    int  codigo;
    char mensaje[256];
} Resultado;

const char* error_strerror(Error e) {
    switch (e) {
        case ERR_OK:      return "Operación exitosa";
        case ERR_MEMORIA: return "Error de memoria";
        case ERR_IO:      return "Error de entrada/salida";
        case ERR_PARAM:   return "Parámetro inválido";
        case ERR_ESTADO:  return "Estado inválido";
        default:          return "Error desconocido";
    }
}

Error leer_archivo(const char* ruta, char** contenido, size_t* tam) {
    if (!ruta || !contenido || !tam) return ERR_PARAM;

    FILE* f = fopen(ruta, "rb");
    if (!f) {
        return ERR_IO;
    }

    fseek(f, 0, SEEK_END);
    long size = ftell(f);
    if (size < 0) {
        fclose(f);
        return ERR_IO;
    }
    rewind(f);

    *contenido = malloc((size_t)size + 1);
    if (!*contenido) {
        fclose(f);
        return ERR_MEMORIA;
    }

    size_t leido = fread(*contenido, 1, (size_t)size, f);
    if (ferror(f)) {
        free(*contenido);
        fclose(f);
        return ERR_IO;
    }

    (*contenido)[leido] = '\0';
    *tam = leido;
    fclose(f);
    return ERR_OK;
}
```

=== 7. ARQUITECTURA EN CAPAS ===

Una arquitectura en capas organiza el código en niveles de abstracción:

```
  Capa 0: Aplicación (main, lógica de negocio)
  Capa 1: Servicios (logging, configuración, comunicación)
  Capa 2: Módulos funcionales (procesamiento, análisis)
  Capa 3: Infraestructura (archivos, red, memoria)
  Capa 4: Hardware/SO (llamadas al sistema, drivers)
```

Cada capa solo puede comunicarse con la capa inmediatamente inferior.
Esto reduce el acoplamiento y facilita el mantenimiento.

Ejemplo completo:

```c
// ===== Capa 3: infraestructura =====
typedef struct {
    FILE* handle;
    char  ruta[512];
} Archivo;

Archivo* archivo_abrir(const char* ruta, const char* modo) {
    Archivo* a = malloc(sizeof(Archivo));
    if (!a) return NULL;
    a->handle = fopen(ruta, modo);
    if (!a->handle) { free(a); return NULL; }
    strncpy(a->ruta, ruta, sizeof(a->ruta) - 1);
    return a;
}

size_t archivo_leer(Archivo* a, void* buf, size_t tam) {
    return fread(buf, 1, tam, a->handle);
}

int archivo_cerrar(Archivo* a) {
    if (!a) return -1;
    int r = fclose(a->handle);
    free(a);
    return r;
}

// ===== Capa 2: módulo procesador =====
typedef struct {
    Archivo* entrada;
    Archivo* salida;
} Procesador;

int procesador_init(Procesador* p, const char* in, const char* out) {
    p->entrada = archivo_abrir(in, "rb");
    if (!p->entrada) return -1;
    p->salida = archivo_abrir(out, "wb");
    if (!p->salida) {
        archivo_cerrar(p->entrada);
        return -1;
    }
    return 0;
}

int procesador_ejecutar(Procesador* p) {
    char buf[4096];
    size_t n;
    while ((n = archivo_leer(p->entrada, buf, sizeof(buf))) > 0) {
        // Procesar (ejemplo: convertir a mayúsculas)
        for (size_t i = 0; i < n; i++)
            buf[i] = toupper((unsigned char)buf[i]);
        fwrite(buf, 1, n, p->salida->handle);
    }
    return 0;
}

void procesador_destruir(Procesador* p) {
    if (p->entrada) archivo_cerrar(p->entrada);
    if (p->salida) archivo_cerrar(p->salida);
}

// ===== Capa 0: aplicación =====
int main(int argc, char** argv) {
    if (argc < 3) {
        fprintf(stderr, "Uso: %s entrada salida\n", argv[0]);
        return 1;
    }
    Procesador p;
    if (procesador_init(&p, argv[1], argv[2]) != 0) {
        fprintf(stderr, "Error al inicializar\n");
        return 1;
    }
    procesador_ejecutar(&p);
    procesador_destruir(&p);
    return 0;
}
```

=== 9. GESTIÓN DE DEPENDENCIAS ===

En proyectos grandes, las dependencias entre módulos deben gestionarse
cuidadosamente. Técnicas fundamentales:

  1. GRÁFICO DE DEPENDENCIAS: Identificar dependencias cíclicas.
  2. INVERSIÓN DE DEPENDENCIAS: Las interfaces definen contratos.
  3. INYECCIÓN DE DEPENDENCIAS: Los módulos reciben sus dependencias.

Para detectar dependencias cíclicas automáticamente:

```bash
#!/bin/bash
# detectar_ciclos.sh
for header in $(find . -name '*.h'); do
    includes=$(grep '#include' "$header" | grep -oP '"[^"]+"' | tr -d '"')
    base=$(basename "$header")
    for inc in $includes; do
        if grep -q "$base" "$(dirname "$header")/$inc" 2>/dev/null; then
            echo "CICLO: $header <-> $(dirname "$header")/$inc"
        fi
    done
done
```

Principio de Inversión de Dependencias (DIP):

```c
// MAL: Dependencia directa con una implementación concreta
#include "mysql_conexion.h"

void guardar_usuario(Usuario* u) {
    mysql_insertar(u);
}

// BIEN: Dependencia abstracta mediante interfaz
typedef struct {
    int (*insertar)(void* db, const Usuario* u);
    int (*actualizar)(void* db, const Usuario* u);
    int (*eliminar)(void* db, int id);
} RepositorioUsuarios;

void guardar_usuario(RepositorioUsuarios* repo, void* db, Usuario* u) {
    repo->insertar(db, u);
}
```

=== 10. CASO PRÁCTICO: ARQUITECTURA DE SERVIDOR CHAT ===

Apliquemos todo lo aprendido diseñando la arquitectura de un servidor de
chat simple pero profesional.

Módulos identificados:

  1. red:     Manejo de sockets y conexiones.
  2. sesion:  Gestión de sesiones de usuario.
  3. sala:    Lógica de salas de chat.
  4. comando: Procesamiento de comandos.
  5. logger:  Registro de eventos.

```c
// red.h - Interfaz del módulo de red
#ifndef RED_H
#define RED_H

#include <stddef.h>

typedef struct Conexion Conexion;
typedef void (*CallbackMensaje)(Conexion* origen,
                                const char* msg, size_t len);
typedef void (*CallbackConexion)(Conexion* conn);

typedef enum {
    RED_OK,
    RED_ERROR_CREAR,
    RED_ERROR_ENLAZAR,
    RED_ERROR_ACEPTAR,
    RED_ERROR_ENVIAR,
    RED_ERROR_RECIBIR
} RedError;

RedError red_iniciar(int puerto, CallbackMensaje cb_msg);
RedError red_ejecutar(void);
RedError red_enviar(Conexion* conn, const char* msg, size_t len);
void     red_difundir(const char* msg, size_t len);
void     red_finalizar(void);
void     red_set_callback_conexion(CallbackConexion on_connect,
                                   CallbackConexion on_disconnect);

#endif
```

```c
// sesion.h
#ifndef SESION_H
```

=== 11. ARQUITECTURA PARA SISTEMAS EMBEBIDOS ===

En sistemas embebidos, la arquitectura debe considerar restricciones
de memoria, tiempo real y consumo energético.

Patrón típico para embebidos:

```c
// Arquitectura de máquina de estados para sistemas embebidos
typedef enum {
    ESTADO_INICIO,
    ESTADO_CONFIG,
    ESTADO_EJECUCION,
    ESTADO_ERROR,
    ESTADO_DORMIR
} Estado;

typedef struct {
    Estado    estado_actual;
    void      (*on_enter)(void);
      Estado    (*on_tick)(void);
    void      (*on_exit)(void);
} EstadoHandler;

static EstadoHandler maquina[] = {
    { ESTADO_INICIO,     inicio_enter,     inicio_tick,     inicio_exit },
    { ESTADO_CONFIG,     config_enter,     config_tick,     config_exit },
    { ESTADO_EJECUCION,  ejecucion_enter,  ejecucion_tick,  ejecucion_exit },
    { ESTADO_ERROR,      error_enter,      error_tick,      error_exit },
    { ESTADO_DORMIR,     dormir_enter,     dormir_tick,     dormir_exit },
};

void bucle_principal(void) {
    Estado actual = ESTADO_INICIO;
    maquina[actual].on_enter();

    while (1) {
        Estado siguiente = maquina[actual].on_tick();
        if (siguiente != actual) {
            maquina[actual].on_exit();
            actual = siguiente;
            maquina[actual].on_enter();
        }
    }
}
```

=== 12. HERRAMIENTAS DE ANÁLISIS ARQUITECTÓNICO ===

Herramientas útiles para analizar y visualizar la arquitectura:

  2. doxygen: Genera documentación y diagramas.

Ejemplo de uso de cflow:

```bash

```

Ejemplo de Makefile con análisis:

```makefile
.PHONY: analisis
analisis:
    @echo "=== Análisis arquitectónico ==="
    @echo "--- Complejidad ciclomática ---"
    @for f in src/*.c; do \
        complexity "$$f" 2>/dev/null || true; \
    done
    @echo "--- Dependencias entre módulos ---"
    @for f in src/*.h; do \
        echo "$$f:"; \
        grep '#include "' "$$f" | sed 's/.*"//' | sed 's/".*//' \
            | sed 's/^/  -> /'; \
    done
```

=== 13. REFACTORIZACIÓN ARQUITECTÓNICA ===

Reconocer cuándo la arquitectura necesita ser refactorizada es una
habilidad crucial. Señales de alerta:

  - Módulo demasiado grande (más de 1000 líneas).
  - Dependencias cíclicas entre módulos.
  - Código duplicado en múltiples módulos.
  - Funciones con demasiados parámetros.
  - Archivos .h que incluyen todo el proyecto.

Proceso de refactorización:

```c
// ANTES: Monolito
int procesar(const char* entrada, const char* salida,
              int opciones, int nivel_log, const char* log_file,
              int (*callback)(int), void* user_data) {
    // 200 líneas de código
    return 0;
}

// DESPUÉS: Modularizado
typedef struct {
    const char* entrada;
    const char* salida;
    int         opciones;
} ConfigProceso;

typedef struct {
    int   (*callback)(int);
    void* user_data;
} Callbacks;

typedef struct {
    ConfigProceso config;
    Callbacks     cbs;
    Logger*       logger;
} ContextoProceso;

int procesar(ContextoProceso* ctx) {
    logger_log(ctx->logger, LOG_INFO, "Iniciando proceso...");
    // Lógica modular
    return 0;
}
```

________________________________________________________________________________
  PREGUNTAS FRECUENTES
________________________________________________________________________________

P: ¿Cuántos módulos debería tener mi proyecto?
R: No hay un número mágico. Cada módulo debe tener una responsabilidad
única. Si tienes dudas, empieza con pocos módulos y divide cuando un
módulo se vuelva demasiado grande o haga demasiadas cosas.

P: ¿Debo usar un solo .h por cada .c?
R: Generalmente sí. Cada módulo tiene su par .h/.c. Sin embargo,
puedes tener .h que agrupen varios módulos (como un API pública)
y .h internos para detalles de implementación compartidos.

P: ¿Cómo manejo las dependencias cíclicas?
R: Las dependencias cíclicas son una mala práctica. Para eliminarlas,
puedes: (1) extraer la interfaz común a un tercer módulo, (2) usar
callbacks para invertir la dependencia, o (3) fusionar los módulos
con dependencia cíclica en uno solo.

P: ¿Qué hago con las estructuras de datos globales?
R: Minimiza su uso. Si necesitas estado global, encapsúlalo en un
módulo singleton con funciones de acceso controladas. Nunca expongas
variables globales en archivos .h.

P: ¿Cómo documento la arquitectura de mi proyecto?
R: Usa Doxygen para documentación en el código. Para diagramas de
alto nivel, herramientas como PlantUML o Mermaid son excelentes.
Mantén un ARCHITECTURE.md con la visión general.
