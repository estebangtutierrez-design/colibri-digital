1. ANTECEDENTES HISTÓRICOS
===============================================================================

Para entender el nacimiento de C, debemos retroceder hasta finales
de la década de 1960. Los ordenadores de la época eran enormes,
costosos y mucho menos potentes que un teléfono móvil actual.

**1.1 La computación en los años 60**

Los programas se escribían principalmente en:

- **Ensamblador**: Lenguaje de bajo nivel específico para cada
  procesador. Era rápido pero terrible para escribir y mantener.
- **FORTRAN**: Orientado a cálculos científicos y matemáticos.
- **COBOL**: Orientado a aplicaciones comerciales y de negocios.
- **ALGOL**: Un lenguaje pionero en estructuras de control, pero
  principalmente teórico.

Ninguno de estos lenguajes era adecuado para programar sistemas
operativos. El ensamblador era el estándar de facto para ese tipo
de trabajo, y eso era un problema enorme.

```c
/* Avance rápido: esto en C sería inimaginable en ensamblador */
#include <stdio.h>

int main(void) {
    printf("Antes de C, los sistemas operativos se escribían\n");
    printf("en ensamblador. Cada instrucción era específica\n");
    printf("para un procesador. No había portabilidad.\n");
    return 0;
}
```

**1.2 El problema de la portabilidad**

En los años 60, si querías que un sistema operativo funcionara en
una nueva máquina, tenías que reescribirlo casi completamente. No
había una forma de escribir código que funcionara en múltiples
arquitecturas sin cambios sustanciales.

Esta falta de portabilidad era:
- **Cara**: Reescribir software para cada máquina multiplicaba los
  costos.
- **Lenta**: Adaptar un sistema operativo a un nuevo hardware podía
  llevar meses o años.
- **Propensa a errores**: Cada reescritura introducía nuevos bugs.

La industria necesitaba desesperadamente un lenguaje de alto nivel
que pudiera acceder al hardware directamente pero que fuera
portátil entre máquinas.

===============================================================================
2. LOS LABORATORIOS BELL Y UNIX
===============================================================================

**2.1 Bell Telephone Laboratories**

Los Laboratorios Bell (Bell Labs) eran el centro de investigación
de AT&T, la compañía telefónica estadounidense. En los años 60 y 70,
Bell Labs era un paraíso para la investigación: científicos e
ingenieros tenían libertad para explorar ideas innovadoras sin
presiones comerciales inmediatas.

Fue en este entorno donde ocurrió una convergencia de talento que
cambiaría la computación para siempre.

**2.2 El proyecto MULTICS**

En 1965, Bell Labs se asoció con el MIT y General Electric para
desarrollar MULTICS (Multiplexed Information and Computing Service),
un ambicioso sistema operativo interactivo que permitía a múltiples
usuarios trabajar simultáneamente.

MULTICS era visionario pero excesivamente complejo. Bell Labs se
retiró del proyecto en 1969. Sin embargo, la experiencia dejó
lecciones valiosas.

```c
#include <stdio.h>

int main(void) {
    printf("MULTICS fue el fracaso que enseñó el camino.\n");
    printf("De sus cenizas nació UNIX.\n");
    printf("Y de UNIX nació C.\n");
    return 0;
}
```

**2.3 El nacimiento de UNIX**

En el verano de 1969, Ken Thompson, Dennis Ritchie y otros
investigadores de Bell Labs comenzaron a trabajar en un nuevo
sistema operativo. Thompson escribió la primera versión de UNIX
en un PDP-7, una computadora relativamente pequeña y económica.

UNIX (originalmente Unics, como un juego de palabras con MULTICS)
fue diseñado desde el principio con tres principios:
- **Simplicidad**: Herramientas pequeñas que hacen una cosa bien.
- **Portabilidad**: Escrito para ser fácil de mover a nueva hardware.
- **Modularidad**: Componentes que se combinan para hacer tareas
  complejas.

La primera versión de UNIX estaba escrita en ensamblador. Pero
Thompson y Ritchie sabían que para que UNIX fuera realmente portátil,
necesitaban un lenguaje de alto nivel.

===============================================================================
3. DENNIS RITCHIE Y LA CREACIÓN DE C
===============================================================================

**3.1 El contexto: el lenguaje B**

Antes de C, existió B. Ken Thompson creó el lenguaje B en 1969,
basado en BCPL (Basic Combined Programming Language) de Martin
Richards. B era un lenguaje tipado débilmente, sin tipos de datos
explícitos, donde todo era una "palabra" (word).

B funcionaba, pero tenía limitaciones serias:
- No soportaba tipos de datos diferentes (solo palabras).
- El código generado era ineficiente.
- No era adecuado para todas las tareas de programación de sistemas.

```c
#include <stdio.h>

int main(void) {
    printf("B fue el precursor inmediato de C.\n");
    printf("Pero B era demasiado limitado.\n");
    printf("C nació de la necesidad de superar esas limitaciones.\n");
    return 0;
}
```

**3.2 El nacimiento de C (1972)**

Dennis Ritchie comenzó a trabajar en una extensión de B que
incorporara tipos de datos. El resultado fue C, cuyo nombre continúa
alfabéticamente donde B terminó.

Las innovaciones clave que Ritchie introdujo en C incluyeron:

- **Sistema de tipos**: `char`, `int`, `float`, `double` y derivados.
- **Estructuras**: Agrupación de datos heterogéneos (`struct`).
- **Punteros**: Acceso directo a memoria con aritmética de punteros.
- **Preprocesador**: Directivas para inclusión de archivos y macros.

**3.3 La personalidad de Dennis Ritchie**

Dennis MacAlistair Ritchie (1941-2011) era conocido por su enfoque
minimalista y su habilidad para diseñar sistemas elegantes. No
buscaba añadir características por añadirlas; cada elemento de C
tenía un propósito claro.

Ritchie dijo una vez: *"C es un lenguaje que combina la eficiencia
de un lenguaje ensamblador con la legibilidad de un lenguaje de alto
nivel. O al menos, esa era la intención."*

```c
#include <stdio.h>

int main(void) {
    printf("Dennis Ritchie (1941-2011)\n");
    printf("━━━━━━━━━━━━━━━━━━━━━━━\n");
    printf("Creador del lenguaje C\n");
    printf("Co-creador del sistema UNIX\n");
    printf("Premio Turing 1983\n");
    printf("Premio IEEE Richard W. Hamming 1990\n");
    printf("Medalla Nacional de Tecnología 1998\n");
    printf("━━━━━━━━━━━━━━━━━━━━━━━\n");
    printf("Su legado vive en cada programa en C.\n");
    return 0;
}
```

===============================================================================
4. LA PUBLICACIÓN DE K&R C (1978)
===============================================================================

**4.1 El libro que definió un lenguaje**

En 1978, Brian Kernighan y Dennis Ritchie publicaron *"The C
Programming Language"*, conocido cariñosamente como **K&R** o
"el libro blanco de C". Este libro, de apenas 228 páginas en su
primera edición, se convirtió en el estándar de facto del lenguaje
durante más de una década.

El libro K&R es famoso por:
- Su claridad y concisión.
- Su enfoque en ejemplos prácticos (incluyendo el legendario
  "Hello, World").
- Su cobertura completa pero sucinta del lenguaje.

**4.2 El estilo K&R**

El libro estableció lo que se conoce como "estilo K&R" para formatear
código C. Aunque hoy hay muchos estilos (Allman, BSD, GNU), el estilo
K&R sigue siendo popular, especialmente en código de sistemas.

```c
/* Estilo K&R: la llave de apertura está en la misma línea */
#include <stdio.h>

int main() {
    int i;
    for (i = 0; i < 5; i++) {
        printf("Clásico e inconfundible: estilo K&R\n");
    }
    return 0;
}
```

**4.3 La influencia de Kernighan**

Brian Kernighan no solo coescribió el libro; también contribuyó al
desarrollo temprano de C y es el creador de muchos programas UNIX
clásicos. Kernighan tiene una habilidad especial para explicar
conceptos complejos de manera simple, y su estilo pedagógico influyó
en generaciones de programadores.

Cuando se le pregunta cómo se pronuncia "C", Kernighan responde:
*"Solo C. No hay 'lenguaje C'. Es C, como una letra del alfabeto."*

5. LA ESTANDARIZACIÓN: ANSI C (1989)
===============================================================================

**5.1 La necesidad de un estándar**

Para finales de los años 80, C se había vuelto inmensamente popular.
Pero cada compilador implementaba características ligeramente
diferentes. Lo que funcionaba con un compilador podía no funcionar
con otro. Esto era un caos para los programadores que querían código
portátil.

**5.2 ANSI C (C89/C90)**

En 1983, el Instituto Nacional Estadounidense de Estándares (ANSI)
formó el comité X3J11 para estandarizar C. El resultado fue aprobado
en 1989 y se conoce como **ANSI C** o **C89**. En 1990, la
Organización Internacional de Normalización (ISO) lo adoptó como
**C90** (ISO/IEC 9899:1990).

Las principales adiciones del estándar ANSI incluyeron:

- **Prototipos de funciones**: Declarar los parámetros que recibe
  una función, permitiendo al compilador verificar tipos.
- **Punteros `void*`**: Tipo genérico para punteros.
- **Tipos enumerados (`enum`)**: Conjuntos de constantes enteras
  con nombre.
- **Calificadores de tipo**: `const` y `volatile`.
- **La biblioteca estándar**: Un conjunto oficial de funciones y
  cabeceras.

```c
/* Los prototipos de función fueron una innovación clave de ANSI C */
#include <stdio.h>

/* Prototipo: declara la función antes de usarla */
int suma(int a, int b);

int main(void) {
    int resultado = suma(5, 3);
    printf("5 + 3 = %d\n", resultado);
    return 0;
}

/* Definición de la función */
int suma(int a, int b) {
    return a + b;
}
```

**5.3 El legado de la estandarización**

La estandarización de C fue un hito en la historia de la programación.
Por primera vez, un lenguaje de sistemas tenía una especificación
precisa que garantizaba que el código escrito según el estándar
funcionaría en cualquier plataforma conforme.

Este concepto de **portabilidad real** cambió la industria. Empresas
como Adobe, Microsoft, Oracle y cientos más adoptaron C para
desarrollar productos que funcionaran en múltiples sistemas
operativos.

===============================================================================
6. C99: LA MODERNIZACIÓN (1999)
===============================================================================

**6.1 Nueve años de evolución**

Después de una década sin cambios importantes, el comité ISO publicó
una nueva revisión del estándar en 1999, conocida como **C99** o
C9X durante su desarrollo.

**6.2 Novedades principales de C99**

C99 introdujo numerosas mejoras que los programadores habían estado
solicitando durante años:

- **Declaraciones en cualquier lugar**: Ya no era necesario declarar
  todas las variables al inicio de un bloque.
- **Comentarios de una línea**: `//` como en C++.
- **Tipos enteros de ancho fijo**: `int8_t`, `int16_t`, `int32_t`,
  `int64_t` y sus versiones sin signo, definidos en `<stdint.h>`.
- **Tipo `_Bool`**: Tipo booleano nativo.
- **Tipo `long long int`**: Entero de al menos 64 bits.
- **Literales de punto flotante hexadecimales**.
- **Arrays de longitud variable (VLA)**: Arrays cuyo tamaño se
  determina en tiempo de ejecución.
- **Funciones inline**: Sugerencia al compilador para expandir
  una función en el lugar de llamada.
- **Inicializadores designados**: Inicializar miembros específicos
  de un array o estructura.

```c
/* C99 permitió declarar variables donde se necesitan */
#include <stdio.h>
#include <stdint.h>
#include <stdbool.h>

int main(void) {
    // Comentario de una línea (nuevo en C99)
    for (int i = 0; i < 5; i++) {  // declaración en el for
        printf("i = %d\n", i);
    }

    // Tipo booleano
    _Bool es_valido = 1;  // o usar <stdbool.h> y 'bool'

    // Enteros de ancho fijo
    int32_t numero = 42;
    uint8_t pequeno = 255;

    printf("int32_t: %d, uint8_t: %d\n", numero, pequeno);
    return 0;
}
```

**6.3 Impacto y recepción**

C99 fue bien recibido, pero su adopción fue lenta. Microsoft, en
particular, nunca implementó completamente C99 en su compilador
MSVC. Esto fragmentó el ecosistema y retrasó la adopción de las
nuevas características.

===============================================================================
7. C11: MADUREZ Y SEGURIDAD (2011)
===============================================================================

**7.1 Contexto**

Doce años después de C99, el comité ISO publicó **C11** (ISO/IEC
9899:2011). Esta revisión se centró menos en añadir características
y más en mejorar la seguridad y la previsibilidad del lenguaje.

**7.2 Principales cambios en C11**

- **Modelo de memoria y multihilo**: Soporte oficial para
  programación concurrente con hilos y variables atómicas.
- **Tipos genéricos**: `_Generic` para escribir macros que se
  comportan diferentemente según el tipo del argumento.
- **`_Static_assert`**: Afirmaciones en tiempo de compilación.
- **`_Noreturn`**: Indicador de funciones que no retornan.
- **`char16_t` y `char32_t`**: Tipos para caracteres Unicode
  (con `<uchar.h>`).
- **Anonimización de estructuras y uniones**: Miembros anónimos
  dentro de estructuras y uniones.
- **Eliminación de características peligrosas**: `gets()` fue
  oficialmente eliminado de la biblioteca estándar.
- **Funciones `_s` (bounds-checking)**: Versiones seguras de
  funciones de cadena (opcionales).

```c
/* C11: aserciones en tiempo de compilación */
#include <stdio.h>
#include <assert.h>

/* Afirmación en tiempo de compilación */
_Static_assert(sizeof(int) >= 4, "Se necesita int de al menos 4 bytes");

int main(void) {
    printf("C11: aserciones en tiempo de compilación verificadas.\n");
    printf("El programa compila porque sizeof(int) >= 4.\n");
    return 0;
}
```

**7.3 Filosofía de C11**

C11 representa la madurez de C. El comité reconoció que el lenguaje
necesitaba adaptarse a la programación moderna (multihilo, Unicode)
pero sin perder su esencia: un lenguaje pequeño, eficiente y cercano
al hardware.

===============================================================================
8. C17 Y C23: LA ERA CONTEMPORÁNEA
===============================================================================

**8.1 C17 (ISO/IEC 9899:2018)**

C17 fue principalmente una revisión correctiva. No introdujo nuevas
características del lenguaje, sino que:
- Corrigió errores e inconsistencias del estándar C11.
- Aclaró comportamientos indefinidos.
- Mejoró la especificación técnica.

**8.2 C23 (ISO/IEC 9899:2023)**

La revisión más reciente, publicada en 2023, trae cambios significativos:

- **`nullptr`**: Una palabra clave para representar punteros nulos
  (similar a C++).
- **`bool`** como palabra clave nativa, reemplazando a `_Bool`.
- **`typeof`**: Operador para obtener el tipo de una expresión.
- **Atributos**: Sintaxis `[[attr]]` similar a C++.
- **`#elifdef` y `#elifndef`**: Nuevas directivas de preprocesador.
- **Mejoras en `constexpr`**: Expresiones constantes más flexibles.
- **Eliminación de características obsoletas**: Eliminación completa
  de funciones peligrosas.

```c
/* C23: nullptr y typeof (ejemplo conceptual) */
#include <stdio.h>

int main(void) {
    /* En C23:
    int* ptr = nullptr;
    typeof(ptr) otro_ptr = nullptr;
    printf("nullptr disponible en C23\n");
    */
    printf("C23 continúa la evolución de C hacia el futuro.\n");
    printf("El lenguaje no se detiene; sigue mejorando.\n");
    return 0;
}
```

10. LA INFLUENCIA DE C EN OTROS LENGUAJES
===============================================================================

C no solo es importante por sí mismo; ha sido la base o la inspiración
para decenas de otros lenguajes:

**10.1 C++ (1985)**

Creado por Bjarne Stroustrup como "C con clases". Originalmente era
un preprocesador que generaba código C. Hoy es uno de los lenguajes
más potentes y complejos.

**10.2 Java (1995)**

La sintaxis de Java está fuertemente inspirada en C y C++. James
Gosling, creador de Java, tomó la sintaxis familiar de C y la combinó
con un modelo de objetos más simple.

**10.3 C# (2000)**

El lenguaje insignia de Microsoft para .NET. Su sintaxis es
directamente derivada de C y C++, con influencias de Java.

**10.4 Python (1991)**

Aunque Python no se parece sintácticamente a C, su implementación
principal (CPython) está escrita en C. Además, muchas de las
bibliotecas nativas de Python están en C.

**10.5 JavaScript (1995)**

La sintaxis de JavaScript (llaves, paréntesis, punto y coma) hereda
directamente de C. Brendan Eich la diseñó para que pareciera familiar
a los programadores de C y Java.

**10.6 Rust (2015), Go (2009), Zig (2016)**

Estos lenguajes modernos compiten directamente con C en el espacio
de los lenguajes de sistemas. Todos reconocen a C como su predecesor
y referente.

```c
#include <stdio.h>

int main(void) {
    printf("Lenguajes influenciados por C:\n");
    printf("C++  Java  C#   Python (CPython)\n");
    printf("JS   Rust  Go   Zig   Perl\n");
    printf("PHP  Swift Objective-C D\n");
    printf("Prácticamente todos los lenguajes modernos\n");
    printf("llevan un poco de C en su ADN.\n");
    return 0;
}
```

===============================================================================
11. EL LEGADO DE C EN LA ACTUALIDAD
===============================================================================

**11.1 ¿Por qué sigue siendo relevante?**

En 2026, C sigue siendo fundamental por varias razones:

- **Infraestructura crítica**: No se puede reescribir el kernel de
  Linux, ni los millones de líneas de código C embebido en sistemas
  de misión crítica.
- **Rendimiento insuperable**: Para aplicaciones donde cada nanosegundo
  cuenta, C sigue siendo la mejor opción.
- **Hardware nuevo**: Cada nuevo procesador, microcontrolador o sistema
  embebido tiene un compilador de C antes que cualquier otro lenguaje.
- **Conocimiento fundamental**: C es la base sobre la que se construye
  la comprensión profunda de la computación.

**11.2 El futuro de C**

Lejos de desaparecer, C continúa evolucionando con el estándar C23.
El lenguaje se adapta a las necesidades modernas (multihilo, Unicode,
seguridad) sin perder su esencia.

Como dijo Dennis Ritchie: *"C es un lenguaje que permite hacer mucho
con poco. Y eso nunca pasará de moda."*

```c
#include <stdio.h>

int main(void) {
    printf("C no es un lenguaje del pasado.\n");
    printf("C es el lenguaje del presente que construye el futuro.\n");
    printf("Y sigue siendo la columna vertebral de la computación.\n");
    return 0;
}
```

RESUMEN DEL CAPÍTULO
===============================================================================

- C fue creado por Dennis Ritchie en 1972 en los Laboratorios Bell
  para desarrollar el sistema operativo UNIX.
- Antes de C existió el lenguaje B de Ken Thompson, basado en BCPL.
- K&R (Kernighan & Ritchie) publicaron "The C Programming Language"
  en 1978, que fue el estándar de facto durante una década.
- ANSI C (C89/C90) fue la primera estandarización oficial del lenguaje.
- C99 introdujo mejoras importantes como `<stdint.h>`, `_Bool`,
  comentarios `//` y declaraciones en cualquier lugar.
- C11 añadió soporte para multihilo, `_Generic`, `_Static_assert`
  y eliminó funciones inseguras como `gets()`.
- C17 fue una revisión correctiva; C23 trae `nullptr`, `typeof`,
  `bool` nativo y atributos `[[...]]`.
- C ha influido en prácticamente todos los lenguajes modernos: C++,
  Java, C#, JavaScript, Python, Rust, Go y muchos más.
- C sigue siendo fundamental para sistemas operativos, embebidos,
  y software de alto rendimiento.

PREGUNTAS FRECUENTES
===============================================================================

❓ **¿Por qué se llama C? ¿Hubo un lenguaje A o B?**
Sí. El lenguaje B de Ken Thompson precedió a C. Antes de B existió
BCPL (Basic Combined Programming Language) de Martin Richards. Y a
su vez, BCPL derivaba de CPL. El nombre "C" continúa alfabéticamente
donde B terminó. No hubo un lenguaje "A" propiamente dicho en esta
línea.

❓ **¿Por qué C no tiene recolector de basura?**
Porque cuando se creó C (1972), los recursos de hardware eran muy
limitados. Un recolector de basura habría consumido memoria y tiempo
de CPU que simplemente no estaban disponibles. Además, C fue diseñado
para programación de sistemas, donde el control preciso de la memoria
es esencial.

❓ **¿Cuál es el futuro de C?**
El futuro de C es brillante en los ámbitos donde siempre ha dominado:
sistemas embebidos, kernels, compiladores y software de alto
rendimiento. El estándar C23 demuestra que el lenguaje sigue
evolucionando. Sin embargo, para aplicaciones de usuario final,
lenguajes más modernos están ganando terreno.

❓ **¿Debo aprender C89, C99, C11 o C23?**
En 2026, deberías aprender C11 o C17 como base, ya que son los
estándares más ampliamente soportados. C23 es el futuro pero su
adopción total tomará años. En este curso usaremos **C11** como
estándar de referencia, con notas sobre características de C23.

🏗️ **Arquitecto de la Historia: El Museo de C Interactivo**

Diseña un programa en C llamado `museo_c.c` que simule un museo
interactivo sobre la historia del lenguaje C. El programa debe
presentar un menú con las siguientes "salas" del museo:

1. **Sala 1969**: El origen — B y los Laboratorios Bell.
2. **Sala 1972**: El nacimiento de C — Dennis Ritchie.
3. **Sala 1978**: K&R — El libro que cambió todo.
4. **Sala 1989**: ANSI C — La estandarización.
5. **Sala 1999**: C99 — La modernización.
6. **Sala 2011**: C11 — Madurez y seguridad.
7. **Sala 2023**: C23 — El futuro.
8. **Salir**

Cuando el usuario selecciona una sala, el programa debe mostrar una
descripción informativa de al menos 10 líneas sobre ese período,
incluyendo datos clave, nombres importantes y el contexto histórico.

El reto: el programa debe ser atractivo visualmente. Usa caracteres
ASCII para dibujar el menú, bordes decorativos (`═`, `║`, `╔`, `╗`,
`╚`, `╝`), y pausas entre pantallas (pista: `getchar()` espera una
tecla). El usuario debe poder navegar entre las salas libremente
hasta que decida salir.

Este desafío pondrá a prueba tu capacidad para estructurar un
programa con múltiples opciones, entrada de usuario y presentación
profesional.
