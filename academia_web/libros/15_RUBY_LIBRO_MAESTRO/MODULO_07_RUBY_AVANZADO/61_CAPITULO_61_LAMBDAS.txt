-------------------------------------------------------------------------------
  1. INTRODUCCIÓN: BLOQUES, PROCS Y LAMBDAS
-------------------------------------------------------------------------------

  En Ruby, todo bloque es una pieza de código anónimo que se pasa a un
  método. Sin embargo, los bloques no son objetos. Para convertirlos en
  objetos, usamos Proc. Los lambdas son un tipo especial de Proc con
  reglas más estrictas.

  Veamos la jerarquía conceptual:

    Bloque (no es objeto)
      ↓
    Proc.new (objeto Proc, comportamiento laxo)
      ↓
    lambda (objeto Proc, comportamiento estricto)

  La diferencia fundamental radica en dos aspectos:
    1. Aridad (cantidad de argumentos)
    2. Comportamiento de return

-------------------------------------------------------------------------------
  2. CREACIÓN DE PROCS CON Proc.new
-------------------------------------------------------------------------------

  Un Proc se crea con Proc.new y se ejecuta con .call:

    mi_proc = Proc.new { |x| puts "Hola, #{x}" }
    mi_proc.call("Mundo")
    # => Hola, Mundo

  También podemos convertir un bloque en proc usando el operador &:

    def convertir(&bloque)
      bloque
    end

    mi_proc = convertir { |nombre| "Hola #{nombre}" }
    puts mi_proc.call("Ruby")
    # => Hola Ruby

  Proc.new ignora los argumentos faltantes (asigna nil) y tolera
  argumentos sobrantes (los ignora).

-------------------------------------------------------------------------------
  3. CREACIÓN DE LAMBDAS
-------------------------------------------------------------------------------

  Un lambda se crea con la palabra clave lambda o con la sintaxis ->:

    mi_lambda = lambda { |x| puts "Hola, #{x}" }
    mi_lambda.call("Mundo")

    # Sintaxis lambda stabby:
    mi_lambda = ->(x) { puts "Hola, #{x}" }
    mi_lambda["Mundo"]

  Los lambdas son estrictos con la aridad: si esperan 1 argumento y
  reciben 0 o 2, lanzan ArgumentError.

-------------------------------------------------------------------------------
  4. DIFERENCIA CLAVE #1: ARIDAD
-------------------------------------------------------------------------------

  Proc.new (laxo):

    proc_laxo = Proc.new { |a, b| "a=#{a}, b=#{b}" }
    proc_laxo.call(1)          # => "a=1, b="
    proc_laxo.call(1, 2, 3)   # => "a=1, b=2"

  lambda (estricto):

    lambda_estricto = ->(a, b) { "a=#{a}, b=#{b}" }
    lambda_estricto.call(1)    # ArgumentError: wrong number of arguments
    lambda_estricto.call(1, 2) # => "a=1, b=2"
    lambda_estricto.call(1,2,3)# ArgumentError

  Esto hace que lambda sea más seguro para composición funcional.

-------------------------------------------------------------------------------
  5. DIFERENCIA CLAVE #2: COMPORTAMIENTO DE RETURN
-------------------------------------------------------------------------------

  Esta es la diferencia más importante y sutil.

  return dentro de un Proc.new:

    def metodo_con_proc
      p = Proc.new { return "salida desde proc" }
      p.call
      "esto nunca se ejecuta"
    end

    puts metodo_con_proc
    # => "salida desde proc"

  El return dentro del proc sale del método que lo contiene (metodo_con_proc).
  A esto se le llama "salto de contexto".

  return dentro de un lambda:

    def metodo_con_lambda
      l = -> { return "salida desde lambda" }
      l.call
      "esto SÍ se ejecuta"
    end

    puts metodo_con_lambda
    # => "esto SÍ se ejecuta"

  El return dentro del lambda solo sale del lambda, no del método
  contenedor.

-------------------------------------------------------------------------------
  6. EJEMPLOS DETALLADOS DE RETURN
-------------------------------------------------------------------------------

  Ejemplo 1: Proc.new con return en método simple

    def probar_proc
      proc = Proc.new { return "PROC: saliendo del método" }
      resultado = proc.call
      "Esto jamás se alcanza: #{resultado}"
    end

    puts probar_proc
    # "PROC: saliendo del método"

  Ejemplo 2: Lambda con return

    def probar_lambda
      lamb = -> { return "LAMBDA: saliendo del lambda" }
      resultado = lamb.call
      "Volvimos al método: #{resultado}"
    end

    puts probar_lambda
    # "Volvimos al método: LAMBDA: saliendo del lambda"

  Ejemplo 3: Proc.new dentro de un bloque iterador

    def iterar_con_proc
      [1, 2, 3].each do |n|
        p = Proc.new { return "corto en #{n}" }
        p.call
      end
      "completado"
    end

    puts iterar_con_proc
    # "corto en 1"

  Esto puede causar bugs difíciles de rastrear si no se entiende bien.
  Por eso muchos estilos de código recomiendan usar lambda cuando se
  necesita un comportamiento predecible.

-------------------------------------------------------------------------------
  7. INVOCACIÓN: .call vs .[] vs .yield vs .()
-------------------------------------------------------------------------------

  Todas estas formas invocan un proc o lambda:

    fn = ->(x) { x * 2 }

    fn.call(5)    # => 10
    fn[5]          # => 10
    fn.(5)         # => 10 (azúcar sintáctico)
    fn.yield(5)    # => 10 (menos común)

  La forma .[] es particularmente común en Ruby por su legibilidad:

    procesador = ->(x, y) { x + y }
    resultado = procesador[3, 4]  # => 7

  También existe ===, que permite usar procs en case/when:

    par = ->(n) { n.even? }
    case 4
    when par then "es par"
    else "es impar"
    end
    # => "es par"

-------------------------------------------------------------------------------
  8. CONVERSIÓN BLOQUE ↔ PROC CON &
-------------------------------------------------------------------------------

  El operador unario & convierte un bloque en un Proc y viceversa.

  De bloque a Proc (parámetro):

    def tomar_proc(&bloque)
      bloque.call(42)
    end

    tomar_proc { |n| n * 2 }
    # => 84

  De Proc a bloque (argumento):

    mi_proc = Proc.new { |n| n ** 2 }
    [1, 2, 3].map(&mi_proc)
    # => [1, 4, 9]

  Con lambdas funciona igual:

    cuadrado = ->(n) { n ** 2 }
    [1, 2, 3].map(&cuadrado)
    # => [1, 4, 9]

  También se puede usar & para convertir un símbolo a proc:

    ["a", "b", "c"].map(&:upcase)
    # => ["A", "B", "C"]

  Esto es equivalente a:

    ["a", "b", "c"].map { |s| s.upcase }

  & funciona llamando a to_proc sobre el objeto:

    class MiConversor
      def to_proc
        ->(x) { "convertido: #{x}" }
      end
    end

    ["a", "b"].map(&MiConversor.new)
    # => ["convertido: a", "convertido: b"]

-------------------------------------------------------------------------------
  9. PROC COMO CIERRE (CLOSURE)
-------------------------------------------------------------------------------

  Tanto procs como lambdas capturan el entorno léxico donde se crean:

    def crear_contador(inicial)
      contador = inicial
      -> { contador += 1 }
    end

    c = crear_contador(10)
    puts c.call  # => 11
    puts c.call  # => 12
    puts c.call  # => 13

  La variable contador permanece viva mientras el lambda exista.

-------------------------------------------------------------------------------
  10. PROCS Y LAMBDAS COMO PARÁMETROS
-------------------------------------------------------------------------------

  Puedes pasar múltiples procs/lambdas como argumentos normales:

    def operacion(a, b, suma, resta)
      puts suma.call(a, b)
      puts resta.call(a, b)
    end

    sum = ->(x, y) { x + y }
    res = ->(x, y) { x - y }

    operacion(10, 5, sum, res)
    # 15
    # 5

  También puedes devolverlos:

    def selector(tipo)
      case tipo
      when :suma  then ->(a, b) { a + b }
      when :resta then ->(a, b) { a - b }
      when :mult  then ->(a, b) { a * b }
      end
    end

    op = selector(:mult)
    puts op.call(6, 7)  # => 42

-------------------------------------------------------------------------------
  11. MÉTODO .lambda? PARA DETECTAR EL TIPO
-------------------------------------------------------------------------------

  Todo proc responde al método lambda?:

    p = Proc.new { |x| x }
    l = ->(x) { x }

    puts p.lambda?  # => false
    puts l.lambda?  # => true

  Esto es útil para librerías que reciben un proc y necesitan saber
  si deben tratarlo como método (lambda) o como bloque (proc).

-------------------------------------------------------------------------------
  12. CURRYING CON LAMBDAS
-------------------------------------------------------------------------------

  Los lambdas (no los procs) soportan currying nativamente:

    sum = ->(a, b, c) { a + b + c }
    sum_curried = sum.curry

    sum_5 = sum_curried.call(5)
    sum_5_3 = sum_5.call(3)
    puts sum_5_3.call(2)  # => 10

  Esto se estudiará en profundidad en el capítulo 63.

-------------------------------------------------------------------------------
  13. COMPOSICIÓN DE LAMBDAS
-------------------------------------------------------------------------------

  Podemos componer lambdas para crear pipelines funcionales:

    doblar = ->(x) { x * 2 }
    sumar_1 = ->(x) { x + 1 }
    cuadrado = ->(x) { x ** 2 }

    compose = ->(f, g) { ->(x) { f.call(g.call(x)) } }

    transformacion = compose.call(doblar, compose.call(sumar_1, cuadrado))
    # (x² + 1) * 2
    puts transformacion.call(3)  # (9 + 1) * 2 = 20

  Este patrón es la base del paradigma funcional en Ruby.

-------------------------------------------------------------------------------
  14. MÉTODOS DE CONVENIENCIA: Proc.new CON BLOQUE IMPLÍCITO
-------------------------------------------------------------------------------

  Si defines un método que acepta un bloque, puedes convertirlo en proc
  y almacenarlo:

    class Almacenador
      attr_reader :procs

      def initialize
        @procs = []
      end

      def guardar(&bloque)
        @procs << bloque
      end

      def ejecutar_todos
        @procs.each(&:call)
      end
    end

    a = Almacenador.new
    a.guardar { puts "Hola" }
    a.guardar { puts "Mundo" }
    a.ejecutar_todos
    # Hola
    # Mundo

-------------------------------------------------------------------------------
  15. LAMBDAS POR DEFECTO EN BIBLIOTECAS
-------------------------------------------------------------------------------

  Muchas gemas de Ruby eligen lambda sobre Proc.new por seguridad:

    # Ejemplo estilo Rack
    Middleware = ->(env) {
      status  = 200
      headers = { "Content-Type" => "text/html" }
      body    = ["<h1>Hola Mundo</h1>"]
      [status, headers, body]
    }

  Los lambdas son la opción preferida cuando se necesita predecibilidad.

-------------------------------------------------------------------------------
  16. ERRORES COMUNES CON PROC Y RETURN
-------------------------------------------------------------------------------

  Error #1: Return en proc fuera de un método.

    proc = Proc.new { return "error" }
    # LocalJumpError: unexpected return

  Un proc con return solo puede existir dentro de un método; si se
  define en el ámbito principal (top-level), lanza error.

  Error #2: Esperar que un proc valide argumentos.

    p = Proc.new { |a, b| a + b }
    p.call(1)  # => TypeError: nil + 1 (b es nil)

  El proc no advierte que falta un argumento.

  Error #3: Usar proc cuando se necesita composición segura.

    # Peligroso:
    proc = Proc.new { |x| return x * 2 if x > 10 }
    # Seguro:
    lamb = ->(x) { x * 2 if x > 10 }

-------------------------------------------------------------------------------
  17. COMPARATIVA COMPLETA: PROC vs LAMBDA
-------------------------------------------------------------------------------

  Característica            | Proc.new          | lambda
  --------------------------|-------------------|-------------------------
  Tipo de objeto            | Proc              | Proc (con flag lambda)
  Aridad                    | Laxa              | Estricta
  return sale del...        | método contenedor | solo del lambda
  Soporta currying          | No                | Sí
  lambda?                   | false             | true
  Uso típico                | callbacks, blocks | métodos anónimos
  Seguridad funcional       | Baja              | Alta
  Sintaxis de creación      | Proc.new { }      | ->() { } o lambda { }

-------------------------------------------------------------------------------
  18. EJEMPLO INTEGRADOR: PROCESAMIENTO DE DATOS
-------------------------------------------------------------------------------

  Combinemos lambdas para construir un pipeline de datos real:

    class DataPipeline
      def initialize
        @transformaciones = []
      end

      def agregar_transformacion(&transformacion)
        @transformaciones << transformacion
      end

      def ejecutar(dato)
        @transformaciones.reduce(dato) do |valor, transformacion|
          transformacion.call(valor)
        end
      end
    end

    pipeline = DataPipeline.new

    pipeline.agregar_transformacion(&->(x) { x.strip })
    pipeline.agregar_transformacion(&->(x) { x.downcase })
    pipeline.agregar_transformacion(&->(x) { x.gsub(/\s+/, "_") })
    pipeline.agregar_transformacion(&->(x) { "procesado_#{x}" })

    resultado = pipeline.ejecutar("  Hola MUNDO  ")
    puts resultado
    # => "procesado_hola_mundo"

-------------------------------------------------------------------------------
  19. CONSIDERACIONES DE RENDIMIENTO
-------------------------------------------------------------------------------

  Aunque la diferencia es marginal, los lambdas son ligeramente más
  rápidos que los procs porque no necesitan gestionar el contexto de
  retorno. Sin embargo, la diferencia rara vez es crítica.

  En benchmarks típicos:

    Lambdas:  ~0.08 μs por llamada
    Procs:    ~0.10 μs por llamada
    Bloques:  ~0.06 μs por llamada

  La claridad del código pesa más que estas diferencias.

-------------------------------------------------------------------------------
  20. BUENAS PRÁCTICAS
-------------------------------------------------------------------------------

  • Usa lambda por defecto para métodos anónimos.
  • Usa Proc.new solo cuando necesites comportamiento de bloque.
  • Evita return dentro de procs a menos que sepas exactamente qué haces.
  • Prefiere stabby lambda ->() sobre lambda {} por claridad visual.
  • Usa .call o .[] consistentemente en todo el proyecto.
  • Documenta la aridad esperada de lambdas públicos.
  • No abuses de los closures; pueden causar fugas de memoria.

puts "Composición (3): #{compuesta.call(3)}"  # (9+1)*2 = 20

# Ruby 2.7+ tiene compose (>>) y chain (<<)
doble_l = doble
increm_l = increm
composicion = doble_l >> increm_l
puts "Composición con >>: #{composicion.call(5)}"  # 5*2+1 = 11

# ---------------------------------------------------------------------------
# 13. COMPARATIVA FINAL
# ---------------------------------------------------------------------------

puts "=" * 60
puts "13. COMPARATIVA FINAL"
puts "=" * 60

================================================================================
  RESUMEN DEL CAPÍTULO
================================================================================

  En este capítulo hemos explorado a fondo las diferencias entre lambdas
  y procs en Ruby:

  • Los lambdas se crean con lambda {} o ->() {} y son estrictos con la
    aridad: verifican que el número de argumentos sea exacto.

  • Los procs se crean con Proc.new {} y son laxos: asignan nil a los
    argumentos faltantes e ignoran los sobrantes.

  • La diferencia de return es crucial: return en un proc abandona el
    método contenedor; return en un lambda solo abandona el lambda.

  • Ambos son objetos de la clase Proc, diferenciables con lambda?.

  • Se invocan con .call, .[], .(), o yield.

  • El operador & convierte bloques a procs y viceversa.

  • Los lambdas soportan currying nativo (.curry), lo que los hace
    ideales para programación funcional.

  • La composición de lambdas permite construir pipelines de datos.

  La regla de oro: usa lambda por defecto para métodos anónimos y
  Proc.new solo cuando necesites comportamiento de bloque (aridad laxa
  o return de método).

================================================================================
  PREGUNTAS FRECUENTES (FAQ)
================================================================================

  P: ¿Puedo convertir un lambda en un proc normal?
  R: No directamente. Debes crear un nuevo Proc.new que envuelva al
     lambda, pero pierdes las garantías de aridad.

  P: ¿Cuál es más rápido, lambda o proc?
  R: Los lambdas son ligeramente más rápidos (~20%), pero rara vez es
     un factor decisivo. Prioriza la claridad del código.

  P: ¿Debo siempre usar lambda en lugar de proc?
  R: Como regla general, sí. Los lambdas se comportan como métodos
     anónimos, que es lo que la mayoría de programadores espera. Usa
     proc solo cuando necesites aridad laxa o return de método.

  P: ¿Qué es el operador stabby lambda?
  R: Es la sintaxis ->() { } introducida en Ruby 1.9. Es la forma
     preferida de crear lambdas por su claridad visual.

  P: ¿Por qué mi proc lanza LocalJumpError?
  R: Porque contiene un return y fue creado en el ámbito principal
     (top-level), no dentro de un método. Los procs con return solo
     funcionan dentro de métodos.

  P: ¿Cómo paso un bloque a un método que espera un proc?
  R: Usa &bloque en la definición y bloque.call internamente. O
     define el parámetro con & y pásalo como &mi_proc.

  P: ¿Los lambdas son objetos de primera clase?
  R: Sí, los lambdas (y procs) son objetos Proc, se pueden asignar,
     pasar como argumentos, devolver y almacenar en colecciones.
