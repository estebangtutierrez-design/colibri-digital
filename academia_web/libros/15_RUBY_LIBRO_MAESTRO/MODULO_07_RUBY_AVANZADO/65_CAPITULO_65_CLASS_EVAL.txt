================================================================================
                        BIENVENIDA AL CAPÍTULO 65
================================================================================

  Bienvenido a uno de los temas más confusos y poderosos de Ruby:
  la diferencia entre class_eval e instance_eval. A simple vista,
  ambos evalúan código en un contexto, pero el "contexto" es muy
  diferente en cada caso.

  class_eval (también llamado module_eval) abre una clase o módulo
  como si estuvieras escribiendo código dentro de su definición.
  instance_eval abre un objeto como si estuvieras dentro de sus
  entrañas, permitiéndote acceder a sus variables de instancia y
  definirle métodos singleton.

  Dominar estos dos métodos es esencial para la metaprogramación seria:
  los DSLs, las configuraciones declarativas y las bibliotecas como
  ActiveRecord o RSpec dependen de esta distinción.

================================================================================
  FRASE DEL CAPÍTULO
================================================================================

  "class_eval abre la puerta a la clase; instance_eval abre la caja
   fuerte del objeto."
   — Paolo Perrotta, "Metaprogramming Ruby"

-------------------------------------------------------------------------------
  1. ¿QUÉ ES CLASS_EVAL?
-------------------------------------------------------------------------------

  class_eval es un método de Module que evalúa un bloque en el contexto
  de una clase o módulo. Dentro del bloque, self es la clase/módulo.

    class MiClase; end

    MiClase.class_eval do
      # self es MiClase
      def mi_metodo
        "hola desde class_eval"
      end
    end

    puts MiClase.new.mi_metodo
    # => "hola desde class_eval"

  Equivale a reabrir la clase y agregar métodos.

-------------------------------------------------------------------------------
  2. ¿QUÉ ES INSTANCE_EVAL?
-------------------------------------------------------------------------------

  instance_eval es un método de Object que evalúa un bloque en el
  contexto de una instancia específica. Dentro del bloque, self es
  el objeto receptor.

    obj = "Hola"
    obj.instance_eval do
      # self es "Hola" (el string)
      puts upcase   # "HOLA"
      puts size     # 4
    end

  instance_eval permite:
    • Acceder a variables de instancia del objeto.
    • Definir métodos singleton (propios del objeto).
    • Ejecutar código en el contexto del objeto.

-------------------------------------------------------------------------------
  3. DIFERENCIA CLAVE: CONTEXTO
-------------------------------------------------------------------------------

  Con class_eval, self es la clase.
  Con instance_eval, self es el objeto receptor.

    class Demo; end

    Demo.class_eval do
      puts "class_eval: self = #{self}"  # Demo
    end

    Demo.new.instance_eval do
      puts "instance_eval: self = #{self}"  # <Demo:0x...>
    end

  class_eval sobre una clase define métodos de instancia.
  instance_eval sobre una clase define métodos de clase (singleton).

-------------------------------------------------------------------------------
  4. CLASS_EVAL: DEFINIR MÉTODOS DE INSTANCIA
-------------------------------------------------------------------------------

  class_eval es ideal para añadir métodos a una clase existente:

    String.class_eval do
      def invertir_palabras
        split.reverse.join(" ")
      end
    end

    puts "Hola Mundo Ruby".invertir_palabras
    # => "Ruby Mundo Hola"

  También puedes usar class_eval con un string:

    String.class_eval("def nuevo; 'nuevo método'; end")
    puts "".nuevo  # => "nuevo método"

  Pero la forma con bloque es más segura y común.

-------------------------------------------------------------------------------
  5. INSTANCE_EVAL: ACCEDER A VARIABLES PRIVADAS
-------------------------------------------------------------------------------

  instance_eval permite ver y modificar variables de instancia privadas:

    class CajaFuerte
      def initialize
        @secreto = "TESORO-12345"
        @pin = 1234
      end

      private

      def pin_correcto?(pin)
        @pin == pin
      end
    end

    caja = CajaFuerte.new
    secreto = caja.instance_eval { @secreto }
    puts secreto  # => "TESORO-12345"

    # También podemos modificar
    caja.instance_eval { @secreto = "NUEVO-TESORO" }
    puts caja.instance_eval { @secreto }

  Esto es extremadamente poderoso y peligroso: rompe el encapsulamiento.

-------------------------------------------------------------------------------
  6. INSTANCE_EVAL SOBRE CLASES (MÉTODOS DE CLASE)
-------------------------------------------------------------------------------

  Cuando usamos instance_eval sobre una clase, self es la clase misma.
  Esto nos permite definir métodos de clase:

    class Config
    end

    Config.instance_eval do
      def api_key
        "clave-secreta"
      end

      def endpoint
        "https://api.example.com"
      end
    end

    puts Config.api_key     # => "clave-secreta"
    puts Config.endpoint    # => "https://api.example.com"

  Los métodos definidos con instance_eval sobre una clase son métodos
  singleton de esa clase (lo que comúnmente llamamos "métodos de clase").

-------------------------------------------------------------------------------
  7. CLASS_EVAL VS INSTANCE_EVAL EN LA PRÁCTICA
-------------------------------------------------------------------------------

  Ejemplo comparativo directo:

    class Ejemplo
      def initialize
        @valor = 42
      end
    end

    # class_eval: define método de instancia que accede a @valor
    Ejemplo.class_eval do
      def valor
        @valor
      end
    end

    # instance_eval en instancia: accede directamente a @valor
    obj = Ejemplo.new
    puts obj.valor                    # 42 (vía class_eval)
    puts obj.instance_eval { @valor } # 42 (vía instance_eval)

    # instance_eval en clase: define método de clase
    Ejemplo.instance_eval do
      def clase_valor
        "método de clase"
      end
    end
    puts Ejemplo.clase_valor  # "método de clase"

-------------------------------------------------------------------------------
  8. CLASS_EVAL CON STRING (VS BLOQUE)
-------------------------------------------------------------------------------

  class_eval acepta un string o un bloque:

    class Calculadora
    end

    # Con string:
    Calculadora.class_eval("def suma(a,b); a+b; end")

    # Con bloque:
    Calculadora.class_eval do
      def resta(a, b)
        a - b
      end
    end

    puts Calculadora.new.suma(5, 3)   # 8
    puts Calculadora.new.resta(5, 3)  # 2

  La forma con bloque es preferida por legibilidad y seguridad.

-------------------------------------------------------------------------------
  9. INSTANCE_EVAL PARA DSLs
-------------------------------------------------------------------------------

  instance_eval es la base de la mayoría de DSLs en Ruby:

    class Configuracion
      attr_reader :server, :database, :debug

      def initialize
        @server = "localhost"
        @database = "dev"
        @debug = false
      end

      def configurar(&bloque)
        instance_eval(&bloque)
      end

      private

      def server(s)
        @server = s
      end

      def database(d)
        @database = d
      end

      def debug(d)
        @debug = d
      end
    end

    config = Configuracion.new
    config.configurar do
      server "prod.example.com"
      database "produccion"
      debug true
    end

    puts config.server   # => "prod.example.com"

  Los métodos server, database, debug son privados y solo se llaman
  dentro del bloque de configuración.

-------------------------------------------------------------------------------
  10. EL PROBLEMA DEL CAMBIO DE self
-------------------------------------------------------------------------------

  Cuando usas instance_eval, self cambia. Esto puede causar problemas
  si dentro del bloque necesitas acceder al self original:

    class Helper
      def saludo
        "Hola desde Helper"
      end
    end

    class Usuario
      def initialize(nombre)
        @nombre = nombre
      end

      def configurar(&bloque)
        instance_eval(&bloque)
      end
    end

    helper = Helper.new
    user = Usuario.new("Ana")
    user.configurar do
      # self es user, no podemos llamar helper.saludo directamente
      puts @nombre
      # puts helper.saludo  # Sí funciona si capturamos helper
    end

  Solución: capturar el helper en una variable antes del bloque.

-------------------------------------------------------------------------------
  11. INSTANCE_EXEC Y CLASS_EXEC
-------------------------------------------------------------------------------

  Ruby también tiene instance_exec y class_exec que permiten pasar
  argumentos al bloque:

    class Ejemplo
      def initialize
        @x = 10
      end
    end

    obj = Ejemplo.new
    obj.instance_exec(5) do |arg|
      puts @x + arg  # 15
    end

  Esto es útil cuando el bloque recibe parámetros del exterior.

-------------------------------------------------------------------------------
  12. CLASS_EVAL CON HERENCIA
-------------------------------------------------------------------------------

  class_eval respeta la herencia:

    class Padre
    end

    class Hijo < Padre
    end

    Padre.class_eval do
      def metodo_padre
        "de padre"
      end
    end

    puts Hijo.new.metodo_padre  # "de padre" (heredado)

  Pero instance_eval sobre una clase hija no hereda los métodos
  singleton (de clase) de la clase padre:

    Padre.instance_eval do
      def metodo_clase
        "clase padre"
      end
    end

    # Hijo.metodo_clase  # NoMethodError (no se hereda)

-------------------------------------------------------------------------------
  13. USO EN GEMAS POPULARES
-------------------------------------------------------------------------------

  RSpec usa instance_eval para los bloques describe/it:

    RSpec.describe "algo" do
      it "funciona" do
        expect(1 + 1).to eq(2)
      end
    end

  ActiveRecord usa class_eval para definir asociaciones:

    class Usuario < ApplicationRecord
      has_many :posts
    end

  # has_many usa class_eval internamente para definir métodos.

  Rails routes usan instance_eval:

    Rails.application.routes.draw do
      resources :usuarios
      get "login", to: "sessions#new"
    end

-------------------------------------------------------------------------------
  14. MODULE_EVAL (ALIAS DE CLASS_EVAL)
-------------------------------------------------------------------------------

  module_eval es un alias de class_eval. Se usa con módulos:

    module MiModulo
    end

    MiModulo.module_eval do
      def metodo_modulo
        "definido en módulo"
      end
    end

    class Incluye
      include MiModulo
    end

    puts Incluye.new.metodo_modulo  # "definido en módulo"

-------------------------------------------------------------------------------
  15. EVAL SIN BLOQUE
-------------------------------------------------------------------------------

  Tanto class_eval como instance_eval pueden recibir un string:

    str_class = "def hola; 'mundo'; end"
    String.class_eval(str_class)
    puts "".hola  # "mundo"

    obj = "test"
    obj.instance_eval("def nuevo; 'nuevo'; end")
    puts obj.nuevo  # "nuevo"

  La forma con bloque es más segura (no evalúa strings arbitrarios).

-------------------------------------------------------------------------------
  16. ACCESO A CONSTANTES
-------------------------------------------------------------------------------

  class_eval respeta las constantes de la clase:

    class Exterior
      CONST = "exterior"
    end

    Exterior.class_eval do
      def obtener_const
        CONST
      end
    end

    puts Exterior.new.obtener_const  # "exterior"

  instance_eval no:

    Exterior.new.instance_eval do
      # puts CONST  # NameError (no encuentra CONST)
    end

-------------------------------------------------------------------------------
  17. PATRÓN: DSL CON INSTANCE_EVAL
-------------------------------------------------------------------------------

  Patrón completo para un DSL:

    class DSLBuilder
      def initialize
        @resultado = {}
      end

      def construir(&bloque)
        instance_eval(&bloque)
        @resultado
      end

      private

      def campo(nombre, valor)
        @resultado[nombre] = valor
      end

      def grupo(nombre, &bloque)
        sub = DSLBuilder.new
        @resultado[nombre] = sub.construir(&bloque)
      end
    end

    config = DSLBuilder.new.construir do
      campo :app, "MiApp"
      campo :version, "2.0"
      grupo :database do
        campo :host, "localhost"
        campo :port, 5432
      end
    end

    puts config.inspect

-------------------------------------------------------------------------------
  18. EVITAR PELIGROS CON INSTANCE_EVAL
-------------------------------------------------------------------------------

  Los principales peligros de instance_eval son:

  1. Rompe encapsulamiento: Puedes acceder a datos privados.
  2. Cambia self: Puede causar confusión en el código.
  3. Seguridad: evaluar strings puede ser riesgoso.
  4. Debugging: Los errores dentro de instance_eval pueden ser
     difíciles de rastrear.

  Buenas prácticas:
    • Usa instance_eval principalmente para DSLs controlados.
    • No uses instance_eval para acceder a datos privados en
      producción (solo en tests si es necesario).
    • Prefiere bloques en lugar de strings.
    • Documenta claramente cuándo cambia self.

-------------------------------------------------------------------------------
  19. COMPARATIVA COMPLETA
-------------------------------------------------------------------------------

  Característica      | class_eval              | instance_eval
  --------------------|-------------------------|-------------------------
  self es             | La clase/módulo         | El objeto receptor
  Define métodos de   | Instancia               | Singleton (del objeto)
  Accede a constants  | Sí                      | No
  Accede a vars inst  | No (solo desde método)  | Sí
  Usado para          | Abrir clases            | DSLs, singleton methods
  Método de           | Module                  | Object
  Alias               | module_eval             | —

class ConfigAnidada
  attr_reader :config

  def initialize
    @config = {}
  end

  def self.construir(&bloque)
    c = new
    c.instance_eval(&bloque)
    c.config
  end

  private

  def setting(nombre, valor = nil, &bloque)
    if bloque
      sub = ConfigAnidada.new
      sub.instance_eval(&bloque)
      @config[nombre] = sub.config
    else
      @config[nombre] = valor
    end

class TestSuite
  def initialize
    @tests = []
  end

  def self.describe(desc, &bloque)
    suite = new
    suite.instance_eval(&bloque)
    suite.ejecutar
  end

  def test(desc, &bloque)
    @tests << [desc, bloque]
  end

  def ejecutar
    @tests.each do |desc, bloque|
      print "  #{desc}... "
      begin
        resultado = instance_eval(&bloque)
        puts resultado ? "PASS" : "FAIL"
      rescue => e
        puts "ERROR: #{e.message}"
      end
    end
  end
end

TestSuite.describe "Matemáticas" do

    1 + 1 == 2
  end

  test "multiplicación" do

  end

class FactoryDSL
  class << self
    def define(nombre, &bloque)
      factory = new
      factory.instance_eval(&bloque)
      Object.const_set(nombre.to_s.capitalize, factory.construct_class)
    end
  end

  def initialize
    @atributos = []
    @metodos = {}
  end

  def attribute(nombre, tipo = nil)
    @atributos << nombre
  end

  def method(nombre, &bloque)
    @metodos[nombre] = bloque
  end

  def construct_class
    attrs = @atributos.dup
    meths = @metodos.dup

    Class.new do
      attrs.each { |a| attr_accessor a }

      define_method(:initialize) do |**vals|
        attrs.each do |a|
          send("#{a}=", vals[a]) if vals.key?(a)
        end
      end

      meths.each do |nombre, bloque|
        define_method(nombre, &bloque)
      end

      define_method(:to_s) do
        attrs.map { |a| "#{a}: #{send(a)}" }.join(", ")
      end
    end
  end
end

FactoryDSL.define :persona do
  attribute :nombre
  attribute :edad
  method :saludar do
    "Hola, soy #{@nombre}"
  end
end

p = Persona.new(nombre: "Ana", edad: 30)
puts p.saludar
puts p.to_s

  RESUMEN DEL CAPÍTULO
================================================================================

  En este capítulo hemos explorado class_eval e instance_eval:

  • class_eval (module_eval) evalúa código en el contexto de una clase
    o módulo. self es la clase. Define métodos de instancia.

  • instance_eval evalúa código en el contexto de un objeto. self es
    el objeto. Puede acceder a variables de instancia y definir métodos
    singleton.

  • instance_eval sobre una clase define métodos de clase (singleton).

  • class_eval respeta constantes; instance_eval no.

  • instance_eval es la base de la mayoría de DSLs en Ruby.

  • instance_exec y class_exec permiten pasar argumentos al bloque.

  • La forma con bloque es más segura que con string.

  • instance_eval rompe encapsulamiento: accede a atributos privados.

================================================================================
  PREGUNTAS FRECUENTES (FAQ)
================================================================================

  P: ¿Cuándo usar class_eval en lugar de reabrir la clase?
  R: Cuando necesitas definir métodos dinámicamente (con variables),
     o cuando el nombre de la clase se determina en tiempo de ejecución.

  P: ¿Puedo usar instance_eval para definir métodos de instancia?
  R: instance_eval define métodos singleton (del objeto), no de
     instancia. Para métodos de instancia usa class_eval.

  P: ¿Qué es más seguro, bloque o string?
  R: El bloque. El string evalúa código Ruby arbitrario y puede ser
     peligroso si incluye datos externos.

  P: ¿Cómo accedo al self original dentro de un instance_eval?
  R: Captura el self original en una variable antes del bloque, o
     usa instance_exec para pasarlo como argumento.

  P: ¿module_eval y class_eval son lo mismo?
  R: Sí, module_eval es un alias de class_eval. Se prefiere module_eval
     para módulos y class_eval para clases.
