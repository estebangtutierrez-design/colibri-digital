================================================================================
                        BIENVENIDA AL CAPÍTULO 63
================================================================================

  Bienvenido al arte de la transformación funcional. El currying, nombre
  que rinde homenaje al lógico Haskell Curry, es una técnica que permite
  descomponer una función de múltiples argumentos en una cadena de
  funciones de un argumento cada una.

  En Ruby, el currying se aplica naturalmente a los lambdas mediante el
  método .curry. Esta técnica no solo hace el código más elegante, sino
  que permite la reutilización parcial de funciones, la composición y
  la creación de DSLs funcionales de una potencia sorprendente.

  Prepárate para ver cómo una función de tres argumentos se convierte
  en tres funciones anidadas, cada una esperando su turno para recibir
  el siguiente parámetro.

================================================================================
  FRASE DEL CAPÍTULO
================================================================================

  "El currying es la descomposición de una función en una cadena de
   funciones, cada una con un solo argumento, revelando la estructura
   subyacente de la computación."
   — Haskell Curry

-------------------------------------------------------------------------------
  1. ¿QUÉ ES CURRYING?
-------------------------------------------------------------------------------

  Currying transforma una función que toma múltiples argumentos en una
  secuencia de funciones que toman un argumento cada una.

  Matemáticamente:
    f(a, b, c) → f(a)(b)(c)

  En Ruby:
    suma = ->(a, b, c) { a + b + c }
    suma_curry = suma.curry
    suma_curry.call(1).call(2).call(3)  # => 6

  Cada llamada devuelve una nueva función que espera el siguiente
  argumento.

-------------------------------------------------------------------------------
  2. CURRYING EN RUBY CON .curry
-------------------------------------------------------------------------------

  El método .curry está disponible en objetos Proc que sean lambdas
  (no funciona con Proc.new).

    suma = ->(a, b, c) { a + b + c }
    curried = suma.curry

    paso_1 = curried.call(1)
    paso_2 = paso_1.call(2)
    paso_3 = paso_2.call(3)

    puts paso_3  # => 6

  También puedes pasar múltiples argumentos en cada paso:

    curried.call(1, 2).call(3)   # => 6
    curried.call(1, 2, 3)        # => 6
    curried[1][2][3]             # => 6 (usando [])
    curried.(1).(2).(3)          # => 6

-------------------------------------------------------------------------------
  3. ARIDAD Y CURRYING
-------------------------------------------------------------------------------

  El currying respeta la aridad del lambda original:

    suma = ->(a, b, c) { a + b + c }
    puts suma.arity  # => 3

    curried = suma.curry

  Si llamas a curried con 3 argumentos, ejecuta directamente.
  Si llamas con 1 o 2, devuelve un nuevo lambda esperando el resto.

  Si superas la aridad, lanza ArgumentError.

-------------------------------------------------------------------------------
  4. APLICACIÓN PARCIAL VS CURRYING
-------------------------------------------------------------------------------

  Aunque relacionados, no son lo mismo:

  Aplicación parcial: Fijas algunos argumentos de una función y obtienes
  una función con menos parámetros.

    suma = ->(a, b, c) { a + b + c }
    suma_con_1_y_2 = ->(c) { suma.call(1, 2, c) }

  Currying: Descompones la función en una cadena de funciones unarias.

    suma.curry.call(1).call(2).call(3)

  El currying es una forma sistemática de lograr aplicación parcial.

-------------------------------------------------------------------------------
  5. EJEMPLOS PRÁCTICOS DE CURRYING
-------------------------------------------------------------------------------

  Ejemplo 1: Logging configurable

    log = ->(nivel, mensaje) { "[#{nivel.upcase}] #{mensaje}" }
    log_curried = log.curry

    info  = log_curried.call("info")
    error = log_curried.call("error")
    debug = log_curried.call("debug")

    puts info.call("Sistema iniciado")
    puts error.call("Fallo crítico")

  Ejemplo 2: Descuentos

    calcular_precio = ->(impuesto, descuento, precio) {
      precio * (1 + impuesto) * (1 - descuento)
    }
    calc = calcular_precio.curry

    precio_base = calc.call(0.21, 0.10)
    puts precio_base.call(100)  # Precio con IVA 21% y 10% desc

-------------------------------------------------------------------------------
  6. CURRYING CON PARÁMETROS POR DEFECTO
-------------------------------------------------------------------------------

  Los parámetros por defecto son ignorados para la aridad del currying:

    con_defecto = ->(a, b = 10) { a + b }
    puts con_defecto.arity  # => -2 (complejo)

  Es mejor evitar parámetros por defecto con currying.

-------------------------------------------------------------------------------
  7. CURRYING MANUAL (SIN .curry)
-------------------------------------------------------------------------------

  Podemos implementar currying manualmente:

    def curry(funcion, aridad = funcion.arity)
      ->(*args) {
        if args.size >= aridad
          funcion.call(*args)
        else
          curry(
            ->(*resto) { funcion.call(*args, *resto) },
            aridad - args.size
          )
        end
      }
    end

    suma = ->(a, b, c) { a + b + c }
    curried_manual = curry(suma)

    puts curried_manual.call(1).call(2).call(3)  # => 6

  Esta implementación revela la esencia del currying.

-------------------------------------------------------------------------------
  8. CURRYING CON PROCS (NO LAMBDAS)
-------------------------------------------------------------------------------

  Los procs creados con Proc.new NO tienen .curry. Pero podemos
  convertirlos a lambda:

    proc_obj = Proc.new { |a, b, c| a + b + c }
    # proc_obj.curry  # NoMethodError

    # Solución: convertir a lambda
    lambda_obj = ->(a, b, c) { proc_obj.call(a, b, c) }
    curried = lambda_obj.curry
    puts curried.call(1).call(2).call(3)

  O implementar curry manual como vimos arriba.

-------------------------------------------------------------------------------
  9. COMPOSICIÓN FUNCIONAL CON CURRYING
-------------------------------------------------------------------------------

  Currying y composición se potencian mutuamente:

    map = ->(fn, lista) { lista.map(&fn) }
    filter = ->(fn, lista) { lista.select(&fn) }
    reduce = ->(fn, inicial, lista) { lista.reduce(inicial, &fn) }

    map_curried = map.curry
    filter_curried = filter.curry
    reduce_curried = reduce.curry

    doblar = ->(x) { x * 2 }
    es_par = ->(x) { x.even? }
    sumar = ->(acc, x) { acc + x }

    pipeline = map_curried[doblar] >> filter_curried[es_par]

    datos = [1, 2, 3, 4, 5, 6]
    resultado = pipeline.call(datos)
    puts resultado.inspect  # => [4, 8, 12] (pares doblados)

-------------------------------------------------------------------------------
  10. CURRYING PARA DSL
-------------------------------------------------------------------------------

  El currying permite crear DSLs muy expresivos:

    consulta = ->(tabla, condicion, campos) {
      "SELECT #{campos.join(', ')} FROM #{tabla} WHERE #{condicion}"
    }.curry

    de_usuarios = consulta.call("usuarios")
    usuarios_activos = de_usuarios.call("activo = true")
    query = usuarios_activos.call(["id", "nombre", "email"])

    puts query
    # SELECT id, nombre, email FROM usuarios WHERE activo = true

-------------------------------------------------------------------------------
  11. CURRYING Y FIBRAS
-------------------------------------------------------------------------------

  Combinado con fibras, el currying permite evaluaciones parciales:

    suma_lazy = ->(a, b) {
      Fiber.new { a + b }
    }.curry

    fib = suma_lazy.call(5)
    # aún no se ha sumado
    resultado = fib.resume
    puts resultado  # 8 (suponiendo suma_lazy = ->(a,b){Fiber.new{a+b}})

-------------------------------------------------------------------------------
  12. MÉTODO curry CON ARGUMENTO DE ARIDAD
-------------------------------------------------------------------------------

  .curry puede recibir un argumento opcional que especifica la aridad:

    suma_flexible = ->(*args) { args.sum }
    curried = suma_flexible.curry(3)

    con_1 = curried.call(1)
    con_2 = con_1.call(2)
    puts con_2.call(3)  # => 6

  Esto es útil cuando el lambda usa argumentos variables (*args).

-------------------------------------------------------------------------------
  13. REUTILIZACIÓN DE CÓDIGO CON CURRYING
-------------------------------------------------------------------------------

  Ejemplo real: validación de formularios

    validar_longitud = ->(min, max, valor) {
      valor.length >= min && valor.length <= max
    }.curry

    usuario_valido  = validar_longitud.call(3, 20)
    password_valida = validar_longitud.call(8, 128)

    puts usuario_valido.call("juan")       # true
    puts usuario_valido.call("a")          # false
    puts password_valida.call("secreta")   # false (< 8)

-------------------------------------------------------------------------------
  14. BENEFICIOS DEL CURRYING
-------------------------------------------------------------------------------

  1. Reutilización: Creas funciones base y las especializas.
  2. Legibilidad: Las intenciones quedan explícitas.
  3. Composición: Las funciones unarias se componen fácilmente.
  4. Testing: Cada paso es testeable de forma aislada.
  5. Inmutabilidad: No hay estado mutable entre pasos.
  6. Pereza: Puedes diferir la ejecución hasta tener todos los args.

-------------------------------------------------------------------------------
  15. LIMITACIONES DEL CURRYING EN RUBY
-------------------------------------------------------------------------------

  1. Solo lambdas: Proc.new no soporta .curry.
  2. Aridad fija: Funciona mejor con aridad conocida.
  3. Rendimiento: Cada nivel de currying añade overhead.
  4. Legibilidad: Demasiado currying puede oscurecer el código.
  5. Estado compartido: Los closures pueden mutar estado si no se
     tiene cuidado.

-------------------------------------------------------------------------------
  16. CURRYING VS APLICACIÓN PARCIAL CON BIND
-------------------------------------------------------------------------------

  Otra forma de aplicar parcialmente es con Method#bind:

    class Calculadora
      def suma(a, b, c)
        a + b + c
      end
    end

    calc = Calculadora.new
    metodo_suma = calc.method(:suma)

    # Aplicación parcial con curry
    suma_parcial = metodo_suma.to_proc.curry.call(1)

    # Method#curry no existe, pero to_proc sí tiene curry

-------------------------------------------------------------------------------
  17. PATRONES FUNCIONALES CON CURRYING
-------------------------------------------------------------------------------

  Pipeline:

    transformar = ->(datos, *fns) {
      fns.reduce(datos) { |d, fn| fn.call(d) }
    }.curry

    pipeline = transformar.call([1, 2, 3])
    |> ->(arr) { arr.map { |x| x * 2 } }
    |> ->(arr) { arr.select(&:even?) }

  Nota: Ruby no tiene operador pipeline nativo, pero podemos simularlo.

-------------------------------------------------------------------------------
  18. EJEMPLO INTEGRADOR: SISTEMA DE NOTIFICACIONES
-------------------------------------------------------------------------------

  Usemos currying para construir un sistema de notificaciones flexible:

    notificar = ->(canal, formato, destinatario, mensaje) {
      msg = case formato
            when :texto then mensaje
            when :html  then "<p>#{mensaje}</p>"
            when :json  then { msg: mensaje }.to_json
            end
      "Enviando por #{canal} a #{destinatario}: #{msg}"
    }.curry

    email = notificar.call(:email)
    sms   = notificar.call(:sms)

    email_html = email.call(:html)
    sms_texto  = sms.call(:texto)

    puts email_html.call("admin@test.com", "Bienvenido")
    puts sms_texto.call("+123456789", "Tu código es 1234")

def curry_para_proc(proc_obj, aridad)
  ->(*args) {
    if args.size >= aridad
      proc_obj.call(*args)
    else
      curry_para_proc(
        ->(*resto) { proc_obj.call(*args, *resto) },
        aridad - args.size
      )
    end
  }
end

  RESUMEN DEL CAPÍTULO
================================================================================

  En este capítulo hemos explorado el currying en Ruby:

  • Currying transforma una función de múltiples argumentos en una
    cadena de funciones de un argumento cada una.

  • Se aplica con .curry sobre lambdas (no sobre Proc.new).

  • La aplicación parcial fija algunos argumentos de una función,
    mientras que el currying descompone la función sistemáticamente.

  • El currying permite crear funciones especializadas a partir de
    funciones generales (logging, descuentos, validaciones).

  • Se combina naturalmente con la composición funcional.

  • Se puede implementar currying manualmente para procs.

  • El método .curry acepta un argumento opcional para especificar
    la aridad cuando la función usa parámetros variables.

  • El currying es una herramienta poderosa para crear DSLs y
    pipelines de procesamiento.

================================================================================
  PREGUNTAS FRECUENTES (FAQ)
================================================================================

  P: ¿Por qué .curry solo funciona con lambdas?
  R: Porque los lambdas tienen aridad estricta, necesaria para saber
     cuándo una función curry está completa. Los procs tienen aridad
     laxa y no pueden determinar cuándo tienen suficientes argumentos.

  P: ¿Currying y aplicación parcial son lo mismo?
  R: No. La aplicación parcial fija algunos argumentos (p.ej., fijar
     a=1 en f(a,b)). El currying descompone en funciones unarias. El
     currying es una forma de lograr aplicación parcial, pero no son
     equivalentes.

  P: ¿Cuándo debo usar currying?
  R: Cuando necesites crear muchas variaciones de una función base
     (como logs con diferentes niveles, descuentos, validaciones),
     cuando construyas DSLs, o cuando trabajes con composición funcional.

  P: ¿El currying tiene impacto en rendimiento?
  R: Sí, cada nivel de currying añade una llamada a función adicional.
     Para la mayoría de aplicaciones no es significativo, pero en
     bucles muy ajustados puede notarse.

  P: ¿Puedo hacer currying con métodos de objetos?
  R: Sí, usa object.method(:nombre).to_proc.curry para convertir un
     método a lambda currificable.
