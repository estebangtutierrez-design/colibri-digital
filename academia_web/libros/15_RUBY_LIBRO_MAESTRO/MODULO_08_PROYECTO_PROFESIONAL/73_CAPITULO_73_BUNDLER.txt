BIENVENIDA
──────────
Bienvenido, maestro orquestador. Bundler es la herramienta que
garantiza que tu aplicación Ruby siempre use las gemas correctas
en las versiones correctas. En este capítulo aprenderás desde la
sintaxis del Gemfile hasta técnicas avanzadas de gem groups,
deployment y resolución de conflictos.

FRASE DEL SABIO
───────────────
"Un Gemfile bien escrito es un mapa de confianza. Tu aplicación
funcionará igual hoy, mañana y en producción." — Profesor Búho AI

OBJETIVOS DEL CAPÍTULO
──────────────────────
1. Dominar la sintaxis completa del Gemfile.
2. Comprender Gemfile.lock y su importancia.
3. Usar gem groups para organizar dependencias.
4. Ejecutar bundle install, exec, update correctamente.
5. Resolver conflictos de dependencias como un experto.
6. Configurar Bundler para entornos de producción.

¿QUÉ ES BUNDLER?
─────────────────

Bundler es el gestor de dependencias de facto de Ruby. Lee tu
Gemfile, resuelve las dependencias de forma transitiva y genera
un Gemfile.lock que congela las versiones exactas.

```ruby
# ============================================================
# Verificar que Bundler está instalado
# ============================================================
# $ bundler --version
# $ gem list bundler

# ============================================================
# Instalar Bundler (si no lo tienes)
# ============================================================
# $ gem install bundler
# $ gem install bundler -v 2.4.0
```

# ============================================================
# Instala las gemas del Gemfile

# $ bundle install --without production

# ============================================================
# Ejecuta un comando en el contexto de Bundler
# $ bundle exec ruby app.rb

# ============================================================
# Actualiza TODAS las gemas (regenera Gemfile.lock)

# ============================================================

RESOLVIENDO CONFLICTOS DE DEPENDENCIAS
───────────────────────────────────────

```ruby
# ============================================================
# Escenario: Dos gemas requieren versiones incompatibles
# ============================================================
# Ejemplo de conflicto:
#   gema_a requiere nokogiri ~> 1.14
#   gema_b requiere nokogiri ~> 1.15

# Solución 1: Especificar versiones compatibles
gem 'nokogiri', '~> 1.15'  # Forzar una versión que ambas acepten

# Solución 2: Usar gemas alternativas
gem 'gema_b', '~> 2.0'  # Versión más nueva que use nokogiri >= 1.14

# Solución 3: Congelar la versión problemática
gem 'loofah', '2.21.0'  # Gema intermedia que causa el conflicto

# ============================================================
# Debugging de dependencias
# ============================================================
# $ bundle why nokogiri
# $ bundle exec gem dependency nokogiri

# ============================================================
# Inspeccionar árbol de dependencias
# ============================================================
# $ bundle exec gem list --local
# $ bundle exec gem dependency rails

# ============================================================
# Solucionar "Gem::LoadError"
# ============================================================
# 1. bundle install
# 2. bundle update gema_problematica
# 3. bundle exec (no olvides usarlo)
# 4. bundle pristine (restaurar gemas)
```

EJEMPLO COMPLETO: GESTOR DE DEPENDENCIAS
─────────────────────────────────────────

```ruby
# ============================================================
# Script que gestiona dependencias automáticamente
# ============================================================
#!/usr/bin/env ruby
# scripts/gestor_dependencias.rb

require 'bundler'
require 'json'

class GestorDependencias
  def initialize
    @definition = Bundler.definition
  end

  def reporte
    {
      gemas: gemas_info,
      conflictos: detectar_conflictos,
      salud: verificar_salud,
      estadisticas: estadisticas
    }
  end

  def gemas_info
    @definition.dependencies.map do |dep|
      spec = @definition.specs[dep.name].first
      {
        nombre: dep.name,
        version_requerida: dep.requirement.to_s,
        version_instalada: spec&.version&.to_s,
        grupos: dep.groups,
        fuente: spec&.source&.to_s || 'rubygems',
      }
    end
  end

  def detectar_conflictos
    conflictos = []
    @definition.dependencies.each do |dep|
      specs = @definition.resolve.materialize([dep.name], false)
      if specs.size > 1
        conflictos << { gema: dep.name, versiones: specs.map(&:version) }
      end
    end
    conflictos
  end

  def verificar_salud
    begin
      Bundler.require
      { estado: 'saludable', mensaje: 'Todas las gemas cargadas correctamente' }
    rescue Bundler::GemNotFound, LoadError => e
      { estado: 'problema', mensaje: e.message }
    end
  end

  def estadisticas
    specs = @definition.specs
    {
      total_gemas: specs.size,
      total_dependencias_directas: @definition.dependencies.size,
      grupos_disponibles: @definition.dependencies.map(&:groups).flatten.uniq,
      plataformas: specs.platforms
    }
  end

  def exportar_json(ruta = 'reporte_gemas.json')
    File.write(ruta, JSON.pretty_generate(reporte))
    puts "Reporte exportado a #{ruta}"
  end
end

if $PROGRAM_NAME == __FILE__
  gestor = GestorDependencias.new
  puts JSON.pretty_generate(gestor.reporte)
end
```

─────────────────────
- Bundler resuelve y congela dependencias Ruby.
- El Gemfile define fuentes, gemas, versiones y grupos.
- Gemfile.lock es el contrato que garantiza reproducibilidad.
- `bundle install` instala gemas; `bundle update` las actualiza.
- `bundle exec` ejecuta comandos en el contexto de Bundler.
- Los grupos organizan gemas por entorno (development, test, prod).
- Bundler en producción usa modo deployment y frozen.
- Los conflictos se resuelven especificando versiones compatibles.
- Los binstubs evitan tener que escribir `bundle exec`.
- Bundler-audit revisa vulnerabilidades en las dependencias.

PREGUNTAS FRECUENTES (FAQ)
───────────────────────────
P: ¿Debo incluir Gemfile.lock en el control de versiones?
R: Sí, siempre. Es el contrato que garantiza que todos los entornos
   tengan exactamente las mismas gemas.

P: ¿Cuál es la diferencia entre `bundle install` y `bundle update`?
R: `bundle install` respeta Gemfile.lock. `bundle update` lo
   regenera resolviendo de nuevo todas las dependencias.

P: ¿Por qué debo usar `bundle exec`?
R: Porque Bundler prepara el entorno para que las gemas correctas
   estén disponibles. Sin él, Ruby usa las gemas del sistema.

P: ¿Cómo manejo gemas específicas de plataforma?
R: Usa `platforms:` en el Gemfile o `bundle lock --add-platform`.

P: ¿Qué hago si `bundle install` falla en producción?
R: Verifica que Gemfile.lock existe, que las gemas están en
   RubyGems.org, y que la versión de Ruby es compatible.

DESAFÍO DEL ORQUESTADOR
────────────────────────
Crea un sistema de gestión de dependencias llamado 'bundle-manager'
que sea una CLI interactiva con las siguientes características:
1. Escanea un proyecto y genera un reporte completo de dependencias.
2. Detecta conflictos potenciales antes de instalar.
3. Sugiere actualizaciones seguras basadas en SemVer.
4. Permite crear snapshots de Gemfile.lock y restaurarlos.
5. Genera gráficos ASCII del árbol de dependencias.
6. Integra con bundler-audit para reportar vulnerabilidades.
7. Envía notificaciones cuando haya actualizaciones críticas.

Este desafío demuestra que dominas Bundler a nivel de arquitecto
y puedes construir herramientas que todo el equipo agradecerá.
