  BIENVENIDA
  ----------
  Bienvenido al Capítulo 12.  En el capítulo anterior dominaste if/else,
  la estructura de decisión fundamental.  Pero hay ocasiones en las que
  la lógica negativa (si NO ocurre algo, entonces haz esto) es más natural
  y expresiva.

  unless es la respuesta de Ruby a esas situaciones.  Mientras if ejecuta
  código cuando una condición es verdadera, unless lo ejecuta cuando la
  condición es falsa.  Es, esencialmente, if !condicion.

  Pero unless no es solo azúcar sintáctica.  Usado correctamente, hace
  que tu código fluya de manera más natural, especialmente en validaciones
  y "guard clauses".  En este capítulo aprenderás cuándo y cómo usar unless,
  cuándo evitarlo, y cómo combinarlo con else y modificadores.

  Prepárate para descubrir por qué los rubyistas aman unless.

  12.1  EL UNLESS MÁS SENCILLO
  --------------------------------------------------

  unless ejecuta un bloque de código solo si la condición es falsa.
  Es el opuesto lógico de if.

  ------------------------------------------------------------------
  Código:
  ------------------------------------------------------------------

  # Ejemplo 1: unless básico
  edad = 15

  unless edad >= 18
    puts "Eres menor de edad."
  end

  # Equivalente con if:
  if !(edad >= 18)
    puts "Eres menor de edad."
  end

  # Ejemplo 2: unless con método booleano
  archivo_existe = false

  unless archivo_existe
    puts "El archivo no existe. Creándolo..."
  end

  # Ejemplo 3: unless con nil
  usuario = nil

  unless usuario
    puts "No hay usuario autenticado."
  end

  # Ejemplo 4: unless con array vacío
  items = []

  unless items.any?
    puts "El carrito de compras está vacío."
  end

  # Ejemplo 5: unless en un método
  def enviar_email(usuario)
    unless usuario.email_confirmado?
      puts "Email no enviado: dirección no confirmada."
      return
    end

    puts "Enviando email a #{usuario.email}..."
  end

  usuario = Struct.new(:email, :email_confirmado?).new("test@test.com", false)
  enviar_email(usuario)

  ------------------------------------------------------------------
  Explicación:
  ------------------------------------------------------------------
  - unless = "a menos que" / "si no es cierto que".
  - El bloque se ejecuta solo si la condición es false o nil.
  - Es más legible que if !condicion en muchos casos.
  - unless no acepta elsif (solo else).
  - Ideal para guard clauses y validaciones rápidas.

  12.2  UNLESS / ELSE
  --------------------------------------------------

  Aunque menos común, unless puede llevar else.  Sin embargo, muchos
  rubyistas consideran que unless/else es CONFUSO y prefieren usar
  if/else en su lugar.

  ------------------------------------------------------------------
  Código:
  ------------------------------------------------------------------

  # Ejemplo 1: unless/else
  edad = 20

  unless edad >= 18
    puts "Menor de edad"
  else
    puts "Mayor de edad"
  end

  # Equivalente (y más claro):
  if edad >= 18
    puts "Mayor de edad"
  else
    puts "Menor de edad"
  end

  # Ejemplo 2: unless/else con variable
  conectado = false

  unless conectado
    estado = "Desconectado"
  else
    estado = "Conectado"
  end

  puts "Estado: #{estado}"

  # Mejor con ternario:
  estado = conectado ? "Conectado" : "Desconectado"
  puts "Estado: #{estado}"

  # Ejemplo 3: unless/else para entender el flujo
  # (NO recomendado, solo para comprensión)
  usuario = "invitado"

  unless usuario == "admin"
    puts "Acceso restringido"
  else
    puts "Bienvenido administrador"
  end

  # Mejor así:
  if usuario == "admin"
    puts "Bienvenido administrador"
  else
    puts "Acceso restringido"
  end

  # Ejemplo 4: caso donde unless/else tiene sentido
  # (condición negativa que fluye naturalmente)
  error = true

  unless error
    puts "Procesando solicitud..."
  else
    puts "Error detectado. Deteniendo proceso."
  end

  # Aún así, es discutible. if !error sería igual de claro.

  ------------------------------------------------------------------
  Explicación:
  ------------------------------------------------------------------
  - unless/else existe pero es controvertido.
  - La regla general: si usas else con unless, probablemente deberías
    usar if/else (es más natural leer la condición positiva).
  - Algunos equipos prohíben unless/else en sus guías de estilo.
  - Prefiere if/else a menos que la negación sea muy natural.

  12.3  EL MODIFICADOR UNLESS
  --------------------------------------------------

  El modificador unless es una de las características más elegantes de
  Ruby.  Permite ejecutar una acción a menos que se cumpla una condición.

  ------------------------------------------------------------------
  Código:
  ------------------------------------------------------------------

  # Ejemplo 1: modificador unless básico
  puts "El archivo no existe" unless File.exist?("datos.txt")

  # Ejemplo 2: guard clause en métodos
  def procesar_pago(usuario, monto)
    return "Saldo insuficiente" unless usuario.saldo >= monto

    # Procesar pago...
    "Pago procesado por $#{monto}"
  end

  usuario = Struct.new(:saldo).new(100)
  puts procesar_pago(usuario, 50)   # => Pago procesado...
  puts procesar_pago(usuario, 200)  # => Saldo insuficiente

  # Ejemplo 3: validación de parámetros
  def crear_usuario(nombre, email)
    return "Nombre requerido" unless nombre && !nombre.empty?
    return "Email requerido" unless email && !email.empty?

    "Usuario #{nombre} creado con email #{email}"
  end

  puts crear_usuario("", "test@test.com")  # => Nombre requerido
  puts crear_usuario("Ana", "")            # => Email requerido
  puts crear_usuario("Ana", "a@b.com")     # => Usuario Ana creado...

  # Ejemplo 4: modificador unless en iteración
  usuarios = ["Ana", nil, "Luis", nil, "Carlos"]
  usuarios.each do |u|
    next unless u  # salta si es nil
    puts "Usuario: #{u}"
  end

  # Ejemplo 5: log condicional
  debug = false
  puts "Valor de x: 42" unless !debug  # Equivalente a: puts ... if debug

  # Pero mejor:
  puts "Valor de x: 42" if debug

  # Ejemplo 6: unless con método que devuelve boolean
  def archivo_valido?(ruta)
    File.exist?(ruta) && File.readable?(ruta)
  end

  puts "Archivo inválido o inaccesible" unless archivo_valido?("/etc/hosts")

  # Ejemplo 7: unless para evitar errores
  datos = nil
  puts "Procesando datos..." unless datos.nil?

  # Más ruby way:
  puts "Procesando datos..." if datos

  ------------------------------------------------------------------
  Explicación:
  ------------------------------------------------------------------
  - El modificador unless ejecuta la acción si la condición es false.
  - Es ideal para guard clauses: return/next unless condición.
  - Mejora la legibilidad cuando la condición es una negación natural.
  - No puede llevar else.
  - Es una de las señas de identidad del código Ruby idiomático.

  12.4  UNLESS CON CONDICIONES COMPUESTAS
  --------------------------------------------------

  Aunque unless puede usarse con && y ||, debes tener cuidado:
  las condiciones compuestas con unless pueden ser difíciles de leer.

  ------------------------------------------------------------------
  Código:
  ------------------------------------------------------------------

  # Ejemplo 1: unless con && (dos condiciones falsas)
  edad = 15
  tiene_permiso = false

  unless edad >= 18 && tiene_permiso
    puts "No puedes acceder."
  end

  # Esto se ejecuta si NO es cierto que (edad >= 18 Y tiene_permiso)
  # O sea: si edad < 18 O !tiene_permiso (Ley de De Morgan)

  # Equivalente con if (más claro):
  if !(edad >= 18 && tiene_permiso)
    puts "No puedes acceder."
  end

  # O mejor (usando Ley de De Morgan):
  if edad < 18 || !tiene_permiso
    puts "No puedes acceder."
  end

  # Ejemplo 2: unless con || (confuso)
  nombre = nil
  email = nil

  unless nombre || email
    puts "Debes proporcionar nombre o email."
  end

  # Equivalente con if (más claro):
  if !nombre && !email
    puts "Debes proporcionar nombre o email."
  end

  # Ejemplo 3: GUÍA — Evita unless con condiciones compuestas
  # ❌ Confuso:
  unless usuario.activo? && !usuario.suspendido? || usuario.admin?
    # ...
  end

  # ✅ Claro:
  if !(usuario.activo? && !usuario.suspendido?) && !usuario.admin?
    # ...
  end

  # ✅ Mejor aún: extraer a método
  def acceso_permitido?(usuario)
    (usuario.activo? && !usuario.suspendido?) || usuario.admin?
  end

  unless acceso_permitido?(usuario)
    # ...
  end

  ------------------------------------------------------------------
  Explicación:
  ------------------------------------------------------------------
  - unless con && y || puede ser confuso por las leyes de De Morgan.
  - unless (A && B) equivale a if !A || !B.
  - unless (A || B) equivale a if !A && !B.
  - Si la condición tiene más de un operador, considera extraerla a
    un método o usar if en su lugar.

  12.5  UNLESS COMO GUARD CLAUSE
  --------------------------------------------------

  El uso más común y aceptado de unless es como "guard clause" al
  inicio de métodos y bloques.  Esta técnica valida condiciones al
  inicio y sale temprano, evitando anidamiento.

  ------------------------------------------------------------------
  Código:
  ------------------------------------------------------------------

  # Ejemplo 1: guard clause con return
  def dividir(a, b)
    return "Error: división entre cero" unless b != 0

    a / b
  end

  puts dividir(10, 2)  # => 5
  puts dividir(10, 0)  # => Error: división entre cero

  # Ejemplo 2: múltiples guard clauses
  def registrar_usuario(nombre, email, edad)
    return "Nombre inválido" unless nombre.is_a?(String) && !nombre.empty?
    return "Email inválido" unless email =~ /\A[\w+\-.]+@[a-z\d\-]+(\.[a-z\d\-]+)*\.[a-z]+\z/i
    return "Debes ser mayor de edad" unless edad >= 18

    "Usuario #{nombre} registrado exitosamente"
  end

  puts registrar_usuario("", "a@b.com", 25)              # => Nombre inválido
  puts registrar_usuario("Ana", "invalido", 25)          # => Email inválido
  puts registrar_usuario("Ana", "a@b.com", 15)           # => Debes ser mayor...
  puts registrar_usuario("Ana", "a@b.com", 25)           # => Usuario registrado...

  # Ejemplo 3: guard clause en iteración
  [1, nil, 2, nil, 3].each do |num|
    next unless num
    puts "Número: #{num}"
  end

  # Ejemplo 4: guard clause con next en archivos
  archivos = ["datos.csv", nil, "config.yml", "secreto.txt"]
  archivos.each do |archivo|
    next unless archivo
    next unless archivo.end_with?(".csv", ".yml")
    puts "Procesando: #{archivo}"
  end

  # Ejemplo 5: guard clause en bloque con select
  usuarios = [
    { nombre: "Ana", activo: false },
    { nombre: nil, activo: true },
    { nombre: "Luis", activo: true }
  ]

  usuarios.each do |u|
    next unless u[:nombre]
    next unless u[:activo]
    puts "Notificando a #{u[:nombre]}..."
  end

  # Ejemplo 6: guard clause en método de clase
  class Validador
    def self.validar_password(password)
      return "Debe tener al menos 8 caracteres" unless password.length >= 8
      return "Debe contener una mayúscula" unless password =~ /[A-Z]/
      return "Debe contener un número" unless password =~ /\d/
      return "Debe contener un carácter especial" unless password =~ /[^a-zA-Z0-9]/

      "Contraseña válida"
    end
  end

  puts Validador.validar_password("abc")          # => Debe tener al menos 8...
  puts Validador.validar_password("abcdefgh")     # => Debe contener una mayúscula
  puts Validador.validar_password("Abcdefgh")     # => Debe contener un número
  puts Validador.validar_password("Abcdef1h")     # => Debe contener un carácter...
  puts Validador.validar_password("Abcd@f1h")     # => Contraseña válida

  ------------------------------------------------------------------
  Explicación:
  ------------------------------------------------------------------
  - Los guard clauses con unless hacen el código más plano y legible.
  - Cada guard clause valida una condición y sale inmediatamente.
  - No hay anidamiento: cada validación está en su propia línea.
  - Este patrón es especialmente útil para validar parámetros.
  - También es ideal para filtrar elementos en iteraciones (next unless).

  12.6  UNLESS VS IF !CONDICION
  --------------------------------------------------

  ¿Cuándo usar unless y cuándo usar if con negación?  La respuesta
  depende de la naturalidad de la expresión.

  ------------------------------------------------------------------
  Código:
  ------------------------------------------------------------------

  # Regla general: usa unless cuando la condición es una negación
  # que se lee naturalmente como "a menos que"

  # ✅ BUEN uso de unless:
  unless usuario.autenticado?
    puts "Por favor inicia sesión"
  end

  # "A menos que el usuario esté autenticado, pide iniciar sesión"
  # Suena natural.

  # ✅ BUEN uso de if !:
  if !usuario.admin? && !usuario.moderador?
    puts "No tienes permisos"
  end

  # "Si no es admin Y no es moderador, no tienes permisos"
  # Con unless sería confuso:
  # unless usuario.admin? || usuario.moderador?  (Ley de De Morgan)

  # ❌ MAL uso de unless (condición compuesta):
  unless usuario.activo? && !usuario.baneado?
    puts "Cuenta no disponible"
  end

  # ✅ Mejor:
  if !usuario.activo? || usuario.baneado?
    puts "Cuenta no disponible"
  end

  # ❌ MAL uso de unless (con operadores de comparación != ya incluidos):
  unless rol != "admin"
    puts "Acceso total"
  end

  # Esto es confuso.  unless + != es doble negación.
  # ✅ Mejor:
  if rol == "admin"
    puts "Acceso total"
  end

  # GUÍA RÁPIDA:
  # unless vacio?  => ✅ (natural)
  # unless existe? => ✅ (natural)
  # unless activo? => ✅ (natural)
  # unless x > 5 && y < 10 => ❌ (confuso)
  # unless x.nil? => ❌ (mejor: if x)
  # unless rol != "admin" => ❌ (doble negación, usa if rol == "admin")

  # Comparación práctica:
  valor = nil

  # Opción 1 (confusa):
  unless valor
    puts "No hay valor"
  end

  # Opción 2 (clara):
  if valor.nil?
    puts "No hay valor"
  end

  # Opción 3 (más ruby way):
  puts "No hay valor" if valor.nil?

  ------------------------------------------------------------------
  Explicación:
  ------------------------------------------------------------------
  - unless es más legible cuando la condición es simple y negativa.
  - if !condicion es mejor cuando la condición es compuesta.
  - Evita unless + doble negación (unless x != y).
  - Si dudas entre unless e if, pregúntate: ¿suena natural en inglés
    o español decir "a menos que..."?

  12.7  UNLESS EN EL MUNDO REAL
  --------------------------------------------------

  Veamos ejemplos prácticos de unless en aplicaciones reales.

  ------------------------------------------------------------------
  Código:
  ------------------------------------------------------------------

  # Ejemplo 1: Control de acceso a API
  class ApiController
    def initialize(token)
      @token = token
    end

    def procesar_solicitud
      return { error: "Token no proporcionado" } unless @token
      return { error: "Token inválido" } unless token_valido?(@token)

      { exito: true, data: "Datos sensibles..." }
    end

    private

    def token_valido?(token)
      token.length >= 32 && token.start_with?("sk_")
    end
  end

  api = ApiController.new(nil)
  puts api.procesar_solicitud  # => {error: "Token no proporcionado"}

  api = ApiController.new("token_invalido")
  puts api.procesar_solicitud  # => {error: "Token inválido"}

  api = ApiController.new("sk_" + "a" * 30)
  puts api.procesar_solicitud  # => {exito: true, data: "Datos sensibles..."}

  # Ejemplo 2: Procesamiento de pedidos
  class Pedido
    attr_reader :items, :pagado, :enviado

    def initialize(items)
      @items = items
      @pagado = false
      @enviado = false
    end

    def pagar(monto)
      return { error: "Sin items en el pedido" } unless items.any?
      return { error: "Monto incorrecto" } unless monto == calcular_total

      @pagado = true
      { exito: true, mensaje: "Pedido pagado: $#{monto}" }
    end

    def enviar
      return { error: "Pedido no pagado" } unless @pagado
      return { error: "Pedido ya enviado" } unless @enviado

      @enviado = true
      { exito: true, mensaje: "Pedido enviado" }
    end

    def calcular_total
      items.sum { |item| item[:precio] * item[:cantidad] }
    end
  end

  pedido = Pedido.new([
    { nombre: "Laptop", precio: 1000, cantidad: 1 }
  ])

  puts pedido.pagar(999)[:error]  # => Monto incorrecto
  puts pedido.pagar(1000)[:mensaje]  # => Pedido pagado: $1000
  puts pedido.enviar[:mensaje]  # => Pedido enviado

  # Ejemplo 3: Sistema de notificaciones
  class Notificador
    def initialize(usuario)
      @usuario = usuario
    end

    def enviar_notificacion(mensaje)
      return "Usuario no configurado" unless @usuario
      return "Email no verificado" unless @usuario[:email_verificado]
      return "Preferencias no configuradas" unless @usuario[:notificaciones_activas]

      enviar_email(@usuario[:email], mensaje)
      "Notificación enviada a #{@usuario[:nombre]}"
    end

    private

    def enviar_email(email, mensaje)
      puts "  [EMAIL enviado a #{email}: #{mensaje}]"
    end
  end

  usuario1 = { nombre: "Ana", email: "ana@test.com",
               email_verificado: false, notificaciones_activas: true }
  usuario2 = { nombre: "Luis", email: "luis@test.com",
               email_verificado: true, notificaciones_activas: true }

  notif = Notificador.new(usuario1)
  puts notif.enviar_notificacion("Bienvenido!")  # => Email no verificado

  notif = Notificador.new(usuario2)
  puts notif.enviar_notificacion("Bienvenido!")  # => Notificación enviada a Luis

  # Ejemplo 4: unless en tests (pseudo-código)
  def test_suma
    resultado = 2 + 2
    assert resultado == 4, "Se esperaba 4"
    assert resultado.is_a?(Integer), "Debe ser entero"
  end

  def assert(condicion, mensaje)
    unless condicion
      puts "FALLO: #{mensaje}"
      return false
    end
    true
  end

  test_suma

  ------------------------------------------------------------------
  Explicación:
  ------------------------------------------------------------------
  - unless es ideal para validaciones al inicio de métodos.
  - Cada guard clause con unless es una línea independiente y clara.
  - Los métodos quedan planos (sin anidamiento) y fáciles de leer.
  - Es un patrón consistente en todo el código Ruby profesional.

  12.8  BUENAS PRÁCTICAS CON UNLESS
  --------------------------------------------------

  ------------------------------------------------------------------
  Código:
  ------------------------------------------------------------------

  # 1. Usa unless para condiciones negativas SIMPLES
  # ✅ Bueno:
  unless usuario.activo?
    puts "Usuario inactivo"
  end

  # 2. NO uses unless con else (usa if/else en su lugar)
  # ❌ Evitar:
  unless x > 5
    puts "menor o igual"
  else
    puts "mayor"
  end

  # ✅ Mejor:
  if x > 5
    puts "mayor"
  else
    puts "menor o igual"
  end

  # 3. NO uses unless con condiciones compuestas complejas
  # ❌ Evitar:
  unless x > 0 && y > 0 || z > 0
    puts "nada positivo"
  end

  # ✅ Mejor (extraer a método):
  def alguno_positivo?(x, y, z)
    x > 0 || y > 0 || z > 0
  end

  unless alguno_positivo?(x, y, z)
    puts "nada positivo"
  end

  # 4. Usa unless con modificador para guard clauses
  # ✅ Bueno:
  return unless usuario
  next unless item.valido?

  # 5. No uses unless con variables de asignación
  # ❌ Evitar (confuso):
  unless x = obtener_valor
    puts "No hay valor"
  end

  # ✅ Mejor:
  x = obtener_valor
  unless x
    puts "No hay valor"
  end

  # 6. Prefiere if valor en lugar de unless valor.nil?
  # ❌ Menos idiomático:
  unless valor.nil?
    puts valor
  end

  # ✅ Más idiomático:
  if valor
    puts valor
  end

  # 7. Consistencia: si usas unless, úsalo en todo el proyecto
  # con el mismo estilo.

  # 8. Documenta el uso de unless si la lógica no es obvia
  # Aunque el código debería documentarse solo, a veces un comentario
  # aclara la intención.

  ------------------------------------------------------------------
  Explicación:
  ------------------------------------------------------------------
  - Las guías de estilo de Ruby (RuboCop) recomiendan unless solo para
    condiciones simples.
  - El uso principal de unless es como guard clause.
  - Si te encuentras escribiendo unless con else, reconsidera.
  - La comunidad Ruby es consistente en estas recomendaciones.

  12.9  EJERCICIOS DE REFACTORIZACIÓN
  --------------------------------------------------

  A continuación, ejercicios para transformar código con if ! a unless.

  ------------------------------------------------------------------
  Código:
  ------------------------------------------------------------------

  # --- EJERCICIO GUIADO: Refactoriza estos if ! a unless ---

  # Original 1:
  if !usuario.autenticado?
    puts "Redirigiendo al login..."
  end

  # Refactorizado 1:
  unless usuario.autenticado?
    puts "Redirigiendo al login..."
  end

  # Original 2:
  if !archivo.exist?
    puts "Creando archivo..."
    File.write(archivo, "")
  end

  # Refactorizado 2:
  unless archivo.exist?
    puts "Creando archivo..."
    File.write(archivo, "")
  end

  # Original 3:
  if !email.empty? && !email.nil?
    puts "Enviando a #{email}"
  end

  # Refactorizado 3:
  # (Nota: aquí unless sería confuso por la condición compuesta)
  # Mejor mantener if ! o refactorizar:
  if email && !email.empty?
    puts "Enviando a #{email}"
  end

  # Original 4 (método):
  def procesar(pedido)
    if !pedido.valido?
      return "Pedido inválido"
    end
    # procesar...
  end

  # Refactorizado 4:
  def procesar(pedido)
    return "Pedido inválido" unless pedido.valido?
    # procesar...
  end

  # Original 5 (iteración):
  lista.each do |item|
    if !item.completado?
      puts "Pendiente: #{item.nombre}"
    end
  end

  # Refactorizado 5:
  lista.each do |item|
    next if item.completado?
    puts "Pendiente: #{item.nombre}"
  end

  # O con unless (menos común en iteraciones, pero válido):
  lista.each do |item|
    puts "Pendiente: #{item.nombre}" unless item.completado?
  end

  ------------------------------------------------------------------
  Explicación:
  ------------------------------------------------------------------
  - La refactorización de if ! a unless es mecánica y simple.
  - Pero debes asegurarte de que el resultado sea más legible.
  - No todos los if ! deben convertirse a unless.
  - La regla de oro: si "a menos que" suena natural, usa unless.

  12.10  COMPARATIVA: IF VS UNLESS
  --------------------------------------------------

  ------------------------------------------------------------------
  Código:
  ------------------------------------------------------------------

  # Misma lógica, diferentes expresiones:

  condicion = false

  # if
  if condicion
    puts "Es verdadero"
  end

  # unless
  unless condicion
    puts "Es falso"
  end

  # if con negación
  if !condicion
    puts "Es falso"
  end

  # Todos son válidos, pero expresan diferente intención.

  # --- TABLA DE EQUIVALENCIAS ---

  # if condicion          ==  unless !condicion
  # if !condicion         ==  unless condicion
  # unless condicion      ==  if !condicion
  # unless !condicion     ==  if condicion

  # En la práctica:
  # - Usa "unless condicion" cuando quieres enfatizar el caso negativo.
  # - Usa "if condicion" cuando quieres enfatizar el caso positivo.

  # --- EJEMPLOS COMPARATIVOS ---

  # Escenario: verificar si un usuario puede acceder

  # Versión con if:
  if usuario.edad >= 18 && usuario.membresia_activa
    puts "Acceso permitido"
  else
    puts "Acceso denegado"
  end

  # Versión con unless (si la denegación es el foco):
  unless usuario.edad >= 18 && usuario.membresia_activa
    puts "Acceso denegado"
  else
    puts "Acceso permitido"
  end

  # ¿Cuál prefieres? La mayoría prefiere la versión con if,
  # porque la condición es compuesta y positiva.

  # Escenario: verificar si un archivo no existe

  # Versión con if:
  if !File.exist?("config.yml")
    puts "Archivo de configuración no encontrado"
  end

  # Versión con unless:
  unless File.exist?("config.yml")
    puts "Archivo de configuración no encontrado"
  end

  # unless es claramente más legible aquí. Suena natural.

  ------------------------------------------------------------------
  Explicación:
  ------------------------------------------------------------------
  - La elección entre if y unless es estilística y contextual.
  - No hay una regla universal; la legibilidad es el criterio.
  - En la comunidad Ruby, unless es bien aceptado para condiciones
    simples y negativas.
  - El código debe leerse como una oración en lenguaje natural.

  12.11  ERRORES COMUNES CON UNLESS
  --------------------------------------------------

  ------------------------------------------------------------------
  Código:
  ------------------------------------------------------------------

  # Error 1: Usar elsif con unless (no existe)
  # ❌ Esto causa SyntaxError:
  unless condicion
    # ...
  elsif otra_condicion
    # ...
  end

  # ✅ Solución: usa if/elsif/else en su lugar.

  # Error 2: Doble negación (unless + negación)
  # ❌ Confuso:
  unless !archivo.exist?
    puts "El archivo existe"
  end

  # ✅ Mejor:
  if archivo.exist?
    puts "El archivo existe"
  end

  # Error 3: Asignación en condición de unless
  # ❌ Peligroso:
  unless x = obtener_valor
    puts "No hay valor"
  end

  # Esto asigna y luego evalúa. Confuso.
  # ✅ Mejor:
  x = obtener_valor
  unless x
    puts "No hay valor"
  end

  # Error 4: unless con condición que siempre es true/false
  # ❌ Siempre se ejecuta ([] siempre es true en Ruby):
  unless []
    puts "Esto nunca se ejecuta? Sí se ejecuta!"
  end

  # [] es true, así que unless no se ejecuta... ¡espera!
  # unless [] significa: ejecuta bloque si [] es false.
  # [] NO es false, así que el bloque NO se ejecuta.
  # Pero si esperabas que un array vacío fuera false... error.

  # ✅ Recuerda: solo nil y false son falsy en Ruby.

  # Error 5: unless con next en lugar de if
  items = [1, 2, 3, 4, 5]

  # ❌ ¿Qué hace esto?
  items.each do |item|
    next unless item.even?  # Salta si NO es par => solo procesa pares
    puts item
  end
  # => 2, 4

  # ✅ Si querías filtrar pares:
  items.each do |item|
    puts item if item.even?
  end

  # Error 6: Olvidar que unless devuelve nil cuando la condición es true
  resultado = unless true
                "esto no se ejecuta"
              end
  puts resultado.inspect  # => nil

  ------------------------------------------------------------------
  Explicación:
  ------------------------------------------------------------------
  - Los errores con unless son comunes en principiantes.
  - El más grave es intentar usar elsif (no existe).
  - La doble negación (unless !cond) es confusa y debe evitarse.
  - Recuerda que unless ejecuta el bloque cuando la condición ES falsa.

  RESUMEN DEL CAPÍTULO
  ====================

  1. unless ejecuta código cuando la condición es false o nil.
  2. unless = if !condicion (equivalente lógico).
  3. unless/else existe pero es controvertido; prefiere if/else.
  4. El modificador unless (acción unless condición) es muy usado.
  5. unless es ideal para guard clauses y validaciones.
  6. No uses unless con condiciones compuestas complejas.
  7. No uses elsif con unless (no existe).
  8. unless con doble negación (unless !x) es confuso; evítalo.
  9. Extrae condiciones complejas a métodos para usar unless.
  10. En la comunidad Ruby, unless es bien aceptado y usado.

  PREGUNTAS FRECUENTES (FAQ)
  ==========================

  P: ¿Puedo usar elsif con unless?
  R: No. unless solo acepta else opcional. Para múltiples condiciones
     usa if/elsif/else.

  P: ¿Cuándo debo preferir unless sobre if?
  R: Cuando la condición es una negación simple y la frase "a menos que"
     suena natural. Ej: "A menos que el usuario esté autenticado,
     redirigir al login" => unless usuario.autenticado?

  P: ¿Es malo usar unless con else?
  R: La mayoría de los rubyistas considera que unless/else es confuso
     porque mezcla lógica negativa (unless) con un caso positivo (else).
     Prefiere if/else en su lugar.

  P: ¿Puedo usar el modificador unless en una línea?
  R: Sí: acción unless condición. Es muy común en guard clauses.

  P: ¿Cómo funciona unless con el operador ternario?
  R: No hay un "ternario inverso". Usa unless o if según el contexto.

  P: ¿Puedo tener un unless sin end?
  R: Solo si usas el modificador (acción unless condición). La forma
     bloque siempre necesita end.

  P: ¿Qué dice RuboCop sobre unless?
  R: RuboCop recomienda usar unless para condiciones simples, y prohibe
     unless con else y unless con condiciones compuestas complejas.

  P: ¿unless es más lento que if?
  R: No, son equivalentes en rendimiento. La elección es estilística.

  P: ¿Puedo anidar unless dentro de if?
  R: Sí, puedes mezclarlos. Pero cuidado con la legibilidad.

  P: ¿Existe until como el opuesto de while?
  R: Sí, lo veremos en el Capítulo 14. until es el opuesto de while,
     así como unless es el opuesto de if.

  Constructo: "SISTEMA DE VALIDACIÓN DE FORMULARIOS"

  Diseña e implementa un sistema de validación de formularios que use
  unless como su principal mecanismo de validación.

  Requisitos:
  1. Crea una clase `Formulario` que acepte un hash de campos con sus
     valores y reglas de validación.
  2. Cada campo puede tener estas reglas:
     - :requerido (booleano) — el campo no debe estar vacío
     - :min_longitud (entero) — longitud mínima
     - :max_longitud (entero) — longitud máxima
     - :formato (regex) — patrón que debe cumplir
     - :tipo (String/Integer/etc) — tipo de dato esperado
  3. El método `validar` debe recorrer todos los campos y, para cada
     regla, usar un guard clause con unless si la validación falla.
  4. Acumula todos los errores en un array y devuélvelos.
  5. Si no hay errores, devuelve true.

  Ejemplo de uso:
    formulario = Formulario.new({
      nombre: { valor: "Ana", reglas: { requerido: true, min_longitud: 3 } },
      email:  { valor: "invalido", reglas: { requerido: true, formato: /\A[^@]+@[^@]+\z/ } },
      edad:   { valor: 25, reglas: { tipo: Integer, min_longitud: 1 } }
    })
    errores = formulario.validar
    # => ["email: formato inválido"]

  Requisitos técnicos:
  - Cada validación debe ser un guard clause con unless.
  - No uses if para las validaciones (solo unless).
  - El método validar no debe tener más de 2 niveles de anidamiento.
  - Al menos una validación debe usar el modificador unless.

  Puntos extra:
  - Soporta reglas personalizadas (un Proc).
  - Soporta validación condicional (solo validar si otro campo tiene X valor).
  - Mensajes de error configurables por regla.
