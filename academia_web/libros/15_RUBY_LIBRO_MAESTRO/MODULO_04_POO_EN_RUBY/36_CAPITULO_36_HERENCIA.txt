   clases compartan comportamiento mientras mantienen sus diferencias."
   — Bertrand Meyer

================================================================================

  BIENVENIDA

  La herencia es uno de los pilares de la POO. Permite que una clase
  (subclase) herede atributos y métodos de otra clase (superclase),
  estableciendo una relación "es-un" (is-a). Un Perro es-un Animal,
  un Coche es-un Vehiculo.

  En Ruby, la herencia se declara con el operador < y cada clase solo
  puede tener una superclase directa (herencia simple). Sin embargo,
  la cadena de ancestros en Ruby es profunda: desde tu clase hasta
  BasicObject, pasando por Object y Kernel.

  En este capítulo exploraremos la herencia en profundidad: cómo
  funciona super, cómo navegar la cadena de ancestros, las clases
  fundamentales de Ruby, y las buenas prácticas para usar herencia
  efectivamente.

================================================================================

  FRASE DEL CAPÍTULO

  "La herencia es para especialización, no para reutilización de código.
   Si quieres reutilizar, prefiere composición. Si quieres especializar,
   la herencia es tu herramienta."
   — Sandi Metz

  1. SINTAXIS BÁSICA DE HERENCIA

  La herencia se declara con < después del nombre de la clase.

  ------------------------------------------------------------------
  Ejemplo 1: Herencia básica
  ------------------------------------------------------------------

  class Animal
    def respirar
      "Respirando..."
    end

    def mover
      "Moviéndose..."
    end

    def dormir
      "Durmiendo..."
    end
  end

  class Perro < Animal
    def ladrar
      "¡Guau!"
    end
  end

  class Gato < Animal
    def maullar
      "¡Miau!"
    end
  end

  fido = Perro.new
  puts fido.respirar  # => Respirando... (heredado)
  puts fido.mover     # => Moviéndose... (heredado)
  puts fido.dormir    # => Durmiendo... (heredado)
  puts fido.ladrar    # => ¡Guau! (propio)

  misifu = Gato.new
  puts misifu.respirar  # => Respirando... (heredado)
  puts misifu.maullar   # => ¡Miau! (propio)

================================================================================

  2. SOBREESCRIBIR MÉTODOS

  Una subclase puede redefinir (override) métodos heredados para
  proporcionar una implementación más específica.

  ------------------------------------------------------------------
  Ejemplo 2: Sobreescribir métodos
  ------------------------------------------------------------------

  class Animal
    def sonido
      "Hace un sonido genérico"
    end

    def moverse
      "Se mueve de forma genérica"
    end
  end

  class Perro < Animal
    def sonido
      "¡Guau! ¡Guau!"
    end
  end

  class Pez < Animal
    def moverse
      "Nada en el agua"
    end
  end

  animal = Animal.new
  puts animal.sonido   # => Hace un sonido genérico

  perro = Perro.new
  puts perro.sonido    # => ¡Guau! ¡Guau!

  pez = Pez.new
  puts pez.sonido      # => Hace un sonido genérico (heredado)
  puts pez.moverse     # => Nada en el agua (sobrescrito)

  3. LA PALABRA RESERVADA SUPER

  super llama al método del mismo nombre en la superclase. Es útil
  cuando queremos extender el comportamiento heredado en lugar de
  reemplazarlo completamente.

  ------------------------------------------------------------------
  Ejemplo 3: super básico
  ------------------------------------------------------------------

  class Animal
    def describir
      "Soy un animal"
    end
  end

  class Perro < Animal
    def describir
      super + " y además soy un perro"
    end
  end

  perro = Perro.new
  puts perro.describir  # => Soy un animal y además soy un perro

  ------------------------------------------------------------------
  Ejemplo 4: super con argumentos
  ------------------------------------------------------------------

  class Vehiculo
    attr_reader :marca, :modelo

    def initialize(marca, modelo)
      @marca = marca
      @modelo = modelo
      puts "Vehiculo#initialize: #{marca} #{modelo}"
    end
  end

  class Coche < Vehiculo
    attr_reader :puertas

    def initialize(marca, modelo, puertas)
      super(marca, modelo)  # Llama a Vehiculo#initialize
      @puertas = puertas
      puts "Coche#initialize: #{puertas} puertas"
    end
  end

  coche = Coche.new("Toyota", "Corolla", 4)
  # => Vehiculo#initialize: Toyota Corolla
  # => Coche#initialize: 4 puertas

  ------------------------------------------------------------------
  Ejemplo 5: super sin paréntesis (hereda todos los argumentos)
  ------------------------------------------------------------------

  class Documento
    def initialize(titulo, autor, contenido = "")
      @titulo = titulo
      @autor = autor
      @contenido = contenido
    end
  end

  class Articulo < Documento
    def initialize(titulo, autor, contenido = "", revista = nil)
      super  # Pasa titulo, autor, contenido (los mismos argumentos recibidos)
      @revista = revista
    end

    def to_s
      "#{@titulo} por #{@autor} (#{@revista})"
    end
  end

  art = Articulo.new("Ruby Avanzado", "Ana García", "Contenido...", "Revista Tech")
  puts art  # => Ruby Avanzado por Ana García (Revista Tech)

  super sin argumentos pasa automáticamente todos los argumentos que el
  método actual recibió. super() pasa cero argumentos.

  4. SUPER Y EL CICLO DE VIDA DE INITIALIZE

  Cuando se hereda, initialize de la subclase debe llamar a super para
  asegurar que la superclase inicialice su parte del estado.

  ------------------------------------------------------------------
  Ejemplo 6: initialize con super obligatorio
  ------------------------------------------------------------------

  class Persona
    attr_reader :nombre, :email

    def initialize(nombre, email)
      @nombre = nombre
      @email = email
    end
  end

  class Empleado < Persona
    attr_reader :salario

    def initialize(nombre, email, salario)
      super(nombre, email)  # Obligatorio si Persona#initialize espera args
      @salario = salario
    end
  end

  emp = Empleado.new("Ana", "ana@empresa.com", 50000)
  puts emp.nombre   # => Ana
  puts emp.salario  # => 50000

  # Si olvidamos super:
  class EmpleadoIncorrecto < Persona
    def initialize(nombre, email, salario)
      # No llamamos a super
      @salario = salario
    end
  end

  emp2 = EmpleadoIncorrecto.new("Luis", "luis@empresa.com", 50000)
  puts emp2.nombre  # => nil (no se inicializó)

  5. CADENA DE ANCESTROS (ANCESTORS)

  Cada clase en Ruby tiene una cadena de ancestros que Ruby recorre
  cuando busca un método. Podemos inspeccionarla con ancestors.

  ------------------------------------------------------------------
  Ejemplo 7: ancestors
  ------------------------------------------------------------------

  class Animal; end
  class Mamifero < Animal; end
  class Perro < Mamifero; end

  puts Perro.ancestors.inspect
  # => [Perro, Mamifero, Animal, Object, Kernel, BasicObject]

  puts Mamifero.ancestors.inspect
  # => [Mamifero, Animal, Object, Kernel, BasicObject]

  puts Animal.ancestors.inspect
  # => [Animal, Object, Kernel, BasicObject]

  puts Object.ancestors.inspect
  # => [Object, Kernel, BasicObject]

  puts BasicObject.ancestors.inspect
  # => [BasicObject]

  La cadena muestra el orden de búsqueda. Ruby busca en Perro, luego
  en Mamifero, luego en Animal, luego en Object, luego en Kernel,
  y finalmente en BasicObject.

  ------------------------------------------------------------------
  Ejemplo 8: Búsqueda de métodos en la cadena
  ------------------------------------------------------------------

  class A
    def test; "A"; end
  end

  class B < A
    def test; "B"; end
  end

  class C < B; end

  class D < C
    def test; "D"; end
  end

  puts D.new.test  # => D (encuentra en D, no sigue buscando)
  puts C.new.test  # => B (encuentra en B)
  puts B.new.test  # => B (encuentra en B)
  puts A.new.test  # => A (encuentra en A)

  puts D.ancestors.inspect  # => [D, C, B, A, Object, Kernel, BasicObject]

  6. LA JERARQUÍA: BASICOBJECT, OBJECT, KERNEL

  Entender la jerarquía base de Ruby es fundamental.

  • BasicObject: la raíz de toda la jerarquía. Es una clase casi vacía
    con solo métodos esenciales como __send__, __id__, equal?, y new.
    Es útil para crear objetos extremadamente ligeros.

  • Object: la clase madre de la mayoría de objetos. Añade métodos como
    to_s, inspect, send, respond_to?, methods, class, etc.

  • Kernel: un módulo incluido por Object. Provee métodos como puts,
    print, require, loop, raise, etc. (los "métodos globales" de Ruby).

  ------------------------------------------------------------------
  Ejemplo 9: Herencia de Object y Kernel
  ------------------------------------------------------------------

  obj = Object.new
  puts obj.respond_to?(:to_s)     # => true (de Object)
  puts obj.respond_to?(:puts)     # => false (puts está en Kernel, pero es privado)
  puts obj.respond_to?(:puts, true) # => true (incluyendo privados)

  # Los métodos de Kernel como puts están disponibles globalmente
  # porque Object incluye Kernel, y main (el contexto global) es Object.

  puts "Esto funciona globalmente"  # Método de Kernel

  # BasicObject es minimalista:
  bo = BasicObject.new
  # bo.to_s  # => NoMethodError! BasicObject no tiene to_s

  7. IS-A Y LA PRUEBA DE TIPO

  Podemos verificar la relación de herencia con is_a? y kind_of?.

  ------------------------------------------------------------------
  Ejemplo 10: is_a? y kind_of?
  ------------------------------------------------------------------

  class Animal; end
  class Perro < Animal; end

  fido = Perro.new

  puts fido.is_a?(Perro)    # => true
  puts fido.is_a?(Animal)   # => true
  puts fido.is_a?(Object)   # => true
  puts fido.is_a?(String)   # => false

  puts fido.kind_of?(Perro)  # => true (sinónimo de is_a?)
  puts fido.instance_of?(Perro)  # => true (solo la clase exacta)
  puts fido.instance_of?(Animal) # => false (clase exacta, no superclase)

  puts Perro === fido       # => true (case equality, operador ===)
  puts Animal === fido      # => true

  8. HERENCIA Y VARIABLES DE INSTANCIA

  Las variables de instancia (@) pertenecen a cada objeto, no a la
  clase. Una subclase hereda los métodos que acceden a @var, pero
  @var no es heredada en sí misma.

  ------------------------------------------------------------------
  Ejemplo 11: Variables de instancia en herencia
  ------------------------------------------------------------------

  class Animal
    def initialize(nombre)
      @nombre = nombre
    end

    def nombre
      @nombre
    end
  end

  class Perro < Animal
    def initialize(nombre, raza)
      @nombre = nombre  # Esto es necesario, no se hereda el initialize
      @raza = raza
    end

    def raza
      @raza
    end
  end

  # Mejor práctica:
  class PerroMejor < Animal
    def initialize(nombre, raza)
      super(nombre)  # Delega a Animal#initialize
      @raza = raza
    end
  end

  fido = PerroMejor.new("Fido", "Labrador")
  puts fido.nombre  # => Fido
  puts fido.raza    # => Labrador

  9. HERENCIA Y MÉTODOS DE CLASE

  Los métodos de clase también se heredan.

  ------------------------------------------------------------------
  Ejemplo 12: Herencia de métodos de clase
  ------------------------------------------------------------------

  class Animal
    def self.categorias
      ["mamífero", "ave", "reptil", "anfibio", "pez"]
    end

    def self.clasificar(tipo)
      categorias.include?(tipo) ? "#{tipo} válido" : "#{tipo} desconocido"
    end
  end

  class Perro < Animal
    def self.categorias
      super + ["canino"]
    end
  end

  puts Animal.categorias.inspect   # => ["mamífero", "ave", ...]
  puts Perro.categorias.inspect    # => ["mamífero", "ave", ..., "canino"]
  puts Perro.clasificar("canino")  # => canino válido

  super también funciona en métodos de clase.

  10. HERENCIA Y CONSTANTES

  Las constantes se buscan en la cadena de herencia.

  ------------------------------------------------------------------
  Ejemplo 13: Constantes y herencia
  ------------------------------------------------------------------

  class Animal
    ESPERANZA_VIDA = 10
  end

  class Perro < Animal
    ESPERANZA_VIDA = 13
  end

  class Pez < Animal; end

  puts Animal::ESPERANZA_VIDA  # => 10
  puts Perro::ESPERANZA_VIDA   # => 13
  puts Pez::ESPERANZA_VIDA     # => 10 (hereda de Animal)

  # Búsqueda de constantes:
  puts Perro::ESPERANZA_VIDA   # => 13 (encuentra en Perro)
  puts Pez::ESPERANZA_VIDA     # => 10 (busca en Animal)

  11. HERENCIA Y MÓDULOS (INCLUDE)

  Los módulos incluidos se insertan en la cadena de ancestros entre la
  clase y su superclase. Esto permite compartir comportamiento entre
  clases no relacionadas por herencia.

  ------------------------------------------------------------------
  Ejemplo 14: Módulos en la cadena de ancestros
  ------------------------------------------------------------------

  module Nadador
    def nadar
      "Nadando..."
    end
  end

  module Volador
    def volar
      "Volando..."
    end
  end

  module Corredor
    def correr
      "Corriendo..."
    end
  end

  class Animal
    def respirar
      "Respirando"
    end
  end

  class Perro < Animal
    include Corredor
    include Nadador
  end

  class Pato < Animal
    include Nadador
    include Volador
  end

  puts Perro.ancestors.inspect
  # => [Perro, Nadador, Corredor, Animal, Object, Kernel, BasicObject]

  puts Pato.new.nadar  # => Nadando...
  puts Perro.new.nadar # => Nadando...

  12. COMPOSICIÓN VS HERENCIA

  La composición (tener-un) es a menudo preferible a la herencia
  (es-un). En composición, un objeto contiene otros objetos que
  realizan parte del trabajo.

  ------------------------------------------------------------------
  Ejemplo 15: Composición vs Herencia
  ------------------------------------------------------------------

  # HERENCIA
  class Reporte
    def generar
      "Reporte genérico"
    end
  end

  class ReportePDF < Reporte
    def generar
      "Reporte en PDF"
    end
  end

  class ReporteHTML < Reporte
    def generar
      "Reporte en HTML"
    end
  end

  # COMPOSICIÓN (mejor)
  class ReporteComposicion
    def initialize(formateador)
      @formateador = formateador
    end

    def generar
      @formateador.formatear("Contenido del reporte")
    end
  end

  class FormateadorPDF
    def formatear(contenido)
      "PDF: #{contenido}"
    end
  end

  class FormateadorHTML
    def formatear(contenido)
      "<html>#{contenido}</html>"
    end
  end

  r1 = ReporteComposicion.new(FormateadorPDF.new)
  puts r1.generar  # => PDF: Contenido del reporte

  r2 = ReporteComposicion.new(FormateadorHTML.new)
  puts r2.generar  # => <html>Contenido del reporte</html>

  # Composición es más flexible: podemos cambiar el formateador en
  # tiempo de ejecución sin cambiar la clase.

  13. PRINCIPIO DE SUSTITUCIÓN DE LISKOV (LSP)

  El LSP establece que los objetos de una subclase deben poder
  sustituir a objetos de la superclase sin alterar el comportamiento
  esperado.

  ------------------------------------------------------------------
  Ejemplo 16: Violación de LSP
  ------------------------------------------------------------------

  class Rectangulo
    attr_accessor :ancho, :alto

    def initialize(ancho, alto)
      @ancho = ancho
      @alto = alto
    end

    def area
      @ancho * @alto
    end
  end

  class Cuadrado < Rectangulo
    def ancho=(valor)
      @ancho = valor
      @alto = valor  # Violación de LSP!
    end

    def alto=(valor)
      @ancho = valor
      @alto = valor  # Violación!
    end
  end

  def calcular_area(rect)
    rect.alto = 5
    rect.ancho = 10
    puts "Área esperada: 50, área obtenida: #{rect.area}"
  end

  calcular_area(Rectangulo.new(1, 1))  # => 50
  calcular_area(Cuadrado.new(1, 1))    # => 100 (violación!)

  # La herencia Rectángulo-Cuadrado es un ejemplo clásico de violación
  # de LSP. Una solución mejor es usar composición o una clase abstracta.

  14. CLASES ABSTRACTAS EN RUBY

  Ruby no tiene clases abstractas como concepto del lenguaje, pero
  podemos simularlas haciendo que métodos lancen NotImplementedError.

  ------------------------------------------------------------------
  Ejemplo 17: Clase abstracta simulada
  ------------------------------------------------------------------

  class Forma
    def initialize(color = "transparente")
      @color = color
    end

    def area
      raise NotImplementedError, "#{self.class} debe implementar area"
    end

    def perimetro
      raise NotImplementedError, "#{self.class} debe implementar perimetro"
    end

    def descripcion
      "#{@color} #{self.class.name.downcase} (área: #{area})"
    end
  end

  class Circulo < Forma
    def initialize(radio, color = "transparente")
      super(color)
      @radio = radio
    end

    def area
      Math::PI * @radio ** 2
    end

    def perimetro
      2 * Math::PI * @radio
    end
  end

  class Rectangulo < Forma
    def initialize(ancho, alto, color = "transparente")
      super(color)
      @ancho = ancho
      @alto = alto
    end

    def area
      @ancho * @alto
    end

    def perimetro
      2 * (@ancho + @alto)
    end
  end

  c = Circulo.new(5, "rojo")
  puts c.descripcion  # => rojo circulo (área: 78.53981633974483)

  r = Rectangulo.new(3, 4, "azul")
  puts r.descripcion  # => azul rectangulo (área: 12)

  # f = Forma.new  # No lanza error (no hay protección), pero:
  # f.area  # => NotImplementedError

  15. EJEMPLO COMPLETO: SISTEMA DE EMPLEADOS

  ------------------------------------------------------------------
  Ejemplo 18: Jerarquía de empleados
  ------------------------------------------------------------------

  class Empleado
    attr_reader :nombre, :id

    @@contador = 0

    def initialize(nombre)
      @@contador += 1
      @id = @@contador
      @nombre = nombre
    end

    def salario_base
      raise NotImplementedError, "Subclase debe implementar salario_base"
    end

    def bono
      0
    end

    def salario_total
      salario_base + bono
    end

    def to_s
      "##{@id}: #{@nombre} (#{self.class}) - $#{salario_total}"
    end
  end

  class Ingeniero < Empleado
    def salario_base
      70000
    end

    def bono
      5000
    end
  end

  class Diseniador < Empleado
    def salario_base
      60000
    end
  end

  class Gerente < Empleado
    def initialize(nombre, empleados = [])
      super(nombre)
      @empleados = empleados
    end

    def salario_base
      90000
    end

    def bono
      @empleados.size * 1000
    end

    def agregar_empleado(emp)
      @empleados << emp
    end

    def equipo
      @empleados.dup
    end
  end

  ing = Ingeniero.new("Ana García")
  dis = Diseniador.new("Carlos López")
  ger = Gerente.new("María Martínez", [ing, dis])

  [ing, dis, ger].each { |e| puts e }
  # => #1: Ana García (Ingeniero) - $75000
  # => #2: Carlos López (Diseniador) - $60000
  # => #3: María Martínez (Gerente) - $92000

  16. TIPS Y BUENAS PRÁCTICAS

  1. La herencia debe modelar "es-un", no "tiene-un".
  2. Prefiere composición sobre herencia para reutilización de código.
  3. No crees jerarquías profundas (>3 niveles). Dificulta el debugging.
  4. Usa super en initialize de la subclase para inicializar la
     superclase correctamente.
  5. El Principio de Sustitución de Liskov: las subclases deben poder
     reemplazar a la superclase sin romper el programa.
  6. Los módulos (include) son una alternativa más flexible que la
     herencia para compartir comportamiento.
  7. No uses herencia solo para reutilizar código. Usa composición.
  8. NotImplementedError es una convención para métodos "abstractos".
  9. Mantén las superclases pequeñas y enfocadas.

  RESUMEN DEL CAPÍTULO

  • class Subclase < Superclase declara herencia en Ruby.
  • super llama al método correspondiente de la superclase.
  • La cadena de ancestros se recorre en orden de búsqueda.
  • ancestors muestra: [Clase, Módulos, Superclase, Object, Kernel,
    BasicObject].
  • Object es la superclase por defecto (si no se especifica herencia).
  • BasicObject es la raíz de toda la jerarquía.
  • Kernel es un módulo incluido por Object con métodos globales.
  • is_a? y kind_of? verifican relación de herencia.
  • La composición (tener-un) es preferible a la herencia para
    reutilización de código.
  • NotImplementedError permite simular clases abstractas.

  PREGUNTAS FRECUENTES (FAQ)

  P: ¿Puede una clase heredar de más de una clase?
  R: No. Ruby tiene herencia simple. Pero puedes incluir múltiples
     módulos (include) para compartir comportamiento de múltiples
     fuentes. Esto se llama herencia múltiple de interfaces.

  P: ¿Qué pasa si no uso super en initialize?
  R: La superclase no inicializará sus variables de instancia. Tus
     objetos pueden terminar en un estado inconsistente con atributos
     nil que deberían tener valores.

  P: ¿Cuál es la diferencia entre super y super()?
  R: super (sin paréntesis) pasa todos los argumentos del método actual
     al método de la superclase. super() no pasa ningún argumento.
     super(a, b) pasa exactamente a y b.

  P: ¿Qué es BasicObject y para qué sirve?
  R: Es la clase raíz de Ruby, casi vacía. Sirve para crear objetos
     extremadamente ligeros o para construir DSLs donde no queremos los
     métodos de Object (como métodos de Kernel). Proxy objects es un
     caso de uso común.

  P: ¿Por qué algunos métodos de Kernel (puts, raise) están disponibles
     globalmente?
  R: Porque el contexto global (main) es un Object, y Object incluye
     Kernel. Además, Kernel hace que puts sea privado, por lo que está
     disponible en cualquier contexto sin receptor explícito.

  P: ¿Herencia o composición?
  R: Regla general: usa herencia cuando hay una relación "es-un"
     genuina y la subclase es un subtipo comportamental de la superclase.
     Usa composición cuando quieras reutilizar código o combinar
     comportamientos.

  P: ¿Cómo sé si estoy violando LSP?
  R: Si una subclase modifica el comportamiento de la superclase de
     manera que rompe las expectativas, estás violando LSP. Señales:
     métodos que no hacen nada, que lanzan excepciones inesperadas,
     o que cambian el estado de manera impredecible.

  DESAFÍO DEL ARQUITECTO

  Construye una jerarquía completa para un sistema de medios digitales.

  1. Clase base Medio (abstracta):
     - attr_reader: titulo, autor, año, duracion
     - initialize(titulo, autor, año)
     - Método abstracto: reproducir (lanza NotImplementedError)
     - Método concreto: info (devuelve título, autor, año)
     - Variable de clase @@total_medios

  2. Subclase Audio < Medio:
     - Añade: genero, calidad_kbps
     - Implementa reproducir: "Reproduciendo audio: #{@titulo}"

  3. Subclase Video < Audio:
     - Añade: resolucion, relacion_aspecto
     - Reescribe reproducir: "Reproduciendo video: #{@titulo}"
     - Usa super para parte del audio si aplica

  4. Subclase Podcast < Audio:
     - Añade: episodio, temporada, invitados (array)
     - Método: agregar_invitado(nombre)

  5. Subclase Pelicula < Video:
     - Añade: director, reparto (array), presupuesto
     - Método: mostrar_reparto

  6. Módulo Recomendable con método recomendar(usuario):
     - Las clases que lo incluyan deben implementar puntuacion

  7. Clase BibliotecaMedios:
     - Contiene array de medios
     - Métodos: agregar, buscar_por_titulo, buscar_por_autor,
       filtrar_por_tipo(tipo), reproduccion_rapida (llama a
       reproducir de cada medio)
     - Método de clase: total_medios_en_sistema

  8. Demuestra LSP: un método que acepte Medios y use polimórficamente
     reproducir debe funcionar con Audio, Video, Podcast, Pelicula.

  9. Composición: BibliotecaMedios debe usar composición, no herencia.

  Este desafío integra: herencia, super, ancestros, módulos, clases
  abstractas, LSP, composición, y la jerarquía completa de Ruby.
