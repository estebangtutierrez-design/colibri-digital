  BIENVENIDA

  La encapsulación es quizás el pilar más importante de la POO. Es la
  práctica de ocultar los detalles internos de un objeto y exponer solo
  una interfaz controlada. Un objeto encapsulado correctamente no revela
  cómo hace lo que hace, solo qué puede hacer.

  En Ruby, la encapsulación se logra mediante el control de acceso a
  los métodos (public, protected, private) y mediante el diseño
  cuidadoso de la interfaz pública. A diferencia de lenguajes como
  Java o C++, Ruby no tiene variables privadas a nivel de sintaxis:
  las variables de instancia @var siempre son privadas a menos que
  se expongan con getters.

  En este capítulo exploraremos la filosofía y la práctica de la
  encapsulación en Ruby: cómo controlar el acceso, cómo diseñar
  interfaces mínimas, cómo proteger invariantes, y cómo el sistema
  de acceso de Ruby difiere de otros lenguajes.

================================================================================

  FRASE DEL CAPÍTULO

  "La encapsulación es el principio de que cada objeto debe tener una
   interfaz bien definida a través de la cual se comunica con otros
   objetos. Lo que ocurre dentro del objeto es asunto suyo."
   — Alan Kay, pionero de la POO

  1. FILOSOFÍA DE ENCAPSULACIÓN EN RUBY

  Ruby sigue la filosofía de "confiar en el programador". No impone
  restricciones absolutas. La encapsulación en Ruby es una convención
  y un mecanismo de protección, no una barrera infranqueable.

  A diferencia de Java (donde private es absoluto), en Ruby:
  - private es recomendación, no ley
  - send puede invocar cualquier método
  - instance_variable_get puede leer cualquier variable
  - Las variables de instancia nunca son públicas automáticamente

  ------------------------------------------------------------------
  Ejemplo 1: La encapsulación es por confianza
  ------------------------------------------------------------------

  class CajaFuerte
    def initialize(codigo)
      @codigo = codigo
      @abierta = false
    end

    def abrir(codigo_intentado)
      if @codigo == codigo_intentado
        @abierta = true
        "¡Caja fuerte abierta!"
      else
        "Código incorrecto"
      end
    end

    def cerrar
      @abierta = false
      "Caja fuerte cerrada"
    end

    private

    def codigo_secreto
      @codigo
    end
  end

  cf = CajaFuerte.new(1234)
  puts cf.abrir(1234)   # => ¡Caja fuerte abierta!

  # Ruby no impide el acceso, solo advierte:
  puts cf.send(:codigo_secreto)  # => 1234 (bypassea private!)
  puts cf.instance_variable_get(:@codigo)  # => 1234

  # La encapsulación en Ruby es una recomendación fuerte, no una ley.

  2. INTERFAZ PÚBLICA VS INTERNA

  Todo objeto tiene dos caras:
  - Interfaz pública: los métodos que otros objetos pueden usar.
  - Interfaz privada/interna: los detalles de implementación.

  ------------------------------------------------------------------
  Ejemplo 2: Diseñando la interfaz pública
  ------------------------------------------------------------------

  class Cafetera
    def initialize
      @agua = 0
      @cafe = 0
      @encendida = false
    end

    # INTERFAZ PÚBLICA
    def encender
      @encendida = true
      calentar_agua
      "Cafetera encendida"
    end

    def preparar_cafe(tipo = :normal)
      return "Encender la cafetera primero" unless @encendida
      return "Sin agua" unless @agua > 0
      return "Sin café" unless @cafe > 0

      usar_agua
      usar_cafe
      moler_granos
      if tipo == :espresso
        hacer_espresso
      else
        hacer_cafe_normal
      end
      "Café #{tipo} listo"
    end

    def agregar_agua(cantidad)
      @agua += cantidad
      "#{cantidad}ml de agua agregados"
    end

    def agregar_cafe(cantidad)
      @cafe += cantidad
      "#{cantidad}g de café agregados"
    end

    # INTERFAZ PRIVADA
    private

    def calentar_agua
      "Calentando agua..."
    end

    def usar_agua
      @agua -= 1
    end

    def usar_cafe
      @cafe -= 1
    end

    def moler_granos
      "Moliendo granos..."
    end

    def hacer_espresso
      "Haciendo espresso..."
    end

    def hacer_cafe_normal
      "Haciendo café normal..."
    end
  end

  cafetera = Cafetera.new
  cafetera.agregar_agua(500)
  cafetera.agregar_cafe(30)
  cafetera.encender
  puts cafetera.preparar_cafe(:espresso)

  # cafetera.moler_granos  # Error! private method

  3. POR QUÉ ENCAPSULAR: PROTEGER INVARIANTES

  Un invariante es una condición que debe cumplirse siempre para que
  el objeto esté en un estado válido. La encapsulación protege los
  invariantes al controlar cómo se modifican los datos internos.

  ------------------------------------------------------------------
  Ejemplo 3: Protegiendo invariantes
  ------------------------------------------------------------------

  class CuentaBancaria
    def initialize(titular, saldo_inicial = 0)
      @titular = titular
      @saldo = saldo_inicial
      @congelada = false
    end

    # Invariante: @saldo nunca debe ser negativo
    # Invariante: si @congelada, no se permiten transacciones

    def depositar(monto)
      return "Cuenta congelada" if @congelada
      raise ArgumentError, "Monto debe ser positivo" unless monto > 0

      @saldo += monto
      registrar_transaccion(:deposito, monto)
      "Depósito exitoso. Saldo: $#{@saldo}"
    end

    def retirar(monto)
      return "Cuenta congelada" if @congelada
      raise ArgumentError, "Monto debe ser positivo" unless monto > 0
      return "Saldo insuficiente" if monto > @saldo

      @saldo -= monto
      registrar_transaccion(:retiro, monto)
      "Retiro exitoso. Saldo: $#{@saldo}"
    end

    def congelar
      @congelada = true
      "Cuenta congelada"
    end

    def descongelar
      @congelada = false
      "Cuenta descongelada"
    end

    def saldo
      @saldo
    end

    private

    def registrar_transaccion(tipo, monto)
      # Lógica de registro (no expuesta)
      puts "[#{Time.now.strftime('%H:%M:%S')}] #{tipo}: $#{monto}"
    end
  end

  cuenta = CuentaBancaria.new("Ana", 1000)
  cuenta.depositar(500)
  cuenta.retirar(200)
  # cuenta.retirar(2000)  # => Saldo insuficiente
  puts cuenta.saldo  # => 1300

  # El invariante de saldo no negativo se mantiene porque el
  # acceso a @saldo está controlado.

  4. PRIVATE EN PROFUNDIDAD

  Los métodos privados en Ruby solo pueden llamarse sin receptor
  explícito (self implícito). Ni siquiera self.metodo_privado es
  válido, solo metodo_privado.

  ------------------------------------------------------------------
  Ejemplo 4: Reglas de private
  ------------------------------------------------------------------

  class Padre
    def test_publico
      "público"
    end

    def llama_privado
      metodo_privado  # OK: self implícito
    end

    def llama_privado_con_self
      self.metodo_privado  # Error! private method called with explicit self
    end

    private

    def metodo_privado
      "privado"
    end
  end

  class Hijo < Padre
    def test_hijo
      metodo_privado  # OK: self implícito (hereda el método)
    end

    def test_hijo_con_self
      self.metodo_privado  # Error! (Ruby >= 2.7 da warning, antes era error)
    end
  end

  p = Padre.new
  # p.metodo_privado  # Error!
  puts p.llama_privado  # => privado

  h = Hijo.new
  puts h.test_hijo  # => privado

  # En Ruby, los métodos privados SÍ se heredan (a diferencia de
  # Java). Pero deben llamarse sin self.

  ------------------------------------------------------------------
  Ejemplo 5: Private no es herencia de visibilidad
  ------------------------------------------------------------------

  class Base
    private

    def saludar
      "Hola desde Base"
    end
  end

  class Derivada < Base
    # Esto NO hereda la visibilidad private
    def saludar
      "Hola desde Derivada"
    end
  end

  # La redefinición en Derivada es pública por defecto
  d = Derivada.new
  puts d.saludar  # => "Hola desde Derivada" (es público!)

  # Si queremos que siga siendo privado:
  class DerivadaCorrecta < Base
    private

    def saludar
      "Hola desde Derivada (privada)"
    end
  end

  5. PROTECTED EN PROFUNDIDAD

  Protected permite llamar al método con receptor explícito, siempre
  que el receptor sea de la misma clase o una subclase. Es útil para
  comparar objetos de la misma clase.

  ------------------------------------------------------------------
  Ejemplo 6: Protected para comparaciones
  ------------------------------------------------------------------

  class Persona
    attr_reader :nombre

    def initialize(nombre, edad)
      @nombre = nombre
      @edad = edad
    end

    def mayor_que?(otra)
      # Necesitamos acceder a @edad de otra persona
      self.edad > otra.edad  # protected permite esto
    end

    def menor_que?(otra)
      self.edad < otra.edad
    end

    def misma_edad?(otra)
      self.edad == otra.edad
    end

    protected

    def edad
      @edad
    end
  end

  ana = Persona.new("Ana", 30)
  luis = Persona.new("Luis", 25)
  carla = Persona.new("Carla", 30)

  puts ana.mayor_que?(luis)    # => true
  puts luis.mayor_que?(ana)    # => false
  puts ana.misma_edad?(carla)  # => true

  # puts ana.edad  # Error! protected method

  # protected permite que una instancia acceda al método de otra
  # instancia de la misma clase. private NO lo permite.

  ------------------------------------------------------------------
  Ejemplo 7: Protected entre clases relacionadas
  ------------------------------------------------------------------

  class Empleado
    def initialize(nombre, salario)
      @nombre = nombre
      @salario = salario
    end

    def compare_salary(other)
      if other.is_a?(Empleado)
        self.salario <=> other.salario
      else
        raise "No se puede comparar"
      end
    end

    protected

    def salario
      @salario
    end
  end

  class Gerente < Empleado
    def salario_mayor_que?(empleado)
      self.salario > empleado.salario  # OK: ambos son Empleado/Gerente
    end
  end

  emp = Empleado.new("Ana", 50000)
  ger = Gerente.new("Luis", 80000)

  puts emp.compare_salary(ger)  # => -1 (50000 < 80000)
  puts ger.salario_mayor_que?(emp)  # => true

  6. ACCESS CONTROL DECLARATIVO CON SÍMBOLOS

  Además de usar private/protected como modificadores de ámbito,
  podemos declarar la visibilidad pasando símbolos.

  ------------------------------------------------------------------
  Ejemplo 8: Declaración con símbolos
  ------------------------------------------------------------------

  class Servicio
    def iniciar
      configurar
      conectar
      "Servicio iniciado"
    end

    def detener
      desconectar
      limpiar
      "Servicio detenido"
    end

    def estado
      @conectado ? "conectado" : "desconectado"
    end

    public :iniciar, :detener, :estado

    private

    def configurar
      @conectado = false
    end

    def conectar
      @conectado = true
    end

    def desconectar
      @conectado = false
    end

    def limpiar
      # cleanup
    end
  end

  s = Servicio.new
  puts s.iniciar  # OK
  puts s.estado   # OK
  # s.configurar  # Error! private

  7. LA LEY DE DEMETER (PRINCIPIO DE MÍNIMO CONOCIMIENTO)

  La Ley de Demeter dice: "No hables con extraños. Solo habla con tus
  amigos inmediatos." En la práctica: no encadenes llamadas como
  objeto.a.b.c.

  ------------------------------------------------------------------
  Ejemplo 9: Violación de la Ley de Demeter
  ------------------------------------------------------------------

  class Cliente
    attr_reader :direccion

    def initialize(nombre, direccion)
      @nombre = nombre
      @direccion = direccion
    end
  end

  class Direccion
    attr_reader :ciudad, :pais

    def initialize(calle, ciudad, pais)
      @calle = calle
      @ciudad = ciudad
      @pais = pais
    end
  end

  # Violación: el cliente conoce a Direccion y a Ciudad y a Pais
  cliente = Cliente.new("Ana", Direccion.new("Calle 123", "Madrid", "España"))
  puts cliente.direccion.ciudad  # Demeter violation!

  # Mejor: encapsular el conocimiento
  class ClienteMejor
    def initialize(nombre, direccion)
      @nombre = nombre
      @direccion = direccion
    end

    def ciudad
      @direccion.ciudad
    end

    def pais
      @direccion.pais
    end
  end

  cliente2 = ClienteMejor.new("Luis", Direccion.new("Av 456", "Lima", "Perú"))
  puts cliente2.ciudad  # OK: no viola Demeter

  ------------------------------------------------------------------
  Ejemplo 10: Demeter en la práctica
  ------------------------------------------------------------------

  # MAL: Tren de llamadas
  # order.customer.address.city

  # BIEN: Encapsular
  class Order
    def customer_city
      customer.address.city
    end

    private

    def customer
      @customer
    end
  end

  # order.customer_city  # No viola Demeter

  8. INMUTABILIDAD COMO ENCAPSULACIÓN

  Hacer objetos inmutables (con freeze) es una forma poderosa de
  encapsulación. Un objeto inmutable no puede modificarse después
  de su creación.

  ------------------------------------------------------------------
  Ejemplo 11: Objetos inmutables
  ------------------------------------------------------------------

  class PuntoInmutable
    attr_reader :x, :y

    def initialize(x, y)
      @x = x
      @y = y
      freeze
    end

    def mover(dx, dy)
      PuntoInmutable.new(@x + dx, @y + dy)
    end

    def to_s
      "(#{@x}, #{@y})"
    end
  end

  p1 = PuntoInmutable.new(3, 4)
  puts p1.x  # => 3

  # p1.x = 5  # Error! (no hay setter)
  # p1.instance_variable_set(:@x, 5)  # Error! frozen object

  p2 = p1.mover(1, 2)
  puts p1  # => (3, 4) (sin cambios)
  puts p2  # => (4, 6) (nuevo objeto)

  Los objetos inmutables son inherentemente thread-safe y más fáciles
  de razonar.

  9. BYPASS DE ENCAPSULACIÓN: SEND Y REFLECTION

  Ruby proporciona mecanismos para bypassear la encapsulación. Son
  herramientas poderosas que deben usarse con cuidado.

  ------------------------------------------------------------------
  Ejemplo 12: Métodos de bypass
  ------------------------------------------------------------------

  class ObjetoPrivado
    def initialize
      @secreto = "información sensible"
    end

    private

    def metodo_privado
      "método secreto"
    end
  end

  obj = ObjetoPrivado.new

  # Bypass #1: send (llama a cualquier método)
  puts obj.send(:metodo_privado)  # => "método secreto"

  # Bypass #2: instance_variable_get
  puts obj.instance_variable_get(:@secreto)  # => "información sensible"

  # Bypass #3: instance_variable_set
  obj.instance_variable_set(:@secreto, "modificado")
  puts obj.instance_variable_get(:@secreto)  # => "modificado"

  # Uso responsable: testing, metaprogramación, debugging
  # Uso irresponsable: romper encapsulación en producción

  ------------------------------------------------------------------
  Ejemplo 13: public_send para acceso controlado
  ------------------------------------------------------------------

  class API
    def publico
      "público"
    end

    private

    def interno
      "interno"
    end
  end

  api = API.new
  puts api.public_send(:publico)    # => "público"
  # puts api.public_send(:interno)  # => Error! NoMethodError (respeta private)

  # public_send solo llama a métodos públicos.
  # send llama a cualquier método (público, protegido, privado).

  10. EJEMPLO COMPLETO: SISTEMA BANCARIO

  ------------------------------------------------------------------
  Ejemplo 14: Sistema bancario con encapsulación
  ------------------------------------------------------------------

  class Banco
    def initialize(nombre)
      @nombre = nombre
      @cuentas = []
    end

    def crear_cuenta(titular, saldo_inicial = 0)
      cuenta = Cuenta.new(titular, saldo_inicial)
      @cuentas << cuenta
      cuenta
    end

    def buscar_cuenta(numero)
      @cuentas.find { |c| c.numero == numero }
    end

    def total_depositos
      @cuentas.sum(&:saldo)
    end

    def reporte
      puts "=== #{@nombre} ==="
      @cuentas.each { |c| puts "  #{c}" }
      puts "Total: $#{total_depositos}"
    end
  end

  class Cuenta
    attr_reader :numero, :titular

    @@contador = 0

    def initialize(titular, saldo_inicial = 0)
      @@contador += 1
      @numero = "CU-#{@@contador.to_s.rjust(5, '0')}"
      @titular = titular
      @saldo = saldo_inicial
      @movimientos = []
      @activa = true
      registrar_movimiento(:apertura, saldo_inicial)
    end

    def depositar(monto)
      return "Cuenta inactiva" unless @activa
      validar_monto_positivo(monto)

      @saldo += monto
      registrar_movimiento(:deposito, monto)
      "Depósito exitoso. Saldo: $#{@saldo}"
    end

    def retirar(monto)
      return "Cuenta inactiva" unless @activa
      validar_monto_positivo(monto)
      return "Saldo insuficiente" if monto > @saldo

      @saldo -= monto
      registrar_movimiento(:retiro, monto)
      "Retiro exitoso. Saldo: $#{@saldo}"
    end

    def transferir(monto, destino)
      return "Cuenta inactiva" unless @activa
      return "Cuenta destino inactiva" unless destino.activa?

      resultado = retirar(monto)
      return resultado if resultado.start_with?("Saldo insuficiente")

      destino.depositar(monto)
      registrar_movimiento(:transferencia_envio, monto, destino.numero)
      "Transferencia de $#{monto} a #{destino.titular} exitosa"
    end

    def saldo
      @saldo
    end

    def activa?
      @activa
    end

    def congelar
      @activa = false
      "Cuenta congelada"
    end

    def descongelar
      @activa = true
      "Cuenta descongelada"
    end

    def estado_de_cuenta
      puts "=== Estado de Cuenta #{@numero} ==="
      puts "Titular: #{@titular}"
      puts "Saldo: $#{@saldo}"
      puts "Movimientos:"
      @movimientos.each { |m| puts "  #{m[:fecha]} #{m[:tipo]}: $#{m[:monto]}" }
    end

    def to_s
      "#{@numero}: #{@titular} — $#{@saldo} [#{@activa ? 'ACTIVA' : 'INACTIVA'}]"
    end

    protected

    def movimientos
      @movimientos.dup
    end

    private

    def validar_monto_positivo(monto)
      raise ArgumentError, "Monto debe ser positivo" unless monto.is_a?(Numeric) && monto > 0
    end

    def registrar_movimiento(tipo, monto, referencia = nil)
      @movimientos << {
        fecha: Time.now.strftime("%Y-%m-%d %H:%M:%S"),
        tipo: tipo,
        monto: monto,
        referencia: referencia
      }
    end
  end

  banco = Banco.new("Banco Master")
  a = banco.crear_cuenta("Ana Pérez", 5000)
  b = banco.crear_cuenta("Luis García", 3000)

  a.depositar(1500)
  a.transferir(2000, b)
  a.retirar(500)

  banco.reporte
  a.estado_de_cuenta

  11. TIPS Y BUENAS PRÁCTICAS

  1. Diseña la interfaz pública primero: ¿qué necesita el mundo exterior?
  2. Expon lo mínimo necesario: todo lo que no sea esencial, hazlo privado.
  3. Protege invariantes: validación en setters y métodos de mutación.
  4. No expongas referencias a objetos mutables internos (usa .dup/).
  5. Prefiere objetos inmutables (freeze) para datos de valor.
  6. La Ley de Demeter: no encadenes más de una llamada.
  7. Usa protected para comparaciones entre instancias de la misma clase.
  8. No abuses de send para bypassear encapsulación en producción.
  9. Recuerda: en Ruby, la encapsulación es por confianza, no por ley.

================================================================================

  RESUMEN DEL CAPÍTULO

  • La encapsulación oculta los detalles de implementación y expone
    solo la interfaz necesaria.
  • public: accesible desde cualquier parte.
  • protected: accesible desde la clase y subclases, con receptor.
  • private: accesible solo sin self explícito.
  • Los métodos privados se heredan (diferente de Java).
  • La Ley de Demeter: no hables con extraños.
  • send y reflection permiten bypassear encapsulación.
  • public_send respeta la visibilidad de los métodos.
  • Los objetos inmutables (freeze) son la encapsulación máxima.
  • En Ruby, la encapsulación es por confianza y convención.

  PREGUNTAS FRECUENTES (FAQ)

  P: ¿Ruby tiene variables privadas como Java?
  R: No. Las variables de instancia (@var) siempre son privadas al
     objeto. No hay una keyword para hacerlas públicas. Para exponerlas,
     necesitas attr_reader, attr_writer o métodos getter/setter.

  P: ¿Por qué send puede llamar a métodos privados?
  R: Porque Ruby confía en el programador. send es una herramienta de
     metaprogramación que está disponible para casos donde necesitas
     bypassear la encapsulación (testing, debugging, serialización).
     Úsalo con responsabilidad.

  P: ¿Private se hereda en Ruby?
  R: El método se hereda (está disponible en la subclase), pero la
     subclase puede redefinirlo con diferente visibilidad. Si la
     subclase reescribe el método sin declararlo private, será público.

  P: ¿Qué es mejor: private o protected?
  R: Usa private para métodos de uso interno (helpers). Usa protected
     cuando necesites que instancias de la misma clase accedan al
     método de otra instancia (comparaciones).

  P: ¿Debo siempre encapsular todo?
  R: Sí, pero con medida. Todo lo que no sea parte de la interfaz
     pública debe ser privado. Pero no te obsesiones: a veces un
     attr_accessor simple es suficiente. La clave es preguntarte:
     "¿realmente necesito exponer esto?"

  P: ¿Qué es la "fuga de referencias"?
  R: Ocurre cuando devuelves una referencia a un objeto mutable interno.
     El exterior puede modificar ese objeto y romper tus invariantes.
     Solución: devuelve una copia (.dup, .clone) o congela el objeto.

  P: ¿Cómo aplico la Ley de Demeter en Rails?
  R: Usa delegados: delegate :city, to: :address, prefix: false.
     O crea métodos de fachada en el modelo principal.

  DESAFÍO DEL ARQUITECTO

  Construye un sistema completo de gestión de proyectos con rigurosa
  encapsulación.

  1. Clase Proyecto:
     - @nombre, @descripcion, @tareas (array), @fecha_creacion
     - Interfaz pública: agregar_tarea, completar_tarea, estado,
       progreso, to_s
     - @tareas solo accesible como copia congelada (tareas)
     - Método privado: validar_nombre, generar_reporte_interno
     - Protected: fecha_creacion (para comparación entre proyectos)

  2. Clase Tarea:
     - @titulo, @descripcion, @responsable, @estado, @fecha_limite
     - Interfaz pública: iniciar, completar, cancelar, reasignar(usr)
     - Getter responsable devuelve copia del objeto Usuario
     - Setter estado valida transiciones permitidas
     - Método privado: validar_transicion, notificar_cambio
     - Protected: prioridad (para comparación entre tareas)

  3. Clase Usuario:
     - @nombre, @email, @rol, @tareas_asignadas
     - attr_reader: nombre, email, rol
     - tareas_asignadas devuelve array congelado
     - Método privado: validar_email, puede_asignar_tarea?
     - Protected: nivel_acceso

  4. Clase Informe:
     - Recibe un proyecto y genera reportes
     - Métodos: progreso, tareas_por_estado, tareas_por_usuario,
       tiempo_estimado_total, exportar(formateador)
     - Usa duck typing para formateador (PDF, HTML, TXT)
     - Todo es público (es la interfaz de reporting)

  5. Invariantes a proteger:
     - Una tarea no puede estar en estado :completada y :cancelada
     - Un proyecto no puede tener más de 100 tareas
     - El email del usuario debe tener formato válido
     - Las transiciones de estado de tarea deben ser válidas

  6. Ley de Demeter:
     - project.task_status  (NO project.tareas.first.estado)
     - user.task_count      (NO user.tareas.size)

  7. Inmutabilidad:
     - Los objetos devueltos por getters deben ser copias o congelados
     - Las fechas se devuelven como strings, no objetos mutables

  Este desafío integra: encapsulación, private, protected, invariantes,
  Ley de Demeter, inmutabilidad, duck typing, y diseño de interfaces.

  CIERRE DEL CAPÍTULO

  Has explorado la encapsulación en Ruby, desde sus fundamentos
  filosóficos hasta su aplicación práctica. Ahora entiendes que la
  encapsulación no es solo una cuestión técnica, sino de diseño: se
  trata de definir qué es público y qué es privado, qué se expone y
  qué se oculta.

  La encapsulación bien aplicada hace que tu código sea más robusto
  (los invariantes se protegen), más mantenible (puedes cambiar
  internamente sin afectar a otros) y más comprensible (la interfaz
  pública es clara y mínima).

  Recuerda: en Ruby, la encapsulación es por confianza, pero también
  es una disciplina. Diseña tus objetos para que sean fáciles de usar
  y difíciles de usar incorrectamente.

  En el próximo capítulo exploraremos self: la palabra reservada que
  referencia al objeto actual. Veremos cómo self cambia según el
  contexto y cómo dominarlo es esencial para escribir código Ruby
  correcto y elegante.
