  BIENVENIDA

  El polimorfismo es el tercer pilar de la POO (tras encapsulación y
  herencia). Literalmente significa "muchas formas". En la práctica,
  es la capacidad de objetos de diferentes tipos de responder al mismo
  mensaje de manera apropiada para cada uno.

  Ruby lleva el polimorfismo al extremo con el duck typing: "Si camina
  como pato y suena como pato, entonces es un pato". No importa la
  clase del objeto, importa qué mensajes puede recibir.

  En este capítulo exploraremos las múltiples formas de polimorfismo
  en Ruby: duck typing, respond_to?, method_missing, herencia, y
  módulos. Aprenderás a diseñar sistemas flexibles y extensibles.

  1. ¿QUÉ ES POLIMORFISMO?

  Polimorfismo significa que objetos diferentes pueden responder al
  mismo mensaje de maneras diferentes. El llamador no necesita saber
  qué tipo de objeto es; solo necesita saber que responde al mensaje.

  ------------------------------------------------------------------
  Ejemplo 1: Polimorfismo básico
  ------------------------------------------------------------------

  class Perro
    def hacer_sonido
      "¡Guau!"
    end
  end

  class Gato
    def hacer_sonido
      "¡Miau!"
    end
  end

  class Vaca
    def hacer_sonido
      "¡Muuu!"
    end
  end

  # Polimorfismo: el mismo mensaje, diferentes respuestas
  animales = [Perro.new, Gato.new, Vaca.new]
  animales.each { |a| puts a.hacer_sonido }
  # => ¡Guau!
  # => ¡Miau!
  # => ¡Muuu!

  2. DUCK TYPING

  Duck typing es la forma de polimorfismo más pura de Ruby. No requiere
  herencia ni interfaces. Solo requiere que el objeto tenga el método
  que se le pide.

  ------------------------------------------------------------------
  Ejemplo 2: Duck typing
  ------------------------------------------------------------------

  class Pato
    def nadar
      "El pato nada en el estanque"
    end

    def sonido
      "¡Cuac!"
    end
  end

  class Bote
    def nadar
      "El bote navega por el río"
    end

    def sonido
      "¡Brum brum!"
    end
  end

  class Persona
    def nadar
      "La persona nada en la piscina"
    end
  end

  def hacer_nadar(obj)
    puts obj.nadar
  end

  # Todos responden a nadar, todos "son nadadores"
  hacer_nadar(Pato.new)    # => El pato nada en el estanque
  hacer_nadar(Bote.new)    # => El bote navega por el río
  hacer_nadar(Persona.new) # => La persona nada en la piscina

  # No importa la clase. Importa que responda a :nadar.

  ------------------------------------------------------------------
  Ejemplo 3: Ejemplo práctico de duck typing
  ------------------------------------------------------------------

  class ReportePDF
    def generar(datos)
      "PDF generado con #{datos.size} datos"
    end

    def exportar(archivo)
      "PDF exportado a #{archivo}.pdf"
    end
  end

  class ReporteHTML
    def generar(datos)
      "<html>#{datos.join(', ')}</html>"
    end

    def exportar(archivo)
      "HTML exportado a #{archivo}.html"
    end
  end

  class ReporteCSV
    def generar(datos)
      datos.join(",")
    end

    def exportar(archivo)
      "CSV exportado a #{archivo}.csv"
    end
  end

  def procesar_reporte(reporte, datos, archivo)
    puts reporte.generar(datos)
    puts reporte.exportar(archivo)
  end

  datos = ["Ruby", "POO", "Polimorfismo"]

  procesar_reporte(ReportePDF.new, datos, "reporte")
  procesar_reporte(ReporteHTML.new, datos, "reporte")
  procesar_reporte(ReporteCSV.new, datos, "reporte")

  # Un nuevo formato solo necesita implementar los mismos métodos.

  3. RESPOND_TO?

  Podemos preguntar a un objeto si responde a un mensaje antes de
  enviarlo, usando respond_to?.

  ------------------------------------------------------------------
  Ejemplo 4: respond_to? para verificar capacidades
  ------------------------------------------------------------------

  def procesar_si_puede(obj)
    if obj.respond_to?(:volar)
      puts obj.volar
    elsif obj.respond_to?(:nadar)
      puts obj.nadar
    elsif obj.respond_to?(:correr)
      puts obj.correr
    else
      puts "No sé qué hacer con #{obj.class}"
    end
  end

  class Ave
    def volar; "Volando alto"; end
  end

  class Pez
    def nadar; "Nadando profundo"; end
  end

  class Guepardo
    def correr; "Corriendo rápido"; end
  end

  procesar_si_puede(Ave.new)      # => Volando alto
  procesar_si_puede(Pez.new)      # => Nadando profundo
  procesar_si_puede(Guepardo.new) # => Corriendo rápido
  procesar_si_puede(Robot.new)    # => No sé qué hacer con Robot

  ------------------------------------------------------------------
  Ejemplo 5: respond_to? con polimorfismo condicional
  ------------------------------------------------------------------

  class GestorArchivos
    def guardar(datos, destino)
      if destino.respond_to?(:write)
        destino.write(datos)
        puts "Datos escritos en #{destino.class}"
      elsif destino.respond_to?(:path)
        File.write(destino.path, datos)
        puts "Datos escritos en #{destino.path}"
      elsif destino.is_a?(String)
        File.write(destino, datos)
        puts "Datos escritos en #{destino}"
      else
        raise "No sé cómo guardar en #{destino.class}"
      end
    end
  end

  gestor = GestorArchivos.new
  gestor.guardar("datos", File.open("/tmp/test.txt", "w"))
  gestor.guardar("datos", "/tmp/test2.txt")

  4. METHOD_MISSING: EL POLIMORFISMO MÁXIMO

  method_missing es un método que se llama cuando un objeto recibe un
  mensaje que no sabe manejar. Por defecto, lanza NoMethodError. Pero
  podemos sobreescribirlo para manejar mensajes dinámicamente.

  ------------------------------------------------------------------
  Ejemplo 6: method_missing básico
  ------------------------------------------------------------------

  class Proxy
    def initialize(objeto_real)
      @objeto_real = objeto_real
    end

    def method_missing(nombre, *args, &block)
      puts "Delegando #{nombre} a #{@objeto_real.class}"
      @objeto_real.send(nombre, *args, &block)
    end
  end

  class Calculadora
    def sumar(a, b)
      a + b
    end

    def restar(a, b)
      a - b
    end
  end

  proxy = Proxy.new(Calculadora.new)
  puts proxy.sumar(3, 4)    # => Delegando sumar a Calculadora / 7
  puts proxy.restar(10, 3)  # => Delegando restar a Calculadora / 7

  ------------------------------------------------------------------
  Ejemplo 7: method_missing para crear métodos dinámicos
  ------------------------------------------------------------------

  class AnimalBuilder
    def method_missing(nombre, *args, &block)
      if nombre.to_s.start_with?("crear_")
        tipo = nombre.to_s.sub("crear_", "")
        Class.new do
          define_method(:sonido) { args[0] || "..." }
          define_method(:tipo) { tipo }
          define_method(:to_s) { "#{tipo}: #{sonido}" }
        end.new
      else
        super
      end
    end

    def respond_to_missing?(nombre, include_private = false)
      nombre.to_s.start_with?("crear_") || super
    end
  end

  builder = AnimalBuilder.new
  perro = builder.crear_perro("Guau!")
  gato = builder.crear_gato("Miau!")

  puts perro  # => perro: Guau!
  puts gato   # => gato: Miau!
  puts builder.respond_to?(:crear_vaca)  # => true

  ------------------------------------------------------------------
  Ejemplo 8: method_missing para registros tipo Hash
  ------------------------------------------------------------------

  class RegistroFlexible
    def initialize
      @datos = {}
    end

    def method_missing(nombre, *args, &block)
      nombre_str = nombre.to_s

      if nombre_str.end_with?("=")
        # Setter dinámico: nombre = valor
        clave = nombre_str[0..-2].to_sym
        @datos[clave] = args.first
      elsif @datos.key?(nombre)
        # Getter dinámico
        @datos[nombre]
      else
        super
      end
    end

    def respond_to_missing?(nombre, include_private = false)
      nombre_str = nombre.to_s
      nombre_str.end_with?("=") || @datos.key?(nombre) || super
    end

    def to_s
      @datos.inspect
    end
  end

  r = RegistroFlexible.new
  r.nombre = "Ana"
  r.edad = 30
  r.email = "ana@ejemplo.com"

  puts r.nombre   # => Ana
  puts r.edad     # => 30
  puts r.email    # => ana@ejemplo.com
  puts r.respond_to?(:nombre)  # => true

  5. RESPOND_TO_MISSING?

  Siempre que sobreescribas method_missing, debes también sobreescribir
  respond_to_missing? para que respond_to? funcione correctamente.

  ------------------------------------------------------------------
  Ejemplo 9: Importancia de respond_to_missing?
  ------------------------------------------------------------------

  class SinRespondToMissing
    def method_missing(nombre, *args)
      if nombre.to_s.start_with?("saludar_")
        "Hola, #{nombre.to_s.sub('saludar_', '')}!"
      else
        super
      end
    end
  end

  class ConRespondToMissing
    def method_missing(nombre, *args)
      if nombre.to_s.start_with?("saludar_")
        "Hola, #{nombre.to_s.sub('saludar_', '')}!"
      else
        super
      end
    end

    def respond_to_missing?(nombre, include_private = false)
      nombre.to_s.start_with?("saludar_") || super
    end
  end

  s = SinRespondToMissing.new
  c = ConRespondToMissing.new

  puts s.saludar_mundo    # => Hola, mundo! (funciona)
  puts s.respond_to?(:saludar_mundo)  # => false! (incorrecto)

  puts c.saludar_mundo    # => Hola, mundo! (funciona)
  puts c.respond_to?(:saludar_mundo)  # => true! (correcto)

  6. POLIMORFISMO CON HERENCIA

  La herencia proporciona polimorfismo cuando las subclases sobreescriben
  métodos de la superclase.

  ------------------------------------------------------------------
  Ejemplo 10: Polimorfismo por herencia
  ------------------------------------------------------------------

  class Empleado
    def calcular_salario
      raise NotImplementedError
    end

    def descripcion
      "#{self.class}: $#{calcular_salario}"
    end
  end

  class TiempoCompleto < Empleado
    def initialize(salario_mensual)
      @salario_mensual = salario_mensual
    end

    def calcular_salario
      @salario_mensual
    end
  end

  class PorHoras < Empleado
    def initialize(horas, tarifa)
      @horas = horas
      @tarifa = tarifa
    end

    def calcular_salario
      @horas * @tarifa
    end
  end

  class Comision < Empleado
    def initialize(ventas, porcentaje)
      @ventas = ventas
      @porcentaje = porcentaje
    end

    def calcular_salario
      @ventas * @porcentaje / 100.0
    end
  end

  empleados = [
    TiempoCompleto.new(5000),
    PorHoras.new(160, 25),
    Comision.new(100000, 5)
  ]

  empleados.each { |e| puts e.descripcion }
  # => TiempoCompleto: $5000
  # => PorHoras: $4000
  # => Comision: $5000.0

  7. POLIMORFISMO CON MÓDULOS

  Los módulos permiten polimorfismo sin herencia.

  ------------------------------------------------------------------
  Ejemplo 11: Polimorfismo con módulos
  ------------------------------------------------------------------

  module Volador
    def volar
      "Volando a #{velocidad_vuelo}"
    end

    def velocidad_vuelo
      raise NotImplementedError
    end
  end

  module Terrestre
    def correr
      "Corriendo a #{velocidad_terrestre}"
    end

    def velocidad_terrestre
      raise NotImplementedError
    end
  end

  class Avion
    include Volador

    def velocidad_vuelo
      "900 km/h"
    end
  end

  class Ave
    include Volador
    include Terrestre

    def velocidad_vuelo
      "40 km/h"
    end

    def velocidad_terrestre
      "10 km/h"
    end
  end

  class Coche
    include Terrestre

    def velocidad_terrestre
      "200 km/h"
    end
  end

  [Avion.new, Ave.new, Coche.new].each do |obj|
    puts obj.volar if obj.respond_to?(:volar)
    puts obj.correr if obj.respond_to?(:correr)
  end

  8. PATRÓN STRATEGY

  El patrón Strategy usa polimorfismo para intercambiar algoritmos.

  ------------------------------------------------------------------
  Ejemplo 12: Strategy pattern
  ------------------------------------------------------------------

  class CalculadoraImpuestos
    attr_writer :estrategia

    def initialize(estrategia)
      @estrategia = estrategia
    end

    def calcular(monto)
      @estrategia.calcular(monto)
    end
  end

  class IVA
    def calcular(monto)
      monto * 0.16
    end
  end

  class ISR
    def calcular(monto)
      if monto > 10000
        monto * 0.30
      elsif monto > 5000
        monto * 0.20
      else
        monto * 0.10
      end
    end
  end

  class IEPS
    def calcular(monto)
      monto * 0.08
    end
  end

  calc = CalculadoraImpuestos.new(IVA.new)
  puts calc.calcular(1000)  # => 160.0

  calc.estrategia = ISR.new
  puts calc.calcular(1000)  # => 100.0

  calc.estrategia = IEPS.new
  puts calc.calcular(1000)  # => 80.0

  9. PATRÓN TEMPLATE METHOD

  Template Method define el esqueleto de un algoritmo y delega partes
  a las subclases.

  ------------------------------------------------------------------
  Ejemplo 13: Template Method
  ------------------------------------------------------------------

  class GeneradorReporte
    def generar
      encabezado
      contenido
      pie
      firmar
    end

    def encabezado
      raise NotImplementedError
    end

    def contenido
      raise NotImplementedError
    end

    def pie
      raise NotImplementedError
    end

    def firmar
      puts "Reporte generado por #{self.class}"
    end
  end

  class ReportePDF < GeneradorReporte
    def encabezado
      puts "=== PDF HEADER ==="
    end

    def contenido
      puts "Contenido del PDF"
    end

    def pie
      puts "=== PDF FOOTER ==="
    end
  end

  class ReporteHTML < GeneradorReporte
    def encabezado
      puts "<header>HTML Header</header>"
    end

    def contenido
      puts "<main>Contenido HTML</main>"
    end

    def pie
      puts "<footer>HTML Footer</footer>"
    end
  end

  ReportePDF.new.generar
  ReporteHTML.new.generar

  10. ELIMINANDO CONDICIONALES CON POLIMORFISMO

  Uno de los mayores beneficios del polimorfismo es eliminar
  condicionales (case/when, if/elsif).

  ------------------------------------------------------------------
  Ejemplo 14: Sin polimorfismo (con condicionales)
  ------------------------------------------------------------------

  class ProcesadorPago
    def procesar(tipo, monto)
      case tipo
      when :tarjeta
        "Procesando pago con tarjeta: $#{monto}"
      when :efectivo
        "Procesando pago en efectivo: $#{monto}"
      when :transferencia
        "Procesando transferencia: $#{monto}"
      when :cripto
        "Procesando criptomoneda: $#{monto}"
      else
        "Tipo de pago desconocido"
      end
    end
  end

  # Cada vez que añadimos un tipo, modificamos el case.

  ------------------------------------------------------------------
  Ejemplo 15: Con polimorfismo (sin condicionales)
  ------------------------------------------------------------------

  class PagoTarjeta
    def procesar(monto)
      "Procesando pago con tarjeta: $#{monto}"
    end
  end

  class PagoEfectivo
    def procesar(monto)
      "Procesando pago en efectivo: $#{monto}"
    end
  end

  class PagoTransferencia
    def procesar(monto)
      "Procesando transferencia: $#{monto}"
    end
  end

  class PagoCripto
    def procesar(monto)
      "Procesando criptomoneda: $#{monto}"
    end
  end

  class ProcesadorPago
    def procesar(metodo_pago, monto)
      metodo_pago.procesar(monto)
    end
  end

  # Nuevo tipo: solo creamos una clase nueva, no modificamos existente.
  # Esto es el Principio Abierto/Cerrado (OCP).

  11. EJEMPLO COMPLETO: SISTEMA DE NOTIFICACIONES

  ------------------------------------------------------------------
  Ejemplo 16: Sistema de notificaciones polimórfico
  ------------------------------------------------------------------

  class Notificador
    def initialize(canales = [])
      @canales = canales
    end

    def notificar(mensaje, destinatario)
      @canales.each do |canal|
        if canal.respond_to?(:enviar)
          canal.enviar(mensaje, destinatario)
        else
          puts "Canal #{canal.class} no implementa enviar"
        end
      end
    end

    def agregar_canal(canal)
      @canales << canal
    end
  end

  class Email
    def enviar(mensaje, destinatario)
      puts "Email a #{destinatario[:email]}: #{mensaje}"
      { canal: :email, enviado: true, timestamp: Time.now }
    end
  end

  class SMS
    def enviar(mensaje, destinatario)
      puts "SMS a #{destinatario[:telefono]}: #{mensaje[0..100]}"
      { canal: :sms, enviado: true, timestamp: Time.now }
    end
  end

  class PushNotification
    def enviar(mensaje, destinatario)
      puts "Push a #{destinatario[:dispositivo_id]}: #{mensaje}"
      { canal: :push, enviado: true, timestamp: Time.now }
    end
  end

  class SlackBot
    def enviar(mensaje, destinatario)
      puts "Slack a #{destinatario[:slack_channel]}: #{mensaje}"
      { canal: :slack, enviado: true, timestamp: Time.now }
    end
  end

  notificador = Notificador.new([Email.new, SMS.new])
  notificador.agregar_canal(SlackBot.new)

  notificador.notificar("Compra confirmada #1234", {
    email: "cliente@ejemplo.com",
    telefono: "+525512345678",
    slack_channel: "#pedidos"
  })

  # Añadir un nuevo canal es tan simple como crear una clase que
  # implemente enviar. No se modifica nada existente.

  12. DUCK TYPING PRÁCTICO: OBJETOS QUE RESPONDEN A +, <<, ETC.

  Ruby usa duck typing extensivamente. Por ejemplo, + funciona con
  números y strings; << con arrays, strings, y buffers.

  ------------------------------------------------------------------
  Ejemplo 17: Duck typing con operadores
  ------------------------------------------------------------------

  class Presupuesto
    def initialize(monto)
      @monto = monto
    end

    def +(otro)
      if otro.respond_to?(:monto)
        Presupuesto.new(@monto + otro.monto)
      elsif otro.is_a?(Numeric)
        Presupuesto.new(@monto + otro)
      else
        raise TypeError, "No se puede sumar #{otro.class} a Presupuesto"
      end
    end

    def monto
      @monto
    end

    def to_s
      "$#{@monto}"
    end
  end

  p1 = Presupuesto.new(1000)
  p2 = Presupuesto.new(500)
  puts p1 + p2    # => $1500
  puts p1 + 200   # => $1200

  13. TIPS Y BUENAS PRÁCTICAS

  1. Prefiere duck typing sobre herencia para polimorfismo.
  2. No preguntes "¿qué eres?" (is_a?), pregunta "¿qué haces?" (respond_to?).
  3. Siempre implementa respond_to_missing? cuando uses method_missing.
  4. Usa polimorfismo para eliminar condicionales (case/when).
  5. Diseña tus objetos pensando en los mensajes que recibirán.
  6. No abuses de method_missing: es poderoso pero puede hacer el
     código difícil de entender y depurar.
  7. El patrón Strategy es excelente para comportamientos intercambiables.
  8. Template Method es bueno cuando el esqueleto del algoritmo es fijo
     pero los detalles varían.

  RESUMEN DEL CAPÍTULO

  • Polimorfismo: objetos diferentes responden al mismo mensaje de
    formas diferentes.
  • Duck typing: si responde al mensaje, es del tipo adecuado.
  • respond_to? verifica si un objeto puede recibir un mensaje.
  • method_missing captura mensajes no definidos para manejo dinámico.
  • respond_to_missing? debe acompañar a method_missing.
  • El polimorfismo elimina condicionales y hace el código extensible.
  • Strategy y Template Method son patrones basados en polimorfismo.
  • El Principio Abierto/Cerrado (OCP) se logra con polimorfismo.

  verificar los plugins.

  PREGUNTAS FRECUENTES (FAQ)

  P: ¿El duck typing reemplaza la herencia?
  R: No la reemplaza, la complementa. Duck typing es ideal para
     comportamientos compartidos entre clases no relacionadas. La
     herencia es mejor para especialización de tipos relacionados.

  P: ¿Cuándo debería usar method_missing?
  R: Úsalo para delegación dinámica (proxies), builders, DSLs, o
     cuando los nombres de métodos no se conocen en tiempo de diseño.
     No lo uses para evitar escribir métodos que podrías definir
     normalmente.

  P: ¿Es necesario implementar respond_to_missing? siempre?
  R: Sí. Si no lo haces, respond_to? devolverá false para métodos
     manejados por method_missing, rompiendo el duck typing.

  P: ¿Polimorfismo vs. case/when? ¿Cuál es mejor?
  R: El polimorfismo es mejor porque respeta el Principio Abierto/
     Cerrado: puedes añadir nuevos tipos sin modificar código existente.
     Un case/when requiere modificación cada vez que añades un tipo.

  P: ¿Cómo sé si estoy usando bien el duck typing?
  R: Si tu código usa is_a? o kind_of? para decidir qué hacer, estás
     haciendo mal duck typing. Usa respond_to? o, mejor aún, confía
     en que el objeto responderá al mensaje (si no, recibirás un
     NoMethodError).

  P: ¿Qué es el "Principio Abierto/Cerrado"?
  R: Establece que las clases deben estar abiertas para extensión pero
     cerradas para modificación. El polimorfismo logra esto: añades
     nuevas clases en lugar de modificar las existentes.

  P: ¿Puedo combinar herencia y duck typing?
  R: Sí, son ortogonales. Puedes tener una jerarquía de herencia y
     también pasar objetos a métodos que esperan un duck type. Cada
     herramienta para su propósito.

  DESAFÍO DEL ARQUITECTO

  Construye un sistema de procesamiento de datos polimórfico.

  1. Define un duck type "Procesador" con métodos:
     - procesar(datos): procesa los datos y devuelve resultado
     - nombre: nombre descriptivo del procesador

  2. Implementa estos procesadores:
     - ProcesadorTexto: mayúsculas, cuenta palabras
     - ProcesadorNumerico: suma, promedio, mediana
     - ProcesadorJSON: parse y extrae campos
     - ProcesadorCSV: parsea, filtra filas
     - ProcesadorCifrado: cifra/descifra con método simple

  3. Clase Pipeline:
     - Mantiene una lista de procesadores
     - Método agregar(procesador): añade al pipeline
     - Método ejecutar(datos): pasa datos por cada procesador
       secuencialmente
     - Método ejecutar_paralelo(datos): ejecuta procesadores en
       paralelo (simulado con iteración)
     - Usa respond_to? para verificar que cada procesador implemente
       la interfaz esperada

  4. Clase ReporteProcesamiento:
     - Recibe el resultado del pipeline
     - Método generar: produce un reporte en texto
     - Método exportar(formato): exporta en formato solicitado
     - Usa duck typing para los formatos (PDF, HTML, TXT)

  5. method_missing en una clase ProxyProcesador que envuelva
     cualquier objeto y delegue procesar y nombre.

  6. Implementa respond_to_missing? en ProxyProcesador.

  7. Añade un procesador que se cargue dinámicamente solo si
     responde a procesar (simula plugin loading).

  8. Demuestra que puedes añadir un nuevo procesador sin modificar
     ninguna clase existente (OCP).

  Este desafío integra: duck typing, respond_to?, method_missing,
  respond_to_missing?, Strategy pattern, Pipeline pattern, y
  Principio Abierto/Cerrado.
