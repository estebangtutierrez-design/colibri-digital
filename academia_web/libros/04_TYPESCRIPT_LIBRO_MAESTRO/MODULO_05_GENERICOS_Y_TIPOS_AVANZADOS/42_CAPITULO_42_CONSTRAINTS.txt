BIENVENIDA DEL PROFESOR BÚHO

Querido estudiante,

En el capítulo anterior aprendiste que los genéricos te permiten escribir
código que funciona con cualquier tipo. Pero hay un problema: muchas veces
necesitamos que ese "cualquier tipo" tenga ciertas capacidades mínimas.
Queremos acceder a una propiedad `.length`, invocar un método `.toString()`,
o garantizar que dos valores sean comparables.

¿Cómo logramos esto sin perder la flexibilidad de los genéricos?

La respuesta son las **restricciones** o **constraints**. Con la palabra
clave `extends` podemos decirle a TypeScript: "El tipo T debe tener al
menos estas características". Así combinamos lo mejor de ambos mundos:
abstracción genérica con garantías concretas.

En este capítulo dominarás el uso de `extends` en genéricos, las
restricciones con `keyof`, las restricciones condicionales, y verás
patrones reales donde las constraints son indispensables.

1. SINTAXIS BÁSICA DE CONSTRAINTS
===============================================================================

Un constraint se declara usando la palabra `extends` después del parámetro
de tipo genérico:

```typescript
function longitud<T extends { length: number }>(item: T): number {
  return item.length;
}

// Uso correcto
longitud("Hola");           // string tiene length, resultado: 4
longitud([1, 2, 3]);        // Array tiene length, resultado: 3
longitud({ length: 10 });   // Objeto con length, resultado: 10

// Error
// longitud(42);
// Error: Argument of type 'number' is not assignable to
// parameter of type '{ length: number }'
```

**Lectura de la declaración:**

`<T extends { length: number }>` se lee: "T es un tipo que debe extender
(o cumplir con) la estructura que tiene una propiedad `length` de tipo
`number`".

**La diferencia clave:**

```typescript
// Sin constraint: acepta cualquier cosa
function imprimirSinConstraint<T>(valor: T): void {
  console.log(valor);
  // No podemos acceder a .length porque T puede ser cualquier tipo
}

// Con constraint: solo tipos con .length
function imprimirConConstraint<T extends { length: number }>(valor: T): void {
  console.log(`Longitud: ${valor.length}`);
}
```

El constraint no limita el tipo a una sola opción concreta. Lo que hace
es establecer un **mínimo requerido**. Cualquier tipo que tenga al menos
las propiedades especificadas será aceptado.

===============================================================================
2. CONSTRAINTS CON INTERFACES
===============================================================================

Es más limpio y reutilizable definir el constraint como una interfaz:

```typescript
interface Medible {
  length: number;
}

function longitud<T extends Medible>(item: T): number {
  return item.length;
}

// Funciona con string, Array, cualquier cosa medible
```

**Interfaz como contrato mínimo:**

```typescript
interface Identificable {
  id: number | string;
}

interface Nombrable {
  nombre: string;
}

function mostrarInfo<T extends Identificable & Nombrable>(item: T): string {
  return `[${item.id}] ${item.nombre}`;
}

// Objeto debe tener BOTH id and nombre
const usuario = { id: 1, nombre: "Ana", email: "ana@test.com" };
console.log(mostrarInfo(usuario)); // "[1] Ana"
```

**Constraint con interface genérica:**

```typescript
interface Comparable<T> {
  esIgualA(otro: T): boolean;
}

function buscarIgual<T extends Comparable<T>>(item: T, lista: T[]): T | undefined {
  return lista.find(elem => elem.esIgualA(item));
}

class Persona implements Comparable<Persona> {
  constructor(public id: number, public nombre: string) {}

  esIgualA(otro: Persona): boolean {
    return this.id === otro.id;
  }
}

const personas = [
  new Persona(1, "Ana"),
  new Persona(2, "Luis")
];

const encontrada = buscarIgual(new Persona(1, "Ana"), personas);
```

===============================================================================
3. CONSTRAINTS CON TIPOS PRIMITIVOS
===============================================================================

Aunque es menos común, puedes restringir un genérico a un tipo primitivo
específico o a una unión de primitivos:

```typescript
// Solo acepta tipos que sean subtipos de string
function procesarTexto<T extends string>(texto: T): T {
  return texto.trim() as T;
}

// Solo acepta tipos numéricos
function operarNumero<T extends number>(a: T, b: T): number {
  return a + b;
}

// Unión de primitivos
function mostrar<T extends string | number>(valor: T): string {
  return String(valor);
}
```

**Uso práctico: literales preservados**

```typescript
function crearColor<T extends "rojo" | "verde" | "azul">(color: T): { color: T } {
  return { color };
}

const c1 = crearColor("rojo");
//    ^? const c1: { color: "rojo" }

const c2 = crearColor("amarillo");
// Error: Argument of type '"amarillo"' is not assignable
// to parameter of type '"rojo" | "verde" | "azul"'
```

Este patrón preserva el tipo literal exacto en lugar de ampliarlo a
`string`. Es muy útil cuando trabajamos con configuraciones o eventos.

===============================================================================
4. CONSTRAINTS CON KEYOF
===============================================================================

`keyof T` produce una unión de las claves de T. Usado como constraint,
permite crear funciones que acceden a propiedades de forma segura:

```typescript
function obtenerPropiedad<T, K extends keyof T>(objeto: T, clave: K): T[K] {
  return objeto[clave];
}

const persona = {
  nombre: "Ana",
  edad: 30,
  activo: true
};

const nombre = obtenerPropiedad(persona, "nombre");
//    ^? const nombre: string

const edad = obtenerPropiedad(persona, "edad");
//    ^? const edad: number

// Error en compilación
// obtenerPropiedad(persona, "email");
// Error: Argument of type '"email"' is not assignable
// to parameter of type '"nombre" | "edad" | "activo"'
```

**¿Por qué es poderoso?**

- El autocompletado muestra solo las claves válidas.
- El tipo de retorno es exactamente el tipo del valor de esa propiedad.
- El compilador detecta errores si la clave no existe.

**Ejemplo: actualizar propiedad**

```typescript
function actualizarPropiedad<T, K extends keyof T>(
  objeto: T,
  clave: K,
  valor: T[K]
): T {
  return { ...objeto, [clave]: valor };
}

const actualizado = actualizarPropiedad(persona, "edad", 31);
//    ^? const actualizado: { nombre: string; edad: number; activo: boolean }

// Error:
// actualizarPropiedad(persona, "edad", "treinta");
// Error: Argument of type 'string' is not assignable to parameter of type 'number'
```

===============================================================================
5. CONSTRAINTS CON TIPOS DE CLASE (CONSTRUCTOR)
===============================================================================

Podemos restringir un genérico para que solo acepte constructores de
clases (funciones que pueden ser instanciadas con `new`):

```typescript
function crearInstancia<T>(
  TipoClase: new (...args: any[]) => T,
  ...args: any[]
): T {
  return new TipoClase(...args);
}

class Producto {
  constructor(public nombre: string, public precio: number) {}
}

const producto = crearInstancia(Producto, "Laptop", 1500);
//    ^? const producto: Producto
```

**Constraint de constructor con parámetros tipados:**

```typescript
interface Constructor<T> {
  new (...args: any[]): T;
}

function crearMultiples<T>(
  Constructor: Constructor<T>,
  cantidad: number,
  ...args: any[]
): T[] {
  return Array.from({ length: cantidad }, () => new Constructor(...args));
}

const productos = crearMultiples(Producto, 3, "Genérico", 100);
//    ^? const productos: Producto[]
```

Este patrón se usa en fábricas, inyección de dependencias y ORMs.

===============================================================================
6. CONSTRAINTS ANIDADOS Y CONDICIONALES
===============================================================================

Puedes combinar múltiples constraints y usar tipos que dependen de otros:

```typescript
// Restricción con dos niveles
interface Serializable {
  toJSON(): Record<string, unknown>;
}

function guardar<T extends Serializable>(item: T): void {
  const datos = item.toJSON();
  localStorage.setItem("datos", JSON.stringify(datos));
}

class Usuario implements Serializable {
  constructor(public id: number, public nombre: string) {}

  toJSON(): Record<string, unknown> {
    return { id: this.id, nombre: this.nombre };
  }
}
```

**Constraint que usa otro parámetro de tipo:**

```typescript
function parejaValida<T, U extends T>(primero: T, segundo: U): [T, U] {
  return [primero, segundo];
}

// U debe ser subtipo de T
const r1 = parejaValida("hola", "mundo");  // OK: ambos string
const r2 = parejaValida("hola", "adiós");  // OK: ambos string
// const r3 = parejaValida("hola", 42);     // Error: number no es subtipo de string
```

**Auto-referencia con constraint:**

```typescript
interface AutoReferenciable<T extends AutoReferenciable<T>> {
  padre: T | null;
  hijos: T[];
}

class Nodo implements AutoReferenciable<Nodo> {
  constructor(
    public valor: number,
    public padre: Nodo | null = null,
    public hijos: Nodo[] = []
  ) {}
}
```

Este patrón se conoce como "F-Bounded Polymorphism" o polimorfismo
acotado recursivo. Es útil para árboles y estructuras jerárquicas.

===============================================================================
7. ERRORES COMUNES CON CONSTRAINTS
===============================================================================

**Error 1: Olvidar que el constraint no es el tipo final**

```typescript
interface ConLongitud {
  length: number;
}

function ejemplo<T extends ConLongitud>(item: T): void {
  // Esto funciona
  console.log(item.length);

  // Esto NO funciona — no todos los tipos con length tienen toUpperCase
  // console.log(item.toUpperCase());
}

// Solución: restringir más o usar type guards
function ejemploString<T extends ConLongitud & { toUpperCase(): string }>(
  item: T
): void {
  console.log(item.toUpperCase());
}
```

**Error 2: Restricción demasiado amplia o demasiado estrecha**

```typescript
// ❌ Demasiado amplia: no obtenemos información útil
function procesoGenerico<T>(valor: T): void {
  // No podemos hacer mucho con T
}

// ❌ Demasiado estrecha: perdemos genericidad
function soloString(valor: string): void {
  // Solo acepta strings
}

// ✅ Balanceado: acepta tipos con length
function procesoMedible<T extends { length: number }>(valor: T): void {
  console.log(valor.length);
}
```

**Error 3: Asignar a una propiedad restringida**

```typescript
interface ConNombre {
  nombre: string;
}

function cambiarNombre<T extends ConNombre>(objeto: T): void {
  // ✅ Esto funciona
  objeto.nombre = "Nuevo";

  // ❌ Esto NO — podríamos estar perdiendo precisión
  // objeto.nombre = "Nuevo" as T["nombre"];
}
```

8. PATRONES AVANZADOS CON CONSTRAINTS
===============================================================================

**Patrón: Fábrica con validación**

```typescript
interface Validable {
  validar(): boolean;
}

function crearValidado<T extends Validable>(
  factory: new (...args: any[]) => T,
  ...args: any[]
): T {
  const instancia = new factory(...args);
  if (!instancia.validar()) {
    throw new Error("La instancia no pasó la validación");
  }
  return instancia;
}
```

**Patrón: Selector de propiedades**

```typescript
function seleccionar<T, K extends keyof T>(
  objeto: T,
  ...claves: K[]
): Pick<T, K> {
  const resultado = {} as Pick<T, K>;
  for (const clave of claves) {
    resultado[clave] = objeto[clave];
  }
  return resultado;
}

const usuario = { id: 1, nombre: "Ana", email: "ana@test.com", password: "123" };
const publico = seleccionar(usuario, "id", "nombre");
//    ^? const publico: Pick<{ id: number; nombre: string; ...}, "id" | "nombre">
```

**Patrón: Actualización parcial con validación**

```typescript
type ActualizacionParcial<T> = {
  [K in keyof T]?: T[K];
};

function actualizar<T>(
  objeto: T,
  cambios: ActualizacionParcial<T>
): T {
  return { ...objeto, ...cambios };
}

const user = { nombre: "Ana", edad: 30, email: "ana@test.com" };
const updated = actualizar(user, { edad: 31 });
//    ^? const updated: { nombre: string; edad: number; email: string }
```

9. CONSTRAINTS CON GENÉRICOS EN CLASES
===============================================================================

Las clases también se benefician de las constraints:

```typescript
interface Almacenable {
  id: string;
}

class Repositorio<T extends Almacenable> {
  private items: Map<string, T> = new Map();

  guardar(item: T): void {
    this.items.set(item.id, item);
  }

  obtener(id: string): T | undefined {
    return this.items.get(id);
  }

  eliminar(id: string): boolean {
    return this.items.delete(id);
  }

  listar(): T[] {
    return Array.from(this.items.values());
  }
}

interface Producto extends Almacenable {
  id: string;
  nombre: string;
  precio: number;
}

const repo = new Repositorio<Producto>();
repo.guardar({ id: "p1", nombre: "Laptop", precio: 1500 });
```

**Clase con múltiples constraints:**

```typescript
interface Claveable {
  obtenerClave(): string;
}

interface Cacheable {
  caducidad: number;
  estaCaducado(): boolean;
}

class CacheManager<T extends Claveable & Cacheable> {
  private cache: Map<string, T> = new Map();

  agregar(item: T): void {
    this.cache.set(item.obtenerClave(), item);
  }

  obtener(clave: string): T | undefined {
    const item = this.cache.get(clave);
    if (item && item.estaCaducado()) {
      this.cache.delete(clave);
      return undefined;
    }
    return item;
  }
}
```

10. EJEMPLO INTEGRADOR: SISTEMA DE VALIDACIÓN
===============================================================================

Unamos todos los conceptos en un sistema práctico:

```typescript
// === Interfaces base ===
interface Validable {
  validar(): boolean;
  errores(): string[];
}

interface Persistible {
  id: string;
  guardar(): Promise<void>;
}

type Entidad = Validable & Persistible;

// === Repositorio genérico con constraint ===
class RepositorioEntidad<T extends Entidad> {
  private entidades: Map<string, T> = new Map();

  async agregar(entidad: T): Promise<boolean> {
    if (!entidad.validar()) {
      console.error("Error de validación:", entidad.errores());
      return false;
    }
    this.entidades.set(entidad.id, entidad);
    await entidad.guardar();
    return true;
  }

  obtenerPorId(id: string): T | undefined {
    return this.entidades.get(id);
  }

  buscarPorCampo<K extends keyof T>(campo: K, valor: T[K]): T[] {
    return Array.from(this.entidades.values()).filter(
      e => e[campo] === valor
    );
  }
}

// === Implementación concreta ===
class Usuario implements Entidad {
  constructor(
    public id: string,
    public nombre: string,
    public email: string,
    private _errores: string[] = []
  ) {}

  validar(): boolean {
    this._errores = [];
    if (!this.nombre || this.nombre.length < 2) {
      this._errores.push("El nombre debe tener al menos 2 caracteres");
    }
    if (!this.email.includes("@")) {
      this._errores.push("Email inválido");
    }
    return this._errores.length === 0;
  }

  errores(): string[] {
    return this._errores;
  }

  async guardar(): Promise<void> {
    console.log(`Guardando usuario ${this.id}...`);
  }
}

// === Uso ===
async function main() {
  const repo = new RepositorioEntidad<Usuario>();

  const ana = new Usuario("u1", "Ana", "ana@test.com");
  const exito = await repo.agregar(ana);
  console.log(`¿Agregado? ${exito}`);

  // Búsqueda con keyof constraint
  const encontrados = repo.buscarPorCampo("nombre", "Ana");
  console.log(`Encontrados: ${encontrados.length}`);
}

main();
```

===============================================================================
RESUMEN DEL CAPÍTULO
===============================================================================

- Las constraints (`extends`) permiten exigir que un tipo genérico cumpla
  con una estructura mínima.
- `T extends Interfaz` significa "T debe tener al menos las propiedades
  de Interfaz".
- `keyof T` como constraint permite acceder a propiedades de forma segura.
- Las constraints pueden combinarse con intersecciones (`&`).
- Los constructores pueden restringirse con `new (...args: any[]) => T`.
- El balance correcto entre genericidad y restricción es clave para un
  buen diseño de APIs.

===============================================================================
PREGUNTAS FRECUENTES
===============================================================================

❓ **¿`extends` en genéricos es lo mismo que `extends` en clases?**
Técnicamente usan la misma palabra clave, pero conceptualmente son
diferentes. En clases, `extends` crea herencia. En genéricos, `extends`
establece un **constraint** o restricción de que el tipo debe cumplir
con cierta estructura.

❓ **¿Puedo tener un valor por defecto y un constraint en el mismo T?**
Sí. La sintaxis es `<T extends SomeType = DefaultType>`.

❓ **¿Qué pasa si paso un tipo que cumple con más de lo requerido?**
Es perfectamente válido. El constraint establece un mínimo, no un máximo.
Mientras el tipo tenga al menos lo requerido, se acepta.

❓ **¿Los constraints afectan el rendimiento?**
No, son solo verificaciones en tiempo de compilación. No hay impacto
en runtime.

❓ **¿Puedo usar `keyof` con cualquier tipo?**
`keyof` sobre un tipo produce la unión de sus claves. Sobre tipos
primitivos como `string` o `number`, produce tipos como `string | number`
(o `never` para `boolean`). Para objetos, produce las claves como
literales.
