1. INTRODUCCIÓN AL TIPO ANY
===============================================================================

El tipo any es el supertipo de todos los tipos en TypeScript. Una variable
declarada como any puede contener cualquier valor y se puede operar sin
ninguna verificación de tipos.

```typescript
let variable: any;

variable = 42;                        // number
variable = "Hola";                    // string
variable = true;                      // boolean
variable = { nombre: "Lucas" };       // object
variable = [1, 2, 3];                 // array

// Puedes hacer cualquier operación sin error de compilación
variable.toUpperCase();               // OK (aunque sea number en runtime)
variable()                            // OK (aunque no sea función)
variable.propiedadInexistente();      // OK
```

Cuando marcas una variable como any, le estás diciendo al compilador:
"Confía en mí, sé lo que estoy haciendo." TypeScript desactiva toda
verificación de tipos para esa variable.

---

📌 **any desactiva completamente el sistema de tipos**

```typescript
function procesar(valor: any): any {
  return valor * 2;
}

let resultado = procesar("texto");
// TypeScript cree que resultado es any, sin errores
// En runtime: NaN
```

---

🔵 **Ejemplo 1: Inferencia a any**

```typescript
// Sin tipo explícito y sin inicialización
let valor;    // type: any (inferido)
valor = 10;
valor = "texto";

// Parámetros sin tipo (modo no-strict)
function imprimir(mensaje) {
  console.log(mensaje);
}

// Arrays sin tipo
let datos = [];    // type: any[]
datos.push(1);
datos.push("texto");
datos.push({});
```

🔵 **Ejemplo 2: any en desestructuración y valores dinámicos**

```typescript
// any se propaga
function obtenerDatos(): any {
  return JSON.parse('{"nombre": "Lucas", "edad": 30}');
}

let datos = obtenerDatos();
console.log(datos.nombre.toUpperCase());  // "LUCAS" (funciona)
console.log(datos.edad().noExiste);       // Error en runtime, no en compilación

// Solución: tipar el JSON.parse con genéricos
function obtenerDatosTipados<T>(json: string): T {
  return JSON.parse(json);
}

interface Usuario {
  nombre: string;
  edad: number;
}

let usuario = obtenerDatosTipados<Usuario>('{"nombre": "Lucas", "edad": 30}');
console.log(usuario.nombre.toUpperCase());  // ✅ "LUCAS" (tipado seguro)
```

2. ¿CUÁNDO ESTÁ JUSTIFICADO USAR ANY?
===============================================================================

Aunque any debe evitarse en la medida de lo posible, existen situaciones
legítimas donde su uso está justificado.

---

📌 **Casos de uso legítimos para any**

1. Migración gradual de JavaScript a TypeScript
2. Bibliotecas de terceros sin tipos
3. Valores verdaderamente dinámicos (entrada de usuario)
4. Pruebas unitarias y mock objects
5. Polyfills y código de compatibilidad
6. Configuración y metaprogramación avanzada

```typescript
// Ejemplo de migración gradual
// Archivo legacy.js (aún no tipado)
function calcularTotal(items) {
  return items.reduce((sum, item) => sum + item.precio, 0);
}

// Durante la migración, aceptamos any temporalmente
function calcularTotalMigrando(items: any[]): number {
  return items.reduce((sum: number, item: any) => sum + item.precio, 0);
}
```

---

🔵 **Ejemplo 3: Bibliotecas sin tipos**

```typescript
// Biblioteca externa sin tipos (@types no disponible)
import * as libreriaAntigua from "libreria-antigua";

// Podemos declarar un módulo con any
declare module "libreria-antigua" {
  export function hacerAlgo(...args: any[]): any;
  export const configuracion: any;
}

// O usar any directamente
let resultado = (libreriaAntigua as any).metodoRaro(params);
```

🔵 **Ejemplo 4: JSON.parse y datos externos**

```typescript
// Datos de una API externa
async function fetchUsuario(id: number): Promise<any> {
  const respuesta = await fetch(`/api/usuarios/${id}`);
  return respuesta.json();
}

// Mejor: tipar después de validar
async function fetchUsuarioSeguro(id: number): Promise<Usuario> {
  const respuesta = await fetch(`/api/usuarios/${id}`);
  const datos: unknown = await respuesta.json();

  // Validación en runtime
  if (!esUsuarioValido(datos)) {
    throw new Error("Datos de usuario inválidos");
  }
  return datos;
}

function esUsuarioValido(datos: any): datos is Usuario {
  return typeof datos === "object"
    && datos !== null
    && typeof datos.nombre === "string"
    && typeof datos.edad === "number";
}
```

===============================================================================
3. LOS PELIGROS DEL TIPO ANY
===============================================================================

El uso excesivo o incorrecto de any elimina los beneficios del sistema de
tipos de TypeScript y reintroduce los mismos problemas que JavaScript tiene.

---

📌 **Any propaga la falta de tipos como un virus**

```typescript
function obtenerPrimero(lista: any[]): any {
  return lista[0];
}

let numeros = [1, 2, 3];
let primero = obtenerPrimero(numeros);  // any

// La falta de tipos se propaga
let doble = primero * 2;                // any (ningún error)
console.log(doble.toUpperCase());        // Error en runtime
```

---

🔵 **Ejemplo 5: Errores silenciosos con any**

```typescript
function duplicar(valor: any): any {
  return valor * 2;
}

// Estos errores NO se detectan en compilación
console.log(duplicar(5));               // 10 (correcto)
console.log(duplicar("texto"));         // NaN (error silencioso)
console.log(duplicar(null));            // 0 (error silencioso)
console.log(duplicar(undefined));       // NaN (error silencioso)
console.log(duplicar({}));              // NaN (error silencioso)

// Versión tipada: los errores se detectan
function duplicarSeguro(valor: number): number {
  return valor * 2;
}

// duplicarSeguro("texto");  // Error en compilación
```

🔵 **Ejemplo 6: Pérdida de autocompletado y refactorización**

```typescript
interface Producto {
  id: number;
  nombre: string;
  precio: number;
  categoria: string;
}

// Con any: sin ayuda del editor
function mostrarProducto(producto: any): void {
  console.log(producto.nomebre);  // Sin error, pero está mal escrito
}

// Con tipo: el editor te guía
function mostrarProductoSeguro(producto: Producto): void {
  console.log(producto.nombre);   // Autocompletado funciona
  // console.log(producto.nomebre); // Error: no existe en Producto
}
```

🔵 **Ejemplo 7: El efecto dominó de any**

```typescript
// Comienza con un any en una función de utilidad
function extraerCampo(objeto: any, campo: string): any {
  return objeto[campo];
}

// Esto contamina todo el código aguas arriba
function procesarUsuario(id: number): string {
  let datos = extraerCampo({ nombre: "Lucas" }, "nombre");
  // datos es any, así que no hay advertencia
  return datos * 2;  // Error en runtime, no en compilación
}
```

4. ALTERNATIVAS A ANY
===============================================================================

TypeScript ofrece alternativas más seguras que any para casi todos los
casos de uso.

---

📌 **unknown es la alternativa segura a any**

```typescript
// any permite todo sin verificación
let valorAny: any = "texto";
valorAny.toFixed(2);  // Sin error (pero falla en runtime)

// unknown requiere verificación de tipo
let valorUnknown: unknown = "texto";
// valorUnknown.toFixed(2);  // Error: Object is of type 'unknown'

// Debes hacer type narrowing primero
if (typeof valorUnknown === "string") {
  console.log(valorUnknown.toUpperCase());  // OK
}
```

---

🔵 **Ejemplo 8: Genéricos como alternativa a any**

```typescript
// ❌ Con any (pierdes el tipo)
function identidadAny(valor: any): any {
  return valor;
}

let resultadoAny = identidadAny("Hola");
// resultadoAny es any, no puedes usar métodos de string

// ✅ Con genéricos (preservas el tipo)
function identidad<T>(valor: T): T {
  return valor;
}

let resultado = identidad("Hola");
// resultado es string: autocompletado y verificación funcionan

let resultadoNum = identidad(42);
// resultadoNum es number
```

🔵 **Ejemplo 9: Tipos específicos en lugar de any**

```typescript
// ❌ any en parámetros
function procesarDatos(datos: any): void {
  for (const item of datos) {
    console.log(item.nombre);
  }
}

// ✅ Tipo específico
function procesarDatosTipado(datos: { nombre: string }[]): void {
  for (const item of datos) {
    console.log(item.nombre);
  }
}

// ✅ O con Record para objetos dinámicos
function procesarMapa(datos: Record<string, unknown>): void {
  for (const clave in datos) {
    console.log(clave, datos[clave]);
  }
}
```

🔵 **Ejemplo 10: any controlado con type guards**

```typescript
// Usa any solo en el punto de entrada, luego valida
function cargarConfiguracion(): Configuracion {
  let raw: any = leerArchivoConfig();
  return validarConfiguracion(raw);
}

function validarConfiguracion(raw: any): Configuracion {
  if (typeof raw !== "object" || raw === null) {
    throw new Error("Configuración inválida");
  }
  return {
    host: String(raw.host ?? "localhost"),
    port: Number(raw.port ?? 3000),
    ssl: Boolean(raw.ssl ?? false)
  };
}

interface Configuracion {
  host: string;
  port: number;
  ssl: boolean;
}
```

5. ESTRATEGIAS PARA MINIMIZAR EL USO DE ANY

🔵 **Ejemplo 11: ESLint y no-explicit-any**

```typescript
// Configuración ESLint recomendada
// .eslintrc.json:
// {
//   "rules": {
//     "@typescript-eslint/no-explicit-any": "warn"
//   }
// }

// En lugar de:
function loguear(datos: any): void {  // Warning de ESLint
  console.log(datos);
}

// Usa unknown si es necesario
function loguearSeguro(datos: unknown): void {
  console.log(datos);
}
```

🔵 **Ejemplo 12: Migración gradual con @ts-expect-error**

```typescript
// Durante la migración, usa @ts-expect-error en lugar de any
function funcionLegacy() {
  // @ts-expect-error - Migración en progreso
  return JSON.parse(someInput);
}

// Mejor que:
function funcionLegacyAny(): any {
  return JSON.parse(someInput);
}
```

🔵 **Ejemplo 13: Patrón de adaptador para librerías sin tipos**

```typescript
// Adaptador con tipos
interface LibreriaAdaptada {
  buscar usuarios(id: number): Usuario;
}

function crearAdaptador(lib: any): LibreriaAdaptada {
  return {
    buscar(id: number): Usuario {
      const resultado = lib.query(`SELECT * FROM users WHERE id = ${id}`);
      return {
        id: resultado[0].id,
        nombre: resultado[0].name,
        email: resultado[0].email
      };
    }
  };
}
```

RESUMEN DEL CAPÍTULO
===============================================================================

- El tipo any desactiva completamente el sistema de tipos de TypeScript
  para la variable donde se aplica, permitiendo cualquier operación sin
  verificación.
- any está justificado en migraciones graduales, bibliotecas sin tipos,
  y situaciones donde el tipo es verdaderamente desconocido.
- El principal peligro de any es que elimina la seguridad que TypeScript
  proporciona, reintroduciendo errores silenciosos en runtime.
- any se propaga como un virus: cualquier valor que toque pierde su
  información de tipo.
- Las alternativas a any incluyen unknown (con type narrowing), genéricos
  (para preservar tipos), y tipos específicos (interfaces, Record, etc.).
- Usa ESLint con la regla no-explicit-any para mantener el uso de any
  bajo control en tu proyecto.
- El objetivo no es eliminar any por completo, sino usarlo de forma
  consciente y controlada.

🔵 **Ejemplo 14: Patrón de caja sellada con any controlado**

```typescript
// A veces necesitas un contenedor que acepte cualquier tipo
// pero quieres mantener el control en los puntos de entrada/salida

class Caja<T> {
  private contenido: T;

  constructor(contenido: T) {
    this.contenido = contenido;
  }

  obtener(): T {
    return this.contenido;
  }

  establecer(nuevo: T): void {
    this.contenido = nuevo;
  }
}

// Caja segura
const cajaNumero = new Caja(42);
console.log(cajaNumero. obtener());  // 42

// Si necesitas flexibilidad, tipa explícitamente
const cajaFlexible = new Caja<number | string | boolean>(42);
cajaFlexible. establecer("texto");  // OK
cajaFlexible. establecer(true);     // OK
console.log(cajaFlexible.obtener()); // true
```

🔵 **Ejemplo 15: any en pruebas unitarias**

```typescript
// En tests, any es útil para mocks parciales
interface ServicioEmail {
  enviar(destino: string, asunto: string, cuerpo: string): Promise<boolean>;
  enviarMasivo(destinos: string[]): Promise<number>;
  obtenerEstadisticas(): Promise<{ enviados: number; fallidos: number }>;
}

// Mock parcial con any (solo necesitamos probar una parte)
function crearMockEmail(): any {
  return {
    enviar: async (destino: string) => {
      console.log(`Mock: email enviado a ${destino}`);
      return true;
    }
    // No implementamos enviarMasivo ni obtenerEstadisticas
  };
}

async function testEnvio() {
  const emailMock = crearMockEmail() as ServicioEmail;
  const resultado = await emailMock.enviar("test@test.com", "Hola", "Cuerpo");
  console.assert(resultado === true, "El email debería enviarse");
}

testEnvio();
```

===============================================================================
PREGUNTAS FRECUENTES
===============================================================================

❓ **¿Es malo usar any en TypeScript?**
No es "malo" per se, pero debe usarse con moderación y de forma consciente.
Cada uso de any desactiva el sistema de tipos en esa porción de código.
La regla general: prefiere siempre la opción más específica antes que any,
y documenta por qué lo usas cuando sea necesario.

❓ **¿Cuál es la diferencia entre any y unknown?**
any permite cualquier operación sin verificación de tipos. unknown representa
un valor desconocido que requiere verificación (type narrowing) antes de
poder operar con él. unknown es la opción segura cuando no conoces el tipo.

❓ **¿Cómo migro un proyecto JavaScript a TypeScript sin usar any?**
El enfoque recomendado es la migración gradual: comienza con
allowJs: true y strict: false, luego ve tipando archivos uno por uno.
Usa @ts-expect-error para puntos difíciles temporalmente. El uso de any
está permitido durante la migración, pero debe reducirse progresivamente.

❓ **¿Cómo evito que mi equipo abuse de any?**
Configura ESLint con @typescript-eslint/no-explicit-any como error.
Establece una política de código donde cada uso de any requiera una
justificación documentada. Realiza code reviews enfocados en el uso de
any. Proporciona alternativas como genéricos, unknown y type guards.
