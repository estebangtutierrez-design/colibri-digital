Bienvenido al tipo más enigmático de TypeScript: never. Si void representa
la ausencia de valor, never representa la ausencia de posibilidad.

never es el tipo de las funciones que nunca terminan, de los bucles infinitos,
de las excepciones siempre lanzadas. Pero también es una herramienta
increíblemente poderosa para escribir código exhaustivo y a prueba de errores.

El verdadero poder de never se revela en los exhaustiveness checks:
cuando cubres todos los casos posibles de un tipo unión y TypeScript te
asegura que no te has olvidado de ninguno.

Prepárate para descubrir por qué nunca es sinónimo de seguridad. 🦉

1. INTRODUCCIÓN AL TIPO NEVER
===============================================================================

El tipo never representa valores que nunca pueden ocurrir. Es el fondo
de la jerarquía de tipos: todos los tipos son asignables a never, pero
never no es asignable a ningún tipo (excepto a sí mismo).

```typescript
// Funciones que siempre lanzan una excepción
function error(mensaje: string): never {
  throw new Error(mensaje);
}

// Funciones con bucles infinitos
function bucleInfinito(): never {
  while (true) {
    console.log("Esto nunca termina");
  }
}

// Una variable de tipo never nunca puede tener un valor real
let imposible: never;
// imposible = 42;    // Error: Type 'number' is not assignable to type 'never'
// imposible = "texto"; // Error
// imposible = undefined; // Error (con strictNullChecks)
```

El compilador usa never automáticamente en ciertos contextos, como cuando
una rama de código es inalcanzable.

---

📌 **never en el sistema de tipos**

```typescript
// En uniones, never desaparece
type Test1 = string | never;   // string
type Test2 = number | never;   // number
type Test3 = never | never;    // never

// En intersecciones, never domina
type Test4 = string & never;   // never
type Test5 = number & never;   // never

// En condicionales, never puede aparecer
type TipoExcluido<T> = T extends string ? never : T;
type Test6 = TipoExcluido<string>;  // never
type Test7 = TipoExcluido<number>;  // number
```

---

🔵 **Ejemplo 1: Funciones que retornan never**

```typescript
// Lanzar excepción
function errorFatal(mensaje: string): never {
  console.error(`ERROR FATAL: ${mensaje}`);
  throw new Error(mensaje);
  // console.log("Esto nunca se ejecuta");
}

// Bucle infinito
function procesarSolicitudes(): never {
  while (true) {
    const solicitud = obtenerSolicitud();
    procesar(solicitud);
  }
}

function obtenerSolicitud(): string { return "data"; }
function procesar(s: string): void { console.log(s); }

// Diferencia: lanzar error permite que el flujo continúe después
function validarOError(valor: number): number {
  if (valor < 0) {
    return errorFatal("Valor negativo no permitido");
    // TypeScript sabe que errorFatal es never, así que esto es válido
  }
  return valor * 2;
}
```

🔵 **Ejemplo 2: throw en expresiones**

```typescript
// never permite usar throw en expresiones ternarias
function asertarExiste<T>(valor: T | null | undefined, mensaje: string): T {
  if (valor == null) {
    throw new Error(mensaje);
  }
  return valor;
}

const config = {
  url: "https://api.example.com",
  timeout: asertarExiste(process.env.TIMEOUT, "TIMEOUT requerido")
};

// En arrow functions
const lanzarError = (msg: string): never => {
  throw new Error(msg);
};

const dividir = (a: number, b: number): number => {
  return b === 0 ? lanzarError("División por cero") : a / b;
};
```

===============================================================================
2. EXHAUSTIVENESS CHECKS CON NEVER
===============================================================================

El uso más poderoso de never es para verificar que todos los casos de un
tipo unión han sido cubiertos.

---

📌 **Patrón de exhaustiveness check**

```typescript
type Color = "rojo" | "verde" | "azul";

function obtenerHex(color: Color): string {
  switch (color) {
    case "rojo":
      return "#FF0000";
    case "verde":
      return "#00FF00";
    case "azul":
      return "#0000FF";
    default:
      // Si agregamos un nuevo color, esto causará error de compilación
      const exhaustivo: never = color;
      return exhaustivo;
  }
}
```

---

🔵 **Ejemplo 3: Exhaustiveness check con uniones discriminadas**

```typescript
type Resultado<T> =
  | { exito: true; datos: T }
  | { exito: false; error: string };

function manejarResultado<T>(resultado: Resultado<T>): string {
  if (resultado.exito) {
    return `Éxito: ${JSON.stringify(resultado.datos)}`;
  } else {
    return `Error: ${resultado.error}`;
  }
}

// Si agregamos un nuevo caso a Resultado, el compilador nos alertará
type ResultadoMejorado<T> =
  | { exito: true; datos: T }
  | { exito: false; error: string }
  | { exito: true; datos: T; advertencia: string };  // Nuevo caso

function manejarResultadoMejorado<T>(resultado: ResultadoMejorado<T>): string {
  if (resultado.exito) {
    // TypeScript no sabe si tiene 'advertencia' o no
    return `Éxito: ${JSON.stringify(resultado.datos)}`;
  } else {
    return `Error: ${resultado.error}`;
  }
  // El compilador debería advertir que falta el nuevo caso
}
```

🔵 **Ejemplo 4: Check exhaustivo con aserción never**

```typescript
type EstadoPago = "pendiente" | "procesando" | "completado" | "fallido";

function obtenerEstadoTexto(estado: EstadoPago): string {
  switch (estado) {
    case "pendiente":
      return "Pago pendiente de confirmación";
    case "procesando":
      return "Pago en proceso";
    case "completado":
      return "Pago completado exitosamente";
    case "fallido":
      return "Pago fallido";
    default:
      // Función helper para asegurar exhaustividad
      asercionNunca(estado);
      return "";
  }
}

function asercionNunca(valor: never): never {
  throw new Error(`Valor inesperado: ${valor}`);
}

// Si eliminamos un caso, el compilador mostrará error en asercionNunca
```

🔵 **Ejemplo 5: never para detectar cambios en tipos**

```typescript
// Útil en APIs que evolucionan
type Accion =
  | { tipo: "crear"; datos: object }
  | { tipo: "actualizar"; id: number; datos: object }
  | { tipo: "eliminar"; id: number };

function ejecutarAccion(accion: Accion): void {
  switch (accion.tipo) {
    case "crear":
      console.log("Creando:", accion.datos);
      break;
    case "actualizar":
      console.log("Actualizando:", accion.id);
      break;
    case "eliminar":
      console.log("Eliminando:", accion.id);
      break;
    default:
      const _exhaustivo: never = accion;
      break;
  }
}

// Si en el futuro se agrega "archivar" a Accion,
// el compilador marcará error en _exhaustivo
```

3. NEVER EN TIPOS CONDICIONALES Y GENÉRICOS
===============================================================================

🔵 **Ejemplo 6: never para filtrar tipos**

```typescript
// Filtrar tipos de una unión condicionalmente
type SoloStrings<T> = T extends string ? T : never;

type TiposMezclados = string | number | boolean | symbol;
type SoloTexto = SoloStrings<TiposMezclados>;  // string

// Filtrar propiedades de un objeto
type PropiedadesString<T> = {
  [K in keyof T]: T[K] extends string ? K : never;
}[keyof T];

interface Usuario {
  id: number;
  nombre: string;
  email: string;
  edad: number;
}

type CamposTexto = PropiedadesString<Usuario>;  // "nombre" | "email"
```

🔵 **Ejemplo 7: Exclude y Extract con never**

```typescript
// TypeScript tiene tipos utilitarios que usan never internamente
type T1 = Exclude<string | number | boolean, boolean>;
// Resultado: string | number (excluye boolean)

type T2 = Extract<string | number | boolean, string | number>;
// Resultado: string | number (extrae solo string | number)

type T3 = NonNullable<string | number | null | undefined>;
// Resultado: string | number (excluye null y undefined)

// Implementación simplificada:
// type Exclude<T, U> = T extends U ? never : T;
// type Extract<T, U> = T extends U ? T : never;
// type NonNullable<T> = T extends null | undefined ? never : T;
```

🔵 **Ejemplo 8: never en inferencia de tipos**

```typescript
// never aparece en la inferencia cuando no hay valor
function obtenerOpcional(valor?: string): string | undefined {
  return valor;
}

// Si extraemos el tipo de retorno de una función que lanza error
type RetornoDe<T> = T extends (...args: any[]) => infer R ? R : never;

declare function fn(): never;
type TipoRetorno = RetornoDe<typeof fn>;  // never

// never en arrays filtrados
const numeros = [1, 2, 3, null, 4, null, 5];
const soloNumeros = numeros.filter((n): n is number => n !== null);
// TypeScript infiere: number[]
```

===============================================================================
4. NEVER VS VOID VS UNDEFINED
===============================================================================

Es común confundir never, void y undefined. Veamos sus diferencias
fundamentales.

---

📌 **Comparación: never vs void vs undefined**

```typescript
// void: la función termina pero no retorna valor útil
function soloEfectos(): void {
  console.log("Hago algo y termino");
}

// never: la función nunca termina exitosamente
function siempreFalla(): never {
  throw new Error("Siempre fallo");
}

// undefined: la función retorna el valor undefined
function retornaUndefined(): undefined {
  return undefined;
}

// Asignabilidad
let n: never;
// n = undefined;    // ❌ Error
// n = (() => {})(); // ❌ Error

let v: void;
v = undefined;      // ✅

let u: undefined;
u = undefined;      // ✅

// never está por debajo de todo en la jerarquía
```

---

🔵 **Ejemplo 9: Casos prácticos de cada uno**

```typescript
// never: operación que nunca se completa
function procesarStream(): never {
  while (true) {
    const data = leerStream();
    console.log(data);
  }
}

// void: operación con efecto secundario
function imprimirReporte(datos: object): void {
  console.table(datos);
}

// undefined: función que puede no tener valor
function buscarUsuario(id: number): Usuario | undefined {
  return usuarios.find(u => u.id === id);
}

// Ejemplo combinado
function manejarResultado(resultado: number | null | undefined): number {
  if (resultado === null) {
    throw new Error("Resultado nulo");
  }
  if (resultado === undefined) {
    return 0;  // valor por defecto
  }
  return resultado;
}
```

🔵 **Ejemplo 10: El operador de aserción never**

```typescript
// Aserción de no retorno (TS 3.7+)
function assert(condicion: boolean, mensaje: string): asserts condicion {
  if (!condicion) {
    throw new Error(mensaje);
  }
}

function dividir(a: number, b: number): number {
  assert(b !== 0, "Divisor no puede ser cero");
  return a / b;  // TypeScript sabe que b !== 0 aquí
}

// never en funciones de aserción
interface UsuarioLogeado {
  nombre: string;
  email: string;
  token: string;
}

function requiereAutenticacion(usuario: UsuarioLogeado | null): asserts usuario is UsuarioLogeado {
  if (usuario === null) {
    throw new Error("Autenticación requerida");
  }
}

function obtenerPerfil(usuario: UsuarioLogeado | null): string {
  requiereAutenticacion(usuario);
  return `Perfil de ${usuario.nombre}`;  // usuario es UsuarioLogeado
}
```

5. PATRONES AVANZADOS CON NEVER
===============================================================================

🔵 **Ejemplo 11: never para eliminar casos de uniones**

```typescript
// Eliminar ciertos tipos de una unión
type SinNull<T> = T extends null | undefined ? never : T;

type Valor = string | number | null | undefined;
type ValorLimpio = SinNull<Valor>;  // string | number

// Función que elimina null y undefined de un array
function limpiarArray<T>(items: (T | null | undefined)[]): SinNull<T>[] {
  return items.filter((item): item is SinNull<T> => item != null);
}

const mixto = [1, null, "hola", undefined, true];
const limpio = limpiarArray(mixto);  // (string | number | boolean)[]
console.log(limpio);  // [1, "hola", true]
```

🔵 **Ejemplo 12: never en tipos mapeados**

```typescript
// Omitir propiedades que cumplen cierta condición
type OmitirFunciones<T> = {
  [K in keyof T]: T[K] extends Function ? never : K;
}[keyof T];

type SoloPropiedades<T> = Pick<T, OmitirFunciones<T>>;

interface Servicio {
  id: number;
  nombre: string;
  iniciar(): void;
  detener(): void;
  estado: string;
}

type PropsServicio = SoloPropiedades<Servicio>;
// { id: number; nombre: string; estado: string }
```

🔵 **Ejemplo 13: never en reducers y state management**

```typescript
type AccionApp =
  | { tipo: "INCREMENTAR" }
  | { tipo: "DECREMENTAR" }
  | { tipo: "REINICIAR" }
  | { tipo: "ESTABLECER"; valor: number };

interface EstadoApp {
  contador: number;
}

function reducer(estado: EstadoApp, accion: AccionApp): EstadoApp {
  switch (accion.tipo) {
    case "INCREMENTAR":
      return { contador: estado.contador + 1 };
    case "DECREMENTAR":
      return { contador: estado.contador - 1 };
    case "REINICIAR":
      return { contador: 0 };
    case "ESTABLECER":
      return { contador: accion.valor };
    default:
      const _exhaustivo: never = accion;
      return _exhaustivo;
  }
}
```

===============================================================================
RESUMEN DEL CAPÍTULO
===============================================================================

- never es el tipo que representa valores que nunca pueden ocurrir.
  Ocupa el fondo de la jerarquía de tipos.
- Las funciones que retornan never son aquellas que siempre lanzan
  excepciones o entran en bucles infinitos.
- El patrón de exhaustiveness check usa never en el default de un switch
  para asegurar que todos los casos de una unión están cubiertos.
- En tipos condicionales, never se usa para filtrar tipos (como en
  Exclude, Extract, NonNullable).
- never, void y undefined son distintos: void indica terminación sin
  valor, undefined es un valor específico, never indica que algo
  nunca ocurre.
- El checker exhaustivo es una herramienta poderosa para mantener
  la seguridad cuando los tipos evolucionan.

===============================================================================
PREGUNTAS FRECUENTES
===============================================================================

❓ **¿Cuándo se usa never en lugar de void?**
Usa never cuando la función nunca completa su ejecución (bucle infinito)
o nunca retorna exitosamente (lanza excepción). Usa void cuando la función
completa pero no produce un valor útil de retorno. La diferencia clave:
void termina, never no termina.

❓ **¿Puedo asignar un valor a una variable never?**
No, excepto otro valor never. Una variable never no puede tener un valor
en runtime. Si intentas asignarle cualquier valor, TypeScript mostrará
un error. Esto es intencional: never representa lo imposible.

❓ **¿Debo siempre usar exhaustiveness checks?**
No es obligatorio, pero es una buena práctica para tipos unión que
evolucionan. El patrón asegura que cuando agregas un nuevo caso al tipo,
el compilador te obliga a manejarlo en todos los switch/if relevantes,
evitando errores silenciosos.

❓ **¿Cómo aparece never en el código sin que yo lo declare?**
TypeScript infiere never en varias situaciones: ramas de código
inalcanzables, tipos condicionales que excluyen todos los casos,
arrays vacíos inferidos como never[], y funciones que lanzan errores
o tienen bucles infinitos.
