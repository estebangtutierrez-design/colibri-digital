===============================================================================
1. ¿QUÉ SON LOS MÓDULOS ES (ECMASCRIPT MODULES)?
===============================================================================

Los Módulos ES (ESM) son el sistema de módulos nativo de JavaScript,
estandarizado en ECMAScript 2015 (ES6). A diferencia de sistemas anteriores
(CommonJS, AMD), los módulos ES son parte del lenguaje y no requieren
librerías externas.

```typescript
// ESM es la sintaxis que hemos estado usando:
export const mensaje = "Hola";
import { mensaje } from "./modulo.js";
```

Características clave de ESM:
- Estáticos: las dependencias se resuelven en tiempo de compilación
- Asíncronos: soporte nativo en navegadores con carga diferida
- Named exports y default exports
- Top-level await (awaits en el módulo sin async function)
- Tree-shaking natural (los bundlers eliminan exports no usados)
- Strict mode por defecto

```typescript
// Top-level await (solo en ESM)
const respuesta = await fetch("https://api.ejemplo.com/datos");
export const datos = await respuesta.json();
```

Para que TypeScript genere ESM, debemos configurarlo apropiadamente:

```json
{
  "compilerOptions": {
    "module": "ESNext",
    "moduleResolution": "bundler",
    "target": "ES2022"
  }
}
```

===============================================================================
2. CONFIGURACIÓN DE TYPESCRIPT PARA ESM
===============================================================================

Para usar ES Modules, debemos ajustar tanto tsconfig.json como package.json.

```json
// ----- tsconfig.json -----
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "ESNext",
    "moduleResolution": "bundler",
    "outDir": "./dist",
    "rootDir": "./src",
    "strict": true,
    "esModuleInterop": true,
    "declaration": true,
    "sourceMap": true
  },
  "include": ["src"]
}
```

```json
// ----- package.json -----
{
  "name": "mi-proyecto",
  "version": "1.0.0",
  "type": "module",
  "scripts": {
    "build": "tsc",
    "start": "node dist/main.js"
  },
  "devDependencies": {
    "typescript": "^5.4.0"
  }
}
```

La clave `"type": "module"` en package.json le dice a Node.js que los
archivos .js del proyecto son ES Modules, no CommonJS.

Si no quieres usar `type: "module"` globalmente, puedes usar la extensión
`.mjs` para archivos ESM y `.cjs` para CommonJS. TypeScript soporta las
extensiones `.mts` (que compila a .mjs) y `.cts` (que compila a .cjs).

```typescript
// ----- archivo: main.mts -----
import { sumar } from "./operaciones.mjs";
console.log(sumar(5, 3));

// Compila a main.mjs
```

```json
{
  "compilerOptions": {
    "module": "ESNext",
    "outDir": "./dist"
  }
}
// tsc generará main.mjs a partir de main.mts
```

===============================================================================
3. IMPORT/EXPORT NATIVO EN JAVASCRIPT
===============================================================================

Cuando TypeScript compila con `module: "ESNext"`, el código resultante
conserva la sintaxis import/export nativa.

```typescript
// ----- Código TypeScript (src/main.ts) -----
import { sumar, PI } from "./matematicas.js";
import Config from "./config.js";
export function iniciar(): void {
  console.log("App iniciada");
  console.log(sumar(5, PI));
}
```

```javascript
// ----- Código JavaScript generado (dist/main.js) -----
import { sumar, PI } from "./matematicas.js";
import Config from "./config.js";
export function iniciar() {
  console.log("App iniciada");
  console.log(sumar(5, PI));
}
```

El código generado es prácticamente idéntico, solo se eliminan los tipos.

Las rutas en ESM deben incluir la extensión del archivo:

```typescript
// ✅ Correcto en ESM
import { algo } from "./modulo.js";

// ❌ Incorrecto en ESM (TypeScript acepta ./modulo, pero el JS generado no)
import { algo } from "./modulo";
```

TypeScript permite omitir la extensión en los imports, pero el archivo .js
generado también la omitirá, lo que causará errores en Node.js ESM.

Para solucionarlo, podemos usar `moduleResolution: "bundler"` o configurar
el servidor de desarrollo para que resuelva sin extensiones.

```json
{
  "compilerOptions": {
    "moduleResolution": "bundler"
  }
}
```

===============================================================================
4. DIFERENCIAS ENTRE ESM Y COMMONJS
===============================================================================

| Aspecto              | ES Modules (ESM)        | CommonJS (CJS)         |
|----------------------|-------------------------|------------------------|
| Sintaxis             | import/export           | require/module.exports |
| Carga                | Estática                | Dinámica               |
| Ámbito               | Archivo                 | Archivo                |
| Top-level await      | Sí                      | No                     |
| Tree-shaking         | Sí                      | No                     |
| Strict mode          | Por defecto             | Opcional               |
| Extensión            | .mjs / .js (type:module)| .cjs / .js (type:commonjs)|
| Resolución circular  | Mejor soporte           | Puede dar undefined    |
| Tiempo de evaluación | En paralelo             | En orden               |

```typescript
// ESM - estático
import { sumar } from "./matematicas.js";
export const resultado = sumar(5, 3);
```

```typescript
// CommonJS - dinámico
const { sumar } = require("./matematicas");
module.exports.resultado = sumar(5, 3);
```

En ESM, los imports son "live bindings": si el valor exportado cambia, el
módulo importador ve el cambio.

```typescript
// ----- contador.ts -----
export let contador = 0;
export function incrementar(): void {
  contador++;
}

// ----- main.ts -----
import { contador, incrementar } from "./contador.js";
console.log(contador); // 0
incrementar();
console.log(contador); // 1 (live binding)
```

En CommonJS, el valor se copia en el momento del require.

===============================================================================
5. INTEROPERABILIDAD ESM Y COMMONJS
===============================================================================

A veces necesitamos importar módulos CommonJS desde un proyecto ESM o
viceversa. TypeScript y Node.js ofrecen mecanismos para esto.

```typescript
// Importar CommonJS desde ESM
import ci from "modulo-commonjs";
// Con esModuleInterop: true, funciona sin problemas
```

```json
{
  "compilerOptions": {
    "esModuleInterop": true,
    "allowSyntheticDefaultImports": true
  }
}
```

Para importar ESM desde CommonJS, podemos usar import dinámico:

```javascript
// CommonJS importando ESM
async function main() {
  const moduloES = await import("./modulo-es.mjs");
  console.log(moduloES.exportacion);
}
main();
```

O podemos usar la función createRequire desde ESM:

```typescript
// ESM usando require (con createRequire)
import { createRequire } from "module";
const require = createRequire(import.meta.url);
const lodash = require("lodash");
```

Node.js soporta ambas extensiones:

```json
{
  "type": "module"  // .js se trata como ESM
}
// Pero archivos .cjs siempre son CommonJS
// Y archivos .mjs siempre son ESM
```

TypeScript genera código ES modules, pero podemos hacer que Node.js lo
ejecute configurando `"type": "module"` en package.json.

===============================================================================
6. TREE-SHAKING CON ES MODULES
===============================================================================

Una de las mayores ventajas de ESM es el tree-shaking: los bundlers (Webpack,
Vite, Rollup) pueden eliminar el código no utilizado, reduciendo el tamaño
del bundle.

```typescript
// ----- libreria.ts -----
export function funcionUsada(): string {
  return "usada";
}

export function funcionNoUsada(): string {
  return "no usada";
}

export const constanteUsada = "valor";
export const constanteNoUsada = "otro";
```

```typescript
// ----- main.ts -----
import { funcionUsada, constanteUsada } from "./libreria.js";
console.log(funcionUsada(), constanteUsada);
```

Con tree-shaking, funcionNoUsada y constanteNoUsada no aparecerán en el
bundle final.

Para que el tree-shaking funcione correctamente:
- Usar ESM (no CommonJS)
- No usar efectos secundarios en módulos (side effects)
- Preferir named exports sobre default exports
- Configurar "sideEffects": false en package.json

```json
{
  "name": "mi-libreria",
  "version": "1.0.0",
  "sideEffects": false
}
```

Si tienes archivos con efectos secundarios, indícalos explícitamente:

```json
{
  "sideEffects": [
    "./src/polyfills.ts",
    "*.css"
  ]
}
```

7. PUBLICAR LIBRERÍAS COMO ES MODULES
===============================================================================

Si publicas una librería TypeScript en npm, debes configurarla para que
soporte tanto ESM como CommonJS (dual package).

```json
// package.json para librería dual
{
  "name": "mi-libreria",
  "version": "1.0.0",
  "type": "module",
  "main": "./dist/cjs/index.js",
  "module": "./dist/esm/index.js",
  "types": "./dist/types/index.d.ts",
  "exports": {
    ".": {
      "import": "./dist/esm/index.js",
      "require": "./dist/cjs/index.js",
      "types": "./dist/types/index.d.ts"
    },
    "./*": {
      "import": "./dist/esm/*.js",
      "require": "./dist/cjs/*.js",
      "types": "./dist/types/*.d.ts"
    }
  }
}
```

Configuración de tsconfig para ESM y CJS:

```json
// tsconfig.esm.json
{
  "extends": "./tsconfig.base.json",
  "compilerOptions": {
    "module": "ESNext",
    "outDir": "./dist/esm"
  }
}
```

```json
// tsconfig.cjs.json
{
  "extends": "./tsconfig.base.json",
  "compilerOptions": {
    "module": "CommonJS",
    "outDir": "./dist/cjs"
  }
}
```

```json
// tsconfig.base.json
{
  "compilerOptions": {
    "target": "ES2020",
    "strict": true,
    "declaration": true,
    "declarationDir": "./dist/types",
    "esModuleInterop": true
  },
  "include": ["src"]
}
```

Script de build en package.json:

```json
{
  "scripts": {
    "build": "tsc -p tsconfig.esm.json && tsc -p tsconfig.cjs.json"
  }
}
```

RESUMEN DEL CAPÍTULO
===============================================================================

- ES Modules (ESM) es el sistema de módulos nativo de JavaScript.
- TypeScript genera código ESM con module: "ESNext" en tsconfig.json.
- package.json necesita "type": "module" para que Node.js trate .js como ESM.
- ESM soporta tree-shaking, live bindings y top-level await.
- La interoperabilidad ESM/CJS se logra con esModuleInterop y createRequire.
- Las librerías pueden publicarse como duales (ESM + CJS) con exports.
- Las extensiones .mts/.mjs y .cts/.cjs ayudan a diferenciar sistemas.

===============================================================================
PREGUNTAS FRECUENTES
===============================================================================

❓ **¿Necesito "type": "module" en package.json?**
Si quieres que Node.js trate los .js como ESM, sí. Sin él, Node.js tratará
los .js como CommonJS por defecto.

❓ **¿Por qué debo poner .js en imports si mi archivo es .ts?**
Porque TypeScript genera archivos .js, y el código JavaScript generado debe
tener rutas válidas para el runtime. Es una regla de ESM.

❓ **¿Qué es moduleResolution: "bundler"?**
Es una estrategia de resolución para proyectos que usan bundlers (Vite,
Webpack). Permite omitir extensiones y resuelve más flexiblemente.

❓ **¿Los ES Modules son más lentos que CommonJS?**
En carga inicial, ESM puede ser más lento por la resolución estática, pero
ofrece mejor rendimiento a largo plazo con tree-shaking y carga asíncrona.

❓ **¿Puedo usar require en un proyecto ESM?**
No directamente. Puedes usar createRequire de 'module' o import() dinámico.
