===============================================================================
1. ¿QUÉ ES TSCONFIG.JSON?
===============================================================================

tsconfig.json es el archivo de configuración del compilador de TypeScript
(tsc). Indica al compilador cómo procesar los archivos .ts del proyecto.

```json
{
  "compilerOptions": {
    "target": "ES2020",
    "module": "ESNext",
    "strict": true,
    "outDir": "./dist",
    "rootDir": "./src"
  },
  "include": ["src/**/*"],
  "exclude": ["node_modules", "dist"]
}
```

Para inicializar un proyecto con tsconfig.json:

```bash
tsc --init
```

Este comando genera un archivo con todas las opciones comentadas.

El tsconfig.json puede ubicarse en la raíz del proyecto o en subdirectorios.
TypeScript lo buscará automáticamente al ejecutar tsc.

```bash
# Compilar usando el tsconfig.json de la raíz
tsc

# Compilar usando un tsconfig específico
tsc --project tsconfig.prod.json
```

===============================================================================
2. OPCIONES DE COMPILADOR (COMPILEROPTIONS)
===============================================================================

Las opciones del compilador se definen dentro de `compilerOptions`. Veamos
las más importantes agrupadas por categoría.

```json
{
  "compilerOptions": {
    // === Opciones básicas ===
    "target": "ES2020",
    "module": "ESNext",
    "lib": ["ES2020", "DOM", "DOM.Iterable"],
    "jsx": "react-jsx",
    "outDir": "./dist",
    "rootDir": "./src",
    "declaration": true,
    "sourceMap": true,
    "removeComments": false,
    "noEmit": false,
    "importHelpers": true,

    // === Opciones estrictas ===
    "strict": true,
    "noImplicitAny": true,
    "strictNullChecks": true,
    "noUnusedLocals": true,
    "noUnusedParameters": true,
    "exactOptionalPropertyTypes": true,
    "noImplicitReturns": true,
    "noFallthroughCasesInSwitch": true,

    // === Resolución de módulos ===
    "moduleResolution": "node",
    "baseUrl": ".",
    "paths": {
      "@/*": ["src/*"]
    },
    "esModuleInterop": true,
    "allowSyntheticDefaultImports": true,
    "resolveJsonModule": true,
    "allowJs": true,
    "checkJs": false,

    // === Salida y generación ===
    "outFile": "",
    "declarationDir": "./types",
    "declarationMap": true,
    "downlevelIteration": true,
    "preserveConstEnums": true
  }
}
```

Cada opción tiene un propósito específico que exploraremos a continuación.

**target**: Define la versión de JavaScript de salida.

```json
"target": "ES5"      // Para navegadores antiguos
"target": "ES2015"   // ES6
"target": "ES2020"   // Moderno
"target": "ESNext"   // Última versión
```

**module**: Define el sistema de módulos de salida.

```json
"module": "commonjs"  // Node.js tradicional
"module": "ESNext"    // ES Modules moderno
"module": "AMD"       // RequireJS
"module": "UMD"       // Universal
"module": "System"    // SystemJS
"module": "NodeNext"  // Node.js con ES Modules
```

**lib**: Especifica las librerías de entorno incluidas.

```json
"lib": ["ES2020"]                    // Solo JS moderno
"lib": ["ES2020", "DOM"]             // Para navegador
"lib": ["ES2020", "DOM", "DOM.Iterable"]  // Con iterables DOM
"lib": ["ESNext", "DOM"]             // Últimas características
```

===============================================================================
3. OPCIONES DE ENTRADA Y SALIDA
===============================================================================

Controlan dónde el compilador busca archivos y dónde coloca la salida.

```json
{
  "compilerOptions": {
    "rootDir": "./src",
    "outDir": "./dist",
    "declaration": true,
    "declarationDir": "./types",
    "sourceRoot": "./src",
    "mapRoot": "./maps",
    "outFile": "./bundle.js"
  }
}
```

```typescript
// rootDir: ./src
// outDir: ./dist
// src/main.ts -> dist/main.js
// src/utils/helper.ts -> dist/utils/helper.js

// declaration: true genera archivos .d.ts
// src/main.ts -> dist/main.js y dist/main.d.ts

// declarationDir: "./types"
// src/main.ts -> types/main.d.ts
```

**noEmit**: Compila sin generar archivos (solo verifica tipos).

```json
{
  "compilerOptions": {
    "noEmit": true
  }
}
```

Útil para linters o herramientas que solo necesitan verificación de tipos.

**declaration**: Genera archivos .d.ts (declaraciones de tipos).

```json
{
  "compilerOptions": {
    "declaration": true,
    "declarationMap": true
  }
}
```

Esencial para librerías que otros consumirán.

===============================================================================
4. MODO ESTRICTO (STRICT MODE)
===============================================================================

El modo estricto activa un conjunto de reglas que hacen el código más seguro.

```json
{
  "compilerOptions": {
    "strict": true
  }
}
```

`strict: true` activa simultáneamente estas opciones:

- **noImplicitAny**: Error si TypeScript infiere `any` implícitamente
- **strictNullChecks**: `null` y `undefined` no son asignables a otros tipos
- **strictFunctionTypes**: Verificación estricta de tipos en funciones
- **strictBindCallApply**: Verificación estricta en bind/call/apply
- **strictPropertyInitialization**: Propiedades de clase deben inicializarse
- **noImplicitThis**: Error si `this` tiene tipo any implícito
- **alwaysStrict**: "use strict" en todo el JS generado

```typescript
// Con strict: true

// strictNullChecks evita esto:
const x: string = null; // Error!

// noImplicitAny evita esto:
function procesar(dato) { // Error: parameter implicitly has 'any' type
  return dato;
}

// strictPropertyInitialization exige inicializar propiedades:
class Usuario {
  nombre: string; // Error si strict: true
  // Solución:
  // nombre!: string; (non-null assertion)
  // o inicializar en constructor
  constructor(nombre: string) {
    this.nombre = nombre;
  }
}
```

Puedes activar reglas individuales sin strict completo:

```json
{
  "compilerOptions": {
    "strict": false,
    "strictNullChecks": true,
    "noImplicitAny": true
  }
}
```

Pero la recomendación es siempre usar `strict: true` en proyectos nuevos.

===============================================================================
5. RESOLUCIÓN DE MÓDULOS
===============================================================================

Controla cómo TypeScript resuelve las rutas de los módulos.

```json
{
  "compilerOptions": {
    "moduleResolution": "node",
    "baseUrl": ".",
    "paths": {
      "@components/*": ["src/components/*"],
      "@utils/*": ["src/utils/*"],
      "@types/*": ["src/types/*"]
    },
    "esModuleInterop": true,
    "allowSyntheticDefaultImports": true,
    "resolveJsonModule": true
  }
}
```

**moduleResolution**: Estrategia de resolución.

```json
"moduleResolution": "node"         // Estilo Node.js (más común)
"moduleResolution": "classic"      // Legacy TypeScript
"moduleResolution": "nodenext"     // Para Node.js con ES Modules
"moduleResolution": "bundler"      // Para bundlers (Vite, Webpack)
```

**paths**: Alias de rutas (requiere baseUrl).

```json
{
  "compilerOptions": {
    "baseUrl": ".",
    "paths": {
      "@/*": ["src/*"],
      "@components/*": ["src/components/*"],
      "@utils/*": ["src/utils/*"]
    }
  }
}
```

Esto permite escribir:

```typescript
// En lugar de: import { Boton } from "../../components/Boton"
import { Boton } from "@components/Boton";
```

**rootDirs**: Múltiples directorios raíz virtuales.

```json
{
  "compilerOptions": {
    "rootDirs": ["src", "generated"]
  }
}
```

Permite que archivos en diferentes carpetas compartan rutas relativas.

===============================================================================
6. INCLUDE, EXCLUDE Y FILES
===============================================================================

Controlan qué archivos incluye el compilador.

```json
{
  "include": [
    "src/**/*",
    "tests/**/*",
    "types/**/*.d.ts"
  ],
  "exclude": [
    "node_modules",
    "dist",
    "**/*.spec.ts",
    "**/*.test.ts",
    "coverage"
  ],
  "files": [
    "src/main.ts",
    "src/utils/helper.ts"
  ]
}
```

- **include**: Glob patterns de archivos a incluir (por defecto: **/*)
- **exclude**: Glob patterns a excluir (node_modules se excluye por defecto)
- **files**: Lista explícita de archivos (útil para proyectos pequeños)

```json
{
  "include": ["src"],
  "exclude": ["src/legacy"]
}
```

Si se usa `files`, solo esos archivos se compilan:

```json
{
  "files": ["core.ts", "types.ts", "main.ts"]
}
```

El orden de prioridad es: `files` > `include`/`exclude`. Si un archivo está
en `files`, se compila independientemente de `exclude`.

===============================================================================
7. EXTENDS: HERENCIA DE CONFIGURACIONES
===============================================================================

Permite extender la configuración de otro tsconfig.json.

```json
// ---------- tsconfig.base.json ----------
{
  "compilerOptions": {
    "target": "ES2020",
    "module": "ESNext",
    "strict": true,
    "esModuleInterop": true
  }
}
```

```json
// ---------- tsconfig.json (raíz) ----------
{
  "extends": "./tsconfig.base.json",
  "compilerOptions": {
    "outDir": "./dist",
    "rootDir": "./src"
  },
  "include": ["src"]
}
```

```json
// ---------- tsconfig.prod.json ----------
{
  "extends": "./tsconfig.base.json",
  "compilerOptions": {
    "outDir": "./dist-prod",
    "rootDir": "./src",
    "removeComments": true,
    "sourceMap": false
  },
  "exclude": ["src/**/*.test.ts", "src/**/__mocks__/**"]
}
```

Se puede extender de paquetes npm:

```json
{
  "extends": "@tsconfig/node20/tsconfig.json",
  "compilerOptions": {
    "outDir": "./dist"
  }
}
```

Paquetes populares: `@tsconfig/node20`, `@tsconfig/strictest`, `@tsconfig/vite`.

8. OPCIONES AVANZADAS Y DE DEPURACIÓN
===============================================================================

```json
{
  "compilerOptions": {
    // === Debug ===
    "diagnostics": true,
    "extendedDiagnostics": true,
    "generateCpuProfile": "profile.cpuprofile",
    "listFiles": true,
    "listEmittedFiles": true,
    "traceResolution": true,

    // === Proyectos de referencia ===
    "composite": true,
    "incremental": true,
    "tsBuildInfoFile": "./.tsbuildinfo",

    // === Optimización ===
    "skipLibCheck": true,
    "forceConsistentCasingInFileNames": true,
    "isolatedModules": true,
    "pretty": true
  }
}
```

**composite** e **incremental**: Aceleran compilaciones en proyectos grandes.

```json
{
  "compilerOptions": {
    "composite": true,
    "incremental": true,
    "tsBuildInfoFile": "./.tsbuildinfo"
  }
}
```

**skipLibCheck**: Omite la verificación de tipos en archivos .d.ts.

```json
"skipLibCheck": true
```

Muy útil en proyectos con muchas dependencias.

**forceConsistentCasingInFileNames**: Fuerza consistencia en mayúsculas.

```json
"forceConsistentCasingInFileNames": true
```

Previene errores en sistemas de archivos case-insensitive (Windows/Mac).

RESUMEN DEL CAPÍTULO
===============================================================================

- tsconfig.json es el archivo de configuración del compilador TypeScript.
- Los ajustes principales incluyen target, module, strict, outDir, rootDir.
- strict: true activa múltiples reglas de seguridad de tipos.
- moduleResolution controla cómo se resuelven las rutas de importación.
- include/exclude/files controlan qué archivos se compilan.
- extends permite heredar y componer configuraciones.
- Opciones como paths y baseUrl permiten crear alias de rutas.
- skipLibCheck acelera la compilación en proyectos grandes.

===============================================================================
PREGUNTAS FRECUENTES
===============================================================================

❓ **¿Qué significa strict: true?**
Activa todas las reglas estrictas de TypeScript (noImplicitAny,
strictNullChecks, etc.) para mayor seguridad de tipos.

❓ **¿Cuál es la diferencia entre module y target?**
target define la versión JS de salida (ES5, ES2020, etc.). module define el
sistema de módulos (commonjs, esnext, etc.).

❓ **¿Paths requiere baseUrl?**
Sí, paths solo funciona si se define baseUrl como referencia.

❓ **¿Puedo tener múltiples tsconfig.json en un proyecto?**
Sí. Puedes tener uno por entorno (tsconfig.json, tsconfig.prod.json,
tsconfig.test.json) y combinarlos con extends.

❓ **¿Qué hace skipLibCheck?**
Omite la verificación de tipos en archivos .d.ts, acelerando la compilación
sin sacrificar la verificación del código propio.
