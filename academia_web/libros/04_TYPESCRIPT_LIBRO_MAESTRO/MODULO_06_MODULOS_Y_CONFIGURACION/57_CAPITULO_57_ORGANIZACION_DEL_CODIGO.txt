1. PRINCIPIOS DE ORGANIZACIÓN DE PROYECTOS
===============================================================================

Antes de ver estructuras concretas, entendamos los principios que guían
una buena organización:

```typescript
// Principio 1: Separación de responsabilidades
// Cada archivo debe tener un propósito claro y único.

// ❌ MAL: Un archivo que hace de todo
export function validarEmail(email: string): boolean { ... }
export function enviarEmail(destino: string, asunto: string): void { ... }
export function formatearEmail(nombre: string, dominio: string): string { ... }

// ✅ BIEN: Separar en archivos especializados
// validacion.ts -> validarEmail
// email.service.ts -> enviarEmail
// email.utils.ts -> formatearEmail
```

```typescript
// Principio 2: Cohesión y acoplamiento
// - Alta cohesión: elementos relacionados están juntos
// - Bajo acoplamiento: módulos dependen poco entre sí

// Principio 3: Consistencia
// - Mismos patrones de nombrado en todo el proyecto
// - Misma estructura de carpetas en todos los módulos
```

```typescript
// Principio 4: Escalabilidad
// La estructura debe funcionar tanto con 10 como con 1000 archivos.
// Principio 5: Navegabilidad
// Un nuevo desarrollador debe entender la estructura en minutos.
```

2. ESTRUCTURA POR CAPAS (LAYERED ARCHITECTURE)
===============================================================================

Una de las formas más comunes de organizar proyectos, especialmente en
aplicaciones backend con Node.js.

```
src/
├── controllers/       # Manejan peticiones HTTP
│   ├── usuario.controller.ts
│   └── producto.controller.ts
├── services/          # Lógica de negocio
│   ├── usuario.service.ts
│   └── producto.service.ts
├── repositories/      # Acceso a datos (base de datos, APIs)
│   ├── usuario.repository.ts
│   └── producto.repository.ts
├── models/            # Definiciones de tipos/modelos
│   ├── usuario.model.ts
│   └── producto.model.ts
├── middleware/         # Middleware Express/fastify
│   ├── auth.middleware.ts
│   └── logger.middleware.ts
├── utils/             # Utilidades generales
│   ├── validators.ts
│   └── formatters.ts
├── config/            # Configuración
│   └── index.ts
├── routes/            # Definición de rutas
│   ├── index.ts
│   ├── usuario.routes.ts
│   └── producto.routes.ts
├── types/             # Tipos globales
│   ├── express.d.ts
│   └── global.d.ts
└── main.ts            # Punto de entrada
```

```typescript
// Ejemplo de flujo por capas:

// controller recibe la petición
// controller llama a service
// service llama a repository
// repository accede a la base de datos

// usuario.controller.ts
import { Router } from "express";
import { UsuarioService } from "../services/usuario.service";

const router = Router();
const usuarioService = new UsuarioService();

router.get("/:id", async (req, res) => {
  const usuario = await usuarioService.obtenerPorId(req.params.id);
  res.json(usuario);
});

export default router;
```

Ventajas: separación clara, fácil de entender, testing individual por capa.
Desventajas: puede volverse rígida, muchas carpetas para proyectos pequeños.

3. ESTRUCTURA POR FUNCIONALIDADES (FEATURE FOLDERS)
===============================================================================

Cada funcionalidad o módulo del negocio tiene su propia carpeta con todo
lo que necesita (controlador, servicio, modelo, pruebas).

```
src/
├── usuarios/                  # Funcionalidad: Usuarios
│   ├── usuario.controller.ts
│   ├── usuario.service.ts
│   ├── usuario.repository.ts
│   ├── usuario.model.ts
│   ├── usuario.routes.ts
│   ├── usuario.test.ts
│   └── index.ts               # Barrel file
├── productos/                 # Funcionalidad: Productos
│   ├── producto.controller.ts
│   ├── producto.service.ts
│   ├── producto.repository.ts
│   ├── producto.model.ts
│   ├── producto.routes.ts
│   ├── producto.test.ts
│   └── index.ts
├── shared/                    # Código compartido
│   ├── middleware/
│   ├── utils/
│   └── types/
├── config/
├── app.ts                     # Configuración de la app
└── server.ts                  # Punto de entrada
```

```typescript
// index.ts (barrel file de usuarios)
export { UsuarioController } from "./usuario.controller";
export { UsuarioService } from "./usuario.service";
export { UsuarioRepository } from "./usuario.repository";
export type { Usuario, CrearUsuarioDTO } from "./usuario.model";
export { default as usuarioRouter } from "./usuario.routes";
```

```typescript
// Import desde fuera del módulo
import { UsuarioService, type Usuario } from "../usuarios";
```

Ventajas: encapsulamiento, fácil de mover/eliminar funcionalidades,
escalable, cada feature es independiente.

Desventajas: puede haber duplicación de código entre features si no se
comparte bien.

===============================================================================
4. BARREL FILES (INDEX.TS)
===============================================================================

Los barrel files son archivos index.ts que re-exportan el contenido de un
directorio, creando un punto de entrada único.

```
src/
├── utils/
│   ├── index.ts              # Re-exporta todo
│   ├── validators.ts
│   ├── formatters.ts
│   └── helpers.ts
```

```typescript
// ----- src/utils/index.ts -----
export { validarEmail, validarTelefono } from "./validators";
export { formatearFecha, formatearMoneda } from "./formatters";
export { sleep, retry } from "./helpers";
export type { ValidacionResultado } from "./validators";
```

```typescript
// ----- main.ts: import limpio -----
import { validarEmail, formatearFecha, sleep } from "./utils";
// No: import { validarEmail } from "./utils/validators";
```

Buenas prácticas con barrel files:

```typescript
// 1. No re-exportar todo ciegamente
export * from "./modulo"; // ❌ Puede causar colisiones

// 2. Re-exportar selectivamente
export { algo } from "./modulo"; // ✅ Controlado

// 3. Separar tipos y valores
export type { MiTipo } from "./modulo";
export { miFuncion } from "./modulo";

// 4. Usar namespaces en re-exportaciones
export * as Utils from "./utils"; // import { Utils } from "./index";
```

Los barrel files mejoran la legibilidad de los imports y centralizan
la API pública de cada módulo.

5. ORGANIZACIÓN DE TIPOS E INTERFACES
===============================================================================

Hay diferentes estrategias para organizar los tipos en TypeScript:

```
src/
├── types/                     # Tipos globales
│   ├── index.ts
│   ├── api.types.ts           # Tipos relacionados con API
│   ├── common.types.ts        # Tipos comunes
│   └── env.d.ts               # Tipos de variables de entorno
├── shared/
│   └── models/
│       ├── index.ts
│       ├── usuario.model.ts   # Interfaces de Usuario
│       └── producto.model.ts  # Interfaces de Producto
├── usuarios/
│   ├── usuario.model.ts       # Tipos específicos de usuario
│   ├── usuario.service.ts
│   └── index.ts
```

Estrategias:

**Opción 1: Tipos junto al feature (recomendada para proyectos medianos)**

```typescript
// usuarios/usuario.model.ts
export interface Usuario {
  id: number;
  nombre: string;
  email: string;
  rol: "admin" | "usuario";
}

export type CrearUsuarioDTO = Omit<Usuario, "id">;

export type ActualizarUsuarioDTO = Partial<CrearUsuarioDTO>;
```

**Opción 2: Tipos centralizados (recomendada para proyectos grandes o librerías)**

```typescript
// types/usuario.types.ts
export interface Usuario {
  id: number;
  nombre: string;
  email: string;
}

export interface UsuarioConRol extends Usuario {
  rol: "admin" | "usuario";
}
```

Regla: si un tipo se usa en múltiples features, ponlo en types/ compartido.
Si es específico de una feature, mantenlo ahí.

6. ORGANIZACIÓN DE TESTS
===============================================================================

Dos enfoques principales para organizar tests en TypeScript:

**Enfoque 1: Tests junto al código (co-located)**

```
src/
├── usuarios/
│   ├── usuario.service.ts
│   ├── usuario.service.test.ts   # Test junto al fuente
│   ├── usuario.model.ts
│   └── index.ts
```

**Enfoque 2: Tests en directorio separado**

```
src/
├── usuarios/
│   ├── usuario.service.ts
│   └── usuario.model.ts
tests/
├── usuarios/
│   └── usuario.service.test.ts
```

El enfoque co-located (junto al código) es cada vez más popular porque:

- Los tests están cerca del código que prueban
- Es fácil ver si un archivo tiene tests
- Los imports son más simples
- Al mover una feature, los tests se mueven con ella

```typescript
// usuarios/usuario.service.test.ts
import { describe, it, expect } from "vitest";
import { UsuarioService } from "./usuario.service";

describe("UsuarioService", () => {
  const service = new UsuarioService();

  it("debe crear un usuario correctamente", () => {
    const usuario = service.crear({ nombre: "Ana", email: "ana@test.com" });
    expect(usuario).toHaveProperty("id");
    expect(usuario.nombre).toBe("Ana");
  });
});
```

Configuración de exclusión en tsconfig.json para no compilar tests:

```json
{
  "exclude": ["node_modules", "dist", "**/*.test.ts", "**/*.spec.ts"]
}
```

7. PATRONES DE NOMBRADO DE ARCHIVOS
===============================================================================

La consistencia en el nombrado es crucial para la navegabilidad.

```
# Convenciones comunes:

usuario.service.ts        # Servicios
usuario.controller.ts     # Controladores
usuario.repository.ts     # Repositorios
usuario.model.ts          # Modelos/Interfaces
usuario.routes.ts         # Rutas
usuario.middleware.ts     # Middleware
usuario.test.ts           # Tests unitarios
usuario.e2e.test.ts       # Tests end-to-end
usuario.dto.ts            # Data Transfer Objects
usuario.factory.ts        # Factories
usuario.config.ts         # Configuración específica
usuario.types.ts          # Tipos específicos
index.ts                  # Barrel file
```

Reglas:
- Usar kebab-case para archivos (usuario.service.ts)
- Usar PascalCase para clases/interfaces/tipos (UsuarioService)
- Usar camelCase para funciones/variables (obtenerUsuario)
- Usar UPPER_CASE para constantes globales (API_URL)
- Los barrel files siempre se llaman index.ts

```typescript
// Consistencia es clave
export class UsuarioService {}       // PascalCase
export function obtenerUsuario() {}   // camelCase
export const MAX_INTENTOS = 3;        // UPPER_CASE
```

Si heredas un proyecto desorganizado, aquí tienes un plan de acción:

```typescript
// Paso 1: Auditoría
// - Identificar archivos y sus responsabilidades
// - Detectar dependencias circulares
// - Mapear la estructura actual

// Paso 2: Definir la estructura objetivo
// - Elegir entre capas o features
// - Definir convenciones de nombrado
// - Crear el árbol de carpetas vacío

// Paso 3: Migración incremental
// - Mover archivos uno por uno
// - Actualizar imports
// - Ejecutar tests después de cada cambio
```

```bash
# Comandos útiles para la migración
# Encontrar dependencias circulares:
npx madge --circular --extensions ts src/

# Visualizar dependencias:
npx madge --image graph.png --extensions ts src/

# Encontrar archivos sin usar:
npx ts-unused-exports tsconfig.json
```

```typescript
// Paso 4: Crear barrel files
// Una vez movidos los archivos, crear index.ts en cada carpeta

// Paso 5: Actualizar tsconfig.json
// - Verificar que paths y baseUrl estén correctos
// - Ajustar include/exclude

// Paso 6: Eliminar archivos y carpetas obsoletas

// - No eliminar sin antes verificar que nada depende de ello
```

RESUMEN DEL CAPÍTULO
===============================================================================

- La organización del código es tan importante como el código mismo.
- Arquitectura por capas: controllers, services, repositories.
- Arquitectura por features: cada funcionalidad es autocontenida.
- Los barrel files (index.ts) simplifican los imports y centralizan APIs.
- Los tipos pueden ir junto al feature o centralizados en types/.
- Los tests co-located (junto al código) son una práctica moderna.
- La consistencia en nombrado mejora la navegabilidad del proyecto.
- La refactorización debe ser incremental y con tests de verificación.

PREGUNTAS FRECUENTES
===============================================================================

❓ **¿Qué es mejor: capas o features?**
Depende del proyecto. Features son mejores para proyectos grandes con
módulos bien definidos. Capas son mejores para proyectos pequeños o con
mucha lógica compartida.

❓ **¿Debo tener un barrel file en cada carpeta?**
No obligatorio, pero muy recomendable. Los barrel files mejoran los imports
y centralizan la API pública de cada módulo.

❓ **¿Cómo manejo tipos que se usan en varios features?**
Crea una carpeta shared/types/ o types/ global. Los tipos específicos de
un feature deben quedarse dentro de ese feature.

❓ **¿Pongo los tests dentro de src o fuera?**
Ambos enfoques son válidos. Co-located (dentro de src) es más moderno y
práctico. Separado (carpeta tests/) es más tradicional.

❓ **¿Cómo evito import paths muy largos como ../../../utils?**
Usa path aliases en tsconfig.json (baseUrl + paths) para crear alias como
@utils/ o @shared/.
