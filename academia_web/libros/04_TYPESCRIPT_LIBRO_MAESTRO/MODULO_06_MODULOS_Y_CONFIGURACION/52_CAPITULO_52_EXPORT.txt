===============================================================================
1. EXPORTACIONES NOMBRADAS (NAMED EXPORTS)
===============================================================================

Las exportaciones nombradas son la forma más común y recomendada de exportar
en TypeScript. Cada elemento se exporta con un nombre explícito.

```typescript
// Exportación inline (directa en la declaración)
export function sumar(a: number, b: number): number {
  return a + b;
}

export const VERSION = "1.0.0";

export interface Usuario {
  id: number;
  nombre: string;
}

export type Estado = "activo" | "inactivo";

export class Servicio {
  iniciar(): void {
    console.log("Servicio iniciado");
  }
}
```

Podemos exportar cualquier declaración: variables, funciones, clases,
interfaces, tipos, enums y constantes.

```typescript
// Exportación de constantes
export const API_URL = "https://api.ejemplo.com";
export const TIMEOUT = 5000;
export const MAX_INTENTOS = 3;

// Exportación de funciones
export function saludar(nombre: string): string {
  return `Hola, ${nombre}`;
}

export function despedir(nombre: string): string {
  return `Adiós, ${nombre}`;
}

// Exportación de clases
export class Punto {
  constructor(public x: number, public y: number) {}
}

// Exportación de interfaces
export interface Configuracion {
  tema: "claro" | "oscuro";
  idioma: string;
}
```

Las exportaciones nombradas son explícitas y permiten a los consumidores
del módulo importar exactamente lo que necesitan.

===============================================================================
2. EXPORTACIONES AGRUPADAS (AGRUPAR AL FINAL)
===============================================================================

Podemos declarar variables, funciones y clases sin exportarlas, y luego
exportarlas todas al final del archivo en una sentencia única. Esto mejora
la legibilidad al concentrar la API del módulo en un solo lugar.

```typescript
// Declaraciones sin export
function sumar(a: number, b: number): number {
  return a + b;
}

function restar(a: number, b: number): number {
  return a - b;
}

function multiplicar(a: number, b: number): number {
  return a * b;
}

const PI = 3.1416;
const E = 2.7183;

interface Resultado {
  operacion: string;
  valor: number;
}

// Exportación agrupada al final
export { sumar, restar, multiplicar, PI, E };
export type { Resultado };
```

También podemos renombrar al exportar:

```typescript
function funcionInternaSuma(a: number, b: number): number {
  return a + b;
}

export { funcionInternaSuma as suma };
```

Esto es útil cuando queremos que el nombre público sea diferente al nombre
interno.

Ventajas de las exportaciones agrupadas:
- La API del módulo queda clara al final del archivo
- Podemos renombrar sin tocar las declaraciones
- Es fácil ver de un vistazo todo lo que el módulo expone

===============================================================================
3. EXPORTACIÓN POR DEFECTO (DEFAULT EXPORT)
===============================================================================

Cada módulo puede tener una única exportación por defecto. Se marca con la
palabra clave default.

```typescript
// Función como default
export default function procesar(datos: unknown): void {
  console.log("Procesando:", datos);
}

// Clase como default
export default class GestorArchivos {
  leer(ruta: string): string {
    return "contenido del archivo";
  }
}

// Objeto como default
export default {
  appName: "MiApp",
  version: "2.0.0",
  debug: false
};
```

Las exportaciones default pueden ser anónimas (sin nombre), ya que quien
importa asigna el nombre que prefiera.

```typescript
// Exportación default anónima
export default class {
  private items: string[] = [];

  agregar(item: string): void {
    this.items.push(item);
  }

  listar(): string[] {
    return this.items;
  }
}
```

Combinación de default y named exports:

```typescript
export default function sumar(a: number, b: number): number {
  return a + b;
}

export function restar(a: number, b: number): number {
  return a - b;
}

export const PI = 3.1416;
// Luego se importa así:
// import sumar, { restar, PI } from "./operaciones";
```

Recomendación: aunque es posible tener ambos tipos, muchos equipos prefieren
usar solo named exports por consistencia y mejor soporte de IDE.

===============================================================================
4. RE-EXPORTACIONES (BARREL FILES)
===============================================================================

Podemos exportar elementos que a su vez fueron importados de otros módulos.
Esto permite crear archivos "barril" (barrel files) que concentran la API
de un directorio.

```typescript
// ---------- archivo: index.ts (barrel file) ----------
export { sumar, restar } from "./matematicas/basicas";
export { calcularMedia, calcularVarianza } from "./matematicas/estadistica";
export type { Operacion, Resultado } from "./matematicas/tipos";

// Los consumidores importan desde un solo punto:
// import { sumar, calcularMedia, Resultado } from "./matematicas";
```

Re-exportación con rename:

```typescript
// Solucionar conflictos de nombres al re-exportar
export { sumar as sumaMatematica } from "./matematicas";
export { sumar as sumaFechas } from "./fechas";
```

Re-exportación completa (wildcard):

```typescript
export * from "./matematicas";
export * from "./estadistica";
```

Pero cuidado: si dos módulos exportan lo mismo, TypeScript lanzará error.

```typescript
// Solución: usar namespace en la re-exportación
export * as Matematicas from "./matematicas";
export * as Estadistica from "./estadistica";
// Consumidor: import { Matematicas, Estadistica } from "./index";
```

Re-exportación selectiva de todo excepto algunos elementos:

```typescript
export * from "./matematicas"; // exporta todo
```

No hay sintaxis directa para "exportar todo excepto", pero podemos omitir
los que no queremos y exportar el resto manualmente.

===============================================================================
5. EXPORT TYPE
===============================================================================

Al igual que import type, existe export type para exportar exclusivamente
tipos. Esto asegura que la exportación será eliminada en el JavaScript final.

```typescript
// ---------- archivo: tipos.ts ----------
export type Id = number | string;

export interface Producto {
  id: Id;
  nombre: string;
  precio: number;
}

export interface Usuario {
  id: Id;
  email: string;
}

// Export type agrupado
export type { Producto, Usuario };
```

```typescript
// En un barrel file, podemos re-exportar tipos de forma explícita
export type { Producto, Usuario } from "./tipos";
export { crearProducto, crearUsuario } from "./servicios";
```

TypeScript 4.5+ permite inline type en re-exportaciones:

```typescript
// Re-exportación mixta inline
export { crearProducto, type Producto, type Usuario } from "./productos";
```

Desde TypeScript 5.0, podemos usar `export type *`:

```typescript
// Exportar solo los tipos de otro módulo
export type * from "./tipos";
```

Esto es especialmente útil cuando queremos que los consumidores tengan
acceso a los tipos pero no queremos exponer implementaciones concretas.

===============================================================================
6. EXPORT EN DECLARACIONES COMPUESTAS
===============================================================================

TypeScript permite exportar múltiples elementos en diferentes estilos.

```typescript
// Export directo en declaración
export const constante1 = "valor1";
export const constante2 = "valor2";

// Export agrupado
export { constante3, constante4 };

// Re-export
export { algo } from "./otro-modulo";

// Export default
export default class ClasePrincipal {}

// Export type
export type { MiTipo };

// Todas estas formas pueden coexistir en un mismo archivo
```

También podemos exportar enums y type aliases:

```typescript
export enum Direction {
  Norte = "N",
  Sur = "S",
  Este = "E",
  Oeste = "O"
}

export type Respuesta<T> = {
  exito: boolean;
  datos: T;
  error?: string;
};
```

```typescript
// Exportar un namespace (legacy, pero aún soportado)
export namespace Utilidades {
  export function formatearFecha(fecha: Date): string {
    return fecha.toISOString();
  }

  export function formatearMoneda(valor: number): string {
    return `$${valor.toFixed(2)}`;
  }
}
```

7. BUENAS PRÁCTICAS EN EXPORTACIONES
===============================================================================

Diseñar una buena API de módulo es tan importante como escribir código
correcto.

```typescript
// 1. Preferir named exports sobre default exports
// ✅ Recomendado
export function procesarPago(monto: number): boolean { ... }
export function validarTarjeta(numero: string): boolean { ... }

// ❌ Menos recomendado (dificulta renombrar en IDE)
export default function procesarPago(monto: number): boolean { ... }
```

```typescript
// 2. Mantener la API pequeña y enfocada
// Exporta solo lo necesario, no todo el archivo

// 3. Usar nombres descriptivos y consistentes
export function obtenerUsuario(id: number): Usuario { ... }
export function crearUsuario(datos: CrearUsuarioDTO): Usuario { ... }
export function actualizarUsuario(id: number, datos: Partial<Usuario>): Usuario { ... }
```

```typescript
// 4. Usar barril (index.ts) para simplificar imports
// En cada carpeta, crea index.ts que re-exporta lo público
export { UsuarioService } from "./usuario.service";
export type { Usuario, CrearUsuarioDTO } from "./usuario.model";
```

```typescript
// 5. No exportar implementaciones internas
// ❌ MAL: exponer funciones auxiliares internas
export function formatearFechaInterna(fecha: Date): string { ... }

// ✅ BIEN: exportar solo lo que otros consumirán
export function obtenerFechaFormateada(id: number): string { ... }
```

```typescript
// 6. Documentar las exportaciones con JSDoc
/**
 * Calcula el precio total con impuestos incluidos.
 * @param precioBase - Precio sin impuestos
 * @param tasaIVA - Tasa de IVA (ej: 0.16 para 16%)
 * @returns Precio total con impuestos
 */
export function calcularPrecioConIVA(precioBase: number, tasaIVA: number): number {
  return precioBase * (1 + tasaIVA);
}
```

8. ERRORES COMUNES CON EXPORT
===============================================================================

```typescript
// Error 1: Múltiples default exports
// ❌
export default function funcion1() {}
export default function funcion2() {} // Error: solo un default por módulo

// ✅
export default function funcion1() {}
export function funcion2() {}
```

```typescript
// Error 2: Exportar y declarar sin nombre en default
// ❌ No tiene sentido, default requiere expresión
export default const x = 5;

// ✅ Correcto
const x = 5;
export default x;
```

```typescript
// Error 3: Confundir export named con default
// ❌ Si el módulo exporta default, no se importa con llaves

// Error 4: Olvidar export type para interfaces
// Puede funcionar sin type, pero usar export type es más explícito
```

```typescript
// Error 5: Re-exportar con nombre duplicado
export { usuario } from "./modulo-a";
export { usuario } from "./modulo-b"; // Error si ambos exportan "usuario"
```

RESUMEN DEL CAPÍTULO
===============================================================================

- Las exportaciones nombradas (named exports) usan export antes de la declaración.
- Las exportaciones agrupadas al final concentran la API del módulo.
- Cada módulo puede tener una única exportación por defecto (default).
- Las re-exportaciones permiten crear barrel files (index.ts) para simplificar imports.
- export type exporta exclusivamente tipos, eliminados en compilación.
- Es mejor práctica preferir named exports sobre default exports.
- Los barrel files consolidan la API pública de un directorio.

===============================================================================
PREGUNTAS FRECUENTES
===============================================================================

❓ **¿Cuántos default exports puede tener un módulo?**
Solo uno. No puede haber más de una exportación por defecto por módulo.

❓ **¿Puedo tener default export y named exports juntos?**
Sí. Un módulo puede tener un default export y múltiples named exports.

❓ **¿Export type y export normal son intercambiables?**
No exactamente. export type garantiza que la exportación será eliminada
en JavaScript. export normal también funciona para tipos, pero es menos
explícito.

❓ **¿Qué es un barrel file?**
Es un archivo (normalmente index.ts) que re-exporta elementos de otros
módulos, creando un punto único de entrada para un directorio.

❓ **¿Puedo re-exportar un módulo completo?**
Sí, con export * from "./modulo" o export * as Alias from "./modulo".
