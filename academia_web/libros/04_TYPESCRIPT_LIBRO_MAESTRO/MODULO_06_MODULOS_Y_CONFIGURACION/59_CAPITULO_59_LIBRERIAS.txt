===============================================================================
1. ¿QUÉ SON LOS ARCHIVOS .D.TS?
===============================================================================

Los archivos .d.ts (declaration files) contienen solo la información de
tipos de un módulo, sin implementación. Son el "contrato" de tipos que
TypeScript usa para entender librerías JavaScript.

```typescript
// ---------- ejemplo.d.ts ----------
declare function sumar(a: number, b: number): number;
declare const PI: number;
declare class Calculadora {
  sumar(a: number, b: number): number;
  restar(a: number, b: number): number;
}
export { sumar, PI, Calculadora };
```

Estos archivos se generan automáticamente al compilar TypeScript con
declaration: true:

```json
{
  "compilerOptions": {
    "declaration": true,
    "declarationDir": "./types",
    "declarationMap": true
  }
}
```

```typescript
// src/operaciones.ts
export function sumar(a: number, b: number): number {
  return a + b;
}

export interface Resultado {
  valor: number;
  operacion: string;
}
```

```typescript
// dist/operaciones.d.ts (generado)
export declare function sumar(a: number, b: number): number;
export interface Resultado {
  valor: number;
  operacion: string;
}
```

Los .d.ts describen la forma de las librerías sin incluir la implementación.
Son esenciales para:
- Librerías publicadas en npm
- Consumir código JavaScript desde TypeScript
- Compartir tipos entre proyectos sin compartir implementación

===============================================================================
2. DEFINITELYTYPED Y PAQUETES @TYPES
===============================================================================

DefinitelyTyped es un repositorio que alberga declaraciones de tipos para
miles de librerías JavaScript. Se publican como paquetes npm con prefijo
@types/.

```bash
# Instalar tipos para una librería existente
npm install --save-dev @types/react
npm install --save-dev @types/node
npm install --save-dev @types/express
npm install --save-dev @types/lodash
npm install --save-dev @types/jest
```

```typescript
// Después de instalar @types/node
import fs from "fs";
import path from "path";

// Todo está tipado gracias a @types/node
const contenido = fs.readFileSync(path.join(__dirname, "config.json"), "utf-8");
```

Cómo saber si una librería necesita @types:
1. Revisa si la librería ya incluye tipos (busca archivos .d.ts en su carpeta)
2. Busca en npm: npm info @types/nombre-de-la-libreria
3. Usa el comando: npx types-pkg-name check

```bash
# Verificar si una librería tiene tipos incluidos
npx types-pkg-name check lodash
# Output: @types/lodash exists
```

```typescript
// Algunas librerías ya incluyen tipos (no necesitan @types)
import axios from "axios";         // Axios incluye tipos propios
import { z } from "zod";           // Zod incluye tipos propios
import { v4 } from "uuid";         // uuid incluye tipos propios
import * as Prisma from "@prisma/client"; // Prisma genera sus tipos

// Otras necesitan @types
import * as React from "react";    // npm install @types/react
import express from "express";     // npm install @types/express
import _ from "lodash";            // npm install @types/lodash
```

===============================================================================
3. DECLARAR MÓDULOS SIN TIPOS (DECLARE MODULE)
===============================================================================

Cuando una librería no tiene tipos ni @types disponibles, podemos declarar
nuestros propios tipos con declare module.

```typescript
// ---------- src/types/mi-libreria.d.ts ----------
declare module "mi-libreria-sin-tipos" {
  export function hacerAlgo(param: string): number;
  export const VERSION: string;
  export class Cliente {
    constructor(config: { apiKey: string });
    conectar(): Promise<void>;
    obtener(url: string): Promise<unknown>;
  }
}
```

Para librerías globales (que se añaden al objeto window):

```typescript
// ---------- src/types/global.d.ts ----------
declare global {
  interface Window {
    miLibreriaGlobal: {
      init: (config: Record<string, unknown>) => void;
      version: string;
    };
  }
}

// Ahora podemos usar:
// window.miLibreriaGlobal.init({ debug: true });
```

Si la librería no tiene ningún tipo y queremos suprimir errores:

```typescript
// ---------- src/types/ambient.d.ts ----------
declare module "libreria-sin-tipos-que-no-me-interesa-tipar" {
  const anything: any;
  export = anything;
  export default anything;
}
```

También podemos declarar módulos con patrones de archivo:

```typescript
// Para archivos .json, .css, .svg, etc.
declare module "*.json" {
  const value: unknown;
  export default value;
}

declare module "*.svg" {
  const content: string;
  export default content;
}

declare module "*.css" {
  const classes: Record<string, string>;
  export default classes;
}
```

4. CREAR TYPES PARA LIBRERÍAS JAVASCRIPT
===============================================================================

Para librerías JavaScript más complejas, podemos crear un archivo de
declaración detallado.

```typescript
// ---------- types/mi-libreria/index.d.ts ----------
declare module "mi-libreria" {
  // Opciones de configuración
  interface Configuracion {
    host: string;
    puerto: number;
    ssl?: boolean;
    timeout?: number;
  }

  // Tipos de respuesta
  interface Respuesta<T> {
    datos: T;
    error: string | null;
    status: number;
  }

  // Funciones principales
  export function inicializar(config: Configuracion): void;
  export function obtener<T>(ruta: string): Promise<Respuesta<T>>;
  export function enviar<T>(ruta: string, datos: unknown): Promise<Respuesta<T>>;

  // Clases
  export class EventEmitter {
    on(evento: string, callback: (...args: unknown[]) => void): void;
    emit(evento: string, ...args: unknown[]): void;
    off(evento: string, callback: (...args: unknown[]) => void): void;
  }

  // Constantes
  export const VERSION: string;
  export const EVENTOS: {
    CONECTADO: "conectado";
    DESCONECTADO: "desconectado";
    ERROR: "error";
  };

  // Namespace (si la librería expone sub-módulos)
  export namespace Utils {
    export function formatearFecha(fecha: Date): string;
    export function sleep(ms: number): Promise<void>;
  }
}
```

Para probar nuestros tipos localmente antes de publicar:

```bash
# Incluir la carpeta types en tsconfig.json
```

```json
{
  "compilerOptions": {
    "typeRoots": ["./node_modules/@types", "./types"]
  }
}
```

5. PUBLICAR LIBRERÍAS CON TIPOS INCLUIDOS
===============================================================================

Si publicas una librería TypeScript, es importante incluir los archivos .d.ts.

```typescript
// ---------- src/index.ts (tu librería) ----------
export function sumar(a: number, b: number): number {
  return a + b;
}

export interface Config {
  debug: boolean;
  version: string;
}

export class Cliente {
  constructor(private config: Config) {}
  async conectar(): Promise<void> {
    // implementación
  }
}
```

```json
// package.json de la librería
{
  "name": "mi-libreria-typescript",
  "version": "1.0.0",
  "main": "dist/index.js",
  "types": "dist/index.d.ts",
  "files": ["dist"],
  "scripts": {
    "build": "tsc",
    "prepublishOnly": "npm run build"
  },
  "devDependencies": {
    "typescript": "^5.4.0"
  }
}
```

```json
// tsconfig.json de la librería
{
  "compilerOptions": {
    "target": "ES2020",
    "module": "CommonJS",
    "outDir": "./dist",
    "rootDir": "./src",
    "declaration": true,
    "declarationMap": true,
    "sourceMap": true,
    "strict": true,
    "esModuleInterop": true
  },
  "include": ["src"]
}
```

Para soporte dual ESM + CommonJS con tipos:

```json
{
  "name": "mi-libreria-dual",
  "version": "1.0.0",
  "type": "module",
  "main": "./dist/cjs/index.js",
  "module": "./dist/esm/index.js",
  "types": "./dist/types/index.d.ts",
  "exports": {
    ".": {
      "import": "./dist/esm/index.js",
      "require": "./dist/cjs/index.js",
      "types": "./dist/types/index.d.ts"
    }
  },
  "files": ["dist"]
}
```

6. EXTENDER TIPOS DE LIBRERÍAS EXISTENTES
===============================================================================

A veces necesitamos añadir propiedades o métodos a tipos existentes.

```typescript
// Extender Express Request para añadir usuario
// ---------- src/types/express.d.ts ----------
import { Usuario } from "./usuario";

declare module "express-serve-static-core" {
  interface Request {
    usuario?: Usuario;
  }
  interface Response {
    enviarExito(datos: unknown): void;
    enviarError(mensaje: string, codigo: number): void;
  }
}
```

```typescript
// Extender una librería de UI
declare module "mi-libreria-ui" {
  export interface BotonProps {
    // Añadir nuevas props a las existentes
    loading?: boolean;
    icon?: string;
  }
}
```

```typescript
// Extender Promise global
declare global {
  interface Promise<T> {
    tiempoLimite(ms: number): Promise<T>;
  }
}

// Implementación
Promise.prototype.tiempoLimite = function <T>(ms: number): Promise<T> {
  return Promise.race([
    this,
    new Promise((_, reject) =>
      setTimeout(() => reject(new Error("Tiempo agotado")), ms)
    ),
  ]) as Promise<T>;
};
```

El module augmentation (extensión de módulos) es una técnica poderosa que
permite adaptar librerías existentes a nuestras necesidades.

===============================================================================
7. CONFIGURACIÓN DE TYPES EN TSCONFIG.JSON
===============================================================================

```json
{
  "compilerOptions": {
    "types": ["node", "express", "jest"],
    "typeRoots": ["./node_modules/@types", "./src/types"],
    "skipLibCheck": false
  }
}
```

**types**: Lista blanca de paquetes @types a incluir. Por defecto, todos
los paquetes @types en node_modules se incluyen automáticamente.

```json
// Solo incluye estos tipos
"types": ["node"]
```

```json
// Incluye todos (por defecto si no se especifica)
// "types": []  <- No incluye ningún @types automáticamente
```

**typeRoots**: Directorios donde TypeScript busca paquetes @types.

```json
"typeRoots": ["./node_modules/@types", "./src/types"]
```

**skipLibCheck**: Omite verificación de tipos en .d.ts. Útil para acelerar
compilación, pero puede ocultar errores en @types.

8. RESOLUCIÓN DE PROBLEMAS CON @TYPES
===============================================================================

Problemas comunes y soluciones:

```bash
# Problema: Versión incompatible de @types
# Error: Type 'X' is not assignable to type 'Y'
npm install @types/react@latest
```

```typescript
// Problema: Una librería no encuentra sus tipos
// Solución: Verificar la configuración de types en package.json de la librería
// La propiedad "types" o "typings" debe apuntar al .d.ts correcto

// Problema: Múltiples versiones del mismo @types
// Solución: Usar npm dedupe o instalar versión específica
```

```bash
# Verificar qué versión de @types está instalada
npm ls @types/react

# Forzar una versión específica
npm install @types/react@18.2.0 --save-dev
```

```typescript
// Problema: El módulo no se encuentra
// Error: Cannot find module 'x' or its corresponding type declarations

// Solución 1: Instalar @types
npm install --save-dev @types/x

// Solución 2: Crear declaración propia
// archivo: x.d.ts
declare module "x";
```

```typescript
// Problema: Conflictos entre @types globales
// Solución: Usar types específicos en tsconfig.json
"types": ["node"] // Solo incluye @types/node
```

===============================================================================
PREGUNTAS FRECUENTES
===============================================================================

❓ **¿Todas las librerías npm tienen @types?**
No. Muchas librerías modernas incluyen sus propios tipos. Otras dependen
de DefinitelyTyped (@types). Algunas no tienen tipos en absoluto.

❓ **¿Qué hago si una librería no tiene tipos?**
Crea tu propio archivo .d.ts con declare module, o usa // @ts-ignore como
último recurso.

❓ **¿Cuál es la diferencia entre "types" y "typings" en package.json?**
Son equivalentes. "types" es el nombre moderno, "typings" es legacy. Ambos
apuntan al archivo .d.ts de entrada.

❓ **¿Puedo tener conflictos entre @types y tipos incluidos?**
Sí. Si una librería incluye sus propios tipos, no instales @types también.
Puede causar duplicación y conflictos.

❓ **¿Qué hace typeRoots?**
Especifica directorios donde TypeScript busca paquetes de tipos. Por defecto
es node_modules/@types.
