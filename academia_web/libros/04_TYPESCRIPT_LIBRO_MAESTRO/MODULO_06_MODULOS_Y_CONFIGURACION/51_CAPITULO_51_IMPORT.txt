===============================================================================
1. ¿QUÉ ES UN MÓDULO EN TYPESCRIPT?
===============================================================================

En TypeScript, cualquier archivo que contenga una declaración import o export
es considerado un módulo. Los módulos tienen su propio ámbito; las variables,
funciones, clases y tipos declarados en un módulo no son visibles fuera de él
a menos que se exporten explícitamente.

Los archivos que no tienen import ni export son tratados como scripts
globales, lo que significa que sus declaraciones comparten el ámbito global.
Esto puede causar conflictos de nombres, por lo que se recomienda usar siempre
el sistema de módulos.

```typescript
// modulo.ts (este archivo es un módulo por su uso de export)
export const mensaje = "Hola desde el módulo";

// main.ts
import { mensaje } from "./modulo";
console.log(mensaje); // "Hola desde el módulo"
```

Un archivo solo necesita un import o export para ser considerado un módulo:

```typescript
// Esto convierte al archivo en un módulo
export const x = 1;
```

```typescript
// Esto también lo convierte en un módulo
import { x } from "./otro-modulo";
```

Cuando usamos la sintaxis de módulos, TypeScript transpila el código según la
configuración de module en tsconfig.json (ES modules, CommonJS, etc.).

===============================================================================
2. IMPORT DE EXPORTACIONES NOMBRADAS (NAMED IMPORTS)
===============================================================================

Las exportaciones nombradas son aquellas que se exportan con un nombre
específico. Para importarlas, usamos llaves con el nombre exacto.

```typescript
// ---------- archivo: utilidades.ts ----------
export function sumar(a: number, b: number): number {
  return a + b;
}

export const PI = 3.1416;

export interface Resultado {
  valor: number;
  mensaje: string;
}

// ---------- archivo: main.ts ----------
import { sumar, PI, Resultado } from "./utilidades";

const resultado: Resultado = {
  valor: sumar(5, 3),
  mensaje: `El resultado es ${sumar(5, 3)} y PI vale ${PI}`
};

console.log(resultado);
```

Podemos importar múltiples elementos separándolos por coma dentro de las
llaves. El orden no importa, solo el nombre.

```typescript
import { PI, sumar, Resultado } from "./utilidades";
```

Si el módulo tiene muchas exportaciones, podemos importar todo el espacio
de nombres con el asterisco (*):

```typescript
import * as util from "./utilidades";

console.log(util.PI);
console.log(util.sumar(2, 3));

const res: util.Resultado = { valor: 5, mensaje: "OK" };
```

La sintaxis `import * as nombre` agrupa todas las exportaciones nombradas
bajo un solo objeto.

===============================================================================
3. RENOMBRAR IMPORTACIONES CON AS
===============================================================================

Podemos renombrar una importación para evitar conflictos o darle un nombre
más descriptivo en nuestro contexto.

```typescript
// Caso típico: dos módulos exportan la misma función
import { sumar as sumarMatematicas } from "./matematicas";
import { sumar as sumarFechas } from "./fechas";

const total = sumarMatematicas(5, 3);   // suma numérica
const totalDias = sumarFechas(new Date(), 5); // suma de días
```

También podemos renombrar varios elementos:

```typescript
import {
  sumar as suma,
  restar as resta,
  multiplicar as multi,
  dividir as div
} from "./operaciones";

console.log(suma(10, 5));  // 15
```

Es una práctica recomendada cuando los nombres genéricos podrían generar
confusión.

===============================================================================
4. IMPORT DE EXPORTACIÓN POR DEFECTO (DEFAULT IMPORT)
===============================================================================

Cada módulo puede tener una exportación por defecto. Se importa sin llaves
y se le puede asignar cualquier nombre.

```typescript
// ---------- archivo: Calculadora.ts ----------
export default class Calculadora {
  sumar(a: number, b: number): number {
    return a + b;
  }
}

// ---------- archivo: main.ts ----------
import Calculadora from "./Calculadora";

const calc = new Calculadora();
console.log(calc.sumar(4, 6)); // 10
```

La exportación por defecto puede ser una función, clase, objeto o cualquier
otro valor:

```typescript
// default como función
export default function saludar(nombre: string): string {
  return `¡Hola, ${nombre}!`;
}

// importación
import saludar from "./saludo";
console.log(saludar("Ana"));
```

```typescript
// default como objeto
export default {
  nombre: "TypeScript",
  version: "5.x"
};

// importación
import config from "./config";
console.log(config.nombre);
```

Podemos combinar import default con importaciones nombradas en una misma
línea:

```typescript
import Calculadora, { sumar, PI } from "./matematicas";
```

Primero va el default import (sin llaves) y luego las nombradas (entre llaves).

===============================================================================
5. IMPORT DE TIPOS (IMPORT TYPE)
===============================================================================

TypeScript permite importar exclusivamente tipos para que el transpilador
los elimine en la compilación. Esto es útil para evitar imports circulares
y para reducir el tamaño del código JavaScript generado.

```typescript
// ---------- archivo: tipos.ts ----------
export interface Usuario {
  id: number;
  nombre: string;
  email: string;
}

export type Estado = "activo" | "inactivo" | "pendiente";

export function crearUsuario(nombre: string): Usuario {
  return { id: 1, nombre, email: `${nombre}@email.com` };
}

// ---------- archivo: main.ts ----------
import type { Usuario, Estado } from "./tipos";
import { crearUsuario } from "./tipos";

const estado: Estado = "activo";
const usuario: Usuario = crearUsuario("Luis");

console.log(usuario, estado);
```

La palabra clave `type` en la importación le indica a TypeScript que estas
declaraciones son solo de tipo y serán eliminadas en el JS resultante.

```typescript
// Alternativa inline (menos común)
import { crearUsuario, type Usuario, type Estado } from "./tipos";
```

Desde TypeScript 4.5, podemos usar la sintaxis inline para mezclar imports
de valor y de tipo en una misma línea.

Beneficios de import type:
- Evita imports circulares (un problema común en TypeScript)
- Reduce el JavaScript generado
- Hace explícito qué es tipo y qué es valor
- Mejora el rendimiento de compilación en proyectos grandes

===============================================================================
6. IMPORT SIDE-EFFECT
===============================================================================

A veces necesitamos importar un módulo solo por sus efectos secundarios (side
effects), sin usar ninguna de sus exportaciones. Esto se logra importando
sin asignar nada:

```typescript
import "./polyfills";
import "./estilos.css";
import "./configuracion-global";
```

Este tipo de import ejecuta el código del módulo importado. Es común para:
- Polyfills que extienden el entorno
- Archivos de estilo (CSS, SCSS)
- Configuración global que debe ejecutarse al inicio
- Registro de service workers o plugins

```typescript
// polyfills.ts
if (!Array.prototype.flat) {
  Array.prototype.flat = function() {
    // implementación del polyfill
  };
}

// main.ts
import "./polyfills"; // ejecuta el polyfill sin importar nada

const arr = [1, [2, [3]]].flat(); // funciona gracias al polyfill
```

Los imports side-effect deben usarse con moderación, ya que dificultan el
tree-shaking y pueden hacer que el orden de carga sea impredecible.

===============================================================================
7. IMPORTACIONES DINÁMICAS (DYNAMIC IMPORT)
===============================================================================

TypeScript soporta importaciones dinámicas con la función import(), que
devuelve una promesa. Esto permite carga bajo demanda (lazy loading).

```typescript
async function cargarModulo() {
  try {
    const modulo = await import("./operaciones");
    console.log(modulo.sumar(10, 5));
  } catch (error) {
    console.error("Error al cargar el módulo", error);
  }
}

// También con sintaxis then/catch
import("./matematicas").then(mod => {
  console.log(mod.PI);
});
```

Las importaciones dinámicas son útiles para:
- Dividir el código en chunks (code splitting)
- Cargar módulos solo cuando se necesitan
- Implementar plugins o extensiones
- Mejorar el rendimiento inicial de la aplicación

```typescript
// Ejemplo práctico: carga condicional
async function procesarArchivo(formato: "pdf" | "csv" | "json") {
  switch (formato) {
    case "pdf":
      const pdf = await import("./procesadores/pdf-processor");
      return pdf.procesar();
    case "csv":
      const csv = await import("./procesadores/csv-processor");
      return csv.procesar();
    case "json":
      const json = await import("./procesadores/json-processor");
      return json.procesar();
  }
}
```

8. BUENAS PRÁCTICAS EN IMPORTACIONES
===============================================================================

Organizar correctamente las importaciones mejora la legibilidad y el
mantenimiento del código.

```typescript
// 1. Orden recomendado de imports
import "modulo-side-effect";          // 1. Side effects

import React from "react";             // 2. Librerías externas
import { useState, useEffect } from "react";
import axios from "axios";

import type { Usuario } from "./tipos"; // 3. Imports de tipo
import type { RespuestaAPI } from "./tipos";

import { sumar, restar } from "./utils"; // 4. Módulos propios
import { formatearFecha } from "./fechas";
import Calculadora from "./Calculadora";
```

```typescript
// 2. Agrupar imports del mismo módulo
// ❌ MAL: imports separados del mismo módulo
import { sumar } from "./matematicas";
import { PI } from "./matematicas";
import { restar } from "./matematicas";

// ✅ BIEN: un solo import
import { sumar, restar, PI } from "./matematicas";
```

```typescript
// 3. No importar lo que no se usa
import { sumar, restar, multiplicar, dividir } from "./operaciones";
// Si solo usamos sumar y restar, eliminemos multiplicar y dividir

// 4. Preferir named exports sobre default exports
// Los named exports permiten autocompletado y renombrado consistente

// 5. Usar path aliases configurados en tsconfig.json
import { Usuario } from "@modelos/usuario";
import { apiHelper } from "@utils/api";
```

===============================================================================
9. ERRORES COMUNES CON IMPORT
===============================================================================

Veamos algunos errores típicos y cómo solucionarlos:

```typescript
// Error 1: Ruta incorrecta
// ❌
import { algo } from "./modulo"; // si modulo.ts está en ../modulo

// ✅
import { algo } from "../modulo";
```

```typescript
// Error 2: Olvidar las llaves en named imports
// ❌
import sumar from "./matematicas"; // sumar es named export, no default

// ✅
import { sumar } from "./matematicas";
```

```typescript
// Error 3: Usar llaves en default imports
// ❌
import { Calculadora } from "./Calculadora"; // Calculadora es default

// ✅
import Calculadora from "./Calculadora";
```

```typescript
// Error 4: Importar algo que no existe
// ❌
import { funcionInexistente } from "./modulo";
// Error: Module '"./modulo"' has no exported member 'funcionInexistente'
```

```typescript
// Error 5: Conflictos de nombre sin resolver
// ❌
import { sumar } from "./matematicas";
import { sumar } from "./fechas"; // Error: duplicate identifier

// ✅
import { sumar as sumarMat } from "./matematicas";
import { sumar as sumarFech } from "./fechas";
```

===============================================================================
RESUMEN DEL CAPÍTULO
===============================================================================

- Los módulos en TypeScript son archivos con import o export y tienen ámbito propio.
- Los named imports usan llaves: import { nombre } from "./modulo".
- Los default imports no usan llaves: import Nombre from "./modulo".
- Se puede renombrar con as: import { x as y } from "./modulo".
- import type permite importar solo tipos, eliminados en compilación.
- import * as agrupa todas las exportaciones nombradas.
- Las importaciones dinámicas con import() permiten lazy loading.
- Los side-effect imports ejecutan código sin importar valores.
- Es buena práctica ordenar y agrupar las importaciones.

===============================================================================
PREGUNTAS FRECUENTES
===============================================================================

❓ **¿Cuál es la diferencia entre import y require?**
import es la sintaxis de ES Modules, estática y analizable en tiempo de
compilación. require es de CommonJS, dinámica y ejecutada en runtime.
TypeScript puede transpilar import a require según la configuración module.

❓ **¿Puedo mezclar import default y named import?**
Sí: import NombreDefault, { named1, named2 } from "./modulo".

❓ **¿import type elimina realmente el código?**
Sí. TypeScript elimina las importaciones marcadas con type del JavaScript
generado, ya que los tipos no existen en runtime.

❓ **¿Qué pasa si importo un archivo que no existe?**
TypeScript reportará un error en tiempo de compilación. Algunos bundlers
(Webpack, Vite) también lanzarán error en tiempo de build.

❓ **¿Es malo usar import *?**
No es malo, pero puede traer más código del necesario y dificultar el
tree-shaking. Es preferible importar solo lo que se necesita.
