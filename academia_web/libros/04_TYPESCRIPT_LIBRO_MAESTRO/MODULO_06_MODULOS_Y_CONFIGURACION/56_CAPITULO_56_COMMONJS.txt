===============================================================================
1. ¿QUÉ ES COMMONJS?
===============================================================================

CommonJS es un sistema de módulos diseñado para entornos fuera del navegador,
principalmente Node.js. Se basa en dos funciones globales:

- `require()`: Para importar módulos
- `module.exports`: Para exportar desde un módulo

```javascript
// ----- matematicas.js (CommonJS) -----
function sumar(a, b) {
  return a + b;
}

function restar(a, b) {
  return a - b;
}

const PI = 3.1416;

module.exports = { sumar, restar, PI };

// ----- main.js (CommonJS) -----
const { sumar, restar, PI } = require("./matematicas");
console.log(sumar(5, 3)); // 8
console.log(PI); // 3.1416
```

TypeScript puede compilar código con sintaxis import/export a CommonJS:

```typescript
// ----- TypeScript src/main.ts -----
export function sumar(a: number, b: number): number {
  return a + b;
}
export const PI = 3.1416;
```

Compilado a CommonJS:

```javascript
// ----- JavaScript dist/main.js -----
"use strict";
Object.defineProperty(exports, "__esModule", { value: true });
exports.PI = exports.sumar = void 0;
function sumar(a, b) {
  return a + b;
}
exports.sumar = sumar;
exports.PI = 3.1416;
```

Configuración para CommonJS en tsconfig.json:

```json
{
  "compilerOptions": {
    "module": "CommonJS",
    "target": "ES2020",
    "outDir": "./dist",
    "rootDir": "./src",
    "strict": true,
    "esModuleInterop": true
  }
}
```

===============================================================================
2. REQUIRE VS IMPORT
===============================================================================

La principal diferencia es sintáctica y semántica:

```typescript
// CommonJS con require (estilo Node.js)
const modulo = require("./modulo");
const { sumar, restar } = require("./modulo");
const { default: MiClase } = require("./modulo");
```

```typescript
// ES Modules con import (estilo TypeScript)
import * as modulo from "./modulo";
import { sumar, restar } from "./modulo";
import MiClase from "./modulo";
```

Con TypeScript, escribimos siempre con sintaxis import/export, y el
compilador transforma a require/module.exports según la opción module:

```json
"module": "CommonJS"   // -> require/module.exports
"module": "ESNext"     // -> import/export nativos
"module": "NodeNext"   // -> según package.json
```

```typescript
// TypeScript (siempre igual, independientemente del target)
import { sumar } from "./matematicas";
import fs from "fs";

export function duplicar(n: number): number {
  return sumar(n, n);
}
```

```javascript
// CommonJS output
"use strict";
Object.defineProperty(exports, "__esModule", { value: true });
exports.duplicar = void 0;
const matematicas_1 = require("./matematicas");
const fs_1 = require("fs");
function duplicar(n) {
  return (0, matematicas_1.sumar)(n, n);
}
exports.duplicar = duplicar;
```

```javascript
// ESM output
import { sumar } from "./matematicas.js";
import fs from "fs";
export function duplicar(n) {
  return sumar(n, n);
}
```

===============================================================================
3. MODULE.EXPORTS Y EXPORTS EN TYPESCRIPT
===============================================================================

En TypeScript, no usamos directamente module.exports o exports. Usamos la
sintaxis import/export de ES Modules, y el compilador traduce a CommonJS.

Sin embargo, a veces necesitamos interoperar con código CommonJS existente.

```typescript
// Para exportar directamente usando module.exports:
// (No recomendado, mejor usar sintaxis ES)
export = function sumar(a: number, b: number): number {
  return a + b;
};

// Esto compila a:
// module.exports = function sumar(a, b) { return a + b; }
```

```typescript
// Importar un módulo que usa module.exports
import sumar = require("./matematicas");
// o
import sumar from "./matematicas"; // con esModuleInterop
```

La sintaxis `export =` es la forma TypeScript de emular module.exports:

```typescript
// ----- archivo: calculadora.ts -----
class Calculadora {
  sumar(a: number, b: number): number {
    return a + b;
  }
  restar(a: number, b: number): number {
    return a - b;
  }
}

export = Calculadora; // Equivalente a: module.exports = Calculadora
```

```typescript
// ----- archivo: main.ts -----
import Calculadora from "./calculadora";
// o:
import Calc = require("./calculadora");
```

La sintaxis `import = require` es específica para módulos CommonJS en
TypeScript y se usa cuando el módulo exporta con `export =`.

4. __DIRNAME Y __FILENAME EN TYPESCRIPT CON COMMONJS
===============================================================================

En Node.js CommonJS, existen las variables globales __dirname (ruta del
directorio actual) y __filename (ruta del archivo actual). TypeScript las
soporta con la configuración adecuada.

```typescript
console.log(__dirname);  // /home/proyecto/src
console.log(__filename); // /home/proyecto/src/main.ts
```

Para que TypeScript reconozca estas variables, necesitas instalar los tipos
de Node.js:

```bash
npm install --save-dev @types/node
```

```typescript
import path from "path";

export function leerConfiguracion(): string {
  const ruta = path.join(__dirname, "config.json");
  // leer archivo...
  return ruta;
}
```

En ESM, __dirname y __filename no existen. Debes usar import.meta.url:

```typescript
// Para ESM, alternativa a __dirname
import { fileURLToPath } from "url";
import { dirname } from "path";

const __filename = fileURLToPath(import.meta.url);
const __dirname = dirname(__filename);
```

Con CommonJS, estas variables están siempre disponibles sin configuración
adicional.

5. COMMONJS Y MODOS DE EXPORTACIÓN
===============================================================================

CommonJS ofrece varias formas de exportar:

```javascript
// Forma 1: Exportar un objeto
module.exports = {
  sumar: (a, b) => a + b,
  restar: (a, b) => a - b
};

// Forma 2: Exportar como propiedades de exports
exports.sumar = (a, b) => a + b;
exports.restar = (a, b) => a - b;

// Forma 3: Exportar una sola función/clase (reemplaza module.exports)
module.exports = function sumar(a, b) { return a + b; };
```

En TypeScript, estas formas se traducen así:

```typescript
// Equivalente TypeScript de forma 1 (objeto)
export = {
  sumar(a: number, b: number): number { return a + b; },
  restar(a: number, b: number): number { return a - b; }
};
```

```typescript
// Equivalente TypeScript de forma 2 (named exports)
export function sumar(a: number, b: number): number { return a + b; }
export function restar(a: number, b: number): number { return a - b; }
```

```typescript
// Equivalente TypeScript de forma 3 (función única)
export = function sumar(a: number, b: number): number {
  return a + b;
};
```

Es importante notar que `exports` (sin module) y `module.exports` son el
mismo objeto inicialmente, pero si reasignas module.exports, pierdes la
referencia a exports.

```javascript
// Esto funciona:
exports.sumar = (a, b) => a + b;
// exports.sumar existe en module.exports

// Esto rompe la referencia:
module.exports = { algo: "diferente" };
// exports.sumar ya no está disponible
```

6. INTEROPERABILIDAD CJS ↔ ESM
===============================================================================

TypeScript facilita la interoperabilidad entre CommonJS y ES Modules
mediante la opción esModuleInterop.

```json
{
  "compilerOptions": {
    "esModuleInterop": true,
    "allowSyntheticDefaultImports": true
  }
}
```

Sin esModuleInterop, importar un módulo CommonJS requiere sintaxis especial:

```typescript
// Sin esModuleInterop
import fs = require("fs");
import express = require("express");
```

```typescript
// Con esModuleInterop
import fs from "fs";
import express from "express";
```

esModuleInterop añade un helper que detecta cómo el módulo exporta y crea
un "default" apropiado.

```javascript
// Cómo compila TypeScript con esModuleInterop
var __importDefault = (this && this.__importDefault) || function (mod) {
  return (mod && mod.__esModule) ? mod : { "default": mod };
};
var express_1 = __importDefault(require("express"));
```

Para proyectos que publican librerías en CommonJS, es recomendable:

1. Incluir archivos .d.ts (declaraciones de tipos)
2. Configurar "types" en package.json
3. Usar esModuleInterop para mejor compatibilidad

```json
{
  "name": "mi-libreria",
  "version": "1.0.0",
  "main": "dist/index.js",
  "types": "dist/index.d.ts",
  "scripts": {
    "build": "tsc"
  }
}
```

7. CÓDIGO TYPESCRIPT QUE SE COMPILA A COMMONJS
===============================================================================

Veamos ejemplos prácticos de TypeScript compilado a CommonJS:

```typescript
// ----- src/libreria.ts -----
export interface Usuario {
  id: number;
  nombre: string;
}

export function crearUsuario(nombre: string): Usuario {
  return { id: Date.now(), nombre };
}

export default function saludar(nombre: string): string {
  return `¡Hola, ${nombre}!`;
}
```

```javascript
// ----- dist/libreria.js (CommonJS output) -----
"use strict";
Object.defineProperty(exports, "__esModule", { value: true });
exports.crearUsuario = void 0;
function crearUsuario(nombre) {
  return { id: Date.now(), nombre };
}
exports.crearUsuario = crearUsuario;
function saludar(nombre) {
  return "¡Hola, " + nombre + "!";
}
exports.default = saludar;
```

```typescript
// ----- src/main.ts -----
import saludar, { crearUsuario, Usuario } from "./libreria";

const usuario: Usuario = crearUsuario("Ana");
console.log(saludar(usuario.nombre));
```

```javascript
// ----- dist/main.js (CommonJS output) -----
"use strict";
Object.defineProperty(exports, "__esModule", { value: true });
const libreria_1 = require("./libreria");
const usuario = (0, libreria_1.crearUsuario)("Ana");
console.log((0, libreria_1.default)(usuario.nombre));
```

Observa que:
- `default` export se compila como `exports.default`
- `esModuleInterop` permite importar default desde CommonJS
- Las interfaces (tipos) desaparecen en el JavaScript generado

8. BUENAS PRÁCTICAS CON COMMONJS EN TYPESCRIPT
===============================================================================

```typescript
// 1. Escribir siempre con sintaxis import/export de ES
export function hacerAlgo(): void {}
import { hacerAlgo } from "./modulo";

// 2. Usar esModuleInterop: true en tsconfig
// Esto simplifica la importación de módulos CommonJS

// 3. Para exportar directamente como module.exports, usar export =
export = class Servicio { /* ... */ };

// 4. Usar @types/node para tener tipado de Node.js
```

```typescript
// 5. Evitar mezclar estilos en el mismo proyecto
// No uses require() si estás usando import/export

// 6. Para dependencias circulares, lazy require:
function procesar() {
  const { helper } = require("./helper"); // lazy require
  helper();
}
```

```typescript
// 7. Declarar módulos CommonJS existentes
declare module "mi-libreria-cjs" {
  export function hacerAlgo(): void;
  export const version: string;
}
```

```typescript
// 8. Usar import type para tipos en CommonJS
import type { Usuario } from "./modelos";
import { crearUsuario } from "./modelos";
```

===============================================================================
PREGUNTAS FRECUENTES
===============================================================================

❓ **¿CommonJS está obsoleto?**
No. Sigue siendo el sistema más usado en Node.js y npm. Sin embargo, se
recomienda migrar gradualmente a ESM para proyectos nuevos.

❓ **¿Puedo usar import y export con CommonJS?**
Sí. TypeScript compila import/export a require/module.exports cuando
module: "CommonJS" está configurado.

❓ **¿Qué hace esModuleInterop?**
Permite importar módulos CommonJS con la sintaxis import default, añadiendo
un helper que adapta module.exports a un objeto con propiedad default.

❓ **¿Por qué mi código CommonJS tiene "__esModule"?**
TypeScript añade exports.__esModule = true para indicar que el módulo
proviene de un ES Module, facilitando la interoperabilidad.

❓ **¿Puedo usar require en un archivo .ts?**
Técnicamente sí, pero no se recomienda. Mejor usa la sintaxis import/export
y deja que TypeScript maneje la transformación.
