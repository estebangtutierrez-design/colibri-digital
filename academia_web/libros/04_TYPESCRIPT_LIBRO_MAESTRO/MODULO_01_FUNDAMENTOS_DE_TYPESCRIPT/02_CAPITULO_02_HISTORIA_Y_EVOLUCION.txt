1. EL ORIGEN: ANDERS HEJLSBERG Y MICROSOFT
===============================================================================

TypeScript fue creado por **Anders Hejlsberg**, un ingeniero de software danés
que también es el diseñador principal de Turbo Pascal, Delphi y C#. Hejlsberg
trabaja en Microsoft desde 1996 y es una de las figuras más influyentes en el
mundo de la programación.

La historia cuenta que Hejlsberg y su equipo estaban desarrollando aplicaciones
web complejas en Microsoft usando JavaScript y se enfrentaban a los mismos
problemas que cualquier desarrollador: código difícil de mantener, errores
silenciosos en producción y falta de herramientas adecuadas para proyectos
a gran escala.

El desarrollo de TypeScript comenzó internamente en Microsoft en **2010**.
Durante dos años, el equipo trabajó en el diseño del lenguaje, el compilador
y el servicio de lenguaje antes de hacerlo público.

---

📌 **El nombre "TypeScript" proviene de la combinación de "Type" (tipo) y
"Script" (guión/script), reflejando su naturaleza de añadir tipos a la
programación tipo script.**

```typescript
// Anders Hejlsberg quería algo como esto...
class Programador {
  constructor(
    public nombre: string,
    public lenguajeFavorito: string
  ) {}
}

const andres = new Programador("Anders Hejlsberg", "TypeScript");
```

---

🔵 **Ejemplo 1: La visión de Hejlsberg**

```typescript
// Él quería que TypeScript se sintiera como JavaScript, no como C#
// Por eso la sintaxis es familiar:
function saludar(nombre: string) {
  return `Hola ${nombre}`;
}

// Y no como:
// function saludar(nombre: String): String { ... }
```

2. EL NACIMIENTO PÚBLICO: OCTUBRE 2012
===============================================================================

TypeScript fue anunciado públicamente el **1 de octubre de 2012** en la
conferencia Build de Microsoft. La versión inicial fue la **0.8**, indicando
que aún estaba en desarrollo activo y no se consideraba lista para producción.

El anuncio causó gran revuelo en la comunidad. Muchos desarrolladores se
mostraron escépticos: "Microsoft va a matar JavaScript", "Es otro intento
de imponer C# en la web". Sin embargo, pronto quedó claro que TypeScript no
competía con JavaScript, sino que lo complementaba.

**Características de TypeScript 0.8:**
- Tipado estático básico
- Clases (antes que ES6 las incluyera oficialmente)
- Interfaces
- Módulos
- Compilador escrito en TypeScript (dogfooding)
- Soporte para ES3 como target de compilación

---

📌 **TypeScript fue uno de los primeros lenguajes en implementar "dogfooding":
el compilador estaba escrito en TypeScript desde el principio.**

```typescript
// Sintaxis que ya existía en 0.8 (válida hoy)
interface Persona {
  nombre: string;
  edad: number;
}

class Empleado implements Persona {
  constructor(
    public nombre: string,
    public edad: number,
    public salario: number
  ) {}
}
```

---

🔵 **Ejemplo 2: Lo que NO tenía TypeScript 0.8**

```typescript
// No tenía:
// - Tipos unión (string | number)
// - Tipos condicionales
// - Mapped types
// - async/await (eso llegó después con ES2017)

// Tenía que escribirse así:
function procesar(dato: any): any {
  return dato;
}
```

3. VERSIONES TEMPRANAS (2012-2015)
===============================================================================

La evolución temprana de TypeScript fue rápida y agresiva:

| Versión | Fecha | Hitos principales |
|---------|-------|-------------------|
| 0.8 | Oct 2012 | Primer lanzamiento público |
| 0.9 | Jun 2013 | Genéricos, enum, typeof |
| 1.0 | Abr 2014 | Estabilidad, soporte en Visual Studio |
| 1.3 | Nov 2014 | Protected, tuplas |
| 1.4 | Ene 2015 | Tipos unión, let/const |
| 1.5 | Jul 2015 | Módulos ES6, decoradores, tsconfig.json |
| 1.6 | Sep 2015 | Type guards, intersection types |

**La versión 1.0** fue un hito importantísimo: señaló que TypeScript era
lo suficientemente maduro para producción. Microsoft se comprometió con la
compatibilidad hacia atrás, algo que han mantenido hasta hoy.

---

📌 **TypeScript 1.0 coincidió aproximadamente con el estándar ECMAScript 5,
que era lo más moderno disponible en navegadores en ese momento.**

```typescript
// TypeScript 1.4 introdujo los tipos unión
let identificador: string | number;
identificador = "ABC-123";  // ✅
identificador = 456;        // ✅
// identificador = true;    // ❌ Error

// TypeScript 1.6 introdujo Type Guards
function esString(valor: any): valor is string {
  return typeof valor === "string";
}
```

---

🔵 **Ejemplo 3: Genéricos llegaron en la versión 0.9**

```typescript
// Los genéricos permiten crear componentes reutilizables
function identidad<T>(arg: T): T {
  return arg;
}

const num = identidad<number>(42);
const str = identidad<string>("Hola");
const bool = identidad(true);  // Inferencia: T = boolean
```

4. LA ERA ES6 Y LA MADUREZ (2015-2019)
===============================================================================

El año **2015** fue transformador tanto para JavaScript como para TypeScript.
ECMAScript 6 (ES2015) se ratificó, trayendo clases, arrow functions, let/const,
promesas y muchas otras características al estándar de JavaScript.

TypeScript, que siempre se había adelantado al estándar (tenía clases desde
2012), ahora se alineaba con ES6 y se convertía en el puente perfecto: los
desarrolladores podían usar características modernas de ES6 y transpilarlas
a ES5 para navegadores antiguos, con tipos adicionales.

**Versiones clave de esta era:**

| Versión | Fecha | Hitos |
|---------|-------|-------|
| 2.0 | Sep 2016 | null/undefined checks, never, `--strictNullChecks` |
| 2.1 | Nov 2016 | Mapped types, keyof, Object spread/rest |
| 2.3 | Abr 2017 | Default type parameters, --strict |
| 2.4 | Jun 2017 | String enums, dynamic import |
| 2.8 | Mar 2018 | Conditional types, --strictFunctionTypes |
| 3.0 | Jul 2018 | Project references, tuple types mejorados |
| 3.2 | Nov 2018 | strictBindCallApply |
| 3.4 | Mar 2019 | --incremental, type-safe F-bounded polymorphism |

---

📌 **`--strictNullChecks` (TS 2.0) fue probablemente la opción de compilador
más impactante: `null` y `undefined` dejaron de ser asignables a cualquier tipo.**

```typescript
// Sin strictNullChecks
let nombre: string = null;  // ✅ Permitido (mal)

// Con strictNullChecks (TS 2.0+)
let nombre: string = null;  // ❌ Error
let nombre2: string | null = null;  // ✅ Explícito
```

---

🔵 **Ejemplo 4: Conditional types (TS 2.8)**

```typescript
// Los tipos condicionales permiten lógica a nivel de tipos
type EsString<T> = T extends string ? "sí" : "no";

type A = EsString<string>;  // "sí"
type B = EsString<number>;  // "no"

// Uso real: extraer el tipo de promesas
type Desempaquetar<T> = T extends Promise<infer U> ? U : T;

type Resultado = Desempaquetar<Promise<number>>;  // number
type Directo = Desempaquetar<string>;             // string
```

5. ACELERACIÓN Y ADOPCIÓN MASIVA (2019-2022)
===============================================================================

La adopción de TypeScript explotó entre 2019 y 2022, impulsada por:

- **Deno** (2018-2020): El runtime de JavaScript creado por Ryan Dahl (creador
  de Node.js) con soporte nativo de TypeScript
- **React con TypeScript**: Create React App añadió soporte oficial con `--template typescript`
- **Vue 3**: Escrito en TypeScript
- **Angular**: Siempre había usado TypeScript como lenguaje principal
- **Next.js y NestJS**: Adopción nativa

**Versiones clave:**

| Versión | Fecha | Hitos |
|---------|-------|-------|
| 3.7 | Nov 2019 | Optional chaining (`?.`), nullish coalescing (`??`), assertion functions |
| 3.8 | Feb 2020 | Type-only imports, export * as, ES2020 target |
| 3.9 | May 2020 | Mejoras de rendimiento en el compilador |
| 4.0 | Ago 2020 | Variadic tuple types, labeled tuple elements |
| 4.1 | Nov 2020 | Template literal types, key remapping in mapped types |
| 4.2 | Feb 2021 | Smarter type alias preservation, `--noPropertyAccessFromIndexSignature` |
| 4.3 | Jun 2021 | Override keyword, `--noImplicitOverride` |
| 4.4 | Ago 2021 | Control flow analysis for aliased conditions, symbol types |

---

📌 **Optional chaining (`?.`) y nullish coalescing (`??`) en TS 3.7 fueron
características muy esperadas que llegaron primero a TypeScript y luego al
estándar ECMAScript.**

```typescript
// Optional chaining (TS 3.7)
const ciudad = usuario?.direccion?.ciudad ?? "Ciudad no disponible";

// Equivalente sin optional chaining:
const ciudad2 = usuario && usuario.direccion && usuario.direccion.ciudad
  ? usuario.direccion.ciudad
  : "Ciudad no disponible";

// Nullish coalescing: solo reemplaza null/undefined, no otros falsy
const contador = 0;
const valor = contador ?? 10;  // 0 (0 no es null/undefined)
const valor2 = contador || 10; // 10 (0 es falsy)
```

---

🔵 **Ejemplo 5: Template literal types (TS 4.1)**

```typescript
// Los template literal types permiten manipular strings a nivel de tipos
type Color = "red" | "blue" | "green";
type Tamanio = "small" | "medium" | "large";

type ClaseCSS = `${Color}-${Tamanio}`;
// "red-small" | "red-medium" | "red-large"
// | "blue-small" | "blue-medium" | "blue-large"
// | "green-small" | "green-medium" | "green-large"

function aplicarClase(clase: ClaseCSS) {
  document.body.classList.add(clase);
}

aplicarClase("red-medium"); // ✅
// aplicarClase("yellow-small"); // ❌ Error
```

6. TYPESCRIPT MODERNO (2022-PRESENTE)
===============================================================================

El desarrollo de TypeScript continúa activamente. Microsoft libera nuevas
versiones aproximadamente cada 2-3 meses, cada una con mejoras de rendimiento,
correcciones de errores y nuevas características del lenguaje.

**Versiones recientes:**

| Versión | Fecha | Hitos |
|---------|-------|-------|
| 4.5 | Nov 2021 | Import type en módulos, `Awaited` type |
| 4.6 | Feb 2022 | Destructuring control flow, `--target es2022` |
| 4.7 | May 2022 | Instantiation expressions, `in` narrowing, ES2022 target |
| 4.8 | Ago 2022 | Improved intersection reduction, `--strict` refinements |
| 4.9 | Nov 2022 | `satisfies` operator, `--autoAccessor` |
| 5.0 | Mar 2023 | Decoradores, `--moduleResolution bundler`, const type parameters |
| 5.1 | Jun 2023 | `undefined`-returning functions, JSX elements |
| 5.2 | Ago 2023 | `using` declarations (TC39 Stage 3) |
| 5.3 | Nov 2023 | Import attributes, `--isolatedDeclarations` |
| 5.4 | Mar 2024 | NoInfer, preserved narrowing, improvements |
| 5.5 | Jun 2024 | Inferred type predicates, control flow narrowing |
| 5.6 | Sep 2024 | Nullish coalescing assignments, iterator helpers |

---

📌 **TypeScript 5.0, lanzado en marzo de 2023, fue una versión histórica que
reimplementó el compilador para ser más rápido y moderno, y añadió decoradores
estándar compatibles con el proposal de TC39.**

```typescript
// TypeScript 4.9: El operador satisfies
type RGB = { red: number; green: number; blue: number };
type Color = RGB | string;

const miColor = { red: 255, green: 200, blue: 100 } satisfies Color;
// Tipo inferido: { red: number; green: number; blue: number }
// satisfies verifica que cumple Color sin cambiar el tipo inferido

// TypeScript 5.2: using declarations
function usarRecurso() {
  using recurso = crearRecurso();
  // recurso se libera automáticamente al salir del ámbito
}
```

---

🔵 **Ejemplo 6: Decoradores estándar en TS 5.0**

```typescript
// TS 5.0 implementó decoradores según el estándar TC39
function registrar(target: any, context: ClassMethodDecoratorContext) {
  const metodoOriginal = target;
  return function (this: any, ...args: any[]) {
    console.log(`Llamando a ${String(context.name)} con:`, args);
    return metodoOriginal.call(this, ...args);
  };
}

class Calculadora {
  @registrar
  sumar(a: number, b: number): number {
    return a + b;
  }
}

const calc = new Calculadora();
calc.sumar(3, 4);
// Output: Llamando a sumar con: [3, 4]
```

7. TYPESCRIPT Y ECMASCRIPT: UNA RELACIÓN SIMBIÓTICA
===============================================================================

Una de las preguntas más comunes es: ¿TypeScript compite con JavaScript/ECMAScript?
La respuesta es no; de hecho, **TypeScript y ECMAScript tienen una relación simbiótica**.

- TypeScript implementa características de ECMAScript antes de que sean
  estándar (como clases, async/await, optional chaining)
- TypeScript propone ideas que a veces llegan a ECMAScript (como decoradores)
- TypeScript sirve como banco de pruebas para ideas que luego se estandarizan
- TypeScript siempre se dirige a versiones específicas de ECMAScript

**Timeline de convergencia:**
- 2012: TypeScript tiene clases → 2015: ES6 añade clases
- 2012: TypeScript tiene módulos → 2015: ES6 añade módulos
- 2014: TypeScript tiene async → 2017: ES2017 añade async/await
- 2015: TypeScript tiene decoradores → 2023: TC39 Stage 3 (TS 5.0)

---

📌 **TypeScript es como un laboratorio de innovación para JavaScript. Muchas
características se prueban primero en TypeScript antes de llegar al estándar.**

```typescript
// Características que TypeScript tuvo antes que ECMAScript:
// - Clases (TS 2012 → ES 2015)
// - Arrow functions (TS 2012 → ES 2015)
// - Optional chaining (TS 2016 → ES 2020) - como propuesta
// - Nullish coalescing (TS 2019 → ES 2020) - como propuesta
```

---

🔵 **Ejemplo 7: Versiones target de TypeScript**

```typescript
// En tsconfig.json
{
  "compilerOptions": {
    "target": "ES2022"
    // Opciones: ES3, ES5, ES2015, ES2016, ..., ES2022, ESNext
    // A mayor versión, menos transformaciones necesita el compilador
  }
}

// Si usas ES2022, características como:
const arreglo = [1, 2, 3];
console.log(arreglo.at(-1)); // 3 (ES2022)
// Se mantienen sin transformar en la salida
```

- TypeScript fue creado por Anders Hejlsberg en Microsoft, con desarrollo
  iniciado en 2010 y lanzamiento público en octubre de 2012

  en el estándar de facto para desarrollo web profesional
- TypeScript 5.x representa la versión más madura, rápida y completa del lenguaje

PREGUNTAS FRECUENTES
===============================================================================

❓ **¿Por qué Microsoft creó TypeScript y no usó C# para la web?**
C# para la web ya existía (Silverlight, Blazor), pero TypeScript nació para
mejorar JavaScript, no para reemplazarlo. La web funciona con JavaScript, y
TypeScript respeta eso añadiendo valor sin cambiar la plataforma subyacente.

❓ **¿TypeScript dejará de ser útil cuando JavaScript tenga tipos?**
Aunque TC39 ha discutido tipos opcionales en JavaScript (Type Annotations
proposal), estos serían solo comentarios o anotaciones sin efecto en
ejecución, muy similar a cómo funciona TypeScript. TypeScript seguirá siendo
necesario como compilador y herramienta de análisis.

❓ **¿Cuál fue la versión más polémica de TypeScript?**
Probablemente la 2.0 con `--strictNullChecks`, que rompió mucho código
existente. Sin embargo, a largo plazo fue la decisión más acertada para
la salud del ecosistema.

❓ **¿Sigue Microsoft desarrollando TypeScript activamente?**
Sí. El equipo de TypeScript en Microsoft continúa lanzando nuevas versiones
cada 2-3 meses con mejoras de rendimiento, correcciones y nuevas
características.
