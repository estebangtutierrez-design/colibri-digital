===============================================================================
1. ¿QUÉ SON LAS ASERCIONES DE TIPO?
===============================================================================

Una **aserción de tipo** (type assertion) es una forma de decirle a
TypeScript: "confía en mí, sé que este valor es de este tipo específico".
Es similar al "type casting" en otros lenguajes, pero sin transformación
en tiempo de ejecución.

**Importante**: Las aserciones no convierten el valor ni realizan
verificación en ejecución. Solo le indican al compilador que trate el
valor como un tipo específico.

```typescript
// Sintaxis con 'as' (recomendada)
let valor: unknown = "Hola, TypeScript";
let longitud = (valor as string).length;
console.log(longitud); // 16

// Sintaxis con '<>' (no recomendada en JSX/TSX)
let longitud2 = (<string>valor).length;
```

---

📌 **La sintaxis `as` es la preferida. La sintaxis `<>` puede causar
conflictos con JSX en React.**

```typescript
// ¿Por qué necesitamos aserciones?
// Cuando nosotros sabemos más que TypeScript sobre un tipo

// Ejemplo clásico: obtener un elemento del DOM
const miInput = document.getElementById("mi-input");
// TypeScript dice: tipo HTMLElement | null

// Nosotros sabemos que existe y es un HTMLInputElement
const input = document.getElementById("mi-input") as HTMLInputElement;
// Ahora TypeScript sabe que es HTMLInputElement
console.log(input.value); // ✅ Sin error
```

---

🔵 **Ejemplo 1: Aserción básica**

```typescript
function procesarEvento(evento: unknown): void {
  // Sin aserción, no podemos acceder a propiedades específicas
  // console.log(evento.key); // ❌ Error

  // Con aserción, le decimos a TypeScript que confíe en nosotros
  const eventoTeclado = evento as KeyboardEvent;
  console.log(`Tecla presionada: ${eventoTeclado.key}`);
}

// Uso simulado
procesarEvento({ key: "Enter", code: "Enter" });
// Output: Tecla presionada: Enter
```

2. ASERCIONES vs CONVERSIÓN EN OTROS LENGUAJES
===============================================================================

Es crucial entender la diferencia entre:

**Type assertion (TypeScript)**: Solo afecta al compilador. No genera código
de verificación en tiempo de ejecución.

**Type casting (Java/C#)**: Puede incluir verificación en tiempo de ejecución
y lanzar excepciones.

```typescript
// En TypeScript: la aserción es "gratis" (solo en compilación)
let dato: unknown = "texto";
let texto = dato as string;  // Sin verificación en ejecución

// Si nos equivocamos:
let numero = dato as number;  // Compila, pero en ejecución no es number
console.log(numero.toFixed(2)); // ❌ Error en ejecución: toFixed no es función
```

---

📌 **Las aserciones de tipo NO realizan conversión de datos. Solo le dicen
al compilador qué tipo tratar.**

```typescript
// ❌ Esto no convierte el valor
let valor: unknown = "123";
let numero = valor as number;

console.log(typeof numero); // "string" (¡sigue siendo string!)
console.log(numero);        // "123"
// No se ha convertido a número

// ✅ Para convertir realmente, usa funciones de JS
let numeroReal = Number("123");   // number 123
let numeroReal2 = parseInt("123"); // number 123
let numeroReal3 = +"123";         // number 123
```

```typescript
// La aserción solo cambia la perspectiva del compilador
const elemento = document.querySelector(".mi-clase") as HTMLElement;
// En ejecución: elemento sigue siendo el mismo nodo del DOM
// En compilación: TypeScript trata elemento como HTMLElement
```

---

🔵 **Ejemplo 2: Aserción incorrecta (peligro)**

```typescript
interface Gato {
  maullar(): void;
  nombre: string;
}

interface Perro {
  ladrar(): void;
  nombre: string;
}

function interactuarConMascota(mascota: Gato | Perro): void {
  // Aserción peligrosa si no verificamos
  (mascota as Perro).ladrar(); // ❌ Puede fallar si es un Gato

  // Mejor: usar type guards
  if ("maullar" in mascota) {
    mascota.maullar(); // ✅ TypeScript sabe que es Gato
  } else {
    mascota.ladrar();  // ✅ TypeScript sabe que es Perro
  }
}
```

3. CUÁNDO USAR ASERCIONES DE TIPO
===============================================================================

Las aserciones son apropiadas en estos escenarios:

**1. DOM: Cuando sabes que un elemento existe y tiene un tipo específico**

```typescript
const canvas = document.getElementById("miCanvas") as HTMLCanvasElement;
const ctx = canvas.getContext("2d");
```

**2. JSON.parse: Cuando sabes la estructura de los datos**

```typescript
interface Usuario {
  nombre: string;
  edad: number;
}

const datos = JSON.parse('{"nombre":"Ana","edad":25}') as Usuario;
console.log(datos.nombre); // "Ana"
```

**3. unknown: Cuando verificaste el tipo manualmente**

```typescript
function procesarDato(dato: unknown): number {
  if (typeof dato === "string") {
    return (dato as string).length;
  }
  if (typeof dato === "number") {
    return dato;
  }
  return 0;
}
```

---

📌 **Regla de oro: usa aserciones solo cuando sepas más que TypeScript sobre
el tipo de un valor.**

```typescript
// ✅ APROPIADO: DOM
const form = document.getElementById("loginForm") as HTMLFormElement;

// ✅ APROPIADO: fetch/API con tipos conocidos
interface ApiResponse {
  success: boolean;
  data: unknown;
}

async function obtenerUsuario(id: number): Promise<Usuario> {
  const res = await fetch(`/api/usuarios/${id}`);
  const data = (await res.json()) as Usuario;
  return data;
}

// ❌ INAPROPIADO: para evitar verificar tipos
function procesar(valor: string | number) {
  (valor as string).toUpperCase(); // ❌ Peligroso si valor es number
}

// ✅ Mejor: type guard
function procesarSeguro(valor: string | number) {
  if (typeof valor === "string") {
    valor.toUpperCase(); // ✅ TypeScript sabe que es string
  }
}
```

---

🔵 **Ejemplo 3: Aserciones en fetch API**

```typescript
interface Producto {
  id: number;
  nombre: string;
  precio: number;
  disponible: boolean;
}

async function obtenerProductos(): Promise<Producto[]> {
  const respuesta = await fetch("https://api.example.com/productos");

  if (!respuesta.ok) {
    throw new Error(`Error HTTP: ${respuesta.status}`);
  }

  // Aserción: confiamos en que la API devuelve Producto[]
  const productos = (await respuesta.json()) as Producto[];

  return productos;
}

// Uso
async function mostrarProductos(): Promise<void> {
  try {
    const productos = await obtenerProductos();
    productos.forEach(({ nombre, precio }) => {
      console.log(`${nombre}: $${precio}`);
    });
  } catch (error) {
    console.error("Error al obtener productos:", error);
  }
}
```

4. NON-NULL ASSERTION OPERATOR (!)
===============================================================================

El operador `!` (non-null assertion) le dice a TypeScript que un valor no es
`null` ni `undefined`. Es una forma de aserción específica para nulabilidad.

```typescript
let nombre: string | null = "Ana";
let longitud = nombre!.length; // Confiamos en que nombre no es null

// Sin !, TypeScript nos obligaría a verificar
if (nombre !== null) {
  let longitudSegura = nombre.length; // ✅ Verificación explícita
}
```

---

📌 **Usa `!` con moderación. La verificación explícita siempre es más segura.**

```typescript
// Escenarios típicos donde se usa !

// 1. Búsqueda en DOM que sabemos que existe
const header = document.getElementById("header")!;
header.textContent = "Nuevo título";

// 2. Propiedades que se inicializan después (post-constructor)
class Configuracion {
  private apiUrl!: string; // Se inicializará en init()

  init(): void {
    this.apiUrl = "https://api.example.com";
  }
}

// 3. Valores que verificamos antes pero TS no lo capta
function encontrarUsuario(id: number): Usuario | undefined {
  // ... búsqueda ...
  return usuario;
}

const usuario = encontrarUsuario(1)!;
console.log(usuario.nombre); // Confiamos en que existe
```

---

🔵 **Ejemplo 4: Non-null assertion en la práctica**

```typescript
// Sin non-null assertion:
function getElementoSeguro(id: string): HTMLElement {
  const elemento = document.getElementById(id);
  if (elemento === null) {
    throw new Error(`Elemento #${id} no encontrado`);
  }
  return elemento; // TypeScript sabe que no es null
}

// Con non-null assertion (más riesgoso pero más corto):
function getElementoRapido(id: string): HTMLElement {
  return document.getElementById(id)!;
  // Si no existe, será undefined en ejecución
}

// Uso responsable:
const app = document.getElementById("app");
if (app) {
  // Dentro de este bloque, app no es null
  // Podemos usar app sin aserción
}

// O con optional chaining y nullish coalescing:
const titulo = document.getElementById("titulo")?.textContent ?? "Sin título";
```

5. DOBLE ASERCIÓN
===============================================================================

A veces necesitas convertir entre dos tipos que no están relacionados.
La doble aserción (primero a `unknown`, luego al tipo deseado) fuerza la
conversión.

```typescript
// Aserción directa entre tipos no relacionados
// const x = "hola" as number; // ❌ Error: Conversión imposible

// Doble aserción (forzada)
const x = "hola" as unknown as number; // ✅ Compila, pero peligroso
```

---

📌 **La doble aserción es una bandera roja. Si la necesitas, probablemente
haya un problema de diseño en tu código.**

```typescript
// Escenario raro pero posible: trabajar con APIs externas
interface UserData {
  id: number;
  nombre: string;
}

interface ApiResponse {
  status: string;
  data: unknown;
}

async function fetchUser(id: number): Promise<UserData> {
  const res = await fetch(`/api/users/${id}`);
  const response = (await res.json()) as ApiResponse;

  // Doble aserción porque sabemos que response.data tiene la estructura
  return response.data as unknown as UserData;
}

// Alternativa mejor: verificación en tiempo de ejecución
function isUserData(data: unknown): data is UserData {
  return (
    typeof data === "object" &&
    data !== null &&
    "id" in data &&
    "nombre" in data &&
    typeof (data as UserData).id === "number" &&
    typeof (data as UserData).nombre === "string"
  );
}

async function fetchUserSeguro(id: number): Promise<UserData> {
  const res = await fetch(`/api/users/${id}`);
  const response = await res.json();

  if (!isUserData(response.data)) {
    throw new Error("Respuesta de API inválida");
  }

  return response.data; // ✅ TypeScript sabe que es UserData
}
```

---

🔵 **Ejemplo 5: Cuándo evitar la doble aserción**

```typescript
// ❌ MAL: doble aserción innecesaria
function procesarEntrada(entrada: string): number {
  return (entrada as unknown as number);
}

// ✅ BIEN: usar conversión real
function procesarEntradaCorrecta(entrada: string): number {
  const num = Number(entrada);
  if (isNaN(num)) {
    throw new Error("Entrada no es un número válido");
  }
  return num;
}

// ❌ MAL: abusar de aserciones para silenciar errores
interface Producto {
  nombre: string;
  precio: number;
}

function mostrarProducto(producto: unknown): void {
  const p = producto as Producto; // Sin verificar
  console.log(p.nombre); // Puede fallar en ejecución
}

// ✅ BIEN: verificar antes de usar
function mostrarProductoSeguro(producto: unknown): void {
  if (producto && typeof producto === "object" && "nombre" in producto) {
    const p = producto as Producto;
    console.log(p.nombre);
  }
}
```

6. ANOTACIONES DE TIPO AVANZADAS
===============================================================================

Más allá de los tipos básicos, TypeScript permite anotaciones detalladas.

**Tipos unión (`|`)**: Una variable puede ser de uno de varios tipos.

```typescript
let id: string | number;
id = "ABC-123";  // ✅
id = 456;        // ✅
id = true;       // ❌ Error

function imprimirId(id: string | number): void {
  console.log(`ID: ${id}`);
}
```

**Tipos intersección (`&`)**: Combina múltiples tipos en uno.

```typescript
interface Nombrable {
  nombre: string;
}

interface Edad {
  edad: number;
}

type Persona = Nombrable & Edad;
// { nombre: string; edad: number }

const persona: Persona = { nombre: "Ana", edad: 25 };
```

---

📌 **Los tipos unión (`|`) e intersección (`&`) son herramientas poderosas
para componer tipos complejos.**

```typescript
// Tipo unión con literales (discriminados)
type EstadoPedido = "pendiente" | "enviado" | "entregado" | "cancelado";

function actualizarEstado(estado: EstadoPedido): void {
  switch (estado) {
    case "pendiente":
      console.log("Pedido pendiente de procesar");
      break;
    case "enviado":
      console.log("Pedido enviado al cliente");
      break;
    case "entregado":
      console.log("Pedido entregado");
      break;
    case "cancelado":
      console.log("Pedido cancelado");
      break;
  }
}
```

---

🔵 **Ejemplo 6: Anotaciones de tipos complejos**

```typescript
// Parámetros con tipos unión y retorno condicional
function obtenerLongitud(valor: string | unknown[]): number {
  return valor.length;
}

// Objetos con propiedades opcionales
interface Configuracion {
  url: string;
  puerto?: number;        // Opcional
  timeout?: number;       // Opcional
  readonly apiKey: string; // Solo lectura
}

function conectar(config: Configuracion): void {
  const puerto = config.puerto ?? 443;
  const timeout = config.timeout ?? 5000;
  console.log(`Conectando a ${config.url}:${puerto}`);
  // config.apiKey = "nueva"; // ❌ Error: readonly
}

// Tipos para funciones
type Callback<T> = (error: Error | null, resultado?: T) => void;

function leerArchivo(ruta: string, callback: Callback<string>): void {
  // Simulación
  if (ruta === "") {
    callback(new Error("Ruta vacía"));
  } else {
    callback(null, "Contenido del archivo");
  }
}

// Uso
leerArchivo("datos.txt", (error, contenido) => {
  if (error) {
    console.error(error.message);
  } else {
    console.log(contenido);
  }
});
```

7. EL OPERADOR SATISFIES (TS 4.9+)
===============================================================================

El operador `satisfies` verifica que el tipo de una expresión cumple con un
tipo, pero infiere el tipo más específico posible, no el tipo general.

```typescript
// Sin satisfies: inferencia general
type Color = string | { red: number; green: number; blue: number };

const color1: Color = "rojo";
// color1.toUpperCase(); // ❌ Error: Object is possibly string

// Con satisfies: inferencia específica
const color2 = "rojo" satisfies Color;
// color2.toUpperCase(); // ✅ TypeScript sabe que es string literal

const color3 = { red: 255, green: 0, blue: 0 } satisfies Color;
console.log(color3.red); // ✅ TypeScript sabe que tiene propiedad red
```

---

📌 **`satisfies` verifica que el valor cumple el tipo, pero mantiene el tipo
inferido más específico, no el tipo general.**

```typescript
// Diferencia clave: type annotation vs satisfies

// Type annotation: fuerza el tipo amplio
type RGB = { red: number; green: number; blue: number };
type Color = RGB | string;

let colorAnotado: Color = { red: 255, green: 0, blue: 0 };
// colorAnotado.red; // ❌ Error: Property 'red' does not exist on type 'Color'

// satisfies: verifica pero mantiene el tipo específico
const colorSatisfies = { red: 255, green: 0, blue: 0 } satisfies Color;
console.log(colorSatisfies.red); // ✅ 255 (RGB específico)

// satisfies con strings literales
let paleta = {
  primary: "blue",
  secondary: "green",
  accent: "#FF0000",
} satisfies Record<string, Color>;

// paleta.primary.toUpperCase(); // ✅ string, no string | RGB
```

---

🔵 **Ejemplo 7: satisfies en la práctica**

```typescript
// Caso 1: Validar estructura de objetos anidados
type Coordenadas = { lat: number; lng: number };
type Lugar = { nombre: string; ubicacion: Coordenadas } | Coordenadas;

const lugar = {
  nombre: "Torre Eiffel",
  ubicacion: { lat: 48.8584, lng: 2.2945 },
} satisfies Lugar;

console.log(lugar.nombre);          // ✅ string
console.log(lugar.ubicacion.lat);   // ✅ number

// Caso 2: Mapas con tipos específicos
type Paleta = Record<string, string | { light: string; dark: string }>;

const colores = {
  primary: "#3498db",
  background: { light: "#fff", dark: "#333" },
} satisfies Paleta;

console.log(colores.primary.toUpperCase());        // ✅ string
console.log(colores.background.light);             // ✅ string

// Caso 3: Eventos tipados
type EventHandler = { (e: MouseEvent): void } | { (e: KeyboardEvent): void };

const manejador = (e: MouseEvent) => {
  console.log(`Click en (${e.clientX}, ${e.clientY})`);
} satisfies EventHandler;
```

8. BUENAS PRÁCTICAS CON ASERCIONES
===============================================================================

**1. Prefiere type guards sobre aserciones**

```typescript
// ❌ Aserción directa
function procesar(dato: unknown): string {
  return (dato as string).toUpperCase();
}

// ✅ Type guard
function esString(valor: unknown): valor is string {
  return typeof valor === "string";
}

function procesarSeguro(dato: unknown): string {
  if (esString(dato)) {
    return dato.toUpperCase(); // ✅ Sin aserción necesaria
  }
  return String(dato);
}
```

**2. Verifica en tiempo de ejecución cuando sea necesario**

```typescript
interface RespuestaAPI {
  exito: boolean;
  datos: unknown;
}

function esUsuario(datos: unknown): datos is Usuario {
  return (
    typeof datos === "object" &&
    datos !== null &&
    "nombre" in datos &&
    "email" in datos
  );
}

function manejarRespuesta(respuesta: RespuestaAPI): void {
  if (respuesta.exito && esUsuario(respuesta.datos)) {
    console.log(respuesta.datos.nombre); // ✅ Seguro
  }
}
```

**3. Evita aserciones en tipos unión cuando puedas usar narrowing**

```typescript
// ❌ Aserción innecesaria
function saludar( persona: { nombre: string } | { alias: string }): string {
  return (persona as { nombre: string }).nombre;
}

// ✅ Type narrowing con 'in'
function saludarSeguro(persona: { nombre: string } | { alias: string }): string {
  if ("nombre" in persona) {
    return persona.nombre; // ✅ TypeScript sabe que es { nombre: string }
  }
  return persona.alias;    // ✅ TypeScript sabe que es { alias: string }
}
```

---

📌 **Las aserciones son una herramienta, no una solución. Abusar de ellas
anula los beneficios de TypeScript.**

```typescript
// Progresión: de inseguro a seguro

// Nivel 1: any (sin seguridad)
function procesar(dato: any): any {
  return dato.nombre;
}

// Nivel 2: unknown + aserción (algo de seguridad)
function procesarMejor(dato: unknown): string {
  return (dato as { nombre: string }).nombre;
}

// Nivel 3: tipo específico + verificación (máxima seguridad)
function procesarMejorAun(dato: unknown): string {
  if (typeof dato === "object" && dato !== null && "nombre" in dato) {
    return (dato as { nombre: string }).nombre;
  }
  return "Desconocido";
}

// Nivel 4: type guard + interface (óptimo)
interface ConNombre {
  nombre: string;
}

function esConNombre(valor: unknown): valor is ConNombre {
  return (
    typeof valor === "object" &&
    valor !== null &&
    "nombre" in valor &&
    typeof (valor as ConNombre).nombre === "string"
  );
}

function procesarOptimo(dato: unknown): string {
  if (esConNombre(dato)) {
    return dato.nombre; // ✅ Sin aserción
  }
  return "Desconocido";
}
```

---

🔵 **Ejemplo 8: Resumen de buenas prácticas**

```typescript
// Sistema de validación con aserciones controladas
interface Producto {
  id: number;
  nombre: string;
  precio: number;
  categoria: "electronica" | "ropa" | "alimentos";
}

function esProducto(valor: unknown): valor is Producto {
  if (typeof valor !== "object" || valor === null) return false;
  const p = valor as Record<string, unknown>;
  return (
    typeof p.id === "number" &&
    typeof p.nombre === "string" &&
    typeof p.precio === "number" &&
    (p.categoria === "electronica" || p.categoria === "ropa" || p.categoria === "alimentos")
  );
}

function validarYProcesar(datos: unknown[]): Producto[] {
  return datos.filter(esProducto).map((producto) => {
    // Aquí la aserción es segura porque filtramos con type guard
    const p = producto as Producto;
    console.log(`Procesando: ${p.nombre} - $${p.precio}`);
    return p;
  });
}

// Uso
const datosEntrada = [
  { id: 1, nombre: "Laptop", precio: 1200, categoria: "electronica" },
  { id: 2, nombre: "Camisa", precio: 50, categoria: "ropa" },
  { id: 3, nombre: "Invalido", precio: "gratis", categoria: "alimentos" },
];

const productosValidos = validarYProcesar(datosEntrada);
console.log(`Productos válidos: ${productosValidos.length}`);
```

RESUMEN DEL CAPÍTULO
===============================================================================

- Las aserciones de tipo (`as`) le dicen al compilador "confía en mí"
- Las aserciones NO transforman datos en tiempo de ejecución
- El non-null assertion operator (`!`) afirma que un valor no es null/undefined
- La doble aserción (`as unknown as T`) fuerza tipos no relacionados (evitar)
- `satisfies` verifica el cumplimiento de un tipo pero mantiene el tipo inferido
- Prefiere type guards sobre aserciones directas
- Verifica tipos en tiempo de ejecución cuando los datos vienen de fuentes externas
- Las anotaciones con tipos unión e intersección permiten tipado flexible y preciso

PREGUNTAS FRECUENTES
===============================================================================

❓ **¿Cuándo debo usar as vs <>?**
Usa siempre `as`. La sintaxis `<>` puede causar conflictos con JSX en
React y es menos legible.

❓ **¿Las aserciones afectan el rendimiento?**
No. Las aserciones solo afectan al compilador. Desaparecen completamente
en el JavaScript generado.

❓ **¿Qué es más seguro: `satisfies` o type annotation?**
`satisfies` es más seguro porque verifica el tipo sin perder información
específica. La type annotation fuerza el tipo general.

❓ **¿Puedo usar aserciones para convertir entre tipos incompatibles?**
Solo con doble aserción (`as unknown as T`), pero es una mala práctica
que indica un problema de diseño.
