===============================================================================
1. ¿QUÉ ES TSCONFIG.JSON?
===============================================================================

`tsconfig.json` es el archivo de configuración de TypeScript. Se coloca en la
raíz del proyecto y le indica al compilador `tsc`:

- Qué archivos compilar
- Hacia qué versión de JavaScript transpilar
- Qué tan estricta debe ser la verificación de tipos
- Dónde colocar los archivos compilados
- Y muchas otras opciones

Sin un `tsconfig.json`, el compilador `tsc` usa valores por defecto y solo
compila los archivos que se le pasan como argumento.

---

📌 **El tsconfig.json es el centro de control de tu proyecto TypeScript. Sin
él, estás volando a ciegas.**

```bash
# Generar un tsconfig.json por defecto
npx tsc --init

# Esto crea un archivo con todas las opciones comentadas
# Es el mejor punto de partida para aprender
```

```json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "commonjs",
    "strict": true,
    "outDir": "./dist",
    "rootDir": "./src",
    "esModuleInterop": true,
    "skipLibCheck": true,
    "forceConsistentCasingInFileNames": true
  },
  "include": ["src/**/*"],
  "exclude": ["node_modules", "dist"]
}
```

---

🔵 **Ejemplo 1: Generando tsconfig.json por primera vez**

```bash
# En la raíz de tu proyecto
$ mkdir mi-proyecto && cd mi-proyecto
$ npm init -y
$ npm install --save-dev typescript
$ npx tsc --init

Created a new tsconfig.json with:
  target: es2016
  module: commonjs
  strict: true
  esModuleInterop: true
  skipLibCheck: true
  forceConsistentCasingInFileNames: true

You can learn more at https://aka.ms/tsconfig
```

===============================================================================
2. OPCIONES ESENCIALES DEL COMPILADOR
===============================================================================

Vamos a explorar las opciones más importantes que podemos configurar:

**target**: Especifica la versión de JavaScript de salida.
- Opciones: ES3, ES5, ES2015, ES2016, ..., ES2022, ESNext
- Recomendación: ES2022 (moderno) o ES6 si necesitas compatibilidad amplia

**module**: Define el sistema de módulos para el JavaScript de salida.
- Opciones: commonjs, amd, umd, es2015, es2020, es2022, esnext, node16, bundler
- Recomendación: "commonjs" para Node.js, "esnext" o "bundler" para frontend

**strict**: Activa todas las opciones de verificación estricta de tipos.
- Incluye: noImplicitAny, strictNullChecks, strictFunctionTypes, etc.
- Recomendación: SIEMPRE true

---

📌 **La opción `strict: true` es la configuración más importante que puedes
activar. Te da el máximo nivel de seguridad de tipos.**

```json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "commonjs",
    "strict": true,
    "outDir": "./dist",
    "rootDir": "./src",
    "esModuleInterop": true
  }
}
```

```bash
# Estructura de proyecto con outDir y rootDir
mi-proyecto/
├── src/              # ← rootDir: código fuente TS
│   ├── index.ts
│   └── utils.ts
├── dist/             # ← outDir: salida compilada JS
│   ├── index.js
│   └── utils.js
├── tsconfig.json
└── package.json
```

---

🔵 **Ejemplo 2: Configuración para Node.js backend**

```json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "commonjs",
    "strict": true,
    "outDir": "./dist",
    "rootDir": "./src",
    "esModuleInterop": true,
    "resolveJsonModule": true,
    "declaration": true,
    "declarationMap": true,
    "sourceMap": true
  },
  "include": ["src/**/*"],
  "exclude": ["node_modules", "dist", "**/*.test.ts"]
}
```

===============================================================================
3. OPCIONES DE SALIDA Y DEPURACIÓN
===============================================================================

**outDir**: Directorio donde se colocarán los archivos JavaScript compilados.
Mantiene la estructura de carpetas del source.

**rootDir**: Directorio raíz del código fuente. tsc preserva la estructura de
carpetas relativa a esta ruta.

**sourceMap**: Genera archivos .js.map que permiten depurar el código
TypeScript original en lugar del JavaScript compilado.

**declaration**: Genera archivos `.d.ts` con las declaraciones de tipos.

**removeComments**: Elimina los comentarios del JavaScript de salida.

---

📌 **Los sourceMaps son esenciales para la depuración. Sin ellos, los errores
en ejecución apuntan al JavaScript compilado, no a tu código TypeScript.**

```json
{
  "compilerOptions": {
    "outDir": "./dist",
    "rootDir": "./src",
    "sourceMap": true,
    "declaration": true,
    "declarationMap": true,
    "removeComments": false
  }
}
```

```bash
# Archivos generados con sourceMap y declaration
dist/
├── index.js          # JavaScript compilado
├── index.js.map      # SourceMap (depuración)
├── index.d.ts        # Declaración de tipos
├── index.d.ts.map    # Mapa de declaración
├── utils.js
├── utils.js.map
├── utils.d.ts
└── utils.d.ts.map
```

---

🔵 **Ejemplo 3: SourceMap en acción**

```typescript
// src/calculadora.ts
export function dividir(a: number, b: number): number {
  if (b === 0) {
    throw new Error("No se puede dividir por cero");
  }
  return a / b;
}
```

```bash
# Sin sourceMap, el stack trace apunta al .js
Error: No se puede dividir por cero
    at dividir (dist/calculadora.js:5:15)

# Con sourceMap, apunta al .ts original
Error: No se puede dividir por cero
    at dividir (src/calculadora.ts:4:11)
```

===============================================================================
4. OPCIONES DE VERIFICACIÓN ESTRICTA
===============================================================================

TypeScript ofrece múltiples niveles de verificación. Con `strict: true` se
activan todas, pero podemos personalizarlas:

| Opción | Descripción | Default sin strict |
|--------|-------------|-------------------|
| `strictNullChecks` | null/undefined no asignables a otros tipos | false |
| `noImplicitAny` | Error si una expresión tiene tipo `any` implícito | false |
| `strictFunctionTypes` | Verificación estricta de tipos en funciones | false |
| `strictBindCallApply` | Verificación de tipos en bind/call/apply | false |
| `strictPropertyInitialization` | Propiedades de clase deben inicializarse | false |
| `noUnusedLocals` | Error si hay variables locales no usadas | false |
| `noUnusedParameters` | Error si hay parámetros no usados | false |
| `noImplicitReturns` | Error si faltan returns en todos los caminos | false |

---

📌 **Puedes activar strict y luego desactivar opciones individuales, pero es
raro que necesites hacerlo. Confía en el modo estricto.**

```json
{
  "compilerOptions": {
    "strict": true,
    // Estas están implícitamente activadas con strict
    // "strictNullChecks": true,
    // "noImplicitAny": true,
    // "strictFunctionTypes": true,
    // "strictBindCallApply": true,
    // "noImplicitThis": true

    // Opciones adicionales recomendadas
    "noUnusedLocals": true,
    "noUnusedParameters": true,
    "noImplicitReturns": true,
    "noFallthroughCasesInSwitch": true
  }
}
```

---

🔵 **Ejemplo 4: Cómo strictNullChecks salva tu código**

```typescript
// Sin strictNullChecks
function obtenerLongitud( texto?: string ) {
  return texto.length;  // ¡Error en ejecución si texto es undefined!
}

// Con strictNullChecks
function obtenerLongitud(texto?: string) {
  if (texto === undefined) {
    return 0;
  }
  return texto.length;  // ✅ TypeScript sabe que aquí texto es string
}

// O usando optional chaining
function obtenerLongitud(texto?: string): number {
  return texto?.length ?? 0;  // ✅ Moderno y seguro
}
```

===============================================================================
5. INCLUDE, EXCLUDE Y FILES
===============================================================================

Controlan qué archivos compila TypeScript:

**include**: Patrones de archivos a incluir en la compilación.
**exclude**: Patrones a excluir (tiene prioridad sobre include).
**files**: Lista explícita de archivos a compilar (menos flexible).

```json
{
  "include": ["src/**/*", "test/**/*"],
  "exclude": ["node_modules", "dist", "**/*.spec.ts", "coverage"],
  // "files": ["src/main.ts", "src/utils.ts"]  // Alternativa explícita
}
```

---

📌 **Si no especificas include, TypeScript compila todos los archivos .ts
en la carpeta del tsconfig.json (excepto los excluidos).**

```json
// Configuración típica para un proyecto
{
  "compilerOptions": { /* ... */ },
  "include": ["src"],
  "exclude": [
    "node_modules",
    "dist",
    "**/*.test.ts",
    "**/*.spec.ts"
  ]
}
```

---

🔵 **Ejemplo 5: Include con múltiples directorios**

```json
{
  "compilerOptions": {
    "outDir": "./dist",
    "rootDir": "./"
  },
  "include": [
    "src",           // Todo el código fuente
    "shared"         // Tipos compartidos
  ],
  "exclude": [
    "node_modules",
    "dist",
    "src/experimental"  // Código experimental que no queremos compilar
  ]
}
```

```bash
# Estructura del proyecto
proyecto/
├── src/
│   ├── main.ts       # ✅ Incluido
│   └── experimental/
│       └── test.ts   # ❌ Excluido
├── shared/
│   └── types.ts      # ✅ Incluido
├── dist/             # ❌ Excluido
└── node_modules/     # ❌ Excluido
```

6. CONFIGURACIONES POR DEFECTO Y PERSONALIZADAS
===============================================================================

TypeScript permite extender configuraciones base, lo que es útil cuando tienes
múltiples proyectos con la misma base de configuración.

```json
// tsconfig.base.json
{
  "compilerOptions": {
    "strict": true,
    "target": "ES2022",
    "module": "commonjs",
    "esModuleInterop": true,
    "skipLibCheck": true,
    "forceConsistentCasingInFileNames": true
  }
}
```

```json
// tsconfig.json (proyecto específico)
{
  "extends": "./tsconfig.base.json",
  "compilerOptions": {
    "outDir": "./dist",
    "rootDir": "./src"
  },
  "include": ["src"]
}
```

---

📌 **TypeScript también soporta configuraciones por referencia de proyecto
(project references), ideal para monorepos.**

```json
// Configuración para frontend (React)
{
  "compilerOptions": {
    "target": "ES2020",
    "module": "ESNext",
    "moduleResolution": "bundler",
    "jsx": "react-jsx",
    "strict": true,
    "outDir": "./dist",
    "sourceMap": true
  },
  "include": ["src"]
}
```

```json
// Configuración para backend (Node.js)
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "commonjs",
    "outDir": "./dist",
    "rootDir": "./src",
    "strict": true,
    "esModuleInterop": true,
    "resolveJsonModule": true,
    "sourceMap": true
  },
  "include": ["src"]
}
```

---

🔵 **Ejemplo 6: Configuración completa comentada**

```json
{
  "compilerOptions": {
    /* Versión de JS de salida */
    "target": "ES2022",

    /* Sistema de módulos */
    "module": "commonjs",

    /* Directorios */
    "rootDir": "./src",
    "outDir": "./dist",

    /* Verificación estricta */
    "strict": true,
    "noUnusedLocals": true,
    "noUnusedParameters": true,

    /* Compatibilidad */
    "esModuleInterop": true,
    "resolveJsonModule": true,

    /* Depuración */
    "sourceMap": true,
    "declaration": true,
    "declarationMap": true,

    /* Misc */
    "skipLibCheck": true,
    "forceConsistentCasingInFileNames": true
  },
  "include": ["src"],
  "exclude": ["node_modules", "dist", "**/*.test.ts"]
}
```

7. SOLUCIÓN DE PROBLEMAS DE CONFIGURACIÓN
===============================================================================

**Problema 1: "Cannot write file" o conflictos de directorio**

```json
// Asegúrate de que rootDir y outDir no se solapen
{
  "compilerOptions": {
    "rootDir": "./src",
    "outDir": "./dist"
    // ✅ No hay solapamiento
  }
}

// ❌ Incorrecto: rootDir incluye a outDir
// "rootDir": ".",
// "outDir": "./dist"
```

**Problema 2: Archivos no se compilan**

```bash
# Verifica la configuración de include
npx tsc --showConfig   # Muestra la configuración efectiva

# Para depurar qué archivos se están compilando
npx tsc --listFiles    # Lista todos los archivos incluidos
```

**Problema 3: Módulos no se resuelven correctamente**

```json
// Para Node.js
{
  "compilerOptions": {
    "module": "commonjs",
    "moduleResolution": "node"
  }
}

// Para frontend con bundler (Vite, Webpack)
{
  "compilerOptions": {
    "module": "ESNext",
    "moduleResolution": "bundler"
  }
}
```

---

📌 **Usa `npx tsc --showConfig` para ver exactamente cómo TypeScript interpreta
tu configuración. Es la mejor herramienta de depuración.**

```bash
$ npx tsc --showConfig
{
  "compilerOptions": {
    "target": "es2022",
    "module": "commonjs",
    "strict": true,
    "esModuleInterop": true,
    "skipLibCheck": true,
    "forceConsistentCasingInFileNames": true,
    "outDir": "./dist",
    "rootDir": "./src"
  },
  "files": [
    "./src/index.ts",
    "./src/utils.ts"
  ],
  "include": ["src"]
}
```

---

🔵 **Ejemplo 7: Depurando configuración**

```bash
# Ver la configuración efectiva después de aplicar extends
$ npx tsc --showConfig

# Ver todos los archivos que se compilarán
$ npx tsc --listFiles | head -20
/home/usuario/proyecto/src/index.ts
/home/usuario/proyecto/src/utils.ts
/home/usuario/proyecto/node_modules/typescript/lib/lib.es2022.d.ts
# ... más archivos de declaración

# Compilar sin generar archivos (solo verificar errores)
$ npx tsc --noEmit
# ✅ No output = no errors!
```

===============================================================================
RESUMEN DEL CAPÍTULO
===============================================================================

- `tsconfig.json` es el archivo de configuración central de TypeScript
- `npx tsc --init` genera un tsconfig.json por defecto
- `target` define la versión de JavaScript de salida (ES2022 recomendado)
- `module` define el sistema de módulos (commonjs para Node.js)
- `strict: true` activa todas las verificaciones estrictas de tipos
- `outDir` y `rootDir` organizan la estructura de entrada/salida
- `sourceMap` habilita la depuración en TypeScript original
- `include` y `exclude` controlan qué archivos se compilan
- Se puede extender configuraciones base con `extends`
- `--showConfig` y `--listFiles` son herramientas de depuración esenciales

===============================================================================
PREGUNTAS FRECUENTES
===============================================================================

❓ **¿Es obligatorio tener un tsconfig.json?**
No, pero sin él tsc usa valores por defecto y debes pasar los archivos a
compilar como argumentos. Para proyectos serios es indispensable.

❓ **¿Puedo tener múltiples tsconfig.json en un proyecto?**
Sí. Es común tener diferentes configuraciones para desarrollo, producción
y pruebas. Se usa `extends` para compartir la base.

❓ **¿Qué hace exactamente strict: true?**
Activa 5 opciones: strictNullChecks, noImplicitAny, strictFunctionTypes,
strictBindCallApply y noImplicitThis. Juntas proporcionan la verificación
de tipos más rigurosa.

❓ **¿Debo incluir node_modules en exclude?**
Aunque tsc ignora node_modules por defecto, es buena práctica incluirlo
explícitamente para claridad.
