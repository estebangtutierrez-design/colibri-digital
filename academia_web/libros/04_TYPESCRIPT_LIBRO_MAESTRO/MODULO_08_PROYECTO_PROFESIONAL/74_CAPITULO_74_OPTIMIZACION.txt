1. TYPESCRIPT Y EL RENDIMIENTO
===============================================================================

TypeScript opera en dos dimensiones de rendimiento: tiempo de compilación
(typescript compiler) y tiempo de ejecución (JavaScript generado).

1.1 RENDIMIENTO EN TIEMPO DE COMPILACIÓN

El compilador de TypeScript (tsc) realiza un trabajo significativo:
análisis sintáctico, resolución de módulos, inferencia de tipos, verificación
de tipos y generación de código. Proyectos grandes pueden tardar minutos en
compilar.

```typescript
// Factores que afectan la velocidad de compilación:
// 1. Tipos complejos con inferencia profunda
// 2. Módulos con muchas importaciones
// 3. Uso excesivo de tipos condicionales
// 4. Proyectos con muchos archivos (10,000+)
// 5. Dependencias con archivos de declaración grandes

// LENTO: Tipos excesivamente complejos
type DeepPartial<T> = T extends object
  ? { [P in keyof T]?: DeepPartial<T[P]> }
  : T;

// Muy lento con tipos grandes
type SuperComplex<T> = T extends (...args: infer A) => infer R
  ? (...args: A) => R
  : T extends Promise<infer U>
  ? Promise<DeepPartial<U>>
  : T;

// RÁPIDO: Tipos planos y específicos
interface UsuarioSimple {
  id: string;
  nombre: string;
  email: string;
}
```

1.2 tsconfig.json PARA RENDIMIENTO

La configuración del compilador impacta directamente en la velocidad.

```json
{
  "compilerOptions": {
    "target": "ES2020",
    "module": "ES2020",
    "moduleResolution": "bundler",
    "strict": true,
    "skipLibCheck": true,
    "forceConsistentCasingInFileNames": true,
    "resolveJsonModule": true,
    "isolatedModules": true,
    "esModuleInterop": true,
    "incremental": true,
    "tsBuildInfoFile": ".tsbuildinfo"
  },
  "include": ["src"],
  "exclude": ["node_modules", "dist", "**/*.test.ts"]
}
```

Opciones clave para rendimiento:

- `skipLibCheck: true` — Omite la verificación de tipos en archivos .d.ts
- `incremental: true` — Compilación incremental, solo recompila lo que cambió
- `isolatedModules: true` — Trata cada archivo como módulo independiente
- `moduleResolution: "bundler"` — Resolución más rápida para bundlers modernos
- `forceConsistentCasingInFileNames: true` — Previene errores en sistemas case-insensitive

1.3 PROJECT REFERENCES

Para proyectos muy grandes, las referencias entre proyectos permiten compilar
por partes.

```json
// tsconfig.base.json
{
  "compilerOptions": {
    "composite": true,
    "declaration": true,
    "declarationMap": true
  }
}

// packages/core/tsconfig.json
{
  "extends": "../../tsconfig.base.json",
  "compilerOptions": {
    "outDir": "./dist",
    "rootDir": "./src"
  },
  "references": []
}

// packages/web/tsconfig.json
{
  "extends": "../../tsconfig.base.json",
  "compilerOptions": {
    "outDir": "./dist",
    "rootDir": "./src"
  },
  "references": [
    { "path": "../core" }
  ]
}
```

2. MODO ESTRICTO (STRICT MODE)
===============================================================================

El modo estricto no solo mejora la seguridad de tipos, sino que también
ayuda al compilador a generar código más eficiente.

2.1 OPCIONES DE STRICT MODE

```json
{
  "compilerOptions": {
    "strict": true
  }
}
```

Esto habilita todas las siguientes opciones:

- `strictNullChecks: true` — Previene null/undefined no controlados
- `strictFunctionTypes: true` — Verificación covariante/contravariante
- `strictBindCallApply: true` — Verificación de bind/call/apply
- `strictPropertyInitialization: true` — Propiedades deben inicializarse

```typescript
// Sin strict mode: errores silenciosos que pueden causar crashes
function longitud(str: string): number {
  return str.length;
}
longitud(null); // Runtime error: Cannot read property 'length' of null

// Con strict mode: error en compilación
function longitud(str: string): number {
  return str.length;
}
// longitud(null); // Error: Argument of type 'null' is not assignable
//                 // to parameter of type 'string'

// Con strictNullChecks, obligas a manejar null
function longitudSegura(str: string | null): number {
  return str?.length ?? 0;
}
```

2.2 NO UNCHECKED INDEXED ACCESS

```typescript
// strict: true permite detectar accesos inseguros a índices
interface Diccionario {
  [clave: string]: number;
}

const dict: Diccionario = {};
const valor = dict["clave"];
// Sin strict, valor es number (aunque realmente es undefined)
// Con strict y noUncheckedIndexedAccess, valor es number | undefined
```

3. TREE SHAKING
===============================================================================

Tree shaking es el proceso de eliminar código muerto (no utilizado) del
bundle final. TypeScript y los bundlers modernos (Webpack, Rollup, esbuild)
lo soportan.

3.1 CONDICIONES PARA TREE SHAKING

```typescript
// ❌ NO se puede hacer tree shaking de:
export default class MiClase { /* ... */ }

// ✅ Sí se puede hacer tree shaking de:
export class MiClase { /* ... */ }
export function miFuncion() { /* ... */ }
export interface MiInterface { /* ... */ }

// Importaciones específicas permiten tree shaking
// ❌ Importa todo el módulo
import * as utils from "./utils";

// ✅ Importa solo lo necesario
import { formatoFecha, formatoMoneda } from "./utils";
```

3.2 SIDE EFFECTS

Para que el tree shaking funcione, los módulos deben ser "pure" (sin efectos
secundarios al importarlos).

```json
// package.json
{
  "sideEffects": false,
  // O específico
  "sideEffects": [
    "**/*.css",
    "**/*.scss"
  ]
}
```

```typescript
// ❌ Efecto secundario: esto impide el tree shaking
Array.prototype.ultimo = function<T>() {
  return this[this.length - 1];
};

// ✅ Sin efectos secundarios: exportación pura
export function ultimoElemento<T>(arr: T[]): T | undefined {
  return arr[arr.length - 1];
}
```

4. LAZY LOADING Y CÓDIGO BAJO DEMANDA
===============================================================================

Cargar módulos solo cuando se necesitan reduce el tamaño inicial del bundle.

```typescript
// CARGA INMEDIATA (EAGER)
import { FuncionPesada } from "./funcion-pesada";
FuncionPesada.ejecutar();

// CARGA DIFERIDA (LAZY) con import() dinámico
async function cargarModuloPesado(): Promise<void> {
  const modulo = await import("./funcion-pesada");
  modulo.FuncionPesada.ejecutar();
}

// Solo se carga cuando el usuario hace clic
document.getElementById("btn-cargar")?.addEventListener("click", async () => {
  const { renderizarGrafico } = await import("./graficos");
  renderizarGrafico("container");
});
```

4.1 PATRÓN DE CARGA DIFERIDA

```typescript
// Servicio de carga diferida
class ServicioDiferido {
  private modulosCache = new Map<string, any>();

  async cargar<T>(nombre: string): Promise<T> {
    if (this.modulosCache.has(nombre)) {
      return this.modulosCache.get(nombre) as T;
    }

    const modulo = await this.importarModulo(nombre);
    this.modulosCache.set(nombre, modulo);
    return modulo as T;
  }

  private async importarModulo(nombre: string): Promise<any> {
    switch (nombre) {
      case "graficos":
        return import("./graficos");
      case "exportacion":
        return import("./exportacion");
      case "admin":
        return import("./admin");
      default:
        throw new Error(`Módulo desconocido: ${nombre}`);
    }
  }
}

const cargador = new ServicioDiferido();

// Uso: el módulo solo se carga cuando se necesita
async function mostrarGrafico(): Promise<void> {
  const { graficoBarras } = await cargador.cargar<typeof import("./graficos")>("graficos");
  graficoBarras.render();
}
```

5. OPTIMIZACIÓN DE TIPOS
===============================================================================

Tipos mal diseñados pueden ralentizar la compilación y confundir al
desarrollador.

5.1 EVITAR TIPOS CONDICIONALES ANIDADOS

```typescript
// ❌ LENTO: Tipos condicionales excesivamente anidados
type TransformarCadena<T> =
  T extends `${infer A}.${infer B}`
    ? A extends `${infer C}-${infer D}`
      ? `${C}${D}${B}`
      : `${A}${B}`
    : T;

// ✅ RÁPIDO: Tipos directos y simples
type IdUsuario = string;
type IdProducto = string;
type Email = string;
```

5.2 USAR type vs interface APROPIADAMENTE

```typescript
// Para objetos y clases: interface es más rápida
interface Usuario {
  id: string;
  nombre: string;
}

// Para uniones y tuplas: type es necesario
type Estado = "activo" | "inactivo" | "pendiente";
type ParNumero = [number, number];
type Resultado<T> = { exito: true; data: T } | { exito: false; error: string };
```

5.3 EVITAR INFERENCIA EXCESIVA

```typescript
// ❌ Inferencia profunda: el compilador trabaja más
const config = {
  servidor: {
    puerto: 3000,
    host: "localhost",
    opciones: {
      timeout: 5000,
      retry: 3,
      backoff: {
        base: 1000,
        max: 30000
      }
    }
  }
};

// ✅ Tipo explícito: el compilador verifica contra el tipo
interface ConfiguracionServidor {
  servidor: {
    puerto: number;
    host: string;
    opciones: {
      timeout: number;
      retry: number;
      backoff: {
        base: number;
        max: number;
      };
    };
  };
}

const config: ConfiguracionServidor = {
  servidor: {
    puerto: 3000,
    host: "localhost",
    opciones: {
      timeout: 5000,
      retry: 3,
      backoff: {
        base: 1000,
        max: 30000
      }
    }
  }
};
```

6. MEMORIA Y EFICIENCIA EN EJECUCIÓN
===============================================================================

El código TypeScript compilado a JavaScript puede optimizarse para usar
menos memoria y ejecutarse más rápido.

6.1 ESTRUCTURAS DE DATOS EFICIENTES

```typescript
// ❌ Maps y Sets cuando arrays son suficientes
const usuariosMap = new Map<string, Usuario>();
usuariosMap.set("1", { id: "1", nombre: "Ana" });
const ana = usuariosMap.get("1");

// ✅ Usar arrays cuando solo necesitas listas
const usuariosList: Usuario[] = [];
usuariosList.push({ id: "1", nombre: "Ana" });
const ana = usuariosList.find(u => u.id === "1");

// ✅ Usar Sets para valores únicos
const idsProcesados = new Set<string>();
idsProcesados.add("1");
idsProcesados.add("2");
const yaProcesado = idsProcesados.has("1");
```

6.2 EVITAR COPIAS INNECESARIAS

```typescript
// ❌ Copia superficial con spread en cada operación
let estado = { contador: 0, usuarios: [] as string[] };
estado = { ...estado, contador: estado.contador + 1 };

// ✅ Mutación controlada + copia final (mejor rendimiento)
function actualizarEstado(prev: Estado): Estado {
  const nuevo = { ...prev };
  nuevo.contador++;
  return nuevo;
}

// ❌ Map + filter en cadena (dos iteraciones)
const resultado = datos
  .filter(d => d.activo)
  .map(d => d.nombre);

// ✅ Reduce (una iteración)
const resultado = datos.reduce<string[]>((acc, d) => {
  if (d.activo) acc.push(d.nombre);
  return acc;
}, []);
```

7. HERRAMIENTAS DE MEDICIÓN
===============================================================================

No optimices sin medir. Usa herramientas profesionales para identificar
cuellos de botella.

```typescript
// Medición de tiempo de ejecución
class MedidorRendimiento {
  private marcas = new Map<string, number>();

  iniciar(nombre: string): void {
    this.marcas.set(nombre, performance.now());
  }

  detener(nombre: string): number {
    const inicio = this.marcas.get(nombre);
    if (!inicio) {
      console.warn(`Marca "${nombre}" no encontrada`);
      return 0;
    }
    const duracion = performance.now() - inicio;
    this.marcas.delete(nombre);
    return Math.round(duracion * 100) / 100;
  }

  medir<T>(nombre: string, fn: () => T): T {
    this.iniciar(nombre);
    const resultado = fn();
    const ms = this.detener(nombre);
    console.log(`${nombre}: ${ms}ms`);
    return resultado;
  }

  async medirAsync<T>(nombre: string, fn: () => Promise<T>): Promise<T> {
    this.iniciar(nombre);
    const resultado = await fn();
    const ms = this.detener(nombre);
    console.log(`${nombre}: ${ms}ms`);
    return resultado;
  }
}

// Uso
const medidor = new MedidorRendimiento();

const resultado = medidor.medir("Procesar datos", () => {
  return datos.map(d => d * 2).filter(d => d > 10);
});
```

8. COMPILACIÓN Y BUNDLE
===============================================================================

El proceso de build tiene un impacto directo en el rendimiento final.

8.1 COMPILADORES ALTERNATIVOS

```bash
# tsc (TypeScript Compiler) - Más lento, verificación completa
npx tsc

# esbuild - Extremadamente rápido, sin verificación de tipos
npx esbuild src/main.ts --bundle --outfile=dist/bundle.js

# swc - Muy rápido, compatible con Rust
npx swc src -d dist

# Bun - Runtime todo-en-uno con TypeScript nativo
bun build src/main.ts --outdir=dist
```

8.2 CONFIGURACIÓN DE BUNDLE

```typescript
// vite.config.ts — Configuración optimizada para producción
import { defineConfig } from "vite";

export default defineConfig({
  build: {
    target: "es2020",
    minify: "terser",
    sourcemap: false,
    rollupOptions: {
      output: {
        manualChunks: {
          vendor: ["react", "react-dom"],
          utils: ["date-fns", "lodash-es"]
        }
      }
    }
  },
  esbuild: {
    target: "es2020"
  }
});
```

RESUMEN DEL CAPÍTULO
===============================================================================

- TypeScript impacta el rendimiento en tiempo de compilación y ejecución
- El modo estricto mejora la seguridad y ayuda a generar código eficiente
- El tree shaking elimina código muerto; usa exportaciones nombradas
- El lazy loading reduce el tamaño inicial del bundle
- Tipos simples compilan más rápido que tipos condicionales complejos
- La medición es esencial antes de optimizar
- Herramientas como esbuild, swc y Bun compilan más rápido que tsc
- Las optimizaciones deben centrarse en cuellos de botella reales, no en micro-optimizaciones

PREGUNTAS FRECUENTES
===============================================================================

❓ **¿Debo preocuparme por micro-optimizaciones?**
No. Las micro-optimizaciones rara vez tienen impacto noticeable. Céntrate
en optimizaciones de alto nivel: estructura de datos, algoritmos y carga
de módulos.

❓ **¿Cuándo debo usar esbuild en lugar de tsc?**
Usa tsc durante el desarrollo (por la verificación de tipos) y esbuild para
el build de producción. O combínalos: tsc para type-check y esbuild para
generar el bundle.

❓ **¿El modo estricto hace que el código sea más lento?**
No. strict mode afecta solo a la compilación (puede hacerla ligeramente más
lenta), pero no tiene impacto en el rendimiento en ejecución. De hecho, al
prevenir errores, evita costosos debugging en producción.

❓ **¿Tree shaking funciona con CommonJS?**
No. El tree shaking requiere módulos ES (import/export). CommonJS (require)
es dinámico y no permite análisis estático de dependencias.
