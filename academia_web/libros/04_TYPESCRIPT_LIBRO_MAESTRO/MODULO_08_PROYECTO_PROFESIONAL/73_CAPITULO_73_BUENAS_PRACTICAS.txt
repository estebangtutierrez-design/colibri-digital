1. PRINCIPIOS SOLID EN TYPESCRIPT
===============================================================================

Los principios SOLID son la base del diseño orientado a objetos moderno.
En TypeScript adquieren una dimensión adicional gracias al sistema de tipos.

1.1 S — RESPONSABILIDAD ÚNICA (SINGLE RESPONSIBILITY)

Cada clase, función o módulo debe tener una única razón para cambiar.

```typescript
// MAL: Una clase que mezcla persistencia, lógica y presentación
class Usuario {
  constructor(
    public nombre: string,
    public email: string
  ) {}

  guardar(): void {
    localStorage.setItem(`usuario_${this.nombre}`, JSON.stringify(this));
  }

  validar(): boolean {
    return this.email.includes("@");
  }

  renderizar(): string {
    return `<div>${this.nombre} - ${this.email}</div>`;
  }
}

// BIEN: Cada responsabilidad en su propia clase
class Usuario {
  constructor(
    public readonly nombre: string,
    public readonly email: string
  ) {}
}

class UsuarioRepository {
  guardar(usuario: Usuario): void {
    localStorage.setItem(`usuario_${usuario.nombre}`, JSON.stringify(usuario));
  }
}

class UsuarioValidator {
  static validar(usuario: Usuario): boolean {
    return usuario.email.includes("@");
  }
}

class UsuarioRenderer {
  static renderizar(usuario: Usuario): string {
    return `<div>${usuario.nombre} - ${usuario.email}</div>`;
  }
}
```

1.2 O — ABIERTO/CERRADO (OPEN/CLOSED)

Las entidades deben estar abiertas para extensión pero cerradas para
modificación.

```typescript
// MAL: Cada nuevo tipo de descuento requiere modificar la clase
class CalculadoraPrecio {
  calcular(precio: number, tipoCliente: string): number {
    if (tipoCliente === "regular") return precio;
    if (tipoCliente === "premium") return precio * 0.9;
    if (tipoCliente === "vip") return precio * 0.8;
    throw new Error("Tipo de cliente desconocido");
  }
}

// BIEN: Abierto para extensión, cerrado para modificación
interface EstrategiaDescuento {
  aplicar(precio: number): number;
}

class SinDescuento implements EstrategiaDescuento {
  aplicar(precio: number): number {
    return precio;
  }
}

class DescuentoPremium implements EstrategiaDescuento {
  aplicar(precio: number): number {
    return precio * 0.9;
  }
}

class DescuentoVIP implements EstrategiaDescuento {
  aplicar(precio: number): number {
    return precio * 0.8;
  }
}

// Nuevo descuento: sin modificar código existente
class DescuentoBlackFriday implements EstrategiaDescuento {
  aplicar(precio: number): number {
    return precio * 0.5;
  }
}

class CalculadoraPrecio {
  constructor(private estrategia: EstrategiaDescuento) {}

  calcular(precio: number): number {
    return this.estrategia.aplicar(precio);
  }

  cambiarEstrategia(estrategia: EstrategiaDescuento): void {
    this.estrategia = estrategia;
  }
}
```

1.3 L — SUSTITUCIÓN DE LISKOV (LISKOV SUBSTITUTION)

Los subtipos deben ser sustituibles por sus tipos base sin alterar la
corrección del programa.

```typescript
// MAL: Violación de LSP
class Rectangulo {
  constructor(public ancho: number, public alto: number) {}

  area(): number {
    return this.ancho * this.alto;
  }
}

class Cuadrado extends Rectangulo {
  constructor(lado: number) {
    super(lado, lado);
  }

  // Cambia el comportamiento esperado
  set ancho(valor: number) {
    super.ancho = valor;
    super.alto = valor;
  }

  set alto(valor: number) {
    super.ancho = valor;
    super.alto = valor;
  }
}

// BIEN: Usar una interfaz común sin herencia problemática
interface Figura {
  area(): number;
}

class Rectangulo implements Figura {
  constructor(
    public readonly ancho: number,
    public readonly alto: number
  ) {}

  area(): number {
    return this.ancho * this.alto;
  }
}

class Cuadrado implements Figura {
  constructor(public readonly lado: number) {}

  area(): number {
    return this.lado * this.lado;
  }
}

function calcularAreaTotal(figuras: Figura[]): number {
  return figuras.reduce((total, f) => total + f.area(), 0);
}
```

1.4 I — SEGREGACIÓN DE INTERFACES (INTERFACE SEGREGATION)

Ninguna clase debe ser forzada a implementar interfaces que no usa.

```typescript
// MAL: Interfaz grande que obliga a implementar métodos no necesarios
interface Trabajador {
  trabajar(): void;
  comer(): void;
  dormir(): void;
  cobrar(): void;
}

class Robot implements Trabajador {
  trabajar(): void { /* trabajar */ }
  comer(): void { throw new Error("Los robots no comen"); }
  dormir(): void { throw new Error("Los robots no duermen"); }
  cobrar(): void { /* cobrar */ }
}

// BIEN: Interfaces pequeñas y específicas
interface Trabajable {
  trabajar(): void;
}

interface Comible {
  comer(): void;
}

interface Dormible {
  dormir(): void;
}

interface Cobrable {
  cobrar(): void;
}

class Robot implements Trabajable, Cobrable {
  trabajar(): void { /* trabajar */ }
  cobrar(): void { /* cobrar */ }
}

class Humano implements Trabajable, Comible, Dormible, Cobrable {
  trabajar(): void { /* trabajar */ }
  comer(): void { /* comer */ }
  dormir(): void { /* dormir */ }
  cobrar(): void { /* cobrar */ }
}
```

1.5 D — INVERSIÓN DE DEPENDENCIAS (DEPENDENCY INVERSION)

Los módulos de alto nivel no deben depender de módulos de bajo nivel. Ambos
deben depender de abstracciones.

```typescript
// MAL: Dependencia directa en implementación concreta
class ServicioNotificacion {
  private email = new EmailService();

  notificar(mensaje: string): void {
    this.email.enviar(mensaje);
  }
}

// BIEN: Dependencia invertida mediante interfaz
interface ServicioMensajeria {
  enviar(destino: string, mensaje: string): Promise<void>;
}

class ServicioNotificacion {
  constructor(private readonly mensajeria: ServicioMensajeria) {}

  async notificar(destino: string, mensaje: string): Promise<void> {
    await this.mensajeria.enviar(destino, mensaje);
  }
}

class EmailService implements ServicioMensajeria {
  async enviar(destino: string, mensaje: string): Promise<void> {
    console.log(`Enviando email a ${destino}: ${mensaje}`);
  }
}

class SMSService implements ServicioMensajeria {
  async enviar(destino: string, mensaje: string): Promise<void> {
    console.log(`Enviando SMS a ${destino}: ${mensaje}`);
  }
}

// La inyección ocurre en el punto de composición
const sms = new SMSService();
const notificador = new ServicioNotificacion(sms);
```

2. PRINCIPIO DRY (DON'T REPEAT YOURSELF)
===============================================================================

Cada pieza de conocimiento debe tener una representación única, no ambigua
y autoritativa dentro del sistema.

```typescript
// MAL: Código duplicado
function validarEmailLogin(email: string): boolean {
  if (!email || email.length < 5) return false;
  if (!email.includes("@")) return false;
  if (!email.includes(".")) return false;
  return true;
}

function validarEmailRegistro(email: string): boolean {
  if (!email || email.length < 5) return false;
  if (!email.includes("@")) return false;
  if (!email.includes(".")) return false;
  return true;
}

// BIEN: Código reutilizado
function esEmailValido(email: string): boolean {
  if (!email || email.length < 5) return false;
  if (!email.includes("@")) return false;
  if (!email.includes(".")) return false;
  return true;
}

function validarEmailLogin(email: string): boolean {
  return esEmailValido(email);
}

function validarEmailRegistro(email: string): boolean {
  return esEmailValido(email);
}
```

2.1 ABSTRACCIÓN ESTRATÉGICA

No te repitas, pero tampoco abstraigas prematuramente. La regla de tres:
si usas el mismo código tres veces, es momento de abstraerlo.

```typescript
// DRY aplicado con tipos compartidos
type ID = string;
type FechaISO = string;

interface Entidad {
  id: ID;
  creadoEn: FechaISO;
  actualizadoEn: FechaISO;
}

interface Usuario extends Entidad {
  nombre: string;
  email: string;
}

interface Producto extends Entidad {
  nombre: string;
  precio: number;
  stock: number;
}
```

3. CÓDIGO AUTODOCUMENTADO
===============================================================================

El mejor código es el que se explica a sí mismo y no necesita comentarios.

```typescript
// MAL: Comentario que explica lo obvio
// Incrementa el contador en 1
contador = contador + 1;

// Suma el item al carrito
function agregarAlCarrito(item: any) { /* ... */ }

// BIEN: El nombre comunica la intención
contador++;
function agregarProductoAlCarrito(producto: Producto): void { /* ... */ }

// MAL: Comentario que debería ser código
// Si el usuario es admin y está activo y no está baneado
if (usuario.rol === "admin" && usuario.activo && !usuario.baneado) { /* ... */ }

// BIEN: Variable con nombre descriptivo
const esAdminActivoYNoBaneado =
  usuario.rol === "admin" && usuario.activo && !usuario.baneado;

if (esAdminActivoYNoBaneado) { /* ... */ }
```

3.1 NOMBRES SIGNIFICATIVOS

Los nombres de variables, funciones y clases deben revelar la intención.

```typescript
// MAL: Nombres crípticos
function proc(d: any[]): any[] {
  return d.filter(x => x.s > 0).map(x => ({ n: x.s, p: x.s * x.q }));
}

// BIEN: Nombres que comunican intención
interface LineaPedido {
  precioUnitario: number;
  cantidad: number;
}

interface ResultadoProcesamiento {
  nombre: string;
  total: number;
}

function procesarLineasConPrecioPositivo(
  lineas: LineaPedido[]
): ResultadoProcesamiento[] {
  return lineas
    .filter(linea => linea.precioUnitario > 0)
    .map(linea => ({
      nombre: `Producto $${linea.precioUnitario}`,
      total: linea.precioUnitario * linea.cantidad
    }));
}
```

4. ANTIPATRONES COMUNES EN TYPESCRIPT
===============================================================================

Conocer los antipatrones te ayuda a reconocer y evitar código problemático.

4.1 ANY CÓMPLICE

Usar any excesivamente desactiva el sistema de tipos, eliminando la principal
ventaja de TypeScript.

```typescript
// MAL: Any elimina la seguridad de tipos
function procesar(data: any): any {
  return data.nombre + data.edad;
}

// BIEN: Tipos específicos
interface DatosPersona {
  nombre: string;
  edad: number;
}

function procesar(data: DatosPersona): string {
  return `${data.nombre} ${data.edad}`;
}

// MAL: any en parámetros de funciones genéricas
function identidad(valor: any): any {
  return valor;
}

// BIEN: Genérico con tipo preservado
function identidad<T>(valor: T): T {
  return valor;
}
```

4.2 OBJETOS DIOS (GOD OBJECTS)

Objetos que concentran demasiada funcionalidad y conocimiento.


4.3 MANEJO SILENCIOSO DE ERRORES

Tragar errores sin registrarlos o sin propagarlos correctamente.

```typescript
// MAL: Error silenciado
try {
  await operacionRiesgosa();
} catch {
  // No hacer nada - el error desaparece
}

// MAL: catch sin tipo específico
try {
  await operacionRiesgosa();
} catch (err) {
  console.log("Algo salió mal");
}

// BIEN: Manejo adecuado de errores
try {
  await operacionRiesgosa();
} catch (error) {
  const mensaje = error instanceof Error
    ? error.message
    : "Error desconocido";
  console.error(`Operación fallida: ${mensaje}`);
  throw new Error(`Error en proceso: ${mensaje}`);
}

// MEJOR AÚN: Result type para manejo funcional
type ResultadoOperacion<T> =
  | { exito: true; data: T }
  | { exito: false; error: string };

async function operacionSegura<T>(
  fn: () => Promise<T>
): Promise<ResultadoOperacion<T>> {
  try {
    const data = await fn();
    return { exito: true, data };
  } catch (error) {
    return {
      exito: false,
      error: error instanceof Error ? error.message : "Error desconocido"
    };
  }
}
```

4.4 ACOPLAMIENTO TEMPORAL

Código que depende del orden de ejecución sin documentarlo.

```typescript
// MAL: Dependencia temporal oculta
class Configuracion {
  private baseDatos: any;

  conectar(): void {
    this.baseDatos = { conectado: true };
  }

  consultar(query: string): any {
    // Explota si no se llamó conectar() primero
    return this.baseDatos.ejecutar(query);
  }
}

// BIEN: La dependencia temporal se resuelve en construcción
class Configuracion {
  private readonly baseDatos: any;

  constructor(baseDatos: any) {
    this.baseDatos = baseDatos;
  }

  consultar(query: string): any {
    return this.baseDatos.ejecutar(query);
  }
}
```

5. CÓDIGO IDIOMÁTICO DE TYPESCRIPT
===============================================================================

Escribir TypeScript idiomático significa usar el lenguaje de la forma en que
fue diseñado, aprovechando sus características únicas.

```typescript
// MAL: Estilo JavaScript plano
function getNombre(data) {
  return data.nombre ? data.nombre : "Sin nombre";
}

// BIEN: Estilo TypeScript idiomático
function obtenerNombre(data: { nombre?: string }): string {
  return data.nombre ?? "Sin nombre";
}

// MAL: Validación manual de tipos
function duplicar(valor: any): any {
  if (typeof valor === "number") return valor * 2;
  if (typeof valor === "string") return valor + valor;
  return null;
}

// BIEN: Sobrecarga de funciones con tipos específicos
function duplicar(valor: number): number;
function duplicar(valor: string): string;
function duplicar(valor: number | string): number | string {
  if (typeof valor === "number") return valor * 2;
  return valor + valor;
}
```

6. MANEJO DE ERRORES PROFESIONAL
===============================================================================

El manejo de errores es una de las señales más claras de un desarrollador
profesional.

```typescript
// Jerarquía de errores personalizados
class ErrorAplicacion extends Error {
  constructor(
    mensaje: string,
    public readonly codigo: string,
    public readonly statusHTTP: number = 500
  ) {
    super(mensaje);
    this.name = "ErrorAplicacion";
  }
}

class ErrorValidacion extends ErrorAplicacion {
  constructor(mensaje: string) {
    super(mensaje, "VALIDATION_ERROR", 400);
    this.name = "ErrorValidacion";
  }
}

class ErrorNoEncontrado extends ErrorAplicacion {
  constructor(entidad: string, id: string) {
    super(`${entidad} con id ${id} no encontrado`, "NOT_FOUND", 404);
    this.name = "ErrorNoEncontrado";
  }
}

class ErrorAutenticacion extends ErrorAplicacion {
  constructor() {
    super("Credenciales inválidas", "AUTH_ERROR", 401);
    this.name = "ErrorAutenticacion";
  }
}

// Uso profesional
class UsuarioService {
  async obtenerUsuario(id: string): Promise<Usuario> {
    const usuario = await this.repo.buscarPorId(id);
    if (!usuario) {
      throw new ErrorNoEncontrado("Usuario", id);
    }
    return usuario;
  }

  async crearUsuario(dto: CrearUsuarioDTO): Promise<Usuario> {
    if (!dto.email.includes("@")) {
      throw new ErrorValidacion("El email debe contener @");
    }
    const existente = await this.repo.buscarPorEmail(dto.email);
    if (existente) {
      throw new ErrorValidacion("El email ya está registrado");
    }
    return this.repo.guardar(new Usuario(dto));
  }
}
```

PREGUNTAS FRECUENTES
===============================================================================

❓ **¿Es necesario aplicar siempre todos los principios SOLID?**
No, la rigidez no es el objetivo. Aplica SOLID cuando resuelva un problema
real. La sobre-aplicación puede generar complejidad innecesaria.

❓ **¿Los comentarios son siempre malos?**
No. Los comentarios son necesarios cuando explican el "por qué" detrás de
una decisión compleja, no cuando describen el "qué" (que debe ser obvio en
el código).

❓ **¿Cómo sé si estoy siendo demasiado DRY?**
Si abstraer el código requiere más esfuerzo mental que tenerlo duplicado,
probablemente estás siendo demasiado DRY. La regla de tres es una buena guía.

❓ **¿Qué hago con un código heredado lleno de malas prácticas?**
Refactoriza de forma incremental. No intentes arreglar todo de una vez.
Identifica los puntos más críticos, escribe tests para protegerte y refactoriza
uno a la vez.
