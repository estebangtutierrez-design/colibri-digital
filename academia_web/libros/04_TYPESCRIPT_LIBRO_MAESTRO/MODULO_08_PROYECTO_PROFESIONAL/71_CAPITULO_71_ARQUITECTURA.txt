1. FUNDAMENTOS DE ARQUITECTURA DE SOFTWARE
===============================================================================

La arquitectura de software es el conjunto de decisiones estructurales que
determinan cómo un sistema está organizado. En TypeScript, estas decisiones
se ven favorecidas por el sistema de tipos, que permite expresar restricciones
arquitectónicas directamente en el código.

Una buena arquitectura debe cumplir con los siguientes atributos:

- Mantenibilidad: El código debe ser fácil de modificar sin romper otras partes
- Escalabilidad: El sistema debe crecer sin requerir reescrituras completas
- Testeabilidad: Cada componente debe poder probarse de forma aislada
- Flexibilidad: Debe poder adaptarse a nuevos requisitos con bajo costo

```typescript
// Ejemplo de una decisión arquitectónica: encapsular la creación de objetos
// usando un Factory Pattern en lugar de instanciar directamente

interface Logger {
  log(mensaje: string): void;
  error(mensaje: string): void;
}

class ConsoleLogger implements Logger {
  log(mensaje: string): void {
    console.log(`[LOG] ${mensaje}`);
  }
  error(mensaje: string): void {
    console.error(`[ERROR] ${mensaje}`);
  }
}

class FileLogger implements Logger {
  log(mensaje: string): void {
    // Simula escritura a archivo
    console.log(`[FILE LOG] ${mensaje}`);
  }
  error(mensaje: string): void {
    // Simula escritura a archivo de errores
    console.error(`[FILE ERROR] ${mensaje}`);
  }
}

class LoggerFactory {
  static createLogger(tipo: "console" | "file"): Logger {
    switch (tipo) {
      case "console":
        return new ConsoleLogger();
      case "file":
        return new FileLogger();
    }
  }
}

// Uso: la aplicación nunca depende de una implementación concreta
const logger = LoggerFactory.createLogger("console");
logger.log("Aplicación iniciada correctamente");
```

Este patrón Factory desacopla la creación del uso, permitiendo cambiar la
implementación sin modificar el código que consume el Logger.

1.1 CICLO DE VIDA DE LA ARQUITECTURA

La arquitectura no es un evento único, sino un proceso continuo:

1. Análisis: Comprender los requisitos funcionales y no funcionales
2. Diseño: Definir la estructura, componentes e interacciones
3. Implementación: Escribir el código siguiendo el diseño
4. Evaluación: Revisar que la implementación cumple con la arquitectura
5. Evolución: Refinar la arquitectura conforme el proyecto crece

```typescript
// Representación de una decisión arquitectónica documentada en tipo
type DecisionArquitectonica = {
  id: string;
  fecha: Date;
  titulo: string;
  contexto: string;
  opciones: string[];
  decision: string;
  consequencias: string[];
  estatus: "propuesta" | "aceptada" | "deprecada";
};

const decision: DecisionArquitectonica = {
  id: "ADR-001",
  fecha: new Date(),
  titulo: "Elegir Logger basado en interfaz",
  contexto: "Necesitamos registrar eventos en consola y archivo",
  opciones: ["ConsoleLogger", "FileLogger", "LoggerFactory"],
  decision: "Usar LoggerFactory con interfaz Logger",
  consequencias: [
    "Fácil de cambiar implementación",
    "Testeable mediante mocks",
    "Bajo acoplamiento"
  ],
  estatus: "aceptada"
};
```

2. PATRONES ARQUITECTÓNICOS EN TYPESCRIPT
===============================================================================

Existen varios patrones arquitectónicos que se adaptan bien a TypeScript.
Analicemos los más importantes para nuestro proyecto profesional.

2.1 ARQUITECTURA POR CAPAS (LAYERED ARCHITECTURE)

Divide la aplicación en capas horizontales donde cada una tiene una
responsabilidad específica. Las capas típicas son:

- Capa de Presentación: Interfaz de usuario (UI)
- Capa de Aplicación: Casos de uso, orquestación
- Capa de Dominio: Lógica de negocio, entidades
- Capa de Infraestructura: Acceso a datos, servicios externos

```typescript
// Capa de Dominio: Entidades puras sin dependencias externas
interface Producto {
  id: string;
  nombre: string;
  precio: number;
  stock: number;
}

// Capa de Aplicación: Casos de uso
class CrearProductoUseCase {
  constructor(
    private readonly repositorio: ProductoRepositorio,
    private readonly notificador: Notificador
  ) {}

  async ejecutar(datos: Omit<Producto, "id">): Promise<Producto> {
    const producto: Producto = {
      id: crypto.randomUUID(),
      ...datos
    };
    await this.repositorio.guardar(producto);
    this.notificador.enviar(`Producto ${producto.nombre} creado`);
    return producto;
  }
}

// Capa de Infraestructura: Implementación concreta
interface ProductoRepositorio {
  guardar(producto: Producto): Promise<void>;
  buscarPorId(id: string): Promise<Producto | null>;
}

class ProductoRepositorioSQL implements ProductoRepositorio {
  async guardar(producto: Producto): Promise<void> {
    // Lógica de base de datos
    console.log(`Guardando producto ${producto.id} en SQL`);
  }

  async buscarPorId(id: string): Promise<Producto | null> {
    // Lógica de base de datos
    return null;
  }
}
```

La regla fundamental de las capas es que una capa solo puede depender de la
capa inmediatamente inferior. Esto crea un flujo unidireccional de dependencias
que facilita el mantenimiento.

2.2 ARQUITECTURA HEXAGONAL (POR PUERTOS Y ADAPTADORES)

También conocida como arquitectura de puertos y adaptadores. El dominio es el
centro y se comunica con el exterior a través de interfaces (puertos). Los
adaptadores implementan esas interfaces.

```typescript
// Puerto: define el contrato que el dominio necesita
interface PuertaDePago {
  cobrar(monto: number, tarjeta: DatosTarjeta): Promise<ResultadoPago>;
}

// Adaptador: implementación concreta del puerto
class StripeAdaptador implements PuertaDePago {
  async cobrar(monto: number, tarjeta: DatosTarjeta): Promise<ResultadoPago> {
    // Lógica específica de Stripe
    return { exito: true, transaccionId: "txn_123" };
  }
}

class MercadoPagoAdaptador implements PuertaDePago {
  async cobrar(monto: number, tarjeta: DatosTarjeta): Promise<ResultadoPago> {
    // Lógica específica de Mercado Pago
    return { exito: true, transaccionId: "mp_456" };
  }
}

// El dominio usa el puerto, sin conocer la implementación
class ProcesadorDePagos {
  constructor(private readonly puerta: PuertaDePago) {}

  async procesar(orden: Orden): Promise<ResultadoPago> {
    return this.puerta.cobrar(orden.total, orden.tarjeta);
  }
}

// La inyección de dependencias conecta todo
const adaptador = new StripeAdaptador();
const procesador = new ProcesadorDePagos(adaptador);
```

2.3 PATRÓN REPOSITORY

El patrón Repository abstrae el almacenamiento y recuperación de datos,
presentando al dominio una colección de objetos en memoria.

```typescript
// Definición genérica del repositorio
interface Repositorio<T extends { id: string }> {
  obtenerPorId(id: string): Promise<T | null>;
  obtenerTodos(): Promise<T[]>;
  guardar(entidad: T): Promise<void>;
  eliminar(id: string): Promise<void>;
}

// Implementación concreta usando un Map en memoria
class RepositorioEnMemoria<T extends { id: string }>
  implements Repositorio<T>
{
  private readonly almacen = new Map<string, T>();

  async obtenerPorId(id: string): Promise<T | null> {
    return this.almacen.get(id) ?? null;
  }

  async obtenerTodos(): Promise<T[]> {
    return Array.from(this.almacen.values());
  }

  async guardar(entidad: T): Promise<void> {
    this.almacen.set(entidad.id, entidad);
  }

  async eliminar(id: string): Promise<void> {
    this.almacen.delete(id);
  }
}

// Uso con una entidad específica
interface Usuario extends Entidad {
  nombre: string;
  email: string;
}

interface Entidad {
  id: string;
}

const usuarioRepo = new RepositorioEnMemoria<Usuario>();
```

2.4 PATRÓN SERVICE

Los servicios encapsulan lógica de negocio que no pertenece naturalmente a una
entidad o a un caso de uso individual.

```typescript
// Servicio de autenticación
interface ServicioAutenticacion {
  registrar(email: string, password: string): Promise<Usuario>;
  iniciarSesion(email: string, password: string): Promise<Sesion>;
  verificarToken(token: string): Promise<boolean>;
}

class ServicioAutenticacionImpl implements ServicioAutenticacion {
  constructor(
    private readonly repoUsuarios: Repositorio<Usuario>,
    private readonly encriptador: Encriptador
  ) {}

  async registrar(email: string, password: string): Promise<Usuario> {
    const existente = await this.repoUsuarios.buscarPorEmail(email);
    if (existente) {
      throw new Error("El email ya está registrado");
    }
    const hash = await this.encriptador.hash(password);
    const usuario: Usuario = {
      id: crypto.randomUUID(),
      email,
      passwordHash: hash,
      creadoEn: new Date()
    };
    await this.repoUsuarios.guardar(usuario);
    return usuario;
  }

  async iniciarSesion(email: string, password: string): Promise<Sesion> {
    const usuario = await this.repoUsuarios.buscarPorEmail(email);
    if (!usuario) {
      throw new Error("Credenciales inválidas");
    }
    const valido = await this.encriptador.comparar(password, usuario.passwordHash);
    if (!valido) {
      throw new Error("Credenciales inválidas");
    }
    return { usuarioId: usuario.id, token: "jwt_token_generado" };
  }

  async verificarToken(token: string): Promise<boolean> {
    // Lógica de verificación JWT
    return token.length > 0;
  }
}
```

3.1 TIPOS PARA MODELAR DOMINIO

Los tipos de TypeScript son excelentes para modelar el dominio del problema.

```typescript
// Modelado de dominio rico con tipos
type Dinero = number & { readonly __brand: "dinero" };

type Email = string & { readonly __brand: "email" };

type Telefono = string & { readonly __brand: "telefono" };

// Funciones constructoras con validación
function crearEmail(valor: string): Email {
  if (!valor.includes("@")) {
    throw new Error("Email inválido");
  }
  return valor as Email;
}

function crearDinero(valor: number): Dinero {
  if (valor < 0) {
    throw new Error("El dinero no puede ser negativo");
  }
  return valor as Dinero;
}

interface Cliente {
  id: string;
  email: Email;
  telefono: Telefono;
  saldo: Dinero;
}
```

3.2 UNIONS DISCRIMINADAS PARA ESTADOS

Las uniones discriminadas modelan perfectamente los estados finitos de una
entidad.

```typescript
type EstadoPedido =
  | { tipo: "pendiente"; fechaCreacion: Date }
  | { tipo: "confirmado"; fechaConfirmacion: Date; metodoPago: string }
  | { tipo: "enviado"; fechaEnvio: Date; codigoSeguimiento: string }
  | { tipo: "entregado"; fechaEntrega: Date; firma: string }
  | { tipo: "cancelado"; fechaCancelacion: Date; motivo: string };

// TypeScript obliga a manejar todos los casos
function procesarEstadoPedido(estado: EstadoPedido): string {
  switch (estado.tipo) {
    case "pendiente":
      return `Pedido creado el ${estado.fechaCreacion}`;
    case "confirmado":
      return `Pagado con ${estado.metodoPago}`;
    case "enviado":
      return `Envío: ${estado.codigoSeguimiento}`;
    case "entregado":
      return `Recibido por ${estado.firma}`;
    case "cancelado":
      return `Cancelado: ${estado.motivo}`;
  }
}
```

3.3 GENÉRICOS PARA COMPONENTES REUTILIZABLES

Los genéricos permiten crear componentes arquitectónicos que funcionan con
cualquier tipo manteniendo la seguridad.

```typescript
// Result es un tipo funcional para manejo de errores
type ResultadoExitoso<T> = { exito: true; valor: T };
type ResultadoFallido = { exito: false; error: Error };
type Resultado<T> = ResultadoExitoso<T> | ResultadoFallido;

async function ejecutarSeguro<T>(
  fn: () => Promise<T>
): Promise<Resultado<T>> {
  try {
    const valor = await fn();
    return { exito: true, valor };
  } catch (error) {
    return { exito: false, error: error as Error };
  }
}

// Uso
const resultado = await ejecutarSeguro(async () => {
  // Operación que puede fallar
  return "datos obtenidos";
});

if (resultado.exito) {
  console.log(resultado.valor); // TypeScript sabe que es string
} else {
  console.error(resultado.error.message);
}
```

4. DISEÑO DE LA ESTRUCTURA BASE DEL PROYECTO
===============================================================================

Para nuestro proyecto profesional, adoptaremos una arquitectura por capas
combinada con elementos de la arquitectura hexagonal. La estructura base será:

```
proyecto/
├── src/
│   ├── domain/          # Entidades, value objects, interfaces del dominio
│   ├── application/     # Casos de uso, servicios de aplicación
│   ├── infrastructure/  # Implementaciones concretas (DB, API, etc.)
│   └── presentation/    # UI, controladores, middlewares
├── tests/               # Pruebas unitarias y de integración
├── config/              # Configuración de la aplicación
└── docs/                # Documentación técnica
```

```typescript
// domain/entidades/Producto.ts
```

5. PRINCIPIOS SOLID EN LA ARQUITECTURA
===============================================================================

Los principios SOLID son fundamentales para una buena arquitectura. Veamos
cómo aplicarlos en TypeScript.

5.1 RESPONSABILIDAD ÚNICA (S)

Cada clase debe tener una única razón para cambiar.

```typescript
// MAL: Una clase que hace demasiadas cosas
class GestorUsuarios {
  constructor(private db: any) {}

  crear(usuario: Usuario) { /* ... */ }
  enviarEmail(usuario: Usuario, mensaje: string) { /* ... */ }
  generarReporte(usuario: Usuario): string { /* ... */ }
  validarPassword(password: string): boolean { /* ... */ }
}

// BIEN: Cada responsabilidad en su propia clase
class RepositorioUsuarios {
  constructor(private db: any) {}
  guardar(usuario: Usuario): void { /* ... */ }
}

class ServicioEmail {
  enviar(destino: string, mensaje: string): void { /* ... */ }
}

class GeneradorReportes {
  generar(usuario: Usuario): string { /* ... */ }
}

class ValidadorPassword {
  validar(password: string): boolean { /* ... */ }
}
```

5.2 ABIERTO/CERRADO (O)

Las entidades deben estar abiertas para extensión pero cerradas para
modificación.

```typescript
interface ProcesadorPago {
  procesar(monto: number): Promise<boolean>;
}

class PagoTarjeta implements ProcesadorPago {
  async procesar(monto: number): Promise<boolean> {
    return true;
  }
}

class PagoCripto implements ProcesadorPago {
  async procesar(monto: number): Promise<boolean> {
    return true;
  }
}

// No necesitas modificar código existente para agregar nuevos métodos
```

PREGUNTAS FRECUENTES
===============================================================================

❓ **¿Qué arquitectura es mejor para un proyecto TypeScript?**
No hay una "mejor" arquitectura universal. La elección depende del tamaño del
proyecto, el equipo, los requisitos y el contexto. La arquitectura por capas
es un buen punto de partida para la mayoría de proyectos.

❓ **¿Debo usar siempre patrones de diseño?**
No. Los patrones son soluciones probadas para problemas recurrentes, pero
usarlos sin necesidad añade complejidad innecesaria. Aplica patrones cuando
resuelvan un problema real, no por moda.

❓ **¿Cómo evito el over-engineering?**
Céntrate en los requisitos actuales y en un futuro inmediato previsible.
Diseña pensando en el cambio, pero no anticipes cambios hipotéticos. La
arquitectura evoluciona con el proyecto.

❓ **¿TypeScript reemplaza la documentación arquitectónica?**
No la reemplaza, pero la complementa. Los tipos documentan contratos y
restricciones que antes requerían documentación externa. Sin embargo,
decisiones arquitectónicas de alto nivel aún deben documentarse.
