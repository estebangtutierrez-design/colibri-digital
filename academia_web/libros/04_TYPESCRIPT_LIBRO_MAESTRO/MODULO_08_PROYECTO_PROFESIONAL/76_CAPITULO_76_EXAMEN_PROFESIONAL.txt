**SECCIÓN 1 — Opción Múltiple:**

1. **c) Error de compilación**
   El tipo `"d"` no está en la unión `"a" | "b" | "c"`. TypeScript rechaza
   valores que no pertenecen al tipo definido. Esto es una de las mayores
   ventajas del sistema de tipos.

2. **b) Verifica que un valor cumple con un tipo sin cambiar su tipo inferido**
   `satisfies` permite verificar que una expresión cumple con un tipo sin
   cambiar el tipo inferido de la expresión. Es útil cuando quieres mantener
   el tipo literal pero verificar que cumple un contrato más amplio.

3. **c) type puede representar uniones y tuplas; interface puede fusionarse**
   `type` puede representar uniones, tuplas, y tipos primitivos. `interface`
   puede fusionarse (declaration merging) y extenderse de forma más explícita.
   Ambos son poderosos y se eligen según el contexto.

4. **b) Impide que la propiedad sea modificada después de la inicialización**
   `readonly` es una restricción en tiempo de compilación. No impide la
   modificación en tiempo de ejecución, pero el compilador previene
   asignaciones después de la inicialización.

5. **b) Promise<void>**
   Todas las funciones async retornan una Promise. Si no devuelven un valor
   explícito, la Promise se resuelve con `undefined`, y el tipo es `Promise<void>`.

6. **c) Nullable<T>**
   `Nullable<T>` no existe en TypeScript. Los tipos utilitarios nativos son:
   `Partial<T>`, `Required<T>`, `Readonly<T>`, `Pick<T, K>`, `Omit<T, K>`,
   `Record<K, T>`, `Exclude<T, U>`, `Extract<T, U>`, `NonNullable<T>`,
   `Parameters<T>`, `ConstructorParameters<T>`, `ReturnType<T>`,
   `InstanceType<T>`, `ThisParameterType<T>`, `OmitThisParameter<T>`,
   `Awaited<T>`.

7. **a) Obtiene las claves de un objeto como un tipo unión**
   `keyof T` produce un tipo unión con los nombres de las propiedades de T.
   Por ejemplo, `keyof { a: number; b: string }` produce `"a" | "b"`.

8. **b) Restringir un tipo más general a uno más específico mediante checks**
   El type narrowing ocurre cuando TypeScript reduce un tipo unión a un
   subtipo específico basándose en checks como `typeof`, `instanceof`,
   discriminantes de uniones, o guards personalizados.

9. **b) Para que null y undefined sean tratados como tipos distintos**
   Con `strictNullChecks: true`, `null` y `undefined` son tipos separados.
   Una variable de tipo `string` no puede contener `null` a menos que se
   declare explícitamente como `string | null`.

10. **c) Strategy**
    El patrón Strategy define una familia de algoritmos intercambiables.
    El contexto delega la ejecución en la estrategia, permitiendo cambiar
    el comportamiento en tiempo de ejecución.

**SECCIÓN 2 — Soluciones de Código:**

11. **Función agruparPor:**

```typescript
function agruparPor<T, K extends string | number | symbol>(
  items: T[],
  extractorClave: (item: T) => K
): Record<K, T[]> {
  return items.reduce((acumulador, item) => {
    const clave = extractorClave(item);
    if (!acumulador[clave]) {
      acumulador[clave] = [];
    }
    acumulador[clave].push(item);
    return acumulador;
  }, {} as Record<K, T[]>);
}

// Uso con inferencia de tipos
const personas = [
  { nombre: "Ana", ciudad: "Madrid" },
  { nombre: "Luis", ciudad: "Barcelona" },
  { nombre: "Eva", ciudad: "Madrid" }
];

const agrupado = agruparPor(personas, p => p.ciudad);
// TypeScript infiere: Record<string, { nombre: string; ciudad: string }[]>
```

12. **ElementoDeArray<T>:**

```typescript
type ElementoDeArray<T> = T extends (infer U)[] ? U : T;

// Casos de prueba
type A = ElementoDeArray<number[]>;     // number
type B = ElementoDeArray<string[]>;     // string
type C = ElementoDeArray<boolean>;      // boolean (se mantiene porque no es array)
type D = ElementoDeArray<(number | string)[]>; // string | number
type E = ElementoDeArray<number[][]>;   // number[] (extrae el tipo interno del primer nivel)
```

13. **Validable<T>:**

```typescript
type Validable<T> = {
  [P in keyof T]?: boolean | string;
};

// Uso
interface Formulario {
  nombre: string;
  email: string;
  edad: number;
}

type ValidableFormulario = Validable<Formulario>;
// Resultado:
// {
//   nombre?: boolean | string;
//   email?: boolean | string;
//   edad?: boolean | string;
// }

// Versión avanzada con mensajes de error específicos
type ValidableAvanzado<T> = {
  [P in keyof T]: { valido: boolean; error?: string };
};
```

// Refactorización:
interface Tarea {
  id: number;
  titulo: string;
  done: boolean;
  createdAt: Date;
}

// Responsabilidad 1: Gestión de tareas
class GestorTareas {
  constructor(private items: Tarea[] = []) {}

  agregar(titulo: string): Tarea {
    const tarea: Tarea = {
      id: Date.now(),
      titulo,
      done: false,
      createdAt: new Date()
    };
    this.items.push(tarea);
    return tarea;
  }

  eliminar(id: number): void {
    this.items = this.items.filter(i => i.id !== id);
  }

  obtenerTodas(): Tarea[] {
    return [...this.items];
  }
}

// Responsabilidad 2: Persistencia
interface RepositorioTareas {
  guardar(items: Tarea[]): Promise<void>;
  cargar(): Promise<Tarea[]>;
}

class LocalStorageRepositorio implements RepositorioTareas {
  async guardar(items: Tarea[]): Promise<void> {
    localStorage.setItem("todos", JSON.stringify(items));
  }
  async cargar(): Promise<Tarea[]> {
    const data = localStorage.getItem("todos");
    return data ? JSON.parse(data) : [];
  }
}

// Responsabilidad 3: Renderizado
class RenderizadorTareas {
  renderizar(items: Tarea[]): string {
    return items.map(i =>
      `${i.id}: ${i.titulo} [${i.done ? "✓" : " "}]`
    ).join("\n");
  }
}

// Responsabilidad 4: Comunicación
class ServicioReportes {

  }
