===============================================================================
1. CONCEPTO DE HERENCIA
===============================================================================

La herencia es un mecanismo que permite a una clase derivada (subclase o
clase hija) heredar propiedades y métodos de otra clase (superclase o
clase padre). La subclase puede:

- Usar los miembros heredados como si fueran propios.
- Añadir nuevas propiedades y métodos.
- Sobrescribir (override) métodos existentes para cambiar su comportamiento.
- Extender la funcionalidad del padre sin modificarlo.

```typescript
class Animal {
  constructor(public nombre: string) {}

  mover(distancia: number = 0): void {
    console.log(`${this.nombre} se movió ${distancia} metros.`);
  }

  emitirSonido(): void {
    console.log(`${this.nombre} hace un sonido.`);
  }
}

class Perro extends Animal {
  constructor(nombre: string) {
    super(nombre);
  }

  emitirSonido(): void {
    console.log(`${this.nombre} dice: ¡Guau!`);
  }
}

const perro = new Perro("Rex");
perro.mover(10);       // ✅ Heredado de Animal: "Rex se movió 10 metros."
perro.emitirSonido();  // ✅ Sobrescrito: "Rex dice: ¡Guau!"
```

En este ejemplo:
- `Perro` extiende `Animal`, heredando la propiedad `nombre` y el método
  `mover`.
- `Perro` sobrescribe `emitirSonido` para personalizar el comportamiento.
- El constructor de `Perro` llama a `super(nombre)` para inicializar la
  parte del padre.

===============================================================================
2. LA PALABRA CLAVE SUPER
===============================================================================

`super` es una palabra clave que permite acceder a miembros de la clase
padre desde la clase hija. Se usa principalmente para:

1. **Llamar al constructor del padre**: `super(...)` debe ser la primera
   línea del constructor de la subclase.
2. **Llamar métodos del padre**: `super.metodo(...)` para invocar la
   versión del padre de un método sobrescrito.

```typescript
class Vehiculo {
  constructor(
    public marca: string,
    public modelo: string,
    public anio: number
  ) {}

  obtenerInfo(): string {
    return `${this.marca} ${this.modelo} (${this.anio})`;
  }
}

class Automovil extends Vehiculo {
  constructor(
    marca: string,
    modelo: string,
    anio: number,
    public puertas: number
  ) {
    super(marca, modelo, anio); // Obligatorio: primera línea
  }

  obtenerInfo(): string {
    // Usamos super para obtener la info base y la extendemos
    return `${super.obtenerInfo()} - ${this.puertas} puertas`;
  }
}

const auto = new Automovil("Toyota", "Corolla", 2024, 4);
console.log(auto.obtenerInfo());
// "Toyota Corolla (2024) - 4 puertas"
```

La regla es clara: si una subclase define un constructor, debe llamar a
`super()` antes de acceder a `this`. TypeScript te lo recordará con un
error de compilación si lo olvidas.

===============================================================================
3. SOBRESCRITURA DE MÉTODOS (METHOD OVERRIDING)
===============================================================================

La sobrescritura de métodos permite a una subclase proporcionar una
implementación diferente de un método definido en la clase padre.

TypeScript verifica que la firma del método sobrescrito sea compatible
con la del padre (mismo nombre, mismos tipos de parámetros, mismo tipo
de retorno o un subtipo).

```typescript
class Empleado {
  constructor(
    public nombre: string,
    public salarioBase: number
  ) {}

  calcularSalario(): number {
    return this.salarioBase;
  }

  describir(): string {
    return `Empleado: ${this.nombre}`;
  }
}

class EmpleadoPorHoras extends Empleado {
  constructor(
    nombre: string,
    public horasTrabajadas: number,
    public tarifaPorHora: number
  ) {
    super(nombre, 0); // El salario base no aplica aquí
  }

  calcularSalario(): number {
    return this.horasTrabajadas * this.tarifaPorHora;
  }

  describir(): string {
    return `${super.describir()} (por horas)`;
  }
}

class EmpleadoComision extends Empleado {
  constructor(
    nombre: string,
    salarioBase: number,
    public ventas: number,
    public porcentajeComision: number
  ) {
    super(nombre, salarioBase);
  }

  calcularSalario(): number {
    return this.salarioBase + (this.ventas * this.porcentajeComision) / 100;
  }
}
```

Puntos importantes sobre la sobrescritura:
- El método debe tener el mismo nombre.
- Los tipos de parámetros deben ser compatibles.
- El tipo de retorno debe ser el mismo o un subtipo (covarianza).
- Puedes llamar a la versión del padre con `super.metodo()`.

4. HERENCIA DE PROPIEDADES
===============================================================================

Las propiedades también se heredan. La subclase puede acceder a las
propiedades públicas y protegidas del padre como si fueran propias.

```typescript
class Persona {
  constructor(
    public nombre: string,
    protected documento: string,
    private _seguroSocial: string
  ) {}
}

class Estudiante extends Persona {
  constructor(
    nombre: string,
    documento: string,
    seguroSocial: string,
    public matricula: string
  ) {
    super(nombre, documento, seguroSocial);
  }

  mostrarInfo(): void {
    console.log(this.nombre);       // ✅ Público: accesible
    console.log(this.documento);    // ✅ Protected: accesible en subclase
    // console.log(this._seguroSocial); // ❌ Private: solo en Persona
    console.log(this.matricula);    // ✅ Propia de Estudiante
  }
}

const estudiante = new Estudiante(
  "Ana", "DNI-12345", "SS-98765", "MAT-001"
);
console.log(estudiante.nombre);    // ✅ Público
// console.log(estudiante.documento); // ❌ Protected: solo en clase y subclases
```

La herencia respeta los modificadores de acceso:
- `public`: Accesible desde cualquier lugar.
- `protected`: Accesible desde la clase y sus subclases.
- `private`: Solo accesible desde la clase donde se define.

===============================================================================
5. CADENA DE HERENCIA
===============================================================================

TypeScript permite cadenas de herencia de cualquier profundidad. Una
clase puede extender a otra que a su vez extiende a otra, formando una
jerarquía.

```typescript
class SerVivo {
  constructor(public estaVivo: boolean = true) {}
}

class Animal extends SerVivo {
  constructor(
    public nombre: string,
    estaVivo: boolean = true
  ) {
    super(estaVivo);
  }

  respirar(): void {
    console.log(`${this.nombre} está respirando.`);
  }
}

class Mamifero extends Animal {
  constructor(nombre: string, public tienePelo: boolean = true) {
    super(nombre);
  }

  amamantar(): void {
    console.log(`${this.nombre} está amamantando.`);
  }
}

class Perro extends Mamifero {
  constructor(nombre: string, public raza: string) {
    super(nombre);
  }

  ladrar(): void {
    console.log(`${this.nombre} dice: ¡Guau!`);
  }
}

const rex = new Perro("Rex", "Pastor Alemán");
console.log(rex.estaVivo);  // ✅ Heredado de SerVivo
rex.respirar();             // ✅ Heredado de Animal
rex.amamantar();            // ✅ Heredado de Mamifero
rex.ladrar();               // ✅ Propio de Perro
```

Jerarquía: `SerVivo → Animal → Mamifero → Perro`

Cada clase añade algo nuevo:
- `SerVivo`: estado de vida
- `Animal`: nombre y respiración
- `Mamifero`: pelo y amamantar
- `Perro`: raza y ladrar

===============================================================================
6. EL PRINCIPIO DE SUSTITUCIÓN DE LISKOV (LSP)
===============================================================================

El Principio de Sustitución de Liskov (LSP) es fundamental en la herencia.
Establece que una subclase debe poder sustituir a su clase padre sin
alterar el comportamiento correcto del programa.

En TypeScript, esto se traduce en que la subclase debe:
- Respetar el contrato de la clase padre.
- No debilitar las precondiciones (no exigir más).
- No fortalecer las postcondiciones (no prometer menos).
- Mantener el mismo tipo de retorno o uno más específico.

```typescript
// ❌ Violación de LSP
class Rectangulo {
  constructor(public ancho: number, public alto: number) {}

  establecerAncho(valor: number): void {
    this.ancho = valor;
  }

  establecerAlto(valor: number): void {
    this.alto = valor;
  }

  calcularArea(): number {
    return this.ancho * this.alto;
  }
}

class Cuadrado extends Rectangulo {
  constructor(lado: number) {
    super(lado, lado);
  }

  establecerAncho(valor: number): void {
    this.ancho = valor;
    this.alto = valor; // ❌ ¡Esto viola LSP!
  }

  establecerAlto(valor: number): void {
    this.ancho = valor;
    this.alto = valor; // ❌ ¡Esto viola LSP!
  }
}

function procesarRectangulo(r: Rectangulo): void {
  r.establecerAncho(5);
  r.establecerAlto(10);
  console.log(r.calcularArea()); // Esperamos 50
}

const cuadrado = new Cuadrado(5);
procesarRectangulo(cuadrado); // ❌ Imprime 100 (25 en realidad...)
```

Para evitar esto, prefiere la composición sobre la herencia o diseña
jerarquías que realmente cumplan con el principio "es-un".

===============================================================================
7. HERENCIA VS COMPOSICIÓN
===============================================================================

Un error común de principiantes es abusar de la herencia. A veces, la
composición (tener un objeto dentro de otro) es una mejor alternativa.

| Herencia                       | Composición                  |
|--------------------------------|------------------------------|
| "Es un" (is-a)                 | "Tiene un" (has-a)           |
| Acoplamiento fuerte            | Acoplamiento débil            |
| Jerarquía rígida               | Flexibilidad                 |
| Reutilización por extensión    | Reutilización por delegación |

```typescript
// ❌ Herencia forzada
class Coche {
  constructor(public marca: string) {}
  arrancar(): void { console.log("Coche arrancado"); }
}

class Motor {
  constructor(public tipo: string) {}
  encender(): void { console.log("Motor encendido"); }
}

// Herencia incorrecta: un coche NO es un motor
class CocheHereda extends Motor {
  // ❌ Esto no tiene sentido conceptual
}

// ✅ Composición correcta: un coche TIENE un motor
class CocheComposicion {
  constructor(
    public marca: string,
    public motor: Motor // Composición
  ) {}

  arrancar(): void {
    this.motor.encender();
    console.log("Coche arrancado");
  }
}

const motor = new Motor("V6");
const coche = new CocheComposicion("Ford", motor);
coche.arrancar();
```

Regla de oro: usa herencia solo cuando exista una relación "es-un"
genuina. Para relaciones "tiene-un", usa composición.

8. MÉTODOS Y PROPIEDADES PROTEGIDAS
===============================================================================

El modificador `protected` permite que ciertos miembros sean accesibles
desde la clase y sus subclases, pero no desde fuera. Esto es útil para
compartir funcionalidad interna dentro de una jerarquía.

```typescript
class CuentaBancaria {
  constructor(
    public readonly numeroCuenta: string,
    public titular: string,
    protected saldo: number
  ) {}

  protected registrarTransaccion(tipo: string, monto: number): void {
    console.log(`${tipo}: $${monto} - Saldo: $${this.saldo}`);
  }

  depositar(monto: number): void {
    if (monto > 0) {
      this.saldo += monto;
      this.registrarTransaccion("Depósito", monto);
    }
  }
}

class CuentaAhorros extends CuentaBancaria {
  private interesAnual: number;

  constructor(
    numeroCuenta: string,
    titular: string,
    saldo: number,
    interesAnual: number
  ) {
    super(numeroCuenta, titular, saldo);
    this.interesAnual = interesAnual;
  }

  aplicarInteres(): void {
    const interes = this.saldo * (this.interesAnual / 100);
    this.saldo += interes; // ✅ protected: accesible en subclase
    this.registrarTransaccion("Interés", interes); // ✅ protected
  }
}

const cuenta = new CuentaAhorros("123", "Ana", 1000, 5);
cuenta.depositar(500);
cuenta.aplicarInteres();
// cuenta.saldo; // ❌ Error: protected
// cuenta.registrarTransaccion("test", 0); // ❌ Error: protected
```

9. HERENCIA CON PARAMETER PROPERTIES
===============================================================================

Las parameter properties funcionan perfectamente con la herencia. La
subclase puede declarar sus propias parameter properties y pasar las
del padre mediante `super()`.

```typescript
class Usuario {
  constructor(
    public id: number,
    public nombre: string,
    public email: string
  ) {}
}

class UsuarioPremium extends Usuario {
  constructor(
    id: number,
    nombre: string,
    email: string,
    public nivelPremium: number,
    public fechaExpiracion: Date
  ) {
    super(id, nombre, email);
  }
}

class UsuarioAdmin extends Usuario {
  constructor(
    id: number,
    nombre: string,
    email: string,
    public permisos: string[],
    public readonly codigoAcceso: string
  ) {
    super(id, nombre, email);
  }
}

const admin = new UsuarioAdmin(
  1, "Admin", "admin@site.com",
  ["crear", "editar", "eliminar"], "SECRET-001"
);
```

Observa que en la subclase no repetimos los modificadores de acceso para
los parámetros que se pasan al padre (a menos que queramos crear una
propiedad adicional con el mismo nombre, lo cual no es recomendable).

10. HERENCIA Y TIPADO
===============================================================================

TypeScript trata las relaciones de herencia de forma estructural y
nominal. Una subclase es un subtipo de su clase padre, lo que significa
que puede usarse en cualquier lugar donde se espere la clase padre.

```typescript
class Animal {
  constructor(public nombre: string) {}
}

class Perro extends Animal {
  constructor(nombre: string, public raza: string) {
    super(nombre);
  }
}

function presentarAnimal(animal: Animal): void {
  console.log(`Animal: ${animal.nombre}`);
}

const perro = new Perro("Rex", "Pastor Alemán");
presentarAnimal(perro); // ✅ Perro es un Animal

// Arreglos polimórficos
const animales: Animal[] = [
  new Animal("Genérico"),
  new Perro("Fido", "Labrador"),
  new Perro("Bruno", "Bulldog"),
];

animales.forEach(a => presentarAnimal(a));
```

Sin embargo, lo contrario no es cierto: no puedes usar un Animal donde se
espere un Perro, a menos que verifiques el tipo primero:

```typescript
function pasearPerro(perro: Perro): void {
  console.log(`Paseando a ${perro.nombre} (${perro.raza})`);
}

// pasearPerro(new Animal("Genérico")); // ❌ Error: Animal no es Perro

// Verificación de tipo
const animales2: Animal[] = [new Perro("Rex", "Pastor")];
animales2.forEach(a => {
  if (a instanceof Perro) {
    pasearPerro(a); // ✅ Verificamos que es Perro
  }
});
```

11. ERRORES COMUNES CON HERENCIA
===============================================================================

**Error 1: Olvidar llamar a super()**
```typescript
class Base {
  constructor(public x: number) {}
}

class Derivada extends Base {
  constructor(x: number, public y: number) {
    // ❌ Error: olvidaste super(x)
  }
}
```

**Error 2: Llamar a super() después de usar this**
```typescript
class Derivada extends Base {
  constructor(x: number, public y: number) {
    this.y = y; // ❌ Error: 'super' debe llamarse antes de acceder a 'this'
    super(x);
  }
}
```

**Error 3: Sobrescritura incompatible**
```typescript
class Base {
  metodo(x: number): string {
    return x.toString();
  }
}

class Derivada extends Base {
  // ❌ Error: los tipos de parámetros son incompatibles
  metodo(x: string): string {
    return x;
  }
}
```

**Error 4: Romper el Principio de Liskov**
Ya lo vimos con el ejemplo de Rectángulo/Cuadrado. Diseña siempre la
herencia pensando en que la subclase debe poder sustituir al padre.

===============================================================================
12. BUENAS PRÁCTICAS CON HERENCIA
===============================================================================

1. **Favorece la composición sobre la herencia**: La composición es más
   flexible y produce menor acoplamiento.

2. **Respeta el Principio de Sustitución de Liskov**: Asegúrate de que
   cualquier subclase pueda reemplazar al padre sin problemas.

3. **Mantén las jerarquías poco profundas**: Más de 3-4 niveles de
   herencia suele ser una señal de alerta (code smell).

4. **Usa protected con moderación**: Prefiere private + métodos públicos
   a menos que necesites compartir intencionadamente con subclases.

5. **Documenta el propósito de la herencia**: Explica por qué una clase
   extiende a otra y qué comportamiento añade o modifica.

6. **No dupliques código**: Si varias clases comparten código, considera
   si una clase base abstracta es apropiada.

```typescript
// Ejemplo de buena práctica: jerarquía plana con composición
class Conectable {
  conectar(): void { console.log("Conectado"); }
  desconectar(): void { console.log("Desconectado"); }
}

class Dispositivo {
  constructor(
    public nombre: string,
    public conectable: Conectable // Composición
  ) {}

  encender(): void {
    this.conectable.conectar();
    console.log(`${this.nombre} encendido`);
  }
}
```

===============================================================================
RESUMEN DEL CAPÍTULO
===============================================================================

- La herencia permite que una clase herede propiedades y métodos de otra
  usando la palabra clave `extends`.
- `super()` debe llamarse en el constructor de la subclase antes de
  acceder a `this`.
- `super.metodo()` permite invocar la versión del padre de un método
  sobrescrito.
- La sobrescritura (override) debe mantener compatibilidad de firma.
- Las propiedades `protected` son accesibles en la clase y subclases.
- `instanceof` permite verificar el tipo en tiempo de ejecución.
- El Principio de Sustitución de Liskov garantiza que una subclase pueda
  reemplazar al padre sin efectos secundarios.
- La composición ("tiene-un") es preferible a la herencia ("es-un")
  cuando no existe una relación jerárquica genuina.
- Las jerarquías profundas son difíciles de mantener.

===============================================================================
PREGUNTAS FRECUENTES
===============================================================================

❓ **¿Puedo extender múltiples clases en TypeScript?**
No, TypeScript (como JavaScript) no soporta herencia múltiple. Una clase
solo puede extender una clase padre. Sin embargo, puedes implementar
múltiples interfaces.

❓ **¿Qué pasa si la clase padre no tiene constructor?**
Si la clase padre no define un constructor, se usa el constructor vacío
por defecto. La subclase puede llamar a `super()` sin argumentos o
incluso omitir la llamada si no necesita inicializar nada del padre.

❓ **¿Puedo evitar que una clase sea extendida?**
Sí, usando el modificador `final` o `sealed`. En TypeScript no existe
un equivalente directo, pero puedes marcar el constructor como `private`
o `protected` para evitar la herencia externa.

❓ **¿Herencia o interfaces? ¿Cuándo usar cada una?**
Usa herencia para compartir implementación concreta (código) entre clases
relacionadas. Usa interfaces para definir contratos que múltiples clases
no relacionadas pueden implementar.

❓ **¿Puedo tener una clase que herede y que a su vez sea heredada?**
Sí, esto es una cadena de herencia. Es común en muchos diseños, pero evita
que sea demasiado profunda (más de 3-4 niveles).
