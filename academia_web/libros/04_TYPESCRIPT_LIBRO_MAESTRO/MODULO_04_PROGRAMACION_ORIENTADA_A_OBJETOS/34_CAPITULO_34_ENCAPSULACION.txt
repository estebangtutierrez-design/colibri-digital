===============================================================================
1. ¿QUÉ ES LA ENCAPSULACIÓN?
===============================================================================

La encapsulación es uno de los cuatro pilares de la POO (junto con
herencia, polimorfismo y abstracción). Consiste en:

1. **Ocultar el estado interno** de un objeto (sus propiedades).
2. **Exponer una interfaz controlada** para interactuar con el objeto
   (métodos públicos, getters y setters).
3. **Proteger la integridad** del objeto evitando estados inválidos.

En TypeScript, la encapsulación se logra mediante:
- **Modificadores de acceso**: `private`, `protected`, `public`.
- **Getters y Setters**: Métodos especiales que controlan el acceso a
  propiedades.
- **Propiedades privadas de JavaScript**: `#` (ver Capítulo 38).

```typescript
class CuentaBancaria {
  // Estado interno oculto
  private _saldo: number;
  private _titular: string;

  constructor(titular: string, saldoInicial: number = 0) {
    this._titular = titular;
    this._saldo = saldoInicial;
  }

  // Interfaz pública controlada
  depositar(monto: number): void {
    if (monto <= 0) {
      throw new Error("El monto debe ser positivo");
    }
    this._saldo += monto;
  }

  retirar(monto: number): void {
    if (monto <= 0) {
      throw new Error("El monto debe ser positivo");
    }
    if (monto > this._saldo) {
      throw new Error("Saldo insuficiente");
    }
    this._saldo -= monto;
  }

  obtenerSaldo(): number {
    return this._saldo;
  }

  obtenerTitular(): string {
    return this._titular;
  }
}

const cuenta = new CuentaBancaria("Ana", 1000);
cuenta.depositar(500);
cuenta.retirar(200);
console.log(cuenta.obtenerSaldo()); // 1300

// cuenta._saldo = 999999; // ❌ Error: _saldo es private
```

La encapsulación nos protege de modificaciones no controladas. Sin ella,
cualquiera podría poner un saldo negativo o modificar el titular sin
restricciones.

===============================================================================
2. PRIVATE VS PROTECTED
===============================================================================

Entender la diferencia entre `private` y `protected` es crucial para
diseñar correctamente la encapsulación:

- **`private`**: Solo accesible dentro de la clase donde se define.
  Ni siquiera las subclases pueden acceder.

- **`protected`**: Accesible dentro de la clase y sus subclases, pero
  no desde fuera.

```typescript
class Empresa {
  public nombre: string;
  private _rfc: string;         // Solo aquí
  protected _empleados: number;  // Aquí y subclases

  constructor(nombre: string, rfc: string) {
    this.nombre = nombre;
    this._rfc = rfc;
    this._empleados = 0;
  }
}

class Startup extends Empresa {
  constructor(nombre: string, rfc: string) {
    super(nombre, rfc);
  }

  contratar(): void {
    this._empleados++; // ✅ protected: accesible en subclase
    // this._rfc; // ❌ private: solo en Empresa
  }

  obtenerInfo(): string {
    return `${this.nombre} - ${this._empleados} empleados`;
  }
}

const empresa = new Empresa("TechCorp", "RFC-123");
console.log(empresa.nombre);     // ✅ Público
// console.log(empresa._rfc);    // ❌ Private
// console.log(empresa._empleados); // ❌ Protected
```

| Modificador | Misma clase | Subclase | Fuera de la clase |
|-------------|-------------|----------|-------------------|
| public      | ✅          | ✅       | ✅                |
| protected   | ✅          | ✅       | ❌                |
| private     | ✅          | ❌       | ❌                |

===============================================================================
3. GETTERS Y SETTERS
===============================================================================

Los getters y setters son métodos especiales que permiten acceder y
modificar propiedades como si fueran propiedades públicas, pero con la
ventaja de poder añadir lógica de validación o transformación.

En TypeScript se definen con las palabras clave `get` y `set`:

```typescript
class Temperatura {
  private _celsius: number = 0;

  // Getter: se accede como propiedad (sin paréntesis)
  get celsius(): number {
    return this._celsius;
  }

  // Setter: permite validación antes de asignar
  set celsius(valor: number) {
    if (valor < -273.15) {
      throw new Error("La temperatura no puede ser menor al cero absoluto");
    }
    this._celsius = valor;
  }

  // Getter para propiedad calculada
  get fahrenheit(): number {
    return (this._celsius * 9 / 5) + 32;
  }

  set fahrenheit(valor: number) {
    this.celsius = (valor - 32) * 5 / 9;
  }

  // Getter sin setter: propiedad de solo lectura calculada
  get descripcion(): string {
    if (this._celsius <= 0) return "Congelante";
    if (this._celsius <= 15) return "Frío";
    if (this._celsius <= 25) return "Templado";
    if (this._celsius <= 35) return "Cálido";
    return "Extremo";
  }
}

const temp = new Temperatura();
temp.celsius = 25;          // Usa el setter
console.log(temp.celsius);  // Usa el getter → 25
console.log(temp.fahrenheit); // 77
console.log(temp.descripcion); // "Templado"

// temp.celsius = -300; // ❌ Error: menor al cero absoluto
// temp.descripcion = "Caliente"; // ❌ Error: no hay setter
```

Los getters y setters se usan como propiedades (sin paréntesis), lo que
hace que el código sea más limpio. Por convención, la propiedad privada
suele prefijarse con `_`.

===============================================================================
4. VALIDACIÓN CON SETTERS
===============================================================================

Los setters son el lugar ideal para validar los datos antes de modificar
el estado interno del objeto:

```typescript
class Usuario {
  private _edad: number = 0;
  private _email: string = "";
  private _nombre: string = "";

  get edad(): number {
    return this._edad;
  }

  set edad(valor: number) {
    if (!Number.isInteger(valor)) {
      throw new Error("La edad debe ser un número entero");
    }
    if (valor < 0 || valor > 150) {
      throw new Error("La edad debe estar entre 0 y 150 años");
    }
    this._edad = valor;
  }

  get email(): string {
    return this._email;
  }

  set email(valor: string) {
    const emailRegex = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;
    if (!emailRegex.test(valor)) {
      throw new Error("Formato de email inválido");
    }
    this._email = valor;
  }

  get nombre(): string {
    return this._nombre;
  }

  set nombre(valor: string) {
    if (valor.trim().length < 2) {
      throw new Error("El nombre debe tener al menos 2 caracteres");
    }
    this._nombre = valor.trim();
  }
}

const usuario = new Usuario();
usuario.nombre = "Ana López";   // ✅ Se guarda "Ana López"
usuario.edad = 28;              // ✅
usuario.email = "ana@mail.com"; // ✅

// usuario.edad = -5;   // ❌ Error: edad fuera de rango
// usuario.email = "invalido"; // ❌ Error: formato inválido
```

La validación en setters garantiza que el objeto nunca entre en un estado
inválido. Esto es mucho más seguro que tener propiedades públicas que
cualquiera pueda modificar sin control.

===============================================================================
5. GETTERS PARA PROPIEDADES CALCULADAS
===============================================================================

Los getters no tienen por qué limitarse a devolver el valor de una
propiedad privada. Pueden calcular valores dinámicamente:

```typescript
class Rectangulo {
  constructor(
    private _ancho: number,
    private _alto: number
  ) {}

  get ancho(): number {
    return this._ancho;
  }

  set ancho(valor: number) {
    if (valor <= 0) throw new Error("El ancho debe ser positivo");
    this._ancho = valor;
  }

  get alto(): number {
    return this._alto;
  }

  set alto(valor: number) {
    if (valor <= 0) throw new Error("El alto debe ser positivo");
    this._alto = valor;
  }

  // Propiedades calculadas (solo getter)
  get area(): number {
    return this._ancho * this._alto;
  }

  get perimetro(): number {
    return 2 * (this._ancho + this._alto);
  }

  get diagonal(): number {
    return Math.sqrt(this._ancho ** 2 + this._alto ** 2);
  }

  get esCuadrado(): boolean {
    return this._ancho === this._alto;
  }
}

const rect = new Rectangulo(10, 5);
console.log(rect.area);       // 50
console.log(rect.perimetro);  // 30
console.log(rect.diagonal);   // 11.18
console.log(rect.esCuadrado); // false

rect.ancho = 5;
console.log(rect.esCuadrado); // true
console.log(rect.area);       // 25
```

Las propiedades calculadas son una excelente manera de exponer información
derivada sin tener que almacenarla y sin necesidad de métodos como
`calcularArea()`.

===============================================================================
6. PROPIEDADES DE SOLO LECTURA Y SOLO ESCRITURA
===============================================================================

Con getters y setters puedes crear propiedades con accesos restringidos:

- **Solo lectura (read-only)**: Getter sin setter.
- **Solo escritura (write-only)**: Setter sin getter (raro pero posible).

```typescript
class TarjetaCredito {
  private _numero: string;
  private _cvv: string;

  constructor(numero: string, cvv: string) {
    this._numero = numero;
    this._cvv = cvv;
  }

  // Solo lectura: el número se muestra pero no se modifica
  get numero(): string {
    return "****-****-****-" + this._numero.slice(-4);
  }

  // Solo escritura: el CVV se puede cambiar pero no leer
  set cvv(nuevoCVV: string) {
    if (nuevoCVV.length !== 3 || !/^\d{3}$/.test(nuevoCVV)) {
      throw new Error("CVV debe ser 3 dígitos");
    }
    this._cvv = nuevoCVV;
  }

  // Método interno que usa el CVV
  autenticar(cvvIngresado: string): boolean {
    return this._cvv === cvvIngresado;
  }
}

const tarjeta = new TarjetaCredito("1234567890123456", "123");
console.log(tarjeta.numero); // "****-****-****-3456"
tarjeta.cvv = "456";         // ✅ Setter sin getter
// console.log(tarjeta.cvv); // ❌ Error: no hay getter
```

Las propiedades de solo lectura son comunes para:
- IDs generados automáticamente.
- Fechas de creación.
- Valores calculados.
- Datos sensibles que se muestran parcialmente.

===============================================================================
7. PATRÓN: PROPIEDAD PRIVADA CON GETTER PÚBLICO
===============================================================================

Este es uno de los patrones más comunes en TypeScript: mantener una
propiedad privada y exponerla mediante un getter público. Esto te permite
cambiar la implementación interna sin afectar a los consumidores.

```typescript
class Almacenamiento {
  private _datos: Map<string, string> = new Map();

  // Getter público que expone una copia (inmutable hacia afuera)
  get datos(): ReadonlyMap<string, string> {
    return this._datos;
  }

  set claveValor(pair: { clave: string; valor: string }) {
    this._datos.set(pair.clave, pair.valor);
  }

  obtener(clave: string): string | undefined {
    return this._datos.get(clave);
  }

  guardar(clave: string, valor: string): void {
    this._datos.set(clave, valor);
  }

  get cantidad(): number {
    return this._datos.size;
  }
}

const almacen = new Almacenamiento();
almacen.guardar("tema", "oscuro");
almacen.guardar("idioma", "es");

// Los consumidores pueden leer pero no modificar directamente
const datos = almacen.datos;
console.log(datos.get("tema")); // "oscuro"
// datos.set("tema", "claro");  // ❌ Error: ReadonlyMap no tiene set
```

Este patrón es especialmente útil cuando trabajas con colecciones o
estructuras de datos que no quieres que sean modificadas externamente.

===============================================================================
8. ENCAPSULACIÓN CON PARAMETER PROPERTIES
===============================================================================

Las parameter properties ya incorporan encapsulación básica mediante los
modificadores de acceso:

```typescript
class Producto {
  constructor(
    public readonly id: number,
    public nombre: string,
    private _precio: number,
    protected descuento: number = 0
  ) {}

  // Getter para precio (con la convención de _)
  get precio(): number {
    return this._precio;
  }

  // Setter con validación
  set precio(valor: number) {
    if (valor <= 0) {
      throw new Error("El precio debe ser positivo");
    }
    this._precio = valor;
  }

  get precioFinal(): number {
    return this._precio - (this._precio * this.descuento) / 100;
  }
}

const prod = new Producto(1, "Laptop", 15000);
console.log(prod.id);         // ✅ 1 (public readonly)
console.log(prod.nombre);     // ✅ "Laptop" (public)
prod.precio = 14000;          // ✅ Usa el setter
console.log(prod.precio);     // ✅ Usa el getter
```

Combinando parameter properties con getters y setters, obtienes un código
muy expresivo y seguro.

===============================================================================
9. ENCAPSULACIÓN VS RENDIMIENTO
===============================================================================

Una pregunta común es: ¿los getters y setters afectan el rendimiento?
La respuesta corta es que en la práctica, el impacto es mínimo y vale
la pena por los beneficios en seguridad y mantenibilidad.

```typescript
class Punto {
  constructor(
    private _x: number,
    private _y: number
  ) {}

  get x(): number { return this._x; }
  set x(valor: number) { this._x = valor; }
  get y(): number { return this._y; }
  set y(valor: number) { this._y = valor; }
}

// vs

class PuntoSimple {
  constructor(
    public x: number,
    public y: number
  ) {}
}
```

El enfoque encapsulado es ligeramente más lento, pero:
- Puedes añadir validación después sin cambiar la API.
- Puedes cambiar la implementación interna (ej: usar array en lugar de
  dos propiedades).
- Puedes añadir logging, disparar eventos, etc.

La regla general: empieza con properties públicas si la clase es simple
y estás seguro de que no necesitarás validación. Usa encapsulación desde
el principio si anticipas crecimiento o si la integridad de los datos
es crítica.

===============================================================================
10. ENCAPSULACIÓN EN JAVASCRIPT VS TYPESCRIPT
===============================================================================

Es importante entender que los modificadores `private` y `protected` de
TypeScript son solo en tiempo de compilación. En JavaScript compilado,
todas las propiedades son públicas.

```typescript
// TypeScript
class EjemploTS {
  private secreto: string = "123";
  public nombre: string = "público";
}

// JavaScript compilado (target ES2015)
class EjemploJS {
  constructor() {
    this.secreto = "123";
    this.nombre = "público";
  }
}
// ¡Ambas propiedades son públicas en JS!
```

Si necesitas verdadera privacidad en tiempo de ejecución, JavaScript
ofrece las propiedades privadas con `#` (desde ES2021):

```typescript
class EjemploRealmentePrivado {
  #secreto: string = "123"; // Privado de verdad en JS
  public nombre: string = "público";

  obtenerSecreto(): string {
    return this.#secreto; // ✅ Acceso interno
  }
}

const ej = new EjemploRealmentePrivado();
console.log(ej.nombre); // "público"
// console.log(ej.#secreto); // ❌ Error de sintaxis en JS
```

Las propiedades con `#` son privadas también en tiempo de ejecución,
no solo en compilación. Sin embargo, las herramientas de TypeScript
suelen ser suficientes y más compatibles con versiones anteriores.

11. BUENAS PRÁCTICAS DE ENCAPSULACIÓN
===============================================================================

1. **Mantén las propiedades privadas por defecto**: Solo haz pública una
   propiedad si realmente necesitas que sea accesible directamente.

2. **Usa getters para propiedades calculadas**: En lugar de métodos
   como `obtenerNombreCompleto()`, usa un getter `nombreCompleto`.

3. **Valida en setters, no en getters**: El getter debe ser rápido y
   sin efectos secundarios. Las validaciones van en el setter.

4. **Nunca expongas referencias mutables directamente**: Si devuelves
   un array o mapa, devuelve una copia o una versión readonly.

5. **Usa el prefijo `_` para propiedades privadas**: Aunque no es
   obligatorio, es una convención ampliamente adoptada.

6. **Documenta el contrato de la clase**: Explica qué hace cada getter
   y setter, especialmente si tienen validaciones o efectos secundarios.

```typescript
class Biblioteca {
  private _libros: string[] = [];

  // Devuelve una copia para evitar modificaciones externas
  get libros(): ReadonlyArray<string> {
    return [...this._libros];
  }

  get cantidad(): number {
    return this._libros.length;
  }

  agregarLibro(titulo: string): void {
    if (!titulo || titulo.trim().length === 0) {
      throw new Error("El título no puede estar vacío");
    }
    if (this._libros.includes(titulo)) {
      throw new Error("El libro ya existe en la biblioteca");
    }
    this._libros.push(titulo.trim());
  }

  eliminarLibro(titulo: string): boolean {
    const index = this._libros.indexOf(titulo);
    if (index === -1) return false;
    this._libros.splice(index, 1);
    return true;
  }
}
```

12. CASO PRÁCTICO: SISTEMA DE CUENTAS
===============================================================================

Veamos un ejemplo completo que integra todos los conceptos de
encapsulación:

```typescript
class CuentaAhorros {
  private _saldo: number;
  private _transacciones: string[] = [];
  private static _tasaInteres: number = 3.5;

  constructor(
    public readonly numeroCuenta: string,
    public readonly titular: string,
    saldoInicial: number = 0
  ) {
    if (saldoInicial < 0) {
      throw new Error("El saldo inicial no puede ser negativo");
    }
    this._saldo = saldoInicial;
    this._registrarTransaccion("APERTURA", saldoInicial);
  }

  static get tasaInteres(): number {
    return CuentaAhorros._tasaInteres;
  }

  static set tasaInteres(valor: number) {
    if (valor <= 0 || valor > 20) {
      throw new Error("La tasa de interés debe estar entre 0 y 20%");
    }
    CuentaAhorros._tasaInteres = valor;
  }

  get saldo(): number {
    return this._saldo;
  }

  get transacciones(): ReadonlyArray<string> {
    return [...this._transacciones];
  }

  get interesGenerado(): number {
    return this._saldo * (CuentaAhorros._tasaInteres / 100);
  }

  depositar(monto: number): void {
    if (monto <= 0) throw new Error("El monto a depositar debe ser positivo");
    this._saldo += monto;
    this._registrarTransaccion("DEPÓSITO", monto);
  }

  retirar(monto: number): void {
    if (monto <= 0) throw new Error("El monto a retirar debe ser positivo");
    if (monto > this._saldo) throw new Error("Saldo insuficiente");
    this._saldo -= monto;
    this._registrarTransaccion("RETIRO", monto);
  }

  private _registrarTransaccion(tipo: string, monto: number): void {
    const fecha = new Date().toISOString();
    this._transacciones.push(`[${fecha}] ${tipo}: $${monto}`);
  }
}

const cuenta = new CuentaAhorros("AHO-001", "María García", 5000);
cuenta.depositar(1000);
cuenta.retirar(2000);
console.log(cuenta.saldo);           // 4000
console.log(cuenta.interesGenerado); // 140
console.log(cuenta.transacciones);   // Copia del historial
```

===============================================================================
RESUMEN DEL CAPÍTULO
===============================================================================

- La encapsulación protege el estado interno del objeto y expone una
  interfaz controlada.
- `private` restringe el acceso a la propia clase; `protected` permite
  acceso también a subclases.
- Los getters (`get`) exponen propiedades con lógica de lectura.
- Los setters (`set`) permiten validación y control al modificar valores.
- Los getters sin setters crean propiedades de solo lectura.
- Las propiedades calculadas (getters) exponen datos derivados sin
  almacenarlos.
- La encapsulación mejora la mantenibilidad al desacoplar interfaz de
  implementación.
- TypeScript ofrece privacidad en compilación; `#` ofrece privacidad en
  tiempo de ejecución.

PREGUNTAS FRECUENTES
===============================================================================

❓ **¿Getter y setter son obligatorios para encapsular?**
No. Puedes tener propiedades privadas y métodos públicos que las modifiquen
sin usar getters/setters. Los getters/setters son útiles cuando quieres
acceder/modificar con sintaxis de propiedad en lugar de método.

❓ **¿Debo usar getters para todo?**
No. Si una propiedad es pública y no necesita validación ni lógica extra,
no necesitas getter/setter. Usa `public` directamente. La encapsulación
no significa que todo deba ser privado con getter.

❓ **¿Los getters pueden ser async?**
No, los getters no pueden ser asíncronos. Si necesitas obtener un valor
de forma asíncrona, usa un método que retorne una Promise.

❓ **¿Cuál es la diferencia entre getter y método?**
Los getters se usan como propiedades (sin paréntesis) y deben ser rápidos
y sin efectos secundarios. Los métodos pueden tener efectos secundarios,
recibir parámetros y ser asíncronos. Como regla: si tiene parámetros o
es async, debe ser método, no getter.

❓ **¿Cómo decido si algo debe ser private o protected?**
Usa `private` por defecto. Usa `protected` solo si estás diseñando una
clase base que será extendida y necesitas que las subclases accedan a
ciertos miembros internos.
