===============================================================================
1. NATURALEZA FUNDAMENTAL: VALOR VS TIPO
===============================================================================

La diferencia más fundamental entre clases e interfaces es:

- **Una clase genera un VALOR** (existe en JavaScript compilado).
- **Una interfaz genera un TIPO** (desaparece al compilar a JS).

```typescript
// INTERFAZ: solo existe en tiempo de compilación
interface Punto {
  x: number;
  y: number;
}

// CLASE: existe en tiempo de ejecución
class PuntoImpl {
  constructor(public x: number, public y: number) {}
}
```

Veamos cómo se ven en JavaScript compilado:

```typescript
// TypeScript
interface UsuarioInterface {
  nombre: string;
  email: string;
}

class UsuarioClass {
  constructor(public nombre: string, public email: string) {}
}

// JavaScript compilado
// La interfaz desapareció completamente
class UsuarioClass {
  constructor(nombre, email) {
    this.nombre = nombre;
    this.email = email;
  }
}
```

Esta diferencia es crucial porque significa:
- Las interfaces no pueden tener constructores, métodos con
  implementación, modificadores de acceso (public/private), o
  propiedades con valores iniciales.
- Las clases pueden tener todo eso y pueden ser instanciadas con `new`.

```typescript
// ❌ Esto NO es posible con interfaces
interface VehiculoInterface {
  private acelerar(): void; // Error: private no existe en interfaces
  constructor(marca: string); // Error: constructor no existe en interfaces
  tipo: string = "vehiculo"; // Error: no se permiten valores iniciales
}

// ✅ Esto SÍ es posible con clases
class VehiculoBase {
  protected acelerar(): void {}
  constructor(public marca: string) {}
  tipo: string = "vehiculo";
}
```

===============================================================================
2. CUÁNDO USAR INTERFACES
===============================================================================

Las interfaces son ideales para:

**1. Definir contratos que múltiples clases implementan:**

```typescript
interface Serializable {
  toJSON(): object;
  fromJSON(datos: object): void;
}

class Usuario implements Serializable {
  constructor(
    public nombre: string,
    public email: string,
    public edad: number
  ) {}

  toJSON(): object {
    return {
      nombre: this.nombre,
      email: this.email,
      edad: this.edad
    };
  }

  fromJSON(datos: object): void {
    const d = datos as any;
    this.nombre = d.nombre;
    this.email = d.email;
    this.edad = d.edad;
  }
}

class Configuracion implements Serializable {
  constructor(
    public tema: string,
    public idioma: string
  ) {}

  toJSON(): object {
    return { tema: this.tema, idioma: this.idioma };
  }

  fromJSON(datos: object): void {
    const d = datos as any;
    this.tema = d.tema;
    this.idioma = d.idioma;
  }
}
```

**2. Definir la forma de objetos literales o datos:**

```typescript
// Ideal para datos que cruzan la red
interface RespuestaAPI<T> {
  exito: boolean;
  datos: T;
  mensaje?: string;
  paginacion?: {
    pagina: number;
    totalPaginas: number;
    total: number;
  };
}

interface ProductoDTO {
  id: string;
  nombre: string;
  precio: number;
  categoria: string;
}

async function obtenerProductos(): Promise<RespuestaAPI<ProductoDTO[]>> {
  const respuesta = await fetch("/api/productos");
  return respuesta.json();
}
```

**3. Tipar parámetros de funciones:**

```typescript
interface OpcionesBusqueda {
  query: string;
  pagina?: number;
  limite?: number;
  filtros?: Record<string, string>;
  ordenar?: "asc" | "desc";
}

function buscar<T>(opciones: OpcionesBusqueda): Promise<T[]> {
  // ...
}
```

**4. Definir tipos para props de componentes (React, Vue, etc.):**

```typescript
interface BotonProps {
  texto: string;
  variante: "primario" | "secundario" | "peligro";
  deshabilitado?: boolean;
  onClick: () => void;
}

// En React:
function Boton(props: BotonProps) {
  return (
    <button
      className={`btn btn-${props.variante}`}
      disabled={props.deshabilitado}
      onClick={props.onClick}
    >
      {props.texto}
    </button>
  );
}
```

===============================================================================
3. CUÁNDO USAR CLASES
===============================================================================

Las clases son ideales para:

**1. Crear múltiples objetos con comportamiento:**

```typescript
class Contador {
  private valor: number = 0;

  incrementar(): number {
    return ++this.valor;
  }

  decrementar(): number {
    return --this.valor;
  }

  obtenerValor(): number {
    return this.valor;
  }

  reiniciar(): void {
    this.valor = 0;
  }
}

const contador1 = new Contador();
const contador2 = new Contador();
contador1.incrementar();
contador1.incrementar();
contador2.incrementar();
console.log(contador1.obtenerValor()); // 2
console.log(contador2.obtenerValor()); // 1
```

**2. Encapsular estado con lógica de validación:**

```typescript
class CuentaBancaria {
  private _saldo: number;

  constructor(saldoInicial: number = 0) {
    if (saldoInicial < 0) {
      throw new Error("El saldo inicial no puede ser negativo");
    }
    this._saldo = saldoInicial;
  }

  get saldo(): number {
    return this._saldo;
  }

  depositar(monto: number): void {
    if (monto <= 0) throw new Error("El monto debe ser positivo");
    this._saldo += monto;
  }

  retirar(monto: number): void {
    if (monto <= 0) throw new Error("El monto debe ser positivo");
    if (monto > this._saldo) throw new Error("Saldo insuficiente");
    this._saldo -= monto;
  }
}
```

**3. Polimorfismo con herencia y métodos compartidos:**

```typescript
abstract class Formateador {
  abstract formatear(datos: any): string;

  formatearYMostrar(datos: any): void {
    console.log(this.formatear(datos));
  }
}

class FormateadorJSON extends Formateador {
  formatear(datos: any): string {
    return JSON.stringify(datos, null, 2);
  }
}

class FormateadorCSV extends Formateador {
  formatear(datos: any): string {
    if (Array.isArray(datos)) {
      const cabeceras = Object.keys(datos[0] || {}).join(",");
      const filas = datos.map(f => Object.values(f).join(","));
      return [cabeceras, ...filas].join("\n");
    }
    return String(datos);
  }
}
```

**4. Patrones que requieren instancias (Singleton, Builder, etc.):**

```typescript
class Singleton {
  private static instancia: Singleton;
  private constructor() {}

  static obtener(): Singleton {
    if (!Singleton.instancia) {
      Singleton.instancia = new Singleton();
    }
    return Singleton.instancia;
  }
}
```

===============================================================================
4. CLASES COMO TIPOS
===============================================================================

Una característica única de TypeScript es que las clases generan
automáticamente un tipo con la misma forma. Esto significa que una
clase puede usarse como tipo:

```typescript
class Producto {
  constructor(
    public id: number,
    public nombre: string,
    public precio: number
  ) {}
}

// La clase Producto también es un tipo
function calcularIVA(producto: Producto): number {
  return producto.precio * 0.16;
}

// Los objetos literales compatibles también son aceptados
const productoLiteral = {
  id: 1,
  nombre: "Laptop",
  precio: 15000
};

console.log(calcularIVA(productoLiteral)); // ✅ 2400
```

Pero hay una diferencia importante: el tipo generado por una clase es
estructural, no nominal. TypeScript solo verifica la forma, no si el
objeto fue creado con `new Producto()`.

```typescript
class Persona {
  constructor(public nombre: string) {}
}

class Empleado {
  constructor(public nombre: string) {}
}

const p: Persona = new Empleado("Juan"); // ✅ Estructuralmente compatible
```

===============================================================================
5. INTERFAZ VS TYPE ALIAS
===============================================================================

Aunque el capítulo se centra en interfaces vs clases, también es
importante entender la diferencia entre `interface` y `type`:

```typescript
// Interface
interface Usuario {
  nombre: string;
  email: string;
}

// Type alias
type UsuarioTipo = {
  nombre: string;
  email: string;
};
```

Diferencias clave:

| Característica           | Interface                    | Type Alias                 |
|--------------------------|------------------------------|----------------------------|
| Extensión                | extends (herencia)           | & (intersección)           |
| Declaración merging      | ✅ Sí                        | ❌ No                      |
| Tipos primitivos         | ❌ No                        | ✅ Sí                      |
| Tuplas                   | ❌ No                        | ✅ Sí                      |
| Union types              | ❌ No                        | ✅ Sí                      |
| Mapped types             | ❌ No                        | ✅ Sí                      |
| Herencia múltiple        | extends A, B                 | A & B                      |

```typescript
// Type alias puede hacer cosas que interface no puede:
type ID = string | number;
type Par<T, U> = [T, U];
type Opcional<T> = T | null | undefined;
type Callback = (error: Error | null, resultado?: any) => void;

// Interface puede hacer cosas que type no puede:
interface Usuario { nombre: string; }
interface Usuario { edad: number; } // ✅ Merging: Usuario tiene nombre y edad

// type UsuarioTipo = { nombre: string; };
// type UsuarioTipo = { edad: number; }; // ❌ Error: duplicado
```

Regla general: prefiere `interface` para objetos/contratos públicos.
Usa `type` para uniones, tuplas, primitivos y tipos complejos.

===============================================================================
6. DECLARATION MERGING DE INTERFACES
===============================================================================

Una característica única de las interfaces es que puedes declararlas
múltiples veces y TypeScript las fusiona automáticamente:

```typescript
// Primera declaración
interface Usuario {
  nombre: string;
  email: string;
}

// Segunda declaración (merging)
interface Usuario {
  edad?: number;
  telefono?: string;
}

// Resultado: Usuario tiene nombre, email, edad (opcional) y telefono (opcional)
const usuario: Usuario = {
  nombre: "Ana",
  email: "ana@mail.com",
  edad: 28
};
```

Esto es muy útil para:
- Extender tipos de librerías externas.
- Añadir propiedades a interfaces globales.
- Separar declaraciones en diferentes archivos.

```typescript
// global.d.ts
interface Window {
  miApp: {
    version: string;
    iniciar(): void;
  };
}

// Ahora window.miApp existe globalmente
window.miApp = {
  version: "1.0.0",
  iniciar: () => console.log("App iniciada")
};
```

Las clases NO tienen esta capacidad. Si declaras una clase dos veces,
obtendrás un error.

===============================================================================
7. PATRÓN: CLASE QUE IMPLEMENTA INTERFAZ
===============================================================================

Este es uno de los patrones más comunes y útiles en TypeScript:
usar una interfaz para definir el contrato y una clase para la
implementación.

```typescript
interface Almacenable {
  guardar(): void;
  cargar(): void;
  eliminar(): void;
}

// La interfaz define el QUÉ, la clase el CÓMO
class AlmacenamientoLocal implements Almacenable {
  constructor(private clave: string) {}

  guardar(): void {
    console.log(`Guardando en localStorage: ${this.clave}`);
  }

  cargar(): void {
    console.log(`Cargando de localStorage: ${this.clave}`);
  }

  eliminar(): void {
    console.log(`Eliminando de localStorage: ${this.clave}`);
  }
}

class AlmacenamientoNube implements Almacenable {
  constructor(
    private ruta: string,
    private token: string
  ) {}

  guardar(): void {
    console.log(`Guardando en la nube: ${this.ruta}`);
  }

  cargar(): void {
    console.log(`Cargando de la nube: ${this.ruta}`);
  }

  eliminar(): void {
    console.log(`Eliminando de la nube: ${this.ruta}`);
  }
}

// Uso polimórfico
function backup(almacenamientos: Almacenable[]): void {
  almacenamientos.forEach(a => a.guardar());
}

const local = new AlmacenamientoLocal("datos");
const nube = new AlmacenamientoNube("/backup", "token-secreto");
backup([local, nube]);
```

Este patrón separa el contrato (interfaz) de la implementación (clase),
permitiendo intercambiar implementaciones sin afectar a los consumidores.

===============================================================================
8. PATRÓN: INTERFAZ CON PROPIEDADES VS CLASE CON MÉTODOS
===============================================================================

A veces tienes datos puros (DTOs) que solo necesitan propiedades, y
otras veces necesitas comportamiento. La decisión entre interfaz y clase
refleja esta diferencia:

```typescript
// ✅ DTO: solo datos, usa INTERFAZ
interface PedidoDTO {
  id: string;
  cliente: string;
  productos: { id: string; cantidad: number }[];
  total: number;
  fecha: string;
}

// ✅ Entidad de dominio: datos + comportamiento, usa CLASE
class Pedido {
  private _productos: Map<string, number> = new Map();

  constructor(
    public readonly id: string,
    public readonly cliente: string,
    public readonly fecha: Date = new Date()
  ) {}

  agregarProducto(productoId: string, cantidad: number): void {
    const actual = this._productos.get(productoId) ?? 0;
    this._productos.set(productoId, actual + cantidad);
  }

  eliminarProducto(productoId: string): void {
    this._productos.delete(productoId);
  }

  get productos(): ReadonlyMap<string, number> {
    return this._productos;
  }

  get total(): number {
    // Cálculo complejo con precios, descuentos, impuestos...
    return 0;
  }

  toDTO(): PedidoDTO {
    return {
      id: this.id,
      cliente: this.cliente,
      productos: Array.from(this._productos.entries()).map(
        ([id, cantidad]) => ({ id, cantidad })
      ),
      total: this.total,
      fecha: this.fecha.toISOString()
    };
  }

  static fromDTO(dto: PedidoDTO): Pedido {
    const pedido = new Pedido(dto.id, dto.cliente, new Date(dto.fecha));
    dto.productos.forEach(p => pedido.agregarProducto(p.id, p.cantidad));
    return pedido;
  }
}
```

Regla: si solo necesitas transportar datos, usa interfaz. Si necesitas
comportamiento, validación, o encapsulación, usa clase.

===============================================================================
9. INTERFAZ VS CLASE PARA MODELOS DE DATOS
===============================================================================

Una pregunta frecuente: "¿Debo modelar mis datos como interfaz o como
clase?" Depende del contexto:

**Caso 1: API responses (datos que llegan del servidor)**
Usa INTERFAZ. Los datos llegan como objetos literales JSON.

```typescript
interface UsuarioAPI {
  id: string;
  nombre: string;
  email: string;
  fecha_registro: string;
}

async function obtenerUsuario(id: string): Promise<UsuarioAPI> {
  const res = await fetch(`/api/usuarios/${id}`);
  return res.json();
}
```

**Caso 2: Modelos de dominio con lógica de negocio**
Usa CLASE. Necesitas comportamiento.

```typescript
class Usuario {
  private _cambios: string[] = [];

  constructor(
    public readonly id: string,
    private _nombre: string,
    private _email: string,
    public readonly fechaRegistro: Date
  ) {}

  get nombre(): string { return this._nombre; }

  set nombre(valor: string) {
    if (valor.trim().length < 2) {
      throw new Error("El nombre debe tener al menos 2 caracteres");
    }
    this._cambios.push(`Nombre cambiado a: ${valor}`);
    this._nombre = valor.trim();
  }

  get email(): string { return this._email; }

  set email(valor: string) {
    const regex = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;
    if (!regex.test(valor)) throw new Error("Email inválido");
    this._cambios.push(`Email cambiado a: ${valor}`);
    this._email = valor;
  }

  obtenerHistorialCambios(): ReadonlyArray<string> {
    return [...this._cambios];
  }
}
```

**Caso 3: State management (Redux, Zustand, etc.)**
Usa INTERFAZ o TYPE. Los estados suelen ser objetos inmutables.

```typescript
interface EstadoApp {
  usuario: UsuarioAPI | null;
  productos: ProductoAPI[];
  carrito: { productoId: string; cantidad: number }[];
  ui: {
    tema: "claro" | "oscuro";
    sidebarAbierta: boolean;
  };
}
```

10. ¿PUEDO USAR UNA CLASE SIN INTERFAZ?
===============================================================================

¡Absolutamente! No todo requiere una interfaz. Puedes usar clases solas
cuando no necesitas el nivel extra de abstracción.

```typescript
// Perfectamente válido: clase sin interfaz
class Calculadora {
  private valor: number = 0;

  agregar(n: number): this {
    this.valor += n;
    return this;
  }

  restar(n: number): this {
    this.valor -= n;
    return this;
  }

  multiplicar(n: number): this {
    this.valor *= n;
    return this;
  }

  dividir(n: number): this {
    if (n === 0) throw new Error("División por cero");
    this.valor /= n;
    return this;
  }

  obtenerResultado(): number {
    return this.valor;
  }

  reiniciar(): void {
    this.valor = 0;
  }
}
```

Y también puedes tener interfaces sin clases:

```typescript
// Interfaz que describe un objeto literal
interface Direccion {
  calle: string;
  numero: number;
  ciudad: string;
  codigoPostal: string;
  pais: string;
}

// Uso directo sin clase
const direccion: Direccion = {
  calle: "Av. Siempre Viva",
  numero: 742,
  ciudad: "Springfield",
  codigoPostal: "12345",
  pais: "EE.UU."
};
```

No hay una regla que obligue a usar interfaz + clase siempre. La
herramienta correcta depende del problema.

===============================================================================
12. GUÍA DE DECISIÓN RÁPIDA
===============================================================================

Regla práctica: ¿new() + estado/comportamiento? → CLASE.
¿Solo forma/contrato? → INTERFAZ. ¿Union/tuple? → TYPE.

Recomendaciones generales:

1. **Por defecto, usa INTERFAZ** para definir la forma de objetos.
2. **Usa CLASE** cuando necesites encapsular estado + comportamiento.
3. **Usa TYPE** para uniones, intersecciones, tuplas y tipos complejos.
4. **No añadas una interfaz por cada clase** - solo si necesitas el
   nivel extra de abstracción.
5. **Para DTOs y datos de API, prefiere INTERFAZ**.
6. **Para entidades de dominio con lógica, prefiere CLASE**.

```typescript
// Ejemplo final: combinación equilibrada

// 1. Interfaces para contratos
interface Repositorio<T> {
  obtenerTodos(): Promise<T[]>;
  obtenerPorId(id: string): Promise<T | null>;
  guardar(item: T): Promise<T>;
  eliminar(id: string): Promise<boolean>;
}

interface Logger {
  info(mensaje: string): void;
  error(mensaje: string, error?: Error): void;
  warn(mensaje: string): void;
}

// 2. Clase que implementa interfaces
class RepositorioUsuarios implements Repositorio<Usuario> {
  private usuarios: Map<string, Usuario> = new Map();

  constructor(private logger: Logger) {}

  async obtenerTodos(): Promise<Usuario[]> {
    this.logger.info("Obteniendo todos los usuarios");
    return Array.from(this.usuarios.values());
  }

  async obtenerPorId(id: string): Promise<Usuario | null> {
    this.logger.info(`Buscando usuario ${id}`);
    return this.usuarios.get(id) ?? null;
  }

  async guardar(usuario: Usuario): Promise<Usuario> {
    this.logger.info(`Guardando usuario ${usuario.id}`);
    this.usuarios.set(usuario.id, usuario);
    return usuario;
  }

  async eliminar(id: string): Promise<boolean> {
    this.logger.info(`Eliminando usuario ${id}`);
    return this.usuarios.delete(id);
  }
}

// 3. Uso
const logger: Logger = {
  info: (m) => console.log(`[INFO] ${m}`),
  error: (m, e) => console.error(`[ERROR] ${m}`, e),
  warn: (m) => console.warn(`[WARN] ${m}`)
};

const repo = new RepositorioUsuarios(logger);
```

===============================================================================
RESUMEN DEL CAPÍTULO
===============================================================================

- Las interfaces generan tipos que desaparecen en tiempo de compilación.
  Las clases generan valores que existen en tiempo de ejecución.
- Usa interfaces para contratos, formas de objetos, DTOs y tipado de
  parámetros.
- Usa clases para encapsulación, comportamiento, herencia y patrones
  que requieren instancias.
- Las clases generan automáticamente tipos (son valor y tipo a la vez).
- `interface` permite declaration merging; `type` no.
- El patrón interfaz + clase implementadora es el estándar para diseño
  desacoplado.
- No es obligatorio usar interfaz con cada clase. Evalúa según el caso.

🧩 **Ejercicio 2: Refactorizar Datos a Interfaz**
Refactoriza el siguiente código para usar una interfaz para los datos
y una clase para la lógica:
```typescript
class Producto {
  constructor(
    public id: number,
    public nombre: string,
    public precio: number,
    public stock: number
  ) {}

  aplicarDescuento(porcentaje: number): number {
    return this.precio * (1 - porcentaje / 100);
  }

  hayStock(cantidad: number): boolean {
    return this.stock >= cantidad;
  }
}
```

PREGUNTAS FRECUENTES
===============================================================================

❓ **¿Puedo usar una interfaz para definir el tipo de una instancia de clase?**
Sí. Toda clase genera un tipo. Puedes usar `typeof MiClase` para el tipo
del constructor, o directamente `MiClase` para el tipo de instancia.

❓ **¿Es siempre mejor usar interfaces que clases para datos?**
Generalmente sí, porque las interfaces no generan código JavaScript y
son más ligeras. Pero si necesitas validación o métodos en los datos,
una clase es mejor.

❓ **¿Puedo hacer que una clase implemente múltiples interfaces?**
Sí, una clase puede implementar múltiples interfaces separadas por coma:
`class MiClase implements InterfazA, InterfazB`.

❓ **¿Interfaz o type para objetos?**
Para objetos, prefiere interfaz porque ofrece declaration merging y
mensajes de error más claros. Usa type para uniones, tuplas y tipos
complejos.

❓ **¿Necesito definir una interfaz para cada clase?**
No. La interfaz es útil cuando esperas tener múltiples implementaciones
del mismo contrato. Si solo tienes una implementación y no anticipas
otras, puedes usar la clase directamente.
