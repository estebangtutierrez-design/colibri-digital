1. ESTÁNDARES DE CODIFICACIÓN PSR-12
===============================================================================

PSR-12 es el estándar de codificación recomendado por PHP-FIG (Framework
Interoperability Group). Define reglas para que el código PHP sea
consistente y legible independientemente de quién lo escriba.

```php
<?php

// Reglas fundamentales de PSR-12

declare(strict_types=1);

namespace App\Services;

use App\Contracts\PaymentGatewayInterface;
use App\Exceptions\PaymentException;
use DateTimeImmutable;

/**
 * Servicio de procesamiento de pagos.
 *
 * Maneja la lógica de negocio para transacciones
 * financieras con múltiples pasarelas.
 */
class PaymentService implements PaymentGatewayInterface
{
    // 1. Líneas de máximo 120 caracteres
    // 2. Identación con 4 espacios (no tabs)
    // 3. Una declaración por línea

    private const MAX_RETRIES = 3;
    private const TIMEOUT_SECONDS = 30;

    public function __construct(
        // 4. Propiedades promovidas desde PHP 8.0
        private array $config,
        private LoggerInterface $logger,
        private HttpClientInterface $httpClient
    ) {
    }

    // 5. Espacio después de palabras clave (if, foreach, while)
    // 6. Llave de apertura en la misma línea para clases/métodos
    public function processPayment(
        float $amount,
        string $currency,
        array $metadata = []
    ): PaymentResult {

        // 7. Espacios alrededor de operadores
        $maxAmount = $this->config['max_transaction'] ?? 10000.00;

        if ($amount > $maxAmount) {
            // 8. Excepciones para flujo de control excepcional
            throw new PaymentException(
                "Monto excede el límite máximo de {$maxAmount}",
                PaymentException::EXCEEDS_LIMIT
            );
        }

        // 9. Llamadas a métodos sin espacio antes del paréntesis
        $this->logger->info('Procesando pago', [
            'amount'   => $amount,
            'currency' => $currency,
            // 10. Coma final en arrays multilínea
            'timestamp' => (new DateTimeImmutable())->format('c'),
        ]);

        // 11. foreach con llaves en la misma línea
        foreach ($this->config['gateways'] as $gateway) {
            $result = $this->tryGateway($gateway, $amount, $currency);

            if ($result->isSuccess()) {
                return $result;
            }
        }

        throw new PaymentException('Todos los gateways fallaron');
    }

    // 12. Métodos con visibilidad explícita (public, protected, private)
    // 13. Tipo de retorno declarado
    private function tryGateway(
        string $gateway,
        float $amount,
        string $currency
    ): PaymentResult {

        try {
            // 14. catch en la misma línea que la llave de cierre
            $response = $this->httpClient->post($gateway, [
                'amount'   => $amount,
                'currency' => $currency,
            ]);

            return PaymentResult::fromResponse($response);

        } catch (ConnectionException $e) {
            $this->logger->warning("Gateway {$gateway} no disponible", [
                'error' => $e->getMessage(),
            ]);

            // 15. throw sin paréntesis alrededor del objeto
            throw $e;
        }
    }

    // 16. Métodos estáticos con visibilidad
    public static function createFromConfig(array $config): self
    {
        return new self(
            $config,
            new FileLogger(),
            new CurlHttpClient()
        );
    }
}
```

2. GIT WORKFLOW PROFESIONAL
===============================================================================

El flujo de trabajo con Git es esencial para la colaboración en equipo.

**Git Flow** (para proyectos con versiones):

```bash
# Ramas principales
main        # Código en producción
develop     # Integración de características

# Ramas de soporte
feature/*   # Nuevas características (desde develop)
release/*   # Preparación de lanzamiento (desde develop)
hotfix/*    # Correcciones urgentes en producción (desde main)

# Ciclo de vida de una característica
git checkout develop
git checkout -b feature/pago-con-paypal

# Trabajar en la característica...
git add .
git commit -m "feat: agregar integración con PayPal"

# Al terminar, fusionar a develop
git checkout develop
git merge --no-ff feature/pago-con-paypal
git branch -d feature/pago-con-paypal

# Preparar release
git checkout develop
git checkout -b release/v1.2.0
# Últimos ajustes, pruebas...
git checkout main
git merge --no-ff release/v1.2.0
git tag v1.2.0
git checkout develop
git merge --no-ff release/v1.2.0

# Hotfix urgente
git checkout main
git checkout -b hotfix/seguridad-login
# Corregir...
git checkout main
git merge --no-ff hotfix/seguridad-login
git tag v1.2.1
git checkout develop
git merge --no-ff hotfix/seguridad-login
```

**Trunk-Based Development** (para entrega continua):

```bash
# Una sola rama principal (main/trunk)
# Ramas de corta duración (max 1 día)
# Integración frecuente (varias veces al día)

# Cycle
git checkout main
git pull --rebase
git checkout -b feat/nueva-funcionalidad

# Trabajar en pequeños commits
git commit -m "refactor: extraer validación a método propio"

# Integrar inmediatamente
git checkout main
git pull --rebase
git merge --squash feat/nueva-funcionalidad
git branch -D feat/nueva-funcionalidad
git push origin main
```

**Conventional Commits**:

```bash
# Formato: tipo(alcance): descripción

feat:         # Nueva característica
fix:          # Corrección de bug
refactor:     # Refactorización sin cambios funcionales
docs:         # Documentación
style:        # Formato, estilos (no funcional)
test:         # Tests
chore:        # Tareas de mantenimiento
perf:         # Mejora de rendimiento
ci:           # Configuración de CI/CD

# Ejemplos
git commit -m "feat(usuarios): agregar endpoint de exportación CSV"
git commit -m "fix(pago): corregir cálculo de impuestos en facturas"
git commit -m "refactor(auth): extraer lógica JWT a servicio separado"
git commit -m "docs(api): actualizar documentación de endpoints de pedidos"
git commit -m "test(usuario): agregar tests unitarios para cambio de password"
```

```markdown
# GUÍA DE CODE REVIEW

## QUÉ REVISAR (en orden de importancia)

### 1. Diseño y arquitectura
- ¿La solución es la adecuada para el problema?
- ¿Sigue los patrones del proyecto?
- ¿Respeta los principios SOLID?
- ¿Tiene la separación de responsabilidades correcta?

### 2. Funcionalidad
- ¿Resuelve el problema que dice resolver?
- ¿Maneja casos borde correctamente?
- ¿Tiene manejo de errores adecuado?
- ¿Considera seguridad (inyección SQL, XSS, CSRF)?

### 3. Legibilidad y mantenibilidad
- ¿El código es fácil de entender?
- ¿Los nombres de variables/métodos son descriptivos?
- ¿Tiene la cantidad correcta de comentarios?
- ¿El flujo lógico es claro?

### 4. Tests
- ¿Cubren los casos principales?
- ¿Cubren casos borde y errores?
- ¿Son fáciles de entender?

### 5. Rendimiento
- ¿Evita consultas N+1?
- ¿Usa índices de base de datos adecuadamente?
- ¿Hay operaciones costosas innecesarias?

## CÓMO DAR FEEDBACK

✅ Bueno:
  "Me gusta cómo encapsulaste la lógica de validación en un servicio separado."
  "Buena idea usar early return aquí, mejora la legibilidad."
  "Este test cubre un caso borde que no habíamos considerado."

⚠ Constructivo:
  "Esta función tiene 50 líneas, ¿podríamos extraer la validación a un método?"
  "Este método devuelve dos tipos diferentes, ¿podríamos hacerlo más predecible?"
  "Faltan tests para el caso cuando el usuario no existe."

❌ Evitar:
  "Está mal." (sin explicación)
  "Hazlo como yo lo haría." (preferencias personales)
  "Esto es obvio." (asume conocimiento)
```

```php
namespace Tests\Unit\Services;

use App\Services\AuthorizationService;
use PHPUnit\Framework\TestCase;

        $usuarioId = 2;

        $resultado = $this->authService->usuarioTienePermiso(

    {
```

```php
        // Crear primer usuario
        self::makeRequest('POST', '/usuarios', [
            'nombre'   => 'User 1',
            'email'    => $email,
            'password' => 'TestPass123!'
        ], self::$adminToken);

        // Intentar crear duplicado
        $response = self::makeRequest('POST', '/usuarios', [
            'nombre'   => 'User 2',
            'email'    => $email,
            'password' => 'TestPass123!'
        ], self::$adminToken);

        $this->assertEquals(422, $response['status']);
    }

    public function test_eliminar_usuario_inexistente(): void
    {
        $response = self::makeRequest(
            'DELETE',
            '/usuarios/99999',
            [],
            self::$adminToken
        );

        $this->assertEquals(404, $response['status']);
    }

    private static function makeRequest(
        string $method,
        string $path,
        array $data = [],
        string $token = ''
    ): array {
        $url = self::$baseUrl . $path;
        $headers = ['Content-Type: application/json'];

        if ($token) {
            $headers[] = "Authorization: Bearer {$token}";
        }

        $ch = curl_init();
        curl_setopt_array($ch, [
            CURLOPT_URL => $url,
            CURLOPT_CUSTOMREQUEST => $method,
            CURLOPT_RETURNTRANSFER => true,
            CURLOPT_HTTPHEADER => $headers,
            CURLOPT_POSTFIELDS => $data ? json_encode($data) : null,
            CURLOPT_TIMEOUT => 10,
        ]);

        $body = curl_exec($ch);
        $status = curl_getinfo($ch, CURLINFO_HTTP_CODE);
        curl_close($ch);

        return [
            'status' => $status,
            'body'   => json_decode($body, true) ?? []
        ];
    }
}
```

5. DOCUMENTACIÓN DE CÓDIGO CON PHPDOC
===============================================================================

La documentación del código debe explicar el "qué" y el "por qué", no el
"cómo" (el código ya muestra el cómo).

```php
<?php
/**
 * Procesador de pagos con soporte multi-gateway.
 *
 * Esta clase orquesta la lógica de pagos a través de múltiples
 * pasarelas configuradas. Implementa failover automático y
 * reintentos con backoff exponencial.
 *
 * @package App\Services
 * @author  EGA Enterprise Team
 * @version 2.0.0
 * @since   1.0.0
 */
class PaymentProcessor
{
    /**
     * Lista de gateways de pago configurados.
     *
     * @var PaymentGatewayInterface[]
     */
    private array $gateways = [];

    /**
     * Procesa un pago a través de los gateways disponibles.
     *
     * Intenta cada gateway en orden hasta que uno procese el pago
     * exitosamente. Si todos fallan, lanza una excepción.
     *
     * @param float        $amount    Monto del pago (debe ser > 0)
     * @param string       $currency  Código ISO de moneda (MXN, USD, EUR)
     * @param array{
     *   reference?: string,
     *   description?: string,
     *   customer_id?: int
     * } $metadata Datos adicionales de la transacción
     *
     * @return PaymentResult Resultado del procesamiento
     *
     * @throws PaymentException     Si todos los gateways fallan
     * @throws \InvalidArgumentException Si amount <= 0
     *
     * @since 2.0.0 Ahora soporta metadata
     */
    public function process(
        float $amount,
        string $currency = 'MXN',
        array $metadata = []
    ): PaymentResult {
        if ($amount <= 0) {
            throw new \InvalidArgumentException(
                'El monto debe ser mayor a cero'
            );
        }

        foreach ($this->gateways as $gateway) {
            try {
                return $gateway->charge($amount, $currency, $metadata);
            } catch (GatewayException $e) {
                $this->logger->warning(
                    "Gateway {$gateway->getName()} falló: {$e->getMessage()}"
                );
                continue;
            }
        }

        throw new PaymentException(
            'No se pudo procesar el pago con ningún gateway disponible'
        );
    }

    /**
     * Registra un nuevo gateway de pago.
     *
     * @param PaymentGatewayInterface $gateway Instancia del gateway
     * @param int $priority Prioridad (menor = más prioritario)
     *
     * @return self Para encadenamiento de métodos
     */
    public function addGateway(
        PaymentGatewayInterface $gateway,
        int $priority = 10
    ): self {
        $this->gateways[$priority][] = $gateway;
        ksort($this->gateways);

        return $this;
    }
}
?>
```

```php
<?php
// src/Quality/TechnicalDebtRegistry.php
namespace App\Quality;

class TechnicalDebtRegistry
{
    private static array $debts = [];

    public static function register(
        string $file,
        int $line,
        string $description,
        string $priority = 'medium'
    ): void {
        self::$debts[] = [
            'id'          => uniqid('debt_'),
            'file'        => $file,
            'line'        => $line,
            'description' => $description,
            'priority'    => $priority,
            'created_at'  => date('c'),
            'status'      => 'open'
        ];

        // Registrar en archivo
        file_put_contents(
            __DIR__ . '/../../storage/technical-debt.json',
            json_encode(self::$debts, JSON_PRETTY_PRINT)
        );
    }

    public static function generateReport(): string
    {
        $debts = self::getAll();
        $byPriority = [];

        foreach ($debts as $debt) {
            $byPriority[$debt['priority']][] = $debt;
        }

        $report = "# Reporte de Deuda Técnica\n\n";
        $report .= "Generado: " . date('Y-m-d H:i:s') . "\n";
        $report .= "Total: " . count($debts) . " items\n\n";

        foreach ($byPriority as $priority => $items) {
            $report .= "## {$priority} (" . count($items) . ")\n\n";
            foreach ($items as $item) {
                $report .= "- [{$item['id']}] {$item['description']}\n";
                $report .= "  Archivo: {$item['file']}:{$item['line']}\n\n";
            }
        }

        return $report;
    }

    public static function getAll(): array
    {
        $file = __DIR__ . '/../../storage/technical-debt.json';
        if (!file_exists($file)) {
            return [];
        }
        return json_decode(file_get_contents($file), true) ?? [];
    }

    public static function markAsResolved(string $id): void
    {
        $debts = self::getAll();
        foreach ($debts as &$debt) {
            if ($debt['id'] === $id) {
                $debt['status'] = 'resolved';
                $debt['resolved_at'] = date('c');
                break;
            }
        }
        file_put_contents(
            __DIR__ . '/../../storage/technical-debt.json',
            json_encode($debts, JSON_PRETTY_PRINT)
        );
    }

    public static function calculateInterest(): array
    {
        $debts = self::getAll();
        $openDebts = array_filter($debts, fn($d) => $d['status'] === 'open');

        $hours = count($openDebts) * 2; // Estimación: 2h por item
        $cost = $hours * 50; // $50 USD/hora

        return [
            'total_items' => count($openDebts),
            'estimated_hours' => $hours,
            'estimated_cost' => $cost,
            'currency' => 'USD'
        ];
    }
}
```

```php
<?php
// 1. S — Single Responsibility Principle
// Una clase debe tener una única razón para cambiar.

// ❌ MAL: La clase hace de todo
class UsuarioService
{
    public function crearUsuario(array $datos): void { /* ... */ }
    public function enviarEmailBienvenida(int $id): void { /* ... */ }
    public function generarReporteExcel(string $formato): void { /* ... */ }
    public function procesarPago(int $userId, float $monto): void { /* ... */ }
}

// ✅ BIEN: Cada clase tiene una responsabilidad
class CrearUsuarioUseCase { /* maneja solo creación de usuarios */ }
class EmailService { /* maneja solo envío de emails */ }
class ReporteService { /* maneja solo generación de reportes */ }
class PaymentService { /* maneja solo procesamiento de pagos */ }

// 2. O — Open/Closed Principle
// Abierto a extensión, cerrado a modificación.

// ✅ BIEN: Usando interfaces y polimorfismo
interface PaymentGatewayInterface
{
    public function charge(float $amount, string $currency): PaymentResult;
}

class StripeGateway implements PaymentGatewayInterface { /* ... */ }
class PayPalGateway implements PaymentGatewayInterface { /* ... */ }
class MercadoPagoGateway implements PaymentGatewayInterface { /* ... */ }

// Nuevo gateway sin modificar el código existente
class OpenPayGateway implements PaymentGatewayInterface { /* ... */ }

// 3. L — Liskov Substitution Principle
// Las subclases deben poder sustituir a sus clases base.

// ✅ BIEN: Square extiende Rectangle sin romper comportamiento
class Shape
{
    public function getArea(): float { return 0; }
}

class Rectangle extends Shape
{
    public function __construct(
        protected float $width,
        protected float $height
    ) {}

    public function getArea(): float
    {
        return $this->width * $this->height;
    }
}

class Square extends Shape
{
    public function __construct(
        protected float $side
    ) {}

    public function getArea(): float
    {
        return $this->side * $this->side;
    }
}

// 4. I — Interface Segregation Principle
// Interfaces específicas, no generales.

// ❌ MAL: Interfaz muy grande
interface UserInterface
{
    public function getProfile(): array;
    public function updateProfile(array $data): void;
    public function getPermissions(): array;
    public function updatePermissions(array $perms): void;
    public function sendNotification(string $message): void;
}

// ✅ BIEN: Interfaces pequeñas y específicas
interface ProfileReadInterface { public function getProfile(): array; }
interface ProfileWriteInterface { public function updateProfile(array $data): void; }
interface PermissionInterface { public function getPermissions(): array; }
interface NotificationInterface { public function send(string $message): void; }

// 5. D — Dependency Inversion Principle
// Depende de abstracciones, no de implementaciones concretas.

// ❌ MAL: Dependencia directa de implementación concreta
class UserReport
{
    private MySQLDatabase $db; // Acoplamiento rígido

    public function __construct()
    {
        $this->db = new MySQLDatabase(); // Hardcoded
    }
}

// ✅ BIEN: Dependencia de abstracción
class UserReport
{
    public function __construct(
        private DatabaseInterface $db  // Inyección de dependencia
    ) {}
}
```

- PSR-12 define estándares de codificación PHP: identación 4 espacios,
  líneas máx 120 caracteres, naming PascalCase/camelCase/UPPER_CASE.
- Git Flow (ramas feature/release/hotfix) y Conventional Commits
  estructuran el trabajo colaborativo en equipo.

  Revisa: arquitectura, funcionalidad, legibilidad, tests, rendimiento.
- Los tests automatizados (unitarios, integración, funcionales) son la
  red de seguridad que permite refactorizar con confianza.
- La documentación PHPDoc explica el qué y el por qué, no el cómo.
- La deuda técnica debe rastrearse y gestionarse como la deuda financiera.
- Los principios SOLID guían el diseño de código mantenible y extensible.

===============================================================================
PREGUNTAS FRECUENTES
===============================================================================

❓ **¿PSR-12 es obligatorio?** No es obligatorio legalmente, pero es el
estándar aceptado por la comunidad PHP. Si trabajas en equipo, úsalo.
Si trabajas solo, úsalo igual; tu yo del futuro te lo agradecerá.

❓ **¿Cada cuánto hacer code review?** Cada Pull Request debe ser revisado
antes de fusionarse. Idealmente, en menos de 24 horas para no bloquear
al equipo. PRs pequeños (< 200 líneas) se revisan más rápido.

❓ **¿Qué cubrimiento de tests es aceptable?** El 80% es un buen objetivo.
Pero más importante que el porcentaje es que los tests cubran las rutas
críticas de negocio y los casos borde más importantes.

❓ **¿Cómo convenzo al equipo de seguir estándares?** Implementa herramientas
automáticas (PHP CS Fixer, ESLint, Prettier) que corrijan el formato al
guardar. La automatización elimina la fricción y las discusiones.

❓ **¿La deuda técnica es mala?** No siempre. A veces es necesario tomar
ataques para cumplir deadlines. Lo malo es no documentarla y no planear
cuándo pagarla. La deuda técnica consciente es estratégica.
