1. ESTRUCTURA DE UN FORMULARIO HTML
===============================================================================

Un formulario HTML se define con la etiqueta <form>. Dentro de ella se colocan
los campos que el usuario rellenará. Cada campo se asocia a un nombre (name)
que PHP utilizará como clave para acceder al valor enviado.

Atributos esenciales:

  - action: Ruta del script PHP que procesará los datos.
  - method: Método HTTP de envío: GET o POST.
  - enctype: Tipo de codificación (solo necesario para archivos).

Sintaxis básica:

```html
<form action="procesar.php" method="POST">
    <label for="nombre">Nombre:</label>
    <input type="text" id="nombre" name="nombre">
    <button type="submit">Enviar</button>
</form>
```

El atributo name es obligatorio si queremos que el dato llegue a PHP. Sin él,
el campo será ignorado en el lado del servidor.

Veamos los tipos de input más comunes:

```html
<!-- Campo de texto simple -->
<input type="text" name="usuario" placeholder="Tu nombre">

<!-- Contraseña (oculta los caracteres) -->
<input type="password" name="clave">

<!-- Correo electrónico (validación HTML5) -->
<input type="email" name="correo">

<!-- Número -->
<input type="number" name="edad" min="0" max="120">

<!-- Área de texto multilínea -->
<textarea name="comentario" rows="4" cols="50"></textarea>

<!-- Lista desplegable -->
<select name="pais">
    <option value="mx">México</option>
    <option value="es">España</option>
    <option value="ar">Argentina</option>
</select>

<!-- Casillas de verificación -->
<input type="checkbox" name="intereses[]" value="php"> PHP
<input type="checkbox" name="intereses[]" value="js"> JavaScript

<!-- Botones de radio -->
<input type="radio" name="genero" value="masculino"> Masculino
<input type="radio" name="genero" value="femenino"> Femenino

<!-- Campo oculto (útil para enviar datos sin mostrar al usuario) -->
<input type="hidden" name="token" value="abc123">

<!-- Botón de envío -->
<button type="submit">Enviar formulario</button>
```

Observa el uso de corchetes en name="intereses[]". Esto indica a PHP que los
múltiples valores seleccionados deben agruparse en un arreglo.

La etiqueta <label> mejora la accesibilidad: al hacer clic en el texto, el
foco se coloca automáticamente en el campo asociado. Usa for para enlazarlo
con el id del input.

```html
<label for="email">Correo electrónico:</label>
<input type="email" id="email" name="email">
```

Un formulario completo y accesible podría verse así:

```html
<!DOCTYPE html>
<html lang="es">
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <title>Registro de Usuario</title>
</head>
<body>
    <h1>Formulario de Registro</h1>
    <form action="registrar.php" method="POST">
        <p>
            <label for="nombre">Nombre completo:</label>
            <input type="text" id="nombre" name="nombre" required>
        </p>
        <p>
            <label for="email">Correo electrónico:</label>
            <input type="email" id="email" name="email" required>
        </p>
        <p>
            <label for="password">Contraseña:</label>
            <input type="password" id="password" name="password" required>
        </p>
        <p>
            <label for="edad">Edad:</label>
            <input type="number" id="edad" name="edad" min="1" max="120">
        </p>
        <p>
            <label>Curso de interés:</label><br>
            <input type="radio" name="curso" value="php" checked> PHP
            <input type="radio" name="curso" value="js"> JavaScript
            <input type="radio" name="curso" value="python"> Python
        </p>
        <p>
            <label>Temas adicionales:</label><br>
            <input type="checkbox" name="temas[]" value="mysql"> Bases de datos
            <input type="checkbox" name="temas[]" value="seguridad"> Seguridad
            <input type="checkbox" name="temas[]" value="frameworks"> Frameworks
        </p>
        <p>
            <label for="pais">País:</label>
            <select name="pais" id="pais">
                <option value="">Seleccione un país</option>
                <option value="mx">México</option>
                <option value="co">Colombia</option>
                <option value="es">España</option>
                <option value="ar">Argentina</option>
                <option value="pe">Perú</option>
            </select>
        </p>
        <p>
            <label for="comentarios">Comentarios:</label><br>
            <textarea name="comentarios" id="comentarios" rows="5" cols="40"></textarea>
        </p>
        <p>
            <button type="submit">Registrarme</button>
            <button type="reset">Limpiar</button>
        </p>
    </form>
</body>
</html>
```

El botón type="reset" restablece todos los campos a su valor inicial. Aunque es
útil para el desarrollo, pocos sitios lo incluyen en producción porque puede
causar frustración al usuario.

El atributo required en los inputs activa la validación HTML5 del navegador. Si
el usuario intenta enviar el formulario con un campo required vacío, el
navegador muestra un mensaje de error y detiene el envío.

===============================================================================
2. EL ATRIBUTO ACTION
===============================================================================

El atributo action define la URL absoluta o relativa del script PHP que
procesará los datos.

Ejemplos:

```html
<!-- Ruta relativa simple -->
<form action="procesar.php" method="POST">

<!-- Ruta relativa con subdirectorio -->
<form action="/app/usuarios/registrar.php" method="POST">

<!-- URL absoluta -->
<form action="https://ejemplo.com/api/usuarios.php" method="POST">

<!-- Vacío: se envía a la misma página -->
<form action="" method="POST">

<!-- Omitido: también se envía a la misma página -->
<form method="POST">
```

Cuando action se deja vacío o se omite, el formulario se envía a la URL actual.
Esto es común en aplicaciones donde el mismo archivo PHP muestra el formulario
y lo procesa.

```php
<?php
// Este archivo se llama a sí mismo
if ($_SERVER['REQUEST_METHOD'] === 'POST') {
    echo "Formulario recibido";
    // procesar...
}
?>
<form method="POST">
    <input type="text" name="nombre">
    <button type="submit">Enviar</button>
</form>
```

La superglobal $_SERVER['REQUEST_METHOD'] nos indica si la petición fue GET o
POST. La usaremos frecuentemente para decidir si mostrar el formulario o
procesarlo.

===============================================================================
3. EL ATRIBUTO METHOD: GET vs POST
===============================================================================

El atributo method tiene dos valores posibles: GET y POST. La elección depende
del tipo de operación que estemos realizando.

GET:

  - Los datos se envían en la URL: ?clave=valor&clave2=valor2
  - Es visible para el usuario en la barra de direcciones.
  - Limitado a unos 2048 caracteres (depende del navegador y servidor).
  - Adecuado para búsquedas, filtros y consultas de solo lectura.
  - No debe usarse para contraseñas o datos sensibles.
  - Puede ser marcado como favorito (bookmark).

POST:

  - Los datos se envían en el cuerpo de la petición HTTP.
  - No son visibles en la URL.
  - Sin límite práctico de tamaño.
  - Adecuado para registros, inicios de sesión y subida de archivos.
  - No puede ser marcado como favorito.
  - Ofrece mayor privacidad (aunque no es cifrado por sí mismo).

```html
<!-- Búsqueda: GET es apropiado -->
<form action="buscar.php" method="GET">
    <input type="text" name="q" placeholder="Buscar...">
    <button type="submit">Buscar</button>
</form>

<!-- Registro: POST es obligatorio -->
<form action="registrar.php" method="POST">
    <input type="text" name="usuario">
    <input type="password" name="password">
    <button type="submit">Crear cuenta</button>
</form>
```

También existe el método DELETE para eliminar recursos, PUT para actualizar y
PATCH para modificar parcialmente. Estos se utilizan principalmente en APIs
RESTful y no están disponibles directamente en formularios HTML estándar.

Para usar DELETE o PUT desde un formulario, necesitaríamos JavaScript y
fetch() o XMLHttpRequest.

===============================================================================
4. RECIBIR DATOS DEL FORMULARIO EN PHP
===============================================================================

Cuando el usuario envía el formulario, PHP guarda automáticamente los datos en
dos superglobales:

  - $_GET: contiene los datos enviados con method="GET".
  - $_POST: contiene los datos enviados con method="POST".
  - $_REQUEST: combina ambos (GET + POST + COOKIE). Su uso es desaconsejado
    por razones de seguridad y claridad.

Ambas son arreglos asociativos donde las claves son los valores de name y los
valores son lo que el usuario escribió o seleccionó.

Ejemplo con POST:

```php
<?php
// procesar.php
if ($_SERVER['REQUEST_METHOD'] === 'POST') {
    $nombre   = $_POST['nombre'];
    $email    = $_POST['email'];
    $password = $_POST['password'];
    $edad     = $_POST['edad'];
    $pais     = $_POST['pais'];
    $comentarios = $_POST['comentarios'];

    echo "Nombre: " . htmlspecialchars($nombre) . "<br>";
    echo "Email: " . htmlspecialchars($email) . "<br>";
    echo "Edad: " . (int)$edad . "<br>";
    echo "País: " . htmlspecialchars($pais) . "<br>";
}
?>
```

Ejemplo con GET (búsqueda):

```php
<?php
// buscar.php
$termino = $_GET['q'] ?? '';

if ($termino !== '') {
    echo "Resultados de búsqueda para: " . htmlspecialchars($termino);
    // consultar base de datos...
} else {
    echo "Ingrese un término de búsqueda.";
}
?>
```

Para campos tipo checkbox con name="intereses[]", PHP recibe un arreglo:

```php
<?php
if (isset($_POST['intereses'])) {
    $intereses = $_POST['intereses']; // es un array
    foreach ($intereses as $interes) {
        echo htmlspecialchars($interes) . "<br>";
    }
}
?>
```

Si el checkbox no está marcado, no se envía nada. Por eso es recomendable
verificar con isset() antes de acceder a la variable.

Para inputs tipo radio, solo se envía el valor del botón seleccionado:

```php
<?php
$genero = $_POST['genero'] ?? 'no especificado';
echo "Género: " . htmlspecialchars($genero);
?>
```

El operador ?? (null coalescing operator) asigna un valor por defecto si la
clave no existe en el arreglo. Es una forma elegante de evitar errores.

===============================================================================
5. PROCESAMIENTO EN LA MISMA PÁGINA
===============================================================================

Un patrón muy común es tener el formulario y el procesamiento en el mismo
archivo PHP. Esto simplifica la estructura y evita redirecciones innecesarias.

```php
<?php
// formulario.php
$mensaje = '';
$nombre  = '';

if ($_SERVER['REQUEST_METHOD'] === 'POST') {
    $nombre = $_POST['nombre'] ?? '';

    if ($nombre !== '') {
        $mensaje = "Hola, " . htmlspecialchars($nombre) . "! Gracias por enviar el formulario.";
    } else {
        $mensaje = "Por favor, ingrese su nombre.";
    }
}
?>
<!DOCTYPE html>
<html lang="es">
<head>
    <meta charset="UTF-8">
    <title>Formulario y Procesamiento</title>
</head>
<body>
    <h1>Formulario en la misma página</h1>

    <?php if ($mensaje !== ''): ?>
        <p style="color: green; font-weight: bold;"><?= htmlspecialchars($mensaje) ?></p>
    <?php endif; ?>

    <form method="POST">
        <label for="nombre">Nombre:</label>
        <input type="text" id="nombre" name="nombre"
               value="<?= htmlspecialchars($nombre) ?>">
        <button type="submit">Saludar</button>
    </form>
</body>
</html>
```

Ventajas de este patrón:

  - Menos archivos que administrar.
  - El contexto PHP está disponible directamente en la vista.
  - Podemos rellenar los campos con los valores anteriores si hay errores.

Desventajas:

  - El archivo crece en tamaño.
  - Mezcla lógica de presentación con lógica de negocio.
  - Dificulta la reutilización.

Para proyectos pequeños está bien. En proyectos grandes se usa un patrón MVC
(Modelo-Vista-Controlador) que separa estas responsabilidades.

Observa que usamos htmlspecialchars() al imprimir $nombre dentro del value del
input. Esto evita que caracteres especiales como comillas rompan el HTML y
previene ataques XSS (Cross-Site Scripting).

```php
value="<?= htmlspecialchars($nombre) ?>"
```

Si el usuario escribe Juan "El Grande", sin htmlspecialchars el HTML quedaría
roto:

```html
<input value="Juan "El Grande""> <!-- HTML inválido -->
```

Con htmlspecialchars:

```html
<input value="Juan &quot;El Grande&quot;"> <!-- HTML válido -->
```

Nunca confíes en los datos del usuario. Sanitiza siempre antes de imprimir.

6. ENVÍO DE FORMULARIOS CON ARCHIVOS
===============================================================================

Para subir archivos necesitamos tres cosas:

  1. method="POST" (GET no soporta archivos).
  2. enctype="multipart/form-data" (codificación especial para binarios).
  3. input type="file" con name para el archivo.

```html
<form action="subir.php" method="POST" enctype="multipart/form-data">
    <label for="archivo">Seleccione un archivo:</label>
    <input type="file" id="archivo" name="archivo">
    <button type="submit">Subir</button>
</form>
```

En PHP, los archivos subidos se reciben en la superglobal $_FILES:

```php
<?php
if ($_SERVER['REQUEST_METHOD'] === 'POST' && isset($_FILES['archivo'])) {
    $archivo = $_FILES['archivo'];

    echo "Nombre original: " . $archivo['name'] . "<br>";
    echo "Tipo MIME: " . $archivo['type'] . "<br>";
    echo "Tamaño: " . $archivo['size'] . " bytes<br>";
    echo "Nombre temporal: " . $archivo['tmp_name'] . "<br>";
    echo "Código de error: " . $archivo['error'] . "<br>";

    if ($archivo['error'] === UPLOAD_ERR_OK) {
        $destino = 'uploads/' . basename($archivo['name']);
        if (move_uploaded_file($archivo['tmp_name'], $destino)) {
            echo "Archivo subido exitosamente a: $destino";
        } else {
            echo "Error al mover el archivo.";
        }
    } else {
        echo "Error en la subida.";
    }
}
?>
```

$_FILES['archivo'] es un arreglo asociativo con las siguientes claves:

  - name: Nombre original del archivo en el cliente.
  - type: Tipo MIME declarado por el navegador.
  - size: Tamaño en bytes.
  - tmp_name: Ruta temporal donde PHP almacenó el archivo.
  - error: Código de error (0 si fue exitoso, 1 si excede upload_max_filesize).

La función move_uploaded_file() mueve el archivo de la ubicación temporal a una
ubicación permanente. Siempre úsala en lugar de rename() por seguridad.

Nunca confíes en el nombre original del archivo. Podría contener rutas maliciosas
como ../../etc/passwd. Usa basename() para extraer solo el nombre.

Para múltiples archivos, usa name="archivos[]" y múltiple atributo en HTML:

```html
<input type="file" name="archivos[]" multiple>
```

PHP recibe un arreglo anidado en $_FILES['archivos'].

===============================================================================
7. VALIDACIÓN EN EL CLIENTE VS SERVIDOR
===============================================================================

La validación puede ocurrir en dos lugares:

Validación del lado del cliente (HTML5 + JavaScript):

  - Ventajas: respuesta instantánea, sin recargar la página.
  - Desventajas: puede ser omitida fácilmente (desactivar JS, modificar HTML).

Validación del lado del servidor (PHP):

  - Ventajas: es la única realmente segura. Los datos pasan por tu control.
  - Desventajas: requiere recargar la página (o petición AJAX).

Regla de oro: nunca confíes en la validación del cliente. Siempre valida en el
servidor. La validación del cliente es solo para mejorar la experiencia de
usuario.

```php
<?php
// Validación del lado del servidor
$errores = [];

if ($_SERVER['REQUEST_METHOD'] === 'POST') {
    $nombre = trim($_POST['nombre'] ?? '');
    $email  = trim($_POST['email'] ?? '');
    $edad   = $_POST['edad'] ?? '';

    if ($nombre === '') {
        $errores[] = 'El nombre es obligatorio.';
    }

    if ($email === '' || !filter_var($email, FILTER_VALIDATE_EMAIL)) {
        $errores[] = 'Ingrese un correo electrónico válido.';
    }

    if ($edad === '' || !is_numeric($edad) || $edad < 1 || $edad > 120) {
        $errores[] = 'La edad debe ser un número entre 1 y 120.';
    }

    if (empty($errores)) {
        echo "Registro exitoso. Bienvenido, " . htmlspecialchars($nombre);
        // guardar en base de datos...
    }
}
?>
```

La función trim() elimina espacios al inicio y final. filter_var() aplica
filtros de validación. Combinadas ofrecen una primera capa defensiva.

8. BUENAS PRÁCTICAS CON FORMULARIOS
===============================================================================

1. Siempre usa htmlspecialchars() al imprimir datos del usuario.
2. Verifica el método HTTP con $_SERVER['REQUEST_METHOD'].
3. Usa isset() o ?? antes de acceder a $_POST o $_GET.
4. Valida todos los datos en el servidor, incluso si ya validaste en cliente.
5. No almacenes contraseñas en texto plano. Usa password_hash().
6. Para archivos, verifica el tipo MIME y el tamaño.
7. Usa tokens CSRF en formularios críticos (registro, login, pagos).
8. Mantén el mismo archivo para formulario y procesamiento en proyectos pequeños.
9. Separa lógica de presentación en proyectos grandes (MVC).
10. Siempre configura el charset a UTF-8 tanto en HTML como en PHP.

Token CSRF básico:

```php
<?php
session_start();
if (empty($_SESSION['csrf_token'])) {
    $_SESSION['csrf_token'] = bin2hex(random_bytes(32));
}
?>
<form method="POST">
    <input type="hidden" name="csrf_token"
           value="<?= $_SESSION['csrf_token'] ?>">
    <!-- campos del formulario -->
</form>
```

En el procesamiento:

```php
<?php
session_start();
if ($_POST['csrf_token'] !== $_SESSION['csrf_token']) {
    die('Token CSRF inválido.');
}
// procesar formulario...
?>
```

El CSRF (Cross-Site Request Forgery) impide que sitios maliciosos envíen
formularios en nombre de tus usuarios. Cada formulario recibe un token único
que el servidor verifica.

===============================================================================
RESUMEN DEL CAPÍTULO
===============================================================================

- La etiqueta <form> con action y method define el punto de envío y la forma de
  transmisión de los datos del formulario.
- GET envía datos visibles en la URL, ideal para búsquedas y consultas.
- POST envía datos en el cuerpo de la petición, obligatorio para información
  sensible o archivos.
- PHP recibe los datos mediante $_GET y $_POST, arreglos asociativos indexados
  por el atributo name del campo.
- La función htmlspecialchars() es esencial para prevenir XSS al imprimir datos
  del usuario.
- La validación del lado del servidor es la única barrera de seguridad real.
- Los archivos se reciben mediante $_FILES y se mueven con move_uploaded_file().
- El patrón de mismo archivo para formulario y procesamiento es práctico para
  proyectos pequeños.
- Siempre usa ?? (null coalescing) o isset() para evitar accesos indefinidos.
- Los tokens CSRF protegen contra ataques de falsificación de peticiones.

===============================================================================
PREGUNTAS FRECUENTES
===============================================================================

❓ **¿Puedo usar GET para enviar contraseñas?** No, nunca. Las contraseñas
enviadas por GET quedan visibles en la URL, en el historial del navegador y en
los logs del servidor. Siempre usa POST.

❓ **¿Qué pasa si un checkbox no está marcado?** No se envía ningún valor para
ese campo. Por eso debes usar isset() o ?? para verificar su existencia.

❓ **¿Cuál es la diferencia entre $_POST y $_REQUEST?** $_REQUEST fusiona $_GET,
$_POST y $_COOKIE. Su uso es desaconsejado porque puede generar ambigüedad y
conflictos de seguridad.

❓ **¿Qué es enctype="multipart/form-data"?** Es el tipo de codificación
necesario cuando el formulario incluye un input type="file". Sin él, el archivo
no se transmite correctamente.

❓ **¿Por qué usar htmlspecialchars()?** Para evitar que el navegador interprete
caracteres especiales como etiquetas HTML. Es la defensa principal contra XSS.

❓ **¿Qué hace trim()?** Elimina espacios en blanco, tabulaciones y saltos de
línea al inicio y final de un string. Útil para limpiar entradas del usuario.

❓ **¿Cuántos datos puedo enviar por GET?** Depende del navegador y servidor, pero
generalmente el límite es de 2048 caracteres para la URL completa.

❓ **¿Puedo tener múltiples formularios en una misma página?** Sí, cada uno puede
tener su propio action y method. Asegúrate de que los name no se confundan.

🏗️ **Sistema de Encuesta Dinámica con Validación y Persistencia en Archivo**

Diseña y construye un sistema de encuesta que conste de:

  1. Un formulario HTML completo con los siguientes campos:
     - Nombre completo (text, requerido).
     - Correo electrónico (email, requerido, validación HTML5 y servidor).
     - Edad (number, rango 1-120).
     - Lenguaje de programación favorito (radio: PHP, JavaScript, Python, Java,
       C#, Otro).
     - Frameworks que domina (checkbox múltiple: Laravel, React, Django, Spring,
       Vue, Angular, Otro).
     - Nivel de experiencia (select: Principiante, Intermedio, Avanzado, Experto).
     - Comentarios adicionales (textarea, opcional).
     - Un campo oculto con un token CSRF generado aleatoriamente.

  2. Procesamiento en la misma página que:
     - Valida todos los campos del lado del servidor con mensajes de error
       específicos.
     - Sanitiza todas las entradas con htmlspecialchars().
     - Si hay errores, muestra el formulario con los valores previos y los
       mensajes de error junto a cada campo.
     - Si no hay errores, guarda los datos en un archivo JSON
       (encuestas.json) y muestra un mensaje de éxito.
     - Verifica el token CSRF antes de cualquier procesamiento.

  3. Una sección que muestre todas las encuestas anteriores leídas del archivo
     JSON, formateadas en una tabla HTML.

  4. Un botón para eliminar todas las encuestas (limpia el archivo JSON).

  Criterios de evaluación:
  - Cobertura completa de validación (todos los campos, no solo los obligatorios).
  - Manejo correcto de errores con retroalimentación visual.
  - Implementación de seguridad CSRF.
  - Persistencia y lectura correcta del archivo JSON.
  - Calidad y claridad del código (comentarios, indentación, nombres).
  - Experiencia de usuario (mensajes claros, diseño limpio).
