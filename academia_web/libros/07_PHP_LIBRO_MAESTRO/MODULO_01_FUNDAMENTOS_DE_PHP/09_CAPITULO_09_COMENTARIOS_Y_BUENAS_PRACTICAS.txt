1. TIPOS DE COMENTARIOS EN PHP
===============================================================================

PHP soporta tres estilos de comentarios:

**1.1. Comentarios de una línea (//)**

```php
<?php
// Esto es un comentario de una línea
echo "Hola";  // También puede ir al final de una línea

// Útil para deshabilitar temporalmente código
// $depuracion = true;
?>
```

**1.2. Comentarios de bloque (/* */)**

```php
<?php
/*
 * Esto es un comentario de bloque
 * que puede abarcar múltiples líneas.
 * Se usa típicamente para explicar
 * secciones complejas de código.
 */

/*
Bloque sin asteriscos también funciona
pero es menos legible
*/
?>
```

**1.3. Comentarios estilo Perl (#)**

```php
<?php
# Esto es un comentario al estilo Perl/Shell
# Menos común en PHP, pero perfectamente válido
echo "Comentario con #";
?>
```

**1.4. Comentarios de documentación (PHPDoc)**

```php
<?php
/**
 * Calcula el precio final con IVA incluido.
 *
 * Esta función toma un precio base y le aplica el Impuesto al Valor
 * Agregado (IVA) según la tasa especificada.
 *
 * @param float $precio    Precio base del producto (sin IVA)
 * @param float $tasa      Tasa de IVA (ej: 0.16 para 16%)
 *
 * @return float           Precio final con IVA incluido
 *
 * @throws InvalidArgumentException Si el precio es negativo
 */
function calcularPrecioFinal(float $precio, float $tasa = 0.16): float
{
    if ($precio < 0) {
        throw new InvalidArgumentException(
            "El precio no puede ser negativo"
        );
    }
    return $precio * (1 + $tasa);
}
?>
```

2. CUÁNDO Y CÓMO COMENTAR
===============================================================================

El mayor desafío no es aprender a comentar, sino saber cuándo hacerlo.

**2.1. Comenta el "por qué", no el "qué"**

```php
<?php
// ❌ Mal comentario (explica el qué, que es obvio)
// Suma 1 al contador
$contador = $contador + 1;

// ✅ Buen comentario (explica el por qué)
// Necesitamos incrementar el contador antes de validar
// porque la validación usa el valor actualizado
$contador = $contador + 1;
?>
```

**2.2. Comenta decisiones complejas**

```php
<?php
// ✅ Bueno: explica una decisión no obvia
// Usamos búsqueda lineal en lugar de binaria porque
// el array normalmente tiene menos de 10 elementos
// y la sobrecarga de ordenar no vale la pena
foreach ($items as $item) {
    if ($item->id === $buscado) {
        return $item;
    }
}
?>
```

**2.3. Comenta código que parece incorrecto pero no lo es**

```php
<?php
// ✅ Bueno: previene que alguien "corrija" el código
// La multiplicación por 100 aquí es intencional:
// el porcentaje viene como decimal (0.15) pero
// necesitamos el valor como entero (15)
$porcentaje = $valorDecimal * 100;
?>
```

**2.4. TODO, FIXME y HACK**

```php
<?php
// TODO: Implementar caché para mejorar rendimiento
// FIXME: Esta función falla cuando el array está vacío
// HACK: Solución temporal hasta que el proveedor arregle el bug
?>
```

3. DOCUMENTACIÓN CON PHPDOC
===============================================================================

PHPDoc es el estándar para documentar código PHP. Los IDE lo usan para
autocompletado y análisis de tipos.

**3.1. Documentación de clases**

```php
<?php
/**
 * Representa un usuario del sistema.
 *
 * Esta clase maneja la autenticación, perfil y permisos
 * de los usuarios registrados en la aplicación.
 *
 * @package App\Models
 * @author  Lucas EGA
 * @version 1.0.0
 */
class Usuario
{
    /** @var string Nombre completo del usuario */
    private string $nombre;

    /** @var string Email del usuario (único en el sistema) */
    private string $email;

    /** @var int Edad del usuario en años */
    private int $edad;
}
?>
```

**3.2. Tags PHPDoc principales**

| Tag | Descripción | Ejemplo |
|-----|-------------|---------|
| @param | Tipo y descripción de parámetro | @param string $nombre |
| @return | Tipo y descripción de retorno | @return bool |
| @throws | Excepción que puede lanzar | @throws \InvalidArgumentException |
| @var | Tipo de una propiedad | @var int |
| @see | Referencia a otro elemento | @see Usuario::login() |
| @deprecated | Marca como obsoleto | @deprecated usar Usuario::login() |
| @since | Versión desde la que existe | @since 2.0.0 |
| @property | Propiedad mágica | @property string $nombre |

**3.3. Documentación de funciones con tipos**

```php
<?php
/**
 * Obtiene los usuarios activos del sistema.
 *
```

4. ESTÁNDARES PSR
===============================================================================

PSR (PHP Standard Recommendations) son recomendaciones de la PHP
Framework Interoperability Group (PHP-FIG). Son el estándar de facto.

**4.1. PSR-1: Basic Coding Standard**

```php
<?php
// <?php debe ser la primera línea (sin espacios ni BOM)

declare(strict_types=1);  // Siempre al inicio

namespace App\Lib;  // Namespace obligatorio para librerías

// Clases en PascalCase
class ControladorBase {}

// Métodos en camelCase
public function obtenerDatos(): array {}

// Constantes en UPPER_CASE
const VERSION = '1.0';

// SIN ETIQUETA DE CIERRE ?>
```

**4.2. PSR-12: Extended Coding Style**

```php
<?php
declare(strict_types=1);

namespace App\Models;

class Producto
{
    public function __construct(
        private string $nombre,
        private float $precio
    ) {
        // Una línea en blanco después de la llave de apertura
        // cuando hay código dentro
    }

    public function calcularDescuento(float $porcentaje): float
    {
        return $this->precio * (1 - $porcentaje / 100);
    }
}
```

**4.3. Reglas PSR-12 esenciales**

```php
<?php
// ✅ Indentación: 4 espacios (no tabs)
if ($condicion) {
    $variable = "valor";  // 4 espacios
}

// ✅ Llaves: misma línea para estructuras de control
if ($a === $b) {
    echo "igual";
} elseif ($a > $b) {
    echo "mayor";
} else {
    echo "menor";
}

// ✅ Llaves: nueva línea para clases y funciones
class MiClase
{
    public function miMetodo(): void
    {
        // contenido
    }
}

// ✅ Espacios alrededor de operadores
$x = 1 + 2;
$y = ($a + $b) * $c;

// ✅ Longitud máxima: 120 caracteres (recomendado 80)
// ✅ Una línea en blanco entre diferentes bloques lógicos
?>
```

5. CONVENCIONES DE NOMENCLATURA
===============================================================================

**5.1. Tipos de nomenclatura en PHP**

```php
<?php
// camelCase — variables, funciones, métodos
$nombreCompleto = "Lucas EGA";
function obtenerUsuario(int $id): ?Usuario {}

// PascalCase — clases, interfaces, traits
class UsuarioActivo {}
interface RepositorioInterface {}
trait Logeable {}

// UPPER_SNAKE_CASE — constantes
define('MAX_INTENTOS', 3);
const LIMITE_PAGINACION = 20;

// snake_case — NO RECOMENDADO en PHP moderno
$nombre_completo = "Lucas";  // Mejor camelCase

// kebab-case — NO USAR en PHP (es para archivos CSS/HTML/URL)
// $nombre-completo = "Lucas";  // ❌ Error sintáctico
?>
```

**5.2. Prefijos y sufijos comunes**

| Elemento | Convención | Ejemplo |
|----------|------------|---------|
| Interfaces | Sufijo `Interface` | `CacheInterface` |
| Traits | Sufijo `Trait` | `Loggeable` (o sin sufijo) |
| Abstractas | Prefijo `Abstract` | `AbstractController` |
| Excepciones | Sufijo `Exception` | `AuthException` |
| Tests | Sufijo `Test` | `UsuarioTest` |

6. MALAS PRÁCTICAS QUE DEBES EVITAR
===============================================================================

**6.1. Uso de `extract()`**

```php
<?php
// ❌ extract(): crea variables sin control
$datos = ['nombre' => 'Lucas', 'admin' => true];
extract($datos);  // Crea $nombre y $admin
// Ahora $admin podría sobrescribir una variable existente

// ✅ Alternativa: acceso directo al array
echo $datos['nombre'];
?>
```

**6.2. Variables globales excesivas**

```php
<?php
// ❌ Variables globales: dependencias ocultas
global $db;
function getUser(int $id): array {
    global $db;  // ¿De dónde viene $db?
    return $db->query("SELECT * FROM users WHERE id = ?", [$id]);
}

// ✅ Inyección de dependencias
function getUser(int $id, \PDO $db): array {
    return $db->query("SELECT * FROM users WHERE id = ?", [$id])->fetch();
}
?>
```

**6.3. No confiar en datos del usuario**

```php
<?php
// ❌ Peligroso: XSS y SQL Injection
echo $_GET['nombre'];                          // XSS
$sql = "SELECT * FROM users WHERE id = " . $_GET['id'];  // SQL Injection

// ✅ Seguro: sanitizar y escapar
echo htmlspecialchars($_GET['nombre'], ENT_QUOTES, 'UTF-8');
$stmt = $db->prepare("SELECT * FROM users WHERE id = ?");
$stmt->execute([$_GET['id']]);
?>
```

**6.4. Código espagueti (HTML + PHP sin estructura)**

```php
<?php
// ❌ Código espagueti: todo mezclado
// index.php con lógica, HTML, SQL y CSS todo junto

// ✅ Separación de responsabilidades
// controladores/UsuarioController.php → lógica
// vistas/usuario/perfil.php → HTML
// modelos/Usuario.php → datos
?>
```

**6.5. Números mágicos**

```php
<?php
// ❌ Números mágicos: sin significado
if ($intentos > 3) {
    bloquearUsuario($id);
}
sleep(900);

// ✅ Constantes descriptivas
define('MAX_INTENTOS_FALLIDOS', 3);
define('TIEMPO_BLOQUEO_SEGUNDOS', 900);

if ($intentos > MAX_INTENTOS_FALLIDOS) {
    bloquearUsuario($id);
    sleep(TIEMPO_BLOQUEO_SEGUNDOS);
}
?>
```

7. HERRAMIENTAS DE CALIDAD DE CÓDIGO
===============================================================================

**7.1. PHP CodeSniffer (PHPCS)**

Detecta violaciones de estándares de codificación:

```bash
# Instalación global
composer global require "squizlabs/php_codesniffer=*"

# Verificar un archivo
phpcs --standard=PSR12 archivo.php

# Corregir automáticamente
phpcbf --standard=PSR12 archivo.php
```

**7.2. PHPStan**

Analiza el código sin ejecutarlo (análisis estático):

```bash
composer require --dev phpstan/phpstan

# Ejecutar
vendor/bin/phpstan analyse src/ --level=max
```

**7.3. Psalm**

Otra herramienta de análisis estático:

```bash
composer require --dev vimeo/psalm

# Ejecutar
vendor/bin/psalm --show-info=true
```

**7.4. PHP-CS-Fixer**

Formatea automáticamente el código:

```bash
composer require --dev friendsofphp/php-cs-fixer

# Ejecutar
vendor/bin/php-cs-fixer fix src/ --rules=@PSR12
```

```php
<?php
// Ejemplo: código que PHPStan detectaría como problemático

/**
 * @param array<string, int> $items
 */
function procesarItems(array $items): int
{
    $total = 0;
    foreach ($items as $item) {
        $total += $item;  // PHPStan sabe que $item es int
        // Pero si $items es mixto, lo detecta
    }
    return $total;
}

// PHPStan reportaría:
// - Parámetro sin tipo (si no tiene PHPDoc)
// - Posible null en operación aritmética
// - Tipos incompatibles en asignaciones
?>
```

RESUMEN DEL CAPÍTULO
===============================================================================

- PHP soporta comentarios de una línea (//, #), bloque (/* */) y PHPDoc.
- Los comentarios deben explicar el "por qué", no el "qué".
- PHPDoc usa tags (@param, @return, @throws) para documentar tipos.
- PSR-1 y PSR-12 son los estándares de codificación de la comunidad.
- Usa PascalCase para clases, camelCase para variables/funciones,
  UPPER_SNAKE_CASE para constantes.
- Evita malas prácticas: extract(), global, datos sin sanitizar, números
  mágicos, código espagueti.
- PHPCS, PHPStan, Psalm y PHP-CS-Fixer mantienen la calidad del código.

===============================================================================
PREGUNTAS FRECUENTES
===============================================================================

❓ **¿Debo comentar todo el código?**
No. Comenta solo lo necesario: decisiones complejas, workarounds,
TODO/FIXME. El buen código debe ser autodocumentado.

❓ **¿Qué nivel de PHPStan debo usar?**
Empieza con level 1 y aumenta progresivamente. Los proyectos maduros
apuntan a level max (10 en PHPStan 1.x).

❓ **¿PHPDoc es obligatorio?**
No, pero es una buena práctica. Los IDEs lo usan para autocompletado y
las herramientas de análisis estático para detectar errores.

❓ **¿Puedo mezclar PSR-12 con otros estándares?**
Elige un estándar y mantenlo consistente en todo el proyecto. PSR-12
es el más adoptado.

❓ **¿Los comentarios afectan el rendimiento?**
No. PHP ignora los comentarios durante la ejecución. Los archivos se
cargan igual de rápido.
