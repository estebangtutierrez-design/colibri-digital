===============================================================================
1. ¿QUÉ SON LOS NAMESPACES?
===============================================================================

Un namespace (espacio de nombres) es un contenedor lógico que agrupa
clases, interfaces, traits, funciones y constantes relacionadas. Los
namespaces resuelven dos problemas fundamentales:

1. **Conflictos de nombres**: Dos clases con el mismo nombre pueden
   coexistir en diferentes namespaces.
2. **Organización**: El código se estructura jerárquicamente, reflejando
   la arquitectura del proyecto.

PHP introdujo los namespaces en la versión 5.3.0. Son el equivalente
a los "packages" de Java o los "modules" de Python.

```php
<?php
// Sin namespaces: conflicto potencial
// class Usuario { ... }
// class Usuario { ... } // Error: ya existe

// Con namespaces: cada uno en su contexto
// Archivo: src/Modelo/Usuario.php
namespace MiApp\Modelo;

class Usuario
{
    public function __construct(
        public string $nombre
    ) {}
}

// Archivo: src/Seguridad/Usuario.php
namespace MiApp\Seguridad;

class Usuario
{
    public function __construct(
        public string $username,
        public string $password
    ) {}

    public function verificarPassword(string $password): bool
    {
        return password_verify($password, $this->password);
    }
}
```

Los namespaces se declaran con la palabra clave `namespace` al inicio
del archivo (después de `<?php` y antes de cualquier otro código):

```php
<?php

namespace MiApp\Servicios;

use MiApp\Modelo\Usuario;

class ServicioUsuarios
{
    public function registrarUsuario(string $nombre): Usuario
    {
        return new Usuario($nombre);
    }
}
```

Un archivo puede contener solo un namespace (es lo más común) o varios
namespaces anidados con llaves (no recomendado).

===============================================================================
2. DECLARACIÓN DE NAMESPACES
===============================================================================

La declaración de namespace debe ser la primera declaración en un archivo
PHP (después de la etiqueta `<?php` y `declare(strict_types=1)`).

```php
<?php

declare(strict_types=1);

namespace App\Modelo;

class Cliente
{
    // ...
}
```

Los nombres de namespace siguen estas convenciones:

- Usan UpperCamelCase para cada segmento.
- La raíz suele ser el nombre del proyecto o empresa (Vendor).
- Los segmentos reflejan la jerarquía del directorio.
- Usan `\` como separador (no `/`).

Ejemplos de namespaces comunes:

```
App\Modelo\Usuario
App\Controlador\UsuarioController
App\Servicio\Autenticacion
App\Repositorio\UsuarioRepositorio
App\Excepcion\UsuarioNoEncontradoException
Vendor\Paquete\Clase
LucasEgaAcademy\PHP\Utilidades
```

Puedes tener múltiples niveles de jerarquía:

```php
<?php

namespace LucasEgaAcademy\PhpMasterBook\Modulo4\Ejemplos;

class EjemploNamespace
{
    public function mostrarInfo(): string
    {
        return "Namespace: " . __NAMESPACE__;
    }
}
```

La constante mágica `__NAMESPACE__` devuelve el namespace actual como
string. Es útil para depuración y logging.

Si un archivo no tiene declaración de namespace, sus clases pertenecen
al espacio global (`\`).

===============================================================================
3. IMPORTACIÓN DE NAMESPACES CON USE
===============================================================================

El operador `use` importa clases, interfaces, traits, funciones y
constantes de otros namespaces al contexto actual, permitiendo usar
nombres cortos en lugar de fully qualified names (FQN).

```php
<?php

namespace App\Controladores;

// Sin use: debemos usar el nombre completamente cualificado
// $usuario = new \App\Modelos\Usuario();

// Con use: podemos usar el nombre corto
use App\Modelos\Usuario;
use App\Modelos\Pedido;
use App\Servicios\Notificador;

class UsuarioController
{
    public function crear(array $datos): string
    {
        $usuario = new Usuario($datos['nombre'], $datos['email']);
        $pedido = new Pedido($usuario);

        $notificador = new Notificador();
        $notificador->notificarBienvenida($usuario);

        return "Usuario creado exitosamente";
    }

    public function listar(): array
    {
        // También podemos usar el FQN si preferimos
        $repositorio = new \App\Repositorios\UsuarioRepositorio();
        return $repositorio->obtenerTodos();
    }
}
```

La declaración `use` puede importar:

- `use Namespace\Clase;` — Importa una clase.
- `use Namespace\Interfaz;` — Importa una interfaz.
- `use Namespace\Trait;` — Importa un trait.
- `use Namespace\function;` — Importa una función (PHP 5.6+).
- `use Namespace\const;` — Importa una constante (PHP 5.6+).

Múltiples imports del mismo namespace pueden agruparse:

```php
<?php

use App\Modelos\{Usuario, Pedido, Producto};
use App\Servicios\{Notificador, Logger, EmailService};

// Equivalente a:
// use App\Modelos\Usuario;
// use App\Modelos\Pedido;
// use App\Modelos\Producto;
// use App\Servicios\Notificador;
// use App\Servicios\Logger;
// use App\Servicios\EmailService;
```

Importación de funciones y constantes:

```php
<?php

namespace App\Util;

// Definición de una función en un namespace
function formatearFecha(\DateTime $fecha): string
{
    return $fecha->format('d/m/Y');
}

const VERSION = '1.0.0';
```

```php
<?php

namespace App\Controladores;

use function App\Util\formatearFecha;
use const App\Util\VERSION;

class ReporteController
{
    public function generar(\DateTime $fecha): string
    {
        return "Reporte v" . VERSION . " - Fecha: " . formatearFecha($fecha);
    }
}
```

La importación `use` solo afecta al archivo actual. No importa el archivo
físicamente (eso es trabajo del autoloader).

===============================================================================
4. ALIAS CON AS
===============================================================================

El operador `as` permite crear alias (nombres alternativos) para clases,
namespaces, interfaces, funciones o constantes importados.

```php
<?php

namespace App\Controladores;

// Alias para resolver conflictos de nombres
use App\Modelos\Usuario as UsuarioModelo;
use App\Seguridad\Usuario as UsuarioSeguridad;

// Alias para acortar nombres largos
use App\Servicios\Notificacion\Email\ProveedorSendGrid as SendGrid;
use App\Repositorios\RepositorioBaseDeDatos as DBRepo;

// Alias para funciones
use function App\Util\formatearFecha as ff;
use const App\Util\VERSION as VER;

class UsuarioController
{
    public function crear(array $datos): void
    {
        $usuarioModelo = new UsuarioModelo($datos['nombre'], $datos['email']);

        $usuarioSeguridad = new UsuarioSeguridad(
            $datos['email'],
            $datos['password']
        );

        $repo = new DBRepo();
        $repo->guardar($usuarioModelo);

        echo ff(new \DateTime()); // Usando alias de función
        echo VER;                  // Usando alias de constante
    }
}
```

Los alias son especialmente útiles cuando:

1. Dos clases importadas tienen el mismo nombre.
2. El nombre de una clase es muy largo.
3. Quieres dar un nombre más semántico en un contexto específico.

También se puede crear un alias para un namespace completo:

```php
<?php

use App\Servicios\Notificacion\Email as EmailServicios;

$proveedor = new EmailServicios\ProveedorSendGrid();
$plantilla = new EmailServicios\Plantillas\Bienvenida();
```

===============================================================================
5. EL NAMESPACE GLOBAL Y EL OPERADOR \
===============================================================================

El namespace global es el espacio sin nombre. Las clases nativas de PHP
(como `Exception`, `DateTime`, `PDO`) viven en el namespace global.

Para acceder a una clase del namespace global desde dentro de un
namespace, se usa el prefijo `\` (backslash):

```php
<?php

namespace App\Servicios;

use App\Modelos\Usuario;

class Procesador
{
    public function procesar(Usuario $usuario): void
    {
        // DateTime es del namespace global
        $fecha = new \DateTime();

        // Exception es del namespace global
        try {
            // Lógica...
        } catch (\Exception $e) {
            echo "Error: " . $e->getMessage();
        }

        // PDO es del namespace global
        $pdo = new \PDO('mysql:host=localhost;dbname=test', 'root', '');
    }
}
```

Sin la `\` inicial, PHP buscaría `App\Servicios\DateTime`, que no existe,
lanzando un error.

Si estás en el namespace global (sin declaración `namespace`), no
necesitas la `\` inicial:

```php
<?php

// Sin namespace: estamos en el global
$fecha = new DateTime();     // Funciona
$excepcion = new Exception(); // Funciona
```

El nombre completamente cualificado (Fully Qualified Name, FQN) incluye
la `\` inicial y la ruta completa del namespace:

```
\App\Modelos\Usuario
\DateTime
\Exception
\App\Servicios\Procesador
```

Dentro de un namespace, las referencias a clases sin `use` ni `\` se
resuelven relativas al namespace actual:

```php
<?php

namespace App\Modelos;

class Usuario {}
class Admin extends Usuario {} // Busca App\Modelos\Usuario (correcto)
```

```php
<?php

namespace App\Servicios;

class Procesador
{
    public function test(): void
    {
        $obj = new Usuario(); // Busca App\Servicios\Usuario (no existe)
    }
}
```

Para referenciar `App\Modelos\Usuario` desde `App\Servicios`, debes
usar `use App\Modelos\Usuario` o `new \App\Modelos\Usuario()`.

===============================================================================
6. AUTOLOADING CON PSR-4 Y COMPOSER
===============================================================================

PSR-4 es un estándar de PHP FIG (Framework Interoperability Group) que
define cómo debe realizarse el autoloading de clases a partir de
namespaces. Es el estándar más utilizado hoy en día, implementado por
Composer.

La regla fundamental de PSR-4:

**El namespace raíz (vendor namespace) se corresponde con un directorio
base. Los sub-namespaces se corresponden con subdirectorios. El nombre
de la clase (sin namespace) se corresponde con el nombre del archivo
más `.php`.**

```
Namespace: App\Modelos\Usuario
Directorio: src/Modelos/Usuario.php

Namespace: App\Controladores\Admin\UsuarioController
Directorio: src/Controladores/Admin/UsuarioController.php

Namespace: LucasEgaAcademy\PhpMaster\Utilidades
Directorio: src/Utilidades.php
```

Para configurar PSR-4 con Composer, edita el archivo `composer.json`:

```json
{
    "name": "mi-app/proyecto",
    "autoload": {
        "psr-4": {
            "App\\": "src/",
            "LucasEgaAcademy\\PhpMaster\\": "lib/",
            "MiApp\\Test\\": "tests/"
        }
    }
}
```

Luego ejecuta `composer dump-autoload` para generar los archivos de
autoloading.

Ejemplo completo de estructura PSR-4:

```
mi-proyecto/
├── composer.json
├── src/
│   ├── Modelos/
│   │   ├── Usuario.php          (namespace App\Modelos)
│   │   └── Pedido.php           (namespace App\Modelos)
│   ├── Controladores/
│   │   ├── UsuarioController.php (namespace App\Controladores)
│   │   └── Admin/
│   │       └── DashboardController.php (namespace App\Controladores\Admin)
│   ├── Servicios/
│   │   └── Autenticacion.php    (namespace App\Servicios)
│   └── Excepciones/
│       └── NotFoundException.php (namespace App\Excepciones)
├── public/
│   └── index.php
└── vendor/
```

Archivo `public/index.php`:

```php
<?php

declare(strict_types=1);

require_once __DIR__ . '/../vendor/autoload.php';

use App\Modelos\Usuario;
use App\Controladores\UsuarioController;
use App\Servicios\Autenticacion;

$usuario = new Usuario('Ana García', 'ana@example.com');
$auth = new Autenticacion();
$auth->login($usuario);

$controlador = new UsuarioController();
$controlador->mostrarPerfil($usuario);
```

Si no quieres usar Composer (no recomendado en proyectos reales), puedes
implementar un autoloader manual basado en PSR-4:

```php
<?php

// Autoloader PSR-4 manual
spl_autoload_register(function (string $clase): void {
    $prefijo = 'App\\';
    $directorioBase = __DIR__ . '/../src/';

    $longitudPrefijo = strlen($prefijo);

    // Verificar si la clase usa el prefijo deseado
    if (strncmp($prefijo, $clase, $longitudPrefijo) !== 0) {
        return; // No es nuestra responsabilidad
    }

    // Obtener el nombre relativo de la clase (sin prefijo)
    $nombreRelativo = substr($clase, $longitudPrefijo);

    // Convertir namespace a ruta de directorio
    $archivo = $directorioBase . str_replace('\\', '/', $nombreRelativo) . '.php';

    if (file_exists($archivo)) {
        require $archivo;
    }
});
```

Sin embargo, en proyectos profesionales, siempre debes usar Composer y
su autoloader, que es mucho más completo y eficiente.

7. NAMESPACE Y LA CONSTANTE __NAMESPACE__
===============================================================================

PHP proporciona la constante `__NAMESPACE__` que contiene el namespace
actual como string. Es útil para:

- Logging y depuración.
- Construir nombres de clase dinámicamente.
- Verificar en qué namespace estamos.

```php
<?php

namespace App\Util\Depuracion;

class Depurador
{
    public static function getInfo(): array
    {
        return [
            'namespace_actual' => __NAMESPACE__,
            'clase_actual' => __CLASS__,
            'metodo_actual' => __METHOD__,
        ];
    }

    public static function crearClaseDinamica(string $nombreClase): string
    {
        // Construir nombre completo con el namespace actual
        $fqn = __NAMESPACE__ . '\\' . $nombreClase;

        if (class_exists($fqn)) {
            return $fqn;
        }

        throw new \RuntimeException("Clase no encontrada: $fqn");
    }
}

$info = Depurador::getInfo();
// namespace_actual: App\Util\Depuracion
// clase_actual: App\Util\Depuracion\Depurador

$clase = Depurador::crearClaseDinamica('Depurador');
// App\Util\Depuracion\Depurador
```

La palabra clave `namespace` también se puede usar para referenciar
explícitamente el namespace actual:

```php
<?php

namespace App\Modelos;

class Usuario
{
    public function clonar(): Usuario
    {
        // namespace\NombreClase es equivalente a App\Modelos\NombreClase
        return new namespace\Usuario();
        // Equivalente a: new \App\Modelos\Usuario()
    }
}
```

===============================================================================
8. BUENAS PRÁCTICAS CON NAMESPACES
===============================================================================

1. **Una sola declaración por archivo**: No uses múltiples namespaces
   en un mismo archivo. Es confuso y contrario a las convenciones.

2. **Correspondencia 1:1 con directorios**: La estructura de directorios
   debe reflejar exactamente la estructura de namespaces (PSR-4).

3. **Vendor prefix único**: Usa un prefijo único para tu proyecto o
   empresa (ej: `App\`, `LucasEgaAcademy\`, `MiEmpresa\`).

4. **Namespace después de declare**: El orden correcto es:

```php
<?php

declare(strict_types=1);

namespace App\Modelos;

use App\Servicios\Notificador;

class Usuario { /* ... */ }
```

5. **Nombres en inglés o español**: Elige un idioma y sé consistente.
   En equipos internacionales, se prefiere inglés. Para proyectos
   locales, español es aceptable.

6. **Agrupa por funcionalidad, no por tipo**: En lugar de:
   ```
   App\Clases\Usuario
   App\Clases\Pedido
   App\Interfaces\RepositorioInterface
   ```
   Prefiere agrupar por módulo:
   ```
   App\Modelos\Usuario
   App\Modelos\Pedido
   App\Repositorios\UsuarioRepositorio
   App\Repositorios\PedidoRepositorio
   ```

7. **Usa `use` en la parte superior**: Todas las declaraciones `use`
   deben ir al inicio del archivo, después de `namespace`.

8. **No uses `use` para clases del mismo namespace**: Si estás en
   `App\Modelos`, no necesitas importar `App\Modelos\Producto` con
   `use`.

9. **Ordena los `use` alfabéticamente** para facilitar la lectura.

10. **Siempre usa Composer para autoloading**: No implementes tu propio
    autoloader a menos que tengas una razón muy específica.

===============================================================================
RESUMEN DEL CAPÍTULO
===============================================================================

- Los namespaces organizan el código y evitan conflictos de nombres.
- Se declaran con `namespace Vendor\Paquete;` al inicio del archivo.
- `use` importa clases, interfaces, funciones y constantes al contexto
  actual.
- `as` crea alias para resolver conflictos o acortar nombres.
- `\` (backslash inicial) accede al namespace global.
- PSR-4 es el estándar de autoloading: namespace = directorio, clase =
  archivo.
- Composer implementa PSR-4 automáticamente con `composer.json`.
- `__NAMESPACE__` es la constante que contiene el namespace actual.
- Las buenas prácticas incluyen: vendor prefix, 1 clase por archivo,
  organización por funcionalidad.

===============================================================================
PREGUNTAS FRECUENTES
===============================================================================

❓ **¿Puedo tener múltiples namespaces en un mismo archivo?**
Técnicamente sí, con la sintaxis de llaves, pero es una mala práctica.
Cada archivo debe tener un solo namespace y una sola clase para
facilitar el autoloading y la navegación.

❓ **¿Qué pasa si no declaro un namespace?**
Las clases viven en el namespace global. Pueden ser referenciadas como
`\NombreClase`. Esto funciona, pero causa problemas de colisión de
nombres en proyectos grandes.

❓ **¿Los namespaces afectan el rendimiento?**
No de manera significativa. El autoloading puede tener un pequeño
impacto en la primera carga (resolución de archivos), pero con opcache
en producción, este impacto es negligible.

❓ **¿Cómo referencio una clase del namespace global desde un namespace?**
Usa el prefijo `\`: `new \DateTime()`, `new \Exception()`, `new \PDO()`.

❓ **¿Composer es obligatorio para usar namespaces?**
No, puedes usar tu propio autoloader con `spl_autoload_register()`. Sin
embargo, Composer es el estándar de la industria y proporciona muchas
ventajas (optimización, generación de classmap, integración con
paquetes de terceros).

❓ **¿Qué diferencia hay entre PSR-0 y PSR-4?**
PSR-0 (obsoleto) requería que el guión bajo en nombres de clase se
convirtiera en separador de directorio. PSR-4 es más flexible porque
mapea directamente el namespace a un directorio base sin esa
transformación. PSR-4 es el estándar actual.

Diseña y ejecuta la migración de un proyecto PHP estructurado

```
proyecto_legacy/

│   ├── db.php
│   ├── funciones.php
│   └── config.php

│   ├── Carrito.php
│   └── Admin/
│       ├── UsuarioAdmin.php
│       └── Reportes.php
├── public/
│   ├── index.php
│   └── admin/
│       └── dashboard.php
└── index.php
```

Tareas a realizar:

1. **Rediseñar la estructura** siguiendo PSR-4 con el vendor `App\`:

```
proyecto_moderno/
├── src/
│   ├── Modelos/
│   │   ├── Usuario.php       (namespace App\Modelos)
│   │   ├── Pedido.php        (namespace App\Modelos)
│   │   ├── Producto.php      (namespace App\Modelos)
│   │   └── Carrito.php       (namespace App\Modelos)
│   ├── Admin/
│   │   ├── UsuarioAdmin.php  (namespace App\Admin)
│   │   └── Reportes.php     (namespace App\Admin)
│   ├── Servicios/
│   │   ├── BaseDatos.php     (namespace App\Servicios)
│   │   └── Autenticacion.php (namespace App\Servicios)
│   └── Util/
│       ├── funciones.php     (namespace App\Util, funciones sueltas)
│       └── Configuracion.php (namespace App\Util)
├── public/
│   ├── index.php
│   └── admin/
│       └── dashboard.php
├── composer.json
├── vendor/
└── config/
    └── database.php
```

2. **Configurar composer.json** con PSR-4 para `App\` → `src/`.
   Incluir también `App\Config\` → `config/` si es necesario.

3. **Refactorizar cada clase** para incluir su namespace y declaración
   `declare(strict_types=1)`.

4. **Crear archivos de arranque**:

   a. `public/index.php` debe:
      - Requerir `vendor/autoload.php`.
      - Usar `use App\Servicios\BaseDatos;` y `use App\Modelos\Producto;`.
      - Mostrar listado de productos, crear carrito, etc.

   b. `public/admin/dashboard.php` debe:
      - Usar `use App\Admin\Reportes;` y `use App\Admin\UsuarioAdmin;`.
      - Mostrar reportes de ventas en los últimos 30 días.

5. **Resolver conflictos**: Si hay clases con el mismo nombre en
   diferentes namespaces (ej: `Usuario` en Modelos y Admin), usar alias
   con `as`.

6. **Importar funciones** con `use function App\Util\...` para las
   funciones de utilidad.

7. **Script de demostración**: Crear un script `demo_migracion.php` que:
   - Demuestre que todas las clases se cargan automáticamente (sin
     requires manuales).
   - Muestre conflictos resueltos con alias.
   - Muestre el uso de funciones importadas con `use function`.
   - Verifique que las clases nativas (DateTime, Exception) se usan
     correctamente desde los namespaces.

Requisitos de validación:

- No debe haber ningún `require_once` (excepto el de `vendor/autoload.php`).
- Todas las clases deben tener `declare(strict_types=1)`.
- El proyecto debe poder moverse de carpeta sin cambiar rutas absolutas.
- Los namespaces deben reflejar exactamente la estructura de directorios.
- Se debe ejecutar `composer dump-autoload -o` para optimizar el
  autoloading.

Este desafío integra todos los conceptos de namespaces y autoloading,
y te prepara para trabajar en proyectos PHP profesionales que utilizan
Composer y PSR-4 como estándar.
