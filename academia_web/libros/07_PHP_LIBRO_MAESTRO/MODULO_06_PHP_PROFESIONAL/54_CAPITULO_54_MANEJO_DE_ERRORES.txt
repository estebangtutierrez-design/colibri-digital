BIENVENIDA DEL PROFESOR BÚHO

En aplicaciones PHP profesionales, los errores no pueden mostrarse
al usuario como mensajes feos de "Fatal error" o "Warning". Un
usuario no debe ver una pila de llamadas (stack trace) ni detalles
técnicos.

El manejo profesional de errores consiste en:
  - Capturar errores y excepciones.
  - Registrar los detalles en un archivo de log.
  - Mostrar al usuario un mensaje amigable.
  - En desarrollo, mostrar información detallada para depurar.
  - En producción, ocultar detalles sensibles.

PHP proporciona herramientas poderosas para todo esto: try/catch,
set_error_handler, set_exception_handler, y Monolog para logging.

1. TIPOS DE ERRORES EN PHP
===============================================================================

PHP clasifica los errores en varios niveles:

Error fatal (E_ERROR):
  Ejecución detenida. No se puede recuperar.
  Ejemplo: require de archivo inexistente, memoria insuficiente.

Advertencia (E_WARNING):
  Ejecución continúa. Problema no crítico.
  Ejemplo: include de archivo inexistente, división por cero.

Noticia (E_NOTICE):
  Ejecución continúa. Posible error o código descuidado.
  Ejemplo: variable no definida, array sin índice.

Error fatal de usuario (E_USER_ERROR):
  Generado con trigger_error() y bandera E_USER_ERROR.

Advertencia de usuario (E_USER_WARNING):
  Generado con trigger_error() y bandera E_USER_WARNING.

Error de parse (E_PARSE):
  Error de sintaxis. PHP no puede analizar el archivo.

Error de compilación (E_COMPILE_ERROR):
  Error durante la compilación del script.

Error recuperable (E_RECOVERABLE_ERROR):
  Error fatal pero recuperable. PHP lanza un Error exception.

Deprecation (E_DEPRECATED):
  Funcionalidad obsoleta que será eliminada en el futuro.

En PHP 8, muchos errores fatales se convirtieron en excepciones
de la clase \Error, que pueden capturarse con try/catch.

```php
<?php
// Ejemplo de diferentes niveles de error
// E_NOTICE: variable no definida
echo $variableNoDefinida;

// E_WARNING: archivo no encontrado
include 'archivo_inexistente.php';

// E_USER_NOTICE: advertencia personalizada
trigger_error('Esto es un aviso', E_USER_NOTICE);

// E_RECOVERABLE_ERROR: type hint incorrecto
function saludar(string $nombre) { echo "Hola $nombre"; }
saludar(42); // TypeError en PHP 8
```

2. CONFIGURACIÓN DE ERROR REPORTING
===============================================================================

La función error_reporting() controla qué tipos de errores se
reportan.

Para desarrollo (mostrar todo):
```php
<?php
// Mostrar todos los errores excepto deprecations
error_reporting(E_ALL & ~E_DEPRECATED);

// Mostrar absolutamente todo
error_reporting(E_ALL);

// Configurar en php.ini
ini_set('display_errors', '1');
ini_set('display_startup_errors', '1');
```

Para producción (ocultar todo):
```php
<?php
// No mostrar errores en pantalla
ini_set('display_errors', '0');
ini_set('display_startup_errors', '0');

// Seguir reportando (para logging)
error_reporting(E_ALL & ~E_DEPRECATED & ~E_NOTICE);
```

Configuración según entorno:

```php
<?php
// config/error_handler.php

define('ENVIRONMENT', $_ENV['APP_ENV'] ?? 'production');

switch (ENVIRONMENT) {
    case 'development':
    case 'local':
        error_reporting(E_ALL);
        ini_set('display_errors', '1');
        ini_set('display_startup_errors', '1');
        break;

    case 'testing':
        error_reporting(E_ALL & ~E_DEPRECATED);
        ini_set('display_errors', '1');
        ini_set('display_startup_errors', '1');
        break;

    case 'production':
    default:
        error_reporting(E_ALL & ~E_DEPRECATED & ~E_NOTICE);
        ini_set('display_errors', '0');
        ini_set('display_startup_errors', '0');
        ini_set('log_errors', '1');
        ini_set('error_log', __DIR__ . '/../logs/php_errors.log');
        break;
}
```

Esta configuración es crítica: en producción NUNCA debes mostrar
errores en pantalla. Podrían revelar credenciales de base de
datos, rutas del servidor, o información sensible.

3. TRY/CATCH PARA EXCEPCIONES
===============================================================================

Las excepciones son la forma moderna de manejar errores en PHP.
Se lanzan con throw y se capturan con try/catch.

```php
<?php
try {
    // Código que puede lanzar una excepción
    $archivo = fopen('config.json', 'r');
    if (!$archivo) {
        throw new Exception('No se pudo abrir el archivo');
    }
    $contenido = fread($archivo, filesize('config.json'));
    fclose($archivo);
} catch (Exception $e) {
    // Manejar la excepción
    echo 'Error: ' . $e->getMessage();
} finally {
    // Este bloque SIEMPRE se ejecuta
    echo 'Proceso finalizado.';
}
```

El bloque finally es opcional pero muy útil para:
  - Cerrar conexiones a base de datos.
  - Cerrar archivos abiertos.
  - Liberar recursos.

Múltiples catches:

```php
<?php
try {
    // Código que lanza diferentes tipos de excepción
    $db = new PDO('mysql:host=localhost;dbname=test', 'user', 'pass');
    $stmt = $db->query('SELECT * FROM usuarios');
    $resultados = $stmt->fetchAll();
} catch (PDOException $e) {
    // Error específico de base de datos
    log_error('Error BD: ' . $e->getMessage());
    echo 'Error de conexión. Intente más tarde.';
} catch (Exception $e) {
    // Cualquier otra excepción
    log_error('Error general: ' . $e->getMessage());
    echo 'Error inesperado.';
}
```

Siempre captura las excepciones más específicas primero y la
genérica (Exception) al final.

4. EXCEPCIONES PERSONALIZADAS
===============================================================================

Crear excepciones personalizadas mejora la claridad del código
y permite capturar errores específicos.

```php
<?php
namespace App\Exceptions;

class DatabaseException extends \RuntimeException
{
    private string $sql;

    public function __construct(string $message, string $sql = '', int $code = 0, ?\Throwable $previous = null)
    {
        parent::__construct($message, $code, $previous);
        $this->sql = $sql;
    }

    public function getSql(): string
    {
        return $this->sql;
    }

    public function getDetailedMessage(): string
    {
        return sprintf(
            '[Database] %s | SQL: %s | File: %s:%d',
            $this->getMessage(),
            $this->sql,
            $this->getFile(),
            $this->getLine()
        );
    }
}
```

```php
<?php
namespace App\Exceptions;

class ValidationException extends \InvalidArgumentException
{
    private array $errors;

    public function __construct(array $errors)
    {
        parent::__construct('Error de validación');
        $this->errors = $errors;
    }

    public function getErrors(): array
    {
        return $this->errors;
    }
}
```

```php
<?php
namespace App\Exceptions;

class AuthenticationException extends \RuntimeException
{
    public function __construct(string $message = 'No autenticado')
    {
        parent::__construct($message, 401);
    }
}
```

Uso en controladores:

```php
<?php
namespace App\Controllers;

use App\Exceptions\ValidationException;
use App\Exceptions\AuthenticationException;
use App\Exceptions\DatabaseException;
use App\Models\Usuario;

class UsuarioController
{
    public function store(): void
    {
        try {
            $this->validarDatos($_POST);

            $usuario = new Usuario();
            $usuario->create($_POST);

            $this->json(['success' => true]);
        } catch (ValidationException $e) {
            $this->json(['errors' => $e->getErrors()], 422);
        } catch (DatabaseException $e) {
            log_error($e->getDetailedMessage());
            $this->json(['error' => 'Error en el servidor'], 500);
        }
    }

    private function validarDatos(array $data): void
    {
        $errors = [];
        if (empty($data['nombre'])) {
            $errors['nombre'] = 'El nombre es requerido';
        }
        if (empty($data['email'])) {
            $errors['email'] = 'El email es requerido';
        }
        if (!empty($errors)) {
            throw new ValidationException($errors);
        }
    }
}
```

5. ERROR VS EXCEPTION EN PHP 8
===============================================================================

En PHP 7 y 8, muchos errores fatales se lanzan como excepciones
de la clase \Error. Esto significa que puedes capturarlos con
try/catch.

```php
<?php
try {
    // Llamar a un método que no existe
    $obj = new stdClass();
    $obj->metodoInexistente();
} catch (\Error $e) {
    // Captura TypeError, DivisionByZeroError, etc.
    echo 'Error capturado: ' . $e->getMessage();
}
```

Jerarquía de Throwable:

```
Throwable (interface)
├── Exception
│   ├── LogicException
│   │   ├── InvalidArgumentException
│   │   ├── LengthException
│   │   ├── OutOfRangeException
│   │   └── BadFunctionCallException
│   └── RuntimeException
│       ├── OutOfBoundsException
│       ├── OverflowException
│       ├── PDOException
│       └── UnexpectedValueException
└── Error
    ├── TypeError
    ├── ArithmeticError
    │   └── DivisionByZeroError
    ├── ParseError
    └── AssertionError
```

Regla general:
  - Exception: errores predecibles (validación, lógica de negocio).
  - Error: errores de sistema (type hints, división por cero).
  - Captura \Throwable si quieres absolutamente todo.

```php
<?php
try {
    // Código que puede lanzar cualquier cosa
} catch (\Throwable $t) {
    // Captura Exception y Error
    echo get_class($t) . ': ' . $t->getMessage();
}
```

6. SET_ERROR_HANDLER: MANEJADOR PERSONALIZADO
===============================================================================

set_error_handler() reemplaza el manejador de errores nativo de
PHP por uno personalizado.

```php
<?php
// config/error_handler.php

/**
 * Manejador personalizado de errores.
 * Convierte errores en excepciones para manejarlos con try/catch.
 *
 * @throws \ErrorException
 */
set_error_handler(function (
    int $severity,
    string $message,
    string $file,
    int $line
): bool {
    // Convertir el error en una excepción
    throw new \ErrorException($message, 0, $severity, $file, $line);

    // Devolver true para indicar que manejamos el error
    return true;
});
```

Ahora cualquier error (E_WARNING, E_NOTICE, etc.) se convierte
en una excepción que puedes capturar:

```php
<?php
// Sin el manejador, esto solo muestra un warning
// Con el manejador, esto lanza una excepción

try {
    include 'archivo_inexistente.php';
    $resultado = 1 / 0; // DivisionByZeroError
} catch (\ErrorException $e) {
    echo 'Error convertido: ' . $e->getMessage();
}
```

Manejador completo con logging:

```php
<?php
// config/error_handler.php

use Monolog\Logger;
use Monolog\Handler\StreamHandler;

$log = new Logger('app');
$log->pushHandler(new StreamHandler(__DIR__ . '/../logs/error.log'));

set_error_handler(function (
    int $severity,
    string $message,
    string $file,
    int $line
) use ($log): bool {
    // Niveles de severidad
    $niveles = [
        E_ERROR           => 'Error',
        E_WARNING         => 'Warning',
        E_NOTICE          => 'Notice',
        E_USER_ERROR      => 'User Error',
        E_USER_WARNING    => 'User Warning',
        E_USER_NOTICE     => 'User Notice',
        E_DEPRECATED      => 'Deprecated',
        E_RECOVERABLE_ERROR => 'Recoverable Error',
    ];

    $nivel = $niveles[$severity] ?? 'Unknown';

    // Registrar en el log
    $log->error("[$nivel] $message in $file:$line");

    // En desarrollo, mostrar en pantalla
    if (ENVIRONMENT === 'development') {
        echo "<div style='color:red;'><b>[$nivel]</b> $message<br>
              <small>$file:$line</small></div>";
    } else {
        // En producción, no mostrar nada sensible
        if ($severity === E_ERROR || $severity === E_USER_ERROR) {
            echo "<div style='color:red;'>Error interno del servidor.</div>";
        }
    }

    // No ejecutar el manejador interno de PHP
    return true;
});
```

7. SET_EXCEPTION_HANDLER: EXCEPCIONES NO CAPTURADAS
===============================================================================

set_exception_handler() captura cualquier excepción que no haya
sido capturada por un bloque try/catch.

```php
<?php
// config/error_handler.php

set_exception_handler(function (\Throwable $e): void {
    // Registrar en log
    $mensaje = sprintf(
        "[%s] %s in %s:%d\nStack trace:\n%s",
        get_class($e),
        $e->getMessage(),
        $e->getFile(),
        $e->getLine(),
        $e->getTraceAsString()
    );

    error_log($mensaje, 3, __DIR__ . '/../logs/exceptions.log');

    // Responder según el entorno
    if (ENVIRONMENT === 'development') {
        // Mostrar detalles en desarrollo
        http_response_code(500);
        echo "<h1>Error Interno</h1>";
        echo "<p><b>" . get_class($e) . ":</b> " . htmlspecialchars($e->getMessage()) . "</p>";
        echo "<pre>" . htmlspecialchars($e->getTraceAsString()) . "</pre>";
    } else {
        // Producción: página amigable
        http_response_code(500);
        require __DIR__ . '/../views/errors/500.php';
    }

    exit(1);
});
```

Registra ambos manejadores al inicio:

```php
<?php
// public/index.php
require_once __DIR__ . '/../vendor/autoload.php';
require_once __DIR__ . '/../config/error_handler.php';
require_once __DIR__ . '/../config/app.php';

// A partir de aquí, todos los errores son capturados
$app = new App\Core\Application();
$app->run();
```

Con esta configuración, cualquier error no capturado termina en
el manejador de excepciones, que decide cómo responder.

8. WHOops: DEBUGGING EN DESARROLLO
===============================================================================

Whoops es una librería que reemplaza el manejador de errores de
PHP con una interfaz visual hermosa y útil para desarrollo.

```bash
composer require --dev filp/whoops
```

```php
<?php
// public/index.php (desarrollo)
require_once __DIR__ . '/../vendor/autoload.php';

$whoops = new \Whoops\Run();

if (ENVIRONMENT === 'development') {
    // Desarrollo: mostrar Whoops pretty error
    $whoops->pushHandler(new \Whoops\Handler\PrettyPageHandler());
} else {
    // Producción: página amigable
    $whoops->pushHandler(function (\Throwable $e) {
        http_response_code(500);
        error_log($e->getMessage());
        require __DIR__ . '/../views/errors/500.php';
        exit(1);
    });
}

$whoops->register();

// El resto de la aplicación
```

Whoops muestra:
  - El mensaje de error y tipo de excepción.
  - La pila de llamadas completa.
  - El código fuente en la línea del error.
  - Variables en cada frame de la pila.
  - Pestañas con request data, server data, etc.

Whoops es solo para desarrollo. Nunca lo uses en producción.

9. LOGGING PROFESIONAL CON MONOLOG
===============================================================================

Monolog es la librería de logging estándar de PHP.

```bash
composer require monolog/monolog
```

```php
<?php
use Monolog\Logger;
use Monolog\Handler\StreamHandler;
use Monolog\Handler\RotatingFileHandler;
use Monolog\Handler\NativeMailerHandler;
use Monolog\Formatter\LineFormatter;

// Crear el logger
$log = new Logger('app');

// Formato personalizado
$formato = new LineFormatter(
    "[%datetime%] %channel%.%level_name%: %message% %context% %extra%\n",
    'Y-m-d H:i:s'
);

// Handler para archivo rotativo (nuevo archivo cada día)
$rotating = new RotatingFileHandler(__DIR__ . '/../logs/app.log', 30);
$rotating->setFormatter($formato);
$log->pushHandler($rotating);

// Handler para errores críticos por email
$emailHandler = new NativeMailerHandler(
    'admin@mi-app.com',
    'Error crítico en la aplicación',
    'errores@mi-app.com',
    Logger::CRITICAL
);
$log->pushHandler($emailHandler);

// Uso
$log->info('Usuario registrado', ['id' => 42, 'email' => 'user@example.com']);
$log->error('Error de base de datos', ['sql' => 'SELECT *', 'error' => 'Table not found']);
$log->critical('Servidor sin espacio en disco', ['disk_free' => disk_free_space('/')]);

try {
    // operación riesgosa
} catch (\Throwable $e) {
    $log->error($e->getMessage(), [
        'exception' => get_class($e),
        'file' => $e->getFile(),
        'line' => $e->getLine(),
        'trace' => $e->getTraceAsString(),
    ]);

    throw $e; // relanzar si es necesario
}
```

Niveles de logging (de menor a mayor severidad):
  - DEBUG: información de depuración detallada.
  - INFO: eventos normales de la aplicación.
  - NOTICE: eventos normales pero significativos.
  - WARNING: eventos que no son errores pero son inusuales.
  - ERROR: errores que deberían investigarse.
  - CRITICAL: condiciones críticas.
  - ALERT: acción requerida inmediatamente.
  - EMERGENCY: el sistema está inservible.

Integración con el manejador de errores:

```php
<?php
// config/error_handler.php

$log = new Logger('app');
$log->pushHandler(new RotatingFileHandler(__DIR__ . '/../logs/error.log', 30));

// Manejador de errores
set_error_handler(function ($severity, $message, $file, $line) use ($log) {
    $log->error("[$severity] $message", ['file' => $file, 'line' => $line]);
    throw new \ErrorException($message, 0, $severity, $file, $line);
});

// Manejador de excepciones
set_exception_handler(function (\Throwable $e) use ($log) {
    $log->critical($e->getMessage(), [
        'exception' => get_class($e),
        'file' => $e->getFile(),
        'line' => $e->getLine(),
        'trace' => $e->getTraceAsString(),
    ]);

    http_response_code(500);
    if (ENVIRONMENT === 'development') {
        // Mostrar error detallado
        echo '<pre>' . $e . '</pre>';
    } else {
        // Página amigable
        require __DIR__ . '/../views/errors/500.php';
    }
    exit(1);
});
```

10. CÓDIGO DE RESPUESTA HTTP Y ERRORES
===============================================================================

Los códigos HTTP son parte integral del manejo de errores en APIs
y aplicaciones web.

```php
<?php
class Response
{
    public static function json(array $data, int $status = 200): void
    {
        http_response_code($status);
        header('Content-Type: application/json');
        echo json_encode($data, JSON_UNESCAPED_UNICODE);
        exit;
    }

    public static function success(mixed $data, string $message = 'OK'): void
    {
        self::json([
            'success' => true,
            'message' => $message,
            'data'    => $data,
        ], 200);
    }

    public static function error(string $message, int $status = 400, array $details = []): void
    {
        self::json([
            'success' => false,
            'error'   => [
                'message' => $message,
                'details' => $details,
            ],
        ], $status);
    }

    public static function notFound(string $message = 'Recurso no encontrado'): void
    {
        self::error($message, 404);
    }

    public static function validationError(array $errors): void
    {
        self::error('Error de validación', 422, $errors);
    }

    public static function serverError(string $message = 'Error interno del servidor'): void
    {
        self::error($message, 500);
    }
}
```

Uso en controlador:

```php
<?php
try {
    $usuario = $model->findById($id);
    if (!$usuario) {
        Response::notFound('Usuario con ID ' . $id . ' no encontrado');
    }
    Response::success($usuario);
} catch (\Throwable $e) {
    log_error($e);
    Response::serverError();
}
```

12. ESTRATEGIA DE MANEJO DE ERRORES
===============================================================================

Un plan completo de manejo de errores para producción:

```
NIVEL 1 - Try/Catch (por el desarrollador)
  ↓ (excepción no capturada)
NIVEL 2 - Manejador de Excepciones (set_exception_handler)
  ↓ (error no convertido en excepción)
NIVEL 3 - Manejador de Errores (set_error_handler)
  ↓ (error fatal de PHP)
NIVEL 4 - shutdown function (register_shutdown_function)
```

Implementación del nivel 4:

```php
<?php
register_shutdown_function(function () {
    $error = error_get_last();
    if ($error !== null && in_array($error['type'], [E_ERROR, E_PARSE, E_CORE_ERROR, E_COMPILE_ERROR])) {
        // Error fatal que no pasó por los manejadores
        $mensaje = sprintf(
            "[FATAL] %s in %s:%d",
            $error['message'],
            $error['file'],
            $error['line']
        );
        error_log($mensaje, 3, __DIR__ . '/../logs/fatal.log');

        http_response_code(500);
        if (ENVIRONMENT !== 'development') {
            require __DIR__ . '/../views/errors/500.php';
        } else {
            echo "<h1>Fatal Error</h1><p>$mensaje</p>";
        }
    }
});
```

Este es el safety net final. Si todos los demás manejadores fallan,
esta función capturará el error fatal y mostrará una página amigable.

===============================================================================
RESUMEN DEL CAPÍTULO
===============================================================================

- PHP tiene múltiples niveles de error: fatal, warning, notice, etc.
- error_reporting() controla qué errores se muestran/registran.
- En desarrollo: mostrar todo. En producción: ocultar todo, loguear todo.
- try/catch captura excepciones. finally siempre se ejecuta.
- Las excepciones personalizadas mejoran la claridad del código.
- set_error_handler() convierte errores en excepciones.
- set_exception_handler() captura excepciones no capturadas.
- register_shutdown_function() captura errores fatales finales.
- Whoops es excelente para debugging en desarrollo.
- Monolog proporciona logging profesional con múltiples handlers.
- Los códigos HTTP (404, 500, 422) son parte del manejo de errores.
- Las vistas de error deben ser amigables y seguir el diseño de la app.

===============================================================================
PREGUNTAS FRECUENTES
===============================================================================

❓ **¿Cuál es la diferencia entre error y excepción?**
   Los errores son problemas tradicionales de PHP (fatales, warnings).
   Las excepciones son objetos lanzados con throw. En PHP moderno,
   los errores fatales se lanzan como objetos \Error que también
   puedes capturar con try/catch.

❓ **¿Debo capturar Exception o Throwable?**
   Si usas PHP 7+, captura \Throwable si quieres absolutamente todo.
   Captura \Exception si solo quieres excepciones (no errores).
   Captura clases específicas si quieres manejo granular.

❓ **¿Por qué no debo mostrar errores en producción?**
   Por seguridad. Los errores pueden revelar rutas del servidor,
   credenciales de BD, estructura de tablas, y otra información
   que un atacante podría usar.

❓ **¿Cómo manejo errores en APIs?**
   Captura la excepción, registra en log, devuelve JSON con el
   código HTTP apropiado (400, 404, 422, 500) y un mensaje genérico.

❓ **¿Qué es mejor: trigger_error o throw Exception?**
   Siempre prefiere throw Exception. trigger_error es el estilo
   antiguo de PHP 4. Las excepciones son más flexibles, se pueden
   capturar selectivamente, y funcionan con la jerarquía de clases.

❓ **¿Debo usar finally o no?**
   finally es útil para liberar recursos (cerrar archivos, conexiones)
   que deben ejecutarse tanto si hay excepción como si no. Si no
   tienes recursos que liberar, no necesitas finally.

❓ **¿Cómo manejo errores en scripts CLI?**
   En CLI, display_errors ya está activo por defecto. Puedes usar
   los mismos manejadores pero adaptando la salida para consola
   en lugar de HTML. Whoops tiene un handler para CLI.

DESAFÍO DEL ARQUITECTO

🏗️ **Sistema Profesional de Manejo de Errores**

   Construye un sistema completo de manejo de errores con:

   1. Clase App\Core\ErrorHandler que registre:
      - set_error_handler
      - set_exception_handler
      - register_shutdown_function
   2. Debug mode: con Whoops y barra de depuración.
   3. Producción: logging a archivo, email para críticos.
   4. Soporte para múltiples canales de log (archivo, consola,
      syslog, Slack).
   5. Integración con la respuesta HTTP: devolver vista de error
      para HTML, JSON para APIs.
   6. Método report() para loguear sin interrumpir.
   7. Método render() para mostrar la vista de error apropiada.
   8. Sistema de eventos: onError, onException disparando
      callbacks registrados.

   API final:
   ```php
   $handler = new App\Core\ErrorHandler();
   $handler->debug();  // Modo desarrollo
   $handler->production('logs/');  // Modo producción

   // Uso
   try {
       // ...
   } catch (\Throwable $e) {
       $handler->report($e);  // Loguea
       $handler->render($e);  // Muestra página
   }
   ```

   Este sistema profesional elevará tu manejo de errores al nivel
   de frameworks como Laravel o Symfony.
