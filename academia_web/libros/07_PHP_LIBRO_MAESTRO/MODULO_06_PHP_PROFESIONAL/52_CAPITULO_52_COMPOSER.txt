===============================================================================
1. ¿QUÉ ES COMPOSER?
===============================================================================

Composer es un gestor de dependencias para PHP. Inspirado en npm
(Node.js) y Bundler (Ruby), Composer resuelve:

- ¿Qué librerías necesita tu proyecto?
- ¿Qué versiones son compatibles entre sí?
- ¿Dónde descargarlas?
- ¿Cómo cargarlas automáticamente?

Composer trabaja con Packagist (https://packagist.org), el repositorio
público de paquetes PHP con más de 300,000 librerías disponibles.

Cuando ejecutas composer require vendor/paquete, Composer:
  1. Consulta Packagist para encontrar el paquete.
  2. Descarga la versión correcta en vendor/.
  3. Actualiza composer.json y composer.lock.
  4. Genera el archivo vendor/autoload.php.

Todo queda listo para usar con un simple require.

2. INSTALACIÓN DE COMPOSER
===============================================================================

Instalación local (en el proyecto):

```bash
php -r "copy('https://getcomposer.org/installer', 'composer-setup.php');"
php -r "if (hash_file('sha384', 'composer-setup.php') ===
  'e21205b207c3ff031906575712edab6f13eb0b361f2085f1f1237b7126d785e8
  26a4922925d1664a0e8747e9848e70e7') { echo 'Installer verified'; } else {
  echo 'Installer corrupt'; unlink('composer-setup.php'); } echo PHP_EOL;"
php composer-setup.php
php -r "unlink('composer-setup.php');"
```

Instalación global (recomendado):

```bash
# Mover composer.phar a /usr/local/bin
sudo mv composer.phar /usr/local/bin/composer

# Verificar instalación
composer --version
```

En Windows, descarga el instalador desde getcomposer.org. En macOS:

```bash
brew install composer
```

Verificar instalación:

```bash
composer --version
# Composer version 2.7.0 2024-09-10 12:00:00

composer --help
# Lista de comandos disponibles
```

===============================================================================
3. COMANDOS ESENCIALES
===============================================================================

Crear un nuevo proyecto:

```bash
# Inicializar composer.json interactivamente
composer init
# Responde: nombre del proyecto, descripción, autor, etc.
```

```bash
# Crear proyecto desde un paquete existente
composer create-project laravel/laravel mi-app
composer create-project symfony/skeleton mi-app-symfony
```

Instalar dependencias:

```bash
# Instalar dependencias de producción
composer require monolog/monolog
composer require guzzlehttp/guzzle

# Instalar dependencias de desarrollo
composer require --dev phpunit/phpunit
composer require --dev phpstan/phpstan

# Instalar una versión específica
composer require monolog/monolog:2.9.*
composer require vendor/paquete:^1.0
```

Actualizar dependencias:

```bash
# Actualizar todas las dependencias según composer.json
composer update

# Actualizar un paquete específico
composer update monolog/monolog
```

Instalar dependencias existentes (sin --dev):

```bash
# Sin composer.lock (instala según composer.json)
composer install

# Con composer.lock (instala versiones exactas)
composer install
```

Composer siempre respeta el archivo composer.lock si existe. En
producción, siempre debes usar composer install con el lock file
para garantizar versiones exactas.

===============================================================================
4. ESTRUCTURA DEL ARCHIVO COMPOSER.JSON
===============================================================================

El archivo composer.json es el corazón del proyecto. Define todo:

```json
{
    "name": "mi-app/proyecto",
    "description": "Aplicación profesional MVC",
    "type": "project",
    "license": "MIT",
    "authors": [
        {
            "name": "Tu Nombre",
            "email": "tu@email.com"
        }
    ],
    "require": {
        "php": ">=8.1",
        "monolog/monolog": "^3.0",
        "guzzlehttp/guzzle": "^7.0"
    },
    "require-dev": {
        "phpunit/phpunit": "^10.0",
        "phpstan/phpstan": "^1.0"
    },
    "autoload": {
        "psr-4": {
            "App\\": "src/"
        }
    },
    "autoload-dev": {
        "psr-4": {
            "Tests\\": "tests/"
        }
    },
    "scripts": {
        "test": "phpunit",
        "phpstan": "phpstan analyse src tests",
        "check": [
            "@test",
            "@phpstan"
        ]
    },
    "config": {
        "optimize-autoloader": true,
        "sort-packages": true
    },
    "minimum-stability": "stable",
    "prefer-stable": true
}
```

Secciones clave:

- name: vendor/proyecto, único en Packagist.
- require: dependencias de producción.
- require-dev: dependencias solo para desarrollo.
- autoload: configuración de carga automática.
- scripts: comandos alias (composer run test).
- config: opciones del autoloader y comportamiento.

El campo require define el corazón de las dependencias. Cada entrada
tiene un nombre de paquete y una restricción de versión.

===============================================================================
5. VERSIONADO SEMÁNTICO (SEMVER)
===============================================================================

Composer usa Semantic Versioning (SemVer): MAJOR.MINOR.PATCH.

  - MAJOR: Cambios incompatibles (1.0.0 → 2.0.0).
  - MINOR: Nuevas funcionalidades compatibles (1.1.0 → 1.2.0).
  - PATCH: Correcciones de bugs compatibles (1.1.0 → 1.1.1).

Operadores de versión:

  - ^1.2.3: >=1.2.3 <2.0.0 (compatible con MAJOR)
  - ~1.2.3: >=1.2.3 <1.3.0 (compatible con MINOR)
  - 1.2.*: >=1.2.0 <1.3.0 (cualquier PATCH)
  - >=1.0: 1.0 o superior
  - 1.0 - 2.0: entre 1.0 y 2.0 inclusive

Ejemplos prácticos:

```json
{
    "require": {
        "monolog/monolog": "^2.0",
        "guzzlehttp/guzzle": "~6.0",
        "phpunit/phpunit": "10.*",
        "symfony/console": ">=5.0 <6.0"
    }
}
```

El operador ^ (caret) es el más usado. Permite actualizaciones de
MINOR y PATCH pero no de MAJOR. Es la opción más segura.

El operador ~ (tilde) es más restrictivo. Solo permite PATCH.

===============================================================================
6. COMPOSER.LOCK Y POR QUÉ ES IMPORTANTE
===============================================================================

El archivo composer.lock registra las versiones EXACTAS de cada
dependencia instalada, incluyendo dependencias de dependencias.

Cuando tú instalas monolog, monolog depende de psr/log. Composer
resuelve todas las dependencias transitivas y las congela en el
lock file.

¿Por qué es importante?

  - Todos los miembros del equipo usan las MISMAS versiones.
  - Producción usa las MISMAS versiones que desarrollo.
  - Los tests pasan en cualquier máquina consistentemente.
  - El deploy es predecible y reproducible.

Reglas:
  - composer.lock se COMMITEA al repositorio (git).
  - En producción, siempre se ejecuta composer install (no update).
  - Cuando quieras actualizar, ejecuta composer update en desarrollo
    y prueba antes de desplegar.

```bash
# Desarrollo: actualizar dependencias
composer update

# Producción: instalar versiones exactas del lock
composer install --no-dev --optimize-autoloader
```

La bandera --no-dev excluye dependencias de desarrollo (PHPUnit,
PHPStan, etc.) en producción, reduciendo el tamaño del proyecto.

La bandera --optimize-autoloader genera un autoload optimizado
que mejora el rendimiento en producción.

7. COMANDOS AVANZADOS

```bash
# Mostrar dependencias instaladas
composer show

composer search logging

composer dump-autoload

# Validar composer.json
composer validate

composer remove phpunit/phpunit

composer run test
composer run phpstan
composer run check

# Ver estado de paquetes desactualizados
composer outdated
composer outdated --direct
```

El comando composer outdated es especialmente útil para saber qué
dependencias necesitan actualización en tus proyectos.

```bash
composer outdated --major-only
# Muestra solo paquetes con nueva versión MAJOR
```

8. BUSCAR LIBRERÍAS EN PACKAGIST
===============================================================================

Packagist (https://packagist.org) es el repositorio oficial de
paquetes PHP. Puedes buscar por:

- Nombre: monolog/monolog
- Palabra clave: logging, routing, template, security
- Categoría: framework, library, bundle

Librerías esenciales para proyectos PHP:

```
Logueo:      monolog/monolog
HTTP:        guzzlehttp/guzzle
Testing:     phpunit/phpunit
Análisis:    phpstan/phpstan
Debug:       symfony/var-dumper
Plantillas:  twig/twig
ORM:         doctrine/orm
Env:         vlucas/phpdotenv
UUID:        ramsey/uuid
CSV:         league/csv
Markdown:    erusev/parsedown
Email:       phpmailer/phpmailer
Caché:       symfony/cache
```

Cada paquete en Packagist muestra:
- Versiones disponibles.
- Dependencias del paquete.
- Estadísticas de descargas.
- Enlace al repositorio GitHub.
- Documentación y changelog.

Siempre revisa las estadísticas: un paquete con millones de
descargas es más confiable que uno con pocas.

===============================================================================
9. COMPOSER Y GITIGNORE
===============================================================================

La carpeta vendor/ NO debe subirse al repositorio. Contiene miles de
archivos que se regeneran con composer install.

.gitignore:

```gitignore
# Ignorar dependencias instaladas por Composer
/vendor/

# Ignorar archivos del gestor de dependencias
composer.lock   # NO! Este SÍ se commitea
```

Corrección: composer.lock NO va en .gitignore. El archivo que
no se commitea sería .env o las credenciales.

```gitignore
/vendor/
/node_modules/
.env
.env.local
.DS_Store
Thumbs.db
*.log
```

Cuando alguien clona tu proyecto:

```bash
git clone https://github.com/usuario/mi-app.git
cd mi-app
composer install      # Descarga vendor/ según composer.lock
cp .env.example .env  # Configurar variables de entorno
```

Este flujo es el estándar en la industria. Nadie sube vendor/ al
repositorio. Es mala práctica y puede causar conflictos.

10. COMPOSER EN PRODUCCIÓN
===============================================================================

El despliegue con Composer tiene consideraciones especiales:

```bash
# En el servidor de producción:
cd /var/www/mi-app

# Instalar solo dependencias de producción
composer install --no-dev --optimize-autoloader --no-interaction

# Opciones:
# --no-dev:       excluye require-dev
# --optimize-autoloader: genera autoload optimizado (más rápido)
# --no-interaction: evita preguntas en modo headless
# --prefer-dist:  descarga versiones empaquetadas (más rápido)
# --no-progress:  evita barras de progreso en logs
```

CI/CD ideal con Composer:

```yaml
# Ejemplo GitHub Actions
name: Deploy

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup PHP
        uses: shivammathur/setup-php@v2
        with:
          php-version: '8.2'

      - name: Install dependencies
        run: composer install --no-dev --optimize-autoloader

      - name: Deploy to server
        run: rsync -avz --exclude 'vendor/' ./ user@server:/var/www/app/
```

En producción, nunca ejecutes composer update. Siempre usa
composer install con el archivo composer.lock.

11. SCRIPTS DE COMPOSER
===============================================================================

Composer permite definir scripts que se ejecutan con comandos
personalizados.

```json
{
    "scripts": {
        "test": "phpunit --colors=always",
        "test:coverage": "phpunit --coverage-html coverage/",
        "phpstan": "phpstan analyse --level max src",
        "cs-fix": "php-cs-fixer fix src/",
        "lint": "parallel-lint src tests",
        "check": [
            "@lint",
            "@phpstan",
            "@test"
        ],
        "pre-install-cmd": [
            "echo 'Instalando dependencias...'"
        ],
        "post-install-cmd": [
            "php artisan migrate",
            "php artisan cache:clear"
        ],
        "post-update-cmd": [
            "php artisan optimize"
        ]
    }
}
```

Ejecutar:

```bash
composer run test
composer run phpstan
composer run check
```

Los hooks (pre-install-cmd, post-install-cmd, etc.) se ejecutan
automáticamente durante install/update. Son útiles para migraciones,
limpieza de caché, o generación de assets.

===============================================================================
12. COMPOSER CON PHP VERSIONES
===============================================================================

Especificar la versión de PHP requerida:

```json
{
    "require": {
        "php": ">=8.1",
        "ext-pdo": "*",
        "ext-mbstring": "*",
        "ext-json": "*"
    }
}
```

Extensiones de PHP como ext-pdo, ext-mbstring, ext-json pueden
requerirse explícitamente. Composer verificará que estén instaladas.

Para plataformas (PHP con configuraciones específicas):

```json
{
    "config": {
        "platform": {
            "php": "8.1.0",
            "ext-pdo": "1.0"
        }
    }
}
```

La sección "platform" engaña a Composer para que crea que el
servidor tiene cierta versión. Útil cuando desarrollas local con
PHP 8.2 pero el servidor tiene 8.1.

RESUMEN DEL CAPÍTULO
===============================================================================

- Composer es el gestor de dependencias estándar de PHP.
- Se instala local o globalmente. La instalación global es
  recomendada para acceso desde cualquier proyecto.
- composer init crea un proyecto nuevo.
- composer require agrega dependencias a composer.json.
- composer install descarga las versiones exactas del lock file.
- composer update actualiza las dependencias a las últimas versiones
  permitidas por las restricciones.
- composer.lock congela versiones exactas y debe committearse.
- Packagist es el repositorio público de paquetes PHP.
- vendor/ no debe ir en el repositorio (está en .gitignore).
- En producción usar composer install --no-dev --optimize-autoloader.
- Los scripts de Composer automatizan tareas comunes.
- SemVer (^, ~, *) controla las versiones permitidas.
- Las extensiones de PHP se declaran en require.

PREGUNTAS FRECUENTES
===============================================================================

❓ **¿Composer reemplaza a npm, pip o gem?**
   No, cada lenguaje tiene su gestor. PHP usa Composer, JavaScript usa
   npm/yarn, Python usa pip, Ruby usa gem. Son equivalentes pero para
   ecosistemas diferentes.

❓ **¿Qué pasa si dos paquetes requieren versiones incompatibles?**
   Composer mostrará un error de conflicto de dependencias y te
   indicará qué paquetes están en conflicto. Deberás buscar versiones
   compatibles o paquetes alternativos.

❓ **¿Puedo tener múltiples proyectos con diferentes versiones?**
   Sí, cada proyecto tiene su propio composer.json y vendor/.
   No hay conflicto entre proyectos porque cada uno gestiona sus
   dependencias de forma independiente.

❓ **¿Cómo actualizo Composer mismo?**
   composer self-update actualiza Composer a la última versión.
   Si lo instalaste globalmente con sudo, usa sudo composer self-update.

❓ **¿Qué es prefer-dist vs prefer-source?**
   --prefer-dist descarga versiones empaquetadas (zip) más ligeras.
   --prefer-source descarga el repositorio Git completo. Para
   producción usa --prefer-dist. Para desarrollo de paquetes usa
   --prefer-source.

❓ **¿Por qué mi instalación local de Composer dice "proc_open" error?**
   En algunos servidores compartidos, proc_open está deshabilitado
   por seguridad. Composer necesita funciones como proc_open,
   exec y system. Contacta a tu hosting para habilitarlas.

❓ **¿Debo versionar composer.lock en librerías?**
   Para librerías publicadas en Packagist, NO se incluye. Para
   aplicaciones, SIEMPRE se incluye. Las librerías definen
   restricciones en composer.json; la aplicación final resuelve
   y congela en el lock file.

🏗️ **Gestor de Dependencias Propio**

   Crea tu propio gestor de dependencias simplificado en PHP que
   emule el funcionamiento básico de Composer. Los requisitos:

   1. Archivo packages.json que sirva como repositorio (similar a

   5. Archivo lock.json que guarde las versiones exactas instaladas.
   6. Autoloader básico que cargue las clases de los paquetes.

   Llamada de ejemplo:
   ```
   php mini-composer install
   php mini-composer require monolog/monolog ^3.0
   php mini-composer update
   ```

   Este desafío te hará entender las complejidades que Composer
   resuelve automáticamente. ¡Nunca más darás por sentado a Composer!
