¡Saludos, constructor de sistemas robustos!

En el capítulo anterior viste `include`, que permite cargar archivos
de forma flexible. Ahora exploraremos su contraparte estricta:
`require` y `require_once`.

La diferencia fundamental es que `require` detiene la ejecución del
script si el archivo no existe, mientras que `include` solo emite
una advertencia.

Esta distinción es crucial: úsala para archivos que son esenciales
para el funcionamiento del sistema, como configuraciones de base de
datos, definiciones de funciones críticas y bootstrap de la aplicación.

===============================================================================
1. QUÉ ES REQUIRE
===============================================================================

`require` es una directiva de PHP similar a `include`, con una
diferencia crucial: si el archivo no existe o no puede ser cargado,
PHP lanza un **error fatal** y detiene la ejecución del script.

```php
<?php

// Si config.php no existe, el script se DETIENE aquí
require 'config.php';

// Este código solo se ejecuta si config.php se cargó correctamente
echo "Configuración cargada exitosamente\n";
```

Sintaxis:

```php
<?php

require 'archivo.php';
require('archivo.php'); // También válido, pero la primera es más común
require_once 'archivo.php';
require_once('archivo.php');
```

Comportamiento:

  1. Evalúa el archivo en el scope actual (igual que include).
  2. Si el archivo NO existe: **Fatal Error** → script termina.
  3. Si el archivo existe: lo carga y ejecuta normalmente.
  4. Al igual que include, puede usarse en cualquier expresión.

===============================================================================
2. INCLUDE VS REQUIRE
===============================================================================

La diferencia fundamental es el manejo de errores:

| Característica        | include              | require              |
|-----------------------|----------------------|----------------------|
| Archivo no existe     | Warning (continúa)   | Fatal Error (detiene)|
| Archivo existe        | Carga normal         | Carga normal         |
| Caso de uso típico    | Plantillas, vistas   | Bootstrap, config    |
| Tolerancia a fallos   | Alta (falla suave)   | Baja (falla dura)    |
| Severidad de error    | E_WARNING            | E_COMPILE_ERROR      |

```php
<?php

// include: el script continúa aunque falle
include 'templates/header.php';
echo "El header puede haber fallado, pero seguimos\n";

// require: el script se detiene si falla
require 'includes/database.php';
echo "Si database.php no existe, esto nunca se ejecuta\n";
```

¿Cuándo usar cada uno?

  • Usa `require` cuando el archivo es **esencial** para que el
    sistema funcione:
    - Conexión a base de datos.
    - Configuración de la aplicación.
    - Definiciones de funciones críticas.
    - Bootstrap del framework.

  • Usa `include` cuando el archivo es **prescindible** o parte
    de la presentación:
    - Plantillas HTML.
    - Fragmentos de interfaz.
    - Contenido opcional.
    - Widgets o módulos no críticos.

```php
<?php

// Archivos ESENCIALES → require
require __DIR__ . '/includes/config.php';
require __DIR__ . '/includes/database.php';
require __DIR__ . '/includes/autenticacion.php';

// Archivos de PRESENTACIÓN → include
include __DIR__ . '/templates/header.php';
include __DIR__ . '/templates/contenido.php';
include __DIR__ . '/templates/footer.php';
```

===============================================================================
3. REQUIRE ONCE
===============================================================================

`require_once` es la combinación de `require` e `include_once`:
requiere el archivo (error fatal si no existe) pero solo lo carga
una vez.

```php
<?php

// require_once: carga una sola vez, error fatal si no existe
require_once 'config.php';
require_once 'config.php'; // No hace nada (ya cargado)
require_once 'config.php'; // No hace nada

echo "Todo correcto\n";
```

¿Cuándo usar `require_once` vs `require`?

  • Usa `require_once` para:
    - Archivos con definiciones de clases.
    - Archivos con funciones globales.
    - Configuraciones que solo deben ejecutarse una vez.

  • Usa `require` para:
    - Archivos que quieres recargar en cada llamada.
    - Archivos que cambian dinámicamente.
    - Plantillas (aunque include es más común).

```php
<?php

// require_once: definiciones que solo cargan una vez
require_once __DIR__ . '/vendor/autoload.php';
require_once __DIR__ . '/includes/class.Usuario.php';
require_once __DIR__ . '/includes/class.Producto.php';

// require: archivos que queremos forzar a recargar
require __DIR__ . '/includes/dinamico_' . $modo . '.php';
```

===============================================================================
4. EL ERROR FATAL DE REQUIRE
===============================================================================

Cuando `require` falla, PHP produce un error fatal:

```
Fatal error: require(): Failed opening required 'archivo.php'
(include_path='.:/usr/share/php') in /ruta/script.php on line 5
```

Este error NO se puede capturar con un bloque try-catch normal
(porque es un E_COMPILE_ERROR). Sin embargo, puedes usar un
manejador de errores personalizado o... no dejar que falle:

```php
<?php

// ❌ Esto NO capturará el error fatal de require
try {
    require 'archivo_inexistente.php';
} catch (Throwable $e) {
    echo "Error capturado: " . $e->getMessage();
}
// En PHP 8+, esto SÍ captura el error porque Throwable atrapa
// \Error que es la clase de los errores fatales.
```

En PHP 8+, los errores fatales son lanzables (Throwable):

```php
<?php

// PHP 8+ — Los errores fatales son capturables
try {
    require 'archivo_inexistente.php';
} catch (\Throwable $e) {
    echo "Error capturado: " . $e->getMessage();
    // En PHP 8+, esto funciona
}
```

La mejor práctica es verificar antes de requerir:

```php
<?php

$archivo = __DIR__ . '/config.php';

if (!file_exists($archivo)) {
    die("Error crítico: No se encuentra $archivo");
}

if (!is_readable($archivo)) {
    die("Error crítico: No se puede leer $archivo");
}

require $archivo;
```

O crear una función helper:

```php
<?php

function requerir_seguro(string $archivo): void
{
    if (!file_exists($archivo)) {
        throw new RuntimeException(
            "Archivo requerido no encontrado: $archivo"
        );
    }

    if (!is_readable($archivo)) {
        throw new RuntimeException(
            "Archivo requerido sin permisos de lectura: $archivo"
        );
    }

    require $archivo;
}

// Uso
try {
    requerir_seguro(__DIR__ . '/config.php');
} catch (RuntimeException $e) {
    http_response_code(500);
    echo "Error interno del servidor";
    error_log($e->getMessage());
    exit;
}
```

===============================================================================
5. PATRÓN BOOTSTRAP CON REQUIRE
===============================================================================

El patrón más común con `require` es el bootstrap: un archivo central
que carga todas las dependencias esenciales de la aplicación.

```php
<?php

// === bootstrap.php ===
// Punto de entrada: carga todas las dependencias del sistema

// 1. Configuración base (SIEMPRE primero)
require_once __DIR__ . '/config/app.php';
require_once __DIR__ . '/config/database.php';

// 2. Funciones globales
require_once __DIR__ . '/includes/funciones.php';
require_once __DIR__ . '/includes/helpers.php';

// 3. Definir manejador de errores
set_error_handler(function(int $severity, string $message, string $file, int $line): bool {
    if (error_reporting() & $severity) {
        throw new \ErrorException($message, 0, $severity, $file, $line);
    }
    return true;
});

// 4. Configurar entorno
date_default_timezone_set(APP_TIMEZONE);
setlocale(LC_ALL, APP_LOCALE);

// 5. Conexión a base de datos
require_once __DIR__ . '/includes/database.php';

// 6. Enrutador
require_once __DIR__ . '/includes/router.php';

// 7. Cargar controladores dinámicamente (seguro)
$controladores_permitidos = ['inicio', 'usuarios', 'productos'];
$controlador = $_GET['c'] ?? 'inicio';

if (in_array($controlador, $controladores_permitidos, true)) {
    require __DIR__ . "/controladores/$controlador.php";
} else {
    require __DIR__ . '/controladores/error.php';
}

// === index.php ===
require_once __DIR__ . '/bootstrap.php';

// El resto del script asume que todo está configurado
echo "Aplicación iniciada correctamente\n";
```

===============================================================================
6. REQUIRE EN LA PRÁCTICA: SISTEMA DE MÓDULOS
===============================================================================

Un patrón común es tener módulos que requieren otros módulos:

```php
<?php

// === modulo/ventas.php ===
require_once __DIR__ . '/../includes/database.php';
require_once __DIR__ . '/../includes/seguridad.php';
require_once __DIR__ . '/../includes/notificaciones.php';

function crear_venta(array $datos): int
{
    $db = conectar_bd();
    $usuario = obtener_usuario_actual();

    $stmt = $db->prepare("INSERT INTO ventas (usuario_id, total, fecha) VALUES (?, ?, NOW())");
    $stmt->execute([$usuario['id'], $datos['total']]);

    $venta_id = (int) $db->lastInsertId();

    notificar_venta_realizada($venta_id, $datos);

    return $venta_id;
}

function obtener_ventas_recientes(int $limite = 10): array
{
    $db = conectar_bd();
    $stmt = $db->prepare("SELECT * FROM ventas ORDER BY fecha DESC LIMIT ?");
    $stmt->execute([$limite]);
    return $stmt->fetchAll(PDO::FETCH_ASSOC);
}
```

Cada módulo declara sus dependencias con `require_once`, garantizando
que todo lo necesario esté disponible sin importar el orden de carga.

===============================================================================
7. AUTOLOADING VS REQUIRE
===============================================================================

En aplicaciones modernas de PHP, el autoloading (carga automática de
clases) ha reemplazado en gran medida a `require` para cargar clases.
Sin embargo, `require` sigue siendo esencial para:

  • Archivos de configuración.
  • Archivos con funciones globales.
  • Bootstrap y punto de entrada.
  • Scripts legacy y sistemas heredados.

```php
<?php

// Carga manual con require (antes de autoload)
require_once 'class/Usuario.php';
require_once 'class/Producto.php';
require_once 'class/Pedido.php';
require_once 'class/Pago.php';
require_once 'class/Notificacion.php';

// Carga automática con spl_autoload_register (moderno)
spl_autoload_register(function(string $clase): void {
    $archivo = __DIR__ . '/class/' . str_replace('\\', '/', $clase) . '.php';
    if (file_exists($archivo)) {
        require_once $archivo;
    }
});

// Ahora PHP carga las clases automáticamente al usarlas
$usuario = new Usuario(); // require_once se llama automáticamente
```

El estándar PSR-4 y Composer han hecho el autoloading casi universal,
pero `require` sigue siendo el mecanismo subyacente.

===============================================================================
8. DIFERENCIAS SUTILES ENTRE INCLUDE Y REQUIRE
===============================================================================

Además del error fatal, hay diferencias en cómo se comportan en
ciertos contextos:

Contexto de inclusión condicional:

```php
<?php

// Con include, el script continúa si falla
if ($condicion) {
    include 'opcional.php'; // Si falla, continúa
}

// Con require, el script se detiene si falla
if ($condicion) {
    require 'esencial.php'; // Si falla, TODO se detiene
}
```

Valor de retorno:

```php
<?php

// include devuelve 1 en éxito, false en fallo
if (include 'archivo.php') {
    echo "Archivo cargado\n";
} else {
    echo "Archivo no cargado\n";
}

// require NO es adecuado para usar en expresiones condicionales
// porque si falla, el script termina antes de evaluar la condición
```

En funciones anónimas:

```php
<?php

// Con include, la función puede fallar silenciosamente
$cargar_vista = function(string $vista): void {
    include "templates/$vista.php";
};

// Con require, la función lanza error fatal si falla
$cargar_modulo = function(string $modulo): void {
    require "modulos/$modulo.php";
};
```

===============================================================================
9. BUENAS PRÁCTICAS CON REQUIRE
===============================================================================

  • Usa `require` solo para archivos críticos (config, bootstrap).
  • Prefiere `require_once` para definiciones de funciones/clases.
  • Verifica la existencia del archivo antes de requerir.
  • Usa `__DIR__` para rutas absolutas.
  • No mezcles require con include para el mismo tipo de archivos.
  • Centraliza los requires en un archivo bootstrap.
  • En proyectos modernos, usa Composer autoload para clases.

```php
<?php

// ✅ Buenas prácticas
require_once __DIR__ . '/../vendor/autoload.php';   // Composer
require_once __DIR__ . '/config/app.php';            // Config
require_once __DIR__ . '/includes/bootstrap.php';    // Bootstrap

// ❌ Malas prácticas
require 'config.php';               // Ruta relativa ambigua
require_once 'config.php';          // ¿Ya incluido? No se sabe
require $_GET['modulo'] . '.php';   // LFI vulnerability
```

10. EJEMPLO COMPLETO: APLICACIÓN MVC CON REQUIRE
===============================================================================

```php
<?php

// === Estructura ===
// app/
//   bootstrap.php
//   config/
//     app.php
//     database.php
//     routes.php
//   includes/
//     funciones.php
//     validacion.php
//     autenticacion.php
//   modulos/
//     usuarios/
//       controlador.php
//       modelo.php
//       vistas/
//     productos/
//       controlador.php
//       modelo.php
//   templates/
//     header.php
//     footer.php
//     layout.php
//   public/
//     index.php
//     .htaccess

// === public/index.php ===
<?php

declare(strict_types=1);

// Punto de entrada único (Front Controller)
require_once __DIR__ . '/../bootstrap.php';

// Enrutar
$ruta = parse_url($_SERVER['REQUEST_URI'], PHP_URL_PATH);
$ruta = rtrim($ruta, '/');

// Mapa de rutas
$rutas = [
    '/' => ['modulo' => 'inicio', 'accion' => 'index'],
    '/usuarios' => ['modulo' => 'usuarios', 'accion' => 'listar'],
    '/usuarios/nuevo' => ['modulo' => 'usuarios', 'accion' => 'nuevo'],
    '/productos' => ['modulo' => 'productos', 'accion' => 'listar'],
    '/productos/nuevo' => ['modulo' => 'productos', 'accion' => 'nuevo'],
];

if (isset($rutas[$ruta])) {
    $modulo = $rutas[$ruta]['modulo'];
    $accion = $rutas[$ruta]['accion'];

    // Cargar el controlador del módulo
    $archivo_controlador = __DIR__ . "/../modulos/$modulo/controlador.php";

    if (file_exists($archivo_controlador)) {
        require_once $archivo_controlador;

        $nombre_funcion = "ejecutar_$accion";
        if (function_exists($nombre_funcion)) {
            $nombre_funcion();
        } else {
            require_once __DIR__ . '/../modulos/error/controlador.php';
            ejecutar_no_encontrado();
        }
    } else {
        require_once __DIR__ . '/../modulos/error/controlador.php';
        ejecutar_no_encontrado();
    }
} else {
    require_once __DIR__ . '/../modulos/error/controlador.php';
    ejecutar_no_encontrado();
}

// === bootstrap.php ===
<?php

declare(strict_types=1);

// ==============================================
// BOOTSTRAP: Carga esencial del sistema
// ==============================================

// 1. Validar versión de PHP
if (PHP_VERSION_ID < 80100) {
    die("Se requiere PHP 8.1 o superior");
}

// 2. Configuración base
require_once __DIR__ . '/config/app.php';
require_once __DIR__ . '/config/database.php';

// 3. Funciones globales del sistema
require_once __DIR__ . '/includes/funciones.php';
require_once __DIR__ . '/includes/validacion.php';
require_once __DIR__ . '/includes/autenticacion.php';

// 4. Configuración de errores (desarrollo vs producción)
if (defined('APP_DEBUG') && APP_DEBUG) {
    error_reporting(E_ALL);
    ini_set('display_errors', '1');
} else {
    error_reporting(0);
    ini_set('display_errors', '0');
}

// 5. Zona horaria y locale
date_default_timezone_set(defined('APP_TIMEZONE') ? APP_TIMEZONE : 'UTC');
setlocale(LC_ALL, defined('APP_LOCALE') ? APP_LOCALE : 'es_MX.UTF-8');

// 6. Sesión
if (session_status() === PHP_SESSION_NONE) {
    session_start();
}

// 7. Conexión a base de datos
require_once __DIR__ . '/config/database.php'; // Define constantes DB
require_once __DIR__ . '/includes/database.php'; // Define funciones DB

// === config/app.php ===
<?php

define('APP_NAME', 'Mi Aplicación MVC');
define('APP_VERSION', '1.0.0');
define('APP_DEBUG', true);
define('APP_TIMEZONE', 'America/Mexico_City');
define('APP_LOCALE', 'es_MX.UTF-8');
define('APP_CHARSET', 'UTF-8');

// === modulos/usuarios/controlador.php ===
<?php

require_once __DIR__ . '/modelo.php'; // Dependencia del modelo

function ejecutar_listar(): void
{
    $usuarios = obtener_usuarios();

    // Cargar vista (include, no require)
    include __DIR__ . '/vistas/listar.php';
}

function ejecutar_nuevo(): void
{
    if ($_SERVER['REQUEST_METHOD'] === 'POST') {
        $datos = [
            'nombre' => $_POST['nombre'] ?? '',
            'email' => $_POST['email'] ?? '',
            'password' => $_POST['password'] ?? '',
        ];

        $errores = validar_usuario($datos);

        if (empty($errores)) {
            $id = crear_usuario($datos);
            redirigir("/usuarios?id=$id");
            return;
        }

        include __DIR__ . '/vistas/formulario.php';
    } else {
        include __DIR__ . '/vistas/formulario.php';
    }
}
```

===============================================================================
ERRORES COMUNES CON REQUIRE
===============================================================================

Error 1: Usar require donde debería ir include:

```php
<?php

// ❌ Un template no debería detener la aplicación
require 'templates/header.php';

// ✅ include es mejor para presentación
include 'templates/header.php';
```

Error 2: No usar __DIR__ en requires:

```php
<?php

// ❌ Depende del directorio de trabajo actual
require 'config.php';

// ✅ Ruta absoluta confiable
require __DIR__ . '/config.php';
```

Error 3: require_once con archivos que cambian:

```php
<?php

// ❌ Si el archivo cambia dinámicamente, require_once no lo recarga
foreach ($modulos as $modulo) {
    require_once "modulos/$modulo.php"; // Solo carga el primero
}

// ✅ Usar require para recargar
foreach ($modulos as $modulo) {
    require "modulos/$modulo.php";
}
```

===============================================================================
RESUMEN DEL CAPÍTULO
===============================================================================

- `require` es como `include` pero con error fatal si el archivo no existe.
- `require_once` combina error fatal con carga única.
- Usa `require` para archivos esenciales (config, bootstrap, database).
- Usa `include` para archivos prescindibles (templates, vistas).
- En PHP 8+, los errores fatales de require son capturables con Throwable.
- Verifica archivos con file_exists() antes de requerir.
- Centraliza los requires en un archivo bootstrap.
- El autoloading (PSR-4/Composer) complementa pero no reemplaza a require.

===============================================================================
PREGUNTAS FRECUENTES
===============================================================================

❓ **¿Puedo usar require y include juntos?**

Sí, es práctica común. require para bootstrap y archivos críticos,
include para plantillas y contenido opcional.

❓ **¿Require es más rápido que include?**

No hay diferencia significativa de rendimiento. La diferencia es
semántica y de manejo de errores.

❓ **¿Qué pasa si requiero un archivo con errores de sintaxis?**

PHP reporta el error de sintaxis en el archivo requerido y el
script se detiene, tanto con require como con include.

❓ **¿Require_once y include_once comparten el registro?**

Sí. Si usas include_once('archivo.php'), luego require_once('archivo.php')
tampoco lo cargará, y viceversa. El registro es único por archivo.

❓ **¿Puedo requerir archivos remotos?**

Técnicamente sí (con allow_url_include), pero está deshabilitado por
defecto y es una mala práctica de seguridad.

DESAFÍO DEL ARQUITECTO
===============================================================================

🏗️ **Framework MVC con Carga Modular**

Diseña un mini-framework MVC que use require y require_once para
gestionar la carga de módulos de forma inteligente:

  • `bootstrap.php` — carga esencial del framework: config, helpers,
    database, session, router. Usa require_once.

  • `cargar_modulo(string $nombre): bool` — carga dinámicamente un
    módulo completo: controlador, modelo, vistas, config. Usa
    require_once para cada componente. Si falta el controlador,
    lanza excepción. Si falta la vista, usa vista por defecto.

  • `cargar_helper(string $nombre): bool` — carga helpers bajo demanda
    (solo cuando se necesitan). Usa require_once con cache estático
    para evitar recargas.

  • `registrar_ruta(string $metodo, string $patron, callable $handler): void`
    — registra rutas con handlers. Los handlers se ejecutan con
    require (no require_once) para permitir regeneración.

  • `ejecutar_app(): void` — ejecuta el ciclo completo: bootstrap,
    carga de módulos, enrutamiento, renderizado.

  • `renderizar_vista(string $vista, array $datos = []): void` — carga
    una vista usando include. Si no existe, usa include de una vista
    de error.

  • Sistema de ejemplo con 3 módulos:
    - `usuarios`: CRUD completo con MySQL simulado.
    - `productos`: listado y detalle con archivo JSON.
    - `reportes`: genera reportes en HTML y PDF simulado.

Requisitos:
  - strict_types en todos los archivos.
  - El bootstrap debe verificar PHP version y extensiones.
  - La carga de módulos debe ser segura (lista blanca).
  - Las vistas deben usar include (falla suave si no existen).
  - Los controladores deben usar require (error fatal si no existen).
  - Documenta con PHPDoc todo el sistema.

Este desafío integra todos los conceptos de require/include en un
framework MVC funcional, demostrando la diferencia práctica entre
dependencias críticas (require) y contenido flexible (include).
