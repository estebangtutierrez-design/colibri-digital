===============================================================================
1. POR QUÉ ORGANIZAR EL CÓDIGO
===============================================================================

Un código bien organizado proporciona beneficios tangibles:

  • **Legibilidad**: cualquier desarrollador entiende la estructura.
  • **Mantenibilidad**: encontrar y modificar código es rápido.
  • **Escalabilidad**: añadir funcionalidades no rompe lo existente.
  • **Reusabilidad**: los componentes se pueden reutilizar fácilmente.
  • **Testabilidad**: el código organizado es más fácil de probar.
  • **Colaboración**: múltiples desarrolladores trabajan sin pisarse.

¿Qué pasa cuando NO organizamos el código?

```
proyecto/
  index.php           # 3000 líneas de todo mezclado
  funciones.php       # 2000 líneas de funciones sin relación
  helpers.php         # "Utilidades varias" — cajón de sastre
  config.php          # Config + consultas DB + HTML
```

Este enfoque (o la falta de él) se conoce como "código espagueti"
y es la pesadilla de todo desarrollador profesional.

Los principios fundamentales de organización son:

  1. **Separación de Preocupaciones** (SoC): cada módulo tiene una
     responsabilidad única y bien definida.
  2. **Alta cohesión**: los elementos dentro de un módulo están
     fuertemente relacionados.
  3. **Bajo acoplamiento**: los módulos dependen mínimamente entre sí.
  4. **Principio de Responsabilidad Única** (SRP): cada archivo o
     función hace una sola cosa.

3. SEPARACIÓN DE PREOCUPACIONES (SoC)
===============================================================================

La Separación de Preocupaciones divide el código en capas o módulos,
cada uno con una responsabilidad específica:

Capas típicas en una aplicación web:

  • **Presentación (Vista)**: HTML, CSS, templates.
    → Archivos en `templates/`.

  • **Lógica de negocio (Modelo)**: reglas de dominio, cálculos.
    → Archivos en `modules/*/modelo.php` o `src/*.php`.

  • **Control**: recibe peticiones, orquesta la respuesta.
    → Archivos en `modules/*/controlador.php`.

  • **Acceso a datos (Persistencia)**: consultas a BD, APIs.
    → Archivos en `src/Database/`.

  • **Configuración**: constantes, parámetros del entorno.
    → Archivos en `config/`.

```php
<?php

// ❌ TODO MEZCLADO (código espagueti)
<?php
$db = new PDO("mysql:host=localhost", "root", "");
$usuarios = $db->query("SELECT * FROM usuarios")->fetchAll();
?>
<!DOCTYPE html>
<html>
<head><title>Usuarios</title></head>
<body>
<table>
<?php foreach ($usuarios as $u): ?>
<tr><td><?= $u['nombre'] ?></td></tr>
<?php endforeach; ?>
</table>
</body>
</html>

// ✅ SEPARADO (capas bien definidas)

// config/database.php
define('DB_HOST', 'localhost');
define('DB_USER', 'root');
define('DB_PASS', '');

// includes/database.php
function conectar_bd(): PDO
{
    return new PDO(
        "mysql:host=" . DB_HOST,
        DB_USER,
        DB_PASS
    );
}

// modules/usuarios/modelo.php
function obtener_usuarios(): array
{
    $db = conectar_bd();
    $stmt = $db->query("SELECT * FROM usuarios");
    return $stmt->fetchAll(PDO::FETCH_ASSOC);
}

// modules/usuarios/controlador.php
function listar_usuarios(): void
{
    $usuarios = obtener_usuarios();
    include __DIR__ . '/vistas/listar.php';
}

// modules/usuarios/vistas/listar.php
<table>
<?php foreach ($usuarios as $u): ?>
<tr><td><?= htmlspecialchars($u['nombre']) ?></td></tr>
<?php endforeach; ?>
</table>
```

===============================================================================
4. ACOPLAMIENTO Y COHESIÓN
===============================================================================

**Alta cohesión**: los elementos dentro de un módulo están relacionados.

✅ ALTA cohesión (correcto):

```
validacion/
  ValidarEmail.php     # Validación de email
  ValidarRFC.php       # Validación de RFC
  ValidarEdad.php      # Validación de edad
  Validador.php        # Orquestador de validaciones
```

❌ BAJA cohesión (incorrecto):

```
utils/
  ValidarEmail.php     # Validación
  FormatearFecha.php   # Formateo
  EnviarCorreo.php     # Envío de emails
  CalcularIVA.php      # Cálculos fiscales
  GenerarPDF.php       # Generación de PDF
  LeerArchivo.php      # Lectura de archivos
```

**Bajo acoplamiento**: los módulos dependen mínimamente entre sí.

```php
<?php

// ❌ ALTO acoplamiento: UsuarioService depende directamente de MySQL
class UsuarioService
{
    private PDO $db;

    public function __construct()
    {
        // Dependencia directa de una implementación concreta
        $this->db = new PDO(
            "mysql:host=localhost;dbname=mi_app",
            "root",
            ""
        );
    }

    public function obtener(int $id): ?array
    {
        $stmt = $this->db->prepare("SELECT * FROM usuarios WHERE id = ?");
        $stmt->execute([$id]);
        return $stmt->fetch(PDO::FETCH_ASSOC) ?: null;
    }
}

// ✅ BAJO acoplamiento: UsuarioService recibe la dependencia
class UsuarioService
{
    public function __construct(
        private PDO $db  // Inyección de dependencia
    ) {}

    public function obtener(int $id): ?array
    {
        $stmt = $this->db->prepare("SELECT * FROM usuarios WHERE id = ?");
        $stmt->execute([$id]);
        return $stmt->fetch(PDO::FETCH_ASSOC) ?: null;
    }
}

// Más fácil de probar y mantener
```

5. NAMESPACES (ESPACIOS DE NOMBRES)
===============================================================================

Los namespaces (PHP 5.3+) son la herramienta fundamental para organizar
código en PHP moderno. Evitan colisiones de nombres y agrupan clases
relacionadas.

```php
<?php

// src/Database/Conexion.php
namespace LucasEga\Database;

class Conexion
{
    public static function obtener(): PDO
    {
        // ...
    }
}

// src/Auth/Login.php
namespace LucasEga\Auth;

use LucasEga\Database\Conexion;

class Login
{
    public function autenticar(string $email, string $password): bool
    {
        $db = Conexion::obtener();
        // ...
        return true;
    }
}
```

Los namespaces siguen la estructura de directorios para facilitar
el autoloading (PSR-4):

```
src/
  LucasEga/
    Database/
      Conexion.php       → namespace LucasEga\Database;
    Auth/
      Login.php          → namespace LucasEga\Auth;
      Registro.php       → namespace LucasEga\Auth;
```

Con Composer y PSR-4 autoload:

```json
{
    "autoload": {
        "psr-4": {
            "LucasEga\\": "src/LucasEga/"
        }
    }
}
```

Luego puedes usar las clases sin require manual:

```php
<?php

require_once __DIR__ . '/vendor/autoload.php';

use LucasEga\Auth\Login;
use LucasEga\Database\Conexion;

$login = new Login();
$login->autenticar('user@mail.com', 'secret');
```

6. PATRONES DE ORGANIZACIÓN MODULAR
===============================================================================

Patrón 1: Módulos Funcionales (por característica)

```
modules/
  usuarios/          # Todo lo relacionado con usuarios
    controlador.php
    modelo.php
    vistas/
    config.php
    helpers.php
  productos/         # Todo lo relacionado con productos
    controlador.php
    modelo.php
    vistas/
    config.php
```

Ventaja: alta cohesión, fácil de entender.
Desventaja: puede duplicar lógica entre módulos.

Patrón 2: Capas Técnicas (por tipo de componente)

```
src/
  Controllers/
    UsuarioController.php
    ProductoController.php
  Models/
    Usuario.php
    Producto.php
  Views/
    usuario/
    producto/
  Services/
    UsuarioService.php
    ProductoService.php
  Repositories/
    UsuarioRepository.php
    ProductoRepository.php
```

Ventaja: separación técnica clara, fácil estandarización.
Desventaja: dispersa lógica de una característica.

Patrón 3: Híbrido (módulos + capas) — el más usado en la práctica

```
src/
  Modules/
    Usuarios/
      Controllers/
        AuthController.php
        PerfilController.php
      Models/
        Usuario.php
      Services/
        UsuarioService.php
      Repositories/
        UsuarioRepository.php
      Views/
        auth/
        perfil/
    Productos/
      Controllers/
        CatalogoController.php
      Models/
        Producto.php
      Services/
        ProductoService.php
```

Ventaja: equilibrio entre cohesión de características y estándares
técnicos.

7. CONVENCIONES DE NOMENCLATURA
===============================================================================

Consistencia en nombres = legibilidad inmediata.

Archivos:

  • Clases: `PascalCase.php` → `class UsuarioController`
  • Funciones: `snake_case.php` → `function obtener_usuario()`
  • Configuración: `snake_case.php` → `database.php`
  • Plantillas: `snake_case.php` → `listar_usuarios.php`

Variables y funciones:

  • Variables: `$snake_case` (PSR) o `$camelCase` (según proyecto).
  • Funciones: `snake_case()` (recomendado por la comunidad PHP).
  • Constantes: `UPPER_SNAKE_CASE`.
  • Clases e interfaces: `PascalCase`.
  • Métodos: `camelCase()` (en POO) o `snake_case()` (según estándar).

```php
<?php

// Constantes
define('MAX_INTENTOS_LOGIN', 3);
const VERSION_APP = '1.0.0';

// Variables
$nombre_usuario = 'Ana';
$edad = 25;

// Funciones
function obtener_usuario_por_email(string $email): ?array
{
    // ...
}

// Clases
class UsuarioController
{
    public function obtenerPerfil(int $id): array
    {
        // ...
    }

    public function actualizarPerfil(int $id, array $datos): bool
    {
        // ...
    }
}
```

8. PRINCIPIO DRY (DON'T REPEAT YOURSELF)
===============================================================================

El principio DRY establece que cada conocimiento debe tener una
representación única, no ambigua y autoritativa dentro del sistema.

```php
<?php

// ❌ VIOLACIÓN DE DRY: validación duplicada

// En usuarios/controlador.php
function crear_usuario(array $datos): array
{
    $errores = [];
    if (empty($datos['nombre'])) $errores[] = 'Nombre requerido';
    if (empty($datos['email'])) $errores[] = 'Email requerido';
    if (empty($datos['password'])) $errores[] = 'Password requerido';
    if (strlen($datos['password']) < 8) $errores[] = 'Password muy corto';
    return $errores;
}

// En productos/controlador.php (misma validación, copiada)
function crear_producto(array $datos): array
{
    $errores = [];
    if (empty($datos['nombre'])) $errores[] = 'Nombre requerido';
    if (empty($datos['precio'])) $errores[] = 'Precio requerido';
    if ($datos['precio'] <= 0) $errores[] = 'Precio debe ser positivo';
    return $errores;
}

// ✅ DRY: validación centralizada

// includes/validacion.php
function validar_campo_requerido(string $campo, string $valor): ?string
{
    if (empty(trim($valor))) {
        return "El campo '$campo' es requerido";
    }
    return null;
}

function validar_longitud_minima(string $valor, int $min, string $campo): ?string
{
    if (strlen($valor) < $min) {
        return "'$campo' debe tener al menos $min caracteres";
    }
    return null;
}

function validar_positivo(float $valor, string $campo): ?string
{
    if ($valor <= 0) {
        return "'$campo' debe ser un número positivo";
    }
    return null;
}

function validar_con_reglas(array $datos, array $reglas): array
{
    $errores = [];
    foreach ($reglas as $campo => $validadores) {
        $valor = $datos[$campo] ?? '';
        foreach ($validadores as $validador) {
            $error = $validador($valor);
            if ($error !== null) {
                $errores[$campo][] = $error;
            }
        }
    }
    return $errores;
}

// Uso en cualquier módulo
$reglas = [
    'nombre' => [
        fn($v) => validar_campo_requerido('nombre', $v),
    ],
    'email' => [
        fn($v) => validar_campo_requerido('email', $v),
        fn($v) => filter_var($v, FILTER_VALIDATE_EMAIL) ? null : 'Email inválido',
    ],
    'password' => [
        fn($v) => validar_campo_requerido('password', $v),
        fn($v) => validar_longitud_minima($v, 8, 'password'),
    ],
];
```

9. ERRORES COMUNES DE ORGANIZACIÓN
===============================================================================

Error 1: El archivo "todo menos el fregadero":

```php
<?php

// ❌ "helpers.php" de 2000 líneas con funciones no relacionadas
function validar_email($email) { ... }
function formatear_fecha($fecha) { ... }
function conectar_bd() { ... }
function enviar_correo($para, $asunto) { ... }
function generar_pdf($html) { ... }

// ✅ Separar en archivos temáticos
// includes/validacion.php
// includes/formateo.php
// includes/database.php
// includes/correo.php
// includes/pdf.php
```

Error 2: Dependencias circulares:

```php
<?php

// a.php
require_once 'b.php';
function funcion_a() { funcion_b(); }

// b.php
require_once 'a.php';
function funcion_b() { funcion_a(); }
// Esto causa un error fatal por dependencia circular
```

Error 3: Mezclar lógica con presentación:

```php
<?php

// ❌ En el mismo archivo: consultas SQL + HTML
$db = new PDO(...);
$stmt = $db->query("SELECT * FROM ventas");
$total = 0;
while ($venta = $stmt->fetch()) {
    $total += $venta['monto'];
}
?>
<h2>Total de ventas: $<?= $total ?></h2>

// ✅ Separar responsabilidades
// modelo.php
function calcular_total_ventas(): float { ... }
// vista.php
<h2>Total de ventas: $<?= calcular_total_ventas() ?></h2>
```

Error 4: No usar namespaces en proyectos grandes:

```php
<?php

// Sin namespaces: conflicto de nombres inevitable
class Usuario {}
class Usuario {} // ❌ Fatal error: Cannot redeclare class

// Con namespaces:
namespace ModuloVentas;
class Usuario {}

namespace ModuloCompras;
class Usuario {} // ✅ Perfectamente válido
```

10. EJEMPLO COMPLETO: APLICACIÓN MODULAR
===============================================================================

A continuación, un ejemplo completo de cómo organizar una aplicación
siguiendo los principios de este capítulo:

```php
<?php

// === ESTRUCTURA ===
// app/
//   config/
//     app.php
//     database.php
//   includes/
//     bootstrap.php
//     database.php
//     validacion.php
//     respuesta.php
//   modules/
//     auth/
//       controlador.php
//       modelo.php
//       vistas/
//         login.php
//         registro.php
//     dashboard/
//       controlador.php
//       modelo.php
//       vistas/
//         inicio.php
//     usuarios/
//       controlador.php
//       modelo.php
//       vistas/
//         listar.php
//         editar.php
//   templates/
//     layout.php
//     header.php
//     footer.php
//   public/
//     index.php
//     .htaccess
//   vendor/
//     autoload.php

// === public/index.php ===
<?php

declare(strict_types=1);

require_once __DIR__ . '/../includes/bootstrap.php';

$modulo = $_GET['modulo'] ?? 'dashboard';
$accion = $_GET['accion'] ?? 'index';

$modulos_permitidos = ['auth', 'dashboard', 'usuarios', 'error'];
$modulo = in_array($modulo, $modulos_permitidos, true) ? $modulo : 'error';

$archivo_controlador = __DIR__ . "/../modules/$modulo/controlador.php";

if (!file_exists($archivo_controlador)) {
    $modulo = 'error';
    $archivo_controlador = __DIR__ . "/../modules/error/controlador.php";
}

require_once $archivo_controlador;

$nombre_funcion = "ejecutar_$accion";

if (!function_exists($nombre_funcion)) {
    $modulo = 'error';
    $accion = 'no_encontrado';
    require_once __DIR__ . "/../modules/error/controlador.php";
    $nombre_funcion = "ejecutar_no_encontrado";
}

$nombre_funcion();

// === includes/bootstrap.php ===
<?php

declare(strict_types=1);

require_once __DIR__ . '/../config/app.php';
require_once __DIR__ . '/../config/database.php';
require_once __DIR__ . '/database.php';
require_once __DIR__ . '/validacion.php';
require_once __DIR__ . '/respuesta.php';

date_default_timezone_set(APP_TIMEZONE);

if (session_status() === PHP_SESSION_NONE) {
    session_start();
}

// === modules/usuarios/controlador.php ===
<?php

declare(strict_types=1);

require_once __DIR__ . '/modelo.php';

function ejecutar_listar(): void
{
    $pagina = max(1, (int) ($_GET['pagina'] ?? 1));
    $limite = 10;
    $offset = ($pagina - 1) * $limite;

    $usuarios = obtener_usuarios_paginados($limite, $offset);
    $total = contar_usuarios();
    $total_paginas = (int) ceil($total / $limite);

    renderizar_vista('usuarios/listar', [
        'usuarios' => $usuarios,
        'pagina_actual' => $pagina,
        'total_paginas' => $total_paginas,
    ]);
}

function ejecutar_editar(): void
{
    $id = (int) ($_GET['id'] ?? 0);

    if ($_SERVER['REQUEST_METHOD'] === 'POST') {
        $datos = [
            'nombre' => $_POST['nombre'] ?? '',
            'email' => $_POST['email'] ?? '',
        ];

        $reglas = [
            'nombre' => [fn($v) => validar_requerido('Nombre', $v)],
            'email' => [
                fn($v) => validar_requerido('Email', $v),
                fn($v) => validar_email($v),
            ],
        ];

        $errores = validar_con_reglas($datos, $reglas);

        if (empty($errores)) {
            actualizar_usuario($id, $datos);
            redirigir('index.php?modulo=usuarios&accion=listar');
            return;
        }

        renderizar_vista('usuarios/editar', [
            'usuario' => $datos,
            'errores' => $errores,
        ]);
    } else {
        $usuario = obtener_usuario_por_id($id);
        if (!$usuario) {
            renderizar_vista('error/no_encontrado', ['mensaje' => 'Usuario no encontrado']);
            return;
        }
        renderizar_vista('usuarios/editar', ['usuario' => $usuario, 'errores' => []]);
    }
}
```

===============================================================================
BUENAS PRÁCTICAS DE ORGANIZACIÓN
===============================================================================

  • Un archivo, una responsabilidad (SRP a nivel de archivo).
  • Nombres de archivos que reflejen su contenido.
  • Un módulo por carpeta con todo lo que necesita.
  • Usa namespaces desde el principio (incluso en proyectos pequeños).
  • Mantén las funciones pequeñas (< 30 líneas cada una).
  • No mezcles lógica con presentación.
  • Centraliza la configuración (no hagas hardcode).
  • Usa require_once para bootstrap, include para vistas.
  • Sigue los estándares PSR para nomenclatura.
  • Documenta la estructura en un README o ARCHITECTURE.md.

===============================================================================
RESUMEN DEL CAPÍTULO
===============================================================================

- La organización del código es fundamental para la mantenibilidad
  y escalabilidad de cualquier proyecto.
- La Separación de Preocupaciones (SoC) divide el código en capas
  con responsabilidades específicas.
- La estructura de directorios debe reflejar la arquitectura del
  proyecto, agrupando por módulos o por capas técnicas.
- Alta cohesión (elementos relacionados juntos) y bajo acoplamiento
  (módulos independientes) son ideales de diseño.
- Los namespaces (PSR-4) evitan colisiones de nombres y facilitan
  el autoloading.
- DRY (Don't Repeat Yourself) centraliza el conocimiento compartido.
- Las convenciones de nomenclatura consistentes mejoran la legibilidad.
- Un archivo bootstrap centralizado gestiona las dependencias iniciales.

===============================================================================
PREGUNTAS FRECUENTES
===============================================================================

❓ **¿Cuántos archivos debería tener un módulo?**

Los que sean necesarios, pero típicamente: controlador, modelo,
configuración y vistas. Si el módulo es complejo, puedes dividirlo
en submódulos.

❓ **¿Cuándo usar namespaces vs prefijos en nombres?**

Siempre prefiere namespaces. Son más limpios, permiten autoloading
y son el estándar moderno. Los prefijos (Ej: `class Usuario_Admin`)
son obsoletos.

❓ **¿Qué hago con funciones que uso en varios módulos?**

Centralízalas en `includes/` o `src/Utils/`. Son helpers globales.
Si son muchas, agrúpalas por tema (validación, formateo, etc.).

❓ **¿Debo usar un framework para organizar mi código?**

No es necesario. Los principios de organización son independientes
del framework. Pero un framework como Laravel o Symfony ya viene
con una estructura organizada que puedes seguir.

❓ **¿Cómo manejo la configuración entre entornos?**

Usa archivos .env (con librería vlucas/phpdotenv) o archivos de
configuración por entorno: `config.prod.php`, `config.dev.php`.

🏗️ **Diseño de una Arquitectura Modular Profesional**

Diseña la arquitectura completa para una aplicación de comercio
electrónico que debe gestionar: productos, carrito, pedidos, pagos,
usuarios, envíos, reseñas, notificaciones y reportes.

Tu entrega debe incluir:

  1. **Estructura completa de directorios** con todos los módulos,
     capas y archivos necesarios. Incluye archivos de configuración,
     bootstrap, helpers, templates, assets, tests, etc.

  2. **Diagrama de dependencias entre módulos** (puede ser en texto):
     - ¿Qué módulos dependen de otros?
     - ¿Qué servicios son compartidos?
     - ¿Dónde está el punto de entrada?

  3. **Convenciones de nomenclatura** documentadas para:
     - Archivos, clases, funciones, variables, constantes, métodos.

  4. **Implementación de 3 módulos completos**:
     - `auth/`: registro, login, logout, recuperación de contraseña,
       verificación de email. Con validación, sesión y middleware.
     - `productos/`: CRUD, categorías, búsqueda, filtros, imágenes,
       inventario. Con paginación y caché estática.
     - `carrito/`: agregar/quitar items, calcular total, aplicar
       descuentos, convertir a pedido. Con sesión como storage.

  5. **Un archivo `bootstrap.php`** que cargue todo el sistema en
     el orden correcto, con verificación de requisitos.

  6. **Un `public/index.php`** que sea el Front Controller que
     enrute las peticiones a los módulos correspondientes.

  7. **Documentación de la arquitectura** explicando:
     - Por qué elegiste esa estructura.
     - Cómo se aplica SoC, alta cohesión y bajo acoplamiento.
     - Cómo escalaría la aplicación a 50 módulos.

Requisitos:
  - strict_types en todos los archivos PHP.
  - Usa require_once para bootstrap, include para vistas.
  - Autoloading PSR-4 para clases.
  - Validación centralizada y reutilizable.
  - Manejo de errores con excepciones específicas.
  - Sin dependencias externas (solo PHP estándar).

Este desafío integra TODO el Módulo III en un diseño arquitectónico
completo que demuestra maestría en organización del código.
