CONTENIDO DEL CAPÍTULO
===============================================================================

1. PULL REQUESTS

Solicitud para integrar cambios de una rama a otra. GitHub/GitLab/Bitbucket.

Título descriptivo, descripción con contexto, referencia a issues.

Template de PR para estandarizar: qué, por qué, cómo, evidencias.



2. CODE REVIEW EFECTIVO

Revisar 200-400 líneas por hora. Enfocarse en lógica, no estilo.

Comentarios constructivos: sugerencias, preguntas, no críticas.

Checklist: seguridad, rendimiento, pruebas, documentación.



3. BUENAS PRÁCTICAS

PRs pequeños y enfocados (< 400 líneas). Aprobar rápido o rechazar.

PR Draft para trabajo en progreso. Asignar reviewers explícitamente.

Responder comentarios. Merge propio después de aprobación.
