CONTENIDO DEL CAPÍTULO
===============================================================================

1. SLIDING WINDOW REVIEW

Revisar en múltiples pasadas: arquitectura, lógica, estilo, tests.

Cada pasada con enfoque diferente. Reduce fatiga de revisión.

Acordar tiempos máximos de revisión (ej: 24h).



2. AUTOMATIZACIÓN DE REVIEW

Code owners automáticos. Linters y formatters en CI. Análisis estático

(SonarQube, CodeQL). Coverage gates. Auto-merge si pasa todo.

Danger.js para reglas automatizadas de PR.



3. CULTURA DE REVIEW

Revisar código, no a la persona. Hacer preguntas, no acusaciones.

Celebrar código bueno. Aprender de revisiones ajenas. Rotar revisores.

Métrica: time-to-review, no líneas revisadas por hora.

EJEMPLO DE CÓDIGO
----------------------------------------

# Danger.js config
module.exports = async () => {
  const hasTests = danger.git.modified_files
    .filter(f => f.includes('.test.')).length > 0
  
  if (!hasTests) {
    warn('No se encontraron tests en este PR')
  }
  
  if (danger.github.pr.body.length < 10) {
    fail('La descripción del PR es demasiado corta')
  }
  
  if (danger.github.pr.additions > 400) {
    warn('PR muy grande. Considera dividirlo.')
  }
}
