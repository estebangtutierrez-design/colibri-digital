===============================================================================
PARTE 1: ¿QUÉ ES EL CONTROL DE VERSIONES?
===============================================================================

El control de versiones es un sistema que registra los cambios realizados en un archivo o conjunto de archivos a lo largo del tiempo.

───────────────────────────────────────────────────────────
DEFINICIÓN
───────────────────────────────────────────────────────────

El control de versiones (version control) es la práctica de gestionar cambios en el código fuente de un proyecto, permitiendo:

Rastrear cambios: Saber qué cambió, quién lo cambió y cuándo.

Volver atrás: Revertir a versiones anteriores.

Colaborar: Trabajar con otros sin conflictos.

Experimentar: Probar ideas sin romper el proyecto principal.

───────────────────────────────────────────────────────────
¿POR QUÉ ES IMPORTANTE?
───────────────────────────────────────────────────────────

Historia: Tienes un registro completo de todos los cambios.

Responsabilidad: Sabes quién hizo cada cambio.

Seguridad: Puedes recuperar versiones perdidas.

Colaboración: Varias personas pueden trabajar en el mismo proyecto.

Experimentos: Puedes probar cosas sin miedo a romper el proyecto.

===============================================================================
PARTE 2: EL PROBLEMA QUE RESUELVE EL CONTROL DE VERSIONES
===============================================================================

───────────────────────────────────────────────────────────
SIN CONTROL DE VERSIONES
───────────────────────────────────────────────────────────

Archivos duplicados: "proyecto_final.txt", "proyecto_final_v2.txt", "proyecto_final_definitivo.txt".

Pérdida de cambios: No sabes qué cambió entre versiones.

Conflictos: Dos personas modifican el mismo archivo.

Sin historia: No sabes quién hizo qué ni cuándo.

Riesgo: No puedes volver atrás si algo falla.

───────────────────────────────────────────────────────────
CON CONTROL DE VERSIONES
───────────────────────────────────────────────────────────

Un solo archivo: Siempre tienes la versión más reciente.

Historia clara: Puedes ver todos los cambios.

Sin conflictos: Los cambios se integran automáticamente.

Responsabilidad: Sabes quién hizo cada cambio.

Seguridad: Puedes volver a cualquier versión.

===============================================================================
PARTE 3: ¿QUÉ ES GIT?
===============================================================================

Git es un sistema de control de versiones distribuido creado por Linus Torvalds en 2005.

───────────────────────────────────────────────────────────
CARACTERÍSTICAS DE GIT
───────────────────────────────────────────────────────────

DISTRIBUIDO: Cada desarrollador tiene una copia completa del repositorio.

RÁPIDO: Las operaciones son muy rápidas.

SEGURO: Usa criptografía para asegurar la integridad de los datos.

FLEXIBLE: Soporta múltiples flujos de trabajo.

GRATUITO: Es software libre y abierto.

───────────────────────────────────────────────────────────
¿QUÉ PROBLEMA RESUELVE GIT?
───────────────────────────────────────────────────────────

Trabajo en equipo: Varias personas pueden trabajar en el mismo proyecto.

Historial: Puedes ver todos los cambios.

Ramas (branches): Puedes trabajar en nuevas funcionalidades sin afectar el proyecto principal.

Revertir: Puedes volver a versiones anteriores.

Colaboración: Puedes compartir tu código con otros.

───────────────────────────────────────────────────────────
¿QUÉ ES GITHUB?
───────────────────────────────────────────────────────────

GitHub es una plataforma en línea para alojar repositorios Git.

Es como un "Dropbox para programadores".

Almacenamiento: Guarda tu código en la nube.

Colaboración: Permite que otros vean y contribuyan a tu código.

Visibilidad: Puedes mostrar tu trabajo al mundo.

Comunidad: Puedes aprender de otros proyectos.

===============================================================================
PARTE 4: ¿CÓMO TRABAJAN VARIOS PROGRAMADORES JUNTOS?
===============================================================================

El control de versiones permite que varios programadores trabajen en el mismo proyecto sin pisarse.

───────────────────────────────────────────────────────────
EL FLUJO DE TRABAJO PROFESIONAL
───────────────────────────────────────────────────────────

Clonar: Cada programador descarga una copia del repositorio.

Trabajar: Cada uno trabaja en su propia copia.

Guardar: Cada uno guarda sus cambios localmente (commit).

Subir: Cada uno sube sus cambios al repositorio central (push).

Bajar: Cada uno descarga los cambios de otros (pull).

───────────────────────────────────────────────────────────
EJEMPLO: DOS PROGRAMADORES
───────────────────────────────────────────────────────────

text
+----------------------------------------------------------+
|  REPOSITORIO CENTRAL (GitHub)                             |
|  +--------------------------------------------------+   |
|  |  Código principal                                |   |
|  +--------------------------------------------------+   |
|         ↑                                      ↑          |
|     (push)                                  (push)        |
|         |                                      |          |
|  +------+------+                    +----------+------+   |
|  | PROGRAMADOR 1|                    | PROGRAMADOR 2 |   |
|  |  Trabaja en  |                    |   Trabaja en  |   |
|  |  su copia    |                    |   su copia    |   |
|  +--------------+                    +--------------+   |
───────────────────────────────────────────────────────────
¿QUÉ PASA SI HAY CONFLICTOS?
───────────────────────────────────────────────────────────

A veces dos programadores modifican el mismo archivo.
Git detecta los conflictos y pide que los resuelvas.

Ejemplo:

Ana modifica la línea 10.

Carlos modifica la línea 10.

Git dice: "Conflicto. Decide qué versión usar."

Es como cuando dos personas escriben en el mismo cuaderno en la misma línea.
Tienes que decidir qué texto queda.

PARTE 5: LA FILOSOFÍA DEL CONTROL DE VERSIONES
===============================================================================

El control de versiones no es solo una herramienta.
Es una forma de pensar.

───────────────────────────────────────────────────────────
PRINCIPIOS DEL CONTROL DE VERSIONES
───────────────────────────────────────────────────────────

PEQUEÑOS PASOS: Guarda cambios pequeños y frecuentes.

Cada "commit" debe ser una unidad lógica de cambio.

No guardes todo al final.

MENSAJES CLAROS: Escribe mensajes que expliquen el cambio.

"Corregir error en el formulario" es mejor que "cambios".

RAMA PRINCIPAL SIEMPRE FUNCIONA: La rama principal (main) debe estar siempre funcionando.

No subas código roto.

EXPERIMENTA EN RAMAS: Crea ramas para nuevas funcionalidades.

Si falla, no afecta a la rama principal.

REVISA EL CÓDIGO: Antes de integrar cambios, revisa el código de otros.

───────────────────────────────────────────────────────────
¿QUÉ PASA SI UN PROYECTO NO TIENE VERSIONES?
───────────────────────────────────────────────────────────

Caos: Nadie sabe qué versión es la última.

Pérdida de trabajo: Si algo se rompe, no puedes volver atrás.

Conflictos: Dos personas modifican el mismo archivo.

Sin colaboración: Es difícil trabajar en equipo.

Sin historia: No sabes cómo evolucionó el proyecto.

PARTE 8: ERRORES COMUNES EN EL CONTROL DE VERSIONES
===============================================================================

───────────────────────────────────────────────────────────
ERROR 1: NO USAR CONTROL DE VERSIONES
───────────────────────────────────────────────────────────

❌ Incorrecto:

text
proyecto/
├── index.html
├── index_v2.html
├── index_final.html
└── index_final_v2.html
✅ Correcto:

text
proyecto/
├── index.html
└── .git/
───────────────────────────────────────────────────────────
ERROR 2: MENSAJES DE COMMIT VAGOS
───────────────────────────────────────────────────────────

❌ Incorrecto:

text
cambios
cosas
arreglos
✅ Correcto:

text
Corregir error de validación en el formulario de contacto
Añadir sección de testimonios en la página principal
Optimizar imágenes para mejorar el rendimiento
───────────────────────────────────────────────────────────
ERROR 3: SUBIR CÓDIGO ROTO A LA RAMA PRINCIPAL
───────────────────────────────────────────────────────────

❌ Incorrecto:

text
git commit -m "cambios"
git push origin main
✅ Correcto:

text
# Probar el código localmente antes de subirlo
# Usar ramas para experimentos
git checkout -b nueva-funcionalidad
git commit -m "Añadir nueva funcionalidad"
git push origin nueva-funcionalidad
───────────────────────────────────────────────────────────
ERROR 4: NO HACER PULL ANTES DE TRABAJAR
───────────────────────────────────────────────────────────

❌ Incorrecto:

text
# Empezar a trabajar sin descargar los últimos cambios
✅ Correcto:

text
# Siempre descargar los últimos cambios antes de empezar
git pull origin main
───────────────────────────────────────────────────────────
ERROR 5: SUBIR ARCHIVOS GRANDES O SENSIBLES
───────────────────────────────────────────────────────────

❌ Incorrecto:

text
git add contraseñas.txt
git commit -m "contraseñas"
✅ Correcto:

text
# Usar .gitignore para excluir archivos sensibles
# .gitignore
contraseñas.txt
*.log
node_modules/
