1. ¿QUÉ ES LA DEPURACIÓN?
===============================================================================

Depurar (debugging) es el proceso de encontrar y corregir errores.

No es magia. Es una habilidad que se aprende.
Y la herramienta principal son los breakpoints.

📌 **Analogía del mecánico**

Imagina que tu coche hace un ruido extraño.

Podrías abrir el capó y mirar fijamente el motor.
Eso es como leer el código sin ejecutarlo.

Podrías poner sensores en todas partes y leerlos después.
Eso es como console.log.

O podrías hacer una prueba de conducción con un mecánico al lado
que va revisando cada pieza mientras el coche está en marcha.
Eso es un breakpoint.

---

📌 **console.log vs breakpoints**

console.log	                    Breakpoints
✅ Simple y rápido	            ❌ Requiere configuración
❌ Tienes que saber qué buscar	✅ Ves todo
❌ Llenas la consola de basura	✅ Solo ves lo relevante
❌ Tienes que modificar código	✅ No modificas código
❌ No puedes pausar la ejecución	✅ Pausas y examinas

===============================================================================
2. HERRAMIENTAS DE DESARROLLO (DEVTOOLS)
===============================================================================

Todos los navegadores modernos tienen herramientas de desarrollo.

📌 **Cómo abrir DevTools**

Chrome / Edge / Firefox:
- F12
- Ctrl + Shift + I (Windows/Linux)
- Cmd + Option + I (Mac)
- Clic derecho → Inspeccionar

📌 **La pestaña Sources (Fuentes)**

Esta es tu herramienta principal para debugging.

Contiene:
1. **File Navigator**: lista de archivos del sitio.
2. **Code Editor**: muestra el código del archivo seleccionado.
3. **Debug Controls**: botones para controlar la ejecución.
4. **Watch**: variables que quieres monitorear.
5. **Call Stack**: pila de llamadas de funciones.
6. **Scope**: variables locales y globales actuales.

===============================================================================
3. BREAKPOINTS: CÓMO USARLOS
===============================================================================

Un breakpoint le dice al navegador:
"Pausa la ejecución en esta línea para que pueda inspeccionar".

---

📌 **Cómo colocar un breakpoint**

1. Abre DevTools (F12).
2. Ve a la pestaña Sources.
3. Busca tu archivo JavaScript.
4. Haz clic en el número de línea donde quieres pausar.
5. Aparece una marca azul → breakpoint activo.
6. Ejecuta tu programa.
7. El código se pausa en esa línea.

---

🔵 **Ejemplo 1: Primer breakpoint**

```javascript
// Código para probar breakpoints
let a = 10;
let b = 20;

// Coloca un breakpoint en la línea de suma
let suma = a + b;

console.log("La suma es: " + suma);

if (suma > 25) {
    console.log("La suma es mayor que 25");
} else {
    console.log("La suma es menor o igual que 25");
}

Cuando el código se pause en la línea de suma, podrás:
- Ver que a = 10 y b = 20 en el panel Scope.
- Ver que suma aún no está calculada.
- Avanzar una línea y ver cómo suma cambia a 30.

===============================================================================
4. CONTROLES DE DEPURACIÓN
===============================================================================

Una vez que el código está pausado, puedes controlar su ejecución.

📌 **Los botones de control**

Botón	                Atajo	        Función
▶️ Resume (Continuar)	F8	            Continúa hasta el próximo breakpoint
⬇️ Step Over (Saltar)	F10	            Ejecuta la línea actual (sin entrar a funciones)
⬇️ Step Into (Entrar)	F11	            Entra a la función que se está llamando
⬆️ Step Out (Salir)	    Shift+F11	    Sale de la función actual
🔁 Step (Paso)	        F9	            Siguiente paso (Firefox)

---

🔵 **Ejemplo 2: Step Over vs Step Into**

```javascript
function duplicar(n) {
    let resultado = n * 2;
    return resultado;
}

let x = 5;
let y = duplicar(x);  // Breakpoint aquí
console.log(y);

// Step Over (F10): ejecuta duplicar(x) sin entrar a la función.
// Step Into (F11): entra a duplicar() y ves paso a paso cómo se ejecuta.

¿Cuándo usar cada uno?
- Step Over: cuando confías en que la función funciona.
- Step Into: cuando crees que el error está DENTRO de la función.

===============================================================================
5. INSPECCIÓN DE VARIABLES
===============================================================================

Cuando el código está pausado, puedes ver el valor de las variables.

📌 **Panel Scope (Ámbito)**

Muestra las variables disponibles en el punto de pausa:
- Local: variables de la función actual.
- Block: variables del bloque actual (let/const en if, for, etc.).
- Global: variables globales (window, document, etc.).

📌 **Watch (Vigilancia)**

Puedes agregar expresiones para monitorear:

1. En el panel Watch, haz clic en +.
2. Escribe una expresión, por ejemplo: suma > 25.
3. DevTools muestra true o false según el valor actual.

Útil para:
- Ver cómo cambia una variable en cada iteración.
- Evaluar condiciones sin poner console.log.
- Monitorear expresiones complejas.

---

🔵 **Ejemplo 3: Depurar un bucle con Watch**

```javascript
let numeros = [10, 20, 30, 40, 50];
let suma = 0;

// Pon un breakpoint en esta línea
for (let i = 0; i < numeros.length; i++) {
    suma += numeros[i];
    console.log("i: " + i + ", suma: " + suma);
}

En el Watch, agrega:
- i
- numeros[i]
- suma

Cada vez que avances (F10), verás cómo cambian los valores.

===============================================================================
6. BREAKPOINTS CONDICIONALES
===============================================================================

A veces solo quieres pausar cuando se cumple una condición.

Por ejemplo, en un bucle de 100 iteraciones,
solo te interesa la iteración 50.

📌 **Cómo crear un breakpoint condicional**

1. Haz clic derecho en el número de línea.
2. Selecciona "Add conditional breakpoint".
3. Escribe la condición, por ejemplo: i === 50.
4. El breakpoint solo se activa cuando i es 50.

---

🔵 **Ejemplo 4: Breakpoint condicional en bucle**

```javascript
// Pon un breakpoint condicional: i === 3
for (let i = 0; i < 10; i++) {
    console.log("Iteración: " + i);
}

// El código solo se pausa en i = 3.
// Las iteraciones 0, 1, 2 no pausan.
// Las iteraciones 4-9 tampoco.

🦉 **Profesor Búho dice:**
"Los breakpoints condicionales son perfectos para bucles grandes.
Si tienes un error en la iteración 999 de 1000,
no quieres presionar F10 999 veces."

---

🔵 **Ejemplo 5: Depurar un if...else**

```javascript
let nota = 85;

// Pon un breakpoint aquí
if (nota >= 90) {
    console.log("A");
} else if (nota >= 80) {
    console.log("B");
} else if (nota >= 70) {
    console.log("C");
} else {
    console.log("D");
}

Cuando el código se pause en el if, puedes:
- Ver que nota es 85.
- Saber que nota >= 90 es false.
- Saber qué rama se va a ejecutar (B).

---

🔵 **Ejemplo 6: Depurar un bucle for**

```javascript
let frutas = ["manzana", "pera", "uva", "naranja", "sandía"];

// Pon un breakpoint aquí
for (let i = 0; i < frutas.length; i++) {
    console.log("Fruta " + i + ": " + frutas[i]);
}

Con Watch configurado para i y frutas[i]:
- i: 0, 1, 2, 3, 4
- frutas[i]: "manzana", "pera", "uva", "naranja", "sandía"

LA ANALOGÍA DEL PROFESOR BÚHO: EL CIRUJANO
===============================================================================

Imagina una cirugía.

console.log es como poner una cámara en la sala de operaciones.
Ves lo que pasa, pero no puedes detener la operación ni examinar detalles.

Breakpoints es como detener la cirugía en un momento clave para examinar:
- "¿El pulso del paciente es correcto?" (Scope).
- "¿La presión sanguínea es estable?" (Watch).
- "¿Qué pasó antes de llegar aquí?" (Call Stack).

```javascript
function cirugia(paciente) {
    console.log("🧑‍⚕️ Comenzando cirugía de " + paciente);
    let pulso = 72;

    // Pon un breakpoint aquí
    let presion = 120 / 80;

    if (pulso > 100) {
        console.log("⚠️ Pulso elevado");
    }

    console.log("✅ Cirugía completada");
}

cirugia("Lucas");
```

Con los breakpoints, eres el cirujano que puede detener
la operación en cualquier momento para examinar al paciente.

===============================================================================
ERRORES COMUNES EN DEPURACIÓN
===============================================================================

❌ **Error 1: Poner breakpoints en la línea incorrecta**

Si pones el breakpoint en una línea vacía, no funcionará.
Colócalo siempre en una línea con código ejecutable.

❌ **Error 2: Olvidar quitar los breakpoints**

A veces pones un breakpoint, olvidas quitarlo,
y luego el código se pausa "misteriosamente".

Solución: Revisa la pestaña Sources → Breakpoints.
Puedes desactivar o eliminar todos.

❌ **Error 3: No refrescar el código**

Si modificas tu archivo JavaScript, DEBES recargar la página
para que los cambios se apliquen.

❌ **Error 4: Usar solo console.log**

console.log está bien para cosas simples.
Pero para errores complejos, breakpoints son mucho más poderosos.

❌ **Error 5: Pánico al ver muchos breakpoints**

Cuando tienes varios breakpoints activos, el código se pausará
en cada uno. Es normal. Usa F8 para saltar al siguiente
o elimina los que ya no necesites.

Crea una función que calcule el factorial de un número.
Coloca un breakpoint dentro de la función y usa Step Into.

Solución:
```javascript
function factorial(n) {
    if (n <= 1) return 1;  // Breakpoint aquí
    return n * factorial(n - 1);
}

let resultado = factorial(5);
console.log("Factorial: " + resultado);

📝 **Ejercicio 3: Breakpoint condicional**

Crea un bucle de 1 a 100. Pon un breakpoint condicional
que se active solo cuando i sea múltiplo de 10.

Solución en DevTools:
1. Clic derecho en el número de línea del bucle.
2. "Add conditional breakpoint".
3. Condición: i % 10 === 0.

📝 **Ejercicio 4: Inspecciona el Call Stack**

Crea tres funciones que se llamen entre sí: A → B → C.
Pon un breakpoint en C y observa el Call Stack.

Solución:
```javascript
function A() { B(); }
function B() { C(); }
function C() {
    console.log("Dentro de C");  // Breakpoint aquí
}

A();
// Call Stack: C → B → A → (anonymous)

📝 **Ejercicio 5: Depura un error**

Introduce un error intencional en tu código y usa
breakpoints para encontrarlo (sin mirar el mensaje de error).

Solución:
```javascript
function calcularPromedio(notas) {
    let suma = 0;

    for (let i = 0; i <= notas.length; i++) {  // ❌ Error: <= en lugar de <
        suma += notas[i];
    }

    return suma / notas.length;
}

let notas = [80, 90, 100];
let promedio = calcularPromedio(notas);
console.log("Promedio: " + promedio);

// El error: notas[3] es undefined, suma se vuelve NaN.
// Con breakpoints, verías que i llega a 3 cuando notas.length es 3.
```

===============================================================================
PREGUNTAS FRECUENTES (FAQ)
===============================================================================

❓ **¿Los breakpoints modifican mi código?**

No. Los breakpoints son solo marcas en DevTools.
Tu código original no se modifica.

❓ **¿Puedo poner breakpoints en HTML o CSS?**

Sí, pero en HTML solo en líneas con JavaScript.
En CSS puedes depurar estilos en la pestaña Styles.

❓ **¿Los breakpoints funcionan en todos los navegadores?**

Sí. Chrome, Firefox, Edge, Safari tienen DevTools.
La interfaz varía, pero los conceptos son los mismos.

❓ **¿Qué hago si el breakpoint no se activa?**

1. Verifica que el archivo esté correctamente cargado.
2. Refresca la página.
3. Verifica que el código realmente se ejecute (no esté en una rama muerta).

❓ **¿Puedo depurar código en producción?**

Sí, pero ten cuidado. Los breakpoints pausan la ejecución
para todos los usuarios en esa sesión de DevTools.
No afecta a otros usuarios.
